## Находка 1. D65 internally inconsistent (моя недоделанная правка)

В `spec/decisions/04-effects.md` D65 (Fail) — Правило 1 я переписал
в раунде 49 на «три разные формы»:

> «`Fail` ≠ `Fail[any]`:
> - `Fail` — программист **не указал** тип, ждёт inference.
> - `Fail[any]` — программист **явно** просит erasure.»

Но **остальной D65 не обновлён** под эту формулировку:

- Правило 2, п. 2: «Catch-all — handler `Fail` (≡ `Fail[any]`)» — т.е.
  обратное Правилу 1.
- Правило 5 (matrix транзитивности): «Caller declared `Fail` (any)» —
  трактует `Fail` без параметра как `Fail[any]`.
- Эволюция: «`Fail` без параметра теперь сахар над `Fail[any]`».

Также **D25** (строки 1003, 1012, 1021–1024) и **`spec/effects.md`**
исходят из старой формулировки `Fail ≡ Fail[any]`.

И **твоя реализация раунда 5** (commit `284b20743`) тоже идёт по
старой семантике — в коммит-сообщении: «E ... не выводится точным
типом — добавляется голый Fail (placeholder по D65)». То есть
компилятор добавляет `Fail` без E, что эквивалентно `Fail[any]`
по runtime.

### Что предлагаю

**Откатить Правило 1 к старой формулировке**: `Fail ≡ Fail[any]`,
плюс отдельная заметка про D28 inference (если private fn делает
throw без `Fail` в effect-row, компилятор добавляет голый `Fail` —
это `Fail[any]` по runtime).

Моя «три разные формы» переусложнённая — в практике `Fail` без
параметра везде означает «catch-all», а D28 inference это
ортогональный механизм (про **наличие** Fail в effect-row, не про
его параметр).

**Если согласен — я фикшу spec.** Это ~10-15 правок в D65 + D25
(вернуть согласованность). Никаких правок в твоей реализации не
требуется — она уже соответствует «`Fail` ≡ `Fail[any]`».

**Если не согласен** (хочешь именно три формы как первоклассную
семантику) — скажи, и я довожу формулировку до D25/D62/effects.md/
spec.effects.md и пишу как должна выглядеть инференс-форма
точнее (передача E через D28 для private fn).

## Находка 2. syntax.md — gap по concurrency-примитивам

`spec/syntax.md` содержит секцию **«spawn / supervised / parallel for
/ detach»** (строка 1244), но **отсутствуют** секции для:

- `cancel_scope { tok => body }` (D75 — реализован 2026-05-06)
- `Channel[T]` (D79 — base реализован раунд 5)
- `select { ... }` (D79 — отложен до spawn-block fix)

Полная семантика всех трёх — в `decisions/06-concurrency.md`. Но
`syntax.md` — это **общая грамматическая справка**, к ней первой
обращается LLM и человек когда учат синтаксис. Сейчас она отстаёт
на 3 решения.

### Что предлагаю

Добавить три коротких параграфа в `syntax.md` после секции про
`detach { body }` (строка 1359). Структура — как у существующих
секций: 5-15 строк с минимальным примером + ссылка на D-decision
для деталей. Канонические примеры уже есть в decisions/06.

**Это моя задача** (spec-сторона). Сделаю одним коммитом если ты
согласен, что место правильное. Альтернатива — оставить как есть
(decisions/06 — единственный источник для concurrency); тогда
syntax.md явно отметить пометкой «полная семантика concurrency —
см. decisions/06».
