# Compiler conventions — как разрабатывать компилятор Nova

Этот документ **директивный**, для всех контрибьюторов (включая AI-агентов). Описывает
**архитектурные принципы и инженерные конвенции** разработки самого компилятора
(`compiler-codegen/` — front-end + codegen на Rust), а не язык Nova.

- Язык Nova (как писать `.nv`) → [`nv-coding-style.md`](nv-coding-style.md).
- Как принимать дизайн-решения языка → [`project-philosophy.md`](project-philosophy.md).
- Нормативная семантика → `spec/decisions/` (D-блоки).
- Тесты → [`test-conventions.md`](test-conventions.md). Перф → [`perf-conventions.md`](perf-conventions.md).
- Языковой сервер (`nova-lsp/`) → [`lsp-conventions.md`](lsp-conventions.md) (наследует этот файл).

---

## 0. Главный принцип: один источник истины для типов

> **Типы резолвятся РОВНО ОДИН РАЗ — в type-checker'е. Он владелец полной картины.
> Codegen ему доверяет и только опускает. C-компилятор НИКОГДА не должен ловить
> ошибку типов — если поймал, это баг фронтенда.**

Зрелые компиляторы (Rust, Go, Swift) устроены так:

```
parse → [SEMANTIC ANALYSIS: резолвит ВСЁ один раз] → полностью типизированный IR → codegen → backend
```

Семантический проход строит **полностью типизированное/проверенное дерево**: у каждого
выражения известен тип, каждый вызов разрешён к конкретному callee (перегрузки выбраны
**по типам**), каждая неявная конверсия либо проверена, либо вставлена явно. Codegen
лишь **лоуэрит уже проверенное** — не перерезолвит и не перепроверяет. Бэкенд ошибок
типов не видит.

**Куда движемся:** к этому единому проходу. Любая новая работа по типам/резолву должна
**сокращать** фрагментацию, а не углублять её.

### Анти-паттерн, который мы НЕ повторяем

Исторически (bootstrap) проверки размазались по трём слоям — это **техдолг**, не образец:

| Что проверяется | Где сейчас (плохо) | Где должно быть |
|---|---|---|
| Свободная fn (одна сигнатура) | Nova-чекер (`E7301`) ✅ | чекер |
| Перегрузки fn/методов по типам | **codegen-резолвер** (`no matching overload`) | чекер |
| Категориальное несовпадение у методов (str↔int) | **C-компилятор** (`CC-FAIL`) | чекер |
| Scalar-narrowing в method-arg | **никто** (дыра) | чекер |

Корень: у Nova **нет** единого полного типизирующего прохода — частичный «best-effort»
чекер + codegen, доводящий типы для своих нужд + C как подстраховка. Три слоя знают
**часть** информации, ни один — всю.

## 1. Где какие проверки живут

1. **Type-checker (`types/mod.rs`)** — владелец **soundness**: типы, эффекты, владение,
   видимость, narrowing, резолв вызовов/перегрузок. Сюда добавляется ЛЮБАЯ новая
   проверка корректности.
2. **Codegen (`codegen/emit_c.rs`)** — **только лоуэринг** уже проверенного. Codegen НЕ
   должен быть местом, где soundness-правило enforce'ится впервые. Если правило живёт
   только в codegen (как D84-overload-резолв) — это долг к переносу в чекер, а не норма.
3. **C-компилятор** — **sanity-net, никогда не первичный чекер**. «C поймает» — запрещённое
   обоснование пропуска проверки. Если на корректный фронтенд C выдаёт ошибку типов —
   это баг фронтенда, заводим issue.

**Правило:** добавляя проверку, спроси «может ли это поймать чекер?». Если да — там и место.
Перенос проверки из codegen/C в чекер — всегда улучшение (чище диагностика, единый путь).

## 2. Никакого хардкода типов, функций и методов Nova

> **Имена и сигнатуры типов / функций / методов Nova НЕ зашиваются в Rust-код
> компилятора. Они резолвятся из своих объявлений (`.nv`-исходник и реестры,
> построенные из него), а не из хардкоженных `match`-веток или списков.**

**Почему:**

- Хардкод даёт встроенным сущностям привилегию, недоступную пользовательскому коду →
  **форкает/ломает** user-код: пользовательский `Stack[T]` обязан вести себя как `Vec[T]`,
  пользовательский метод — проверяться как встроенный. Проверка/резолв для `Vec` и для
  `MyType` должны идти **одним механизмом**.
- Хардкоженные списки **дрейфуют** от реальности (списки методов в LSP, method-таблицы,
  имена в codegen) → устаревают и врут.
- Противоречит §0 (единый источник истины) и принципу «общий фикс, не X-специфичный».

**Правило:**

- Новая проверка/резолв работает над **общим механизмом** (любой тип/fn/метод), управляемым
  объявлениями/реестрами — **никогда** через name-keyed спец-кейс под конкретный builtin.
- Примеры из проекта: проверка generic-arg mismatch ловит **любой** generic-тип (`Vec`,
  user `Stack`, `Option`) — не захардкожена под `Vec`. LSP обязан резолвить методы из `.nv`,
  а не из захардкоженного списка (такой список был удалён).

**Исключения (минимально возможные):**

- Допускаются **только** для настоящих языковых builtin'ов, у которых нет `.nv`-объявления
  (например: sugar-алиас `[]T ≡ Vec[T]` (D239); модель ширины/знака примитивов; `int`≡`i64`).
- Любое исключение требует: (1) **явного согласования с владельцем проекта**, и
  (2) **документального оформления** — D-блок / маркер `[M-*]`, фиксирующий **что** именно
  захардкожено и **почему** общий механизм неприменим.
- **Дефолт — «без хардкода».** Исключение нужно **доказать и согласовать**, а не предположить.

## 3. Никаких тихих дыр (`_ => return` — под маркер)

«Best-effort пропуск» (`match … { _ => return }`, «проверим позже», «обычно ловит C»)
создаёт soundness-дыры, проваливающиеся между слоями (как scalar-narrowing в method-arg,
Plan 175.2).

**Конвенция:**

- Если проверку нельзя сделать сейчас — это **не молчаливый skip**, а **зафиксированный
  gap**: маркер `[M-*]` в коде + запись в [`backlog-followups.md`](backlog-followups.md)
  с точной формулировкой (что именно не покрыто и почему).
- **Не выдумывай покрытие.** Формулируй дыру **узко и проверенно** (эмпирически), а не
  «методы не типизируются вообще». Неточная формулировка = неверный план.
- Permissive-ветки (`Other`, alias/generic/unknown → пропуск) допустимы **только** там, где
  проверка принципиально неразрешима, и это прокомментировано.

## 4. Spec-first

- Поведение решается в `spec/decisions/` (D-блок), **потом** кодируется. Код **конформит**
  спеке, не наоборот. Расхождение spec↔impl — баг (чинится одна из сторон, осознанно).
- Каждое наблюдаемое поведение (новый код ошибки, правило конверсии, диагностика) →
  отражено в D-блоке + error-index ([`09-tooling.md`](../spec/decisions/09-tooling.md)).
- Никогда не выдумывай синтаксис/семантику — смотри `spec/` и `examples/` ([feedback]).

---

## 5. Диагностика

- Код ошибки — inline-префикс сообщения: символьный `[E_SCREAMING_SNAKE]` (предпочтительно
  для новых) или числовой `[Ennnn]` (call-arg/type — legacy). Пример: `[E_IMPLICIT_NARROWING]`.
- Сообщение: что не так + **как починить** (`use \`x as u8\``). Span — на провинившееся
  выражение; `.with_note_at(...)` — на объявление/контекст (`parameter \`x\` declared here`).
- Каждый новый код → в error-index спеки. Сообщения — производственного качества (как для
  пользователя языка), не «TODO: fix message».

## 6. Методология изменения чекера/codegen (де-риск ПЕРЕД правкой)

Soundness-правка системы типов рискованна (легко сломать std или насыпать ложных ошибок).
Порядок:

1. **Карта (understand).** Найти ТОЧНУЮ точку интеграции (file:line), как туда попадают
   нужные данные (raw `TypeRef`, scope, generic-scope), где параллельные пути. Для крупного
   — параллельные агенты-ридеры (workflow), но решение принимаешь ты.
2. **Замер blast-radius ДО реализации.** Сколько сайтов в std/nova_tests новое правило
   сломает? Реализуй в **детект-режиме** (count/non-fatal), прогони корпус, посчитай. Это
   определяет объём миграции и саму жизнеспособность правила.
3. **Калибровка против ложных срабатываний.** Правило должно ловить **только** реальный
   класс бага. Permissive на неоднозначном (generic-параметры, alias, unknown). Цель —
   **ноль ложных срабатываний** на позитивном корпусе.
4. **Миграция зависимостей.** Если правило ломает легитимный код (std) — мигрируй его в том
   же изменении (или явными followup'ами там, где правка нетривиальна). Не оставляй красным.
5. **Верификация против ЧИСТОГО бинаря.** Падения отличай от pre-existing: `git stash` своей
   правки → rebuild → прогон тех же тестов. Совпало на чистом → не твоя регрессия.
   ([feedback-codegen-dce-verification]: baseline = kill-switch на ТОМ ЖЕ бинаре, не sibling.)
6. **Полный регресс, не сэмпл, для claim'а корректности.** Сэмпл — для скорости итерации;
   утверждение «ничего не сломал» требует широкого прогона + явного учёта pre-existing.

## 7. Тесты компилятора

- Каждое правило — **позитив + негатив**. Негатив через `EXPECT_COMPILE_ERROR <CODE>`
  ([test-conventions.md](test-conventions.md)), запуск с `--compile-error`.
- Проверять через **релизный** `nova` & компилятор (C-codegen pipeline), не интерпретатор.
- Негатив-фикстуры — в `neg/` (отдельный folder-module), чтобы не ломать позитивный
  folder-module соседей (folder = ОДИН модуль из co-equal файлов).
- Rust-unit-тесты компилятора (`cargo test`) — для изолированной логики (хелперы, резолверы).
- Регресс-гард на КАЖДЫЙ исправленный баг (фикстура, воспроизводящая исходное падение).

## 8. Процесс и гигиена коммитов

- **Spec/D → impl → tests → docs → error-index** — полный цикл, не половина.
- Коммит **на задачу**; несколько задач → несколько коммитов. `git add` **конкретных
  файлов** (рядом работают другие сессии — никогда `git add -A`).
- **Перед commit — `git diff --cached --stat`**: в индексе бывают pre-staged изменения чужих
  сессий; не утаскивай их в свой коммит ([feedback-verify-index-before-commit]).
- Не ломать std/прелюд молча: измеряй blast-radius, мигрируй зависимости.
- Маркеры `[M-*]` + `backlog-followups.md` для каждого осознанного пропуска; `simplifications.md`
  — для упрощений с обоснованием.
- `hot`-файлы (`emit_c.rs`, `types/mod.rs`) часто правят параллельно — проверяй `git status`,
  координируй регионы, не затирай чужие hunk'и.

## 9. Анти-паттерны (чек-лист «так нельзя»)

- ❌ Хардкод имён/сигнатур типов/функций/методов Nova в Rust (без согласованного и
  задокументированного исключения, §2) — особенно name-keyed спец-кейс под один builtin.
- ❌ «C поймает» как обоснование пропуска проверки в чекере.
- ❌ Soundness-правило, enforce'ящееся впервые в codegen (а не в чекере).
- ❌ Тихий `_ => return` без маркера и backlog-записи.
- ❌ Дублирование резолва типов между чекером и codegen.
- ❌ Утверждение о поведении без эмпирической проверки (тестом), «по памяти».
- ❌ Включение строгого правила без замера blast-radius и миграции зависимостей.
- ❌ Минимальный обход вместо правильного варианта ([project-philosophy] §«минимальный vs правильный»).

---

## 10. Резюме одной фразой

**Двигай резолв и проверки в чекер (единый источник истины), не хардкодь типы/функции/методы
Nova, не полагайся на C-подстраховку, не оставляй тихих дыр, измеряй blast-radius до правки и
верифицируй против чистого бинаря.**
