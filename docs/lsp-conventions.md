# LSP conventions — как разрабатывать nova-lsp

Этот документ **директивный**, для всех контрибьюторов (включая AI-агентов). Описывает
конвенции разработки **языкового сервера** (`nova-lsp/` — completion, hover, diagnostics,
goto, rename, …) поверх базовых конвенций компилятора.

- Базовые принципы компилятора → [`compiler-conventions.md`](compiler-conventions.md) (**обязательны и здесь**).
- Язык Nova → [`nv-coding-style.md`](nv-coding-style.md). Дизайн-решения → [`project-philosophy.md`](project-philosophy.md).
- План LSP V2 → [`plans/104.10-lsp-v2-production.md`](plans/104.10-lsp-v2-production.md).

## 0. LSP наследует конвенции компилятора

Следующие разделы [`compiler-conventions.md`](compiler-conventions.md) **обязательны** для nova-lsp
без исключений:

- **§0 единый источник истины** и **§2 никакого хардкода типов/функций/методов** — LSP был
  **худшим нарушителем** (захардкоженные списки методов в `completion.rs` удалены 2026-06-19).
- **§4 spec-first** (поведение — из спеки, не из догадок), **§5 диагностика** (качество сообщений),
  **§7 тесты**, **§8 гигиена коммитов**.

Где compiler-conventions.md и этот файл расходятся — данный файл **уточняет** под LSP, но
не отменяет базовых принципов.

## 1. Главный принцип: LSP — тонкий потребитель резолва компилятора

> **LSP не реализует и не хардкодит знание о языке (типы, методы, сигнатуры, completions,
> hover) — он ВЫВОДИТ его из резолва компилятора (единого источника истины, §0/§2).**

- Completion методов после `.`, hover, signature-help — из **реального типа** выражения
  (`expr_types` компилятора), а не из захардкоженного списка и не из текстовой эвристики.
  Захардкоженные таблицы методов/имён в `completion.rs` — **запрещены** (Plan 104.10 Ф.5).
- Пользовательские типы/методы должны предлагаться **так же**, как stdlib — единым механизмом
  (если работает только для `Vec`/builtin, но не для user-`Stack` — это нарушение §2).
- `compiler-codegen` подключается как **библиотека** (M-11): LSP переиспользует парсер/чекер/
  резолвер, не дублирует их логику. Дрейф между LSP-знанием и реальностью компилятора = баг.

## 2. LSP-специфика (чего нет у компилятора)

LSP работает в условиях, противоположных батч-компилятору, — отсюда отдельные правила:

### 2.1 Латентность важнее полноты

- Интерактивные запросы (completion/hover/signature) должны быть **быстрыми**: нельзя гонять
  полную компиляцию/codegen на каждый ввод. Тяжёлое — за `resolve`-провайдером / ленивым doc/detail.
- Дебаунс диагностик (≈200ms), инкрементальный rope, кэш документов — обязательны для отзывчивости.
- Бюджет латентности — критерий приёмки фич (см. Plan 104.10), а не «потом оптимизируем».

### 2.2 Degraded mode на невалидном коде

- Пользователь печатает → ввод **почти всегда синтаксически/семантически неполон**. LSP обязан
  давать полезный результат на **частичном** коде, не падать и не молчать.
- Partial-parse / error-recovery: при ошибке парсинга — максимально полное частичное дерево, а
  не «нет результата». Любой запрос на сломанном вводе — **graceful**, никогда паника.
- Это контраст с компилятором (§ compiler-conventions): тот требует валидный вход и имеет право
  на hard error; LSP — нет.

### 2.3 Инкрементальность и состояние

- Текст — `ropey`-rope; правки применяются инкрементально, не реparse всего файла без нужды.
- Позиции — **UTF-16** (LSP-протокол) ↔ байтовые офсеты Nova: маппинг обязателен и протестирован.
- Состояние документов — потокобезопасный кэш (`dashmap`); никакой глобальной мутабельной утечки
  между документами/запросами.

## 3. Диагностика и тесты LSP

- Диагностики LSP = диагностики **компилятора**, смаппленные в LSP-формат (коды/спаны/ноуты
  сохраняются). LSP **не выдумывает** свои сообщения об ошибках типов — это работа компилятора (§0).
- Каждая фича (completion/hover/goto/rename/action) — с тестами: позитив + edge (пустой ввод,
  курсор в комменте/строке, незавершённое выражение, offset за концом) → **никогда не паника**.
- Тесты — на реальном движке (компилятор-как-библиотека), не на захардкоженных ожиданиях имён
  (иначе тест закрепляет хардкод, который §2 запрещает).

## 4. Анти-паттерны (чек-лист «так нельзя»)

- ❌ Захардкоженные списки методов/типов/сигнатур/completions в LSP (вместо вывода из компилятора, §1/§2).
- ❌ Текстовая эвристика типа выражения там, где доступен реальный тип из `expr_types`.
- ❌ Спец-кейс под `Vec`/builtin, не работающий для пользовательских типов.
- ❌ Полная компиляция/codegen в интерактивном (per-keystroke) пути.
- ❌ Паника / пустой результат на синтаксически неполном вводе вместо graceful-degradation.
- ❌ Своё, отличное от компилятора, сообщение об ошибке типов.
- ❌ Тест, закрепляющий захардкоженные имена вместо проверки реального резолва.

---

## 5. Резюме одной фразой

**LSP — тонкий клиент над компилятором: выводи знание о языке из его резолва (не хардкодь и не
дублируй, §0/§2), будь быстрым и graceful на неполном коде, диагностику бери у компилятора.**
