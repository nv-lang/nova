# Conventions governance — изменения и отклонения ТОЛЬКО по согласованию

Этот файл управляет **всеми** конвенционными документами проекта Nova. Он нормативен и
сам подчиняется правилу ниже (правка этого файла — тоже по согласованию).

## Конвенционные документы (нормативные)

Обязательны для любого контрибьютора, включая AI-агентов:

- [compiler-conventions.md](compiler-conventions.md) — разработка компилятора (front-end + codegen).
- [test-conventions.md](test-conventions.md) — написание и запуск тестов.
- [nv-coding-style.md](nv-coding-style.md) — стиль кода на Nova (`.nv`).
- [perf-conventions.md](perf-conventions.md) — производительность компилятора.
- [bench-conventions.md](bench-conventions.md) — бенчмарки.
- [lsp-conventions.md](lsp-conventions.md) — языковой сервер (наследует compiler-conventions).
- [project-philosophy.md](project-philosophy.md) — принятие дизайн-решений языка.
- `spec/decisions/` (D-блоки) — нормативная семантика языка (spec-first).

## Главное правило

> **Конвенции НОРМАТИВНЫ. Любое ИЗМЕНЕНИЕ конвенционного документа и любое ОТКЛОНЕНИЕ от
> конвенции — ТОЛЬКО по явному согласованию с владельцем проекта. Не править и не
> отклоняться самовольно.**

Конкретно:

- **Изменение текста конвенции** (правка/добавление/смягчение/ужесточение правила) — сначала
  согласовать с владельцем, потом править. Коммит правки конвенции упоминает согласование.
- **Отклонение в коде/тестах** от конвенции — только как **согласованное исключение**,
  задокументированное на месте (комментарий + маркер `[M-*]`, где уместно) со ссылкой на причину.
- **Нашёл ошибку / противоречие / устаревание** в конвенции — НЕ чинить молча. Вынести владельцу
  с конкретным предложением (что не так + варианты + рекомендация), получить решение, затем править.
- **Конвенция vs реализация инструмента** (напр. `test-conventions.md` ↔ `test_runner.rs`): перед
  правкой конвенции проверить, что она не противоречит фактическому поведению инструмента;
  расхождение — повод согласовать, какую сторону менять.
- **Спека (`spec/decisions/`)** — spec-first: поведение решается в D-блоке по согласованию, потом
  кодируется (compiler-conventions §5).

## Почему

Конвенции — единый источник правды процесса. Самовольная правка/отклонение их форкает: разные
агенты следуют разным версиям, фикстуры/код дрейфуют, теряется доверие к документу. Согласование
держит конвенции консистентными и авторитетными.

## Лог согласованных изменений конвенций

Существенные согласованные правки конвенций — здесь, с датой и сутью (для аудита).

- **2026-06-20 · test-conventions.md** — устранены два внутренних противоречия (правило↔пример),
  смягчением (согласовано с владельцем; проверена совместимость с `test_runner.rs` — классификация
  по маркеру `EXPECT_*`, группировка TU по равенству `module`-деклараций; раннер НЕ читает суффикс
  `_neg` и имя папки `neg/`):
  - `test`-блок в `EXPECT_COMPILE_ERROR`-файле **допустим** (файл не компилируется → блок не
    исполняется; раннер проверяет ошибку на этапе codegen независимо от блока).
  - Суффикс `_neg` **необязателен внутри `neg/`** (путь `neg/` + `module neg.<name>` уже сигналят);
    **рекомендуется/обязателен для neg-файлов ВНЕ `neg/`** (где иного сигнала нет).
