================================================================================
КАК СОЗДАВАЛСЯ NOVA-LANG
================================================================================

Дневник работы над языком программирования Nova. Формат: по шагам, от
первой идеи до текущего состояния. Цель — дать материал для статьи и
для понимания принятых решений.

Среда: Claude Code (Sonnet 4.6), Windows, MSVC, Rust.

================================================================================
ПРЕДЫСТОРИЯ
================================================================================

Nova — новый язык программирования, созданный с нуля. Цель: язык для
прагматичных разработчиков с чистой моделью эффектов, без исключений,
со структурированным concurrency и GC-управлением памятью.

Ключевые идеи:
  - Эффекты вместо исключений (Fail, Async, IO — все через эффекты)
  - spawn { } вместо async/await (структурированный concurrency)
  - Records + sum types, pattern matching
  - Интерпретируемый + компилируемый через C-бэкенд

================================================================================
1. СПЕЦИФИКАЦИЯ ЯЗЫКА (spec/)
================================================================================

Написана параллельно с реализацией. Ключевые решения:

D6  — realtime{} блоки как альтернатива `timing` эффекту
D14 — `spawn { }` как выражение, возвращающее результат
D50 — убраны `async`/`await`, весь concurrency через spawn + Async эффект
D59 — паттерн-матчинг на массивах: `[x, ..]`, `[head, ..rest]`
D60 — spread в массивах: `[0, ...arr, 4, 5]`
D61 — handler keyword, прямой вызов методов handler-объекта (`h.flip()`)
D62 — Async как ambient-эффект, не требует явного объявления
D66 — Self universal: `Self` в типах методов указывает на тип реализатора
D67 — `?` оператор через Fail эффект и ранний return из Option
D68 — stateful handlers через closure capture или record + @method
D69 — variadic параметры `...items []T`
D70 — ToStr протокол + @to_str() метод

2. КОМПИЛЯТОР-BOOTSTRAP (compiler-bootstrap/)
================================================================================

Первая реализация: лексер, парсер, AST, type checker, treewalk-интерпретатор.
Написан на Rust. Покрывает весь синтаксис Nova достаточно для прототипирования.

================================================================================
3. C-БЭКЕНД (compiler-codegen/)
================================================================================

Форк compiler-bootstrap с добавленным модулем codegen/emit_c.rs.
Компилирует Nova AST → C → нативный бинарь через MSVC.

Runtime (nova_rt/):
  - alloc.{h,c}   — три реализации: malloc, ref-counting, Boehm GC
  - fibers.{h,c}  — stackful coroutines через minicoro (header-only, x64 asm)
  - effects.{h,c} — Fail/interrupt механизм через setjmp/longjmp + NovaTestFrame
  - array.h       — NOVA_ARRAY_DECL/IMPL макросы для generic массивов
  - nova_rt.h     — единый include

Фазы реализации:

  Фаза 1 (синхронный subset): fn, let, арифметика, if/else, return, println
    Примеры: hello.nv, arithmetic.nv

  Фаза 2 (типы данных): records → C struct, sum types → tagged union, match
    Примеры: records.nv, match_demo.nv

  Фаза 3 (GC): три заменяемые реализации alloc.c
    alloc.c (malloc), alloc_rc.c (ref-counting), alloc_boehm.c (Boehm)

  Фаза 4 (эффекты): with/handler → thread-local vtable pointer + ctx struct
    D61: handler keyword, interrupt, прямой вызов h.flip() минуя with
    Захваченные переменные через ctx-struct + #define cap (*_c->cap)
    effects_d61.nv: Counter с return, Switch с interrupt, прямой vtable call
    Фиксы: empty ctx struct → char _dummy (MSVC), infer_expr_c_type для vtable,
    zero_literal_for_type для handler trailing NOVA_UNIT, last_is_return check

  Фаза 5 (fibers): spawn { body } → nova_fiber_run(entry, &ctx) через minicoro
    Закрытые переменные через ctx-struct с pointer fields
    Pre-scan для stable handler/spawn IDs (два прохода по AST)
    Pre-scan расширен на While/For/Loop/Match/Test тела
    Wildcard binding через fresh_tmp() вместо фиксированного "_nova_unused"

  Тест-блоки: test "name" { body } → статическая функция + setjmp-runner
    assert(expr) → nova_assert(cond, "expr text") с текстом условия
    Runner: N/M passed, exit(1) при провале
    Примеры: with_tests.nv (4 теста, 1 намеренный fail)
             gc_coroutines_test.nv (13 тестов — GC stress, spawn captures,
             spawn mutations, pipeline, nested, record fields, factorial)

  Фаза 6 (массивы): NovaArray_T через NOVA_ARRAY_DECL/IMPL макросы
    ArrayLit → nova_array_new_T + push loop
    Spread → inline while-loop копирования
    .len → arr->len, .get(i) → NovaOpt_T с NOVA_TAG_Option_Some/None
    Pattern::Array: [] / [x, ..] / [head, ..rest]
    Some(n)/None на NovaOpt_T value struct (не указатель)
    Пример: demo.nv (records, spread update, массивы, match на массиве, .get)

Ключевые технические решения:
  - MSVC-совместимость: нет пустых struct, нет nested function-scope typedef,
    нет non-constant static initializers
  - build_c.bat: объектники в %TEMP%\nova_build_%RANDOM% → нет коллизий имён
  - Генерируемые .c файлы не хранятся в git (в .gitignore)

9. STRUCTURED CONCURRENCY: supervised + cooperative scheduling (2026-05-06)
================================================================================

Минимальная реализация structured concurrency по D50.

ЦЕЛЬ: тесты interleave корутин в 38_deep_spawn.nv должны давать
наблюдаемое переключение fiber'ов, а не sequential subroutine calls.

РЕАЛИЗОВАНО:
  Runtime (nova_rt/fibers.h):
    - NovaFiberQueue { mco_coro* fibers[64]; int count; }
    - nova_fiber_spawn_into(q, entry, user) — создать fiber, в очередь без resume
    - nova_supervised_run(q) — round-robin resume пока все не MCO_DEAD

  AST + parser:
    - токен KwSupervised
    - ExprKind::Supervised(Block)
    - parse_supervised: `supervised { body }`

  Codegen:
    - current_scope_queue: Option<String> — активная очередь supervised
    - emit_supervised: открывает локальную NovaFiberQueue, в конце scheduler
    - emit_spawn маршрутизирует:
      * есть scope → nova_fiber_spawn_into(&queue, ...)
      * нет scope → старая eager-blocking nova_fiber_run (legacy)

  Time.sleep — yield-point:
    - builtin вместо полноценного эффекта (упрощение для bootstrap)
    - Time.sleep(N) для любого N → nova_fiber_yield(), ms игнорируется
    - корректно по D14/D62: Time — yield-point, sleep(0) ≈ runtime.Gosched в Go

  Capture access без #define (важный фикс):
    - старая схема: `#define cap (*_c->cap)` ломалась в nested supervised
      (cap-имя расширялось в struct field-declarator)
    - новая: current_spawn_captures — set имён, ExprKind::Ident при попадании
      эмитит `(*_c->name)` напрямую
    - bonus: nested spawn внутри fiber-body передаёт уже-указатель (_c->name),
      не пытается повторно `&` (это было бы &(*_c->name) что норм, но
      теперь без macro-расширения это тривиально)

  Прочие фиксы по дороге:
    - collect_idents_expr расширен: While, WhileLet, For, Loop, Match, Range,
      Lambda, ArrayLit, RecordLit, Spawn, With, Coalesce, Try, Index — раньше
      пропускались, capture-сборка работала случайно через Block trailing
    - infer_expr_c_type: While/Loop/For/Supervised → "nova_unit"
      (раньше fallback "nova_int", давало MSVC C2440 при cast в nova_int)
    - typedef ctx-struct → lambda_forward_decls (не deferred_impls), теперь
      виден до использования в spawn-instance

ТЕСТЫ (новая секция в 38_deep_spawn.nv, 9 штук):
  - "two fibers interleave at every yield"      log == 142536
  - "three fibers interleave round-robin"       log == 135246
  - "fiber without yield runs to completion"    log == 123456
  - "yield count equals interleave depth"       log == 123456
  - "yields uneven, fiber A finishes first"     log == 13245
  - "producer-consumer pattern via shared mut"  received == 15
  - "4 fibers, 1 yield each, round-robin"       log == 13572468
  - "yield inside loop, lifetime spans many"    order == 121212
  - "nested supervised — inner finishes first"  order == 123456

Все тесты проходят детерминированно. FIFO scheduler даёт точный observable order.

MAIN-LEVEL Time.sleep(0) (продолжение):
  - Time.sleep(0) теперь работает на уровне supervised-body (не только в fiber).
    Раньше main-flow между spawn'ами не мог yield'нуть → не было реального interleave
    с участием main.
  - Runtime: добавлена nova_supervised_step(q) — один round-robin pass очереди,
    возвращает количество живых fiber'ов. Используется как для main-yield'а внутри
    scope, так и внутри nova_supervised_run (do { step } while alive).
  - Codegen Time.sleep(0) теперь context-sensitive:
    * внутри fiber (current_spawn_captures.is_some()) → nova_fiber_yield()
    * вне fiber, внутри scope (current_scope_queue.is_some()) → nova_supervised_step
    * полностью вне → no-op
  - Тесты: nova_tests/42_main_yield.nv — 11 штук:
    * spawn + main sleep + read result
    * main interleaves with two spawns (1234675 trace)
    * counter ping-pong main↔fiber (123456 alternating)
    * fiber finishes early before main yields again
    * sleep(0) over empty queue is no-op
    * nested supervised — outer/inner queue isolation
    * producer-consumer in both directions (main as consumer, main as producer)
    * 5 fibers each yields twice, main drives 2 rounds (total = 165)
    * 10-round ping-pong (counter = 20)

PARALLEL FOR (продолжение):
  - Keyword `parallel for x in iter { body }`, ExprKind::ParallelFor.
  - Codegen — синтетический desugar в supervised { for x in iter { spawn { body } } }.
  - Capture-by-value для immutable scalars (int/bool/f64/byte):
    * spawn-capture теперь различает let vs let mut по var_mutable HashSet.
    * Immutable scalar → snapshot value в ctx struct (не pointer).
    * Mutable / non-scalar → pointer как раньше.
  - Heap-allocated ctx в supervised:
    * ctx-struct в spawn внутри scope аллоцируется через nova_alloc.
    * Без этого все queued fibers внутри loop разделяют stack-slot ctx →
      все видят последнее значение → bug.
    * Stack ctx сохраняется для eager-blocking spawn (вне scope).
  - Loop-var регистрируется в var_types при range-for (nova_int) — без
    этого capture-detection не находил переменную цикла.
  - Тесты (41_parallel_for.nv, 12 штук):
    * sum [1..5], sum-of-squares, empty/single
    * range 0..5 и 1..=10
    * snapshot: каждый fiber видит свой x
    * cooperative yield внутри parallel for body
    * nested: parallel for в for, обычный for в parallel for
    * mut-outer + immutable-inner mix

DETACH (продолжение в той же сессии):
  - Keyword `detach { body }`, AST::Detach(Block), парсер, interp-стаб.
  - Codegen: default SyncDetach handler = inline-block. Никакого fiber wrapping —
    body исполняется в потоке caller'а. Spec D50 явно описывает SyncDetach как
    тестовый default ("with Detach = SyncDetach { ... }"), bootstrap = это.
  - Эффект `Detach` НЕ объявлен; compile-time проверка требования отсутствует.
  - Тесты: nova_tests/40_detach.nv — 13 штук:
    * базовое исполнение, capture (ints, records, arrays)
    * detach с if/while внутри
    * detach внутри supervised — встречается inline ДО scope queue
    * nested detach (inner inline после outer-первого write)
    * sequential detach order
    * detach вне любого scope (валиден, в отличие от spawn)

ОТЛОЖЕНО (см. simplifications.md):
  - parallel for, race, select, cancel_scope, with_timeout
  - cancellation propagation, error-propagation между fibers
  - реальный async detach: OS-thread + global supervisor + LogAndDrop policy
  - effect Detach в effect-system + compile-time проверка
  - timer-wheel для Time.sleep(N>0)

PARALLEL FOR → []T (тест 50, 2026-05-06)
================================================================================

D71 array-mode parallel for.

Spec D14 говорит: parallel for x in iter { body } — это expression типа
[]T, где T — тип body. До этого момента bootstrap-codegen возвращал unit
(статусный desugar в supervised{for{spawn{...}}}), что нарушало spec.

Реализация:
  - В CEmitter добавлено поле current_parfor_slot: Option<(idx_var,
    result_var, elem_ty)>. Когда оно Some, emit_spawn:
      * добавляет в ctx два extra-field (_nova_par_idx, _nova_par_result)
      * копирует в ctx значения idx и pointer на result-array
      * trailing-expression body эмитируется как
        _c->_nova_par_result->data[_c->_nova_par_idx] = trailing
        вместо (void)(trailing)
  - emit_parallel_for детектирует array-mode по body.trailing.is_some().
    Если elem_ty не из {nova_int, nova_bool, nova_f64, nova_str} —
    degrade to statement-mode (unit).
  - Pre-allocate NovaArray_T* размера N (для Range — end-start[+1],
    для ArrayLit — длина литерала, для Ident — len arr_var->len).
  - Поддерживаются итераторы: Range a..b, RangeInclusive a..=b,
    ArrayLit (без spread), Ident bound к array.
  - infer_expr_c_type(ParallelFor) возвращает NovaArray_T* когда body
    имеет trailing, иначе nova_unit. Это включает proper let-binding
    propagation: let xs = parallel for ... { ... } получает корректный тип.
  - array_element_types регистрируется для result_var, чтобы xs[i]
    типизировался правильно при последующих операциях.

Тест: nova_tests/50_parallel_for_array.nv — 6 тестов:
  * range → []int
  * inclusive-range → []int
  * yield-stable ordering (Time.sleep внутри body не ломает порядок slots)
  * array-literal → []int
  * mut-capture одновременно с array-result
  * regression: без trailing — старая semantic (unit)

Регрессия: тест 41_parallel_for (12 тестов statement-mode) продолжает
проходить без изменений.

D75 CANCEL_SCOPE (тест 52, 2026-05-06)
================================================================================

cancel_scope { tok => body } — supervised-scope с first-class CancelToken.
Внешнее cancel из другого fiber'а через tok.cancel(); каскадная отмена
через tok.bind(parent).

Runtime (nova_rt/fibers.h):
- NovaCancelToken struct: scope-pointer + linked[NOVA_CANCEL_LINKED_CAP=8].
- nova_cancel_token_init/cancel/is_cancelled/bind функции.
- cancel(t): scope.cancel_requested=true (idempotent), walks linked[].

Codegen (emit_c.rs):
- Lexer: keyword cancel_scope (KwCancelScope).

HANDLER-WRAPPER PATTERN — D10 DEMO (2026-05-07)
================================================================================

Демонстрационный пример «всё — handler» (D10) — cleanup ресурсов через
функцию-обёртку с body-lambda, без defer/RAII.

with_resource_safe(id, t, body):
  1. opened += 1.
  2. body внутри `with Fail = handler { fail → caught + interrupt }`.
  3. На любом пути выхода (happy + throw) делает cleanup (closed += 1).
  4. Если был throw — re-throw наружу.

vs naive-версия (без внутреннего handler'а): на throw'е closed остаётся
0, ресурс leaked.

Тесты (nova_tests/effects/handler_wrappers.nv, 4 passing):
  - happy path safe + naive (oба opened==1, closed==1).
  - naive throw → leak (closed==0).
  - safe throw → cleanup (closed==1).

При написании выявлены bootstrap-ограничения (документированы в файле):
  1. mut в свободных fn-параметрах не парсится.
  2. fn T @method(...) Fail[E] -> R парсер не любит throw-методы.
  3. Trailing-block с non-int closure-параметром падает в codegen
     (нет type-erasure для closures).

     TurboFish; иначе rollback к Index.

     имеет comma).
   - Goldens: `xs[i].field` остаётся Index→Member (post-`]` это
     `.field` без `(`), `Type[K, V].method(...)` — TurboFish,
     `parse[int]("42")?` — TurboFish.

3. Codegen (emit_c.rs):
   - ExprKind::TurboFish arm в emit_expr — делегирует в
     self.emit_expr(base).
   - Expr::unwrap_turbofish() helper в AST — распаковывает узел.
   - Используется в:
     * emit_call_with_trailing (для mangling имени)
     * infer_expr_c_type (для возвращаемого типа call)
     * Stmt::Let (для generic_fn_tuple_arity lookup и bytes/chars
       array_element_types).

4. Interp: ExprKind::TurboFish arm — self.eval_expr(base, env).

5. Type-checker: has_throw_in_expr — TurboFish-arm проксирует в base.

6. Spec D38 в 03-syntax.md дополнен Bootstrap-секцией.

7. Тесты в nova_tests/types/generics.nv (4 шт.):
   - parse_one[int](42) — single-arg function turbofish
   - pair[int, str](7, "hi") — two-arg function turbofish
   - xs[0] / xs[1] + xs[2] — Index regression (no continuation)

D77 описывал 4-way auto-derive: программист пишет одну форму конверсии,

2026-05-08 Plan 04 зафиксирован: Buffer split на StringBuilder/WriteBuffer/ReadBuffer
================================================================================

Текущий унифицированный Buffer (Q-buffer ✅) смешивает text-domain и
binary-domain в одном типе. Обсуждение через несколько итераций
naming/семантики (add_*/append_*/write_*/put_*) выявило что главное —
не naming, а **разделение типов**.

Финальная архитектура (зафиксирована в docs/plans/04-buffer-split-and-external.md):

- StringBuilder — UTF-8 string accumulator. @into() -> str infallible
  (UTF-8 invariant поддерживается каждым @append).
- WriteBuffer — binary serialization. @write_byte/@write_bytes + 18
  endianness writes (u8/i8/u16/u32/u64/i16/i32/i64/f32/f64 × LE/BE).
  @into() -> []byte.
- ReadBuffer — binary read cursor. View над []byte. Pair @read_*
  (Fail[ReadBufferError]) / @try_read_* (Result), auto-derive на C-runtime
  уровне.

Plus:
- Новый keyword `external` для stdlib runtime-implemented функций.
  external только для функций; типы (StringBuilder/WriteBuffer/ReadBuffer)
  built-in opaque как примитивы.
- str.from(c char) — UTF-8 encode codepoint в str через D73 auto-pair.
- ReadBuffer.@into() — Nova-функция с throw'ом, блокирует D73 auto-derive.
- D30 расширение: «полные слова, не сокращения», `len`/`iter` mainstream
  exceptions.

Делать после Plan 08 (D73+D77 4-way auto-derive infrastructure уже
работает + 18 числовых типов registered — Plan 04 переиспользует).

Урок: длинная итерация дизайна (10+ поворотов в naming) показала что
«как назвать» — вторично; «какие домены/типы» — первично. Когда
зафиксировали split на 3 типа, naming решился сам (StringBuilder.@append,
WriteBuffer.@write_*, ReadBuffer.@read_*).
