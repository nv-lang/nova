## Scope

Production-grade реализация LSP rename и formatting для nova-lsp:

1. `textDocument/prepareRename` — validate cursor position (only identifiers).
2. `textDocument/rename` — cross-file rename через find-references; atomic post-rename check.
3. `textDocument/formatting` + `rangeFormatting` — invoke `nova fmt` (existing CLI).
4. `textDocument/onTypeFormatting` — auto-indent/close for key characters.

## Архитектурные решения

### D-block: Atomic rename contract

**D296 — LSP Rename Atomicity Contract (Plan 104.6)**

Операция rename в nova-lsp выполняется атомарно в три шаги:

1. **Prepare phase** (`prepareRename`): Cursor position → identifier span validation.
   - Only identifiers are renameable (not literals, comments, keywords).
   - Returns `PrepareRenameResponse::RangeWithPlaceholder { range, placeholder }`.
   - Error if position is not on an identifier.

2. **Collect phase** (`rename`): Find all references to the symbol.
   - Uses regex-based identifier scan across all open workspace documents.
   - Respects scope: generic params [T] only within their declaring fn/type scope.
   - Receiver methods: only renames the exact receiver-qualified method, not homonyms.
   - Cross-file: all open documents + files on disk in workspace root.

3. **Atomic check** (`rename` post-step): After computing edits, apply them to an
   in-memory copy of each file and re-typecheck. If ANY file fails type-check after
   the rename, the entire rename is rejected with `ResponseError` code -32803
   (RequestFailed) and an explanatory message listing the conflicting symbols.
   - This prevents rename from introducing `E_UNDECLARED` or name-collision errors.

**Conflict detection:** If new_name already exists in the same scope as old_name,
prepareRename is not affected (we can't know the new name yet), but the rename step
will catch it via the atomic post-check.

**Doc-comment links:** `[[old_name]]` patterns in doc-comments are updated to
`[[new_name]]` as part of the rename edits.

**WorkspaceEdit format:** Uses `documentChanges` array (with `TextDocumentEdit` +
`OptionalVersionedTextDocumentIdentifier`) for editor version tracking, not the
deprecated `changes` map.

### V1 Simplifications (documented)

- Rename uses regex-based occurrence scan (not full symbol table from type-checker).
  Full symbol-resolution rename deferred to V2 (requires compiler API extension for
  per-position symbol resolution across files).
- `onTypeFormatting` handles only `\n` (auto-indent) and `}` (dedent) in V1.
- `nova fmt` must be on PATH for formatting; graceful degradation if not found.
- rangeFormatting V1: formats entire file (nova fmt is a whole-file formatter),
  returns edits for only the requested range.

## Фазы

### Ф.1 — prepareRename (≤50ms, identifier validation)

**Реализация** в `nova-lsp/src/rename.rs`:
- `prepare_rename(text: &str, pos: Position) -> PrepareRenameResponse`
- Extract word at cursor position.
- Validate: must be ASCII-identifier chars (alphanumeric + `_`), not keyword, not digit-start.
- Return `RangeWithPlaceholder { range: word_range, placeholder: word }`.
- Error cases: position in comment, string literal, keyword, whitespace → `ResponseError`.

**Tests:** 4 pos + 4 neg (see test suite below)

### Ф.2 — rename cross-file (atomic, ≤2s/100 files)

**Реализация** в `nova-lsp/src/rename.rs`:
- `collect_renames(docs: &[(Uri, &str)], old_name: &str, def_uri: &Uri, def_range: Range) -> WorkspaceEdit`
- Scan each file for word occurrences of `old_name`.
- Scope guard: for generic params, only match within the declaring type/fn body span.
- Build `documentChanges` with `TextDocumentEdit` per file.
- Atomic check: apply edits to in-memory copies, run `check_source_inner` on each changed file.
  If any fails → return `ResponseError { code: -32803, message: "rename rejected: post-rename type errors in <file>: <diags>" }`.

**Tests:** 6 pos + 4 neg (see test suite below)

### Ф.3 — formatting (nova fmt invocation)

**Реализация** в `nova-lsp/src/format.rs`:
- `format_document(text: &str, nova_fmt_path: Option<&str>) -> Result<Vec<TextEdit>>`
- Spawn `nova fmt --stdin` (if supported) or write to temp file + invoke `nova fmt <file>`.
- Compare original vs formatted text → compute minimal text edits (single whole-file edit in V1).
- Timeout: 5s max.
- Graceful: if `nova fmt` not found → return empty Vec (no error, log warning).

**rangeFormatting V1:** call `format_document`, then clip the result edit to the requested range.

**Tests:** 3 pos + 2 neg

### Ф.4 — onTypeFormatting

**Реализация** в `nova-lsp/src/format.rs`:
- `on_type_format(text: &str, pos: Position, trigger_char: &str) -> Vec<TextEdit>`
- Trigger `\n`: detect indentation of current block (scan backward for `{` or `=>`), insert matching indent.
- Trigger `}`: dedent to match opening `{`.
- Trigger `,` and `;`: noop in V1 (return empty).

**Tests:** inline unit tests в format.rs

### Ф.5 — Server integration + capability advertisement

- Wire handlers in `server.rs`: `prepare_rename`, `rename`, `formatting`, `range_formatting`, `on_type_formatting`.
- Update `initialize` ServerCapabilities:
  ```
  rename_provider: Some(RenameOptions { prepare_provider: Some(true), .. })
  document_formatting_provider: Some(OneOf::Left(true))
  document_range_formatting_provider: Some(OneOf::Left(true))
  document_on_type_formatting_provider: Some(DocumentOnTypeFormattingOptions {

      more_trigger_character: Some(vec!["}".to_string()]),
  })
  ```
