## 1. Bug + симптом

**Тест:** `main_yield`.

**Симптом под активным flip:**
Plan 83.4.3 B4 (commit 7fa8e8af81c) добавил branch в `nova_fiber_yield`
для main thread'а: `if (!mco_running()) uv_run(loop, UV_RUN_NOWAIT)`.
Под bootstrap'е (runtime НЕ armed) это работает: один turn libuv
loop'а — pending callbacks fire, return.

Под flip'е (runtime armed by default): main yield → uv_run(NOWAIT).
В этот момент: worker's `wake_handle` async-event может fire (если
у worker'а есть pending fiber для main wake). Main's evloop процессит
async — пытается re-queue fiber в main scope → conflict с supervised
drain'ом который ждёт pending_remote==0 → race / re-entrance.

## 2. Cross-runtime reference

### Go `runtime.Gosched()`

Yields current goroutine to scheduler. Если caller — main goroutine
(G #1), scheduler выбирает следующую runnable G. Main goroutine —
обычная goroutine, никаких specifics.

Если main G — единственная runnable, Gosched possible no-op (но
возвращается immediately).

### tokio `tokio::task::yield_now()`

```rust
async fn yield_now() {
    YieldNow { yielded: false }.await
}
```

`YieldNow` future returns `Poll::Pending` first call (registers waker
on executor), `Poll::Ready` second call. Effect: executor budgets one
slot для другой task.

На main task НЕТ specifics — main task сам task.

### Node `setImmediate(cb)`

```js
await new Promise(r => setImmediate(r));
```

Yields back to event loop; pending I/O / queued microtasks run; cb
fires next tick.

Полная аналогия `uv_run(NOWAIT)` — Nova's B4 fix corresponds.

### Nova bootstrap

Pre-B4: `nova_fiber_yield` на main = no-op.
Post-B4: `nova_fiber_yield` на main = `uv_run(NOWAIT)`.

Pre-flip: main thread единственный, никаких worker async events.
`uv_run(NOWAIT)` процессит ТОЛЬКО main's pending I/O (timers, sleep
callbacks). Safe.

Post-flip: main armed, workers могут signal main через
`_main_wake` async handle. `uv_run(NOWAIT)` на main процессит этот
async → возможно re-queue fiber в main scope → race с supervised
drain'ом (который тоже на main делает `uv_run`).

## 3. Root cause analysis

Сейчас supervised drain loop (`nova_supervised_drain_main_scope` в
`fibers.h`) сам делает `uv_run(loop, UV_RUN_ONCE)` чтобы waited на
pending events. Если ВО ВРЕМЯ drain main code путь вызывает
`runtime.yield()` → второй `uv_run` recursive. libuv allows recursive
`uv_run`, но if events fire during recursive run, drain'овский outer
loop может пропустить wake.

Detailed scenario:
1. Main thread: enters `nova_supervised_run`.
2. Drain loop: `nova_supervised_step` → no work; `uv_run(loop, UV_RUN_ONCE)`
   — blocks until next event.
3. (Между: user-code called `runtime.yield()` inside main scope body
   before supervised_run started.) → B4 fix: `uv_run(loop, UV_RUN_NOWAIT)`.
4. If worker sent `_main_wake` async event between (2) and (3): yield's
   uv_run consumes it. Drain's uv_run blocks forever (event already
   consumed).

Это race-window. Не fatal по типу crash, но **fiber потерян** —
worker сделал spawn → main wake → main consumed wake в yield, drain
sat блокированный.

## 4. Дизайн решения

### Опция A — `runtime.yield()` на main с armed runtime = no-op

Restore previous bootstrap-семантику. Production-grade обоснование:
main fiber-flow drains через `supervised_run` сам; explicit yield на
main не имеет definite-семантики под M:N. **Most direct fix.**

```c
static inline void nova_fiber_yield(void) {
    mco_coro* co = mco_running();
    if (!co) {
        /* Plan 83.4.5.5 (2026-05-23): на main thread под armed
         * runtime'ом yield — no-op. uv_run(NOWAIT) консумит async
         * events которые нужны supervised drain'у. main code-path
         * сам drain'ится через supervised_run; explicit yield не имеет
         * defined semantics. */
        return;
    }
    /* fiber-context: cancel-check + mco_yield (Plan 83.4.3 B5). */
    if (_nova_active_scope && nova_abool_load(&_nova_active_scope->cancel_requested)) {
        nova_throw_cancel_reason(...);
    }
    mco_yield(co);
}
```

Тест `main_yield` если ожидает что main yield делает something —
переписать на «инсайд supervised body, yield на main = no-op,
explicit drain via supervised_run».

### Опция B — `runtime.yield()` на main = drain step через
`nova_supervised_step`, не uv_run

```c
if (!co) {
    /* Plan 83.4.5.5: drive supervised_step (если есть active scope)
     * вместо uv_run — избегает consume async events нужных drain'у. */
    if (_nova_active_scope) {
        nova_supervised_step(_nova_active_scope);
    }
    return;
}
```

Semantically — main flow toggles cycle: запустить один step через
existing fibers, обработать что готово. Без consume libuv-events
которые drain ждёт.

**Минус:** supervised_step не процессит libuv-events (timers, IO).
Если main waiting на Time.sleep на main flow — yield не продвинет
timer.

### Опция C — Distinguish «main inside supervised body» vs
«main top-level»

В `nova_supervised_run` set flag indicating «drain in progress».
Когда true, yield-on-main = no-op. Когда false (top-level user
code), `uv_run(NOWAIT)`.

**Самый аккуратный**, но добавляет flag-management.

**Рекомендация:** **B** или **C** в зависимости от Ф.0 dignosis. Если
test семантически требует timer-progress — C; иначе — B.

## 6. Acceptance

- `main_yield` PASS под active flip.
- 0 регрессий single-thread пути (B4 fix остаётся для bootstrap'е).
- 0 регрессий existing tests которые используют `runtime.yield()`
  внутри fiber'ов.

## 7. Risk

- **Option B (supervised_step) не процессит timers** — если main code
  waits на Time.sleep через yield, не продвигается. Mitigation:
  пользователи должны использовать `Time.sleep(N)` (который сам
  park'ится корректно), не `runtime.yield()` для timing-purposes.
- **Option C flag — race на flag** под M:N? Flag устанавливается main
  thread'ом из supervised_run, read'ится main thread'ом из
  fiber_yield → same-thread, no race.
