## 1. Цель

Компилятор обращается со всеми value-типами **единообразно** (один путь representation +
приёмник + передача) и **автоматически** выбирает передачу/размещение (by-ref vs by-value,
heap vs stack) — без явных аннотаций ([Q29](../../spec/open-questions.md): нет `ref T`). ABI/placement
**выводятся** из `ResolvedType` + escape/recursion-анализа (D315: «ABI выводится, не хранится»),
а не разбросаны по веткам codegen.

## 2. Acceptance-критерий (от владельца, 2026-06-21)

ЭТО должно компилироваться, запускаться и проходить:

```nova
type Point value { x int, y int }
fn Point mut @inc(a int) -> @ { @x += a; @y += a; @ }

mut p Point = { x: 0, y: 0 }
ro p2 = p.inc(10).inc(20)
assert(p == p2 && p.x == 30)
```

**Сейчас НЕ работает** (диагностика 2026-06-21, release nova на ветке plan-172):
1. **Fluent `mut @ -> @` на value-record** → CC-FAIL: codegen путает `NovaValue_Point` (by value)
   и `NovaValue_Point*` (by pointer) в цепочке (`initializing 'NovaValue_Point *' with
   'NovaValue_Point'`). Heap-record (`type Point {}`) тот же паттерн **проходит** → пробел
   value-record-специфичен.
2. **`==` на value-record** → CC-FAIL: `==` на C-структуре. Heap-record `==` компилится, но это
   **reference-eq** (два разных `{1,2}` → не равны). Sum-типы имеют авто-структурное `==`
   (tag+payload, [02-types.md](../../spec/decisions/02-types.md) ~:6797) — value-records НЕ имеют.

## 2.1. Подход (важно — §0/§10): НЕ патчить старый codegen, чинить ПОСТРОЕНИЕМ

Формулировка владельца (2026-06-21): **«поломано в СТАРОМ компиляторе; главное — чтобы работало
в НОВОМ»**. Эти CC-FAIL'ы — артефакт **фрагментированного** value-ABI (раздельные пути `NovaValue`
/ `NovaTuple` / heap-record — источник постоянных багов). **Запрещено** чинить их band-aid'ом в
старом codegen-пути ([`compiler-conventions.md`](../compiler-conventions.md) §10-анти-паттерн
«углублять фрагментацию»; §0 «сокращать»). Правильно: единый носитель `ResolvedType` (172.1/D315)
даёт **ОДИН** value-ABI-лоуэринг, и acceptance-кейс проходит **ПО ПОСТРОЕНИЮ** — потому что
раздельности нет. То есть **Ф.3 (единый value-ABI) — первичная работа**; fluent `mut @` (Ф.1) и
структурное `==` (Ф.2) — её **следствия/acceptance**, а не отдельные заплатки. Зависимость на 172.1
(единый носитель authoritative) — жёсткая: без него «починка» свелась бы к патчу старого пути.

## 3. Фазы

- **Ф.1 — value-record fluent `mut @ -> @` работает (acceptance).** Рассинхрон `NovaValue_X` ↔
  `NovaValue_X*` в цепочке — следствие фрагментации; устраняется **единым value-ABI-лоуэрингом
  (Ф.3)**, а НЕ точечным патчем старого `NovaValue`-пути (§2.1). Регресс-гард = acceptance-кейс.
- **Ф.2 — value-record структурное `==` (+ дизайн ref-vs-structural).** Авто-вывод структурного
  `==` для value-records (tag/поля, как sum-типы), а не `==` на C-структуре. Дизайн-вопрос (D-блок):
  value-record `==` = **структурное** (они значения); heap-record `==` = текущее reference — оставить
  или тоже структурить? → решить spec-first.
- **Ф.3 — единый value-ABI representation.** Свести `value-record` / `named tuple` / `Tuple(int,str)`
  к ОДНОМУ пути representation + приёмника. Убрать раздельность (named-tuple `ro @` by-value vs
  value-record by-pointer — [Plan 128](128-mut-receiver-abi.md)). Вектор — 172.1 U.5/U.6 (свернуть
  tuple/value-record/sum-schema дубли); 172.4 завершает на ABI-уровне.
- **Ф.4 — авто-by-ref передачи параметров.** Большие value передавать указателем автоматически,
  когда копия не наблюдаема (закрывает ЗА-аргумент Q29 без `ref T`). Foundation: `escape_analyze.rs`
  (Plan 127) уже авто-промоутит value-records — обобщить на by-ref-передачу.
- **Ф.5 — авто heap↔stack placement, в т.ч. нерекурсивный sum→стек.** Sum по умолчанию heap
  (`Nova_X*`) — правильно для **рекурсивных** (деревья/связи); **нерекурсивные/перечисления** →
  авто-свёртка в **стек-value (tagged-union)**, как билтин Option (`NovaOpt` stack) vs Result (heap),
  [Plan 62.A.bis](62.A.bis-sum-schema-registry.md). Детектор рекурсии есть (size/GC,
  [08-runtime.md](../../spec/decisions/08-runtime.md) ~:8211). Консервативно: рекурсия/неопределённость
  → heap (Plan 127 over-promote stance).
- **Ф.6 — copy-elision: RVO/NRVO + move-on-last-use для value-типов** (перф; добавлено 2026-06-26
  по итогам обсуждения с владельцем). Свежие/уникальные value-значения (результат `return`, литералы,
  временные), у которых escape-анализ (Plan 127) доказал **отсутствие второго владельца**, **строятся
  прямо в слоте-назначении** (RVO/NRVO через скрытый `sret`-указатель) либо **перемещаются на
  последнем использовании** (move-on-last-use) — вместо `memcpy`. **Прозрачно** (для `value`-типов
  копия ≡ move наблюдательно: нет идентичности; heap — moot, by-ref). **БЕЗ нового синтаксиса** (вывод,
  как Rust/Swift; НЕ `T&&`/`tmp`-маркер): едет на ОБЫЧНОМ by-value параметре, на `consume` (там даже
  move-`memcpy` не нужен), и на `return`. **Ортогонально `consume`:** `consume` = контракт владения
  (affine); это — оптимизация копий БЕЗ контракта (владелец 2026-06-26 развёл оси «владение» vs «не
  копировать лишнее»). **Обобщает D326 R7/R8** (`-> @` value-record decay = частный случай). GC проще
  C++: нет деструкторов → move = чистый перенос байтов, без moved-from-состояния (нет move-ctor).
  **Инвариант** (узкий D-блок, если понадобится): компилятор ВПРАВЕ move'ить доказанно-уникальные
  value-временные; программа НЕ зависит от идентичности копии (для value-record уже истинно).
  Маркер `[M-172.4-rvo-move-on-last-use]`. **Приоритет P3** (перф; важно для крупных value-record,
  мелкие/heap ≈ ноль); делать ПОСЛЕ Ф.3-Ф.5 (единый ABI), мерить blast-radius. **Корректность не
  затрагивает** — чистая оптимизация.

## 4.5. DESIGN-LOCK (2026-06-28, старт 172.4 — greenlight владельца, пивот с gated-172.1.2)

**Dep-статус (verified):** U.5 ✅ (`ResolvedType` — единственный носитель чекера, `Ty`/`TyCat`/`cat_of`
снесены); U.6 lowering-консолидация ✅ в основном (`resolved_type_to_c` ПОЛНЫЙ паритет U.4.6b `a6f645a3`;
3 type→C таблицы → 1; финальный `type_ref_to_c`-full-retire = endgame U.6.1, OUT of MVP но НЕ блокирует
value-ABI). ⇒ **единый ABI-лоуэринг (`resolved_type_to_c`) в месте** — proper-fix Ф.3 feasible, НЕ band-aid.

**Acceptance CC-FAIL'ы ВОСПРОИЗВЕДЕНЫ** (standalone `nova build`, `fn main`, обход test-harness/assert):
- **Ф.1 fluent** (`va.c:1778`): `NovaValue_Point* _nv_tmp = (*nova_self);` — `-> @` decay путает
  by-value (`NovaValue_Point`) ↔ by-pointer (`NovaValue_Point*`). Сайт: `-> @`/`=> @` mut-receiver
  return decay, **D181/Plan 91.7** (`emit_c.rs:~25292` «mut method returns @ (receiver pointer) for
  fluent chain» + value-used-as-value `:21973`). Корень — фрагментация receiver-ABI (Plan 128
  named-tuple by-value vs value-record by-pointer): цепочка `.inc(10).inc(20)` смешивает формы.
- **Ф.2 ==** (`va.c:1801`): `nova_bool eq = (p == p2);` — сырой C-`==` на `NovaValue_Point` struct.
  Корень: `emit_field_eq` (`emit_c.rs:12904`, §0 ЕДИНЫЙ structural-eq dispatch-by-C-type) **НЕ имеет
  `NovaValue_`-арма** → value-record падает в fallback `(l==r)`; И top-level `==`-лоуэринг value-record'а
  не роутит в структурный eq (в отличие от sum/record/Vec/HashMap — eq-консолидация simplifications.md).
- (StringBuilder-interp `consume_into_str→int` в репро — ОТДЕЛЬНЫЙ extern-return баг, НЕ value-ABI.)

**Подход (§2.1 — fix by construction):**
- **Ф.2 (первый срез, §0-clean, tractable):** РАСШИРИТЬ единый `emit_field_eq` арм'ом `NovaValue_<Name>`
  → структурный field-by-field eq (как record, но value-placement: поля из `record_schemas`/
  `value_record_names`), + роутить top-level `==` value-record'а в тот же диспетчер (НЕ raw C-`==`).
  Это РАСШИРЕНИЕ единого источника (§0), НЕ band-aid отдельным value-record-eq-путём. **Spec-first
  (D-блок):** value-record `==` = СТРУКТУРНОЕ (они значения — Q-value-abi, естественный ответ); heap-record
  `==` остаётся reference (отдельная ось — решить в D-блоке: оставить ref или тоже структурить).
- **Ф.1/Ф.3 (основная работа, после Ф.2):** единый value-ABI representation — свести
  value-record/named-tuple/`Tuple` к ОДНОМУ пути representation+receiver через `resolved_type_to_c`
  (он уже паритетен). `-> @` decay перестаёт путать by-value/by-pointer ПО ПОСТРОЕНИЮ (нет раздельности).
  Это behavior-change поверх единого носителя; де-риск §7 (detect → blast-radius → 0 регрессий vs чистый).
- **Acceptance-кейс → регресс-гард** (`nova_tests/`, после фикса D78-module-path: `module parent.target`).

**Первый implementation-срез:** Ф.2 (== structural для value-record) — bounded, §0-clean, spec-D-блок +
`emit_field_eq` NovaValue_-арм + top-level-`==`-routing. **✅ ЗАЛЭНДЖЕН 2026-06-28 (`4c375169`, D328):**
0-NEW-FAIL + **починил 13 pre-existing** (вся net_v2/net-семья plan91_12/15/16 — это были value-record
`SocketAddr` ==/!= CC-FAIL'ы, §1210). PASS 477→490.

### Ф.1/Ф.3 (fluent value-record `mut @ -> @`) — ДВА interacting блокера (разведано 2026-06-28)
Acceptance `mut p Point = {0,0}; ro p2 = p.inc(10).inc(20); assert(p == p2 && p.x == 30)` упирается в
ДВА **независимых** codegen-блокера (оба — value-record-vs-heap ABI-раздельность, §2.1 fix-by-construction):

1. **Self-return CC-FAIL (`inc` тело):** `NovaValue_Point* _nv_tmp = (*nova_self)` — return-тип `-> @`
   (`receiver_c_type` → `NovaValue_X*` ВСЕГДА, независимо от mut, emit_c.rs:11939) = pointer, но
   SelfAccess-decay (emit_c.rs:22061) для value-record даёт `(*nova_self)` (value) → рассинхрон
   infer(pointer)/emit(value). **ФИКС (разведан, рабочий, НЕ закоммичен):** return-site-scoped флаг
   `in_recv_ptr_return_position` (set в `emit_return_expr` когда `ret_c == var_types["nova_self"]` —
   оба `NovaValue_X*`), в SelfAccess → `nova_self` (pointer). Применять в ОБОИХ body-emit путях:
   `emit_block_stmts` (emit_c.rs:18291, non-generic emit_fn) + 3 mono/method-копии (`emit_monomorphized_fn`/
   `_method`/`emit_generic_method_erased`). **Устраняет CC-FAIL** (проверено: компилируется).

2. **Chain-materialization КОПИЯ (chain_norm):** `p.inc(10).inc(20)` нормализуется (`chain_norm.rs:542`,
   `_chain_root_N_<field> = root`) в `{ let _chain_root = p; _chain_root.inc(10); _chain_root.inc(20);
   _chain_root }`. Для value-record `p` (value) `_chain_root = p` = **КОПИЯ** → обе мутации на копии,
   оригинал `p` нетронут → runtime `p≠p2 && p.x≠30`. chain_norm doc ПРЯМО признаёт «Value-type fields
   not handled — V2 followup» (chain_norm.rs:44); рассчитан на reference-roots (`@F`-pointer-copy =
   тот же heap-объект). **ФИКС (TODO):** value-record chain-root = **reference** (codegen эмитит
   `NovaValue_X* _chain_root = &p` + method-calls передают pointer напрямую + trailing decay-в-value
   при binding), ЛИБО chain_norm не нормализует value-record + codegen нитит return-pointer (но тогда
   prepare_method_recv не должен re-address уже-pointer return — double-ptr hazard). Интрикат.

**⚠️ ПОРЯДОК (§1 soundness):** блокер-1-фикс БЕЗ блокер-2-фикса превращает CC-FAIL (громкий) в
**silent-wrong-behavior** (мутация копии) — РЕГРЕССИЯ звучности. ⇒ Ф.1/Ф.3 коммитить ТОЛЬКО ОБА вместе
(self-return + chain-reference), под полным регрессом (blast-radius: ВСЕ value-record `-> @` методы +
ВСЕ value-record chains — §7 detect-mode). Self-return-фикс разведан/откатан (не оставлять partial).
Гейт: detect172 + 0-CC-FAIL + 0-NEW-FAIL + acceptance runtime `p.x==30`.
