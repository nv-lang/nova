# Plan 33.7: Bit-vectors + integer overflow theory

> Big initiative из [Plan 33.6](33.6-contracts-production-hardening.md) §Ф.6.
> Реализует `BitVec N` sort в SMT-IR для проверки битовых операций и
> переполнения целочисленных типов фиксированной ширины.

## Цель

1. **Sized integer types в контрактах** — `u8`/`u16`/`u32`/`u64`/`i8`/
   `i16`/`i32` мэпятся в `BitVec N` Z3-sort, а не в `Int`. Это даёт
   точную битовую семантику: `255u8 + 1u8 == 0u8` (wrap-around).

2. **Битовые операции** — `&`/`|`/`^`/`<<`/`>>` в контрактах были
   `Unsupported` (compile error). После Plan 33.7 они кодируются в
   SMT BitVec теорию: `bvand`, `bvor`, `bvxor`, `bvshl`, `bvlshr`.

3. **Overflow detection** — атрибут `#nooverflow` на функции добавляет
   дополнительные VC (verification conditions) к каждой арифметической
   операции: для подписанных целых — signed overflow (SOVERFLOW), для
   беззнаковых — unsigned overflow (UOVERFLOW). Если VC не доказывается
   → compile error.

## Принципы

- **`int` = Z3 Int (unbounded)** — глобальное решение плана 33 неизменно.
  `int` используется для general-purpose; `u8`/`u64`/etc. — для
  low-level / crypto / packed data / FFI.
- **Нет inference**: если тип параметра `int` → Int sort. Если `u32` → BitVec 32.
  Нет auto-promotion.
- **`#nooverflow` opt-in**: overflow-VC только если явно запрошено.
  Default — wrap-around (2's complement) как в C/Rust.
- **TrivialBackend**: для BitVec vars возвращает `Unknown(UnsupportedTheory)`,
  с HINT «use REQUIRES_SMT_BACKEND z3 for bitvector reasoning».
- **Production-grade**: Z3 proves, тесты positive (wrap semantics)
  + negative (overflow detection) + edge cases (shift by ≥ width).

### Ф.3 — Encode + Pipeline: BitVec typing + bitwise ops (~1 день)

**Ф.3.1 — `encode.rs`:**

`is_bv_term(t, ctx)` — аналог `is_fp_term`:
```rust
fn is_bv_term(t: &SmtTerm, ctx: &EncodeCtx) -> bool {
    match t {
        SmtTerm::BitVecLit(_, _) => true,
        SmtTerm::Var(name) => matches!(ctx.var_sorts.get(name), Some(SortRef::BitVec(_))),
        SmtTerm::App(op, _) if op.starts_with("bv") => true,
        _ => false,
    }
}
```

`bin_op_to_bv_smt(op, is_signed)` — для BitVec ops:
```rust
fn bin_op_to_bv_smt(op: BinOp, is_signed: bool) -> Result<&'static str, EncodingError> { ... }
```

Dispatch в `encode_expr` для `Binary`:
1. Проверить `is_bv_term(left_term, ctx)` → если да → `bin_op_to_bv_smt`.
2. Иначе → `is_fp_term` → fp-ops.
3. Иначе → int-ops.

`BitVecLit` encoding:
- `ExprKind::IntLit(n)` — если контекст BitVec (определяется через `EncodeCtx.var_sorts` для текущего param) → `SmtTerm::BitVecLit(n as u64, width)`.
- Пока проще: **только через explicit typed var declaration**.

**Ф.3.2 — `pipeline.rs`: `type_ref_to_sort` расширение:**
```rust
"u8"  | "byte" => SortRef::BitVec(8),
"i8"            => SortRef::BitVec(8),
"u16"           => SortRef::BitVec(16),
"i16"           => SortRef::BitVec(16),
"u32"           => SortRef::BitVec(32),
"i32"           => SortRef::BitVec(32),
"u64"           => SortRef::BitVec(64),
// "int" | "i64" — остаётся SortRef::Int (глобальное решение 33.1).
```

**Ф.3.3 — TrivialBackend**: при `declare_var` с `SortRef::BitVec(_)` — return `Unknown(UnsupportedTheory("bitvector; use REQUIRES_SMT_BACKEND z3"))` для любой verify.

Честный `NotAttempted`/`UnsupportedTheory` уже есть в тривиальном бэкенде — достаточно передать правильный hint при encode с BitVec vars.

### Ф.4 — `#nooverflow` attribute + overflow VCs (~1 день)

**AST:**
- `FnAttrKind::NoOverflow` — новый variant.
- `FnDecl.no_overflow: bool` — флаг.

**Parser:**
- `#nooverflow` атрибут на fn-declaration.

**Pipeline:**
- При `verify_fn` + `fn.no_overflow == true`: после основных VCs
  для каждой BinOp `+`/`-`/`*` на BitVec vars —
  emit дополнительный overflow VC:
  ```
  VC: bvadd_no_overflow_s(a, b)   // если signed
  VC: bvadd_no_overflow_u(a, b)   // если unsigned
  VC: bvadd_no_underflow(a, b)    // для вычитания
  ```
- Failure → Diagnostic с hover `"integer overflow detected: ..."`.

**Acceptance:**
```nova
// u8 wrap-around (no #nooverflow — OK):
#verify
fn add_u8(a u8, b u8) -> u8
    requires a == 200u8 && b == 100u8
    ensures result == 44u8      // 300 % 256
=> a + b

// #nooverflow — overflow → compile error:
#nooverflow #verify
fn safe_add(a u8, b u8) -> u8
    requires a == 200u8 && b == 100u8
    ensures result == 300u8     // impossible
=> a + b
// Expected: compile error "integer overflow: u8 addition may overflow"
```

### Ф.5 — Тесты (0.5 дня)

#### Тесты (все `// REQUIRES_SMT_BACKEND z3`):

**Позитивные:**
- `f57_bv_arith_positive.nv` — u32 addition/comparison proves.
- `f57_bv_bitwise_positive.nv` — `&`/`|`/`^`/`<<`/`>>` proves.
- `f57_bv_wrap_positive.nv` — u8 wrap-around (255+1==0).
- `f57_bv_nooverflow_safe.nv` — `#nooverflow` fn с non-overflowing input PASS.

**Негативные (EXPECT_COMPILE_ERROR):**
- `f57_bv_overflow_fail.nv` — `#nooverflow` + overflowing args → error.
- `f57_bv_bitwise_trivial_fail.nv` — BitVec var + TrivialBackend → REQUIRES_SMT_BACKEND.

## Acceptance полного Plan 33.7

1. `u32`/`u64`/`i32` параметры в контрактах → BitVec sort в Z3.
2. `&`/`|`/`^`/`<<`/`>>` в contracts не `Unsupported`.
3. `#nooverflow #verify fn f(a u8, b u8) => a + b` → overflow-VC проверяется.
4. 5+ тестов PASS через `NOVA_SMT_BACKEND=z3`.
5. TrivialBackend: BitVec contracts → `Unknown(UnsupportedTheory)` + hint.
6. V20 (BitVec theory) в simplifications.md.

## Не входит (отдельные milestone'ы)

- Синтаксис `<literal>u8` / `<literal>u32` — Nova пока нет (int-лит
  получает тип из контекста; explicit cast `x as u8` существует).
- BitVec в axiom binders — Plan 33.7 V2.
- Mixed Int/BitVec arithmetic в одном выражении (требует explicit cast).
- Overflow для `int` (i64) — не нужен, bounded axiom уже есть в 33.1.
