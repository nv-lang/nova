# Plan 83.5: Boehm `THREAD_LOCAL_ALLOC` — ❌ REJECTED (wrong hypothesis)

> **Создан:** 2026-05-24 (out of Plan 83 audit, см. [83-audit-2026-05-24.md](83-audit-2026-05-24.md) §1 пункт 1).
> **Статус:** ❌ **REJECTED 2026-05-24 same-day** — wrong hypothesis.
> Audit-агент утверждал что Boehm TL_ALLOC не включён. Verification во
> время implementation показала **TL_ALLOC уже active** в Nova build
> (vcpkg bdwgc 8.2.12 defaults `enable_thread_local_alloc=ON`, gc.lib
> symbols `GC_init_thread_local` + `GC_thr_init` present, Nova workers
> уже регистрируются через `GC_register_my_thread` который автоматически
> вызывает `GC_init_thread_local`). Spawn-cost bottleneck —
> `GC_malloc_uncollectable` (Plan 83.4.5.8), и её путь by Boehm design
> НЕ имеет thread-local cache (uncollectable allocs persistent через
> collections). Реальный fix — Plan 83.6 (per-worker SpawnCtx pool).
> **Родитель:** [Plan 83](83-mn-default-on-gomaxprocs.md).

---

## 1. Original hypothesis (REJECTED)

Per audit §6.1: "Boehm scale.html says `THREAD_LOCAL_ALLOC` +
`GC_REDIRECT_TO_LOCAL` 'greatly reduces locking'. Verified Grep
alloc_boehm.c — defines не set. Therefore TL fast path is OFF. Enable →
2-5× spawn-cost speedup."

## 2. Empirical verification — почему hypothesis неверна

### 2.1 Boehm CMake defaults (8.2.12)

`/d/Sources/vcpkg/buildtrees/bdwgc/src/v8.2.12-fde2beb7fe.clean/CMakeLists.txt`:

```cmake
option(enable_threads "Support threads" ON)                                # default ON
option(enable_parallel_mark "Parallelize marking..." ON)                  # default ON
option(enable_thread_local_alloc "Turn on thread-local alloc..." ON)      # default ON
option(enable_threads_discovery "Enable threads discovery in GC" ON)      # default ON
```

**Все 4 thread-related options default = ON.** vcpkg port
(`ports/bdwgc/portfile.cmake`) не overrides → дефолты применяются.

### 2.2 Installed gc.lib verification

```
$ cat compiler-codegen/vcpkg_installed/vcpkg/blds/bdwgc/x64-windows-static-rel/CMakeCache.txt
enable_thread_local_alloc:BOOL=ON
enable_parallel_mark:BOOL=ON
enable_threads:BOOL=ON
enable_threads_discovery:BOOL=ON
```

Symbols в `gc.lib`:
```
$ grep -ao 'GC_thr_init\|GC_init_thread_local' compiler-codegen/vcpkg_installed/x64-windows-static/lib/gc.lib
GC_init_thread_local
GC_thr_init
```

Symbols **присутствуют** — TL_ALLOC compiled в библиотеку.

### 2.3 Boehm internal automatic activation

В `win32_threads.c:851,3194` (verified):

```c
me = GC_register_my_thread_inner(sb, thread_id);
GC_init_thread_local(&me->tlfs);   /* sets up thread-local freelists */
```

`GC_register_my_thread` **автоматически** инициализирует `tlfs`
(thread-local freelists) для thread'а. Nova **уже** делает
`GC_register_my_thread` в каждом worker (`runtime.c:335-343` per audit §3.4).

**Conclusion: TL fast path active для Nova workers.** Regular `GC_malloc`
calls на worker thread'е берут из per-thread freelist без global lock.

### 2.4 Почему `GC_REDIRECT_TO_LOCAL` define не нужен

В Boehm 8.2.12 этот macro **deprecated/removed**. Заменён internal
automatic redirect через `THREAD_LOCAL_ALLOC` build flag (compiled-in).
Define в client code (`alloc_boehm.c`) **не required и не имеет effect**.

```
$ grep -rn 'GC_REDIRECT_TO_LOCAL' /d/Sources/vcpkg/buildtrees/bdwgc/src/v8.2.12*/include/ /d/Sources/vcpkg/buildtrees/bdwgc/src/v8.2.12*/*.c
(0 hits)
```

Macro existed в Boehm 7.x era; 8.x — automatic.

### 2.5 Почему spawn-cost не уменьшается от TL_ALLOC

`GC_malloc_uncollectable` (`mallocx.c` в Boehm) **by design идёт под global
allocation lock** — uncollectable allocs persistent через все collections,
thread-local cache не applicable (caches recycled per-collection).

Nova Plan 83.4.5.8 routes ALL armed-M:N SpawnCtx через
`nova_alloc_uncollectable` → `GC_malloc_uncollectable`. **Это путь,
обходящий TL_ALLOC fast path по дизайну Boehm.**

Когда audit-агент claimed "spawn cost = Boehm GC global lock", это
**частично верно — но только потому что SpawnCtx идёт через uncollectable**.
Regular `nova_alloc` (managed, user-code) **уже** бенефицирует от TL_ALLOC
автоматически.

## 3. Implications

1. **Bench result не изменится от Plan 83.5** — TL_ALLOC already active
   для regular allocs.
2. **Real fix spawn-cost — Plan 83.6** (per-worker SpawnCtx pool).
   Bypass `GC_malloc_uncollectable` для hot path entirely.
3. **Документация audit (§6.1) — incorrect.** Update simplifications +
   83-audit-2026-05-24.md §6.1 mark REJECTED.

## 6. Lessons learned (для будущих audits)

Это **второй wrong-hypothesis** в Plan 83 series (после Plan 83.4.5.10
Ф.2 stack-size reduction). Pattern:

1. Audit/analysis claim основан на static reading кода + assumed defaults.
2. Empirical verification — defaults уже applied, claim неверен.

**Mitigation для будущих audits:** verify **installed library binary
flags** перед formulating perf plan. Не полагаться на:
- Static defines в client `alloc_boehm.c` (clients don't enable Boehm
  internals; activation is library-side).
- Documentation phrasing ("greatly reduces locking" — assumed off
  by reader, actually on by default).
- Verbal agent claims without explicit `grep symbol_name actual_library`
  verification.

**Recommended audit-protocol** (going forward):
1. Static claim → "X is off because grep didn't find Y in our code".
2. **Required next step:** grep symbol_name actual installed library
   binary. CMake cache. Config files installed by build.
3. **If claim survives binary verification → proceed.** Otherwise mark
   REJECTED + post-mortem (Plan 83.4.5.10 Ф.2 style).
