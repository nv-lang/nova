# Plan 172.3 — Type-set bounds (Go-style generic constraints)

**Статус:** ✅ Ф.0 designed 2026-06-28 (owner sign-off; Q1–Q6 закреплены — §3.1) · impl pending
**Umbrella:** [Plan 172 — Переработка компилятора: типовая система](172-compiler-rework.md).

## 1. Мотивация

Сейчас generic-bound в Nova — это **протокол(ы)**: `[T Hash]`, multi-bound `[T A + B + C]`
(D72 / D145). Нельзя ограничить параметр **набором конкретных типов** (как Go 1.18
`type IntNumber interface { ~int8 | ~int16 | … }`).

Из-за этого код, общий для всех целочисленных типов, приходится писать **per-type
тонкими обёртками** (×~10 типов × N методов). Прямой триггер — Plan 174.1 (primitive
parse API): `int.try_parse` / `u32.try_parse` / … хочется выразить одним
`fn[T IntNumber] T.try_parse(s str, radix int) -> Result[T, ParseIntError]`,
но без type-set bound это невозможно — приходится плодить обёртки (вариант A в Plan 174.1).

Type-set bound нужен шире parse: `clamp`, `min`/`max`, checked-арифметика, numeric
algorithms — всё, что обобщается над множеством примитивов с одинаковой семантикой.

## 2. Что есть сейчас (опора)

- `GenericParam { name, bounds: Vec<TypeRef> }` ([ast/mod.rs:752](../../compiler-codegen/src/ast/mod.rs#L752)) — bound = список протокол-TypeRef (conjunction).
- `type X | A | B` уже занят под **sum-type** (union вариантов) — **конфликт синтаксиса** для наивного `type IntNumber | i8 | i16`.
- `numeric_type_constant_mapping` ([emit_c.rs:35113](../../compiler-codegen/src/codegen/emit_c.rs#L35113)) — `int.MAX→INT_MAX`, `f32.MAX→FLT_MAX` и т.д. по **конкретному** имени типа. Под мономорфизацию это значит: после подстановки конкретного T в инстансе `T.MAX` резолвится — фундамент для overflow-чека внутри generic-тела есть.
- Generic static methods на типе работают: `export fn[T] T.from(t T) -> T => t` ([protocols.nv:109](../../std/prelude/protocols.nv#L109)), `fn int.new() -> Self => 0`.

## 3. Открытые дизайн-вопросы (решить в Ф.0)

- **Q1 — синтаксис объявления.** `type X | A | B` уже = sum. Кандидаты:
  - `bound IntNumber = i8 | i16 | … | uint` (новое ключевое слово `bound`)
  - `type IntNumber set i8 | i16 | …` (контекстное `set`)
  - inline в позиции параметра: `fn[T in i8|i16|…]` / `fn[T (i8|i16)]`
  Критерий: нулевая неоднозначность с sum-type; читаемость; reuse в нескольких сигнатурах.
- **Q2 — семантика тела.** Мономорфизация по каждому типу набора (как у обычных generic). `T.MAX`/`T.MIN`/`T.new`/литералы резолвятся per-instantiation через существующий numeric mapping. Проверить, что mapping срабатывает на подставленном T, а не только на синтаксически-конкретном имени.
- **Q3 — операторы в generic-теле.** Для overflow-чека нужны `* radix`, `+ digit`, сравнения с MIN/MAX. Какие операции легальны над type-set-параметром? Минимум: арифметика + сравнения, разрешённые ВСЕМИ членами набора (целочисленные — да; если в наборе float — другой профиль).
- **Q4 — композиция с protocol-bound.** Допускать ли `[T IntNumber + Hash]` (type-set ∧ protocol)? Предв. да — type-set сужает множество, протокол добавляет требование; обе проверки независимы.
- **Q5 — только примитивы или произвольные типы?** Go допускает `~T` (underlying). У Nova нет newtype-underlying — на V1 ограничиться **именованными конкретными типами** (примитивы + объявленные типы), без `~`.
- **Q6 — uint/u64 граница.** Прямой урок Plan 174.1: `max(u64/uint) = 2^64−1` НЕ влезает в `int`(i64). Значит generic над знаковыми и беззнаковыми **не может** делить одно ядро с границами-`int`. Решение на уровне type-set: либо два отдельных bound (`SignedInt` / `UnsignedInt`), либо разрешить телу ветвиться по `T.MIN == 0`. Зафиксировать в Ф.0.

## 3.1. Ф.0 РЕШЕНО (2026-06-28, owner sign-off + multi-agent design+adversarial-review)

Все Q1–Q6 закреплены, grounded в spec/conventions. Полный design-note + D310-драфт — в harvest; ключевое:

- **Q1 синтаксис →** контекстный kind-токен `set`: `type SignedInt set i8 | i16 | i32 | i64 | int`.
  Диспетчеризуется существующим D52 first-token-after-name (как `alias`/`protocol` под D53) → ноль
  бэктрекинга, ноль конфликта с sum-`type X | A | B` (у sum нет kind-токена). НЕ новое top-level
  слово `bound` (фрагментирует D52-диспетч). НЕ inline `fn[T in …]` на V1 (нет reuse). В param-позиции
  type-set = как protocol-bound: `fn[T SignedInt] T.try_parse(...)`.
- **Q2 тело →** мономорфизация per член (Plan 48 worklist). `T.MAX`/`T.MIN` требуют **Nova-name subst-канал**
  (T→"i8") ПЕРЕД `numeric_type_constant_mapping` (emit_c.rs:35654, keyed по Nova-имени) — это НОВАЯ
  lowering-плита, отличная от C-name subst (T→"int8_t"). Без `nova_int`-fallback (§1).
- **Q3 операторы →** ПЕРЕСЕЧЕНИЕ легальных для ВСЕХ членов, enforce в чекере на use-site внутри тела;
  чекер МАТЕРИАЛИЗУЕТ resolved-тип в per-ExprId канал (§1, codegen не ре-резолвит).
- **Q4 композиция →** через существующий D145 `+` (conjunction), без новой грамматики: `[T SignedInt + Hash]`
  = T ∈ set И impl Hash (проверки независимы, per-member). Два type-set в одном листе запрещены (`E_MULTIPLE_TYPE_SETS`).
- **Q5 члены →** по ИДЕНТИЧНОСТИ: примитивы + любой объявленный конкретный тип (явным листингом). `~underlying`
  НЕТ (нет implicit-coercion; Q-representation-bound → Plan 102). Резолв членов через тот же символьный
  путь, что protocol-bound (`check_generic_bound_declarations`, types/mod.rs) — без второго хардкод-списка (§3).
- **Q6 знаковость →** РАЗДЕЛИТЬ по знаку на уровне декларации (u64.MAX=2^64−1 ∉ i64). Stdlib: `SignedInt`,
  `UnsignedInt`. Чекер: `E_TYPE_SET_MIXED_SIGNEDNESS` на смешанном объявлении. Без рантайм-ветки по `T.MIN==0` (§2).

**Review-правки (folded):** (a) §5 spec-first — нужно **AMEND D72 + D145** («bound = только protocol» →
«protocol ИЛИ type-set»), не только добавить D310; (b) Q-representation-bound НЕ метить closed (D310 —
related, но ≠ representation-bound); (c) D237 цитировать только для capitalization, не как «1 метод»;
(d) `numeric_type_constant_mapping` **неполна для unsigned MIN** и части unsigned MAX — Ф.3 добавляет строки
(не assumed-ready); (e) проверить, что `set` не используется как идентификатор в std/examples перед Ф.1.

**Новые коды (все чекер-emitted, §1/§6):** `E_TYPE_NOT_IN_SET`, `E_TYPE_SET_MEMBER_NOT_CONCRETE`,
`E_TYPE_SET_MIXED_SIGNEDNESS`, `E_MULTIPLE_TYPE_SETS`.

- [ ] `fn[T IntNumber] T.try_parse(...)` (показательный кейс) компилируется и инстанцируется для всех членов; `T.MAX`/`T.MIN` корректны per-type.
