## Проблема (verified state, 2026-05-18)

### 1. Opaque-types не имеют file-based declarations

D26 (spec/decisions/08-runtime.md:463-465) фиксирует:
```nova
// Built-in opaque accumulator/buffer типы (Plan 04, D82).
type StringBuilder    // UTF-8 string accumulator, @into() -> str (infallible)
type WriteBuffer      // binary write buffer, @into() -> []byte
type ReadBuffer       // cursor-style binary reader, view над []byte
```

Эти спецификации **bare-name`type X`** (без body) — но parser отвергает
такой синтаксис (см. §3 ниже). На practice типы существуют **только**:
- В D26 как **prose example** (декларация формы, реально не парсится).
- В `std/runtime/*.nv` как `external fn X.method(...)` — методы есть, но
  декларации **самого типа** нет.
- В compiler как **implicit recognition по имени** (см. §2 ниже).
- В D82 (spec/decisions/08-runtime.md:2547-2548) сказано прямо: «Программист
  **не пишет** `type StringBuilder { ... }` block — этот тип known-by-name
  как `int`».

После Plan 62 (Option/Result/Error/RuntimeError/Iter/Range/From/Into/Fail/Time/Mem
все в file-based form), эти три типа — **единственные** D26 prelude items с
known-by-name статусом. Все остальные `nova doc` видит canonically;
StringBuilder/WriteBuffer/ReadBuffer — нет canonical declaration anywhere.

### 3. Parser отвергает `external type X`

`compiler-codegen/src/parser/mod.rs:933-948`:
```rust
let is_external = self.eat(&TokenKind::KwExternal).is_some();
if is_external {
    // Только `fn` допустимо после `external`.
    if !matches!(self.peek().kind, TokenKind::KwFn) {
        let span = self.peek().span;
        return Err(Diagnostic::new(
            format!(
                "`external` is only valid before `fn`, got {}",
                self.peek().kind.name()
            ),
            span,
        ));
    }
}
```

D82 (spec/decisions/08-runtime.md:2748-2750) явно говорит:
> «**`external type`** — отложено. Если когда-то появятся opaque user-defined типы
> (Channel, mmap'ed Region), вернёмся; сейчас built-in только.»

Plan 62 main (62-prelude-hardcode-migration.md:864-868) предложил **D113**
для `external type`; в 62.F.bis spec amendments номера D124/D125 уже заняты,
**следующий свободный номер — D126** (проверено grep'ом по всем
`spec/decisions/*.md`, D123 — Tuple monomorphization, D124 — edition, D125 —
prelude shadow lint).

### 4. Bare-name `type X` тоже отвергается parser'ом

Альтернативный путь — добавить bare `type X` (без body, без `external`) тоже
не сработает. `parse_type_decl` (parser/mod.rs:2085-2127) после имени и
optional generics всегда требует один из:
- `effect { ... }`
- `protocol { ... }`
- `alias TYPE`
- `{ fields }` (record)
- `| variant | ...` (sum)
- иначе newtype `TYPE`

При отсутствии body parser падает на ветке newtype с `parse_type()` —
требует **другой** type. Bare `type Name` без правой части — syntax error.

То же ограничение блокирует `Never` и `RuntimeNoneError` (см. Plan 62.A
DEFER и 62.C DEFER) — empty-sum syntax (`type Name`) не поддерживается.

### 5. `infer_expr_c_type` имеет 4 hardcoded блока для opaque-типов

Помимо method dispatch (который уже generic через external_registry), есть
**статический name-based inference** в `infer_expr_c_type`:
- L18915-18920: `n == "StringBuilder"` для static-методов `.new/.with_capacity/.from`.
- L18922-18927: `n == "WriteBuffer"` analogically.
- L18928-18933: `n == "ReadBuffer"` analogically.
- L18978-19040: instance-method ret-type inference (40+ methods).

Эта логика **уже** дублирует то, что external_registry знает через
`ExternalDecl::return_c_type`. Migration возможна (через
`self.external_registry.lookup(n, method).and_then(...return_c_type...)`),
но это **separate cleanup**, не требуется для 62.D.bis closure.

## Что НЕ делаем (rejected alternatives)

| Alternative | Почему отвергнут |
|---|---|
| **Bare-name `type X`** (no `external` keyword, no body) | Конфликт с newtype-ветвью parser'а — после `type X` ожидается body или другой type. Различить «opaque» и «error: missing body» невозможно без explicit marker. `external` keyword даёт unambiguous signal. |
| **`opaque type X`** (новый keyword `opaque`) | Дополнительный keyword без явной потребности. `external` уже семантически правильный («реализация во внешнем — runtime — контексте»). Re-use over invent. Прецеденты: OCaml `external`, Dart `external`, Kotlin `external`. |
| **`#external` attribute** на bare `type X` | Atrs зарезервированы для tests/dev-tools (Q-attributes) и для type-attrs типа `#from_fields`. Modifier-форма единообразна с `export`/`external` (которые уже modifier'ы по D82). |
| **`type X { _ runtime }` body-syntax** | Magic body, не более clear чем `external`. Добавляет parsing complexity (special `_ runtime` token). Не используется в других языках. |
| **Migrate в `std/runtime/string_builder.nv`** (декларация типа рядом с методами) | Type-declaration логически принадлежит к prelude (D26), не к runtime/* (где живут implementation details). Mixing — violates Plan 62 stratification rule. |
| **Полностью удалить hardcode'ы в `infer_expr_c_type`** в этом sub-plan'е | Out-of-scope. После file-based declaration появится TypeDecl в `module.items`, но `infer_expr_c_type` всё равно использует external_registry для method ret-types. Cleanup — `infer_expr_c_type` → ExternalRegistry lookup — отдельная refactoring task. |
| **Делать `external type` user-facing** (разрешить в произвольных модулях) | Per D82: «**Программистский Nova-код не пишет `external fn`. Этот keyword — экспозиционный**». Same для `external type` — только в `std.*` (через `is_stdlib_runtime_module` / `is_prelude_self_module` whitelist). User-facing opaque types — отдельная Q-future. |
| **Generic `external type X[T]` support в bootstrap** | Bootstrap parser/codegen уже поддерживает `[T]` generic params на TypeDecl (per D16). Required для StringBuilder/WriteBuffer/ReadBuffer — не requires (они не generic). Но **синтаксически принять** `external type X[T]` важно — это primary use-case для future opaque user-types (Channel[T], Region[T]). Acceptance: parser принимает generics, type-checker регистрирует, codegen не emit'ит struct (runtime decides). |
| **Удалить `nogc_blacklisted_call` hardcode в Plan 62.D.bis** | Эта list — capability check (Plan 16 D64 realtime nogc), не связан с type-declaration migration. После 62.D.bis имена-в-matches остаются как литералы (`matches!(ty, "StringBuilder" \| ...)`), но это не conflict с file-based declarations. Cleanup — отдельный sweep если nogc rules станут data-driven. |

## Решение — архитектурные правила

### D126: `external type X [Generics]` syntax

**Семантика:** opaque type known to compiler by name; backing implementation
живёт в runtime (`nova_rt/*.h`/`.c`); type-checker регистрирует имя для name
resolution; codegen **не эмитит** struct definition (runtime header дублирует).

**Грамматика** (extension D-block — параллельно D82 для типов):
```
type-decl = ['export'] ['external'] 'type' name [generic-params] [body]
```

- `external` modifier — между `export` и `type` (mirror D82 fn-decl).
- `body` должен **отсутствовать** при `external` (никакого `{ }`, `|`, alias,
  newtype). Иначе compile error «external type cannot have a body».
- `generic-params` (`[T]`, `[T1, T2 Hashable]`) допустимы — extends к future
  Channel[T] / Region[T] use-cases. Bootstrap не emit'ит generic
  monomorphization для external types (runtime backing single-erased).
- `methods` объявляются отдельно через `external fn X.method(...)` /
  `external fn X mut @method(...)` (existing D82 mechanism). Связь
  declaration ↔ methods — по receiver-type name.

**Семантика — три уровня:**

| Уровень | Поведение |
|---|---|
| **Parser** | Принимает `external type X[T1, T2]`. Body — error. После — newline/EOF expected. |
| **Type-checker** | Регистрирует имя в `env.types`. `is_external_type: bool` flag в TypeDecl AST. Cross-file resolve работает (R26+R27). Instances типа — opaque pointer (`Nova_X*` mangling). Variants/fields отсутствуют (нет `Sum`/`Record` kind). |
| **Codegen** | `emit_type_decl` skip emission (struct definition в runtime header). Все ref'ы → `Nova_X*` pointer. Forward-decl: skip через `BUILTIN_RUNTIME_TYPES` (already в list). Method calls — через existing `external_registry.lookup()` path. |

**Restricted scope (bootstrap):** `external type` валиден **только** в:
- `std.runtime.*` (где живёт implementation в runtime)
- `std.prelude.*` (где живёт canonical declaration)

В других модулях — compile error «`external type` is only allowed in
`std.runtime.*` / `std.prelude.*` modules; for FFI to external C libraries
a future `extern("C") type` keyword will be added (Q-ffi)».

Точно тот же whitelist что у `external fn` (см. types/mod.rs:85-115, check
через `is_stdlib_runtime_module || is_prelude_self_module`).

**Связь с D82:** `external type` — type-analog `external fn`. Те же
архитектурные принципы:
- Modifier (не attribute) — единообразие с `export`.
- No body (компилятор не emit'ит, runtime реализует).
- Whitelist (только stdlib).
- Validation: compiler знает свой runtime, declaration vs реализация
  cross-check'ается при `emit_type_decl` skip (передаёт BUILTIN_RUNTIME_TYPES).

### Stratification (consistent с Plan 62 «zero-imports prelude rule»)

`std/prelude/collections.nv` уже имеет одно `import std.prelude.core.{Option}`
(для signature `Iter[T].next() -> Option[T]`). Добавление 3 `external type`
deklarations — **zero new imports** (типы opaque, без параметров другого
prelude type'а). Cycle protection preserved.

### Codegen impact — minimal

1. `emit_type_decl` — добавить early-return для `is_external_type`:
   ```rust
   if t.is_external_type { return Ok(()); }   // before any emit
   ```
   `BUILTIN_RUNTIME_TYPES` skip-list (emit_c.rs:1212-1218) уже содержит
   `StringBuilder`/`WriteBuffer`/`ReadBuffer` для fwd-decl skip — не требует
   изменений.

2. `RUNTIME_DEFINED_TYPES` skip-list (emit_c.rs:4998-5037) — добавить
   `StringBuilder`, `WriteBuffer`, `ReadBuffer` для full emit-skip
   (избыточно с p.1, но keeps consistency с pattern Option/Result/Error/Fail).
   Можно opt'нуть либо за `is_external_type`-driven skip (cleaner) либо
   за skip-list extension (consistent с existing pattern). **Recommended:**
   add to RUNTIME_DEFINED_TYPES — matches existing convention для всех
   runtime-backed types.

3. Generic templates registration (emit_c.rs:1258-1272): уже skip'ает
   Option/Result. Если future external type будет generic — skip extension
   needed. Bootstrap: 3 opaque types non-generic — no change required.

4. `external_registry` (emit_c.rs:632 `load_builtins`) — без изменений.
   Registry парсит `external fn` declarations, не type-declarations.

5. `infer_expr_c_type` hardcodes (emit_c.rs:18915-19040) — **остаются
   без изменений** в этом sub-plan'е (отдельный cleanup task, без
   regressions).

### Type-checker impact

1. **AST** (`ast/mod.rs`): добавить `is_external_type: bool` field в `TypeDecl`
   (default `false`, backward-compat). Альтернатива — новый
   `TypeDeclKind::External` variant без content; но `kind` уже семантический
   («что в теле») а `external` — модификатор источника. Flag-field семантически
   точнее. **Recommended:** добавить field.

2. **Parser** (`parser/mod.rs`):
   - L935-947 (existing `external` check): relax — разрешить `external` перед
     `type`, аналогично существующей логике для `fn`.
   - L1001-1003 dispatch: `KwType` branch принимает `is_external` flag.
   - `parse_type_decl(is_export, is_external, attrs, ...)` — добавить
     `is_external: bool` параметр.
   - Внутри `parse_type_decl`: если `is_external`, после `[generic-params]`
     **запретить** body — `expect newline_or_eof()`. Иначе compile error.
     Создать TypeDecl с `is_external_type: true` и **placeholder kind**
     (см. п.3 ниже).

3. **Placeholder `kind`** для external types: проще всего —
   `TypeDeclKind::Sum(Vec::new())` (пустой variants — uninhabited form
   semantically) либо новый `TypeDeclKind::External` variant.
   **Recommended:** новый variant `TypeDeclKind::Opaque` (one-word marker)
   без content. Чище separation; `if is_external_type` flag всё ещё нужен —
   но `kind` остаётся семантически-honest («что в теле»).
   ```rust
   pub enum TypeDeclKind {
       Record(Vec<RecordField>),
       Sum(Vec<SumVariant>),
       Effect(Vec<EffectMethod>),
       Protocol(Vec<EffectMethod>),
       Alias(TypeRef),
       Newtype(TypeRef),
       Opaque,  // NEW: external type, no body
   }
   ```
   Тогда `is_external_type` flag — избыточен; всё encoded в `kind == Opaque`.
   **Final recommended:** **только** новый variant `TypeDeclKind::Opaque`,
   без duplicate flag. Cleaner.

4. **`types/mod.rs::check_module`** (L85-115): existing whitelist для
   `external fn` — расширить аналогичным check'ом для `external type`.
   Use same `is_stdlib_runtime_module || is_prelude_self_module` test.
   ```rust
   for item in entry_items {
       if let Item::Fn(fd) = item {
           if fd.is_external { /* existing error */ }
       }
       if let Item::Type(td) = item {
           if matches!(td.kind, TypeDeclKind::Opaque) {
               errors.push(Diagnostic::new(
                   format!(
                       "`external type` is only allowed in `std.runtime.*` / \
                        `std.prelude.*` modules (this module is `{}`); for FFI \
                        to external C libraries a future `extern(\"C\") type` \
                        keyword will be added (Q-ffi)",
                       module.name.join(".")
                   ),
                   td.span,
               ));
           }
       }
   }
   ```

5. **Name resolution** (NameResCtx L2030+): `Item::Type(td)` уже регистрируется
   в `shared_decls`/`all_decls` — no special-case для `Opaque` kind required.
   Cross-file resolve работает идентично normal type-decl'ам.

### Methods (no change)

StringBuilder methods declared в `std/runtime/string_builder.nv` (62 lines,
14 methods) — **уже работают**:
- `external fn StringBuilder.new() -> Self`
- `external fn StringBuilder mut @append(s str) -> Self`
- и т.д.

После 62.D.bis эти methods продолжают работать **без изменений** через
`external_registry` lookup. Receiver-type matching уже идёт по string
name `"StringBuilder"`, не зависит от наличия formal type-decl.

WriteBuffer (40 methods) + ReadBuffer (75 methods) — аналогично.

## Spec changes (D126 full content)

### D126. `external type` — opaque types declared without body

#### Что

`external type X [Generics]` — модификатор type-декларации, означающий
что **тип реализован в runtime (C-коде `nova_rt/`)**, а Nova-уровневая
декларация даёт только имя + optional generic параметры. Тело
(variants/fields/protocol/effect/alias/newtype) **отсутствует** —
type «opaque». Аналог `external fn` (D82) но для типов.

`external` применяется к **типам** через D126; к **функциям** — через
D82. Один и тот же keyword, два валидных позиционирования
(`external fn ...` / `external type ...`).

#### Правило

##### Грамматика

```
type-decl = ['export'] ['external'] 'type' name [generic-params] [body]
```

Порядок modifiers строгий: `export` первым, `external` вторым. Body у
`external type` **должен отсутствовать** (никаких `{ ... }`, `| variant`,
`effect { ... }`, `protocol { ... }`, `alias TYPE`, или newtype `TYPE`),
иначе compile error «external type cannot have a body».

##### Примеры

```nova
// Public external (built-in)
export external type StringBuilder
export external type WriteBuffer
export external type ReadBuffer

// Generic external (future Channel use-case)
export external type Channel[T]

// Two-param generic external (future Region use-case)
export external type Region[T, Capability]

// Private external (внутри runtime module'а)
external type Nova_intrinsic_buffer
```

##### Связь с D26 prelude

Built-in opaque-типы из D26 (`StringBuilder`, `WriteBuffer`, `ReadBuffer`)
объявляются через `external type` в `std/prelude/collections.nv`
(Plan 62.D.bis, 2026-05-18). Раньше (Plan 04) типы были «known-by-name»
без formal declaration; D126 даёт canonical source-of-truth + `nova doc`
surface + eligible для type-annotations / cross-file resolve.

```nova
// std/prelude/collections.nv
module std.prelude.collections

export external type StringBuilder
export external type WriteBuffer
export external type ReadBuffer
// + Iter[T] protocol (D58)
```

Methods на opaque-типах объявляются **отдельно** через `external fn`
(D82) в `std/runtime/<type>.nv`:

```nova
// std/runtime/string_builder.nv
module std.runtime.string_builder

export external fn StringBuilder.new() -> Self
export external fn StringBuilder mut @append(s str) -> Self
export external fn StringBuilder @into() -> str
// ... 11 more methods
```

Связь декларация ↔ methods — по receiver-type name (`StringBuilder`).
Нет syntactic block'а, объединяющего type-decl с methods (по D52 это
правильно — methods orthogonal к declarations, free-fn-style).

##### Связь с D5/D47 видимостью

`export external type` — публичный: имя видно из других модулей. `external
type` без `export` — модуль-private. Те же правила, что для обычных
type-декл. `external` ортогонален `export`.

##### Связь с D52 kind-tokens

D52 фиксирует kind-tokens `type` / `protocol` / `effect`. D126 **не**
добавляет нового kind-token'а — `external` это **модификатор** перед
`type` (mirror D82 для `fn`), не отдельный kind.

В AST это кодируется через `TypeDeclKind::Opaque` (новый variant),
параллельный existing `Record` / `Sum` / `Effect` / `Protocol` /
`Alias` / `Newtype`. С точки зрения user'а — `external type X` это
specialised type-declaration формы, не отдельный kind.

##### Связь с будущим FFI

`external type` — для типов, реализованных **в Nova-runtime**
(`nova_rt/*.h`/`.c`). Для типов, импортируемых из **сторонних
C-библиотек** (libuv handles, OS-libs), будет отдельный keyword
`extern("C") type` (Q-ffi, не реализуется сейчас). Семантика разная:

| Keyword | Реализация | C-name | Разрешён программисту |
|---|---|---|---|
| `external type` | Nova-runtime (`nova_rt/`) | `Nova_<Name>*` mangled | **нет** (только в `std.runtime.*` / `std.prelude.*`) |
| `extern("C") type` (TBD) | сторонний C/lib | as-is | да (FFI) |

##### Restriction: только `std.*`-whitelist

Программистский Nova-код **не пишет** `external type`. Этот keyword —
**экспозиционный**: только модули в `std.runtime.*` и `std.prelude.*`
имеют право его использовать. Компилятор **отклоняет** `external type`
в любом другом namespace'е:

```
error: `external type` is only allowed in `std.runtime.*` / `std.prelude.*`
       modules (this module is `myapp.foo`); for FFI to external C libraries
       a future `extern("C") type` keyword will be added (Q-ffi)
```

Whitelist реализуется через `manifest::is_stdlib_runtime_module ||
is_prelude_self_module` (тот же check что для `external fn` per D82).

##### Mangling и codegen

`external type X` **не эмитит** struct definition в C output —
определение живёт в runtime header (`nova_rt/<x>.h`):

```c
// nova_rt/string_builder.h
typedef struct {
    char*  data;
    size_t len;
    size_t cap;
} Nova_StringBuilder;
```

Codegen reference на `external type X` использует mangling `Nova_X*`
(pointer, opaque). Это идентично mangling user-defined record-типов
(`type Foo { ... }` → `Nova_Foo*`), что обеспечивает consistency.

| Nova-form | C-name |
|---|---|
| `let sb StringBuilder = ...` | `Nova_StringBuilder* sb = ...` |
| `fn f(sb StringBuilder)` | `void f(Nova_StringBuilder* sb)` |
| `external type Channel[T]` | `Nova_Channel*` (T erased в bootstrap) |

`emit_type_decl` **skip'ает** emission для `TypeDeclKind::Opaque`. Forward-
declarations (`typedef struct Nova_X Nova_X;`) skip'аются через
`BUILTIN_RUNTIME_TYPES` skip-list — runtime header сам предоставляет.

##### Validation

Аналогично D82, компилятор validate'ит что декларированный `external
type` **реально существует** в runtime (через `BUILTIN_RUNTIME_TYPES`
list + at-emit-time check). Если user добавит `external type FooBar`,
но `nova_rt/foo_bar.h` отсутствует → C-toolchain ошибётся при линковке
с `undefined reference to Nova_FooBar` при первом методе.

Полная Nova-side validation (компилятор знает все runtime-implemented
типы и заранее ошибётся «type 'FooBar' not implemented in runtime») —
требует **registry runtime types**, который сейчас живёт неявно в
`BUILTIN_RUNTIME_TYPES`. Q-codegen-runtime-types-registry — отдельная
задача аналогично D82 builtins.nv validation; bootstrap relies на
list maintenance.

#### Почему

##### Зачем нужен `external type`

1. **Source-of-truth for `nova doc`**. Программист (и AI) видит формальную
   декларацию типа в одном месте — `nova doc std.prelude.collections`
   покажет StringBuilder/WriteBuffer/ReadBuffer как canonical API.
   Раньше (Plan 04, до Plan 62.D.bis) типы существовали только как
   bare-name строки в D26 spec'е — не visible в tooling.

2. **Eligibility для cross-file resolve**. После formal declaration
   типы участвуют в R26+R27 resolve (Plan 35). User-код может писать
   `import std.prelude.collections.{StringBuilder}` или полагаться на
   auto-import через prelude.

3. **D29 W_PRELUDE_SHADOW работает**. User declaration `type StringBuilder
   { ... }` теперь генерирует warning (mirror Plan 62.A behavior для
   Option/Result). Раньше silent shadow.

4. **Symmetry с D82 `external fn`.** Если методы opaque-типа объявляются
   через `external fn`, сам тип должен иметь parallel form. Без D126
   semantic asymmetry: methods are first-class, type itself isn't.

5. **Future-proof для opaque user-types** (Channel, Region, mmap'ed buffers).
   Когда возникнет use-case, mechanism уже есть — нужно только relax
   whitelist (или ввести `extern("C") type` для FFI).

##### Почему не `opaque type`

- Один keyword (`external`) для двух concepts (`fn` и `type`) — снижает
  cognitive load. Прецедент: OCaml `external`, Dart `external`, Kotlin
  `external` — все используют один keyword для функций и (когда уместно)
  типов.
- `opaque` подразумевает abstraction-from-user-code, а semantic нужный
  здесь — **implementation-elsewhere**. `external` точнее семантически.

##### Почему не `#external` attribute

- Per D82 уже decision: «Атрибуты в Nova зарезервированы для тестов /
  dev-tools (Q-attributes). Modifier-форма единообразна с `export`/`mut`».
  D126 follows тот же principle.
- `#external` дублировал бы syntax: `#external type X` vs `external type X`.
  Choose one — modifier form for consistency.

##### Почему restrict scope

- Bootstrap MVP — программист **не должен** объявлять opaque types
  произвольно. Runtime backing — это **compiler-versioned artefact**, не
  user-extensible (см. D82 same argument). User-extensibility опасна:
  declaration без runtime impl приведёт к undefined-reference C errors.
- Future relaxation требует either:
  - **Plugin mechanism** (compiler plugin defines runtime — too heavy для
    bootstrap).
  - **FFI keyword `extern("C") type`** (Q-ffi) — для внешних libs, не
    Nova-runtime.

#### Что отвергнуто

- **Bare-name `type X`** (no modifier, no body) — parser ambiguity с
  newtype branch (`type X SomeType`).
- **`opaque type X`** — separate keyword без явного gain.
- **`#external` attribute** — modifier consistency lost.
- **`type X { _ runtime }` body** — magic body, parsing complexity.
- **Auto-discovery по runtime header presence** — magic, debugging
  nightmare. Explicit `external` лучше.
- **Включить methods в декларацию типа** (`external type X { fn @method ... }`):
  per D52 + Plan 11, methods orthogonal к type-decl (free-fn-style). Не
  ломаем consistency только для opaque types.

## Open questions с recommendations

1. **Кодген ABI: pointer vs value for opaque types?**
   Recommended: pointer (`Nova_X*`). Все 3 текущих типа уже pointer-based
   в runtime (`Nova_StringBuilder*` etc.) — consistency. Future opaque
   user-types (Channel, Region) — same pattern. Если когда-то потребуется
   value-type opaque (rare), добавить `external value type X` modifier —
   но не в bootstrap.

2. **Generic external types: monomorphize or erase?**
   Recommended: **erase** в bootstrap (как существующий `Option[T]` /
   `Result[T, E]` value-mono compromise). 3 opaque types non-generic —
   не actual issue. Future Channel[T] — runtime backing single (`Nova_Channel*`
   stores `void*` value); T-erased в bootstrap. Plan 59 Phase 7.5 если
   per-T mono нужен — отдельный задача.

3. **Pre-population infrastructure для `external type`?**
   62.A.bis ввёл `sum_schema_registry` с baseline `HardcodedBaseline` source.
   Аналог для opaque types — **не нужен** (нет variants/methods inside
   declaration, нет state to pre-populate). External_registry уже handles
   methods. Skip.

4. **`Display` protocol + `StringBuilder`-based pipeline для print/println**
   (Plan 62.B.bis dependency)?
   После 62.D.bis StringBuilder formally declared → Plan 62.B.bis может
   использовать `Display { fmt(sb StringBuilder) -> () }` (existing
   `std/prelude/protocols.nv` decl с правильным type-ref). До 62.D.bis
   Display.fmt'ы `sb StringBuilder` parameter резолвится через legacy
   path; после — через cross-file resolve. **Plan 62.B.bis может стартовать**
   независимо или waiting для 62.D.bis (recommended — после, cleaner).

5. **Update `infer_expr_c_type` hardcodes в emit_c.rs:18915-19040**?
   Out-of-scope для 62.D.bis. Recommended separate refactor (e.g.
   Plan 12 Ф.6 continuation): replace name-based `n == "StringBuilder"`
   matches на `self.external_registry.lookup(n, method).map(...return_c_type...)`.
   Не requires for 62.D.bis closure — текущий hardcoded path keeps working.

6. **Bare-name `type Name` (для Never/RuntimeNoneError) — same syntax?**
   No. `Never` / `RuntimeNoneError` — uninhabited sum types (empty-sum
   syntax, e.g. `type Never`); они НЕ opaque (runtime impl trivial — нет
   struct, нет storage). Different syntax extension required (Q-empty-sum).
   62.D.bis закрывает только `external type` для opaque-runtime-backed.

7. **Что если user пишет `external type Foo` в `myapp/foo.nv`?**
   Compile error per whitelist enforcement (см. Ф.1 step 3). Error message
   explicit: «only allowed in `std.runtime.*` / `std.prelude.*` modules».
   Future Q-ffi предложит `extern("C") type Foo` для bridges к C-libs.

### Common pitfalls

1. **`TypeDeclKind::Opaque` match exhaustiveness** — Rust will flag every
   non-exhaustive match site. Don't add `_ => ...` catch-all; explicitly
   match `Opaque => /* no-op semantically meaningful */`. Helps reviewers
   verify intent.

2. **Parser receiver-syntax overlap** — `external type StringBuilder` parses
   `StringBuilder` as type-name. Don't confuse with `external fn
   StringBuilder.new() -> Self` where `StringBuilder` is receiver-type
   (different parsing rules per D82).

3. **`is_prelude_self_module` covers `std.prelude.collections`** — verified
   via manifest.rs:392-402. `external type` whitelist check (`is_stdlib_runtime_module
   || is_prelude_self_module`) will allow declarations in collections.nv
   correctly.

4. **`PRELUDE_VERSION` bump in 2 test files** — easy to miss
   `prelude_auto_import.nv:17` (legacy test) когда обновляешь только
   `iter_protocol_from_prelude.nv:99`. Both must match.

5. **Generic params не emitted в C** — `external type Foo[T]` parser
   принимает, type-checker регистрирует `generics: [T]`, codegen skip'ает
   emission. T erased в bootstrap. If/when generic opaque types become
   relevant — mono path requires separate work.

6. **D29 W_PRELUDE_SHADOW false-positives** — после миграции 3 имени
   станут prelude-visible. Если user-test случайно re-declare'ит — fixe
   локально or add `module X allow_prelude_shadow`. Likely zero hits
   (типы были известны компилятору, user их не shadow'ил).

7. **Spec D126 ordering** — table-of-contents в `03-syntax.md` следует
   alphabetical/numerical order но D126 после D108 в table. Insertion
   point — после D108 row. Body — append к концу файла (после D108
   body), consistent с pattern D124/D125 в `08-runtime.md`.
