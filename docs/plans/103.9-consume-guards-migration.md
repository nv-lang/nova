# Plan 103.9 — Consume guards migration V2 (GATED on Plan 100.7)

> **Статус:** 📋 proposed 2026-05-25, **GATED** — не starts пока Plan 100.7 ready
> **Приоритет:** P2 (V2 enhancement; V1 уже shipped через 103.1-103.8)
> **Оценка:** ~2-2.5 dev-day
> **Зависимости (hard gates):** Plan 100.1 ✅ (D133 consume foundation), Plan 100.7 ✅ (stdlib migration playbook with Mutex pilot)
> **Зависимости (соft):** Plan 103.3-103.5 ✅ (V1 типы для миграции), Plan 62.F.bis ✅ (edition versioning для deprecation)
> **Master:** [Plan 103](103-sync-primitives-spec-formalization.md). Design M5/M15/M16 применимы.

## Зачем

Plan 103 V1 (sub-plans 103.1-103.8) ship'ит sync primitives с
традиционным `lock()/unlock()`, `acquire()/release()` pattern. **V1
mitigation:** `with_lock(fn)` / `with_read(fn)` / `with_permit(fn)`
helpers — closure-form preferred pattern, защищает от забытого release
через Plan 20 `defer`.

**V2 (Plan 103.9, gated on Plan 100.7):** переводит на **`MutexGuard
consume`** API — guard-returning `lock()` с RAII drop = unlock через
Plan 100 consume-mechanism. Защита statically (compile-time) от:
- unlock-без-lock (impossible — guard required для unlock);
- double-unlock (consume = single-use);
- забытого unlock (linear-type enforcement обязывает consume);
- aliasing guard'а между fibers (move-only semantics).

**Plan 100.7 явно перечисляет** «Mutex+Lock-guard (std/sync)» как
один из **4 pilot migrations** stdlib. Plan 103.9 — реализация этого
pilot (плюс расширение на RwLock, Semaphore, Once).

**M16 (master):** Atomics НЕ получают consume никогда (shared-state
primitive, не resource).

## Scope

### In scope

**4 семейства мигрируются на consume guards:**

1. **`Mutex` → `MutexGuard consume`:**
   - `Mutex.lock() -> MutexGuard consume` (new primary API)
   - `Mutex.try_lock() -> Option[MutexGuard consume]`
   - `Mutex.try_lock_for(Duration) -> Option[MutexGuard consume]`
   - `MutexGuard.unlock(consume self)` (explicit release; auto-drop
     also works через Plan 100.4 cleanup-on-drop)
   - `with_lock(fn)` helper становится thin wrapper над guard-returning core.

2. **`RwLock` → `ReadGuard consume` / `WriteGuard consume`:**
   - `RwLock.read() -> ReadGuard consume`
   - `RwLock.write() -> WriteGuard consume`
   - `try_*` variants → `Option[Guard consume]`
   - `*Guard.unlock(consume self)`
   - `with_read(fn)` / `with_write(fn)` — thin wrappers.

3. **`Semaphore` → `Permit consume`:**
   - `Semaphore.acquire() -> Permit consume`
   - `Semaphore.try_acquire() -> Option[Permit consume]`
   - `Semaphore.try_acquire_for(Duration) -> Option[Permit consume]`
   - `Permit.release(consume self)` (explicit release; auto-drop
     possible через Plan 100.4)
   - `with_permit(fn)` — thin wrapper.

4. **`Once` → `OnceGuard consume`:**
   - `Once.try_start() -> Option[OnceGuard consume]` (returns Some
     для exactly-one fiber; None for subsequent — wait for completion).
   - `OnceGuard.commit(consume self)` — successfull completion; transitions
     Once to DONE.
   - `OnceGuard.abort(consume self)` — failed initialization; transitions
     Once to POISONED (или back to NEW — D-block decides, similar to
     Plan 100.7 commit/rollback methods как distinct methods).
   - `Once.call_once(fn)` (V1) → thin wrapper using try_start + commit/abort
     based on closure result.

**Edition-versioning migration (Plan 62.F.bis):**

- **Edition 2026-Q3 (V2 introduction):**
  - Guard-returning API становится primary.
  - `Mutex.unlock()` (bare, без guard) → `#deprecated(since = "0.1")`
    + warning `W_BARE_UNLOCK_DEPRECATED`.
  - `with_lock(fn)` etc. — без изменения (continue работать через
    new wrapper).
- **Edition 2026-Q4 (V2 enforcement):**
  - Bare `unlock()` removed (or remains как escape with explicit
    `#unstable` marker).
  - Linear-type enforcement strict: missing `consume MutexGuard` →
    compile error.

**Spec:**
- **D174** «Sync primitives consume integration» в `spec/decisions/
  06-concurrency.md`. References D131-D166 (Plan 100 consume foundation),
  D169 (V1 Mutex), D170 (Semaphore), D171 (Once).
- Update D169-D171 cross-refs to D174 for V2 contracts.

**Pilot migrations** (4+ sites в `nova_tests/concurrency/`):
- `sync_mutex.nv` — migrate `m.lock()/m.unlock()` → `let g = m.lock();
  g.unlock()` или `let g = m.lock(); defer g.unlock()` или auto-drop.
- `sync_once.nv` — migrate `if o.run() { ...; o.done() }` →
  `match o.try_start() { Some(g) => { ...; g.commit() } | None => {} }`.
- New: `sync_rwlock_guards.nv`, `sync_semaphore_permit.nv`.

### Out of scope

- **Atomics** consume migration — M16 reject (never применимо).
- **WaitGroup / Barrier / CountDownLatch / Lazy / OnceCell** — не resource-
  like, no guard pattern. Stays as V1.
- **Condvar** consume — wait(guard) → new guard semantics tricky;
  defer to future plan (V3?) if user demand.
- **Performance regression** check — separate plan if measured.
- **LSP quick-fix** автоматическая миграция `lock()/unlock()` →
  guard form — Plan 100.8 scope (LSP quick fixes для consume),
  не 103.9 scope.

## API design

### `Mutex` V2

```nova
module runtime.sync

/// Guard для locked Mutex. Consume-typed (Plan 100 D131): обязан
/// быть consumed (через unlock() или auto-drop).
#stable(since = "0.1")
type MutexGuard consume {
    // opaque; internal ref к Mutex
}

#stable(since = "0.1")
export external fn Mutex mut @lock() -> MutexGuard consume

#stable(since = "0.1")
export external fn Mutex mut @try_lock() -> Option[MutexGuard consume]

#stable(since = "0.1")
export external fn Mutex mut @try_lock_for(timeout Duration) -> Option[MutexGuard consume]

#stable(since = "0.1")
export external fn MutexGuard @unlock(consume self)

// V1 helper — preserved, becomes thin wrapper
#stable(since = "0.2")
export fn Mutex mut @with_lock[R](body fn() -> R) -> R {
    let guard = self.lock()
    defer guard.unlock()
    body()
}

// V1 bare API — deprecated
#deprecated(since = "0.1", suggest = "Mutex.lock() returns MutexGuard; use guard.unlock() or with_lock(fn)")
#stable(since = "0.1")
export external fn Mutex mut @unlock()
```

### `RwLock` V2

```nova
#stable(since = "0.1")
type ReadGuard consume { /* opaque */ }

#stable(since = "0.1")
type WriteGuard consume { /* opaque */ }

#stable(since = "0.1")
export external fn RwLock mut @read() -> ReadGuard consume

#stable(since = "0.1")
export external fn RwLock mut @write() -> WriteGuard consume

#stable(since = "0.1")
export external fn ReadGuard @unlock(consume self)

#stable(since = "0.1")
export external fn WriteGuard @unlock(consume self)
```

### `Semaphore` V2

```nova
#stable(since = "0.1")
type Permit consume { /* opaque */ }

#stable(since = "0.1")
export external fn Semaphore mut @acquire() -> Permit consume

#stable(since = "0.1")
export external fn Permit @release(consume self)
```

### `Once` V2

```nova
#stable(since = "0.1")
type OnceGuard consume { /* opaque */ }

#stable(since = "0.1")
export external fn Once mut @try_start() -> Option[OnceGuard consume]

/// Successful completion — transitions Once to DONE, wakes waiters.
#stable(since = "0.1")
export external fn OnceGuard @commit(consume self)

/// Failed init — transitions Once to POISONED (or back to NEW —
/// see D174 decision). Wakes waiters с error throw on resume.
#stable(since = "0.1")
export external fn OnceGuard @abort(consume self)

// V1 helper — thin wrapper
#stable(since = "0.2")
export fn Once mut @call_once(body fn() -> ()) {
    match self.try_start() {
        Some(g) => {
            // try/catch для panic-safety
            // success → commit; panic → abort
            g.commit()
        }
        None => {}  // already done or in progress
    }
}
```

## Industry comparison

| Feature | Nova 103.9 V2 | Rust | Java | Kotlin | Go |
|---|---|---|---|---|---|
| **MutexGuard RAII unlock** | ✅ via consume | ✅ Drop trait | ❌ try/finally | ❌ withLock {} | ❌ Lock/Unlock pair |
| **Linear-type guard (cannot leak)** | ✅ via consume | ⚠ Drop ≠ linear (can be `forget`'d) | ❌ | ❌ | ❌ |
| **Distinct commit/abort на init guard** | ✅ M of Plan 100 unique | ❌ | ❌ | ❌ | ❌ |
| **No lifetime annotations** | ✅ (M of Plan 100 D157) | ❌ `MutexGuard<'a, T>` | n/a | n/a | n/a |

**Nova edge:** consume-types unique (Rust Drop ≠ guaranteed via
`mem::forget`; Java/Kotlin/Go нет statically-enforced); commit/abort
distinct methods (vs single drop) — Plan 100 P-преимущество.

## Tests breakdown

### Positive (8 файлов в `nova_tests/plan103_9/`)

1. `mutexguard_basic_unlock.nv` — `let g = m.lock(); g.unlock();` —
   compile + runtime correct.
2. `mutexguard_auto_drop.nv` — `{ let g = m.lock(); /* implicit drop */ }`
   — guard auto-released at scope end (Plan 100.4 cleanup).
3. `mutexguard_with_lock_wrapper.nv` — backward compat `with_lock(fn)`
   still works (preferred V1 pattern).
4. `readguard_writeguard_separated.nv` — RwLock guards distinct types.
5. `permit_release_explicit.nv` — Semaphore Permit.
6. `onceguard_commit_path.nv` — `try_start → Some(g) → g.commit()`.
7. `onceguard_abort_path.nv` — `try_start → Some(g) → g.abort()` —
   Once transitions to POISONED (or NEW; D174 decides).

   one file (deprecation warning emitted).

### Negative (10 файлов)

1. `mutexguard_double_unlock_neg.nv` — `let g = m.lock(); g.unlock();
   g.unlock();` → `EXPECT_COMPILE_ERROR E_CONSUMED_AFTER_USE`
   (Plan 100 D133).
2. `mutexguard_leak_neg.nv` — `let g = m.lock();` (без unlock) +
   scope end без auto-drop guard config → `EXPECT_COMPILE_ERROR
   E_CONSUME_NOT_CONSUMED` (Plan 100 strict mode).
3. `mutexguard_share_cross_fiber_neg.nv` — попытка передать `MutexGuard`
   через channel → `EXPECT_COMPILE_ERROR E_CONSUME_CROSS_FIBER`
   (Plan 100 D157 borrow).
4. `bare_unlock_deprecation_warn.nv` — `m.unlock()` (без guard) →
   `EXPECT_WARNING W_BARE_UNLOCK_DEPRECATED`.
5. `onceguard_no_commit_no_abort_neg.nv` — `try_start → Some(g);`
   scope end без commit/abort → `E_CONSUME_NOT_CONSUMED` (Plan 100).
6. `permit_release_then_use_neg.nv` — `let p = s.acquire(); p.release();
   /* try use p */ p.something()` → `E_CONSUMED_AFTER_USE`.
7. `readguard_with_writeguard_mix_neg.nv` — `let r = rw.read(); let w =
   rw.write();` ?? — single-thread blocking forever или error
   (D174 decides).
8. `mutexguard_passed_to_other_fiber_neg.nv` — same as #3 cross-fiber.
9. `mutexguard_in_realtime_neg.nv` — `realtime { m.lock() }` —
   `E_REALTIME_SYNC_PARK` (existing 103.6).
10. `onceguard_commit_and_abort_neg.nv` — try commit + abort one
    guard sequence — second one fails compile (`E_CONSUMED_AFTER_USE`).

| Риск | Mitigation |
|---|---|
| Plan 100.7 не закрыт — гейт держится | Plan 103.9 GATED placeholder; periodic re-check после Plan 100 progress |
| Migration ломает backward compat | Edition-versioning (Plan 62.F.bis); `with_lock(fn)` wrappers preserve old usage |

| `OnceGuard.commit()` vs `.abort()` semantics — какое default для Drop без commit/abort? | D174 decision: drop без явного → `abort` (safer; Once back to NEW or POISONED, user may retry); Plan 100.4 cleanup-on-failure has precedent |

| Cross-fiber guard passing — Plan 100 D157 view borrow handles | Test #3 explicitly verifies |
| User confusion `with_lock` vs `lock()` guard | D173 (Plan 103.7) AI-guidance updated to recommend `with_lock(fn)` as default V1+V2; bare `lock()` для advanced control |
