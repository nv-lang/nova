## 1. Проблема

Сейчас (D29) импорт **всегда** — полный путь от корня пакета:
`import std.encoding.hex.{decode}`. Относительные пути отвергнуты.

Цена:
- **Refactor-fragility.** Перемещаешь поддерево модулей — ломаются
  **все** импорты на него: и внутренние (между модулями поддерева), и
  внешние. Внутренние ломаться не должны — поддерево самосогласовано.
- **Многословность** для тесно связанных соседних модулей.
- D29 rev-3 уже сделал **декларацию** модуля короткой (`parent.target`,
  refactor-safe) — но **импорт** остался длинным и от корня. Асимметрия.

Индустрия: Go — относительных нет (строго, неудобно). TS — безграничные
`./`/`../` (удобно, но `../../../../`-hell + ломкость при смене глубины).
**Rust — золотая середина:** `crate::` (абсолютный) + `super::`/`self::`
(относительный, **в пределах крейта**); наружу крейта — имя крейта.

## 2. Решение — package-scoped относительные импорты

Добавить `./` и `../` префиксы импорта, **резолвящиеся относительно
директории импортирующего файла**, но **строго в пределах своего
пакета**. Между пакетами — по-прежнему полный путь от корня.

- Внутри пакета поддерево **перемещаемо целиком** — относительные
  импорты переживают перемещение.
- Межпакетная location-independence сохранена (единица, важная для
  «откуда это» — пакет, а не файл).
- Nova оказывается **между Go (слишком строго) и TS (слишком вольно)** —
  паритет с Rust-моделью.

### Почему это не противоречит AI-first (примирение с D29)

D29 отвергал относительные пути ссылкой на локальность («LLM не должен
искать, откуда имя»). Scoped-форма это **сохраняет**: явный префикс
`./`/`../` сам по себе — сигнал «это сосед, внутри пакета»; LLM видит
ровно это, без поиска. Нарушал бы принцип безграничный *межпакетный*
относительный импорт — его план **не** вводит. `./foo` прозрачно
говорит «сосед» — это «видно откуда», а не магия.

## 3. Дизайн (зафиксирован)

### 3.1. Синтаксис

```nova
import ./sibling.{Name}            // модуль в директории текущего файла
import ../neighbor.{Name}          // на уровень вверх
import ../../shared.types.{T}      // на два уровня вверх, затем путь
export import ./facade_part.{X}    // re-export относительного — допустим
import ./sub.{Y} as Z              // алиасы работают как обычно
```

- Префикс — одна или более групп `./` / `../`. `./` = директория
  текущего файла, `../` = родительская, и т.д.
- После префикса — обычный точечный путь модуля, резолвящийся **от
  директории, на которую указал префикс** (теми же правилами, что
  абсолютный путь резолвится от корня).
- **Bare-путь без префикса остаётся абсолютным** (от корня пакета) —
  поведение не меняется.

### 3.2. Отсутствие неоднозначности — by construction

Префикс `./`/`../` — **синтаксический дискриминатор**: `import a.b` —
всегда абсолютный (от корня), `import ./a` / `import ../a` — всегда
относительный. Правило приоритета «локальное vs от корня» **не нужно** —
их нельзя перепутать. (Тот же приём у TS: `import "foo"` vs
`import "./foo"`.) Это сознательно лучше, чем неявная относительность
Python (`import foo` может быть и тем, и тем).

### 3.3. Граница пакета — жёсткая

Относительный импорт **не может выйти за корень своего пакета**. Пакет =
директория workspace-member (D78; `nova.toml` members — `std/`,
`examples/`, `nova_tests/`, …). Если цепочка `../` поднимается **выше**
корня пакета → **compile error** «relative import выходит за границу
пакета `X`; используйте полный путь от корня». Межпакетные ссылки —
только абсолютным путём.

### 3.4. Взаимодействие с существующими правилами

- **Folder-modules / peers.** Peer-файлы одного folder-модуля делят
  namespace — импорт между ними не нужен и так. `./` из peer-файла
  резолвится в директории folder-модуля → достаёт nested sub-модули
  (`import ./billing.{Invoice}` из `admin/users.nv` → `admin.billing`).
  Если относительный импорт указывает на peer **того же** модуля →
  диагностика «это peer текущего модуля — namespace общий, импорт не
  нужен».
- **`internal/` (rule H).** Относительный импорт `internal/`-модуля
  по-прежнему проверяется правилом H (доступ только из descendants
  `<owner>/`). Относительная форма границу H **не ослабляет**.
- **Циклы.** Резолвятся к canonical path; cycle-detection уже
  canonical-path-based (`imports.rs`) → относительный и абсолютный
  импорт одного файла дают один узел графа. Детекция циклов — **бесплатно**.
- **Декларация модуля.** `module parent.target` (D29 rev-3) — не
  меняется; относительные импорты влияют только на **запись импорта**,
  не на identity-декларацию. Вместе rev-3 (короткая декларация) + rev-4
  (относительный импорт) делают поддерево пакета полностью
  relocatable.
- **`export import ./X`** — re-export относительного модуля: относительная
  резолюция происходит в re-export'ящем файле; наружу имя уходит как
  часть API этого модуля. Без особенностей.

## 4. Фазы

### Ф.1 — Лексер/парсер: relative-prefix в import-path (~1-2 дня)

- Парсер `import` принимает в начале пути `( ./ | ../ )+` затем точечный
  путь. `/` валиден только в этой позиции (вне import — оператор
  деления; парсер импорта в отдельном состоянии — без конфликта).
- AST: `ImportPath` получает поле `relative_prefix: Vec<RelSeg>` (`Cur`/
  `Parent`-сегменты) либо `None` для абсолютного.
- Негативы парсера: `./` в середине пути, `.../`, пустой `./.{X}`.

### Ф.2 — Резолвер: re-rooting + граница пакета (~2-3 дня)

- `resolve_import_path` (`imports.rs`, candidate search roots ~`:1240`):
  при наличии `relative_prefix` стартовый корень = директория
  импортирующего файла, сдвинутая по `../`-цепочке; иначе — корень
  пакета (как сейчас).
- **Package-boundary enforcement (§3.3):** определить корень пакета
  (workspace-member dir, через `manifest.rs`/`find_repo_root`); если
  `../`-цепочка вышла выше — ошибка E84xx.
- Canonical-path резолюция → cycle-detection и inline-expansion
  (`resolve_imports_inline`) работают без изменений.

### Ф.3 — Production-диагностика (~1 день)

Каталог ошибок (production-grade, с suggestion'ами):
- escape-boundary: «выходит за границу пакета `X` — полный путь».
- not-found: «модуль `foo` не найден в директории `<dir>`» + did-you-mean
  по соседям.
- peer-collision: «относительный импорт указывает на peer текущего
  модуля — namespace общий, импорт не нужен».
- intermediate-not-dir: промежуточный сегмент не директория.

### Ф.4 — Аудит взаимодействий (~1 день)

Эмпирически verify каждый пункт §3.4: folder-modules/peers, `internal/`
rule H, `export import` re-export, cycle-detection, `_test.nv`-peers.
Где правило уже работает «бесплатно» — зафиксировать тестом-якорем.

### Ф.5 — Тесты (~1-2 дня)

- Позитив: `./sibling`, `../neighbor`, `../../shared`, nested sub-module
  через `./`, `export import ./X` re-export, алиасы.
- Негатив (EXPECT_COMPILE_ERROR): escape-boundary, not-found, peer-

- Фикстуры — folder-module `nova_tests/plan84/` (несколько пакетов-

## 5. Acceptance

- `import ./X` / `import ../X` резолвятся относительно директории файла;
  поддерево пакета перемещается без правки внутренних импортов.
- Относительный импорт за границу пакета → детерминированная
  compile-error.
- Абсолютные импорты не затронуты (фича аддитивна); 0 регрессий.
- Циклы через относительные импорты ловятся (тест-якорь).
- `internal/` rule H соблюдается и для относительной формы.
- spec D29 rev-4 опубликован.

## 6. Честная рамка

- **Аддитивно** — старый код (всё на абсолютных путях) не меняется;
  миграция не требуется.
- **Scoped, не unrestricted** — `../` за границу пакета запрещён
  сознательно (избегаем TS `../../../../`-hell). Если в будущем
  понадобится — отдельное решение, но по умолчанию: нет.
- Паритет: Rust (`super`/`self` в крейте) — прямой аналог; Go (нет
  относительных) — Nova даёт больше эргономики осознанно; TS
  (безграничные) — Nova строже на межпакетной границе.
