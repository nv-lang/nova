# Plan 85.3 — Полное тестовое покрытие протоколов конверсии

> **Статус:** ✅ ЗАКРЫТ 2026-05-22 (Ф.1–Ф.3)
> **Приоритет:** P1 | **Оценка:** ~1 день
> **Родительский план:** [Plan 85](85-builtin-types-test-coverage.md)
> **Зависимости:** нет (standalone)
>
> **Итог:** каталог `nova_tests/protocols/conversion/` — 9 файлов
> (8 позитивных + 1 негативный). From / Into / TryFrom / TryInto.
>
> **Найден и исправлен codegen-баг:** D73 `.into()` auto-derive не делал
> overload-резолюцию — при нескольких `from`-overload'ах на одном типе
> (`MyNum.from(int)` + `from(str)` + `from(bool)`) все `.into()`
> резолвились в первый overload → CC-FAIL. Фикс в `emit_c.rs` —
> into-derive (и симметрично try_into-derive) выбирает mangled C-имя по
> типу источника через `method_overloads`-реестр.
>
> **Известное ограничение** (documented, не fixed): вызов `T.from(...)`
> на type-параметре ВНУТРИ тела generic-функции не мономорфизируется
> — `[M-generic-static-method-on-typevar]` в simplifications.md.
> generic_from_bound / generic_try_bound тестируют bound parse +
> инстанцирование + enforcement (что работает).
>
> **Корректировки синтаксиса плана:** примеры в плане были в Rust-стиле
> (`impl From[f64] for Celsius { fn from }`) — в Nova `From` это
> static-метод `fn T.from(v X) -> Self`, `Into` — instance `@into`,
> структурно, без `impl`-блоков. Тесты — по фактическому синтаксису.

## Протоколы (std/prelude/protocols.nv)

```nova
type From[T] protocol {
    from(t T) -> Self
}

type Into[U] protocol {
    into() -> U
}

type TryFrom[T, E] protocol {
    try_from(t T) -> Result[Self, E]
}

type TryInto[U, E] protocol {
    try_into() -> Result[U, E]
}
```

## Примеры тестов

```nova
// from_basic.nv
type Celsius { value f64 }
impl From[f64] for Celsius {
    fn from(v f64) -> Celsius => Celsius { value: v }
}

test "Celsius.from(f64)" {
    ro c = Celsius.from(36.6)
    assert(c.value == 36.6)
}

// into_derive.nv
test "f64.into() gives Celsius via From auto-derive" {
    ro c Celsius = (36.6).into()
    assert(c.value == 36.6)
}

// try_from_ok.nv
type Port { n u16 }
impl TryFrom[int, str] for Port {
    fn try_from(v int) -> Result[Port, str] {
        if v >= 1 && v <= 65535 {
            Ok(Port { n: v as u16 })
        } else {
            Err("port out of range")
        }
    }
}

test "Port.try_from valid" {
    match Port.try_from(8080) {
        Ok(p) => assert(p.n == 8080 as u16)
        Err(_) => assert(false)
    }
}

test "Port.try_from invalid" {
    assert(Port.try_from(99999).is_err())
}
```

## Acceptance criteria

- [x] User-defined `From` — от ≥3 типов-источников (int/str/bool, from_multi)
- [x] `Into` auto-derive — вызов `into()` работает (обе формы пары)
- [x] `TryFrom` — Ok и Err пути
- [x] `TryInto` — вызов `try_into()` работает (прямой impl)
- [x] Generic bound `From`/`TryFrom` — парсится, инстанцируется, enforced
- [x] Compile-error при несоответствии bound — `neg_missing_impl.nv`
- [x] Ограничение `T.from()` на typevar в generic-теле — задокументировано
      (`[M-generic-static-method-on-typevar]`)

## Known limitations (pre-existing)

- `match Result[RecordType, E]` с binding Ok-payload → field-access через nova_int — pre-existing codegen bug (зафиксирован в simplifications.md [M-result-record-payload-match]). Тесты пишутся с is_ok()/is_err() или pattern без field-access до фикса в Plan 59.

## Non-scope

- TryFrom для числовых типов (overflow) — Plan 75 ✅
