# Plan 91.8c — Generic array API: sort/min/max/binary_search + _by (D185)

> **Подплан Plan 91** ([91-stdlib-mvp-for-0.1.md](91-stdlib-mvp-for-0.1.md)).
> Статус: ✅ CLOSED (2026-06-17, branch plan-91-8c).
> Зависимости: **Plan 91.8a** (Compare.compare -> int), желательно 91.8b закрыт но не блокер.

## Мотивация

Plan 91.3 даёт `[]int @sort()` (concrete). Plan 91.7 закрепил `-> @` mut chain
для array builtins. Остаётся **generic API** для `[]T` где `T Compare`:

- Сейчас `vs.sort()` работает только для `[]int`. Для `[]str`, `[]u64`, user types — compile error.
- В Plan 91.7 followup `[M-91.7-sort-generic]` зафиксирован — теперь делаем.
- После 91.8a `str.compare() -> int` consistent с Compare; sort на `[]str` должен работать без особых обёрток.

## Scope

### Generic methods (Compare bound)

```nova
fn[T Compare] []T mut @sort() -> @
fn[T Compare] []T @binary_search(target T) -> Option[int]
fn[T Compare] []T @min() -> Option[T]
fn[T Compare] []T @max() -> Option[T]
```

### Callback methods (без bound на T)

```nova
fn[T] []T mut @sort_by(cmp fn(T, T) -> int) -> @
fn[T] []T @min_by(cmp fn(T, T) -> int) -> Option[T]
fn[T] []T @max_by(cmp fn(T, T) -> int) -> Option[T]
fn[T] []T @binary_search_by(cmp fn(T) -> int) -> Option[int]
```

`binary_search_by` принимает 1-arg cmp возвращающий int сравнивая target с element (как Rust's `binary_search_by`).

### Algorithm

- **sort:** stable insertion sort для MVP (Plan 91.3 уже это для int). Для generic — тот же подход, но через `@compare`. Followup `[M-91.8c-pdq-sort]` для pdq-sort на больших данных.
- **binary_search:** classic O(log n) с `@compare` для сравнения с target.
- **min/max:** linear scan, O(n).
- **min_by/max_by:** linear scan с callback.

### Backward compat

Existing `fn []int mut @sort() -> @` в `std/sort.nv` (Plan 91.3) — оставляем как **concrete fast-path**. Generic версия добавляется параллельно; overload resolution выбирает concrete для `[]int`.

После 91.8c можно concrete удалить (D183 fast-path не нужен — generic mono per T даёт identical C code). Решение: **оставить пока** для ясности.

## Spec / D-block

### D185 — Generic array methods bound на Compare

**Файл:** `spec/decisions/02-types.md`.

**Title:** `D185. Generic array API: sort/min/max/binary_search + _by callbacks`

**Content:**

1. Generic surface API:
   - `fn[T Compare]` для bound-based methods
   - `fn[T]` + cmp callback для `_by` variants
2. Stable sort guarantee (insertion sort для MVP).
3. binary_search behavior: returns `Option[int]` с found-index; `None` если absent.
4. min/max: `Option[T]` — `None` для empty.
5. Algorithm complexity: O(n²) sort, O(log n) search, O(n) min/max. Followup `[M-91.8c-pdq-sort]`.
6. Связь: D183 (Compare), D181 (array mut `-> @`), D178 (str.compare).

## Effort + risk

**Estimated:** 1-2 dev-days. Generic mono для builtin `[]T` — нетривиально (Plan 101 покрывает user types; builtin требует additional emit_c.rs paths). Plus 11+ tests, spec, docs.
**Risk:** medium-high. Generic mono per T для array builtin может зацепить existing array-method dispatch. Type-checker `[T Compare]` bound enforcement через protocol satisfaction — fragile.

## Принцип реализации — production-grade

> Implementation должна быть **production-grade**. Никаких MVP shortcut'ов
> вроде «hardcoded для []int, []str и потом extend». Generic dispatch
> должен работать **uniformly** для любого T satisfying Compare
> (int/str/u64/user records). Insertion sort алгоритм — для MVP OK,
> но не «temporary» — это canonical implementation для small/medium arrays.
>
> Все edge cases (empty, single-element, all-equal, reversed, with NaN/INF
> для float) — должны быть covered тестами. Stable property — формально
> проверена через property test.
>
> Generic mono per T — concrete C code для каждого T, **NOT** type erasure
> через void*. Performance должна быть равна или лучше manual implementation.
>
> Если упирается в архитектурную проблему — **поднять вопрос пользователю**,
> не обходить.

## Followup: [M-91.8c-direct-index-method] — CLOSED 2026-06-17

**Проблема:** `@[i].method()` (вызов метода прямо на результате индексирования self) не работал
без промежуточного биндинга — `ro tmp = @[i]; tmp.method()` требовалось как workaround.

**Root cause:** `compute_array_elem_type_for_obj` в emit_c.rs (~line 14248) не имел ветки для
`ExprKind::SelfAccess`, поэтому `@[j]` Index-выражение не могло вычислить тип элемента через
self-ресивер. Кроме того, `emit_monomorphized_method` не устанавливал
`array_element_types["nova_self"]` при входе в метод.

**Fix (два изменения в compiler-codegen/src/codegen/emit_c.rs):**
1. Добавлена ветка `ExprKind::SelfAccess` в `compute_array_elem_type_for_obj` (~line 14248),
   возвращает `array_element_types.get("nova_self")`.
2. В `emit_monomorphized_method` после регистрации `var_types["nova_self"]` вычисляется конкретный
   C-тип элемента из recv_c: `Nova_Vec____<elem>*` — через `generic_type_instance_info` с fallback
   на `current_type_subst`; `NovaArray_<elem>*` — strip prefix + restore `*` для struct-pointer.
   Результат регистрируется под `array_element_types["nova_self"]` и очищается при выходе.

**Тест:** `nova_tests/plan91_8c/neg_direct_baseline.nv` — `@[mid].compare(target)` напрямую
без промежуточного биндинга.
