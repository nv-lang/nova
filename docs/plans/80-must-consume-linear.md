# Plan 80: must-consume — линейные значения (обязательное потребление)

> **Создан 2026-05-21.** Выделен из обсуждения Plan 73 (`consume`, D131)
> и Plan 77 (`-> @`, раздел «Потенциал лучше Rust»).
>
> **Статус:** 📋 proposed, не начат.
>
> **Цель:** дать языку *линейные* значения — значение, которое
> компилятор **обязывает** потребить ровно один раз до конца области
> видимости.

---

## Идея простыми словами

`consume` (D131) сейчас **affine**: значение можно использовать **не
больше одного раза** (0 или 1). Совсем не трогать — допустимо.

Линейное значение — это **ровно 1**: его **обязаны** потребить один раз
до конца scope. Забыл потребить → ошибка компиляции.

Классический пример — `Transaction`:

```
let tx = db.begin()
tx.exec("...")
// конец функции — tx не закоммичен и не откачен → ОШИБКА КОМПИЛЯЦИИ
```

Компилятор требует завершить: `tx.commit()` **или** `tx.rollback()`.

## Зачем — где это лучше Go/Rust/TS

| Язык | Механизм | Сила |
|---|---|---|
| **Rust** | `#[must_use]` | ⚠️ только **warning**, легко проигнорировать; `Drop` срабатывает всегда, но не может *выбрать* `commit` vs `rollback` |
| **Go** | — | нет; полагается на код-ревью / `defer` |
| **TS** | — | нет |
| **Nova** | линейный тип | ✅ **compile error**; заставляет выбрать консьюмер |

Rust `Drop` гарантирует, что *что-то* произойдёт на выходе, но не может
заставить программиста выбрать между двумя завершениями
(`commit` / `rollback`). `#[must_use]` — лишь warning. Линейность Nova —
**жёсткая** гарантия выбора, которой нет ни у Go, ни у Rust, ни у TS.

## Use-cases

- `Transaction` — обязан `.commit()` / `.rollback()`.
- File / socket handle — обязан `.close()`.
- Одноразовый токен / разрешение (capability) — обязан быть предъявлен
  / отозван.
- Builder, который обязан завершиться `.build()` (стыкуется с Plan 77
  `-> @`).

## Дёшево — почему

Consume-checker (Plan 73) **уже** отслеживает состояние каждой
переменной: `Live` / `Consumed` / `MaybeConsumed`, со слиянием по
ветвям и пессимистичными циклами. Линейность добавляет **одну проверку
на выходе из scope**:

- линейная переменная `Live` на выходе → ошибка «не потреблена»;
- `MaybeConsumed` (потреблена не на всех путях) → ошибка «потреблена
  не на всех путях».

Новой dataflow-машинерии не нужно — только финальная проверка scope.

## Открытые решения (закрыть в Ф.1)

1. **Где метить линейность — на ТИПЕ или на binding'е?**
   Рекомендация — **на типе**: `Transaction` линеен по своей природе,
   независимо от того, в какой переменной лежит; пометка на типе
   «заразна» правильно (получил `Transaction` где угодно — обязан
   потребить). Пометка на binding'е (`linear let tx = ...`) — слабее и
   локальнее.
2. **Каким синтаксисом метить тип?** НЕ выдумывать — свериться с
   существующим синтаксисом квалификаторов Nova (`consume` —
   keyword-квалификатор; у `type` атрибутов сейчас нет). Кандидат —
   keyword-квалификатор на объявлении `type`, по аналогии с `consume`.
   Финальное решение — за автором языка в Ф.1.
3. **Линейный тип в поле структуры** — consume-checker видит только
   локальные переменные, поля не отслеживает. Рекомендация —
   **запретить** линейный тип в полях структур (compile error), чтобы
   не было тихой дыры. Полноценная field-линейность — отдельный
   будущий план (honest defer).
4. **Что считается «потреблением»?** Вызов `consume`-метода, передача в
   `consume`-параметр, `return <линейная-переменная>` (обязательство
   переходит к caller'у). Переприсваивание `Live` линейной переменной
   (`tx = new()`) = выброс непотреблённого значения → ошибка.

## Фазы

- **Ф.1 — spec + решения.** D-блок (резерв **D133**): семантика
  линейности (ровно 1), отличие от affine `consume` (≤ 1), список
  «что потребляет». Закрыть 4 открытых решения выше (синтаксис метки,
  тип vs binding, поля структур, переприсваивание).
- **Ф.2 — AST + parser.** Распознать пометку линейного типа на
  объявлении `type`; хранить флаг в AST (`TypeDecl.linear: bool` или
  аналог).
- **Ф.3 — consume-checker: проверка на выходе из scope.** Линейная
  переменная в состоянии `Live` на выходе из её scope → ошибка
  «значение типа T обязано быть потреблено». Корректно обработать
  `return <линейная>` (= потребление, обязательство → caller'у) и
  переприсваивание `Live` линейной переменной (= ошибка).
- **Ф.4 — ветвления и циклы.** Линейная переменная обязана быть
  потреблена на **всех** путях: `MaybeConsumed` на выходе → ошибка
  «потреблена не на всех путях». Циклы — существующая пессимистичная
  логика (потребление в теле цикла, который может не выполниться, →
  `MaybeConsumed` → ошибка) переиспользуется как есть.
- **Ф.5 — запрет линейного типа в полях структур** (per Ф.1 решение):
  compile error при объявлении поля линейного типа.
- **Ф.6 — тесты + пример.** Позитив: `tx.commit()`; `if c { commit }
  else { rollback }`; `return tx`. Негатив: забыл потребить; потребил
  не на всех путях; потребил в необязательном цикле; переприсвоил
  `Live`; линейный тип в поле структуры. Полноценный
  `Transaction`-like пример в фикстуре / `examples/`.
- **Ф.7 — spec sync** + `simplifications.md` (если что-то отложено —
  honest defer) + обновление `README.md`.
