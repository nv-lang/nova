# Plan 140.1 — Custom contract message (`requires expr, "message"`)

> **Создан:** 2026-06-12.  **Статус:** 📋 PLANNED (gated на Plan 140).
> **Эстимат:** ~1–1.5 dev-day.  **Model:** Opus + Thinking ON.
> **Зависит от:** Plan 140 (contracts enforced in release — сообщение проходит через
> `nova_contract_violation`, который после 140 срабатывает и в release).
> **Закрывает:** `[M-140-contract-message]` (backlog).

---

## Проблема

Нарушение `requires`/`ensures`/`invariant` печатает только **авто-текст** из выражения:
```
contract requires failed in `withdraw`: amount > 0 at bank.nv:42
```
(`nova_contract_violation(kind, fn, contract_src, file, line)`, contracts.h).

`assert`/`debug_assert` **уже** принимают пользовательское сообщение (`assert(cond, "msg")`,
Plan 62 / D84 arity-overload). Декларативные контракты — **нет**. Человеко-читаемое сообщение
(«amount must be positive») улучшает диагностику и доменную ясность.

---

## Решение — опциональная строка после выражения контракта

```nova
fn withdraw(amount int) -> ()
    requires amount > 0, "amount must be positive"
    ensures @balance >= 0, "balance never goes negative"
{ ... }
```

`, "<message>"` — **опциональный** строковый литерал после выражения. При нарушении
сообщение показывается **вместе** с авто-контекстом (kind / fn / выражение / file:line):
```
contract requires failed in `withdraw`: amount must be positive (amount > 0) at bank.nv:42
```
Без сообщения — поведение как сейчас (back-compat).

  `requires`/`ensures`/`invariant`. Семантика: строковый литерал; показывается в violation.
  Композиция с Plan 140 (release-enforcement) — сообщение в release-abort.
- Q-block если возникнет открытый вопрос (напр. интерполяция в сообщении — предложить defer).
- **Commit:** `spec(plan140.1): optional custom message on requires/ensures/invariant`

### Ф.1 — Parser + AST (~0.5d)
- Парсер: после contract-выражения опциональный `, <string-literal>`. Грамматика
  `requires <expr> ("," <string-lit>)?` (и ensures/invariant).
- AST: contract-узел несёт `message: Option<String>` (или Span на литерал).
- Негатив: не-строковый-литерал в позиции сообщения → `E_CONTRACT_MESSAGE_NOT_STRING`.
- **Commit:** `feat(plan140.1 Ф.1): parse optional contract message`

### Ф.2 — Codegen + runtime (~0.5d)
- `nova_contract_violation` — добавить опциональный `const char* user_msg` (NULL если нет).
  Формат: при наличии — `... failed in `fn`: <user_msg> (<expr>) at file:line`; иначе —
  текущий формат. Back-compat сигнатуры (все call-сайты обновить).
- Codegen эмитит user_msg-аргумент из AST. Работает и для proven-elision (нет проверки —
  нет сообщения) и для unproven-в-release (Plan 140).
- **Commit:** `feat(plan140.1 Ф.2): thread custom message into contract violation`

### Ф.3 — Тесты + docs + close (~0.25d)

  neg (`E_CONTRACT_MESSAGE_NOT_STRING`), back-compat (контракт без сообщения).
- spec финал; project-creation.txt; simplifications.md; nova-private/discussion-log.md;
  backlog: закрыть `[M-140-contract-message]`; статус этого плана → CLOSED.
