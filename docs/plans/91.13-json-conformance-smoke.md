## Зачем

Plan 91 (std MVP для 0.1 release) держит `Ф.3 JSON conformance` как
deferred. JSON `Json.parse` / `JsonValue.@into()` — критический shipping
API; conformance test (roundtrip `parse → encode == original`) подтверждает
готовность к 0.1.

При попытке написать conformance test обнаружено что **`std/encoding/json.nv`
использует `str.from_codepoint(code)` для Unicode escape parsing (`\uXXXX`),
но функция НЕ объявлена в stdlib**. Compiler выдавал auto-generated
`Nova_str_static_from_codepoint(...)` call но C runtime не имел этого
символа → linker errors при попытке полной conformance проверки.

## Что вошло (V1 partial — этот PR)

### Ф.1 — `str.from_codepoint(int) -> str` API

**Зачем:** D54 (Plan 34 Ф.5.2) запретил `int as char` direct cast. Workaround
— `char.try_from(n)?` — adds Fail effect indirection. Иногда caller уже
validated codepoint range (JSON parser проверяет hex digits `⊆ [0,
0x10FFFF]`) — нужен **прямой "known-valid" path** без Fail unwrap.

**Реализация:**
- `std/runtime/char.nv`: `export external fn str.from_codepoint(cp int) -> str`
- `compiler-codegen/src/codegen/runtime_registry.rs`: добавлен RuntimeFn
  entry (rooted в `std.runtime.char`)
- `compiler-codegen/nova_rt/string_builder.h`: inline alias
  `Nova_str_static_from_codepoint(cp) → Nova_str_static_from_char(cp)`.
  Тот же UTF-8 encode impl (Plan 04, ~80 строк up в file).
- Semantics: invalid codepoint (`< 0` или `> 0x10FFFF`) → empty `""`
  (silent — predictable degradation, не panic). Strict variant — followup
  если потребуется.

### Acceptance criteria (V1 — этот PR)

- [x] `str.from_codepoint(65) == "A"` (ASCII)
- [x] `str.from_codepoint(0x041F) == "П"` (BMP — Cyrillic, 2-byte UTF-8)
- [x] `str.from_codepoint(0x1F600) == "😀"` (supplementary — emoji,
      4-byte UTF-8)
- [x] `str.from_codepoint(0) == "\\x00"` (NUL — 1-byte, len = 1)
- [x] `str.from_codepoint(-1) == ""` (negative — invalid)
- [x] `str.from_codepoint(0x110000) == ""` (out-of-range — invalid)
- [x] Targeted regression: plan48_1 / plan90_1 / plan73 / plan77 /
      plan91_8a_2 — 0 regressions

## Что НЕ вошло (V2 followup'ы)

### Ф.2 — divergence-aware if-expr type inference (BLOCKED)

**Что:** `if cond { throw e } else { compute() }` → result type
inference must use else-branch's type (then-branch diverges, его
implicit unit не должен dominировать).

**Status:** Initial attempt в session 2026-06-03 (`block_diverges`
check в `emit_if_expr`) показал **24 регрессии в Plan 108 binding rules
+ concurrency** при full nova test. Fix слишком aggressive — нужен
более targeted approach (only specific divergent patterns).

**Blocker для:** JSON `read_hex_quad` / `read_unicode_escape` functions
имеют exactly эту pattern. Без divergence-aware fix, `_nv_if_<n>`
declared как `nova_unit` но else-branch returns `nova_str` → CC-FAIL
type mismatch.

**Followup:** `[M-91.13-if-expr-divergence-aware-inference]` — careful
targeted fix:
- Detect only `if cond { throw|panic|exit } else { expr }` pattern
- Whitelist instead of blanket divergence check
- Per-pattern incremental test (Plan 108 binding rules + concurrency
  flaky-recover testing after each step)

### Ф.3 — full JSON roundtrip conformance suite (BLOCKED on Ф.2)

**Что:** `nova_tests/plan91_13/json_roundtrip_*.nv`:
- primitives (null/bool/num/str) — encode → decode = original
- arrays — `[1, "a", true]` parse + stringify
- objects — `{"k": v}` parse + stringify (requires Plan 48.1 ✓)
- nested — `{"arr": [1, {"k": "v"}]}` deep recursion
- edge cases — escapes (`\"`, `\n`, `\t`, `\\`), unicode (`\uXXXX`),
  whitespace, malformed input

**Followup:** `[M-91.13-json-roundtrip-full]` после Ф.2.
