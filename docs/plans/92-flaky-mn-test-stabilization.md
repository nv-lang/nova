# Plan 92 — стабилизация флаки M:N-тестов

> **Статус:** ✅ ЗАКРЫТ 2026-05-22 (Ф.0-Ф.3; ветка `plan-92`)
> **Приоритет:** P2 (CI-надёжность — флаки-тест роняет «зелёный»
> прогон и маскирует настоящие регрессии)
> **Оценка:** ~0.5–1 dev-day (Ф.0 диагностика может re-scope'нуть)
> **Зависимости:** Plan 44.x (M:N runtime — work-stealing scheduler,
> preemption) ✅
> **Источник:** наблюдение при закрытии Plan 89 —
> `concurrency/mn_runtime_actual_workload` упал 1× в полном `nova test`
> (16 параллельных job'ов), при этом 3/3 PASS при изолированном
> перезапуске.

## Зачем

`concurrency/mn_runtime_actual_workload` **периодически падает** в
полном прогоне `nova test` под нагрузкой (16 параллельных job'ов),
но стабильно проходит изолированно (наблюдение Plan 89: 1 FAIL в
полном прогоне → 3/3 PASS изолированно).

Флаки-тест — это **производственная проблема, а не косметика**:

- роняет «зелёный» CI на ровном месте → доверие к прогону падает;
- **маскирует настоящие регрессии**: при `1 FAIL` нельзя без ручного
  разбора отличить «флаки» от «реальный баг» — каждый красный прогон
  требует человека;
- эрозия дисциплины: «а, это просто флаки» → пропускают реальный FAIL.

Два возможных корня, и они **принципиально разные по тяжести**:

1. **Test-harness flakiness** — тест делает timing-предположение
   (sleep-based синхронизация, ожидание «к этому моменту N fiber'ов
   отработали»), которое нарушается под CPU-contention. Чинится
   детерминизацией теста.
2. **Настоящая гонка в M:N work-stealing runtime** — тогда это
   **soundness-дефект** рантайма, и «1 FAIL раз в N прогонов» — его
   видимая верхушка. Это эскалация, не косметика.

Plan 92 обязан **различить эти два случая** прежде чем чинить.

## Сравнение с Go / Rust / TS — обращение с флаки-тестами

| Экосистема | Практика |
|---|---|
| **Go** | `go test -race` для гонок; флаки → чинят либо `t.Skip` с issue; `-count=N` для воспроизведения. Молча падать — не норма. |
| **Rust** | флаки → `#[ignore]` + tracking issue; `cargo test` детерминированный по умолчанию; гонки — `cargo test` под `tsan`/loom. |
| **Industry** | флаки-тест либо **чинят root cause**, либо **quarantine'ят с tracking-issue** — но никогда не оставляют молча мигать в CI. |
| **Nova (цель)** | то же: root cause → fix; если runtime-гонка — escalation. Retry как **временная** мера, не замена фикса. |

## Scope

- **Диагностировать** `mn_runtime_actual_workload`: harness-flakiness
  vs реальная runtime-гонка.
- **Стабилизировать**: если harness — детерминировать тест; если
  runtime — завести escalation (fix или отдельный план Plan 44-lineage).
- **Аудит** прочих `concurrency/*` и `mn_runtime_*` тестов на
  аналогичную флакость (один симптом редко единичен).

## Итог Ф.0

Диагностика проведена 2026-05-22. Тест собран в exe, прогнан в трёх
режимах:

| Режим | Прогонов | PASS | FAIL | Симптом FAIL |
|---|---|---|---|---|
| sequential (без нагрузки) | 60 | 60 | 0 | — |
| parallel (24 копии × 8 раундов) | 192 | 192 | 0 | — |
| **под устойчивой CPU-нагрузкой** (24 burner-процесса) | 100 | 34 | **66** | `assert(on_w0 < 16)` ×66 |

**Вывод — Case A (test-harness flakiness).** Все 66 отказов —
`assert(on_w0 < 16)`; `assert(total_ran == 16)` **не падал ни разу**
(0 из 352 прогонов).

- `total_ran == 16` (все 16 fibers отработали на worker'ах) — **держится
  всегда**. M:N-рантайм корректно выполняет всю поставленную работу:
  потери работы нет, выполнения на main вместо worker'а нет. **Не баг
  рантайма, soundness-дыры нет.**
- `on_w0 < 16` (work-stealing раскидал fibers по >1 worker'у) — падает
  66% под устойчивой нагрузкой. Причина: 16 spawn-bodies микроскопичны
  (`= current_worker_id()`); под CPU-starvation worker-потоков ОС
  worker 0 законно успевает выполнить все 16 раньше, чем ОС вообще
  планирует потоки worker'ов 1-3. Это **валидное поведение
  work-stealing** (оппортунистичная кража), не поломка round-robin.
  Тест ошибочно трактовал эвристику планировщика как контракт.

**Decision point Ф.0.4 → подход A** (детерминизация теста). Корень в
самом тесте, не в рантайме — runtime-эскалация (Case B) не нужна.

Распределение **нельзя** сделать детерминированным «в лоб»: ОС владеет
планированием потоков; под starvation один worker законно делает всё.
Поэтому распределение проверяется **статистически** (bounded-sampling,
см. Ф.1).

**Прочие concurrency-тесты (Ф.2).** Source-review 75 тестов
`nova_tests/concurrency/`: единственный, проверявший распределение
work-stealing'а как hard-assert — `mn_runtime_actual_workload`.
`parallel_for.nv` проверяет порядок чередования (`log == 121212`) —
опирается на детерминированный cooperative-yield (`Time.sleep(0)` в
bootstrap-планировщике), сейчас **не флаки** (прошёл в полном прогоне
1058 тестов); латентный риск при переходе `parallel for` на
true-parallel M:N — отмечен, но вне scope Plan 92 (менять зелёный тест
нельзя). Остальные `assert`'ят конфиг-запросы (`worker_count`,
`maxprocs`) и вычисленные результаты — детерминированы.
