# Plan 92 — стабилизация флаки M:N-тестов

> **Статус:** 📋 proposed 2026-05-22, не начат
> **Приоритет:** P2 (CI-надёжность — флаки-тест роняет «зелёный»
> прогон и маскирует настоящие регрессии)
> **Оценка:** ~0.5–1 dev-day (Ф.0 диагностика может re-scope'нуть)
> **Зависимости:** Plan 44.x (M:N runtime — work-stealing scheduler,
> preemption) ✅
> **Источник:** наблюдение при закрытии Plan 89 —
> `concurrency/mn_runtime_actual_workload` упал 1× в полном `nova test`
> (16 параллельных job'ов), при этом 3/3 PASS при изолированном
> перезапуске.

## Зачем

`concurrency/mn_runtime_actual_workload` **периодически падает** в
полном прогоне `nova test` под нагрузкой (16 параллельных job'ов),
но стабильно проходит изолированно (наблюдение Plan 89: 1 FAIL в
полном прогоне → 3/3 PASS изолированно).

Флаки-тест — это **производственная проблема, а не косметика**:

- роняет «зелёный» CI на ровном месте → доверие к прогону падает;
- **маскирует настоящие регрессии**: при `1 FAIL` нельзя без ручного
  разбора отличить «флаки» от «реальный баг» — каждый красный прогон
  требует человека;
- эрозия дисциплины: «а, это просто флаки» → пропускают реальный FAIL.

Два возможных корня, и они **принципиально разные по тяжести**:

1. **Test-harness flakiness** — тест делает timing-предположение
   (sleep-based синхронизация, ожидание «к этому моменту N fiber'ов
   отработали»), которое нарушается под CPU-contention. Чинится
   детерминизацией теста.
2. **Настоящая гонка в M:N work-stealing runtime** — тогда это
   **soundness-дефект** рантайма, и «1 FAIL раз в N прогонов» — его
   видимая верхушка. Это эскалация, не косметика.

Plan 92 обязан **различить эти два случая** прежде чем чинить.

## Сравнение с Go / Rust / TS — обращение с флаки-тестами

| Экосистема | Практика |
|---|---|
| **Go** | `go test -race` для гонок; флаки → чинят либо `t.Skip` с issue; `-count=N` для воспроизведения. Молча падать — не норма. |
| **Rust** | флаки → `#[ignore]` + tracking issue; `cargo test` детерминированный по умолчанию; гонки — `cargo test` под `tsan`/loom. |
| **Industry** | флаки-тест либо **чинят root cause**, либо **quarantine'ят с tracking-issue** — но никогда не оставляют молча мигать в CI. |
| **Nova (цель)** | то же: root cause → fix; если runtime-гонка — escalation. Retry как **временная** мера, не замена фикса. |

## Scope

- **Диагностировать** `mn_runtime_actual_workload`: harness-flakiness
  vs реальная runtime-гонка.
- **Стабилизировать**: если harness — детерминировать тест; если
  runtime — завести escalation (fix или отдельный план Plan 44-lineage).
- **Аудит** прочих `concurrency/*` и `mn_runtime_*` тестов на
  аналогичную флакость (один симптом редко единичен).

## Декомпозиция (фазы и шаги)

### Ф.0 — Воспроизведение + диагностика (~0.3 д) — GATE

- **Ф.0.1** Воспроизвести: прогнать `mn_runtime_actual_workload` под
  искусственной CPU-нагрузкой (параллельно с `nova test` либо
  stress-нагрузкой на все ядра) N×; зафиксировать частоту FAIL.
- **Ф.0.2** Снять симптом каждого FAIL'а: assert-fail (какой именно
  invariant) / timeout / panic / CC-FAIL. Классифицировать.
- **Ф.0.3** Прочитать сам тест (`nova_tests/concurrency/
  mn_runtime_actual_workload.nv`) — есть ли timing-предположение
  (`Time.sleep`-синхронизация, ожидание «N задач завершились»),
  которое нарушается под contention.
- **Ф.0.4** **Decision point:** определить корень —
  - **(A) test-harness flakiness** — тест полагается на тайминг →
    Ф.1 детерминирует тест.
  - **(B) реальная гонка M:N-runtime** — invariant рантайма
    нарушается → Ф.1 = escalation (отдельный fix рантайма либо
    Plan 44-lineage), Plan 92 фиксирует диагноз и НЕ маскирует.
  Зафиксировать вывод + доказательство в «Итог Ф.0».

### Ф.1 — Стабилизация по итогам Ф.0 (~0.3 д)

- **Ф.1.1 (если A)** Детерминировать тест: убрать timing-assumption —
  заменить sleep-синхронизацию на явную (канал/`supervised`-scope
  exit/счётчик с барьером). Тест должен быть зелёным **независимо** от
  CPU-нагрузки.
- **Ф.1.2 (если B)** Зафиксировать runtime-гонку: минимальный
  воспроизводитель, escalation в Plan 44-lineage (или отдельный план).
  `mn_runtime_actual_workload` временно — `quarantine` с явной
  ссылкой на escalation-план (НЕ молчаливый skip).
- **Ф.1.3** Verify: N прогонов под нагрузкой подряд зелёные.

### Ф.2 — Аудит прочих concurrency-тестов (~0.2 д)

- **Ф.2.1** Прогнать весь `concurrency/` + `mn_runtime_*` под
  нагрузкой N× — выявить другие флаки.
- **Ф.2.2** Найденные — занести в «Итог Ф.0»-таблицу; чинить по тому
  же критерию A/B.
- **Ф.2.3** Оценить, нужен ли test-runner `--stress`/`--count`-режим
  (повторный прогон под нагрузкой) для отлова флаки в CI. Retry-режим
  (`TestAllOpts.retries`) уже существует — зафиксировать, что он
  **маскирует**, а не чинит: для known-флаки до фикса допустим как
  явная quarantine-мера с tracking, не как тихий дефолт.

### Ф.3 — Spec / docs (~0.1 д)

## Acceptance criteria

- [ ] Root cause `mn_runtime_actual_workload` установлен и доказан —
      harness-flakiness либо runtime-гонка (не «вероятно»).
- [ ] Тест больше не флачит: N прогонов подряд под CPU-нагрузкой —
      все зелёные.
- [ ] Если исход B (runtime-гонка) — заведён escalation-план/issue;
      тест в явном quarantine со ссылкой, не молча.
- [ ] Прочие `concurrency/` / `mn_runtime_*` тесты проверены на
      флакость; найденное — учтено.
- [ ] Политика флаки-тестов задокументирована в `test-conventions.md`.

## Non-scope

- **Полная ревизия M:N work-stealing scheduler** (Plan 44.x) — Plan 92
  диагностирует и стабилизирует, не переписывает рантайм. Если Ф.0
  вскроет runtime-гонку, её фикс — отдельная escalation-единица.
- **Введение нового test-framework / CI-инфраструктуры** — Plan 92
  работает в рамках существующего test-runner'а.
