# Plan 100.4 (umbrella): cleanup-on-failure — production-grade defer/errdefer rework

> **Создан 2026-05-23. Ред. 2 (2026-05-23) — production-grade rewrite
> через декомпозицию.** Исходный Plan 100.4 (D136 single) попытался
> интегрировать consume с существующим defer/errdefer (D90, Plan 20),
> но столкнулся с **5 серьёзными ограничениями D90**, делающими
> integration unergonomic / unsound в production:
>
> 1. **D90 §4** «defer body INFALLIBLE» — `tx.commit()` / `tx.rollback()`
>    производственных consume-типов почти всегда `Fail[E]`-fallable.
>    Чтобы вызвать в defer — обязательно 6-строчный handler-wrap.
> 2. **D90 §5** «no-suspend» — graceful socket close / DB connection
>    drain / async transaction commit требуют I/O. Production cleanup
>    не помещается.
> 3. **D90 §7** `interrupt` не триггерит errdefer — consume на handler-
>    flow-interrupt-path не покрывается.
> 4. **Panic в defer body** — D90 не специфицирует; implicit «cleanup
>    partially done», нет error composition.
> 5. **Multi-defer partial failure** — если defer N fail'ит, что с
>    defer N-1? Undefined. Cleanup может остановиться на полпути.
>
> Эти ограничения — **не только consume-проблема**. Они блокируют
> production-grade defer/errdefer для всех use-case'ов (graceful
> shutdown, distributed-cleanup, supervised-scope teardown). Plan
> 100.4 amend'ит D90 системно через **5 sub-sub-plan'ов**.
>
> **Статус:** 📋 roadmap, не начат. **GATED на [100.1](100.1-core-must-consume.md)**.
> **Оценка:** ~14 dev-day total (5 sub-sub-plan'а).
> **Зависимости:** [Plan 20 / D90](20-defer-implementation.md) ✅
> (defer/errdefer foundation), [Plan 49 / D85](49-cancel-throw-routing.md)
> ✅ (kinded throws, cancel-routing, multi-error composition), [Plan
> 100.1](100.1-core-must-consume.md) (consume foundation).

## Vision

Production-grade cleanup на каждом code-path'е — включая failable
cleanup, suspend-able cleanup (graceful drain), interrupt, supervised-
cancel, panic-in-cleanup. **Не хуже Rust async Drop / Kotlin coroutine
use{} / TS await-using; превосходит на 3 capabilities** (per-exit-path
distinction defer/errdefer/okdefer; error composition via Plan 49
вместо Rust double-panic-abort; type-effect on cleanup-fail видно из
сигнатуры).

После закрытия 5 sub-sub-plan'ов defer/errdefer становятся:
- Полноценным mechanism для production cleanup (включая network/IO).
- Foundation для consume-integration (Plan 100.4.5).
- General-purpose tool (применим везде, не только consume).

| Capability | Go | Rust | TS (ES2024) | Kotlin | Nova (после Plan 100.4) |
|---|---|---|---|---|---|
| Scope-exit cleanup на любом path'е | ✅ defer (fn-level only) | ✅ Drop | ✅ using | ✅ use{} | ✅ defer (**scope-level** — лучше Go) |
| Error-only cleanup distinction | ❌ | ❌ | ❌ | ❌ | ✅ **errdefer** (unique) |
| Success-only cleanup distinction | ❌ | ❌ | ❌ | ❌ | ✅ **okdefer** (Plan 100.4.3, unique) |
| Cleanup body может fail | ✅ (return err) | ⚠️ panic in Drop = abort | ⚠️ SuppressedError | ⚠️ throws | ✅ **Plan 49 composition** (100.4.1) |
| Cleanup body async/suspend | ⚠️ через manual sync | ⏳ Rust 2024+ work-in-progress | ✅ await using | ✅ coroutine use{} | ✅ **Plan 100.4.2** |
| LIFO error accumulation | ⚠️ partial | ⚠️ single-Drop | ✅ SuppressedError chain | ⚠️ partial | ✅ **Plan 49 multi-error** (100.4.4) |
| Panic-in-cleanup композиция | ✅ recover() | ❌ double-panic-abort | ⚠️ SuppressedError | ⚠️ try-catch | ✅ **Plan 49 composition** (100.4.4) |
| Interrupt / handler-flow integration | n/a | n/a | n/a | n/a | ✅ **Plan 100.4.5** |
| Supervised-cancel preservation | n/a | ⚠️ async Drop caveats | ⚠️ AbortController manual | ✅ structured | ✅ **Plan 49 + 100.4.5** |
| Exit-reason доступен в cleanup-body | ❌ | ❌ | ❌ | ❌ | ✅ **`defer \|result\|`** (Plan 100.4.3) |
| Compile-time enforcement consume-cleanup | n/a | ✅ Drop guarantee | n/a | n/a | ✅ **100.4.5** через check_consume |

**Свод:** Nova matches Rust по 6 capabilities; **превосходит** на 5
(scope-level vs Go fn-level; errdefer/okdefer distinction unique; Plan
49 composition vs Rust double-panic; effect-typed cleanup-fail видно
из сигнатуры; supervised-cancel native).

## Что меняется в D90 (Plan 20 ✅)

D90 остаётся в силе, но **5 пунктов amend'аются**:

| Пункт D90 | Текущее | После Plan 100.4 |
|---|---|---|
| §4 defer body INFALLIBLE | Fail effect запрещён | **Fail допускается**, ошибка композируется (Plan 49); см. 100.4.1 |
| §5 defer body NO-SUSPEND | sleep/Channel.recv/spawn запрещены | **Suspend допускается** с cancel-safe semantics; см. 100.4.2 |
| §7 errdefer triggers | throw + panic | + **`interrupt`** (для cleanup на handler-flow-path); см. 100.4.5 |
| (новое) Multi-defer LIFO | implicit partial-failure | **Explicit error accumulation** (LIFO с продолжением после fail); см. 100.4.4 |
| (новое) Panic-in-defer | implicit composition | **Plan 49 multi-error composition** explicit; см. 100.4.4 |

Плюс **2 новых construct'а** добавляются:

| Construct | Назначение | Sub-sub-plan |
|---|---|---|
| `okdefer { ... }` | runs только при normal exit (complement к errdefer) | 100.4.3 |
| `defer \|result\| { ... }` (опциональная форма) | exit-reason доступен в body для разных cleanup-actions | 100.4.3 |

## Acceptance (umbrella, across все 5 sub-sub-plan'ов)

После закрытия:

- [ ] **defer/errdefer body может Fail**: `defer { db.commit() }` где
      commit имеет `Fail[CommitErr]` — валиден. Cleanup-fail
      композируется с propagating error через Plan 49 multi-error.
- [ ] **defer/errdefer body может suspend**: `defer { socket.graceful_close() }`
      где close использует Net.* — валиден. Cancel-safe (на cancel
      cleanup completes-then-propagates).
- [ ] **`okdefer { ... }`** — выполняется только на normal/return exit
      (success); на throw/panic/interrupt не выполняется.
- [ ] **`defer |result| { ... }`** — body имеет доступ к exit-reason
      (`Ok(())` / `Err(e)` / `Interrupt(v)`); pattern-match на result
      внутри.
- [ ] **Multi-defer partial failure**: если defer N fail'ит — defer
      N-1, N-2, ... продолжают LIFO; ошибки собираются в composite.
- [ ] **Panic в defer body** — композируется с propagating через Plan
      49 (никаких double-panic-abort'ов).
- [ ] **`interrupt v`** триггерит **errdefer** (изменение D90 §7) —
      нужно для consume на handler-flow-interrupt-path.
- [ ] **check_consume** распознаёт defer/errdefer/okdefer cover на
      consume-vars (Plan 100.4.5).
- [ ] **Supervised cancel** preservation: spawned-fiber cleanup
      выполняется на cancel; consume-vars закрываются (errdefer
      срабатывает).
- [ ] **Cross-language matrix** (см. выше) — каждая «✅ Nova» строка
      подтверждена ≥1 фикстурой.
- [ ] **Полный `nova test`** → 0 регрессий.
- [ ] **Spec sweep**: D90 (amended), D131, D132, D133, D136, D137,
      D138, D139, D140 — cross-ref'ы согласованы.
- [ ] **Pilot stdlib**: `Transaction` mock с failable commit/rollback,
      `File` mock с graceful_close, `Connection` mock с drain — все
      работают end-to-end.

## Risks (cross-cutting)

1. **Backward-compat с Plan 20 (existing defer/errdefer code)**.
   Сейчас существующий код полагается на D90 §4 (тело infallible) —
   не пишут handler-wrap. Plan 100.4.1 расширяет capabilities;
   existing код продолжает работать (no Fail effect — не используется
   composition). **Митигация:** zero migration cost; new capabilities
   opt-in.

2. **`interrupt` triggers errdefer (изменение D90 §7)** — может
   изменить поведение existing handler-flow кода. **Митигация:**
   100.4.5 Ф.0 GATE — audit existing handler-flow-using fixtures;
   verify behavior change ожидаем.

3. **Cancel-safe suspend semantics** (100.4.2) — сложная часть.
   **Митигация:** reuse Plan 49 cancel-routing; не вводим новые
   primitives.

4. **`okdefer` — new keyword burden**. Языку добавляется ещё одно
   слово. **Митигация:** symmetry с errdefer; оба — короткий
   composable family.

5. **Plan 49 composition semantics** должны быть production-grade к
   моменту 100.4.4. **Митигация:** Plan 49 ✅ already; verify
   multi-error пользователю UX'ит правильно (диагностика).

6. **D-block coordination**: резервируем D136/D137/D138/D139/D140.
   **Митигация:** проверка перед каждым sub-sub-plan'ом start.
