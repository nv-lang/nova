# Plan 172.1.1 — Generic-mono / erased-stub lowering reconciliation (U.4.5 substrate)

**Статус:** 📋 proposed 2026-06-27 (выделено из 172.1 U.4.5 — решение владельца)
**Родитель:** [172.1 §9](172.1-unified-type-engine.md) U.4.5; маркер `[M-172.1-U4-typedir-substrate]`.
**Конвенции:** §0 (один лоуэринг `resolved_type_to_c`, `type_ref_to_c`/`erased_type_ref_c` ретайрятся),
§7 (де-риск: detect → full-.c-diff baseline), §1 (нет авто-fallback / `nova_int`).

## 0. Зачем отдельный суб-план

U.4.5 (удалить codegen-re-derive `infer_expr_c_type_legacy`) требует, чтобы `resolved_type_to_c`
(единый лоуэринг) был **byte-identical** с текущим codegen-выводом ДЛЯ ВСЕХ видов выражений
(приёмка U.4.5: «full .c diff vs clean = пусто»). §7-измерение (`NOVA_U45_GAP`, 2026-06-27) показало:
доминирующие невыфлипнутые виды — **Ident 34% + SelfAccess 28% + Call 12% + Binary 9% + Member 8% =
~90%**. Проба-flip SelfAccess (28%) дала **2327 дивергенций** — корень ОДИН: **generic-mono / erased-stub
рассинхрон лоуэринга**. Это keystone-блокер U.4.5 (62% gap = Ident+SelfAccess) — отдельный фокус-атом.

## 0.7. DETECT Call-GAP (§7.2, 2026-06-27) — scope U.1-выигрыша

Инструмент `NOVA_U45_GAP` расширен (Call: callee-shape + receiver C-type + `rc=` membership в
resolved_callees). Замер (modules/generics/plan153_1/strings/plan108):
- **rc=false (28523) — CHECKER-gap** (чекер НЕ записал callee → method_table его не имеет → U.1).
- rc=true (5916) — CODEGEN-gap (записал, но нет `fn_ret_by_span` → generic-mono method, span не в индексе).

Топ по receiver: **StringBuilder 13985 (ВСЕ rc=false)** — registry-only concrete-тип (грузится
`load_builtins`/ExternalRegistry, codegen-only; чекер его НЕ видит). Vec/контейнеры — generic
(rc=true codegen-gap ИЛИ resolved). free-fn 6056 (5885 rc=false).

**Вывод:** крупнейший Call-GAP (~14k StringBuilder + др. registry-only concrete) = ЧИСТО U.1
(дать чекеру registry-only сиги). Lifetime: `SigRegistry<'a>` (`'a`=module) — external .nv (include_str!
`'static`) парсятся в `'static` OnceLock → `'static: 'a` → merge в sig OK (без 'a-конфликта). Это
конкретный, scoped U.1-шаг (НЕ generic-mono codegen-gap, тот отдельно). Гейт: soundness (§10 — чекер
теперь резолвит StringBuilder; новые ошибки/мис-резолв ловит 0-new-FAIL + clean-DELTA).

## 0.10. ОПРЕДЕЛЯЮЩИЙ finding (2026-06-27): большой Call-GAP = CODEGEN-ГЕНЕРИРУЕМЫЕ вызовы

Гипотеза «receiver-inference глубина» (§0.9) — ОПРОВЕРГНУТА. Swap `infer_arg_ty`→`infer_expr_type`
(богатый inference) = **NO-OP** (Call-GAP rc=false 28523→28522). Верификация: `modules/*.nv`
содержат StringBuilder **1 раз** (grep), но GAP = **11542 StringBuilder-вызовов в modules**. ⇒ эти
вызовы **codegen-ГЕНЕРИРУЕМЫЕ** (auto-derive `Debug`/`Display` контейнеров + str-interpolation
десугар), НЕ в исходном AST. **Чекер их НИКОГДА не видит** → не может записать `resolved_callees` →
ни U.1-sigs, ни receiver-inference (оба checker-side) их НЕ закроют (red herring).

**ОПРЕДЕЛЯЮЩЕЕ следствие для удаления legacy (U.4.5):** GAP — НЕ проблема полноты чекерского канала
(U.1/U.3). `infer_expr_c_type_legacy` зовётся codegen'ом для ДВУХ классов: (а) source-exprs (покрыты
каналом — армы+merge) и (б) **codegen-генерируемые exprs** (Debug/Display auto-derive, str-interp
StringBuilder) — их чекер не видит. Удаление legacy требует, чтобы (б) эмитили типы **напрямую при
генерации** (codegen знает их при генерации типизированного кода), не через `infer_expr_c_type`. Это
**bounded codegen-рефактор ГЕНЕРАТОРОВ** (Debug/Display/interp), а НЕ checker-completeness и НЕ
receiver-inference.

**Пересмотренный roadmap U.4.5-удаления:** (1) source-exprs → канал ✅ (армы+Call-channel+merge, в
работе); (2) **codegen-генераторы (auto-derive Debug/Display + str-interp) → эмитят C-типы напрямую**,
не legacy-infer — bounded; (3) тогда legacy удаляем. Спина U.1/U.2 (единый реестр) — для (1)
source-резолва, НЕ для (2). U.4.5-удаление НЕ гейтнуто на полный U.1 — гейтнуто на (2) генераторы.

## 0.9. U.1 type+method merge ✅ ЗАЛЭНДЖЕН (2026-06-27) + СТРАТЕГ-finding

type+method merge РЕАЛИЗОВАН (external_registry::builtin_sig_modules 'static OnceLock парс
StringBuilder/WriteBuffer/ReadBuffer; SigRegistry::merge_module_fns; TypeCheckCtx::build merge
Item::Type в self.types; check_module merge sig). **§7.2 DETECT: blast-radius = 0** (PASS 94 FAIL 1,
baseline-DELTA=0 vs чистый бинарь — §10-hazard НЕ материализовался, merge ЗВУЧЕН). Чекер теперь
знает StringBuilder/buffers как ТИПЫ + их method-сиги (§3 «один реестр», целевая форма).

**СТРАТЕГ-finding (меняет roadmap удаления):** GAP StringBuilder 13985→13582 — **лишь ~400 (~3%)**.
97% StringBuilder-вызовов ОСТАЛИСЬ rc=false ДАЖЕ с типом+сигами в чекере. Корень: `check_instance_
overload` берёт receiver через `infer_arg_ty(obj)` — резолвит ТОЛЬКО let-bound простые var
(`mut sb = StringBuilder.new()`). 97% receiver'ов — **method-chains** (`sb.append(x).append(y)`),
**self-fields** (`@buf.append` в типе с StringBuilder-полем), сложные exprs → `infer_arg_ty` их НЕ
резолвит. ⇒ **большой Call-GAP блокирован ГЛУБИНОЙ receiver-type-inference, НЕ отсутствием sigs.**
U.1-sigs (этот merge) = ~3% + фундамент; **настоящий рычаг = полный receiver-inference** (chains/
fields/complex) в чекере — фундаментальная checker-способность, гораздо глубже. (Альтернатива: эти
97% не баги — legacy/codegen их резолвит через ExternalRegistry+type-info; они блокеры ТОЛЬКО для
DELETION U.4.5, не корректности.)

**Вывод для roadmap U.4.5-удаления:** не «дать чекеру sigs» (сделано, ~3%), а «полный receiver-type-
inference» (chains/fields) ИЛИ codegen-side канал для temp-receiver вызовов. Это пересматривает спину
(§0.6): корень удаления — receiver-inference depth, не единый-реестр-sigs. Merge — фундамент под это.

## 0.8. U.1 method-merge ПРОБА (2026-06-27) — недостаточна, type-merge = high-risk

Попытка: `external_registry::builtin_sig_modules()` ('static OnceLock парс string_builder/write_buffer/
read_buffer.nv) + `SigRegistry::merge_module_fns` (merge method-сигов, skip-known-type guard) +
wire в check_module ПОСЛЕ Bound/Cap, ПЕРЕД TypeCheckCtx. Собралось, sound (94/1), но **GAP
StringBuilder НЕ сдвинулся (13986 rc=false)** → merge методов в `method_table` НЕ помог.

**Корень глубже (локализован):** `check_instance_overload` (types/mod.rs:7700) берёт receiver через
`BoundCtx::infer_arg_ty(obj)` → для `sb.append()` нужен `sb: StringBuilder`, а `sb` инферится из
`mut sb = StringBuilder.new()` → `infer_expr_type(StringBuilder.new())` требует StringBuilder в
**`self.types`** (TypeCheckCtx), не только в `method_table`. ⇒ цепочка: **type-decl в self.types +
static-ctor резолв + receiver-inference**, не просто method-сиги.

**Почему НЕ session-end push:** type-merge StringBuilder в self.types = чекер НАЧНЁТ полностью
type-check'ить StringBuilder-usage (поля, арги, статика) — **blast-radius 13985 вызовов** → массовые
потенциальные новые ошибки при ЛЮБОМ checker↔codegen mismatch (§10-hazard в обратную сторону:
добавление знания вскрывает рассинхрон). Требует §7 **detect-mode** (прогон с type-merge в
non-fatal-режиме, замер новых ошибок, калибровка) ПЕРЕД enable. Это фокус-многосессионная де-риск-
работа, не одна правка. Откатил no-effect method-merge (чистый стейт).

**Вывод для следующей попытки U.1 (де-рискнуто):** (1) merge external TYPE-decls в `self.types`
(не только методы); (2) detect-mode: посчитать новые E_* на корпусе ПЕРЕД enable; (3) калибровать
(вероятно нужен per-type opt-in, начать с 1 типа); (4) только потом — Call-channel выигрыш. Метод-
merge инфраструктура (builtin_sig_modules + merge_module_fns) — переиспользуема, но одна не даёт эффекта.

## 0.6. КРИТПУТЬ к удалению legacy (определён 2026-06-27 — спина 172.1)

Per-arm аннотация (SelfAccess/Ident/Index/Member/Block/control-flow/Binary/Unary — все ✅ залэнджены,
baseline-DELTA=0) **упёрлась в убывающую отдачу**: GAP остаётся **Ident 47k** (имена fn/type/module —
нет value-типа, НЕ аннотируемы), **Call 28k**, **Binary 21k** (сложные операнды), **Member 18k**.
Полное удаление legacy НЕ закрывается аннотацией листьев. Реальная цепочка зависимостей (спина):

1. **U.1/U.2 — ЕДИНЫЙ реестр сигнатур (корень).** Сейчас чекер (`sig.method_table`, TypeRef-уровень)
   и codegen (`ExternalRegistry` с `return_c_type`/`c_name`/span, построен из `.nv`) — **ДВА
   параллельных реестра** (§3-анти-паттерн). Чекер НЕ резолвит primitive/builtin-method callees
   (`is_primitive_recv_name → return`, types/mod.rs:7708; «external overloads не в method_table») →
   28k Call не попадают в `resolved_callees`. **Фикс:** один реестр (с FnDecl.span) питает ОБА слоя.
2. **U.3.2/U.3.3 — резолв вызовов в чекере** (свободные fn + instance/static + перегрузки): снять
   gate 7708, чекер пишет `resolved_callees` для ВСЕХ Call (через единый реестр). Закрывает Call-GAP.
3. **U.4.5 — Call-канал полон + удаление legacy.** Когда `resolved_callees` покрывает все Call →
   `fn_ret_by_span` канал полон → legacy для Call не нужен. Non-value-Ident (47k) — отдельный
   вопрос (имя fn/type не несёт value-тип; codegen-потребление их `infer_expr_c_type` надо
   аудитить — возможно результат не используется). typedef-side-effects → §0-фаза-корректность
   (forward-decl рано/define поздно), не lazy-в-inference.
4. **U.6.1 — `type_ref_to_c` ретайрится** → единый `resolved_type_to_c` для binding И literal
   (снимает dual-lowering #23-класса by construction).

**Вывод:** fastest-whole = спина **U.1/U.2 (единый реестр) → U.3 → U.4.5 → U.6.1**, НЕ добивание
листьев. Корень — единый реестр (U.2.2 build включая external-сиги со spans + U.2.3 consume в
чекере). Это фокус-многосессионная работа с §10-осторожностью (не удалять registry-only снабжение
до звучности инлайн-пути). Independent-атомы (U.1.4 intrinsic-table, U.6.4, U.7.2/3, U.8) —
параллельно, не гейтнуты на спину.

## 0.5. Состояние 2026-06-27 (ВЕРИФИЦИРОВАНО по коду, не по докам)

### Разбор бага #23 (plan-169.2, пустой `[]fn`→SEGV) — это «ДВА ОКНА ПРАВДЫ», не просто nova_int

Владелец указал docs/plan-169.2-compiler-fixes.md (баги легаси). #23 разобран до корня (2026-06-27):
`mut arr []fn()->int = []` даёт в .c **РАССОГЛАСОВАНИЕ ЛОУЭРИНГА**:
- **binding** `arr` → `NovaArray_void_p*` (через `type_ref_to_c(decl.ty)` — declared-тип).
- **literal** `[]` → `Nova_Vec____nova_int* _nv_tmp = ..._static_new()` (через codegen element-derive
  пустого литерала → fallback `nova_int`, + другой container-KIND: Vec, не NovaArray).
⇒ `nova_array_push_void_p(arr, closure)` на физическом `Nova_Vec____nova_int` → layout-mismatch на
realloc → SEGV. Это **§0/D315 «два окна правды»**: `[]fn` лоуэрится ДВУМЯ путями (`type_ref_to_c`
binding vs literal-emit element-derive). **Point-fix невозможен заплаткой** (пробовал hint void_p
перед value-emit — container всё равно Vec____nova_int из другого emit-пути; KIND-mismatch Vec↔Array
не снимается хинтом element-типа). Подтверждает doc «point-fix продублирует U.4». **Настоящий фикс =
единый лоуэринг `resolved_type_to_c` для ОБОИХ (binding И literal)** — U.4.5 (literal через канал) +
U.6.1 (`type_ref_to_c` ретайрится). Т.е. #23 — частный случай keystone, не отдельный баг.

Док-статусы устарели — **код опережает**. Реальное состояние `infer_expr_c_type` (consumer):

**Армы УЖЕ флипнуты на канал** (возвращают channel, не legacy re-derive):
- **Call** — `resolved_callees[id]`→`fn_ret_by_span[span]` (U.4.5a, было сделано ранее; Explore-агент
  ошибочно счёл U.4.3 «многонедельным неначатым» — он СДЕЛАН для resolved-callee-вызовов).
- **leaf/number_exprs** (literals/arith/As/ctor/bool-ops) — byte-identical (U.4.1/4.2).
- **RecordLit / TupleLit** — срезы (byte-identical).
- **SelfAccess** ✅ `985035ff` (var_types-gated), **Ident** ✅ `5c73f22e` (var_types-gated, non-primitive),
  **Index/Member/Block/If/IfLet/Match/Path** ✅ (структурные+control-flow, аннотированы).

**КЛЮЧЕВОЙ ИНСАЙТ (исправляет прежнюю гипотезу):** gap #2 (bare-`@`→erased-схема→nova_int element,
CC-FAIL lru) был **ТОЛЬКО в SelfAccess-flip-каскаде** (flip типизировал `@` → downstream field/index
брали erased-схему). **Прямая аннотация Index/Member НЕ триггерит gap #2** — канал берёт element/field-тип
ПРЯМО из чекера (`infer_expr_type`), корректно. Доказано: Index+Member+control-flow = **0-new-FAIL/34 дир**
(baseline-DELTA=0). Значит структурные армы НЕ блокированы subst-reliability.

**ДВА оставшихся блокера удаления `infer_expr_c_type_legacy`:**
1. **Un-annotated surface** (GAP-замер, fallback-в-legacy по KIND): Call-без-`resolved_callees` (~30k),
   Binary (~23k), non-value Ident (~53k — имена fn/type/module, у них нет value-типа → `infer_expr_type`
   None), Member-без-field-type (~20k), TurboFish/Closure/ArrayLit/MapLit. Каждый нужно либо покрыть
   каналом, либо доказать, что legacy для него = надёжный codegen-state (как Ident/SelfAccess через
   var_types), не re-derive.
2. **Side-effect side-channel (typedef-emit + mono-registration в `legacy`).** `legacy` крутится ВСЕГДА
   (emit_c.rs:36135) РАДИ side-effects. Два теста 2026-06-27 + ПЕРЕОСМЫСЛЕНИЕ (владелец 2026-06-27:
   «в легаси были ошибки — не делай legacy эталоном»):
   - **NOLEG-тест** (`NOVA_U45_NOLEG`: пропуск legacy для resolved_types-аннотированных не-gated армов):
     FAIL-set ИДЕНТИЧЕН (121/1, **компилируется И тесты проходят**), .c-diff = 31 файл. **Изначально
     счёл это «side-effects влияют → нельзя удалять». ОШИБКА интерпретации:** раз channel-only вывод
     КОМПИЛИРУЕТСЯ и поведение корректно (121/1), то .c-различия — это НЕ дропнутые нужные typedef
     (иначе CC-FAIL), а ИЗБЫТОЧНЫЕ typedef, которые legacy эмитил, а channel-only корректно опускает
     (или benign reorder). ⇒ channel-only side-effects **ЗВУЧНЫ** (sound).
   - **Order-тест** (lazy-legacy после resolved_type_to_c): .c-diff = 28/28, но тоже компилируется+проходит
     → benign reorder, тоже sound.
   **ВЫВОД (исправленный): byte-identical — НЕВЕРНЫЙ гейт.** Он (а) сохранял бы баги легаси, (б) over-
   reject'ил ЗВУЧНЫЕ channel-выводы. Правильный гейт (= [feedback-nova-tests-not-correctness-gate],
   conventions §1): **SOUNDNESS — detect172 pos+neg + 0 CC-FAIL + тесты проходят**, НЕ byte-identical.
   Этот гейт ЛОВИТ настоящие регрессии (gap #2 lru дал CC-FAIL = unsound → отвергнут корректно) и
   ПРИНИМАЕТ benign-различия (NOLEG/order компилируются+проходят). ⇒ **удаление legacy ФИЗИБЕЛЬНО**
   под soundness-гейтом; «byte-identical для всех армов» — отброшено как ошибочная установка.

⇒ U.4.5-удаление = многосессионное: покрыть un-annotated surface + relocate side-effects. Армы-покрытие
(этот раунд) — необходимая подготовка, sound (baseline-DELTA=0), но НЕ достаточно для удаления.

## 1. Корень (локализован 2026-06-27)

**U.4.8-тождество `type_ref_to_c(T) ≡ resolved_type_to_c(from_type_ref(T))` НАРУШЕНО для generic-
контейнеров в erased-template-теле:**
- **`erased_type_ref_c` (emit_c.rs:~12089) + `receiver_c_type` (:11803)** — для generic с type-param
  args дают **BARE** `Nova_Deque*` (erased-stub имя без мэнглинга). Это то, что codegen реально эмитит
  в erased generic-method-теле (`emit_generic_fn_erased`, recv_c :12273 → var_types["nova_self"]).
- **`resolved_named_to_c` (:2227)** — мэнглит ВСЕ args, включая erased type-param → **MANGLED**
  `Nova_Deque____Nova_T_p*`.
- ⇒ два naming-scheme одного erased-контейнера. resolved_type_to_c(scope["@"]=Deque[T]) даёт mangled,
  а erased-тело объявило receiver как bare → flip SelfAccess/Ident рушится.

**Доп. внутренняя несогласованность legacy** (которую U.4.5 и устраняет): `erased_type_ref_c` даёт
bare для Box[T] (param), а `infer_expr_c_type` RecordLit-арм (:36426) даёт mangled для того же Box[T] —
т.е. сам legacy непоследователен; цель — ОДНА форма из `resolved_type_to_c`.

## 3. Подход (целевая форма, §0)

1. **Erased-context-aware `resolved_type_to_c`** (не прокси): ввести в CEmitter флаг/состояние
   «in erased generic body» (ставится в `emit_generic_fn_erased`), и в `resolved_named_to_c` для
   generic-контейнера с type-param args лоуэрить в **bare erased-stub** под этим контекстом, в
   **mangled mono** иначе. Это делает `resolved_type_to_c` способным **субсумировать**
   `erased_type_ref_c`/`receiver_c_type` (§0 — один лоуэринг; шаг к ретайру `type_ref_to_c`).
2. **Flip SelfAccess + Ident** (62% gap) на канал — теперь byte-identical для container-типов.
3. **Класс `nova_int`-fallback (Cat-B13)** — отдельно: это §1-ФИКСЫ (legacy выдал `nova_int`, канал
   корректен), принимаются как documented behavior-change (прецедент U.4.4b), НЕ требуют ==legacy;
   верификация = soundness-аргумент + 0-new-FAIL + detect172, а не байт-идентичность.

## 4. Приёмка

- `resolved_type_to_c` byte-identical с codegen-выводом для generic-контейнеров в erased И mono телах
  (detect `NOVA_U45_RLCHECK` = 0 для container-класса; full-.c-diff vs clean для не-§1-fix набора = пусто).
- SelfAccess + Ident зафлиплены на канал; §1-fix набор (nova_int→корректный) задокументирован, 0 нов. FAIL.
- Разблокирует U.4.5 (после Call/Binary/Member — отдельные срезы): удаление `infer_expr_c_type_legacy`.
