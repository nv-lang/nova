# Plan 172.1.1 — Generic-mono / erased-stub lowering reconciliation (U.4.5 substrate)

**Статус:** 📋 proposed 2026-06-27 (выделено из 172.1 U.4.5 — решение владельца)
**Родитель:** [172.1 §9](172.1-unified-type-engine.md) U.4.5; маркер `[M-172.1-U4-typedir-substrate]`.
**Конвенции:** §0 (один лоуэринг `resolved_type_to_c`, `type_ref_to_c`/`erased_type_ref_c` ретайрятся),
§7 (де-риск: detect → full-.c-diff baseline), §1 (нет авто-fallback / `nova_int`).

## 0. Зачем отдельный суб-план

U.4.5 (удалить codegen-re-derive `infer_expr_c_type_legacy`) требует, чтобы `resolved_type_to_c`
(единый лоуэринг) был **byte-identical** с текущим codegen-выводом ДЛЯ ВСЕХ видов выражений
(приёмка U.4.5: «full .c diff vs clean = пусто»). §7-измерение (`NOVA_U45_GAP`, 2026-06-27) показало:
доминирующие невыфлипнутые виды — **Ident 34% + SelfAccess 28% + Call 12% + Binary 9% + Member 8% =
~90%**. Проба-flip SelfAccess (28%) дала **2327 дивергенций** — корень ОДИН: **generic-mono / erased-stub
рассинхрон лоуэринга**. Это keystone-блокер U.4.5 (62% gap = Ident+SelfAccess) — отдельный фокус-атом.

## 0.5. Состояние 2026-06-27 (ВЕРИФИЦИРОВАНО по коду, не по докам)

### Разбор бага #23 (plan-169.2, пустой `[]fn`→SEGV) — это «ДВА ОКНА ПРАВДЫ», не просто nova_int

Владелец указал docs/plan-169.2-compiler-fixes.md (баги легаси). #23 разобран до корня (2026-06-27):
`mut arr []fn()->int = []` даёт в .c **РАССОГЛАСОВАНИЕ ЛОУЭРИНГА**:
- **binding** `arr` → `NovaArray_void_p*` (через `type_ref_to_c(decl.ty)` — declared-тип).
- **literal** `[]` → `Nova_Vec____nova_int* _nv_tmp = ..._static_new()` (через codegen element-derive
  пустого литерала → fallback `nova_int`, + другой container-KIND: Vec, не NovaArray).
⇒ `nova_array_push_void_p(arr, closure)` на физическом `Nova_Vec____nova_int` → layout-mismatch на
realloc → SEGV. Это **§0/D315 «два окна правды»**: `[]fn` лоуэрится ДВУМЯ путями (`type_ref_to_c`
binding vs literal-emit element-derive). **Point-fix невозможен заплаткой** (пробовал hint void_p
перед value-emit — container всё равно Vec____nova_int из другого emit-пути; KIND-mismatch Vec↔Array
не снимается хинтом element-типа). Подтверждает doc «point-fix продублирует U.4». **Настоящий фикс =
единый лоуэринг `resolved_type_to_c` для ОБОИХ (binding И literal)** — U.4.5 (literal через канал) +
U.6.1 (`type_ref_to_c` ретайрится). Т.е. #23 — частный случай keystone, не отдельный баг.

Док-статусы устарели — **код опережает**. Реальное состояние `infer_expr_c_type` (consumer):

**Армы УЖЕ флипнуты на канал** (возвращают channel, не legacy re-derive):
- **Call** — `resolved_callees[id]`→`fn_ret_by_span[span]` (U.4.5a, было сделано ранее; Explore-агент
  ошибочно счёл U.4.3 «многонедельным неначатым» — он СДЕЛАН для resolved-callee-вызовов).
- **leaf/number_exprs** (literals/arith/As/ctor/bool-ops) — byte-identical (U.4.1/4.2).
- **RecordLit / TupleLit** — срезы (byte-identical).
- **SelfAccess** ✅ `985035ff` (var_types-gated), **Ident** ✅ `5c73f22e` (var_types-gated, non-primitive),
  **Index/Member/Block/If/IfLet/Match/Path** ✅ (структурные+control-flow, аннотированы).

**КЛЮЧЕВОЙ ИНСАЙТ (исправляет прежнюю гипотезу):** gap #2 (bare-`@`→erased-схема→nova_int element,
CC-FAIL lru) был **ТОЛЬКО в SelfAccess-flip-каскаде** (flip типизировал `@` → downstream field/index
брали erased-схему). **Прямая аннотация Index/Member НЕ триггерит gap #2** — канал берёт element/field-тип
ПРЯМО из чекера (`infer_expr_type`), корректно. Доказано: Index+Member+control-flow = **0-new-FAIL/34 дир**
(baseline-DELTA=0). Значит структурные армы НЕ блокированы subst-reliability.

**ДВА оставшихся блокера удаления `infer_expr_c_type_legacy`:**
1. **Un-annotated surface** (GAP-замер, fallback-в-legacy по KIND): Call-без-`resolved_callees` (~30k),
   Binary (~23k), non-value Ident (~53k — имена fn/type/module, у них нет value-типа → `infer_expr_type`
   None), Member-без-field-type (~20k), TurboFish/Closure/ArrayLit/MapLit. Каждый нужно либо покрыть
   каналом, либо доказать, что legacy для него = надёжный codegen-state (как Ident/SelfAccess через
   var_types), не re-derive.
2. **Side-effect side-channel (typedef-emit + mono-registration в `legacy`).** `legacy` крутится ВСЕГДА
   (emit_c.rs:36135) РАДИ side-effects. Два теста 2026-06-27 + ПЕРЕОСМЫСЛЕНИЕ (владелец 2026-06-27:
   «в легаси были ошибки — не делай legacy эталоном»):
   - **NOLEG-тест** (`NOVA_U45_NOLEG`: пропуск legacy для resolved_types-аннотированных не-gated армов):
     FAIL-set ИДЕНТИЧЕН (121/1, **компилируется И тесты проходят**), .c-diff = 31 файл. **Изначально
     счёл это «side-effects влияют → нельзя удалять». ОШИБКА интерпретации:** раз channel-only вывод
     КОМПИЛИРУЕТСЯ и поведение корректно (121/1), то .c-различия — это НЕ дропнутые нужные typedef
     (иначе CC-FAIL), а ИЗБЫТОЧНЫЕ typedef, которые legacy эмитил, а channel-only корректно опускает
     (или benign reorder). ⇒ channel-only side-effects **ЗВУЧНЫ** (sound).
   - **Order-тест** (lazy-legacy после resolved_type_to_c): .c-diff = 28/28, но тоже компилируется+проходит
     → benign reorder, тоже sound.
   **ВЫВОД (исправленный): byte-identical — НЕВЕРНЫЙ гейт.** Он (а) сохранял бы баги легаси, (б) over-
   reject'ил ЗВУЧНЫЕ channel-выводы. Правильный гейт (= [feedback-nova-tests-not-correctness-gate],
   conventions §1): **SOUNDNESS — detect172 pos+neg + 0 CC-FAIL + тесты проходят**, НЕ byte-identical.
   Этот гейт ЛОВИТ настоящие регрессии (gap #2 lru дал CC-FAIL = unsound → отвергнут корректно) и
   ПРИНИМАЕТ benign-различия (NOLEG/order компилируются+проходят). ⇒ **удаление legacy ФИЗИБЕЛЬНО**
   под soundness-гейтом; «byte-identical для всех армов» — отброшено как ошибочная установка.

⇒ U.4.5-удаление = многосессионное: покрыть un-annotated surface + relocate side-effects. Армы-покрытие
(этот раунд) — необходимая подготовка, sound (baseline-DELTA=0), но НЕ достаточно для удаления.

## 1. Корень (локализован 2026-06-27)

**U.4.8-тождество `type_ref_to_c(T) ≡ resolved_type_to_c(from_type_ref(T))` НАРУШЕНО для generic-
контейнеров в erased-template-теле:**
- **`erased_type_ref_c` (emit_c.rs:~12089) + `receiver_c_type` (:11803)** — для generic с type-param
  args дают **BARE** `Nova_Deque*` (erased-stub имя без мэнглинга). Это то, что codegen реально эмитит
  в erased generic-method-теле (`emit_generic_fn_erased`, recv_c :12273 → var_types["nova_self"]).
- **`resolved_named_to_c` (:2227)** — мэнглит ВСЕ args, включая erased type-param → **MANGLED**
  `Nova_Deque____Nova_T_p*`.
- ⇒ два naming-scheme одного erased-контейнера. resolved_type_to_c(scope["@"]=Deque[T]) даёт mangled,
  а erased-тело объявило receiver как bare → flip SelfAccess/Ident рушится.

**Доп. внутренняя несогласованность legacy** (которую U.4.5 и устраняет): `erased_type_ref_c` даёт
bare для Box[T] (param), а `infer_expr_c_type` RecordLit-арм (:36426) даёт mangled для того же Box[T] —
т.е. сам legacy непоследователен; цель — ОДНА форма из `resolved_type_to_c`.

## 2. Проба (A) 2026-06-27 — частичный результат

Зеркало `erased_type_ref_c` в `resolved_named_to_c` через **прокси** `is_generic_stub_c(arg)` (erased
type-param лоуэрится в stub C-тип `Nova_T*`; concrete `nova_int` — нет; контекст берётся из
`current_type_subst`: erased→stub→bare, mono→concrete→mangled):
- ✅ **container-erasure класс почти ушёл** (`Nova_Deque____Nova_T_p*` vs `Nova_Deque*`: 233 → ~4).
- ✅ byte-identical для текущих consumer'ов (RecordLit/Tuple detect = 0 div).
- ✅ FAIL'ы SelfAccess-пробы 5 → 2.
- ⚠️ НЕ чисто: остаток `Nova_Lru*` ⟷ `Nova_Lru____…*` REVERSED (3+3) — **прокси `is_generic_stub_c`

- ⚠️ Доминирующий остаток SelfAccess (~2100) — ОТДЕЛЬНЫЙ класс `ir=<sized-int> legacy=nova_int`
  (legacy Cat-B13 nova_int-fallback :36729) = §1-ФИКСЫ, не container-substrate.

Проба откатана (интерим-прокси, §0 «не патчь дубль») — чистый фикс ниже.

**ВТОРАЯ переплетённая проблема (выявлена reversed-кейсами `Nova_Lru*` ⟷ `Nova_Lru____nova_str__nova_int*`):
доступность `current_type_subst` в точке потребления канала.** В некоторых MONO-телах на момент вызова
`resolved_type_to_c` (consume канала) `current_type_subst` НЕ populated → args лоуэрятся в stub
(`Nova_K_p`), тогда как legacy на том же сайте имеет concrete (`nova_str`) через иной путь. ⇒ даже
mangled-форма выходит неверной (stub-args вместо concrete). Значит substrate = НЕ только bare/mangled-
выбор, но и **гарантия populated mono-subst к моменту consume** (фаза-корректность §0: факт=тип-аргументы
готов до потребления). in_erased_body-флаг решает bare/mangled; subst-availability — отдельная под-задача.

**Проба (B) 2026-06-27 — `in_erased_body`-флаг в 2 эмиттерах НЕДОСТАТОЧЕН.** Добавил флаг
`CEmitter.in_erased_body` (save/restore вокруг body-emit в `emit_generic_method_erased`:12379 +
`emit_generic_fn_erased`:16536) + gate в `resolved_named_to_c` (`in_erased_body && is_generic_stub_c
→ bare`). Результат: 2163 дивергенции (≈прокси), **container-класс НЕ убран чисто** —
`Nova_Set____Nova_T_p*` vs `Nova_Set*` (12), `Nova_Stack…` (5) остались + 5 FAIL. Вывод: SelfAccess
для Set/Stack потребляется при `in_erased_body=FALSE` → **есть ДРУГИЕ erased-emit/consume контексты
помимо 2 эмиттеров** (или consume в иное время, чем body-emit этих функций). ⇒ substrate
**многоконтекстный**; чистый фикс требует ЛИБО (i) полного покрытия всех контекстов, где
`resolved_type_to_c` потребляется для erased-body-выражений (энумерация), ЛИБО (ii) более
фундаментального подхода — напр. канал несёт уже-erasure-разрешённую форму, либо `resolved_type_to_c`
выводит erasure из СОСТОЯНИЯ типа (а не из emit-контекста). Обе пробы (A прокси / B флаг-2-эмиттера)
откатаны (§0 «не интерим-патч»). Это и есть focused-work суб-плана: §7 — энумерировать consume-контексты
(NOVA_U45_RLCHECK по kind=self + трассировка in_erased_body) ПЕРЕД фиксом.

**УНИФИЦИРУЮЩИЙ КОРЕНЬ обеих проб (главный инсайт 2026-06-27):** и прокси (`is_generic_stub_c`), и
флаг (`in_erased_body`) падают по ОДНОЙ причине — **`current_type_subst` не синхронен с emit-контекстом**:
- erased-тело с CONCRETE-arg (`Nova_Deque____nova_int*` residual): subst populated там, где контекст
  erased → arg concrete → proxy не сработал, но legacy = bare;
- mono-тело без populated subst (`Nova_Lru` reversed): arg лоуэрится в stub → proxy сработал ложно,
  legacy = mangled-mono.
⇒ инвариант «stub-arg ⟺ erased-контекст», на который опирался прокси, **НАРУШЕН рассинхроном
subst↔контекст**. ⇒ **целевой фикс (§0, фаза-корректность):** привести `current_type_subst` к
инварианту «populated РОВНО в mono-эмиссии, пусто в erased» (фаза-корректность §0 — тип-аргументы
готовы/отсутствуют согласно контексту), ТОГДА `resolved_type_to_c` выводит bare/mangled из СОСТОЯНИЯ
типа (концерт→mangled, stub→bare) консистентно — без emit-context-флага и без энумерации. Это и есть
«resolved_type_to_c выводит erasure из состояния типа» (вариант ii). Под-задача 172.1.1: аудит всех
set/clear `current_type_subst` (emit_monomorphized_fn + erased-эмиттеры) → привести к инварианту →
proxy-лоуэринг становится корректным → flip SelfAccess/Ident.
