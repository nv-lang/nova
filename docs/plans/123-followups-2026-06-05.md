## 2. Sub-task details

### 2.1 V7.6 refactor — TypeDecl-driven ref-type classification

**Closes:** `[M-123.7.6-generic-ref-types]` AND внедряет value-record
поддержку из D228.

**Problem:** текущая `is_reference_type_ref` использует **хардкод
list** для именованных типов (`String`, `Map`, `Vec`, ...). Это:
- ✗ ловит только builtin'ы (user generic wrappers `Box<T>` / `Container[T]`
  пропускаются → пессимизация при правильном решении);
- ✗ не различает heap-record (`type X { ... }`) и value-record
  (`type X value { ... }`) — оба попадут в "конс. unknown → не реф."
  ветку, теряя точность;
- ✗ ломает spec D52/D215/D228 contract — реф./value определяется
  декларацией, не списком.

**Fix:** заменить hardcoded leaf-name lookup на TypeDecl registry
lookup. Spec D52 + D215 + D228 точно определяют классификацию:

| TypeRef | Kind / decl | Ref-type? | Spec |
|---|---|---|---|
| `int`, `bool`, `f64`, `char`, `u8`, `()` | primitive | **no** | D32 |
| `[]T` (Array) | heap handle | **yes** | D32 |
| `[N]T` (FixedArray) | inline | **no** | D52 |
| `str` | heap immutable handle | **yes** (immut → cache always safe) | D32 |
| `*T` (Pointer) | reference | **yes** | D216 |
| Tuple `(A,B,C)` | stack | **no** | D123 |
| Func / Protocol / Unit / Unsafe / Mut / Readonly | recurse inner / no | recurse | — |
| `type X(T1,T2)` (Tuple decl) | stack | **no** | D52 |
| `type X(n T)` (NamedTuple) | stack | **no** | D215 |
| `type X { ... }` (Record, AllocKind::Heap) | heap | **yes** | D52 |
| `type X value { ... }` (Record, AllocKind::Value) | stack | **no** | D228 |
| `type X \| A \| B` (Sum) | heap tagged union | **yes** | D52 |
| `type X T` (Newtype) | recurse inner | recurse | D52 |
| `type X alias T` (Alias) | recurse inner | recurse | D52 |
| Effect / Opaque | external / capability | **yes** (handle/ptr) | D126 |

**Approach:**
1. `is_reference_type_ref(t, type_registry)` — принимает TypeRef +
   ссылку на `HashMap<String, &TypeDecl>` built из всех Module items
   (peer_files included).
2. Для `TypeRef::Named { path, .. }`:
   - Если leaf — primitive (`int|bool|f64|f32|i8|i16|i32|i64|u8|u16|u32|u64|isize|usize|char|()|Never`) → false.
   - Если leaf — builtin reference (`str`, других нет — `[]T` через
     Array, `*T` через Pointer) → true.
   - Иначе lookup в registry:
     - `Record + AllocKind::Heap` → true
     - `Record + AllocKind::Value` → false (D228)
     - `Sum` → true
     - `NamedTuple` → false (D215 stack)
     - `Newtype(inner)` → recurse `is_reference_type_ref(inner, reg)`
     - `Alias(inner)` → recurse
     - `Effect | Opaque | Protocol` → true (handle/ptr)
   - Если type не найден (cross-module import без peer-file)
     → **conservative true** (assume ref so invalidation is sound; за-
     консервативно подтверждает cache invalidation = safe behavior).
3. Builder `build_registry` сразу строит `HashMap<String, TypeDecl>`
   side-by-side с FieldRegistry и держит owned `TypeDecl` clones —
   field_cache pass mutates FnDecls, не TypeDecls, поэтому клон cheap.

**Tests** (unit, `tests/v7_6_typedecl_refclass.rs`):
- value-record (D228): `type Pt value { x int }` field → not ref.
- heap-record: `type C { v int }` field → ref.
- named-tuple: `type Pt(x int, y int)` field → not ref.
- sum: `type R | A | B(int)` field → ref.
- newtype `type Id u64` → ref (primitive, no mut-method-modify-slot).
- newtype `type Box []u8` → ref (recurse → array).
- alias `type ListInt = []int` → ref.
- unknown cross-module type → ref (conservative).
- str → ref (immutable per spec D26).
- ptr → ref (primitive pointer).

**str/primitive immutability — hard spec contract:**

`str` classification (TRUE = cache survives) is correct because:
1. **Spec contract** (`08-runtime.md:658` D26): «str — immutable» normative
   statement; (`08-runtime.md:823-825` D73): O(n) copy at `str↔[]u8`
   justified by str immutability invariant; (Plan 91 §«Принцип: Nova-
   first» lines 196-199): «сознательный дизайн, как в Rust».
2. **Architecture**: StringBuilder (D179) + as_bytes() → `ro []u8`
   (D178) — built on str-immutable invariant.
3. **Stdlib evidence**: `runtime_registry.rs::str_runtime()` — 0 entries
   с `is_mut: true`; `std/runtime/string.nv` — 0 `fn str mut @`.
4. **Codegen safety**: even if user writes `fn str mut @hack(...)` (parser
   permissive — see followup below), `emit_c.rs::prepare_method_recv`
   passes primitive receivers **by value** (`&obj` only для `NovaValue_*`)
   — mutation silently no-ops. Cache survives all three scenarios.

**New followups spawned** (mut-receiver ABI gaps — out of Plan 123 scope,
discovered via 6-agent ultracode audit 2026-06-05):

- **`[M-D26-primitive-mut-method-diag]`** — add `E_PRIMITIVE_MUT_METHOD`
  diagnostic in `types/mod.rs::TypeCheckCtx::build` to reject user-
  written `fn str mut @...` / `fn int mut @...` / etc. per Plan 91
  §«Принцип: Nova-first» rationale. Currently parser accepts +
  codegen silently no-ops. Target = Plan 108/91 hardening.

- **`[M-D215-mut-receiver-pointer-codegen]`** — fix codegen for D215
  named-tuple mut-receiver. Spec (Plan 124.8 §2.7 lines 184): «Method
  receiver: pointer (как value-record)». Reality (`emit_c.rs:9710-9712`):
  `NovaTuple_X` emitted as VALUE (no pointer), `prepare_method_recv`
  skips `NovaTuple_*` prefix. Result: `fn Vec3 mut @rotate()` writes
  to a copy, caller's slot unchanged. Fix: extend `prepare_method_recv`
  to match `NovaTuple_*` when `recv.mutable=true`; switch
  `receiver_c_type` to emit `NovaTuple_X*` for mut receivers. Target =
  Plan 120 / 124.8 hardening.

- **`[M-codegen-recv-mutable-flag-unwired]`** — `recv.mutable` flag is
  NEVER read в emit_c.rs (0 grep hits для `recv.mutable` / `receiver.*
  mutable`). ABI driven entirely by type-shape, не mut-marker. This is
  root cause of `[M-D215-mut-receiver-pointer-codegen]` AND
  `[M-D26-primitive-mut-method-diag]`. Fix: thread `recv.mutable` into
  signature emission + call-site adapter, gating pointer-receiver ABI.

**Plan 123 V7.6 impact assessment:**

My V7.6 classification `is_reference_type_ref(value-record/tuple) = FALSE`
is **principally correct per spec D228 + §2.7**: mut-method on value-
type modifies the receiver's stack-slot bits через pointer ⇒ cache
must invalidate.

Coincidentally, this matches *current broken codegen behaviour* for
named-tuples (`NovaTuple_X` by-value copy = mutation lost = caller's
slot unchanged = «as if cache were valid»). When `[M-D215-mut-receiver-
pointer-codegen]` lands, the cache invalidation V7.6 already mandates
becomes the *required* behaviour. No V7.6 change needed.

For value-records (D228), V7.6 = FALSE matches V2 LANDED codegen.

For str/int/primitives, V7.6 = TRUE relies on by-value primitive
receiver passing (mutation no-op). If `[M-D26-primitive-mut-method-
diag]` lands (rejects mut-methods на primitives outright), this
becomes trivially correct (no method to consider).

### 2.2 V5.4 region-tag in explain

**Closes:** `[M-123.5.4-explain-region-tagging]`.

**Problem:** V5.4 deep-walk сейчас всё пихает в одно `mut_caches`
поле без distinction outer-region (V1.1, `_at_F_r<N>`) vs nested-

**Fix:** split `FnCacheInfo` field:

- `pub nested_region_caches: Vec<String>` — names from `_at_F_n<N>`

`mut_caches` оставляем backward-compat (= outer + nested merged) —
deprecate в V5.5.

**Tests:** обновить existing V5 reports test'ы + новый test для V5.4
nested counts.

### 2.3 Chain-norm fluent detection via FnDecl

**Closes:** `[M-123.4.4-user-fluent-detection]`.

**Problem:** `chain_norm.rs` имеет hardcoded `FLUENT_BUILTIN_METHODS`
list (~30 entries). User custom builders с `-> @` ret type не
рассматриваются.

**Fix:** pre-build `HashMap<(TypeName, MethodName), bool>` of fluent
methods, где bool = `fn.ret_ty == @`. Pre-pass через все Items —
match on Item::Fn где receiver=Some(self), parse ret-ty signature.
Lookup в normalize вместо list-match.

Hardcoded list — backup (для builtin'ов без FnDecl); union с
registry.

**Tests:** chain-norm для `type Bldr { @new() Bldr ... @push(a) @ ... }`
кастомного fluent builder.

### 2.4 Chain-norm non-self receivers

**Closes:** `[M-123.4.4-non-self-receivers]`.

**Problem:** chain-norm V1 detect-ит chains только когда root recv
содержит `@` (self-field access). `local.push(a).push(b)` не triggers.

**Fix:** расширить `try_extract_outer_fluent_chain` — accept root recv
если root = `Ident{...}` (local var) OR `Member{Ident, _}` (local
field). Skip `Call{...}` (already chain-flattened).

**Tests:** chain-norm для `let mut v = [1,2,3]; v.push(4).push(5).push(6)`.

### 2.5 V2 LICM weighted threshold

**Closes:** `[M-123.2.1-v2-licm-threshold-integration]`.

**Problem:** `licm_fn` uses raw read-count threshold (`reads >= 2`).
V2.1 ввёл weighted counter (read inside loop body counted ×N). Но
LICM threshold check сам по себе не uses weighted counter — он
смотрит на reads outside loop bodies (его roi).

Wait: LICM hoists из loops; нужно weight reads внутри loop body для
точного cost/benefit (N reads × N iters / hoist 1 vec preload).

**Fix:** в `LicmAnalysis::count`-pass передавать `cfg.loop_iters_estimate`
и весить reads-inside-iter (multiplier).

**Tests:** loop body с 1 read field accessed once → не hoist (1×8 = 8
benefit, hoist costs 1 var = noise). Loop body с 1 read field
accessed 5 times → hoist (5×8 = 40 benefit).

### 2.6 V2.1 dynamic loop count

**Closes:** `[M-123.2.1-dynamic-loop-count]`.

**Problem:** V2.1 использует env var `NOVA_FC_LOOP_ITERS` (constant
8) для multiplier. Для `for i in range(100) { ... }` это under-
estimate ×12.5; для `for i in range(3) { ... }` over-estimate ×2.7.

**Fix:** parse loop bound expression. Если для-цикла `for x in
range(N)` где N — integer literal → use N. Иначе fallback default
env. AST: detect `For { src: Expr::Call{ callee: Ident("range"),
args }, ... }` где args[0] = `Expr::IntLit(n)` OR args[0]+args[1]
для range(lo, hi).

**Tests:** для loop с range(3) → 3-iter weight; range(100) → 100;
range(x) → fallback.

### 2.7 V7.6 method-realloc flag

**Closes:** `[M-123.7.6-method-realloc-flag]`.

**Problem:** V7.6 текущий: ref-type field → @F.method() **никогда**
не инвалидиэт cache of @F (потому что ref slot stays). Но методы
типа `set_x(new)`, `replace(new)` — replace slot entirely. Cache
of @F стаёт stale.

**Fix:** TypeDecl analysis — для каждого method кодекса определить
«replaces self» flag: method body содержит `*self = other` /
`self = other.into()` / explicit self-assignment. Список таких
methods → `HashSet<(TypeName, MethodName)>` в FieldRegistry.

V7.6 lookup logic:
- IF method ∈ replaces_self → **invalidating** (как до V7.6).
- ELSE (push/extend/clear/...) → **non-invalidating** (V7.6 default).

Также сюда добавляются user-tagged `#realloc` attr methods (future).

**Tests:** type с `replace(other)` method → cache invalidated;
тот же type с `push(item)` method → cache survives.
