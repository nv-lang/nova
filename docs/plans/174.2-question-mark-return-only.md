# Plan 174.2 — `?` строго return-only (Rust-стиль) + чистка spec

> **Top-level план.** Создан 2026-06-20. **Статус:** 📋 PLANNED (компиляторная фича +
> amend D85; реализация — зона 172/владельца). **Маркер:** `[M-174.2-question-return-only]`.
> **Решение (2026-06-20):** `?` = **return-only** (только `Result`/`Option`, проброс
> значением, как Rust); throw-режим D165 убрать. Конверсия типа ошибки — **руками
> `.map_err`** (авто-`From` рассмотрен и ОТКЛОНЁН, см. Часть B).

## Часть A — `?` строго return-only (ДЕЛАЕМ)

`?` работает **только** в функциях, возвращающих `Result[T,E]` / `Option[T]`, и пробрасывает
**значением** (`return Err/None`). В Fail-эффект-функциях `?` **запрещён** — там ошибка
пробрасывается прямым `throw` или оператором `!!`. Один оператор — одна семантика.

- **Обоснование:** `never` — это тип операции эффекта `fail(value E) -> never`
  ([04-effects.md:922](../../spec/decisions/04-effects.md#L922)), а не функции (у неё свой
  `-> T`); смешивать value-propagation (`?`) с effect-throw (`fail`/`!!`) незачем. Это
  каноническая [D85](../../spec/decisions/04-effects.md#d85) («`?` не задействует Fail»).
- **Fallout минимальный:** в компилируемом корпусе `?` уже используется только на
  Result/Option-функциях (`nova_tests/effects/throws.nv`) — throw-режим D165 не задействован,
  «нечего откатывать». Мигрировать лишь: aspirational examples (`http.nv`, `oxsar_port.nv`)
  + doc-comment-примеры on_exit в spec/stdlib (`@commit()?` → `@commit()!!`).

### Изменения по слоям

1. **Checker** (`compiler-codegen/src/types/mod.rs`): `?` в функции на Fail-эффекте
   (return-тип не Result/Option) → `[E_TRY_IN_FAIL_FN]` с подсказкой «используй `!!` или `throw`».
2. **Codegen** (`compiler-codegen/src/codegen/emit_c.rs:21236-21340`): убрать ветку
   Fail-context throw для `?` (`in_fail_ctx`) — станет недостижимой (checker отверг раньше);
   оставить только return-Err/None.
3. **Спека:** amend [D85](../../spec/decisions/04-effects.md#d85) — `?` return-only, запрещён
   в Fail-fn. Починить stale `## D4` (04-effects.md:290) + дубль `####` (:950) (до-D85,
   противоречат канону). Поправить doc-comment on_exit-примеры (`@commit()?` → `!!`) в
   spec D188 + `std/prelude/{core,protocols,errors}.nv`.
4. **Тесты** `nova_tests/q_return_only/`: (a) `?` на Result → return Err; (b) `?` на Option →
   return None; (c) neg `?` в Fail-fn → `[E_TRY_IN_FAIL_FN]`.

## Часть B — авто-`From` конверсия ошибки: ОТКЛОНЕНО (2026-06-20)

Рассматривалось: `?` сам конвертит тип ошибки через `From` (как Rust:
`Err(e) => return Err(From::from(e))`). **Отклонено.** Причины:

1. **Противоречит принципу Nova** «не магия / single canonical path» ([D85](../../spec/decisions/04-effects.md#d85)) —
   авто-From это спец-правило (компилятор ищет `From`-impl за тебя).
2. **Боль меньше, чем в Rust.** У Rust нет эффектов → ошибки текут только значениями, конверсия
   на каждом слое задалбывает. У Nova Fail-эффект пробрасывает сам, ремаппинг типов — явно в
   `with Fail`-хендлере; авто-From помог бы только value-style `?` (ýже, не самый идиоматичный путь).
3. **Скрывает конверсию** в точке вызова (`parse(s)?` не показывает смену типа ошибки; может
   незаметно терять поля).
4. **Асимметрия обратимости:** добавить авто-From позже (если explicit задолбает) — легко и
   неломающе; убрать магию, когда на неё завязан код, — больно. Молодому языку — начать строго.

**Замена:** при `E ≠ E'` — явный `expr.map_err(|e| E'.from(e))?`. Видно на месте, не теряет данные молча.

> Если на практике boilerplate станет реальной болью — пересмотреть (маркер сохранён). Тогда
> учесть **ловушку identity-blanket**: `From[T] for T` есть для всех `T`, поэтому конверсия
> допустима только при `E ≠ E'` И наличии **не-identity** `From[E]` на `E'`.
