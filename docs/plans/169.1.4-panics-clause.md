# Plan 169.1.4 — `panics`-клаузула: консолидация runtime-panic тестов

> **Под-план [Plan 169.1](169.1-test-suite-profiling-and-speedup.md)** (ось «скорость»).
> **Создан:** 2026-06-19. **Статус:** 📋 proposed — самостоятельная задача.
> **Приоритет:** P2 — после консолидации (A)/(D); средний ROI, рантайм-риск.
> **Выигрыш:** −78 CU (114 runtime-panic тестов в 36 папках → folder-module).
> **Оценка:** база ~2–3 ч; реалистично ~4–5 ч; пессимистично — день (см. риск).

## Идея

`nv_panic` (`effects.h:566`) при наличии `_nova_test_frame` делает `longjmp` в
test-frame, **НЕ `abort()`**. Каждый `test {}` уже обёрнут в `setjmp`
(`emit_c.rs:17218-17253`). Значит panic внутри `test {}` УЖЕ ловится — сейчас
всегда трактуется как FAIL. Введём ожидание паники на уровне теста → 114
runtime-panic тестов можно держать в folder-module (паника одного не убивает
остальные, exit=0 при успехе).

Заодно устраняет рассогласование: сейчас runtime-panic раскиданы между
`expected_runtime/` (глобальный bucket) и `<plan>/neg/` (вперемешку с
compile-error). Единая конвенция: runtime-panic = `test "…" panics "…"`.

## Синтаксис

Контекстное KW `panics` (как `raw`/`bench`):
```nova
test "Channel.new(0) panics" panics "capacity must be >= 1" {
    ro _ = Channel.new(0)
}
```

Трактовка (инверсия PASS/FAIL для panic-expecting test):

| Тело | обычный test | `panics "X"` test |
|------|-------------|-------------------|
| panic, msg ⊇ X | FAIL | **PASS** |
| panic, msg ≠ X | FAIL | FAIL (wrong panic) |
| нет паники | PASS | **FAIL** (expected panic) |

## Фазы

- **Ф.1 parser + AST** (~30–45 мин): после `test "<name>"` опц. `panics "<str>"`;
  `TestDecl { …, panics: Option<String> }` (default None).
- **Ф.2 codegen** (~30–45 мин): в `emit_c.rs:17218-17253` при `t.panics.is_some()`
  инвертировать ветки setjmp + `strstr(msg, pattern)` внутри бинаря. exit=0 при
  успехе.
- **Ф.3 миграция** (~45–90 мин): скрипт `fn main`+`// EXPECT_RUNTIME_PANIC <pat>`
  → `test "<stem>" panics "<pat>" { <тело> }`, смена `module` на folder-group,
  удаление маркера. Пропускать `exit()`-тесты.
- **Ф.4 верификация** (~20–30 мин): прогон затронутых папок + регресс.

`test_runner.rs` — НЕ меняется (обычный folder-module, «N/M passed»).
`EXPECT_RUNTIME_PANIC` остаётся для legacy `fn main`-файлов и для `--panic`
селектора ([169.1.1](169.1.1-test-lane-flags-and-ci.md)).

## ГЛАВНЫЙ РИСК — GC/fail-frame consistency между тестами

Сейчас каждый panic-тест = свой процесс (`fn main`→abort), рантайм не переживает
панику. После консолидации несколько паник идут в ОДНОМ процессе подряд, а
`longjmp` пропускает деструкторы. Проверить корректность МЕЖДУ тестами:
- `_nova_fail_top` (стек fail-фреймов) — восстанавливается?
- `_nova_current_handler_iframe` — уже чинил его leak (STATUS_BAD_STACK,
  Plan 169.2 §16/runtime).
- GC-arena, scheduler-state.

Тот же класс, что race-investigations. Особо подозрительны concurrency-паники:
`supervised_cancel_double_bind`, `fiber_stack_overflow`, `select_all_closed`.
**Связь с [Plan 83.14](83.14-reentrant-mutex-mn-owner.md)** — supervised-panic-
propagation. Если рантайм рассыпается — нужен `nova_runtime_reset()` между
тестами (доп. инфра-задача → затея дороже −78 CU, отложить).

## Стратегия (ранняя проверка риска)

1. Ф.1+Ф.2 + мигрировать ОДНУ простую группу (`expected_runtime` contract/assert
   паники, без concurrency) → проверить, что 5–10 паник сосуществуют в бинаре.
2. Держится → катить группами, **concurrency-паники ПОСЛЕДНИМИ**.
3. Рассыпается на шаге 1 → нужен runtime-reset (отдельная задача), отложить.
