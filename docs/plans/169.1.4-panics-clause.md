# Plan 169.1.4 — `panics`-клаузула (консолидация runtime-panic тестов)

> **Под-план [Plan 169.1](169.1-test-suite-profiling-and-speedup.md)** (ось «скорость»).
> **Создан:** 2026-06-20 — извлечён из [169.1.2](169.1.2-consolidate-tests.md) Ф.2 по
> запросу (исходно был 169.1.4 до сворачивания в 169.1.2, теперь развёрнут обратно).
> **Статус:** 📋 PLANNED (отложен — компиляторная фича + рантайм-риск; гейт владелец/172).
> **Маркер:** `[M-169.1.4-panics-clause]`. **Выигрыш:** −78 CU (114 panic-тестов / 36 папок).

## Зачем

Каждый runtime-panic тест сейчас — отдельный файл `fn main` + `// EXPECT_RUNTIME_PANIC <pat>`
= **отдельный compile-unit** (~114 штук в 36 папках). `panics`-клаузула складывает их в
folder-module по теме: одна паника не убивает остальные (ловится test-frame'ом, exit=0)
→ **−78 CU**. Заодно устраняет рассогласование расположения panic-тестов (см. §«Конвенция»).

## Идея (разбор 2026-06-19)

`nv_panic` (`effects.h:555`) при наличии `_nova_test_frame` делает `longjmp` в test-frame,
НЕ `abort()`. Каждый `test {}` уже обёрнут в `setjmp` (`emit_c.rs:17218-17253`). Значит panic
внутри `test {}` УЖЕ ловится — сейчас всегда трактуется как FAIL. Вводим ожидание паники на
уровне теста → инверсия вердикта.

## Синтаксис (контекстное KW `panics`, как `raw`/`bench`)

```nova
test "Channel.new(0) panics" panics "capacity must be >= 1" {
    ro _ = Channel.new(0)
}
```

## Трактовка (инверсия PASS/FAIL)

| Тело | обычный `test` | `panics "X"` test |
|------|----------------|-------------------|
| panic, msg ⊇ X | FAIL | **PASS** |
| panic, msg ≠ X | FAIL | FAIL (wrong panic) |
| нет паники | PASS | **FAIL** (expected panic) |

## Изменения по слоям

- **`parser/mod.rs`:** после `test "<name>"` опц. `panics "<str>"`.
- **`ast/mod.rs`:** `TestDecl { …, panics: Option<String> }` (default `None`).
- **`emit_c.rs:17218-17253`:** при `t.panics.is_some()` инвертировать setjmp-ветки +
  `strstr(msg, pattern)` внутри бинаря; exit=0 при успехе.
- **`test_runner.rs`:** НИЧЕГО (обычный folder-module, «N/M passed»).
- **Миграция:** `fn main` + `// EXPECT_RUNTIME_PANIC <pat>` → `test "<stem>" panics "<pat>"`.

## Граница

Работает ТОЛЬКО для `EXPECT_RUNTIME_PANIC` (114). НЕ применимо: `EXPECT_COMPILE_ERROR`
(compile-fail) / `EXPECT_TIMEOUT` (зависание) / `EXPECT_EXIT` (process-exit) — остаются
`fn main`/`neg/`. `EXPECT_RUNTIME_PANIC` сохраняется для legacy `fn main` + селектора
`--panic` (169.1.1, постоянный — не переходный).

## ГЛАВНЫЙ РИСК — ре-вход рантайма после N пойманных паник в одном процессе

**Уточнено 2026-06-20 (формулировка исправлена).** НЕ «longjmp пропускает деструкторы»:
`nv_panic` идёт через `_nova_fail_top` ПЕРВЫМ → `defer` и `Consumable.@on_exit(Panic)`
корректно отрабатывают на unwind (в Nova нет RAII-деструкторов; `errdefer`/`okdefer`
ретракнуты D189). Реальный риск — **рантайм не рассчитан на чистый ре-вход после N
пойманных паник подряд в ОДНОМ процессе** (folder-module = один бинарь):

- **`_nova_fail_top` / handler-iframe** может остаться висеть на мёртвом стек-кадре между
  тестами → следующая паника `longjmp`-нет в мусор (класс бага, чинился Plan 169.2 §16).
- **GC / uncollectable** — Boehm-куча ок (консервативный сбор), но `uncollectable`-состояние
  (Mutex-owners, sync-примитивы) и глобальные реестры между тестами не сбрасываются.
- **fiber/scheduler** (concurrency-паники) — другой routing (паника остаётся на fiber-стеке,
  не пересекает mco-границу); полу-исполненные fiber'ы + слоты шедулера портят следующий тест.

**Стратегия:** катить группами — сначала ОДНА простая группа (contract/assert-паники без
concurrency); если 5-10 паник сосуществуют — расширять, concurrency-паники последними; если
рассыпается → нужен `nova_runtime_reset()` между тестами (отдельная инфра, тогда отложить).

> **Смежный баг (нашёлся при разборе 2026-06-20):** `[M-172-with-fail-swallows-panic]` —
> `with Fail[E]`-handler ошибочно ловит панику (D13 violation), т.к. re-dispatch
> (`emit_c.rs:6648-6675`) ре-throw'ит только `NOVA_THROW_CANCEL`, не `NOVA_THROW_PANIC`.
> Та же зона (panic ↔ fail-frame) — фикс панического routing'а логично делать вместе с этой фичей.

## Конвенция расположения (заодно)

Сейчас runtime-panic живёт в двух непоследовательных местах: `expected_runtime/` (глобальный
bucket, `fn main`) и `<plan>/neg/` (вперемешку с compile-error). `panics`-клаузула даёт единую
конвенцию: **runtime-panic = `test "…" panics "…"` в обычном folder-module** (не `fn main`,
не `neg/`). `neg/` остаётся строго под `EXPECT_COMPILE_ERROR`.

## Гейт / зависимости

- Компиляторная фича + рантайм-риск → согласование с владельцем; реализацию может вести
  агент 172 (вместе с фиксом панического routing'а `[M-172-with-fail-swallows-panic]`).
- НЕ блокирует остальную консолидацию [169.1.2](169.1.2-consolidate-tests.md) (та идёт без
  panics-клаузулы; достигнуто ~−102 CU без неё).
