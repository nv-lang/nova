   - `_worker_main` main + cleanup loops: `nova_free_uncollectable(ctx)`
     ПОСЛЕ `mco_destroy(co)`. State_store(DEAD) — BEFORE free (accesses
     ctx). mco_destroy — frees co's arena slot (separate alloc from
     ctx). free ctx — releases uncollectable memory.

2. **Orphan-scope pending_remote tracking under armed M:N.**

   До Plan 83.4.5.8: detach{} спавнил orphan через worker pool, но
   `runtime.drain_orphans()` waited только cooperative `q->fibers[]`
   (которая под armed empty). Result — drain returns immediately while
   orphan fiber still running на worker. Test fails (assert before
   orphan dec'нул counter).

   - `nova_runtime_orphan_scope()` API — returns pointer на global
     `_nova_orphan_scope` (lazy-inited).
   - `nova_runtime_orphan_scope_init()` — explicit init без spawn.
   - codegen `emit_detach`: под armed set `ctx->_nova_parent_scope =
     nova_runtime_orphan_scope()` + inc `pending_remote` BEFORE
     `spawn_orphan`. Worker fiber's epilogue dec + signal_main как
     emit_spawn.
   - `nova_runtime_drain_orphans` calls `nova_supervised_drain_main_scope`
     (без изменений) — оно уже ждёт `pending_remote == 0`.

3. **Flip activation** в `emit_main_wrapper`: uncomment

## 2. Симптом

Под armed M:N (uncomment `nova_runtime_auto_arm()` в `emit_c.rs::emit_main_wrapper`):

**Main thread:**
```c
ctx = nova_alloc(sizeof(NovaSpawnCtx));   // GC_malloc, zero-init
ctx->_nova_parent_scope = &q;             // WRITE
nova_runtime_spawn_into(&q, entry, ctx);  // push to worker deque
```

Diagnostic print: `MAIN ctx=0x...FC0 parent=0x...580 raw[0]=0x...580`

**Worker thread:**
```c
NovaSpawnCtx* _c = mco_get_user_data(_co);
// _c == 0x...FC0 (SAME ADDRESS)
// _c->_nova_parent_scope == NULL (!!)
```

Diagnostic print: `worker 0 picked user_data=0x...FC0 raw[0]=0x0`

**Entire `NovaSpawnCtxBase` struct reads as ZERO on worker side** —
not just one field. Same virtual address shows different memory content.

## 3. Симптом-цепочка

1. Worker reads `_c->_nova_parent_scope == NULL`.
2. Fiber entry function skip'ает preamble (нет alloc_slot).
3. Body тривиально выполняется (no preamble setup needed для simple body).
4. Fiber entry function skip'ает epilogue:
   ```c
   if (_c->_nova_parent_scope) {  // false!
       nova_scope_free_slot(...);
       nova_aint_fetch_sub_release(&_c->_nova_parent_scope->pending_remote);
       nova_runtime_signal_main();
   }
   ```
5. `pending_remote` НЕ декрементится → main thread hangs forever в
   `supervised_run_impl` wait-loop'е (`alive=0 remote=1` forever).
6. TIMEOUT (test_runner kills after ~70s).

## 4. Hypothesis

Под Windows fiber arena (`fiber_arena_win.c`):

```c
static void _nova_fw_gc_push_other_roots(void) {
    for (NovaFiberArenaWin* a = _nova_fw_arena_list; a; a = a->next) {
        ...
        for (size_t slot = 0; slot < a->high_water; slot++) {
            if (!((a->used_bits[slot >> 6] >> (slot & 63)) & 1)) continue;
            ...
            mco_coro* co = (mco_coro*)(slot_base + NOVA_FIBER_GUARD_SIZE);
            if (mco_status(co) == MCO_DEAD) continue;
            _nova_fw_gc_push_region(slot_base + NOVA_FIBER_GUARD_SIZE,
                                    slot_base + a->slot_size);
            ...
        }
    }
}
```

This callback pushes fiber stack regions as GC roots. `co->user_data`
pointer (which points to ctx in GC heap) IS within the pushed region.
**По идее GC должен trace через co→user_data→ctx → ctx marked live.**

Но empirically ctx memory zeroed на worker side. Возможные причины:

### 4.1. Hypothesis A: GC sweep race
GC runs between main's write и worker's read. ctx определяется как
unreachable (хотя должен быть reachable через co's region). GC sweep
zeroes block.

### 4.2. Hypothesis B: Arena push timing
arena's `_nova_fw_arena_list` build'ится при первом spawn. До этого
arena не в списке. Если GC fires до registration, ctx unreachable.

### 4.3. Hypothesis C: Windows TLB / cache coherency
Maybe Windows VirtualAlloc lazy-commit semantics interact badly с
Boehm's GC scan. Each thread might see different physical pages
mapped at same virtual address (very unlikely on x86, but possible
under some exotic VirtualAlloc scenarios).

### 4.4. Hypothesis D: nova_alloc thread-local cache
Boehm GC may have thread-local allocation buffers (TLABs). Maybe
ctx is allocated from main's TLAB but worker reads from worker's
TLAB cache. Unlikely — TLAB is just allocator optimization, не
влияет на memory visibility.

## 5. Что было пробовано (НЕ помогло)

В Plan 83.4.5.7 Ф.1 work:

1. ❌ Atomic fiber state machine (`_nova_fiber_state` field, CAS guards

2. ❌ `nova_scope_pin_ctx(scope, ctx)` call в `nova_runtime_spawn_into`
   ДО push'а — pins ctx в scope's ctx_pins array (GC-managed). Scope
   на main's stack, stack scanned by GC → ctx_pins reachable → ctx
   reachable. **Не помогло** — worker всё равно reads NULL.

3. ❌ `__atomic_thread_fence(SEQ_CST)` ДО `nova_deque_push` в spawn_global
   — defensive против Chase-Lev single-owner contract violation. **Не
   помогло** — visibility issue не связан с deque memory ordering.

4. ❌ `nova_runtime_shutdown()` call ДО `nova_evloop_close()` в main —
   fix'ит unrelated libuv assertion (workers still alive when evloop
   closes). Helps shutdown but doesn't fix visibility.

## 6. Рекомендуемые подходы

### 6.1. Approach A: `GC_malloc_uncollectable` для ctx (~0.5 dev-day)

Allocate ctx via `GC_malloc_uncollectable` (un-collectable allocation).
Memory никогда не reclaimed GC'ем до явного `GC_free`. После fiber
completion epilogue делает `GC_free(_c)`.

Pros:
- Простой fix; обходит GC race полностью.
- Production paradigm — task ctx в Go/tokio тоже uncollectable до
  task completion.

Cons:
- Требует explicit free → leaks if fiber doesn't reach epilogue
  (throw'ом catches up но edge case).
- Если ctx содержит pointers на user-data, GC still scans → но
  uncollectable не tracked в normal alloc count.

### 6.2. Approach B: Расширить `_nova_fw_gc_push_other_roots` на ctx_pins (~1 dev-day)

Per Plan 44.5 L5 design, `scope->ctx_pins[]` array хранит pinned
ctx pointers. GC scans scope (on main's stack) → finds ctx_pins
pointer → traces array → marks ctx live.

Но empirically не работает. Возможные исправления:
1. Verify scope is actually on main's stack (not in register).
2. Add explicit `GC_add_roots` для ctx_pins array.
3. Расширить gc_push_other_roots callback на explicit ctx_pins walk
   (для каждого worker's scope).

### 6.3. Approach C: Mutex-protected pending queue для cross-thread push (~1.5 dev-day)

Chase-Lev deque designed для single-owner push/pop + multi-thief
steal. Main's `nova_runtime_spawn_global` push'ит в worker's deque —
нарушение контракта.

Replace с mutex-protected pending list (как существующий wake_pending):
```c
typedef struct {
    nova_mutex_t spawn_mu;
    mco_coro** spawn_pending;
    int spawn_pending_count;
    int spawn_pending_cap;
} NovaWorker;
```

Main's spawn pushes под mutex. Worker drains spawn_pending в начале
каждой loop iteration (как wake_pending). Memory ordering guaranteed
mutex release-acquire pair.

Это может также покрыть visibility issue — mutex unlock-lock pair
flushes all main's writes → worker's reads.

Pros:
- Соответствует Chase-Lev contract.
- Memory ordering bullet-proof.

Cons:
- Performance regression (mutex contention на каждый spawn).
- Может скрывать GC issue без fix'а root cause.

### 6.4. Approach D: Diagnostic + root cause (~1-2 dev-day)

Перед implementation fix'а — verify hypothesis A/B/C/D через:

1. `GC_set_oom_fn` callback для tracking когда GC clears memory.
2. `GC_register_finalizer` на ctx — callback fires когда ctx
   finalized → log сообщает "ctx collected at T1, worker read at T2".
3. Explicit `GC_gcollect()` ПОСЛЕ main's write — verify ctx state
   AFTER collection (если zeroed → confirmed GC race).
4. Add per-thread `_nova_thread_id` to ctx → verify worker reads
   correct value AT MINIMUM for that field (rules out TLB/cache).
5. Strace / etl tracing на VirtualProtect / VirtualAlloc calls в
   arena — verify no unexpected re-protection / decommit.
