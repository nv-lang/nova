## 1. Контекст и мотивация

Plan 124 V1 (124.1-124.7) closed 2026-06-02 с tuple-priv enforcement
(D222 + D225). Post-merge research (`docs/research/07-unified-record-tuple-syntax.md`)
выявил два design issues:

### 1.1 Asymmetric parser surface

Records `{}` поддерживают 4 формы (single-line / multi-line с commas /
newline separator / trailing comma); named tuples `()` — только single-line.

Probe-fixtures (см. `nova_tests/syntax_probe/`):
- `record_multiline_ok` ✅ — все 4 формы.
- `named_tuple_singleline_ok` ✅ — single-line.
- `named_tuple_multiline_commas_neg` ✅ NEG — multi-line ❌.
- `named_tuple_multiline_no_commas_neg` ✅ NEG — без commas ❌.

### 1.2 Conceptual overlap «stack vs heap» через скобки

Nova — единственный mainstream-язык, где **форма скобок** кодирует
allocation. Эталоны:
- Go: ONE syntax + escape analysis.
- Rust: `()` vs `{}` — оба stack, отличаются access pattern.
- Kotlin: `value class` keyword.
- Java Valhalla: `value class` (incoming).
- C#: `class` vs `struct` keyword.

User decision (после discussion): унифицировать через **`value`
modifier** для records, **upgrade tuples** до first-class data type
(no priv, multi-line OK).

### 1.3 New semantic axes (per user)

- **Tuples** = pure data carriers, all-public, stack-only, no priv,
  no per-field mut (binding-level mutability).
- **Records** = encapsulation-capable, heap by default, optional
  `value` keyword делает stack.
- **`ro`/`mut` распространяются по default** на тип справа в binding;
  explicit duplication — `E_REDUNDANT_TYPE_MODIFIER`.

## 2. Design

### 2.1 Tuple syntax — multi-line + restrictions

```nova
// Single-line (current)
type Vec3(x f64, y f64, z f64)

// Multi-line с commas (NEW)
type Vec3(
    x f64,
    y f64,
    z f64
)

// Trailing comma (NEW)
type Vec3(x f64, y f64, z f64,)

// Multi-line с trailing comma (NEW)
type Vec3(
    x f64,
    y f64,
    z f64,
)
```

**Запрещено:**
- `priv` / `pub` modifiers на tuple field → `E_TUPLE_NO_PRIV`.
- `mut` / `ro` per-field → tuple fields всегда без per-field modifier
  (binding-level mutability).

**Разрешено:**
- Newlines между полями (после comma — for now keep comma required).
- Trailing comma после последнего поля.

**Mutability via binding (Rust-style, NEW):**
```nova
ro p = Vec3(x: 1.0, y: 2.0, z: 3.0)
// p.x = 5.0    // ❌ E_READONLY_FIELD — ro binding

mut p = Vec3(x: 1.0, y: 2.0, z: 3.0)
p.x = 5.0       // ✅ — mut binding allows ALL fields mut
```

### 2.2 Value-record — unified syntax

```nova
// Heap record (current default, unchanged)
type Account {
    ro id u64
    mut balance f64
}

// Value record (stack-allocated, NEW)
type Vec3 value {
    mut x f64
    mut y f64
    mut z f64
}

// Composable с priv (Plan 124 carries over)
type Secret value priv {
    pub ro id u64
    key str
}

// Composable с consume — auto-becomes consume (per user)
type Token value {
    consume raw [32]u8     // consume field → Token implicitly consume
}
```

### 2.3 Modifier ordering (canonical)

`type X [value] [consume] [priv] { ... }` — **allocation → ownership → visibility** (outer to inner).

Parser **order-independent** (accept any order). Lint `W_NON_CANONICAL_TYPE_MODIFIER_ORDER` — V2.

### 2.4 Binding mutability propagation rules (NEW, D33/D176 amend)

`ro`/`mut` на binding **по default распространяется** на тип справа.
Explicit duplication = redundant error.

| Декларация | Парсится | Семантика |
|---|---|---|
| `ro x T` | ✅ default | binding ro, type implicit ro |
| `mut x T` | ✅ default | binding mut, type implicit mut |
| `ro x ro T` | ❌ `E_REDUNDANT_TYPE_MODIFIER` | то же что `ro x T` |
| `mut x mut T` | ❌ `E_REDUNDANT_TYPE_MODIFIER` | то же что `mut x T` |
| `ro x mut T` | ✅ NEW | binding ro, content mut (cannot reassign, can mutate) |
| `mut x ro T` | ✅ existing | binding mut, content ro (can reassign, cannot mutate) |

→ Закрывает **partial implementation gap** (D176 §«type-modifier в любой
позиции» декларирует все 4, парсер реально принимает только 2).

### 2.5 D175 amend — binding dominates

| Field declaration | `ro acc` binding | `mut acc` binding |
|---|---|---|
| `field T` | переприсвоить ❌, мутировать ❌ | ✅ / ✅ |
| `ro field T` | ❌ / ❌ | ❌ / ❌ (always frozen) |
| `field ro T` | ❌ / ❌ | ✅ / ❌ |
| `mut field T` | ❌ / ❌ (**binding dominates**) | ✅ / ✅ |
| `mut field ro T` | ❌ / ❌ (**binding dominates**) | ✅ / ❌ |

**Changed:** `mut field` теперь НЕ "always-mutable" — `ro acc` блокирует
переприсвоение и мутацию (Rust-style).

### 2.6 Value-record semantics

- **Allocation:** stack (inline struct в callee frame).
- **Pass semantics:** copy on parameter pass (D32 amend).
- **Method receiver `@`:** pointer на stack-slot (mutates оригинал).
- **`[]Vec3` storage:** inline (Vec3 instances лежат рядом в массиве).
- **Reference fields:** хранятся как handles inline (ptr+len+cap для `[]T`,
  ptr+len для `str`); data на heap (GC-tracked).
- **Fixed array fields:** fully inline (`[32]u8` = 32 bytes inline).
- **Auto-equality:** через `#impl(Equatable)` — **Plan 126**, не 124.8.

### 2.7 Tuple semantics (existing + clarification)

- **Allocation:** stack (как Plan 120).
- **Pass semantics:** copy on parameter pass.
- **Method receiver:** pointer (как value-record).
- **Mutability:** binding-level (`mut p` → все поля mut; `ro p` → frozen).
- **All-public:** `priv` / `pub` запрещены.
- **No per-field modifier:** consistent с Rust.
- **Methods supported:** `fn Vec3 mut @rotate()` валиден.

### 2.8 Composability matrix

| Modifier | Tuple `()` | Record `{}` | Value record `value {}` |
|---|---|---|---|
| `priv` (Plan 124) | ❌ `E_TUPLE_NO_PRIV` | ✅ | ✅ |
| `consume` (Plan 100) | ❌ (tuple is value data, no ownership) | ✅ | ✅ |
| `mut`/`ro` per-field | ❌ | ✅ | ✅ |
| `#test_access` (Plan 124.6) | n/a (no priv) | ✅ | ✅ |
| `#visible_to` (Plan 124.6) | n/a (no priv) | ✅ | ✅ |
| Methods `fn T @method()` | ✅ | ✅ | ✅ |
| Generic `T[A, B]` | ✅ | ✅ | ✅ |
| Protocol `#impl(P)` | ✅ | ✅ | ✅ |

### 2.9 Plan 124 retractions

- **D222 amend:** «Named tuple priv + protocol impl boundary» —
  retract «Named tuple priv» part; keep «protocol impl boundary»
  semantics (applies к records). Documented как design refinement.
- **D225 retract:** «Type-level priv flip для named tuples» —
  fully retracted. Records keep type-level priv flip (D220 §3.3.1).

### 2.10 Followups (out of scope 124.8)

- **Plan 126** — Cloneable + auto-derive Equatable/Hashable/Comparable/Printable.
- **`#zero_on_move`** attribute — opt-in для security-critical consume.
- **`&` heap-promote** — coordinate с Plan 118 (typed pointers).
- **stdlib migration** — `()` → `value {}` где encap нужен (TBD on demand).
- **Display → Printable cleanup** в спеке (dead refs).

## 4. Acceptance criteria (A8.1-A8.20)

### 4.1 Tuple syntax

- **A8.1** ✅ Multi-line tuple `type X(\n a int,\n b int\n)` парсится.
- **A8.2** ✅ Trailing comma `type X(a int, b int,)` парсится.
- **A8.3** ✅ Multi-line + trailing comma `type X(\n a int,\n b int,\n)` парсится.
- **A8.4** ✅ `type X(priv f int)` → `E_TUPLE_NO_PRIV`.
- **A8.5** ✅ `type X(pub f int)` → `E_TUPLE_NO_PRIV` (pub запрещён, т.к. редкий случай — всё уже public).
- **A8.6** ✅ `type X(mut f int)` → `E_TUPLE_NO_PER_FIELD_MOD` (per-field mut запрещён).

### 4.2 Tuple binding mutability

- **A8.7** ✅ `mut p = Vec3(...)` + `p.x = 5.0` работает.
- **A8.8** ✅ `ro p = Vec3(...)` + `p.x = 5.0` → `E_READONLY_FIELD`.

### 4.3 Value record

- **A8.9** ✅ `type Vec3 value { mut x f64 }` парсится.
- **A8.10** ✅ Value-record codegen = inline struct (не pointer).
- **A8.11** ✅ `fn Vec3 mut @rotate()` мутирует оригинал (через `@` pointer).
- **A8.12** ✅ `mut arr []Vec3; arr[0].rotate()` мутирует элемент.
- **A8.13** ✅ `mut v Vec3` параметр = local copy (мутации не видны caller'у).
- **A8.14** ✅ Value + consume composition: `type Token value { consume raw [32]u8 }` works.
- **A8.15** ✅ `value priv` composition works (Plan 124.7 D225 для records preserved).

### 4.4 Binding rules

- **A8.16** ✅ `ro x ro T` → `E_REDUNDANT_TYPE_MODIFIER`.
- **A8.17** ✅ `mut x mut T` → `E_REDUNDANT_TYPE_MODIFIER`.
- **A8.18** ✅ `ro x mut T` парсится + позволяет mutate content.

### 4.5 D175 binding dominates

- **A8.19** ✅ `ro acc` блокирует `acc.mut_field = X` (даже если field is `mut`).

### 4.6 Regression

- **A8.20** ✅ plan124_1/2/3/5/6 unchanged (61 PASS → 61 PASS).
