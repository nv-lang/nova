## 1. Контекст

Per Plan 83 audit §4 + Plan 83.4.5.10:

Boehm conservative GC — **production-blocker для:**
- **Dynamic stack growth** (Go-style 2KB → grow). Conservative GC не может move pointers
  → stack copy невозможен. Nova вынужден держать 8MB virtual reserve.
- **Concurrent GC** (sub-1ms STW). Boehm STW phase scales O(heap).
  Plan 32 measurement: <16ms на tested workloads, но без concurrent
  mark — будущие большие heaps будут проблемой.
- **Per-thread fast-path** (без Boehm `THREAD_LOCAL_ALLOC` — Plan 83.5).
  Даже с Plan 83.5 — fallback на global heap при pool miss.

**Industry alternative:**
- **Go**: precise tri-color concurrent, write barriers, stack maps emit'нутые
  compiler-ом. STW <1ms.
- **JVM (ZGC, Shenandoah)**: concurrent compacting, sub-ms на TB heaps.
- **MMTk** (Memory Management Toolkit) — pluggable GC framework на Rust;
  used by Ruby JIT, V8 spike, OpenJDK research. Bindings для C-runtime
  возможны.

**Nova v1.0 production-grade требует replacement Boehm.**

## 3. Research areas

### 3.1 Pluggable GC frameworks

- **MMTk** — Rust framework, multiple GC plans (NoGC, MarkSweep,
  SemiSpace, GenCopy, Generational Immix). C binding existing
  (mmtk-core/mmtk-capi). Active development by Mozilla / academia.
  - Pro: production-quality, well-tested.
  - Con: requires precise stack maps от Nova codegen — major refactor.

- **Shenandoah** — JVM GC, port to C unlikely (deeply integrated с JVM).

- **Lazy Lisp/SBCL-style** — copying GC, precise. Reference implementations
  в Lisp world.

### 3.2 Custom precise GC

Nova-specific design:
- Codegen-emit'нутые stack maps (per function, per safepoint).
- Generational young/old.
- Write barriers через `__attribute__((nonnull))` либо TLS-flag check
  (Go style).
- Concurrent mark via incremental marking.

Pro: full control, optimized для Nova semantics.
Con: years of engineering. Not feasible до v2.0.

### 3.3 Hybrid: Boehm + Nova-managed арены

Альтернатива replacement — **сосуществование**:
- Boehm для general allocations.
- Nova-managed precise arena для hot-path (SpawnCtx, mco_coro, fiber
  stacks).

Arena уже частично implemented (Plan 82 fiber arena). Расширить к
SpawnCtx pool (Plan 83.6 это первый шаг).

Pro: incremental migration, Boehm не throw'нут.
Con: cross-arena pointers — careful design.

### 3.4 Codegen prerequisites

Independent of выбранного GC:
- **Precise stack maps**: Nova codegen уже emit'ит ABI metadata.
  Extension к включить per-frame pointer maps — significant codegen work
  (~3-6 months).
- **Write barriers**: function-local store-barrier insertion. Plan 27
  возможный отправной point.
- **Safepoints**: уже есть Plan 44.7 sysmon preemption. Generalize к
  GC safepoint.

### 8.3 Deliverable structure

Создать **`docs/research/precise-gc-decision-2026.md`** (главный document)
со следующими sections:

#### Section 1: Executive summary (~300 words)
- Текущее состояние Nova GC (Boehm conservative, multi-MB heap, <16ms STW).
- 3 production blockers (dynamic stack growth, concurrent GC, per-thread fast path).
- 1-line recommendation (replace / hybrid / defer).
- Timeline cost estimate.

#### Section 2: Status quo problems (~500 words)
- Boehm conservative scan limitations (cannot move pointers, no stack copy).
- Global allocation lock (per `nova_alloc_uncollectable` — Plan 83.4.5.10 §2).
- STW pause O(heap) — Plan 32 measurement.
- Plan 83.6 pool — first workaround, но limited к SpawnCtx.

#### Section 3: Industry survey (~1000 words, ≥4 GC frameworks)
- **Go**: precise tri-color concurrent, write barriers, contiguous-grow stacks
  via copystack + pointer rewrite. STW <1ms. Stack maps по safepoint.
- **JVM ZGC**: load-barriers, colored pointers, sub-ms на TB heaps.
- **JVM Shenandoah**: brooks-style forwarding pointers, concurrent compacting.
- **MMTk**: pluggable framework, Rust core, C-API binding. Used by Ruby JIT,
  Julia, JikesRVM. NoGC/MarkSweep/SemiSpace/GenCopy/Immix plans.
- **V8 Orinoco**: concurrent marking + parallel scavenge, write barriers.
- **Lisp/SBCL**: copying generational, precise via tagged pointers.

Per framework: licensing, integration cost, GC plans supported, моving
support, write barrier requirements, used-by, maintenance status.

#### Section 4: Nova-specific constraints (~700 words)
- **Effect system** — does GC interaction break effects? Probably not, но verify.
- **Structured concurrency** — fiber stacks (8MB virtual) need GC roots
  registration (Plan 82 GC_push_other_roots). Moving GC would need
  precise fiber stack scan.
- **Contracts proof** (Plan 33) — pure functions, ghost erasure: how GC
  affects verification? Minimal.
- **minicoro coroutines** — saved register state (CPU context) holds GC
  pointers. Precise marking must include suspended coroutine roots.
- **Cross-language FFI** (libuv, vcpkg deps) — opaque pointers held by
  external libraries. Precise GC needs pin/unpin protocol для FFI args.

#### Section 5: Migration paths (~800 words, 3 options)
- **Option A: Replace Boehm с MMTk.** Steps: precise stack maps codegen
  (3-6 months), write barriers (1-2 months), MMTk binding (1-2 months),
  testing (1-2 months). Total ~6-12 months. Risk: rewrite много alloc
  code paths, breaking changes.
- **Option B: Hybrid Boehm + Nova-managed arenas.** Plan 83.6 spawn pool
  precursor. Extend к: mco_coro pool (Plan 83.4.5.10 V3), effect handler
  storage, channel buffers, scope state. Boehm только для user-code
  allocations. ~3-6 months. Lower risk.
- **Option C: Defer.** Continue с Boehm + TL_ALLOC + pool + arenas.
  Pin v1.0 GC pause budget acceptance (<50ms). Re-evaluate after v1.0
  workload data.

For each: pros, cons, blocking dependencies, exit criteria.

#### Section 6: Codegen prerequisites cost (~500 words)
- **Precise stack maps**: per safepoint, emit pointer offsets within
  current frame. Nova safepoints (Plan 44.7): function prologue + loop
  backedge. Add: alloc site, call site. ~3-6 months.
- **Write barriers**: store-barrier on `field = ptr` для generational GC.
  Codegen insertion ~1-2 months.
- **Per-fiber stack maps**: fiber suspension points (mco_yield).
  Annotate saved register state с pointer mask. ~1-2 months.

#### Section 7: Recommendation + timeline (~400 words)
- Concrete recommendation (1 of 3 options).
- Why (rationale).
- Phases (post-v1.0):
  - Phase 1 (Q1): write barriers (preparatory).
  - Phase 2 (Q2-Q3): precise stack maps emit.
  - Phase 3 (Q4): GC framework integration либо custom precise GC core.

#### Section 8: References

Все cited URLs + papers + Nova internal plans.
