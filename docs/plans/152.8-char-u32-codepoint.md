# Plan 152.8 — `char` и code-point буферы: переход на 32-бит (оптимизация памяти)

> **Родитель:** [Plan 152 (umbrella)](152-string-coordinate-model.md).
> **Создан:** 2026-06-16. **Статус:** ✅ ЗАКРЫТ 2026-06-16. Слой 1 + Слой 2 в ветке `plan-152.8`.
> **Зависит от:** Plan 152 (merge в main), Plan 70.3 (D128 — `nova_char` typedef).
> **Не блокирует:** ничего в Plan 152 Phase B (все sub-plans закрыты без этой оптимизации).
> **Маркер:** `[M-152-unicode-codepoint-u32]`.

---

## Проблема

`char` в Nova = `typedef int64_t nova_char` (Plan 70.3, D128). Все буферы code point'ов
в `std/unicode` (`std/unicode/*_data.nv`, `collate_data.nv`, `category_data.nv`) и
CE-поток коллации — `Vec[int]` (64-бит). Это **корректно** (U+10FFFF = 21 бит, влезает
с запасом), но субоптимально: ×2 память vs индустрийный стандарт.

**Индустрия единогласна:** Go `rune` = `int32`, Rust `char` = `u32`, ICU `UChar32` =
`int32`, Java `codePointAt` = `int` (32-бит), Swift `Unicode.Scalar` = `UInt32`. Никто
не использует 64-бит для code point.

## Два слоя (независимые, разный масштаб риска)

### Слой 1 — Узкий: `Vec[int]` → `Vec[u32]` в `std/unicode` (внутренняя оптимизация)

Заменить буферы code point'ов и значения таблиц в `std/unicode` на `u32`. Не трогает
ABI `char`, не затрагивает пользовательский код.

**Что меняется:**
- `*_data.nv` range-таблицы (`norm_data`, `grapheme_data`, `word_data`, `sentence_data`,
  `category_data`, `case_data`) — значения cp как `u32` вместо `int`
- `normalize.nv` — внутренние буферы decomp/recomp: `Vec[u32]` вместо `Vec[int]`
- `collate_data.nv` — CE-поток: упаковать каждый CE в `u64` (primary 16 + secondary 16
  + tertiary 16 + flags 16) вместо плоского `Vec[int]` четвёрками → 4–8× компактнее
- Генераторы (`unicode-gen` в `nova-codegen`) — эмитировать `u32`-литералы

**Риск:** низкий. Чисто внутренний рефактор `std/unicode`. ABI `char` не трогается.
Требует обновить генераторы + вспомогательные `parse_*`-хелперы + сигнатуры функций.

**Выигрыш:** ×2 меньше памяти для таблиц и CP-буферов нормализации/коллации.

### Слой 2 — Глубокий: `nova_char` = `uint32_t` (ABI-breaking)

Переопределить `nova_char` с `int64_t` на `uint32_t` в `nova_rt.h`. Тогда `char`
одновременно и типобезопасный, и 4 байта — как Rust/Go/ICU.

**Что затрагивается:**
- `nova_rt.h` — typedef + все `NOVA_ARRAY_DECL(nova_char)`, `NovaOpt_nova_char`
- `array.h`, `conv.h` — хелперы с `nova_char`
- `emit_c.rs` — эмиссия char-литералов (`((nova_char)<cp>)` убрать `LL`-суффикс)
- `external_registry.rs` — сигнатуры внешних функций с `nova_char`
- Plan 70.3 (D128) — AMEND: `nova_char = uint32_t`, не `int64_t`
- Все сгенерированные C-файлы — пересборка

**Риск:** средний. ABI-breaking в C-layer (все `.c`-фикстуры требуют пересборки, не
перекомпиляции Nova-исходников). Функционально безопасно: U+10FFFF < 2²¹ < 2³² —
никакого truncation. Главный risk — MSVC: `uint32_t` vs `int64_t` в printf/cast в
ручном C-коде (если есть).

**Выигрыш:** `char` занимает 4 байта (как индустрия); `[]char` вдвое компактнее;
согласованность с Go/Rust/ICU/Java.

## Порядок выполнения

Слой 1 можно делать отдельно и раньше — он не зависит от Слоя 2. Слой 2 логичнее
делать после Слоя 1 (тогда таблицы уже на `u32`, миграция согласованная).

Оба слоя имеет смысл делать **после merge plan-152 в main** — пока ветка не слита,
нет смысла менять ABI поверх незамёрженного фундамента.
