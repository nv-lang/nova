## Контекст и связь с Plan 154

Plan 154 закрыл **cross-module** silent-no-op override (юзер переопределяет std-метод
— codegen тихо берёт std, тело юзера мёртвое → D267 громкая ошибка). Этот sub-план
закрывает **родственный** класс из той же зоны диспатча — **primitive-element
protocol-метод** тихо роутится в erased-стаб контейнера — плюс добавляет языковой
механизм **явной opt-in конформности** (`#impl(P)`), которого Nova не хватает.

Триггер (репортнул автор): `Vec[int].from([10,20,30]).debug(sb)` должен дать
`"Vec[10, 20, 30]"`, а выдаёт мусор/no-op (`plan131/vec_debug_pos` падает). Корень —
**НЕ** в `str.from_debug`, а в мис-диспатче элемента (диагностика, high confidence,
подтверждено пробой с концретным `@debug`).

## Проблема (репро)

```nova
ro v = Vec[int].from([10, 20, 30])
consume sb = StringBuilder.new()
v.debug(sb)
assert(sb.as_str() == "Vec[10, 20, 30]")   // FAIL: мусор/no-op
```

Тесты `display` через `${v}` / `str.from` ПРОХОДЯТ — но это **другой путь** (interp,
emit_c.rs ~25085), не вызов метода. Прямой `v.debug(sb)` / `v.display(sb)` (вызов
метода) мис-диспатчит. То есть «display работает» вводит в заблуждение: метод-путь
ломается одинаково для Display и Debug.

## Корневая причина (диагностика, high confidence)

Вызов `@data[i].debug(sb)` в mono-теле `Vec[T] @debug` идёт по 3 шагам диспатча в
`Call`→`Member` арме `emit_expr`
([emit_c.rs](../../compiler-codegen/src/codegen/emit_c.rs)):

1. **direct-dispatch** ([emit_c.rs:23042](../../compiler-codegen/src/codegen/emit_c.rs#L23042))
   — ПРОМАХ: у `int` нет конкретного `@debug`; `method_overloads[("int","debug")]`
   пуст, `all_methods` не содержит `("int","debug")`.
2. **default-body synthesis** `try_synthesize_default_method`
   ([emit_c.rs:23098](../../compiler-codegen/src/codegen/emit_c.rs#L23098) →
   [4644](../../compiler-codegen/src/codegen/emit_c.rs#L4644)) — ПРОМАХ дважды:
   - гейт `gate_on_impl=true`
     ([4690](../../compiler-codegen/src/codegen/emit_c.rs#L4690)) требует
     `type_impl_protocols.get("int")`, а примитивы туда не опт-инятся (карта
     заполняется ТОЛЬКО из `#impl(P)` на `type`-декларациях,
     [2063](../../compiler-codegen/src/codegen/emit_c.rs#L2063)) →
     `candidates.is_empty()` → `None`;
   - + мис-ключ: передаётся C-имя `"nova_int"` вместо Nova-имени `"int"`
     ([23093](../../compiler-codegen/src/codegen/emit_c.rs#L23093)).
3. **last-wins fallback** ([23110](../../compiler-codegen/src/codegen/emit_c.rs#L23110))
   — `method_receivers["debug"]=("Vec",true)` → эмитит erased-стаб
   `Nova_Vec_method_debug(int_tmp, sb)` — **no-op** (стаб генерится потому что у Vec
   поле `data *mut T` → `has_void_ptr_fields`,
   [10790](../../compiler-codegen/src/codegen/emit_c.rs#L10790)).

**И Display, и Debug** идут этим путём и падают идентично — баг **НЕ**
Debug-специфичный.

**Ключевой вывод (проба подтвердила):** дать элементу **конкретный** `@debug` →
mono-тело контейнера эмитит корректный `Nova_<T>_method_debug((self->item), sb)` через
direct-dispatch (шаг 1) вместо self-рекурсивного erased-стаба. То есть **Variant B
(конкретные impl примитивов) чинит мис-диспатч ДАРОМ** — отдельный
dispatch-resolution фикс НЕ нужен.

## Дизайн-решения (согласованы с автором)

1. **Конформность — opt-in** (НЕ номинальная/required). Структурная конформность
   («есть подходящий метод → тип годится») работает как сейчас; `#impl(P)` —
   **необязательная** пометка: написал → компилятор проверяет подпись + явно
   привязывает протокол к типу; не написал → работает по-старому. Не ломает
   существующий std/тесты, миграция не нужна. Путь к required — отдельным шагом позже
   (`[M-154.1-required-conformance]`).
2. **Синтаксис — `#impl(P)` ВЕДУЩИЙ атрибут** (как `#prelude`/`#stable`), НЕ хвостовой
   `@impl`:
   ```nova
   #impl(Debug)
   fn int @debug(mut sb StringBuilder) -> () { ... }
   ```
   Применим к **метод**-декларациям (новое) И **тип**-декларациям (уже частично есть —
   `type_impl_protocols` заполняется из `#impl` на типах,
   [emit_c.rs:2063](../../compiler-codegen/src/codegen/emit_c.rs#L2063)). Метод-уровень
   `#impl(P)` привязывает P к receiver-типу «как если бы протокол был описан в типе».
3. **Один план** (этот): `#impl` + Variant B вместе (фича + её первый потребитель).

## Фазы

### Ф.1 — Dispatch-hardening (Fix 1): громкая ошибка вместо тихого мис-диспатча [D269]

Зеркало строгого static-guard'а
([emit_c.rs:23122](../../compiler-codegen/src/codegen/emit_c.rs#L23122)) для
INSTANCE-вызовов. Вставить СРАЗУ после `let is_generic_type = self.generic_types.contains(&type_name);`
(~[emit_c.rs:23145](../../compiler-codegen/src/codegen/emit_c.rs#L23145)), ДО bare-T и
`[]T` веток. **Точный код (проверен сборкой, готов к вставке):**

```rust
                    let is_generic_type = self.generic_types.contains(&type_name);
                    // Fix 1 [robustness (C), 2026-06-13]: instance-call counterpart of
                    // the static guard above. If the receiver is a PRIMITIVE C-type but
                    // the single-key `method_receivers` fallback resolved `method` to a
                    // GENERIC/erased container type (e.g. `Vec`), the primitive cannot be
                    // an instance of it. Primitive-direct-dispatch (above, ~23042) and
                    // default-body synthesis (~23091) both already MISSED, so emitting
                    // `Nova_<Generic>_method_*(prim, ...)` would silently call an erased
                    // no-op stub with a type-confused receiver — the long-standing
                    // `vec_debug_pos` mis-dispatch (a primitive element's `.debug(sb)`
                    // inside `Vec[T] @debug` routed to `Nova_Vec_method_debug`). Fail
                    // LOUDLY at compile time instead of emitting a wrong/no-op call.
                    if is_instance && is_generic_type {
                        let recv_ty = self.infer_expr_c_type(obj);
                        let prim_name = match recv_ty.as_str() {
                            "nova_int" => Some("int"),
                            "nova_char" => Some("char"),
                            "nova_bool" => Some("bool"),
                            "nova_f32" => Some("f32"),
                            "nova_f64" => Some("f64"),
                            "nova_str" => Some("str"),
                            _ => None,
                        };
                        if let Some(pn) = prim_name {
                            return Err(format!(
                                "[E_PRIMITIVE_NO_PROTOCOL_METHOD] примитив `{prim}` не \
                                 имеет метода `{m}` — `{prim}` не реализует протокол, \
                                 предоставляющий `.{m}(...)`, а generic-тип `{reg}` \
                                 (single-key fallback) не применим к примитивному \
                                 ресиверу. Подсказка: дать `{prim}` конкретный метод \
                                 через `#impl(<Protocol>)` для `{m}`, либо это опечатка.",
                                prim = pn, m = method, reg = type_name,
                            ));
                        }
                    }
```

- **Узость:** срабатывает только при `is_instance && is_generic_type && receiver —
  примитивный C-тип`. Легитимные Vec/HashMap-вызовы (receiver `Nova_Vec*` / `Nova_Vec____*`,
  не примитив) НЕ задеты. bare-T (`type_name` — typevar, НЕ в `generic_types`) и `[]T`
  (`type_name` начинается с `[]`) — НЕ задеты (их ветки идут после и обрабатывают сами).
- **Почему безопасно:** примитив, имеющий реальный метод, ловится на 23042 до этого
  места; значит дойти сюда с примитивным ресивером = метода нет = вызов через
  generic-fallback всегда мис-диспатч → ошибка корректна.
- **Эффект:** `vec_debug_pos` из ТИХОГО RUN-FAIL → ГРОМКИЙ CC-FAIL (правильное
  промежуточное состояние; Ф.3 его зеленит). Самостоятельный compiler-correctness
  фикс, переживает Variant B как safety-net (после B у int есть метод → guard для int
  не срабатывает).
- **Verify:** broad regression (plan131, contracts, plan90, plan90_1, plan140_2,
  plan138_2, plan128, plan99, basics, generics, plan62, plan101, plan91). Pass-count НЕ
  должен УПАСТЬ нигде, КРОМЕ `vec_debug_pos` (RUN-FAIL → CC-FAIL, тот же 1 fail). Если
  падает что-то ещё — это false-positive guard'а, разобрать (вероятно мис-инференс
  `infer_expr_c_type`). **Коммит — ОТДЕЛЬНЫЙ** (compiler-correctness, до Ф.2/Ф.3).

### Ф.2 — `#impl(P)` opt-in конформность [D268]

- **Парсер:** принять `#impl(ProtocolName)` как ведущий атрибут на (а) метод-декларациях
  (новое), (б) тип-декларациях (уже частично — унифицировать). Сохранить в AST-узле
  декларации (список протоколов).
- **Чекер** (при `#impl(P)` на методе `fn T @m(...)`):
  - P — существующий протокол? иначе `E_IMPL_UNKNOWN_PROTOCOL`.
  - имя `@m` ∈ методов P? иначе `E_IMPL_NOT_A_PROTOCOL_METHOD`.
  - сигнатура `@m` матчит метод P с подстановкой `Self`→`T` / `@`→receiver? иначе
    `E_IMPL_SIGNATURE_MISMATCH` (проверять параметры + возврат + receiver-mutability).
  - **привязать P к типу T** — наполнить `type_impl_protocols[T] += P` (та же карта,
    что [emit_c.rs:2063](../../compiler-codegen/src/codegen/emit_c.rs#L2063)).
  - зарегистрировать метод в `method_overloads`/`all_methods` (чтобы direct-dispatch
    23042 его находил).
- **Opt-in инвариант:** отсутствие `#impl` — НЕ ошибка; структурная конформность
  по-прежнему удовлетворяет бонд `[T P]`. `#impl` лишь ДОБАВЛЯЕТ проверку + явную
  привязку. Никакой существующий код не должен сломаться.

### Ф.3 — Variant B: конкретные Display/Debug примитивов [D269]

Дать `int, f64, f32, bool, char, str` конкретные `#impl(Display)` + `#impl(Debug)`
методы, роутящие в [conv.h](../../compiler-codegen/nova_rt/conv.h)-форматтеры (они уже
существуют):

| тип | `@display` → | `@debug` → |
|-----|--------------|------------|
| int | `nova_int_to_str` | `nova_int_to_debug_str` |
| f64 | `nova_f64_to_str` | `nova_f64_to_debug_str` |
| f32 | (f32→f64 path) | (f32→f64 path) |
| bool | `nova_bool_to_str` | `nova_bool_to_debug_str` |
| char | `nova_char_to_str` | `nova_char_to_debug_str` (кавычки `'c'`) |
| str | identity | `nova_str_to_debug_str` (кавычки+escape) |

- **Где:** примитивы prelude-global → в `std/prelude/` (рядом с
  [protocols.nv](../../std/prelude/protocols.nv)). Нужен способ из Nova-тела дёрнуть
  конкретный conv.h-форматтер — либо `external fn`-декларация на conv.h-функцию, либо
  интринсик-метод (как сейчас делает interp-путь `${x:?}`). **Решение реализатора:**
  предпочесть external-fn-декларацию (явно, без новой codegen-магии); если затратно —
  узкий интринсик `str.from_debug_<prim>` только для conv.h-роутинга.
- **Эффект:** `int.debug` теперь КОНКРЕТНЫЙ → direct-dispatch (23042) ловит → mono
  `Vec[int] @debug` элемент-вызов резолвится корректно → **Display+Debug работают,
  `vec_debug_pos` ЗЕЛЁНЫЙ, Ф.1-guard для int не срабатывает** (у int есть метод).
- **Проверить взаимодействие с interp:** `${x}` / `${x:?}` (emit_c.rs ~25085) НЕ должны
  сломаться — они идут своим путём; конкретные impl не должны их перехватить/задублить.
