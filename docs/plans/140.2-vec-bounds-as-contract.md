## Investigation findings (2026-06-13) — почему prerequisite-first

Попытка реализовать буквальную форму маркера (`requires 0 <= i && i < @len` на `Vec @index`) вскрыла
**два жёстких блокера**:

1. **Блокер A (фатальный для contract-подхода):** контракты **не поддерживают `@field` (self-access)**.
   `encode.rs:508` (`ExprKind::SelfAccess => Err(Unsupported(...))`) отвергает `@len` с
   `[E2401] @field в контракте не поддерживается SMT-encoder'ом`. Без `@len`/`@method()` в контракте
   Plan 140 не может **ни доказать, ни даже закодировать** bounds-контракт → элизия через контракты
   **недостижима в принципе**. Добавление контракта на `@index` **ломает сборку** (basics 0/8, проверено).
2. **Блокер B (codegen):** inline `v[i]` — это **lvalue** statement-expr ([emit_c.rs:18920](../../compiler-codegen/src/codegen/emit_c.rs#L18920));
   нужно для `v[i].field = x` (проверено: `v[0].x = 5` работает) и `&v[i]`. De-inline на
   value-возвращающий метод-вызов сломал бы lvalue-контексты → нужен **context-sensitive codegen**
   (rvalue → вызов; lvalue → inline / `mut @index`). Write `v[i] = val` уже отдельным путём
   (Stmt::Assign, [emit_c.rs:16047](../../compiler-codegen/src/codegen/emit_c.rs#L16047)).

**Текущее состояние:** `v[i]` **уже безопасен** (always-on inline bounds-check → `nv_panic_index_oob`).
Не хватает только **элизии** доказуемого доступа (perf), а это и есть ценность маркера.

**Решение автора (2026-06-13):** реализовать **пререквизит (Part A: `@field`/`@method()` в контрактах)
ПЕРВЫМ**, затем Part B (bounds-as-contract + context-sensitive de-inline). НЕ срезать через
codegen-only элизию (sibling `[M-opt-elide-proven-overflow-checks]`).

## Part A (ПРЕРЕКВИЗИТ) — `@field` / `@method()` self-access в контрактах

Расширить контракт-checker + SMT-encoder, чтобы контракт мог ссылаться на поля/чистые методы receiver'а
(`@len`, `@len()`, `@cap`, …). Это **широко полезно** (любой метод-контракт о состоянии self), не только
для bounds.

### A.0 — Spec / D / Q (~0.5d)
- D-блок: контракты МОГУТ ссылаться на `@field` (self-field read) и `@method()` (если метод `#pure`),
  receiver моделируется как SMT-сущность. Ограничения (только read; field — `#pure`-визибилен; метод —
  `#pure`). Amend D24 / D106 (verify_status). Q-блок про cross-function call-site instantiation (см. A.2).
- **Решить:** MVP scope — (a) within-function (метод доказывает о своём `@field` в ensures/invariant) vs
  (b) full call-site instantiation (caller доказывает контракт callee, подставляя свой receiver) —
  **(b) нужен для элизии bounds**; (a) проще. Зафиксировать в Q.

### A.1 — Encoder: `@field` → SMT term (~1d)
- `encode.rs`: вместо `Err` на `ExprKind::SelfAccess` / `SelfFieldAccess` / `Member{obj:SelfAccess}` —
  кодировать `@field` как SMT-терм. Модель: receiver `self` — uninterpreted-сущность; `@f` — либо
  uninterpreted function `f(self)`, либо свежая SMT-переменная per-field, типизированная из record-schema.
- `@method()` (e.g. `@len()`): если метод `#pure` с простым телом (`=> @len`) — inline/uninterpreted;
  иначе reject (как сейчас). MVP: поддержать прямое `@field`; `@len()` → опционально (через `@len`).
- Типы: вывести SMT-sort поля из record-schema (int → Int/BV, bool → Bool, …).

### A.2 — Call-site instantiation (~1d, ХАРД, нужно для элизии)
- При проверке контракта callee на CALL-сайте — подставить **фактический receiver** (`v`) и его поля:
  `@len` callee ↦ `len(v)` caller'а. Чтобы доказать `0 <= i && i < v.len()` в `for i in 0..v.len()`,
  Z3 связывает loop-bound `v.len()` с полем `@len`. Требует: receiver-aware подстановка в encode +
  loop-invariant поддержка (Plan 33/140) для индекса.
- **Риск-узел:** моделирование состояния объекта в SMT + соответствие `v.len()` (метод) ↔ `@len` (поле).
  Возможен MVP: только когда индекс — loop-var с явной границей-выражением `v.len()`/`@len`.

### A.3 — Checker (~0.25d)
- Разрешить `@field`/`@method()` в contract-expr на этапе type/purity-check (сейчас, видимо, доходит до
  encoder и падает E2401 — но проверить, нет ли раннего reject в checker). Field — read-only; `@method()`
  — `#pure`. Негатив: write/`mut @field` в контракте → ошибка.

### A.4 — Тесты + acceptance (~0.5d, релизный nova)
- Pos: метод с `requires @field > 0` / `ensures result <= @cap` доказывается (Z3); call-site элизия
  proven; `@len()` если поддержан.
- Neg: `@field` non-pure-метод в контракте → внятная ошибка; write в контракте → ошибка.
- **Commit'ы Part A:** A.0 spec → A.1 encoder → A.2 call-site → A.3 checker → A.4 tests (раздельно).

## Part B — bounds-as-contract + context-sensitive de-inline (зависит от Part A)

### B.0 — Spec D-блок (~0.25d)
- `Vec @index`/`mut @index` несут `requires 0 <= i && i < @len`; bounds = элидируемый контракт (proven →
  zero-cost, unproven → release abort). `@get`/`@first`/`@last` (Option) — без контракта. (Это и был
  отозванный преждевременный D249 — внести когда Part A разблокирует.)

### B.1 — stdlib (~0.25d)
- `Vec[T] @index(i int)` + `mut @index(i int, val T)`: убрать ручной `if...panic`, добавить
  `requires 0 <= i && i < @len`. (Зависит от Part A — иначе E2401.)

### B.2 — codegen: context-sensitive route (~1d, риск B)
- **Rvalue** `v[i]` (read, emit_c.rs:18920): route через монолизированный `@index` (reuse
  `vec_method_call` / `register_vec_mono_method`, 27303/27275) → Plan 140 элизия на call-site.
- **Lvalue** `v[i]` (`v[i].field=x`, `&v[i]`): сохранить inline statement-expr (lvalue-совместимо), но
  bounds-check сделать элидируемым (consult proven при Part A.2) ИЛИ оставить always-on.
- **Write** `v[i]=val` (Stmt::Assign 16047): route через `mut @index` ИЛИ inline+элидируемый check.
- Проверить: монолизированный `@index` эмитит контракт-enforcement (register_mono_method_instance →
  contract-aware fn-emission). Per-fix targeted + **basics обязательно** (урок 139.2).

### B.3 — Тесты + perf + close (~0.5d, C-codegen + interp)
- Pos: in-bounds `v[i]`/`v[i]=x`; `for i in 0..v.len()` — proven, спот-чек C: нет bounds-branch (B4).
  `@get` всё ещё None на OOB.
- Neg/trap: OOB `v[i]`/`v[i]=x` → runtime abort в debug И release.
- Perf-sanity: unproven `v[i]` не регрессит (метод заинлайнен C-компилятором).
- **Commitّ'ы Part B:** B.0 → B.1 → B.2 → B.3.

## Критерии приёмки

**Part A:**
- **A1** — контракт может ссылаться на `@field` (self-field read); кодируется в SMT (нет E2401).
- **A2** — `@method()` для `#pure`-метода в контракте поддержан ИЛИ внятно отклонён (зафиксировано).
- **A3** — call-site instantiation: контракт callee с `@field` доказывается у caller'а (подстановка
  receiver'а) — минимум для loop-var индекса с границей `v.len()`/`@len`.
- **A4** — write/`mut @field`/non-pure-`@method()` в контракте → внятная ошибка (не E2401-crash).
- **A5** — pos+neg релизным nova; 0 регрессий (basics + contracts + широкий sweep).

**Part B:**
- **B1** — `@index`/`mut @index` несут `requires 0<=i && i<@len`; ручной `if...panic` убран; сборка зелёная.
- **B2** — `v[i]`/`v[i]=val` enforce bounds; **lvalue** `v[i].field=x` / `&v[i]` НЕ сломаны.
- **B3** — OOB → runtime abort в debug И release; сообщение с index/len.
- **B4** — доказуемый индекс (`for i in 0..v.len()`) — проверка **элидируется** (спот-чек C: нет branch).
- **B5** — `@get`/`@first`/`@last` — поведение не изменилось (None на OOB).
- **B6** — pos+neg (C-codegen + interp); 0 регрессий (basics + широкий sweep); perf unproven `v[i]` не регрессит.

- **Build-unblock (доп. к плану):** D: — exFAT, junction/symlink невозможны; z3-`build.rs` ищет

  на call-site» (из плана) **несостоятелен** для per-site элизии (per-method ключ элидит «всё-или-ничего») и
  ломает lvalue-контексты. Выбран предложенный планом альтернативный путь: inline + per-site элидируемый
