## 1. Цель

Свести типовую систему Nova из **четырёх независимых движков над разными представлениями** к
**одному** семантическому проходу, который строит **типизированный IR**; codegen этот IR
**потребляет** (только лоуэрит), C-компилятор **никогда не ловит ошибку типов**.

Это прямая реализация [`compiler-conventions.md`](../compiler-conventions.md) §0: «типы резолвятся
ровно один раз — в чекере; codegen ему доверяет; C никогда не должен ловить ошибку типов».

## 2. Мотивация — фрагментация «как есть»

Полная карта (3-агентный аудит 2026-06-19) — в [Plan 172.2 §8](172.2-method-arg-type-checking.md).
Кратко, **четыре** независимых движка типов:

1. **Checker** (`types/mod.rs`) — `Ty`/`TyCat`/`TypeRef`, best-effort inference, **частичный**
   single-overload резолв; знает stdlib через **хардкоженный** `builtins: HashSet` (:11497).
2. **Codegen** (`emit_c.rs`) — типы как C-строки, `infer_expr_c_type` (:35718, ~1900 строк
   re-derive), **настоящий** резолв перегрузок (`resolve_overload` :11140), знает stdlib через
   `ExternalRegistry` (parsed из `.nv`, :634).
3. **Const-fn pipeline** (`const_fn_closure.rs`/`const_fn_trampoline.rs`) — **третий** TypeRef
   unification/substitution + свой sig-registry.
4. **Sum-schema** (`codegen/sum_schema_registry.rs`) — layout sum-типов «параллельно» legacy
   `sum_schemas`.

**Следствия (то, что нас и привело сюда):**

- **Soundness-дыры между слоями.** Generics ловятся (raw-`TypeRef`-сравнение), скаляры — отдельной
  веткой, method-args — вообще в codegen, scalar-narrowing в method-arg — **никем**.
- **Рваная диагностика.** Часть ошибок — чистые Nova-ошибки (чекер), часть — codegen-ошибки
  (`no matching overload`), часть — уродливые `CC-FAIL` (категория str↔int).
- **Дублирование/дрейф.** Inference выражения — дважды; type→C-mapping — трижды (`simple_type_ref_to_c`
  **уже дрейфил**: u32 отсутствовал → `Vec[u32]` мис-манглился); метод-таблицы — ×4; знание stdlib —
  **дважды** (худшая поверхность: checker `builtins` ↔ codegen `ExternalRegistry`).

## 3. Целевая архитектура

```
parse → [SEMANTIC ANALYSIS: один движок] → типизированный IR → codegen (лоуэринг) → C
```

- Один резолвер/инференсер: тип каждого выражения и выбранный callee каждого вызова (перегрузки/
  методы/builtin — по типам) вычисляются **один раз**.
- Типизированный IR — единственный источник истины: каждое выражение аннотировано типом, каждый
  вызов — выбранным callee + моно-substitution.
- Один реестр методов/сигнатур/stdlib (включая builtin — из `.nv`, §2), читаемый обоими слоями.
- Codegen — только лоуэринг: читает IR, не выводит/не резолвит типы заново.
- C не ловит ошибки типов; все классы (`no matching overload`, D124, `E_IMPLICIT_NARROWING`) —
  диагностики **чекера**.

### U.1 — Единое знание stdlib

> **Summary.** Phase U.1 makes the checker draw its stdlib knowledge from ONE registry built from `.nv` sources (ExternalRegistry, codegen/external_registry.rs:57, built via load_builtins :110) instead of the hardcoded `builtins: HashSet` in the checker. Current state: the checker's stdlib-name hardcode lives in `NameResCtx.builtins` (types/mod.rs:11497 field, :11681 populated literal, :11849 constructed, used at exactly 3 sites — `is_known` :12544, `is_variant_like` :12485, path-head undefined-name check :12070). The checker has ZERO access to ExternalRegistry today (grep confirms load_builtins is called only in CEmitter::new, emit_c.rs:1335). The `builtins` literal mixes three classes: (a) GENUINE language builtins with no `.nv` (primitives int/u32/.../never, bool literals true/false, Self/self, any) — a legitimate conventions §2 exception; (b) stdlib intrinsic namespaces gc/bench/fibers/runtime — these ARE declared in std/.nv as `extern "nova" fn gc.heap_size()` etc. (std/runtime/gc.nv:74, fibers.nv, runtime.nv, std/bench.nv) and so are derivable from a registry, BUT load_builtins' current source list (external_registry.rs:112-126) does NOT include those four files; (c) builtin effect/handle types Fail/Detach/CancelToken and free fn `fence` (Fail/Detach are prelude effects; fence is in sync.nv → already in ExternalRegistry). Empirically verified hot constraint: nova_tests/concurrency/gc_differential.nv calls `gc.collect()` with NO `import std.runtime.gc`, so the `"gc"` builtins entry is load-bearing — removing namespace names without first making the checker's registry cover them breaks every namespace-without-import test. NOTE: `is_intrinsic_namespace` (types/mod.rs:8706) is a SEPARATE overlapping hardcoded list (adds Channel/Time/Monotonic/Duration which have no `extern fn` `.nv`); per Plan 172.1 it belongs to U.2 (dedup vs codegen guard :23338) and is OUT of U.1 scope except where it must stay consistent.
>
> **Feasibility.** Realistically completable as a self-contained production-grade phase — it is narrowly scoped to ONE hardcode (the checker's `builtins: HashSet`, exactly 3 read sites, all in types/mod.rs) and a registry already exists and is cheap to build. No task here is a multi-week effort. Two real hazards, both de-risked into their own tasks: (1) U.1.2 touches the SHARED ExternalRegistry/load_builtins which codegen also consumes (emit_c.rs is hot, method_overloads is order-sensitive) — mitigated by a byte-compare of generated C and a documented fallback to a checker-private registry if any drift appears; this fallback keeps codegen untouched and is the safe default if measurement shows ANY codegen delta. (2) U.1.4 (Fail/Detach/CancelToken) is the subtlest: these effect/handle names may currently resolve ONLY through the hardcode (effects have no `extern fn` .nv), so the task is explicitly gated on grep-proving each one's non-hardcode resolution path BEFORE migrating; if any is a true language-builtin with no `.nv`, it stays in the documented §2 set rather than being forced into the registry. IMPORTANT SCOPE CLARIFICATION vs the prompt/audit: the `builtins` HashSet is a NAME-RESOLUTION set, not a method-signature store; verified empirically that `method_table` (TypeCheckCtx :2509) IS already built from inlined prelude `Item::Fn` (so Vec.push/str.* method SIGNATURES are already present to the checker). Therefore U.1 alone removes the worst drift surface and gives the checker registry-backed stdlib NAME knowledge, but it does NOT by itself make the checker resolve instance-method receiver TYPES — that blocker (the `_ => return` at :6476) is U.3/Plan 172.2 and must not be conflated into U.1. The audit §8.3 phrasing 'builtin Vec/str methods are not in method_table' is imprecise for Nova-body prelude methods; the genuine, verified gap U.1 closes is the hardcoded NAME set. `is_intrinsic_namespace` (:8706) is a related-but-separate hardcoded list owned by U.2 and is intentionally left in place here (only kept consistent), to avoid scope creep.
>
> **Критерий приёмки фазы.** U.1 is done when: (1) The checker's stdlib-name knowledge is sourced from ONE registry built from `.nv` (ExternalRegistry, extended in U.1.2) consumed at all three name-resolution sites (types/mod.rs :12070, :12485, :12544); the `builtins` literal (:11681) retains ONLY genuine language builtins with no `.nv` (primitives, true/false, Self/self, any, never), reframed and documented as the conventions §2 exception with a `[M-*]` marker. (2) The checker visibly resolves builtin stdlib names (gc/bench/fibers/runtime namespaces, CancelToken/fence, Fail/Detach effects) via the registry — proven by positive fixtures that previously relied on the hardcode (notably gc-without-import). (3) Production-grade, NO simplifications: real registry lookups, no name-keyed special-cases; the user-type-with-same-method-name fixture proves the mechanism is type-agnostic (conventions §2). Any permissive/fallback (load_builtins Err → default; checker-private registry if codegen byte-drift) is documented with a marker, not silent. (4) Conforms to compiler-conventions.md §0 (moves toward single source of truth), §1 (checker owns name-resolution), §2 (no stdlib hardcode beyond the documented primitive exception), §3 (no new silent `_ => return`). (5) Positive AND negative tests via RELEASE nova (C-codegen pipeline, EXPECT_COMPILE_ERROR in neg/ folder-module) — U.1.7. (6) 0 regressions vs a clean binary across a FULL `nova test` regression (not a sample) for the removal task U.1.6, verified by stash+rebuild baseline (conventions §6.5-§6.6); diagnostic-class diff (undefined identifier / undefined module-name) is empty-delta. (7) Docs/spec/error-index/D-block updated (U.1.8).

#### U.7.3 — Assert the zero-type-CC-FAIL guarantee as a gating regression check
- **Что:** Turn the U.7.1 leak count into an enforced gate: a test/CI check that the positive corpus produces ZERO un-expected type-class CC-FAILs, and that the U.7.2 negative suite produces zero CC-FAIL stages (all clean CODEGEN-FAIL). The gate is parameterized by the marker set from U.7.2 so that not-yet-enforced classes are listed as KNOWN gaps with a shrinking allowlist — every later phase (U.3, Plan 172.2, U.4) that converts a CC-FAIL class to a checker diagnostic removes one allowlist entry. Final state (post-U.3/U.4): allowlist empty, gate fully green.

Production-grade: gate runs in CI; allowlist entries each cite a `[M-*]`/phase that will remove them.
- **Blast-radius:** CI/test-gate. Once green, any future change that pushes a type error to the C stage is caught immediately. This is the permanent guard the whole umbrella is for.

De-risk: the gate keys ONLY on type-class CC error patterns (incompatible type / passing X to / unknown type name), not on link/toolchain CC failures; toolchain CC failures remain a separate (non-type) bucket per the U.7.1 classification.


## 5. Критерии приёмки umbrella

1. **Единый источник истины:** один реестр методов/сигнатур/stdlib; checker и codegen — консьюмеры,
   не два движка.
2. **Codegen не резолвит и не выводит типы заново** — потребляет типизированный IR.
3. **C не ловит ошибки типов** (U.7-гарантия).
4. **Никакого хардкода** типов/функций/методов Nova в Rust (§2) — stdlib из `.nv`, не из
   `builtins: HashSet`/списков.
5. Дублирование из §2-таблицы Plan 172.2 §8.4 устранено (inference, type→C, метод-таблицы, width,
   intrinsic-список, sum-layout).
6. На каждом этапе: 0 регрессий против чистого бинаря; spec/доки/error-index обновлены; D312 написан.

## 6. Принципы выполнения

- **Этапами, не «большим взрывом».** Каждый U.x — отдельная ветка/коммит, измеримый, реверсивный.
- **Регресс против чистого бинаря** (stash+rebuild) на каждом шаге — отличать pre-existing от регрессий.
- **Blast-radius до правки** — особенно где перенос проверки в чекер вскроет молчавшие ошибки
  (например U.3 даст реальные диагностики там, где раньше был тихий codegen-dispatch).
- **Не ломать std молча** — мигрировать зависимости в том же шаге.
- `emit_c.rs`/`types/mod.rs` — hot, часто правятся параллельно: координировать регионы.

## 7. Риски

- **Объём.** Многомесячный рефактор ядра — поэтому umbrella с независимыми фазами, а не один план.
- **U.3 (резолв в чекер)** вскроет ошибки, которые раньше тихо проходили до codegen/C → blast-radius
  + миграция (часть — Plan 172.2).
- **U.5 (единое представление)** трогает `Ty`/`TyCat` — широко используемые; делать с полным регрессом.
- Координация с параллельными сессиями на `emit_c.rs`/`types/mod.rs`.
