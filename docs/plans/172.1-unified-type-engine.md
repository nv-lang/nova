## 1. Цель

Свести типовую систему Nova из **четырёх независимых движков над разными представлениями** к
**одному** семантическому проходу, который строит **типизированный IR**; codegen этот IR
**потребляет** (только лоуэрит), C-компилятор **никогда не ловит ошибку типов**.

Это прямая реализация [`compiler-conventions.md`](../compiler-conventions.md) §0: «типы резолвятся
ровно один раз — в чекере; codegen ему доверяет; C никогда не должен ловить ошибку типов».

## 2. Мотивация — фрагментация «как есть»

Полная карта (3-агентный аудит 2026-06-19) — в [Plan 172.2 §8](172.2-method-arg-type-checking.md).
Кратко, **четыре** независимых движка типов:

1. **Checker** (`types/mod.rs`) — `Ty`/`TyCat`/`TypeRef`, best-effort inference, **частичный**
   single-overload резолв; знает stdlib через **хардкоженный** `builtins: HashSet` (:11497).
2. **Codegen** (`emit_c.rs`) — типы как C-строки, `infer_expr_c_type` (:35718, ~1900 строк
   re-derive), **настоящий** резолв перегрузок (`resolve_overload` :11140), знает stdlib через
   `ExternalRegistry` (parsed из `.nv`, :634).
3. **Const-fn pipeline** (`const_fn_closure.rs`/`const_fn_trampoline.rs`) — **третий** TypeRef
   unification/substitution + свой sig-registry.
4. **Sum-schema** (`codegen/sum_schema_registry.rs`) — layout sum-типов «параллельно» legacy
   `sum_schemas`.

**Следствия (то, что нас и привело сюда):**

- **Soundness-дыры между слоями.** Generics ловятся (raw-`TypeRef`-сравнение), скаляры — отдельной
  веткой, method-args — вообще в codegen, scalar-narrowing в method-arg — **никем**.
- **Рваная диагностика.** Часть ошибок — чистые Nova-ошибки (чекер), часть — codegen-ошибки
  (`no matching overload`), часть — уродливые `CC-FAIL` (категория str↔int).
- **Дублирование/дрейф.** Inference выражения — дважды; type→C-mapping — трижды (`simple_type_ref_to_c`
  **уже дрейфил**: u32 отсутствовал → `Vec[u32]` мис-манглился); метод-таблицы — ×4; знание stdlib —
  **дважды** (худшая поверхность: checker `builtins` ↔ codegen `ExternalRegistry`).

## 3. Целевая архитектура

```
parse → [SEMANTIC ANALYSIS: один движок] → типизированный IR → codegen (лоуэринг) → C
```

- Один резолвер/инференсер: тип каждого выражения и выбранный callee каждого вызова (перегрузки/
  методы/builtin — по типам) вычисляются **один раз**.
- Типизированный IR — единственный источник истины: каждое выражение аннотировано типом, каждый
  вызов — выбранным callee + моно-substitution.
- Один реестр методов/сигнатур/stdlib (включая builtin — из `.nv`, §2), читаемый обоими слоями.
- Codegen — только лоуэринг: читает IR, не выводит/не резолвит типы заново.
- C не ловит ошибки типов; все классы (`no matching overload`, D124, `E_IMPLICIT_NARROWING`) —
  диагностики **чекера**.

### U.1 — Единое знание stdlib (ПЕРЕРАБОТАНО: uniform-пакет-резолв, std не особый)

> **Переформулировка (2026-06-19).** Исходная U.1 «дать чекеру `ExternalRegistry`» **отвергнута**: `ExternalRegistry::load_builtins` сам хардкодит список std-файлов (`include_str!`), и её экспонирование благословило бы анти-паттерн (conventions §2). Корень: prelude **уже** импортируется обычным import-путём; std — это **путь** `stdlib_dir`, не inline-список; единственное дублирование — `load_builtins()` парсит std-файлы **отдельно** в codegen параллельно уже-инлайненному модулю. Keystone — **U.1.3**: удалить отдельный парс, построить ОДИН реестр из import-resolved модуля и отдать его и чекеру, и codegen.
>
> **Целевая модель.** CONCRETE TARGET UNIFORM-PACKAGE MODEL (verified against the actual code, not the audit's framing).  Crucial finding that reframes U.1: the uniform mechanism the principle demands ALREADY EXISTS for the prelude and std — and ExternalRegistry::load_builtins is the ONE redundant copy on top of it. Specifically: - std is already a search-path-located folder: `stdlib_dir = repo.join("std")` (nova-cli/src/main.rs:1109; threaded as `stdlib_dir: &Path` through every pipeline). It is NOT yet *configurable*, but it is already a path, not an include_str! list, for the IMPORT side. - prelude is already implicitly-imported via the NORMAL import path: imports.rs:174-189 injects `import std.prelude` (resolved as `stdlib_dir/prelude.nv`) into the entry module unless `#no_prelude`; #prelude(a,b) selects `std/prelude/<name>.nv`. resolve_module_paths(imports.rs:2144) maps a dotted module path to physical .nv files under entry_dir/repo/stdlib_dir/dep_root by ONE mechanism for user, std and deps. So criterion (2) of the principle is already met by the import resolver; the checker and codegen both consume the resulting INLINED module. - The duplication is exactly ONE thing: ExternalRegistry::load_builtins() (external_registry.rs:110-132) — a hardcoded `include_str!` array of 10 std files (string_builder/write_buffer/read_buffer/char/sync/raw_mem/ffi.cstr/net.addr/net.tcp/net.udp), parsed SEPARATELY in codegen (emit_c.rs:1335) PARALLEL to the already-inlined prelude module that codegen also receives. This is the "separate std parse in codegen" + "hardcoded std file list" the principle bans. The checker never sees it at all (its stdlib NAME knowledge is the hardcoded `builtins: HashSet`, types/mod.rs:11681). - `if name=="gc"/"bench"/"fibers"/"runtime"` (emit_c.rs:24649/24699/24727/24749) and is_intrinsic_namespace (types/mod.rs:8706 ↔ emit_c.rs:23338) are a DIFFERENT class: these lower to C runtime calls (nova_gc_collect() etc.). The .nv files (std/runtime/gc.nv etc.) are signature-only `extern "nova" fn gc.heap_size()` — genuine COMPILER INTRINSICS, not .nv-bodied. gc.nv:47-54 literally documents the hand-sync requirement. These cannot be "resolved away"; they need a declared-intrinsic mechanism, not deletion.  TARGET MECHANISM (how each principle clause is concretely met): (A) std = just-another-package via configurable search-path. Replace the hardcoded `repo.join("std")` (main.rs:1109, and the duplicate `repo.join("std")` at main.rs:1436/1546, bench/field_cache) with a resolved `StdSearchPath` that comes from: CLI `--std-path` > env `NOVA_STD_PATH` > `nova.toml [package] std` (manifest.rs) > default `repo/std`. "WHERE to find std" becomes config (allowed by §2 boundary); "WHAT is in std" (file names/signatures) stays in the folder, never in Rust. No behavior change at the default; the include_str! list is what gets deleted. (B) prelude = implicitly-imported packages via the normal import path. KEEP imports.rs:174-189 exactly as the mechanism (it already is the normal import path). The only change: source the prelude module path(s) from the same StdSearchPath, and make the set of implicitly-imported packages (currently just `std.prelude`) a named, documented set rather than a literal — but resolution stays the existing import resolver. (C) ONE resolution feeds both checker and codegen. The single resolution is the ALREADY-INLINED module (parse → resolve_imports_inline merges std/prelude/dep items into module.items). Build ONE registry from THAT module (ExternalRegistry::from_module already does exactly this for codegen at emit_c.rs:2157) and hand the SAME built registry to the checker. Delete load_builtins() (the separate std parse). For std namespaces used WITHOUT import (gc/fibers/runtime/bench in tests), the principled fix is: make them reachable through the import resolver as implicitly-available intrinsic packages OR via a single declared-intrinsic table built from their .nv signatures — NOT a checker-private hardcoded name set. (D) No builtins:HashSet of stdlib names. The checker's stdlib NAME knowledge comes from the inlined-module-derived registry + the import scope (peer_module_names already tracks imported namespaces). Only genuine language builtins with no .nv (primitives int..f64, true/false, Self/self, any, never) remain — the §2-sanctioned exception, renamed `language_builtins`. (E) No `if name=="gc"` duplication of KNOWLEDGE. The intrinsic LOWERING (gc.collect → nova_gc_collect()) is legitimately codegen's job and stays, BUT it must be driven by a SINGLE declared-intrinsic registry built from the .nv `extern "nova" fn` signatures (one source of truth for both: checker resolves the call's type from the signature; codegen looks up the intrinsic c-lowering by (namespace,method) key). The two is_intrinsic_namespace lists collapse to one shared const. This is the genuine "no parallel stdlib registry" fix and is the bridge between U.1 and U.2.5/U.6.  NET: the target is not "expose ExternalRegistry to the checker" (which would bless the hardcoded file list). It is "delete the hardcoded file list; build ONE registry from the already-inlined std module that the import resolver produces; make std-path config; hand that ONE registry to the checker; keep intrinsic lowering as a single declared-intrinsic table." A real package/search-path abstraction is MOSTLY already present (stdlib_dir + resolve_module_paths); the only NEW abstraction needed is (1) making stdlib_dir config-resolved and (2) a single declared-intrinsic registry built from .nv to replace both the checker name-hardcode and the codegen separate-parse for the namespaces that have no .nv body.

## 5. Критерии приёмки umbrella

1. **Единый источник истины:** один реестр методов/сигнатур/stdlib; checker и codegen — консьюмеры,
   не два движка.
2. **Codegen не резолвит и не выводит типы заново** — потребляет типизированный IR.
3. **C не ловит ошибки типов** (U.7-гарантия).
4. **Никакого хардкода** типов/функций/методов Nova в Rust (§2) — stdlib из `.nv`, не из
   `builtins: HashSet`/списков.
5. Дублирование из §2-таблицы Plan 172.2 §8.4 устранено (inference, type→C, метод-таблицы, width,
   intrinsic-список, sum-layout).
6. На каждом этапе: 0 регрессий против чистого бинаря; spec/доки/error-index обновлены; D312 написан.

## 6. Принципы выполнения

- **Этапами, не «большим взрывом».** Каждый U.x — отдельная ветка/коммит, измеримый, реверсивный.
- **Регресс против чистого бинаря** (stash+rebuild) на каждом шаге — отличать pre-existing от регрессий.
- **Blast-radius до правки** — особенно где перенос проверки в чекер вскроет молчавшие ошибки
  (например U.3 даст реальные диагностики там, где раньше был тихий codegen-dispatch).
- **Не ломать std молча** — мигрировать зависимости в том же шаге.
- `emit_c.rs`/`types/mod.rs` — hot, часто правятся параллельно: координировать регионы.
- **Тесты — в своей папке `detect172/`; `nova_tests/` НЕ чинить.** Фикстуры `nova_tests/`
  помечены «сломаны» (есть pre-existing CC-FAIL, напр. `plan59/f10_deeply_nested_tuple_mangle`,
  `plan65/f10_select_cancel_propagation`) — в рамках 172.1 они **не чинятся**; используются
  ТОЛЬКО как корпус для byte-identical дельты (pass/fail + multiset-.c vs чистый бинарь — их
  pre-existing FAIL должны остаться ровно такими же). Свои pos/neg-тесты каждой задачи кладём в
  `detect172/` (своя папка ветки; neg — отдельный `neg/`-folder-module по
  [`test-conventions.md`](../test-conventions.md) §7/§8), а не в `nova_tests/`.

   несколько коммитов** (по одному на атомарную задачу, `git add` конкретных файлов).
