# Plan 100.5: FFI / external integration — consume-типы через C-границу

> **Создан 2026-05-23 как sub-plan [Plan 100 umbrella](100-linear-must-consume.md).**
> **Статус:** 📋 proposed. **GATED на [100.1](100.1-core-must-consume.md)**.
> **Оценка:** ~4 dev-day.
> **D-блок:** **D163**.

---

## Цель

Закрыть критическую production-дыру: **существующие `external fn`
(C-runtime, libuv, OS API) не интегрированы с consume-семантикой**.
Без этого Plan 18 stdlib (File, Socket, Mutex, Connection) не может
быть consume-типами — а это весь смысл Plan 100.

Текущее (Plan 100.1):
```nova
external fn nova_file_open(path str) -> *File          // no consume tracking
external fn nova_file_close(f *File) -> ()
```

Compiler не знает, что `nova_file_open` возвращает obligation; не
знает, что `nova_file_close` — единственный валидный consume-метод.
Result: file leaks остаются runtime-only checks.

Целевое:
```nova
external consume fn nova_file_open(path str) -> File   // ← consume marker
type File consume

fn File consume @close() -> Result[(), IoErr]          // Nova-side wrapper
    external_runtime: nova_file_close
```

Compiler tracks через FFI границу; consume guarantees end-to-end.

## Сравнение

| Capability | Rust | Kotlin/JNI | Go cgo | TS (Node N-API) | Nova target |
|---|---|---|---|---|---|
| Ownership tracking через FFI | ✅ `unsafe fn` + manual contract | ⚠️ JNI manual | ⚠️ manual | ⚠️ manual | ✅ **`external consume fn` declaration** |
| Auto-close на panic при FFI handle | ✅ Drop (если wrapped в RAII) | ⚠️ manual try-finally | ⚠️ defer | ⚠️ try-finally | ✅ **через 100.4.5 defer/errdefer** |
| Compile-time enforcement FFI consume | ❌ только через wrapper | ❌ | ❌ | ❌ | ✅ **D163 native** |
| Capability tracking (FFI = unsafe) | ⚠️ `unsafe fn` | ⚠️ manual | ⚠️ manual | n/a | ✅ **Plan 16 + D163** |
| Resource handle types (fd, mutex pointer) | ⚠️ via newtype | ⚠️ HANDLE wrapping | ⚠️ unsafe.Pointer | n/a | ✅ **opaque type через D126** |

Nova **превосходит Rust** на одной оси — нет нужды в `unsafe` keyword
для FFI consume (D6 «no unsafe» + capability tracking через Plan 16).

## Дизайн-решения

### D1 — `external consume fn` синтаксис

```nova
external consume fn nova_file_open(path str) -> File
//       ^^^^^^^ — функция возвращает consume-obligation
//                 caller обязан закрыть результат

external fn nova_file_close(consume f File) -> ()
//                          ^^^^^^^ — параметр консьюмится
//                                    (mirror D131 на FFI границе)

external consume fn nova_socket_accept(srv ServerSocket) -> ClientSocket
//                                     ↑ borrow source (read-only через FFI)
```

Marker `consume` на `external fn` declaration говорит compiler'у, что
**C-runtime соблюдает обязательство** (return = caller-owns; param =
callee-consumes). Это **contract** — compiler доверяет; runtime может
defense-in-depth проверять.

### D2 — Opaque type integration (Plan 62.D.bis)

`external type X consume` — opaque consume-type с runtime backing:

```nova
external type File consume                              // opaque, runtime в nova_rt/file.h
external type Mutex consume
external type Socket consume

// Использование (через obvious external fn'ы):
let f = nova_file_open("foo.txt")?                      // File (consume)
defer { f.close()? }                                     // covered (Plan 100.4.5)
```

Combines D126 (opaque types) + D133 (consume) + D163 (FFI consume).

### D3 — Capability требование

`external consume fn` **обязан** объявлять capability (Plan 16 D63
`forbid`/D64 `realtime`-аналог):

```nova
external consume fn nova_file_open(path str) -> File
    needs Fs                                             // capability
```

Это force'ит explicit FFI = capability-confined. Compiler ловит
вызов `nova_file_open` в контексте без `Fs`-capability → error.

Marker `external consume` = «I touch raw OS resource + carry consume
obligation» — combine'ит две responsibility'и в одной декларации.

### D4 — C runtime side — defense-in-depth

C-сторона `nova_file_close(consume f File)` обязана:
- После работы `memset` поля `File*` в zero / NULL (defense per D131
  Plan 73 pattern).
- Validate `f != NULL` на entry (assert); двойной consume → C-side
  assert failure, не silent corruption.

`nv_consume_validate(ptr)` runtime helper — single point для C-side
defensive checks.

### D5 — Cross-fiber FFI safety

FFI-call может суспендиться (через libuv async I/O). Если consume-var
живёт через suspend, fiber migrated на другой worker — runtime should
preserve consume-tracker.

Plan 22/47/49 fiber infra уже handles migration; Plan 100.5 verifies
consume-state preserved (через unit tests).

### D6 — Return-type variance

```nova
external consume fn nova_open() -> Result[File, IoErr]
//                                ^^^^^^^^^^^^^^^^^^^^ — Result wraps consume
//                                                       generic-заразность (100.2 D6)
//                                                       Result сам consume.
```

`Result[File, IoErr]` где File consume → Result сам consume через
generic-заразность. Caller обязан consume Result (через match-Some/Err
с consume в Some-arm; или explicit error-handling).

Это **uniform** с Nova-side generic-propagation — никаких FFI-
специфичных правил.

### D7 — `external consume fn` без consume return / consume param — warning

Если `external consume fn` ничего не consume-возвращает и не consume-
принимает — `consume` маркер избыточен:

```nova
external consume fn nova_get_pid() -> int               // ❌ warning W (D163-vacuous-consume)
//                                    ^^^ — non-consume return, no consume params
```

Force programmer'а быть честным о contract'е.

### D8 — FFI panic / longjmp interaction

Если C-side вызывает `nv_panic()` (или OS longjmp) во время обработки
consume-param — defer/errdefer (Plan 100.4) должны срабатывать.
Verify через runtime tests.

### Ф.2 — Lexer + parser (~0.4 dev-day)

- Existing `external fn` parser → extend `consume` qualifier на
  declaration AND на parameters.
- Parser:
  - `external consume fn name(params) -> R` — return-side consume.
  - `external fn name(consume p T)` — param-side consume.

  warning trigger.

### Ф.3 — Type-checker integration (~0.8 dev-day)

- `external consume fn` registered как consume-method в `LinearityRegistry`.
- Call-site analysis:
  - Return: caller получает Live consume-binding.
  - Param: argument consumed (transition к `Consumed` state).
- Capability check (D3): `external consume fn` без declared cap →
  error E (D163-missing-cap).

### Ф.4 — C runtime defensive helpers (~0.5 dev-day)

- `nv_consume_validate(ptr)` — assert ptr != NULL, zero internal
  fields после use.
- `nova_rt/file.h`, `nova_rt/mutex.h` (для pilot) — pattern: validate
  on entry, zero on exit.
- `nv_panic_during_consume_param` — special panic handler для FFI
  panic mid-consume.

### Ф.5 — Pilot stdlib types — File / Mutex (~0.8 dev-day)

## Границы (deferred)

- **`external consume fn` callbacks** (FFI вызывает Nova-callback,
  который touches consume) — bootstrap не поддерживается; nova-callback
  через FFI с consume — future plan.
- **Multi-handle consume** (один external call returns несколько
  consume handles) — поддерживается через tuple-return + generic-
  заразность.
- **`extern "C++"` interop** — out of scope (Nova FFI bootstrap = C
  ABI only).

## Risks

1. **C-side correctness** — programmer'у нужно ПРАВИЛЬНО implementовать
   defensive helpers. **Митигация:** `nv_consume_validate` обязателен
   для consume-FFI; linter warning если не использован в nova_rt/*.h.
2. **Capability burden** — каждый external consume fn нуждается в cap
   declaration. **Митигация:** sensible defaults (Fs/Net/Sys); generic
   `Unsafe` cap для legacy.
3. **Vacuous marker false-positives** — может быть legitimate
   `external consume fn` где consume — implicit semantic (например,
   global allocator). **Митигация:** warning, не error.
4. **Plan 16 capability infra production-readiness** — already ✅
   (Plan 16 closed Ф.1-Ф.9).
