# Plan 101.2 — Bound integration `fn[T Hashable]` (reuse D72)

> **Sub-plan [Plan 101](101-receiver-generic-prefix.md) (master).**
> Reuse existing [D72](../../spec/decisions/02-types.md#d72) bound
> syntax (`[T Bound]`) в новой позиции — `fn[T Bound]` prefix. Не
> требует новых грамматических конструкций, только integration.
> **Status:** 📋 proposed 2026-05-24, не начат.
> **Eval:** ~0.5 dev-day.
> **GATED на:** [Plan 101.1](101.1-fn-prefix-core.md) ✅.

## 1. Зачем

D72 уже даёт `[T Bound]` syntax для **free fn** и **type-decl**:

```nova
fn dedup[T Hashable](xs []T) -> []T => ...    // free fn with bound
type HashMap[K Hashable, V] { ... }            // type-decl with bound
```

После Plan 101.1 появляется новая позиция — `fn[T] ...` prefix.
Логично — bound syntax работает и здесь:

```nova
fn[T Hashable] []T @dedup() -> []T => ...
fn[T Ord] []T @sort() -> []T => ...
fn[T From[K], K] T @construct(k K) -> T => T.from(k)   // parametric protocol bound
```

Это **integration**, не новая фича. Парсер уже знает синтаксис (D72),
нужно убедиться что hooks работают в новой позиции и codegen
respects bounds.

## 2. Phases

### Ф.0 — Probe

Создать `nova_tests/plan101_2/probe/`:
- `fn_prefix_bound_pre.nv` — `fn[T Hashable] []T @dedup()` — verify
  parser статус ДО Plan 101.1 (likely accepts via D72 parser reuse).

### Ф.1 — Type-checker integration

**Acceptance:** bound в `fn[T Bound]` prefix enforced — call с T который
не satisfies Bound → loud error (как для free fn `dedup[T Hashable]`).

**Actions:**
1. Verify bound-resolution logic для free fn (D72 active) **переиспользуется** в method-context (Plan 101.1 added).
2. Если нет — port logic.
3. Tests:
   - `[1,2,3].dedup()` где `T = int` satisfies `Hashable` (assumed; int — Hashable per primitive protocols) — ✓
   - Custom type `User` без `Hashable` impl — `[users].dedup()` → error.

### Ф.2 — Codegen

**Acceptance:** mono-dispatch учитывает bound — dispatch на bound-method'ы T'а
работает.

Обычно D72 + Plan 48 уже сделали базу. Probe в Ф.0 покажет — если работает «бесплатно», Ф.2 — пустой.

### Ф.3 — Tests positive + negative

#### Positive (6)

- `bound_hashable_dedup.nv` — `fn[T Hashable] []T @dedup() -> []T`.
- `bound_ord_sort.nv` — `fn[T Ord] []T @sort() -> []T`.
- `bound_parametric_from.nv` — `fn[T From[K], K] T @construct(k K) -> T`.
- `bound_in_tuple.nv` — `fn[T Hashable, U] (T, U) @hash_pair() -> u64`.
- `bound_in_bare.nv` — `fn[T Hashable] T @self_hash() -> u64 => @hash()`.
- `bound_with_method_gen.nv` — `fn[T Hashable] []T @group_by[K Hashable](f fn(T) -> K) -> HashMap[K, []T]`.

#### Negative (4)

- `bound_violation_neg.nv` — `.dedup()` на type без Hashable → error.
- `bound_unknown_protocol_neg.nv` — `fn[T NotAProtocol] []T @method` → error.
- `bound_concrete_type_neg.nv` — `fn[T int] []T @method` → error «bound must be protocol-type» (refers Q-representation-bound).
- `bound_with_carrier_conflict_neg.nv` — `fn[K Hashable, V] HashMap[K, V] @method` → `E_DUPLICATE_GENERIC_DECL` (101.1 rule).

#### Acceptance Ф.3

- [ ] 6/6 positive PASS.
- [ ] 4/4 negative → expected error.
