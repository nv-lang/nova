## 1. Контекст

Согласно audit (§4 сравнения с Go/tokio/Kotlin/Node):

| Runtime | Deadline API |
|---------|--------------|
| Go | `context.WithDeadline()` / `context.WithTimeout()` — runtime only |
| Tokio | `tokio::time::timeout(d, future)` — runtime only |
| Kotlin | `withTimeout(d) { … }` — runtime only |
| Node | `AbortSignal.timeout(d)` — runtime only |

**Все 4 — runtime-only:** компилятор не знает что вы вошли в `100ms` бюджет
и не warn'ит если вы вызываете функцию документированную как `worst-case
500ms`. Bug discovered только под load test.

**Nova может быть лучше:** effect-aware deadline propagation —
compile-time check.

```nv
fn slow_io() -> Result[String, IoError] [Time, Blocking, worst_case(500ms)]

#deadline(100ms)
fn handler() {
    let r = slow_io()  // WARN W-DEADLINE: worst-case 500ms exceeds parent 100ms
}
```

Это **не имеет аналога** ни у Go, ни у Tokio, ни у Kotlin, ни у Node.

## 2. Дизайн

### 2.1 Extension к `CancelToken[T]` (Plan 49)

```nv
// std/runtime/cancel.nv

record CancelToken[T] {
    pub deadline: Option[Time]   // Plan 83.11: optional absolute deadline
    pub cancel_after: Option[Duration]  // OR relative-from-bind
    // ... existing fields ...
}

fn deadline_token[T](d: Duration, value: T) -> CancelToken[T]
fn timeout[T](d: Duration, value: T) -> CancelToken[T]  // alias

impl[T] CancelToken[T] {
    fn deadline_remaining(self) -> Option[Duration] { ... }
    fn deadline_exceeded(self) -> bool { ... }
}
```

### 2.2 Supervised syntax extension

```nv
supervised(deadline: 100ms) {
    spawn { ... }
}

supervised(cancel: tok, deadline: deadline) {
    ...
}
```

Compiler auto-derives CancelToken[Timeout] из `deadline:` argument либо
binds existing token.

### 2.3 Effect annotation

Документация worst-case latency через effect attribute:

```nv
#worst_case(500ms)
fn slow_io() -> Result[String, IoError] [Time, Blocking]
```

Это аналог attribute Plan 03.4 effect-surface tooling. Latency tracked
в module metadata.

### 2.4 Compile-time check W-DEADLINE

```nv
#deadline(100ms)
fn handler() {
    let r = slow_io()  // WARN W-DEADLINE
                       // worst-case 500ms exceeds enclosing deadline 100ms
}
```

Algorithm (type-checker pass):
1. Build call-graph for current function.
2. Sum worst-case latencies (annotated `#worst_case` или conservative
   bound из effect tracking).
3. Compare с enclosing deadline annotation.
4. If sum > deadline → emit W-DEADLINE warning.

**Conservative defaults:**
- No `#worst_case` annotation + Blocking effect → ∞ (assume any duration).
- No annotation + pure → 0.
- No annotation + Time effect → "non-trivial" (warn под strict mode).

### 2.5 Runtime API

```nv
use std::time

fn handler_with_remaining() {
    if let Some(rem) = Time.deadline_remaining() {
        // Use remaining budget for IO retry strategy.
    }
}
```

Runtime stores deadline in TLS (per-fiber); `Time.deadline_remaining()`
reads.

### 2.6 Spec D-block

New D-block для deadline propagation rules:
- Nested supervised(deadline:) inherit shorter of parent либо own.
- Deadline expiry triggers CancelToken cancel(Timeout reason) — kind=CANCEL,
  не USER (Plan 49 cascade).
- Implicit handler conversion если scope already bound CancelToken[T] +
  T contains Timeout case.

## 3. Тесты

### Positive (новые):

**`deadline_propagation_test.nv`**:
```nv
use std::time

test "supervised deadline triggers cancel" {
    let start = Time.now()
    let cancelled = supervised(deadline: 50ms) {
        spawn {
            Time.sleep(500ms)  // would exceed
            42
        }
    }
    let elapsed = Time.now() - start
    assert(elapsed < 100ms)  // cancelled before 500ms
    assert(cancelled == CancelReason::Timeout)
}
```

**`deadline_remaining_test.nv`**:
```nv
test "Time.deadline_remaining returns correct value" {
    supervised(deadline: 100ms) {
        Time.sleep(30ms)
        let rem = Time.deadline_remaining()
        // 70ms ± noise
        assert(rem.is_some())
        let r = rem.unwrap()
        assert(r > 50ms && r < 90ms)
    }
}
```

**`deadline_nested_inherit_test.nv`**:
```nv
test "nested supervised inherits shorter deadline" {
    let outer_done = false
    supervised(deadline: 200ms) {
        supervised(deadline: 50ms) {
            // inner has 50ms (shorter)
            spawn {
                Time.sleep(100ms)  // exceeds inner
            }
        }
        outer_done = true  // outer не cancelled
    }
    assert(outer_done == true)
}
```

### Negative (новые):

**`deadline_exceeded_no_user_error.nv`**:
```nv
test "deadline expiry produces CANCEL kind не USER" {
    use std::throw
    try {
        supervised(deadline: 10ms) {
            spawn {
                Time.sleep(100ms)
                throw "should not fire"  // cancel beats throw
            }
        }
        // CANCEL: scope cancelled, наружу ничего не летит
    } catch e {
        assert(false)  // no USER error expected
    }
}
```

**`deadline_nested_exceeds_parent_compile_warn.nv`**:
```nv
// EXPECT_COMPILE_WARN W-DEADLINE

#deadline(100ms)
fn parent() {
    nested_slow()  // WARN: nested_slow worst_case 500ms exceeds parent 100ms
}

#worst_case(500ms)
fn nested_slow() -> int [Time, Blocking] { ... }
```

**`deadline_negative_value_compile_fail.nv`**:
```nv
// EXPECT_COMPILE_ERROR E5005  -- negative deadline

fn invalid() {
    supervised(deadline: -10ms) { }  // ERROR: negative deadline
}
```

**`deadline_zero_immediate_cancel.nv`**:
```nv
test "deadline=0 cancels before any spawn body runs" {
    let executed = false
    supervised(deadline: 0ms) {
        spawn {
            executed = true
        }
    }
    assert(executed == false)  // никогда не запустилось
}
```

## 4. Acceptance

### Core:
- ✅ Syntax `supervised(deadline: Duration) { … }` parsed + lowered.
- ✅ `CancelToken[T]` extended с `deadline` field.
- ✅ Runtime: deadline expiry triggers cancel (CANCEL kind).
- ✅ `Time.deadline_remaining()` API.
- ✅ Nested deadline inheritance (min).
- ✅ Compile-time W-DEADLINE warning lint.
- ✅ Spec D-block для deadline semantics.
- ✅ ≥5 positive tests + ≥4 negative tests.

### Stretch:
- `#worst_case(D)` attribute propagation through call-graph + summing
  inferred worst-case.

## 5. Риски

- **Effect-system complexity** — нужно ввести latency как первоклассный
  effect (или metadata в Time effect). Может конфликтовать с existing
  effect-aware tooling (Plan 03.4).
- **Conservative bounds** — без annotation conservative = ∞ может
  завалить W-DEADLINE на 90% existing code. Mitigation: lint default
  off, opt-in через `#strict_deadlines` module attribute.
- **Time source virtualization** — virtual clock в тестах (Plan 34 §3
  `mut_clock`) должен interact correctly с deadline timing.

## 6. Differentiator framing

Это **Nova-unique**: compile-time deadline awareness не имеют ни Go, ни
Tokio, ни Kotlin, ни Node. Связь с effect system делает Nova единственным
production-grade языком с этой capability.

Близайший конкурент — Loom `StructuredTaskScope.ShutdownOnFailure` +
`joinUntil(deadline)` — но без compile-time check.

## 7. Out-of-scope

- **Probabilistic latency bounds (p99 vs worst-case)** — V2 followup.
- **Runtime adaptive deadline shortening (Go-style ctx.WithDeadline
  inheritance)** — нет необходимости, наш min-inheritance проще.
