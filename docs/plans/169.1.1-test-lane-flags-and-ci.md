## Ф.1 — CLI category-селекторы

### Модель (аддитивные селекторы, OR)

Две независимые оси:
- **Тип** (взаимоисключающий, по `EXPECT_*` маркеру заголовка; positive = нет
  маркера): positive · compile-error · panic · timeout · exit.
- **Скорость** (по суффиксу `*_slow.nv`): fast · slow.

Флаги (каждый = «добавить категорию», результат = объединение):
```
nova test                        → positive ∩ fast  (дефолт)
nova test --positive             → то же явно
nova test --slow                 → все *_slow.nv
nova test --compile-error        → только EXPECT_COMPILE_ERROR
nova test --panic                → только EXPECT_RUNTIME_PANIC
nova test --timeout              → только EXPECT_TIMEOUT
nova test --exit                 → только EXPECT_EXIT
nova test --slow --compile-error → объединение
nova test --full                 → всё (все типы + slow)
```

### Факты о текущем коде (опоры)

- **`SlowLane`** enum: `test_runner.rs:3363` (`Exclude`/`Include`/`Only`).
  Поле `slow_lane: SlowLane` в `TestAllOpts` (`:2634`).
- **Discovery:** `walk_nv_filtered(root, out, lane)` (`:3392`) — фильтрует
  `*_slow.nv` по СУФФИКСУ имени БЕЗ чтения тела.
- **CLI:** подкоманда `Test` (`nova-cli/src/main.rs:414`), флаги `include_slow`
  (`:487`), `slow_only` (`:490`). `cmd_test` (`:4300`), маппинг в `slow_lane`
  (`:4461-4463`).
- **EXPECT_* маркеры** парсятся в `test_runner.rs` (`EXPECT_RUNTIME_PANIC`
  ~`:210`, плюс `parse_expect_*` для COMPILE_ERROR/TIMEOUT/EXIT) — логика чтения
  заголовка уже есть; переиспользовать для discovery-фильтра.

### Реализация

1. **Категориальная модель.** Заменить одиночный `slow_lane: SlowLane` на набор
   выбранных категорий. Ввести:
   ```rust
   struct TestSelection {
       types: TypeSet,   // битсет/HashSet: Positive|CompileError|Panic|Timeout|Exit
       include_slow: bool,
   }
   ```
   Дефолт = `{Positive}`, `include_slow=false`. `--full` = все типы + slow.
2. **Discovery читает маркер.** `walk_nv_filtered` (или новый
   `walk_nv_selected`): для каждого файла определить ТИП по `EXPECT_*` маркеру
   заголовка (читать первые ~30 строк, как уже делает `parse_expect_*`) +
   скорость по суффиксу `_slow`. Включать файл если `type ∈ selection.types`
   (с учётом slow). **Фильтр по МАРКЕРУ, не по папке** — 128 compile-error +
   27 panic лежат ВНЕ `neg/` (перепись 169.1).
3. **CLI:** добавить флаги `--positive`/`--compile-error`/`--panic`/`--timeout`/
   `--exit`/`--full` в подкоманду `Test` рядом с `--slow` (переименовать
   `--include-slow`→`--slow` для единообразия аддитивной модели; сохранить
   `--include-slow` как алиас для backward-compat). Собрать `TestSelection`,
   пробросить в `TestAllOpts`.
4. Без флагов → `{Positive} ∩ fast`. Несколько флагов → объединение.

### Тесты Ф.1 (вручную проверить вывод)

- `nova test --compile-error nova_tests` — только compile-error (число ≈674).
- `nova test --panic nova_tests` — только panic (≈114).

- `nova test nova_tests/plan107` (дефолт) НЕ включает neg/panic.

## Ф.2 — CI авто-регресс

Файл `.github/workflows/nova-test-regression.yml`. Образец существующих
workflow — `.github/workflows/contracts-z3.yml` (сборка nova-cli + вызов
`nova test`).

- **PR / push** (на каждый коммит, быстро):
  - собрать nova-cli (release, clang toolchain);
  - `nova test nova_tests` (дефолт = positive-fast);
  - `nova test std`;
  - падение → red CI.
- **Nightly** (`on: schedule: cron: '0 3 * * *'`):
  - `nova test --full nova_tests` + `nova test std`;
  - `--results-file` → upload artifact с отчётом о падениях.
- Job-timeout: GitHub до 6ч; полный прогон ~час — ОК. При риске — дробить
  `--full` по группам категорий (`--compile-error`, затем `--panic` и т.д.).
- Toolchain/env: clang; если нужно — `NOVA_GC_LIB_DIR`/`INCLUDE_DIR` (см. как в
  contracts-z3.yml / project-worktree-nova-test-setup).

Зависимость: Ф.2 использует `--full` из Ф.1 → Ф.1 первой.
