# Plan 83.2: Флип дефолта M:N — вкл по умолчанию

> **Создан:** 2026-05-22 (декомпозиция [Plan 83](83-mn-default-on-gomaxprocs.md)).
> **Статус:** 🟡 **Ф.0 ✅ + Ф.1 infrastructure ✅, full flip отложен.**
> Worktree `nova-p83`, ветка `plan-83-2`. **GATED-явно был зелёный**
> (Ф.0 gate, 83.2-artifacts/f0-readiness.md), но реальная попытка
> активации (auto-arm в main()-codegen) вскрыла 9+ pre-existing M:N
> багов в supervised-drain / D93 sleep-wake / per-fiber handlers,
> которые до 83.2 проявлялись только при explicit `runtime.init` и
> теперь требуют отдельных таргетных фиксов (каждый ~1-2 dev-day).
> Инфраструктура (`nova_runtime_auto_arm` + auto-arm в spawn-путях)
> подготовлена и закоммичена; codegen-emit `main()` НЕ вызывает
> auto-arm до закрытия M:N багов. Полный clang `nova test`: **1071/0/56**.
> **Приоритет:** P2.
> **Жёсткие зависимости:** [Plan 82](82-windows-fiber-arena.md) ✅ +
> [83.1](83.1-mn-infrastructure.md) ✅ + GC-safety под multi-worker.
> **Родитель:** [Plan 83](83-mn-default-on-gomaxprocs.md).

## 1. Scope

Собственно изменение дефолта: M:N **вкл** по умолчанию для
compiled-бинарей, worker'ов = resolved `maxprocs` ([83.1](83.1-mn-infrastructure.md)).
Плюс миграция кодовой базы, spec и измеренный speedup-бенчмарк. Вся
конфигурационная инфраструктура — уже в 83.1; здесь — маленькое, но
рисковое изменение поверх неё.

## 2. Почему GATED

Флип раздаёт M:N **всем** compiled-программам сразу. До него обязательно
(детально — gate Ф.0):
- **Plan 82 ✅** — иначе каждая Windows-программа побежит параллельно на
  56 КБ calloc-стеках без overflow-detection и без GC-интеграции
  fiber-стеков (Plan 82 §1.5) → silent corruption под конкуренцией.
- **GC-safety под multi-worker ✅** — stop-the-world корректен при N
  worker'ах. Сейчас безопасность calloc-пути на Windows опирается на
  «single-thread cooperative — GC не запускается между yield/resume»
  (`fibers.h`); под реальным M:N это допущение исчезает.
- **83.1 ✅** — worker-resolution, `NOVA_MAXPROCS`, lazy-spawn,
  auto-shutdown, thread-budget для `nova test` готовы.
- **[83.3](83.3-blocking-effect-threadpool.md)** — желательно ✅; если
  нет — флип допустим с **documented-ограничением**: под genuinely-
  blocking нагрузкой (FFI) параллелизм проседает ниже `NOVA_MAXPROCS`.

## 3. Фазы

### Ф.0 — Readiness gate (decision point, ~1-2 дня)

Аудит зрелости M:N: что реально работает; известные race'ы; статус
Windows (Plan 82); GC под multi-worker; статус [83.3](83.3-blocking-effect-threadpool.md);
soak/stress (10⁶ spawn, work-stealing migration — no race/leak/deadlock).
**Выход:** документ с go/no-go и точными критериями. Без зелёного gate
Ф.1 не стартует — 83.2 откладывается, 83.1/83.3 остаются ценны.

### Ф.1 — Флип дефолта — 🟡 ИНФРАСТРУКТУРА ГОТОВА (commit b72ce59b475)

> Эта фаза оказалась НЕ "одним обозримым изменением": попытка
> активировать auto-arm в main() через `nova_runtime_auto_arm()`
> вскрыла серию pre-existing M:N багов в supervised-drain layer
> (см. acceptance §4). 9+ concurrency-тестов регрессируют (cancel,
> parallel_for, sleep_bench D93 protocol, per_fiber_handlers,
> fiber_arena_stats main-thread query, time_handler, etc.). Каждый
> — самостоятельный 1-2-day fix в runtime/codegen, накопительно ~2
> dev-week. До 83.2 эти баги были невидимы (все эти сценарии шли
> single-thread cooperative-fallback codegen-веткой).
>
> **Сделано:**
> - `nova_runtime_auto_arm()` (runtime.h + runtime.c) — public idempotent
>   API, эквивалент `runtime.init(0)` без обязательности явного вызова.
> - `_auto_arm_if_needed()` встроен в `nova_runtime_spawn_global` и
>   `nova_runtime_spawn_into` (commit 0af6e6ba482).
>
> **НЕ сделано** (открытый followup):
> - codegen-emit `nova_runtime_auto_arm()` в начале main() (одна строка
>   в emit_main_wrapper — закомментирована с указанием на фоллапы).
> - Фиксы 9+ M:N багов в supervised-drain, D93, handler-scoping.

Одно обозримое ревью-able изменение: дефолт = M:N вкл, worker'ов =
resolved `maxprocs`. Авто-материализация пула через lazy-spawn 83.1 Ф.4
(hello-world по-прежнему 0 потоков). `nova run` (интерпретатор) —
не затронут (scope-граница Plan 83 §1).

### Ф.2 — Миграция + escape hatch — ⏸ ОТЛОЖЕНО (зависит от Ф.1 full-flip)

> Сделать без Ф.1 full-flip — невозможно: sweep и escape-hatch имеют
> смысл только если M:N по умолчанию реально активен. Откладывается до
> закрытия supervised-drain M:N багов.

- Sweep `runtime.init(n)` в `std/`/`nova_tests/`/`examples/` — теперь
  опциональны; убрать обязательные вызовы.
- Escape hatch: `NOVA_MAXPROCS=1` — без параллелизма (Plan 83 §3 П5 —
  честная формулировка: убирает параллелизм, **не** даёт
  scheduling-детерминизм). Задокументировать.
- Fallout: тесты, молча полагавшиеся на single-threaded порядок
  исполнения → починить либо явно пометить `NOVA_MAXPROCS=1`.

### Ф.3 — spec + docs + benchmark — ⏸ ОТЛОЖЕНО (зависит от Ф.1 full-flip)

> Spec-D-block «M:N как дефолтная модель» нельзя написать честно пока
> M:N не дефолт по факту. Speedup-бенчмарк требует работающего default-on.
> Откладывается до закрытия supervised-drain M:N багов (Ф.1).

- Spec: M:N — дефолтная модель исполнения для compiled-бинарей
  (`nova run` — однопоточный); D-block по `NOVA_MAXPROCS` + порядок
  разрешения; правило «worker не делает блокирующую работу инлайн»
  (Plan 83 §3 П2 / [83.3](83.3-blocking-effect-threadpool.md)).
- Benchmark: измеренный parallel speedup на N ядрах. `bench/micro/`
  сейчас не имеет параллельного workload'а — добавить (CPU-bound
  fan-out, замер speedup vs `NOVA_MAXPROCS=1`).
- `docs/concurrency*`, README.

## 4. Acceptance

- Compiled-программа без единого `runtime.*` вызова использует все CPU
  при fiber-нагрузке (паритет Go/tokio).
- Hello-world (без `spawn`) — 0 worker-потоков.
- `NOVA_MAXPROCS=1` → один worker, без параллелизма; escape hatch
  работает и задокументирован.
- `nova test` после флипа **не регрессирует** по времени (83.1 Ф.5).
- `nova run` остаётся однопоточным.
- Измеренный speedup на multi-core workload'е (Ф.3).
- Gate Ф.0 зелёный; 0 регрессий (Windows + Linux).

## 5. Риск

- **Рисковый по построению** — раздаёт M:N всем. Снижается gate'ом Ф.0
  (front-load: дёшево сказать no-go) и тем, что вся инфраструктура уже
  обкатана в 83.1.
- Если gate показывает неготовность — Ф.1 не делается; 83.2 ждёт; ценность
  83.1/83.3 не теряется.
- Дефолт-параллелизм делает пользовательский код race-aware (как Go) —
  сознательная языковая позиция, фиксируется в spec (Ф.3).
