## 1. Bugs (один комплексный фикс, ОДИН review-unit)

A2, A3, B2 имеют **общую причину** — handler/scope-state не путешествует
с fiber'ом между main и worker'ами. Их фиксы пересекаются в
`nova_supervised_step`, ownership-протоколе resume и handler-snapshot
data layout. Делать в одном PR.

### A2 — Supervised drain double-resume

**Симптом** (`supervised_errors`, `supervised_cancel_stress_test`):
```
nova: fiber stack overflow in slot 0 (access violation in fiber arena)
```

**Причина:** main thread в `nova_supervised_step` пытается
`mco_resume(fiber)` для fiber'а, которого уже стащил worker через
work-stealing. mco_coro state становится `MCO_RUNNING` на worker'е;
main resume даёт re-entrance → corrupt state → stack overflow at
some `__chkstk` probe.

### A3 — Per-fiber handlers под M:N

**Симптом** (`per_fiber_handlers`):
```
FAIL: inner with в spawn перекрывает outer для своего fiber — outer_seen == 111
```

**Причина:** handler-scope-snapshot save/restore в `nova_supervised_step`
работает через TLS-массив `_nova_handler_*`. Когда worker запускает
spawned fiber, его handler-snapshot должен быть restored ДО mco_resume;
но это делает main-thread step, который не вызывает worker'а.

### B2 — `time_handler` storage swap

**Симптом** (`time_handler`): Time-effect handler не виден из
spawn'ed fiber.

**Причина:** аналог A3 для конкретно `_nova_handler_Time` TLS-storage.

## 2. Root cause (общий)

Сейчас supervised-execution предполагает:
1. Main thread (или ОДИН worker) — единственный resume'ер fiber'ов
   scope'а.
2. Handler-storage в TLS, swap'ается main thread'ом перед каждым
   `mco_resume` через `nova_supervised_step`.

Под M:N (work-stealing активен, флип в дефолт):
1. **Нарушено:** worker steal'ит fiber, main и worker конкурируют за
   resume. Drain ownership не сформулирован.
2. **Нарушено:** TLS handler-storage main thread'а не виден worker'у;
   воркер resume'ит fiber без правильного handler-context.

## 3. Подход

### 3.1. Ownership-протокол `nova_supervised_step`

Drain main thread'а должен **проверять**:
- `mco_status(co) == MCO_SUSPENDED` — fiber не активно бежит на
  worker'е.
- `fiber_owner_worker_id == -1` — fiber not currently owned by a
  worker (`-1` = no owner, main can resume).

Если worker украл fiber через `nova_deque_steal`, он атомарно
выставляет `fiber_owner_worker_id = w->id` ДО `mco_resume`. После
fiber'а завершения / yield'а — `= -1` обратно.

Main `nova_supervised_step` пропускает fiber'ы у которых
`owner != -1`. Cross-worker fiber'ы main ждёт через `pending_remote`
без попытки локального resume.

### 3.2. Handler-snapshot в `mco_coro` user_data

Сейчас handler-snapshot хранится в TLS-массиве `_nova_handler_*`.
Поднять snapshot в **per-fiber data**:

- `NovaSpawnCtxBase` (`fibers.h`) расширить полем `handler_snapshot`
  — массив pointer'ов handler-storage.
- `nova_supervised_step` сохраняет snapshot **в** fiber's ctx перед
  выходом из fiber'а (yield/end).
- Воркер перед `mco_resume(fiber)` восстанавливает snapshot **из**
  fiber's ctx в TLS.
- TLS остаётся «scratchpad» текущего активного fiber'а — он же main
  scope's effective context.

Refactor затрагивает:
- `nova_register_effect_storage` (regis pointer-pointers → массив).
- `nova_supervised_step` save/restore (`fibers.h:1135+`).
- `nova_runtime.c::_worker_main` (snapshot restore перед resume).

## 4. План реализации

### Ф.1 — A2 ownership (~2 dev-day)

1. Добавить `volatile int fiber_owner_worker_id` в
   `NovaSpawnCtxBase` (default `-1`).
2. `nova_deque_steal` (`deque.h`) — atomic CAS на `owner_id`: если
   уже `!= -1`, steal fails (NULL).
3. `_worker_main` (`runtime.c`) — после resume set `= -1`.
4. `nova_supervised_step` — skip fibers с `owner_id != -1`.
5. Targeted: `supervised_errors` + `supervised_cancel_stress_test`
   под explicit `runtime.init(2)`.

### Ф.2 — A3 + B2 handler-storage migration (~2-3 dev-day)

1. Расширить `NovaSpawnCtxBase` полем `void** handler_snapshot[K]`
   (K — число зарегистрированных effect-storage).
2. `nova_register_effect_storage` — заполняет таблицу с известным
   индексом.
3. В `nova_supervised_step` (~`fibers.h:1180+`) save/restore из TLS
   ↔ fiber's snapshot ПЕРЕД/ПОСЛЕ `mco_resume`.
4. `_worker_main` — снимает текущий fiber's snapshot в TLS перед
   resume, восстанавливает после.
5. Targeted: `per_fiber_handlers` + `time_handler` под explicit
   `runtime.init(2)`.

## 5. Acceptance

- `supervised_errors` + `supervised_cancel_stress_test` PASS под M:N.
- `per_fiber_handlers` + `time_handler` PASS под M:N.
- Никаких `fiber stack overflow in slot 0` access-violation'ов.
- 0 регрессий single-thread пути (без `runtime.init`).
- 0 регрессий полного `nova test` clang.

## 6. Риск

- **Handler-storage refactor** — затрагивает фундаментальный механизм
  algebraic effects; ошибка регрессирует handler-aware пользовательский
  код. Mitigation: incremental approach — сначала ownership (Ф.1)
  изолированно (его acceptance отдельный); затем handler-storage с
  detailed targeted test'ом для каждого effect.
- Atomic `owner_id` CAS под high steal rate может стать contention
  hot-path. Mitigation: профиль `mn_runtime_cpu_workload` ДО и ПОСЛЕ.
