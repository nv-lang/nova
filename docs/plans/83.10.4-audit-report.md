## Root cause summary

### RCA-1: TLS handler snapshot race (pre-preamble path)

**Локация:** `runtime.c` lines 694-700 (`_worker_main`) и 1452-1461 (`_worker_run_one_fiber`).

**Суть:** При первом `mco_resume` fiber'а на воркере (до preamble):
- `base->_nova_worker_slot == -1` → `base->_nova_fiber_scope == NULL`
- Попадаем в `else if (base)` ветку
- Восстанавливаются `_nova_fail_top` и `_nova_interrupt_top`
- **НО НЕ восстанавливается** `nova_effect_snapshot_restore(base->_nova_init_snapshot)`
- Fiber видит handler state воркера (предыдущего fiber'а) вместо parent's

**Fix:** Добавить `if (base->_nova_init_snapshot) { nova_effect_snapshot_restore(base->_nova_init_snapshot); }` в обе `else if (base)` ветки.

### RCA-2: longjmp cross-mco boundary (I-3 violation)

**Локация:** Fiber throw path — `longjmp` к handler frame на другом потоке.

**Суть:** `with Fail = handler { ... }` устанавливает `jmp_buf` через `setjmp`
на main-thread. Под armed M:N fiber бросает throw на worker-thread, пытаясь
`longjmp` к main-thread's `jmp_buf` → undefined behavior (I-3 нарушение).

**Fix:** Worker обнаруживает USER_THROW → сигналит main-thread через
`nova_runtime_report_throw_to_scope` → main делает локальный `longjmp`.

### RCA-3: Cancel propagation race под armed M:N

**Локация:** `nova_sched_cancel_all_pending`, cancel token state visibility.

**Суть:** `tok.cancel()` из main-thread устанавливает state, но worker-thread
fibers не видят обновление достаточно быстро из-за missing acquire barrier или
неправильного wake dispatch.

**Fix:** Определяется в Ф.5 после детального RCA.

### RCA-4: Nested supervised TIMEOUT (deep_spawn)

**Локация:** `nova_runtime_worker_pump_scope`, nested fiber completions.

**Суть:** Fiber X на worker создаёт inner supervised scope и ждёт Y1/Y2.
Y1/Y2 завершаются, но pending_remote не достигает 0 или signal_main не
активирует pump_scope loop exit → X ждёт вечно → TIMEOUT.

**Fix:** Определяется в Ф.6 после детального RCA.
