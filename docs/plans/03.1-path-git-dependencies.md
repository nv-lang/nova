## 1. Scope и почему «можно сейчас»

Первый исполнимый срез экосистемы пакетов: **внешние зависимости через
`path` и `git`** в bootstrap-Rust `nova` CLI + воспроизводимый
`nova.lock` для них. **Без** registry, **без** version-ranges, **без**
self-hosting.

Почему near-term (вопреки редакции 1 Plan 03, гейтившей всё на
self-host): `path`/`git`-резолв не требует ни registry-инфраструктуры,
ни version-resolution (источник пинится точно — путём или commit'ом),
ни Nova-компилятора (bootstrap-Rust справляется). Self-hosting — про
*язык инструмента*, не про *функциональность* (Plan 03 §7).

**Разблокирует:** multi-package разработку до v2.0 — напр. сайт
[Plan 64](64-nova-lang-website.md) сможет зависеть от `std` как от
пакета; прикладные проекты — делиться кодом через `git`.

**НЕ входит** (следующие под-планы Plan 03): version-ranges + PubGrub
(03.2), registry (03.3), `nova info`/effect-diff/`audit` + capability-
confined deps (03.4). 03.1 **резолюцию зависимостей доводит до конца**
(без упрощений) — но *ограничение* зависимостей и registry — отдельные
концерны.

## 2. Текущее состояние

- ✅ Workspace (`nova.toml` members), резолв модулей **внутри** дерева
  (Plan 81), `resolve_imports_inline` (`imports.rs`).
- ❌ Внешних зависимостей нет: `[dependencies]` не парсится, `nova.lock`
  не генерируется, межпакетный резолв за пределы workspace отсутствует.

## 3. Дизайн

### 3.1. `nova.toml` `[dependencies]`

```toml
[dependencies]
foo = { path = "../foo" }
bar = { git = "https://example.org/bar.nv", rev = "a1b2c3d" }
baz = { git = "https://example.org/baz.nv", tag = "v0.3.0" }
qux = { git = "https://example.org/qux.nv", branch = "main" }
```

- `path` — локальная директория (другой пакет на диске).
- `git` — репозиторий; пин через `rev` / `tag` / `branch`. Даже при
  `branch` **lockfile фиксирует точный commit** — ветка не «уедет».
- **Forward-compat поля резервируются** (для 03.4, без enforcement в
  03.1): `forbid = ["Net","Fs"]` (capability-confined dep, Plan 03 §4).
  Парсятся и сохраняются, но в 03.1 не enforce'ятся — чтобы 03.4 не
  ломал формат.

### 3.2. Межпакетный резолв

Первый сегмент import-пути — имя пакета. `import foo.mod.{X}` → `foo`
ищется среди **объявленных зависимостей** (`[dependencies]`); резолв — в
исходном дереве зависимости теми же правилами D29/Plan 81.

**Дизайн-решение: импорт между пакетами требует явной `[dependencies]`-
записи — даже для членов одного workspace.** Workspace-membership сам по
себе **не** делает пакет импортируемым (модель Cargo). Причины: explicit
dependency-граф (AI-first «видно откуда»); член workspace, от которого
ты не зависишь, не должен засорять твой namespace; единая точка для
версии / `forbid` / effect-surface. Текущий резолвер неявно находит
модули через repo-root — это looseness, 03.1 её **ужесточает**.

**`std` — исключение:** остаётся неявным standard-library-пакетом,
доступным без `[dependencies]`-записи (как Rust `std`). Машинерия 03.1 —
для **не-`std`** пакетов.

- Коллизия имени dependency и workspace-member → **config error**.
- Импорт из зависимости — всегда **полным путём** (относительные импорты
  [Plan 84](84-relative-imports.md) package-scoped, границу пакета не
  пересекают; 03.1 и 84 композируются чисто).
- `internal/` (rule H) зависимости — приватная граница соблюдается:
  потребитель не видит `dep.internal.*`.

### 3.3. `nova.lock`

Формат — D78. 03.1-подмножество (без registry/version-ranges):
- `git`-deps: имя + URL + **resolved commit hash** + **sha256 дерева
  исходников**. Воспроизводимо и tamper-evident (ветка не подменится).
- `path`-deps: имя + путь, **без hash-пина** — path-deps локальны и
  мутабельны, всегда берётся текущее содержимое (как в Cargo).
- Поле под effect-surface зависимости — **зарезервировано** (заполняет
  03.4), формат сразу с ним.

### 3.4. Кэш git

Fetched git-репозитории кэшируются (`~/.nova/git/<hash>/` либо
`target/`), пин по commit'у. Повторная сборка — без сети, если lock и
кэш на месте. Offline-режим: build из lock+cache без обращения к сети.

## 4. Фазы

### Ф.0 — Свериться/расширить D78 schema (~0.5 дня)

Проверить, что D78 покрывает `[dependencies]` с `path`/`git` и формат
`nova.lock`; дописать недостающее (резерв полей `forbid` и effect-
surface — §3.1/§3.3). Spec-only.

### Ф.1 — Парсинг `nova.toml` `[dependencies]` (~1 день)

`path`/`git` (+ `rev`/`tag`/`branch`); резерв `forbid`-поля. Валидация:
взаимоисключение `path`/`git`, требование пина для `git`, понятные
ошибки.

### Ф.2 — Git fetch + кэш (~1-2 дня)

Клон/fetch git-репо в локальный кэш, checkout пина, разрешение
`branch`/`tag` → commit. Offline-режим (build из кэша). Ошибки сети —
диагностируемы.

### Ф.3 — Межпакетный резолв ✅ (сделано)

Резолвер (`imports.rs`): первый сегмент import → объявленная
`[dependencies]`-зависимость (+ неявный `std`); candidate-root = source
root этой зависимости.

**Сделано:**
- `lookup_dependency(importer, dep_name) -> DepLookup` — ищет первый
  сегмент среди `[dependencies]` пакета импортирующего файла; для
  `path`-deps возвращает source root дерева зависимости. `git`/registry/
  invalid/missing-path/no-manifest/name-mismatch — диагностируемые
  ошибки. `git`-резолв — заглушка с честной ошибкой (включится в Ф.2).
- `dep_root` проброшен `resolve_one` → `resolve_module_paths`: при
  импорте из зависимости первый сегмент (имя пакета) отбрасывается,
  единственный root = source root зависимости.
- **Ужесточение repo-root looseness (§3.2):** non-`std`, non-relative,
  non-dep импорт, резолвящийся в файл ДРУГОГО пакета (иной
  `package_root_of`), отвергается — межпакетное обязано идти через
  `[dependencies]`.
- **Cross-package cycle detection** (A→B→A): переиспользует
  module-name-keyed `in_progress`-механизм — он cross-package по
  построению, отдельного кода не потребовал.
- **Rule H `internal/`** через границу пакета: `find_internal_owner_dir`
  работает по filesystem-containment — `dep.internal.*` снаружи
  недоступен без дополнительного кода.
- Валидация `[dependencies]`: имя `std` зарезервировано, дубли запрещены.
- `verify_case` (Plan 81 Ф.4) сделан dep-aware: логический первый
  сегмент (`std` / имя пакета) не сверяется с именем директории на диске.
- 11 фикстур `nova_tests/plan03_1/` (см. Ф.6); **983 PASS / 0 FAIL**.

**Отложено в Ф.2:** фактический git-резолв (сейчас — честная ошибка).
**Снято с 03.1:** коллизия имени dependency vs workspace-member — под
explicit-dependency-моделью (§3.2: членство ≠ импортируемость) коллизии
нет, пункт vestigial.

### Ф.4 — `nova.lock` (~1-2 дня)

Генерация/чтение `nova.lock`: git commit + sha256 дерева; path-deps
unpinned. Воспроизводимость: при наличии lock+cache сборка детерминирована
и offline. `nova.lock` коммитится в репозиторий (как `Cargo.lock` для
бинарей).

### Ф.5 — Минимальный CLI: `nova add` / `nova update` (~1-2 дня)

- `nova add foo --path ../foo` / `nova add foo --git URL --tag v1`
  — правит `[dependencies]` + обновляет lock.
- `nova update [foo]` — пере-резолвит git-пины, обновляет lock.
- (Полный `publish`/`search`/`info` — registry, под-план 03.4.)

### Ф.6 — Тесты + spec + docs (~1-2 дня)

- Фикстуры: multi-package проект (`path`-dep + локальный «git»-dep
  через `file://` или локальный bare-репо); воспроизводимая сборка из
  lock; cross-package cycle → error; коллизия имён → error; `internal/`
  зависимости недоступна снаружи.
- spec D78 sync; `docs/nova-cli*`, README.

## 5. Acceptance

- Проект с `path`- и `git`-зависимостями собирается; импорт имён из
  зависимости работает полным путём.
- `nova.lock` генерируется; повторная сборка воспроизводима и offline
  (из lock+cache, без сети).
- `git`-dep на `branch` — lock пинит commit; ветка не «уезжает».
- Cross-package cycle / коллизия имён → детерминированная ошибка.
- `internal/` зависимости снаружи недоступна.
- Формат `nova.toml`/`nova.lock` сразу содержит зарезервированные поля
  под 03.4 (effect-surface, `forbid`) — без будущего format-break.
- 0 регрессий в одиночном workspace (фича аддитивна).

## 6. Честная рамка

- 03.1 **полностью** закрывает резолюцию `path`/`git`-зависимостей — это
  не «упрощённая версия пакетного менеджера», а законченный срез
  (resolution + lockfile + reproducibility).
- Версионные диапазоны (`^1.2`) и registry — **сознательно** не здесь:
  для `path`/`git` источник пинится точно, SAT-resolver не нужен; он —
  03.2. Это декомпозиция, не упрощение.
- `git`-deps тянут произвольный код — но lock пинит commit + sha256 →
  tamper-evident. Полная supply-chain модель (подпись, transparency log,
  `nova audit`, effect-diff) — 03.3/03.4.
