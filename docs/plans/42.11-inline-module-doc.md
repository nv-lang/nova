# Plan 42.11: Inline module-level doc syntax

> **Создан 2026-05-14** — stub. Дополняет Plan 45 (nova doc) inline-доком.
>
> **Приоритет:** MEDIUM. Дешёвая фича с высоким impact для AI-first
> workflow.

---

## Зачем

`nova doc <module>` (Plan 45) собирает API из exports — отличный CLI
tool. Но **в самом файле** нет места для «что этот модуль делает» —
LLM/human при чтении peer видит только imports + items, без context.

Сравнение:
- **Rust:** `//!` inner doc-comment в начале `lib.rs` — module-level
  doc inline, IDE hover показывает.
- **Go:** `doc.go` отдельный файл (правило G нашей спеки — отвергнуто
  как failure mode).
- **TS:** JSDoc `/** @module */` блок.

**Наш отказ от `overview.nv`** (правило G) был правильным — отдельный
файл забывают обновлять. Но **inline doc-comment в любом peer** ≠
overview file. Doc живёт **рядом с кодом**, не дублирует API.

---

## Архитектурное решение

Один из двух синтаксисов:

### Вариант A: Rust `//!`-style

```nova
//! Модуль `admin` — управление пользователями и аудит.
//!
//! Поведение:
//! - Все операции требуют `Auth` capability.
//! - `create_user` логирует в audit.

module src.admin
```

### Вариант B: `#doc` attribute

```nova
#doc "Модуль admin — управление пользователями и аудит."
module src.admin
```

**Pros A:** comment-style, IDE-friendly, multi-line natural.
**Pros B:** unified attribute syntax, parseable как regular attribute.

**Вероятно A** — multi-line doc более natural. Привязан к `module`
declaration в том же peer.

**Multi-peer merging:** если несколько peers имеют `//!` — concat в
alphabetical filename order (deterministic). Один peer = один блок.
