# Plan 162 — Rust-модель резолва: ленивый резолв модулей + «методы едут с типом» + снятие Ф.4-хардкода

## Проблема (замерено/подтверждено)

1. **Жадный рекурсивный резолв импортов → циклы падают стеком.** Сейчас `import` = inline-merge: «встретил `import X` → загрузи и вмерджи X прямо сейчас», рекурсивно. Любой цикл в графе модулей → бесконечная рекурсия. Из-за этого char-методы нельзя положить в prelude (цикл `prelude → std.unicode → std.collections → prelude` → stack overflow), что и породило хак ниже.
2. **No-import char-методы (Plan 159 Ф.4) сделаны хардкодом в компиляторе.** В `imports.rs` зашиты:
   - `CHAR_UNICODE_METHOD_SELECTORS` — список из 10 имён (`is_alphabetic`, …);
   - имя модуля `"std"/"unicode"`;
   - детектор `needs_unicode_injection` + блок авто-инъекции `import std.unicode`.
   **Никто не следит за актуальностью** (нет теста синхронизации списка с `std/unicode/category.nv`); **непоследовательно** (работает только форма `значение.метод()` и только для этих 10 имён; новый метод молча не заработает; голый `is_alphabetic(0x41)` — нет); **источник истины раздвоен** (методы в `.nv` + их имена в Rust).
3. **Методы доступны только при импорте модуля метода** (следствие жадной модели), а не «по типу значения» — в отличие от Rust/Swift, где inherent-метод приезжает вместе с типом.

## Цель (Rust-модель)

- **Резолвер: «объяви-всё → резолви-тела-лениво».** Сначала собрать **сигнатуры** всех модулей в сборке (declaration collection), и только потом **лениво** резолвить/тайп-чекать тела при обращении. Тогда цикл модулей — просто взаимная ссылка в готовой таблице, **безвреден** (как внутри одного крейта в Rust).
- **«Методы едут с типом».** Inherent `@`-метод (объявлен в том же модуле, что и его тип-получатель) **вызывается без импорта** модуля метода — единообразно для **std И пользовательских** типов. Источник истины — `.nv`-объявление, **никакого хардкод-списка**.
- **Char базовые методы → в prelude рядом с `char`** → они становятся inherent → no-import «бесплатно» через общий механизм; тяжёлые Unicode-таблицы остаются в `std.unicode`, **DCE (Plan 159) их срезает**, если не вызваны.
- **Снять весь Ф.4-хардкод** (`CHAR_UNICODE_METHOD_SELECTORS`, `needs_unicode_injection`, блок инъекции, и `@method:`-тег в `collect_expr`, если он больше нигде не нужен).
- **Extension-методы (метод на ЧУЖОМ типе из другого модуля)** — политика как в Rust: inherent (тот же модуль, что тип) = без import; extension (другой модуль) = **требует импорта** этого модуля (защита от «спуки-методов»). Финал — `Q-module-resolution-model` (рекомендация: Rust-strict, НЕ Swift «extension повсюду»).

## Дизайн / Алгоритм

Фазовое разделение резолва (модель крейта Rust):

1. **Collect (сигнатуры).** Распарсить все модули, **транзитивно входящие в сборку** (пакет + deps + встроенный std), и собрать **две** таблицы:
   - `name → decl` для **имён** (типы, свободные функции) — видимость по import/prelude-скоупу (это и есть «что требует import»);
   - `type → methods` — **глобальная** карта `@`-методов по имени типа-получателя (это «что НЕ требует import» — резолв по типу значения).
   «В сборке» = модуль, на который есть транзитивная ссылка (его кто-то импортнул, чтобы НАЗВАТЬ тип/функцию). Тип не может появиться у тебя, не будучи в сборке → его методы гарантированно в `type → methods`. **Не** парсим всю файловую систему — только достижимые модули; std/prelude — всегда (встроены).
2. **Name resolution (имена).** Bare-имя (`T`, свободная `foo`) резолвится по import/prelude-скоупу (как сейчас). Это поверхность, где import **нужен**.
3. **Method resolution (методы).** `x.foo()` → берём тип `x` = `T` → ищем `foo` в `type→methods[T]` **без оглядки на import**. Inherent (метод из модуля типа) — всегда; extension (из другого модуля) — гейтится import'ом по политике.
4. **Lazy body typecheck.** Тела функций/методов резолвятся/тайп-чекаются по обращению, поверх готовых таблиц → цикл не вызывает рекурсивной загрузки.
5. **Codegen + DCE (Plan 159).** Без изменений семантики: эмитим достижимое от roots; таблицы (`*_DATA`) срезаются, если читающий метод недостижим.

## Фазы

- **Ф.1 — Раскол резолвера на collect-signatures / resolve-bodies.** Рефакторинг `resolve_imports_inline*`: отделить сбор сигнатур всех in-build модулей от резолва тел. Foundational; без смены пользовательской семантики (existing DAG-код компилируется как раньше).
- **Ф.2 — Поддержка циклов модулей.** Снять stack-overflow на цикле (collect до bodies). Тест: prelude↔unicode-style цикл компилируется.
- **Ф.3 — Method-resolution-by-type.** Построить `type→methods`; `x.foo()` резолвится по типу `x` без import. Гейт inherent vs extension.
- **Ф.4 — Char-методы в prelude + снятие Ф.4-хардкода.** Перенести базовые `char @`-методы в prelude/core (рядом с `char`); удалить `CHAR_UNICODE_METHOD_SELECTORS` + `needs_unicode_injection` + блок инъекции (+ `@method:`-тег, если осиротел); таблицы остаются в `std.unicode` под DCE. Проверить: `'Ω'.is_alphabetic()` без import работает через **общий** механизм, не спецкейс.
- **Ф.5 — Политика extension-методов** (`Q-module-resolution-model`): enforcement (`E_EXTENSION_METHOD_NEEDS_IMPORT` или решение по Q) + диагностика ambiguity (два extension-`foo` на один тип из разных модулей).
- **Ф.6 — Тесты + замер компайл-тайма + миграция + close.**

## Тесты (через релизный nova & компилятор)

- **POS:** `'Ω'.is_alphabetic()` без import (общий механизм, не хардкод); user inherent `@`-метод вызывается cross-module без импорта модуля метода (значение «привезло» метод); 3-модульная цепочка (`m: type T+foo` → `a: import m, -> T` → `b: import a, x.foo()`) компилируется без `import m` в `b`; цикл модулей компилируется.
- **NEG:** extension-метод на чужом типе из неимпортированного модуля → ошибка по политике; ambiguity двух extension-`foo` → ошибка; `CHAR_UNICODE_METHOD_SELECTORS` удалён (grep = 0).
- **Регресс:** полный `nova test` по батчам + **kill-switch методология** (см. Plan 159: baseline = переключатель на ТОМ ЖЕ бинаре, НЕ устаревший sibling-бинарь) — zero NEW FAIL.
- **Замер:** компайл-тайм до/после (стоимость collect-all-signatures) + размер C (DCE-профиль не деградирует: char no-import = байт-идентичен explicit-import).

## Критерии приёмки

- **A1.** Цикл модулей (prelude↔unicode-style) **компилируется** — нет stack overflow.
- **A2.** No-import char работает **без единого хардкод-имени** в компиляторе (`CHAR_UNICODE_METHOD_SELECTORS` и `needs_unicode_injection` удалены).
- **A3.** Inherent `@`-метод **любого** типа (std/user) вызывается на значении cross-module **без** импорта модуля метода.
- **A4.** Extension-метод гейтится по согласованной политике (`Q-module-resolution-model`), ambiguity ловится.
- **A5.** Plan 159 DCE по-прежнему срезает неиспользуемые таблицы; замер не деградировал.
- **A6.** Полный регресс zero NEW FAIL (kill-switch методология); компайл-тайм-регресс измерен и в приемлемых рамках.
- **G0 (обязательный, «без упрощений как для прода»):** корректность резолва — **никогда** не разрезолвить в неверный метод, **никогда** не сломать существующий cycle-free код; перф-стоимость collect-all измерена, не «на глаз»; источник истины методов — `.nv`, не Rust.

## Риски

- **Компайл-тайм:** collect-all-signatures для всех in-build модулей. Митигация: собирать сигнатуры (дёшево), тела — лениво; std встроен. Замерить; при нужде — кэш сигнатур (пересекается с «вариантом B» Plan 159).
- **Ambiguity методов:** один селектор на один тип из двух модулей (extension) → нужна явная ошибка, не «тихий выбор».
- **Взаимодействие с mono/generics и folder-module peer-моделью** (Plan 42 D29) — `type→methods` должна учитывать peer-файлы и мономорфизацию.
- **Перф «парсим всё в сборке»** vs текущая ленивость по import — контролировать, что «в сборке» = транзитивно достижимое, а не вся ФС.

## Спека (D-блоки — заводить при реализации, не преждевременно)
