## Цель

Снять ограничение D90 §5 «defer body NO-SUSPEND» — production cleanup
часто требует I/O (graceful socket close с FIN+ACK; DB connection
drain; async transaction commit; fiber join). Реализовать cancel-safe
async-aware cleanup через Plan 49 + Plan 22 (libuv async).

Текущее (D90 §5):
```nova
defer { socket.close() }                    // ❌ если close использует Net.* (suspend)
defer { db_conn.drain() }                   // ❌ если drain через Channel.recv
defer { Time.sleep(100) }                   // ❌ explicit suspend
```

Целевое (после Plan 100.4.2):
```nova
defer { socket.graceful_close() }            // ✅ Net.* допустим
defer { db_conn.drain(timeout: 5_s) }        // ✅ Channel.recv допустим
                                             //    cleanup completes-then-propagates
```

## Дизайн-решения

### D1 — Suspend allowed in defer body

D90 §5 amended:

```
БЫЛО: defer body no-suspend (Time.sleep, Channel.recv, Net.*, spawn,
      parallel for запрещены).
СТАЛО: defer body разрешает suspend; cancel-safe semantics обеспечивается
       runtime'ом.
```

### D2 — Cancel-safe semantics: cleanup completes before cancel-propagation

Когда supervised-scope cancelled, текущий fiber unwinding. Если в
defer body suspend — что происходит?

**Решение:** **cleanup completes** — runtime ждёт cleanup до конца
несмотря на cancel signal. После cleanup-completion cancel-propagation
продолжается.

```nova
supervised(cancel: tok) {
    spawn {
        consume conn = open_db()
        defer { conn.drain(timeout: 5_s) }   // graceful, может suspend
        long_op()                              // running...
    }
    // outer cancels:
    //   1. spawned fiber начинает unwinding
    //   2. defer срабатывает: conn.drain() — может suspend (await complete)
    //   3. drain() completes (или timeout 5s)
    //   4. fiber завершается; cancel propagates наверх
}
```

Это аналог Kotlin `withContext(NonCancellable) { cleanup() }` —
cleanup IS shielded от cancel.

### D3 — Cleanup timeout — `Time.timeout` integration

Чтобы cleanup не зависал навсегда — у programmer есть option ограничить:

```nova
defer {
    with Time.timeout(5_s) {
        socket.graceful_close()                // если closing >5s — abort
    }
}
```

`Time.timeout` (Plan 22 sleep-libuv-integration) уже существует и
работает в async context'е. Plan 100.4.2 просто использует.

### D4 — `shielded { }` для explicit shielding (необязательно)

В большинстве случаев cleanup-shielding автоматический (D2). Но
программист может явно пометить блок как «shielded»:

```nova
defer {
    shielded {                                  // explicit shielding marker
        conn.drain()
    }
}
```

Это essentially no-op (defer body уже shielded), но добавляет
читаемость. **Bootstrap:** не вводим `shielded` keyword — D2
покрывает; future plan если возникнет давление.

### D5 — Spawn в defer body — запрещён

`spawn` в defer body спорный. Если cleanup creates new fiber, тот fiber
живёт после exit scope'а — это leak supervised hierarchy.

**Решение:** spawn в defer body — **error** E (D159-spawn-in-defer).
Cleanup должен быть synchronous-flow (suspend OK, новые fiber'ы — нет).

Parallel for тоже запрещён по той же причине.

### D6 — Channel.recv в defer body — разрешено с caveat

`Channel.recv` suspend'ит. В defer body OK, но осторожно: если channel
никогда не получит value — defer зависнет. Programmer должен
использовать `Time.timeout` или `Channel.try_recv()` для bounded
cleanup.

### D7 — Await preservation: defer пересекает await

```nova
fn process() {
    consume tx = begin()
    errdefer { tx.rollback() }
    await long_async_op()                       // await might cancel
    tx.commit()
}
```

Если cancel приходит во время await:
- `long_async_op` cancellation propagates.
- errdefer fires: `tx.rollback()` — может содержать await сам.
- Cleanup completes (shielded).
- Cancel propagates наверх.

Plan 49 cancel-routing уже это покрывает; Plan 100.4.2 verifies через
тесты.

## Границы

- **`shielded { }` keyword** — не вводим в bootstrap; D2 покрывает
  автоматически. Future plan если давление.
- **Distributed cleanup** (cleanup через cross-machine RPC) — out of
  scope; reuse standard async-cleanup.

## Risks

1. **Cancel-shielding deadlock** — если cleanup сам ждёт ресурс
   который cancel-related (закрытый channel и т.п.), deadlock.
   **Митигация:** `Time.timeout` обязателен в production-grade cleanup;
   диагностика «cleanup running >5s» log-warning.
2. **Infinite-loop в defer body** — programmer pre-existing problem,
   не Plan 100.4.2 issue. **Митигация:** `Time.timeout`.
3. **Plan 22 (libuv async) production readiness** — already ✅.
