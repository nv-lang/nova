# Plan 100.4.4: multi-defer LIFO error accumulation + panic-in-defer

> **Создан 2026-05-23 как sub-sub-plan [Plan 100.4 umbrella](100.4-cleanup-on-failure.md).**
> **Статус:** 📋 proposed. **GATED на [100.4.1](100.4.1-failable-cleanup-body.md)**.
> **Оценка:** ~3 dev-day.
> **D-блок:** **D150**.

---

## Цель

Закрыть 2 production-safety hole'а в defer-семантике:

1. **Multi-defer partial-failure** — если N-й defer в LIFO fail'ит, что
   с N-1, N-2, ...? Сейчас D90 не специфицирует; implicit «stop further
   LIFO». Production-grade требует **продолжение LIFO даже при partial
   failure**, со сбором всех ошибок (chain).
2. **Panic в defer body** — Rust = double-panic-abort; D90 не покрывает.
   Production-grade требует **Plan 49 composition** вместо abort.

Закрывает Rust-уровень nightmare (double-panic-abort одного из самых
неприятных bug patterns) и приводит Nova к TS/Java-уровню safety
(SuppressedError chain / addSuppressed) — на самом деле даже **строже,
через effect-typed visibility** (Plan 49).

---

## Сравнение

| Capability | Go | Rust | TS | Kotlin | Java | Nova target |
|---|---|---|---|---|---|---|
| Multi-cleanup LIFO continues после partial fail | ⚠️ defer continues но errors lost | ❌ first-panic-abort sometimes | ✅ SuppressedError chain | ⚠️ partial | ✅ addSuppressed chain | ✅ **Plan 49 multi-error** |
| Panic в cleanup body | ✅ recover() catches | ❌ **double-panic-abort** | ⚠️ SuppressedError | ⚠️ try-catch | ⚠️ silent if not addSuppressed'd | ✅ **Plan 49 composition + no abort** |
| Error chain inspection в caller | ⚠️ через wraperror | ⚠️ chain via source() | ✅ `error.cause` | ✅ getCause | ✅ getSuppressed | ✅ **MultiError API** (Plan 100.4.1) |
| All N cleanups attempted на failure | ⚠️ depends | ❌ first-Drop-only-tries | ⚠️ depends | ✅ try-finally chain | ✅ try-with-resources | ✅ **LIFO guaranteed completion** |

Nova **превосходит Rust** уверенно (no double-panic-abort + all-cleanups-
attempted); **matches TS/Java** на composition; **превосходит** на
visibility (effect-typed).

## Дизайн-решения

### D1 — LIFO continues после partial failure

Правило: каждый defer body run'ится **независимо**. Если defer N
fail'ит — runtime catches его error, continues с defer N-1 LIFO.
Все собранные ошибки accumulate в multi-error structure (Plan 49).

```nova
fn process() Fail[MultiErr] -> () {
    defer A_runs                                 // fail E_a
    defer B_runs                                 // fail E_b
    defer C_runs                                 // success
    body                                          // fail E_main
}
// Exit semantics:
//   1. body throws E_main
//   2. C_runs — success; no contribution
//   3. B_runs — fails E_b; accumulated as suppressed
//   4. A_runs — fails E_a; accumulated as suppressed
//   5. caller receives MultiError { primary: E_main, suppressed: [E_b, E_a] }
//      (LIFO order: B first to fail, then A)
```

### D2 — Panic в defer body — composes, не abort

D90 + Plan 49: panic в defer body **composes** с propagating error
(если есть) либо initiates throw (если был normal exit). **Никаких
abort**.

Сценарии:

**Сценарий A: panic в defer на normal exit:**
```nova
fn process() -> () {
    defer { panic("cleanup broken") }
    do_normal()                                  // success
}
// Exit:
//   1. normal exit reached
//   2. defer fires — panic("cleanup broken")
//   3. fiber unwinding с panic
//   4. caller catches panic (или fiber dies при no handler)
```

**Сценарий B: panic в defer во время error-propagation:**
```nova
fn process() Fail[Err] -> () {
    defer { panic("cleanup broken") }
    do_fails()?                                  // throws Err1
}
// Exit:
//   1. body throws Err1
//   2. unwinding starts
//   3. defer fires — panic("cleanup broken")
//   4. panic composes с Err1: MultiError { primary: Err1, suppressed: [Panic("cleanup broken")] }
//   5. propagation continues with composed error
```

Plan 49 multi-error already поддерживает panic-as-throw (panic =
unhandled cancel-like throw). Plan 100.4.4 расширяет composition на
panic.

### D3 — Defer-stack runtime structure

Runtime maintains per-fiber defer-stack. При exit:

```
for entry in stack.reverse() {
    let result = run_defer_body(entry)
    match result {
        Ok(()) => continue
        Err(e) => {
            propagating_error = compose(propagating_error, e)
            continue                              // continue LIFO!
        }
        Panic(m) => {
            propagating_error = compose(propagating_error, Panic(m))
            continue                              // panic does NOT stop LIFO!
        }
    }
}
throw propagating_error
```

Это **fundamental change**: defer-stack walk completes даже при
ошибках. Rust does NOT do this (panic-in-drop-during-panic = abort).

### D4 — Diagnostic — chain visibility

Multi-error message format:

```
error: composite error during scope exit
  primary error:
    Err1 ("operation failed")
        at do_work (process.nv:12)
  suppressed during defer LIFO (in order of failure):
    [1] Err_B ("cleanup B failed")
        at B_cleanup() in defer (process.nv:10)
    [2] Err_A ("cleanup A failed")
        at A_cleanup() in defer (process.nv:8)
    [3] Panic("cleanup C broken")
        at panic() in defer (process.nv:11)
```

LIFO order shows failures в порядке firing. Каждая ошибка имеет
attribution (where).

### D5 — Limits — `Time.timeout` для protected cleanup

Если defer body suspend'ит (Plan 100.4.2) и hangs — это reverses
benefit. Programmer обязан:

```nova
defer {
    with Time.timeout(5_s) {
        long_cleanup()                            // bounded
    }
}
```

Если timeout fires — cleanup fail'ит с TimeoutError; LIFO continues.
Это правильное поведение — нет infinite-hang.

### D6 — Memory safety: ensure all cleanups attempted

В Rust panic-in-drop-в-stack-unwinding ведёт к leak других ресурсов
(unwinding aborts, остальные Drop'ы не запускаются). Nova D1 garantees
each defer attempted, regardless of failures.

Trade-off: composite-error может быть **большим** (N defers all fail
→ N entries in suppressed). Acceptable cost для memory safety.

### D7 — Composition order

Multi-error composition:
- Primary: первая ошибка (либо body throw, либо первый defer fail).
- Suppressed: остальные failures (LIFO order: first to fail = first in
  suppressed list).
- Если body succeeded но defers fail'ат: первый defer fail становится
  primary, rest — suppressed.

### D8 — Чтение / inspection в caller

Caller catches composite error:

```nova
match process() {
    Ok(_) => Log.info("done"),
    Err(MultiError { primary, suppressed }) => {
        Log.error("primary: ${primary}")
        for s in suppressed {
            Log.error("  suppressed: ${s}")
        }
    }
}
```

Или для convenience — иметь `MultiError.fmt_chain() -> str`.

- [ ] Panic в defer body **композируется** с propagating, **не abort**

## Границы

- **Cleanup body recursion / infinite-loop** — programmer responsibility
  (use `Time.timeout`). Runtime detects hangs >10s — log warning.
- **Error chain depth limit** — bootstrap unlimited; future plan может
  add depth-cap.

## Risks

1. **Memory pressure** при many cleanup failures — composite-error
   carries N suppressed pointers. **Митигация:** GC handles; cleanup
   eventually frees.
2. **Caller не expects composite** — может handle только primary.
   **Митигация:** AI-first diagnostic подсказывает посмотреть suppressed.
3. **Plan 49 composition** — must support panic as composable throw.
   **Митигация:** Ф.3 verifies; extends if needed.
4. **Programmer confused о LIFO с partial fail** — non-obvious что defer
   N-1 runs даже если N fail'нул. **Митигация:** docs + idiom doc.
