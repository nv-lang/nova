# Plan 179.1 — Гражданское (календарное) время: `Date`/`TimeOfDay`/`DateTime`/`ZonedDateTime`/`TimeZone`/`Period` + ISO-8601/RFC-3339

> **Под-план [179](179-time-system-rework.md).** Создан 2026-06-22. **Статус:** 📋 PROPOSED.
> **Маркер:** `[M-179.1-civil-time]`. **Запуск:** «**выполни план 179.1**».
> **Зависит от 179** (типизированный `Timestamp`/`Time`-эффект — фундамент; civil-now резолвится через `Time.timestamp()`).
> **Координация:** value-ABI новых типов = Plan 172.4; str/format = [91.18](91.18-str-unicode-api.md); cron
> ([std/_experimental/time/cron.nv](../../std/_experimental/time/cron.nv)) — потенциальный потребитель.
> **Эталон:** java.time (JSR-310) / TC39 Temporal; Go `time`; Rust `chrono`/`time`.
> **Сквозной критерий:** «без упрощений, как для прода».

## 1. Зачем

После 179 у Nova есть **instant** (`Timestamp` — Unix-ns) и **interval** (`Duration`), но **нет «человеческого»
календаря**: год/месяц/день, час/минута, таймзоны, форматирование/парсинг. Нельзя ни вывести `Timestamp` как
`2026-06-22T14:00:00Z`, ни распарсить дату, ни прибавить «1 месяц». Это закрывает большой пласт реальных задач
(логи, API, расписания, БД-таймстемпы) и снимает зависимость, отмеченную в 179 (full-datetime `@display`).

## 2. Эталон (Rust / Go / java.time / Temporal)

| Аспект | Rust | Go | java.time / Temporal |
|---|---|---|---|
| где живёт | **не в std** (crate `chrono`/`time`) | в `time` (встроено) | в JDK / в языке (Temporal) |
| instant vs civil | `SystemTime`(instant); civil — в crate | всё в одном `Time` (wall+mono+civil) | `Instant` ↔ `LocalDate`/`LocalDateTime`/`ZonedDateTime` (раздельно) |
| plain (без зоны) | `NaiveDate`/`NaiveDateTime` (chrono) | — (нет «plain») | `LocalDate`/`LocalTime`/`LocalDateTime` (`Plain*` в Temporal) |
| zoned | `DateTime<Tz>` | `Time` + `Location` | `ZonedDateTime` |
| таймзоны | crate `chrono-tz` (IANA) | OS tzdata / embedded IANA | IANA (`ZoneId`) |
| календарная величина | `chrono::Months`/`Days` | ❌ (только `Duration`) | **`Period`** (years/months/days) ≠ `Duration` |
| формат | crate (`format!`/strftime-like) | reference-layout `2006-01-02` (Go-изм) | `DateTimeFormatter` (pattern) + ISO |
| leap seconds | игнор | игнор (smear) | игнор |

**Три урока (беру за основу):**
1. **Instant ≠ civil.** Не смешивать Unix-наносекунды с полями календаря (Nova уже так: `Timestamp` отдельно).
2. **Plain vs Zoned.** `DateTime` (без зоны, «настенный календарь») отдельно от `ZonedDateTime` (= `DateTime` + зона →
   резолвится в `Timestamp`). Никаких неявных дефолтных зон.
3. **`Period` ≠ `Duration`.** `Duration` = точный elapsed (ns); `Period` = календарная величина (мес./годы, переменной
   длины). «+1 месяц» — только через `Period`. **У Nova `Period` нет — вводим здесь.**

## 3. Модель типов (proposal)

Все — `value`-records (стек), immutable. Имена под bikeshed (Ф.0); `Time` занят эффектом → «время суток» = `TimeOfDay`.

| Тип | java.time | Представление | Суть |
|---|---|---|---|
| `Date` | `LocalDate` | `epoch_day i64` (дни от 1970-01-01) | дата без времени/зоны; **range шире `Timestamp`** (±млн лет, не ±292) |
| `TimeOfDay` | `LocalTime` | `nanos_of_day i64` (0 … 86_399_999_999_999) | время суток без даты |
| `DateTime` | `LocalDateTime` | `Date` + `TimeOfDay` | гражданские дата+время, **без зоны** |
| `Offset` | `ZoneOffset` | `seconds i32` (±18h) | фикс. UTC-сдвиг (`+03:00`) |
| `TimeZone` | `ZoneId` | IANA-имя + правила | именованная зона с DST (Ф.5, gated) |
| `ZonedDateTime` | `ZonedDateTime` | `DateTime` + `TimeZone`/`Offset` | резолвится ↔ `Timestamp` (instant) |
| `Period` | `Period` | `years i32, months i32, days i32` | **календарная** величина (≠ `Duration`) |
| `Weekday`/`Month` | enum | sum-type | `Mon…Sun` / `Jan…Dec` |

**Ключевое представление (java.time-урок):** civil-типы хранят `epoch_day`+`nanos_of_day`, **не** ns-от-эпохи →
range гражданских дат **не упирается в ±292 года** (это лимит только `Timestamp`-instant'а). Лимит бьёт лишь при
конверсии civil ↔ `Timestamp`.

## 4. Методы (эскиз — финал при реализации)

- **`Date`:** `Date.of(y, m, d) -> Result[Date, DateError]` (валидация — Feb 30 → ошибка), `Date.today(tz) ZonedTime`,
  `@year/@month/@day/@weekday/@day_of_year`, `@plus(Period)/@minus(Period) -> Date`, `@minus(Date) -> Period` (или дни),
  `@compare`, `@iso() -> str` / `Date.parse_iso(s)`, ISO-week.
- **`TimeOfDay`:** `of(h, m, s, ns) -> Result`, `@hour/@minute/@second/@nano`, `@plus(Duration)` (с wrap/overflow-флагом),
  `@compare`.
- **`DateTime`:** `of(Date, TimeOfDay)` / `Date.at(TimeOfDay)`, `@date/@time`, компоненты, `@plus(Period|Duration)`,
  `@to_zoned(tz) -> ZonedDateTime`, ISO-8601 format/parse.
- **`ZonedDateTime`:** `now(tz) => Time.timestamp().to_zoned(tz)`, `@to_instant() -> Timestamp`, `@offset()`,
  `@with_zone(tz)`, арифметика с DST-resolution, RFC-3339.
- **`Timestamp` (из 179):** `+@to_zoned(tz) -> ZonedDateTime`, `+@to_rfc3339() -> str`, `+Timestamp.parse_rfc3339(s)`,
  full-datetime `@display` (закрывает зависимость из 179).
- **`Period`:** `of(y, m, d)`, `@plus/@minus`, `@normalized`; **тип-разделён** с `Duration` (нельзя сложить напрямую).
- **`Weekday`/`Month`:** `@next/@prev`, `@number`, `Into[str]`.

## 5. Дизайн-вопросы (решить в Ф.0)

1. **Именование.** `Date`/`TimeOfDay`/`DateTime` (plain, без префикса — в Nova нет неявной зоны) + явный `ZonedDateTime`.
   vs `Local*`/`Plain*`-префиксы (java.time/Temporal). Рекоменд.: без префикса + явный `Zoned`.
2. **Таймзоны — вес.** MVP = `Offset` + `UTC` + system-local (через эффект); **IANA tz-db по имени** (`Europe/Moscow` +
   DST-правила) — **Ф.5, gated/опц.** (тяжело: читать OS-tzdata через libc vs embed-snapshot). Rust std вообще без зон;
   Go тащит OS-tzdata. Рекоменд.: MVP без IANA, IANA отдельной фазой.
3. **Формат.** MVP = **ISO-8601 / RFC-3339** (канон, как `time.RFC3339`); общий pattern-формат (strftime /
   `DateTimeFormatter`) — followup. Go-reference-layout (`2006-01-02`) — **не брать** (Go-изм).
4. **BigDate.** Civil-типы на `epoch_day`+`nanos_of_day` → range уже не ±292 (см. §3). Конверсия в `Timestamp` за
   пределами ±292 → `Result`/clamp (решить).
5. **Leap seconds** — **игнорировать** (как java.time/Go/Unix; день = 86400s). Зафиксировать.
6. **Валидация конструкторов** — `Result[T, DateError]` (Feb 30 → ошибка), не panic. DST gap/overlap при резолве
   `ZonedDateTime` — java.time-правила (gap→сдвиг вперёд, overlap→ранний offset; опц. явный resolver).
7. **Calendar-арифметика.** `Date + 1.month` через `Period`; month-add **клампит** к последнему валидному дню
   (Jan 31 + 1mo → Feb 28/29) — java.time-семантика.

## 6. Фазы

- **Ф.0 — дизайн-решения (gate):** §5 (имена, tz-стратегия, формат-scope, BigDate, leap, валидация, calendar-rules).
  Черновик D-block. **Без кода.**
- **Ф.1 — Plain civil:** `Date`/`TimeOfDay`/`DateTime` (value-records; epoch-day/ns-of-day), конструкторы-с-валидацией,
  компоненты, `@compare`, `@weekday`/ISO-week. Алгоритмы date↔epoch-day (Howard Hinnant `days_from_civil`).
- **Ф.2 — `Period` + calendar-арифметика:** тип-разделение с `Duration`; month/year-add с клампом.
- **Ф.3 — Offset + UTC + Local + `ZonedDateTime`:** конверсии `Timestamp` ↔ `ZonedDateTime`; system-local — через
  эффект-оп (коорд. 179: возможно `Time.local_offset() -> Offset`). **MVP без IANA-by-name.**
- **Ф.4 — Format/parse ISO-8601 / RFC-3339:** `Timestamp.to_rfc3339`/`parse_rfc3339`, `Date`/`DateTime` ISO; full
  `@display`/`@debug` для civil + `Timestamp` (закрывает зависимость 179).
- **Ф.5 — IANA tz-db (gated/опц.):** `TimeZone.of("Europe/Moscow")` + DST-правила (OS-tzdata vs embed). Тяжёлая —
  возможно отдельный под-план.
- **Ф.6 — тесты + spec + docs.** pos+neg (вкл. високосные годы, DST gap/overlap, invalid-date, ISO round-trip).

**Вне scope (followup):** общий pattern-формат (strftime/`DateTimeFormatter`); локализация (месяцы/дни на языках);
не-григорианские календари; cron-интеграция (`std/_experimental/time/cron.nv` может стать потребителем).

## 7. Spec / D / docs

- **NEW D-block** — «гражданское время: plain vs zoned, `Period` ≠ `Duration`, epoch-day представление, leap-seconds
  ignore, calendar-арифметика (month-clamp), DST-resolution». error-index: `DateError`/parse-ошибки.
- `docs/` — новый `docs/datetime.md` (модель + примеры + таблица «Nova ↔ java.time»).

## 8. Критерии приёмки

1. `Date`/`TimeOfDay`/`DateTime`/`Offset`/`ZonedDateTime`/`Period` + `Weekday`/`Month` реализованы (value-records).
2. ISO-8601 / RFC-3339 format+parse round-trip; `Timestamp ↔ ZonedDateTime` конверсия корректна (вкл. ±292 граница).
3. `Period` тип-разделён с `Duration`; calendar-арифметика (month-clamp, weekday) корректна на високосных/DST-кейсах.
4. full-datetime `@display`/`@debug` (закрывает зависимость из 179).
5. Полный регресс зелёный; **без упрощений**. IANA tz-db — отдельно (Ф.5/под-план), не блокирует 1-4.
