## 1. Bugs + симптомы

**Тесты:** `parallel_for` (даже с NOVA_MAXPROCS=1 directive после
flip), `sleep_precision_bench`, `sleep_real_clock`, `plan40_perf_bench`.

### parallel_for ordering

Encoded-log тесты (`assert(log == 121212)`) — порядок исполнения
проверяется явно. Под cooperative drain (без armed runtime)
scheduler — round-robin строго детерминированный. Под flip
активированный, даже с `NOVA_MAXPROCS=1` (1 worker), worker
scheduling использует **work-stealing deque**, иной order vs main-
thread cooperative drain.

### sleep_precision_bench / sleep_real_clock

Тесты wall-clock precision: `Time.sleep(N)` → `elapsed ≈ N ms`,
budget e.g. `elapsed < N+2`. Под M:N добавляется scheduler jitter
(worker thread не сразу resume'ит fiber после wake). Budgets
overrun на 5-20 ms.

### plan40_perf_bench

Latency measure'ы channel send/recv. Под M:N добавляется cross-
thread wake overhead. Budgets overrun.

## 2. Cross-runtime reference

### Go testing — set-equality для parallel ordering

Go test convention:
```go
got := []int{}
var mu sync.Mutex
wg := sync.WaitGroup{}
for _, x := range input {
    wg.Add(1)
    go func(x int) {
        defer wg.Done()
        mu.Lock()
        got = append(got, x*2)
        mu.Unlock()
    }(x)
}
wg.Wait()
sort.Ints(got)
if !reflect.DeepEqual(got, expected) { t.Fail() }
```

Ordering не проверяется. Sort + compare.

### tokio benchmarks — relaxed budgets

`criterion` benchmarks выполняют warmup + N iterations + statistical
analysis (median, outliers). Тесты `assert(time < N+slack)` слабо
типичны; обычно бенч печатает median и outliers для review.

Production tests timing-precision (где важно):
- `tokio::time::Sleep` budget `± 10-20ms` обычная практика.
- Tighter budgets — отдельный test category, под `#[ignore]` либо
  CI-skipped.

### JS testing — `expect(time).toBeCloseTo(N, slack)`

```js
const start = Date.now();
await sleep(100);
expect(Date.now() - start).toBeGreaterThanOrEqual(100);
expect(Date.now() - start).toBeLessThan(150);
```

Slack типично 30-50% от target для setTimeout (event-loop jitter
~4-15ms на загруженном loop'е).

### Nova сейчас

Tight budgets (`< target + 2ms`) — не оставляет slack для M:N
scheduler jitter. Подходит для cooperative single-thread drain где
overhead ≤1ms.

## 3. Дизайн решения

### parallel_for тесты

Переписать на **set-equality**:
```nova
// До:
let mut log = 0
parallel for x in [1, 2] {
    log = log * 10 + x
    Time.sleep(0); log = log * 10 + x
}
assert(log == 121212)  // ordering-specific

// После:
let mut counts = HashMap[int, int]
parallel for x in [1, 2] {
    counts.set(x, (counts.get(x) ?? 0) + 1)
}
assert(counts.get(1) == 1)
assert(counts.get(2) == 1)
```

ИЛИ для тестов, специально проверяющих parallelism (interleaving):
переместить в `bench/micro/parallel_*` workload-бенчи (Plan 83.4.5.6
Ф.4 speedup-bench будет их использовать); из `nova_tests/` убрать.

### Precision benches

Два пути:
- **`// ENV NOVA_MAXPROCS=1` directive** — оставляет 1-worker
  scheduler; jitter минимальный (single-thread cooperative-like).
  Budgets можно держать tight.
- **Relax budgets** — позволить ±10-20ms slack под M:N. Документация
  «эта precision-bench — M:N-aware budget; tighter precision проверяет
  bench/micro/sleep_precision_single.nv» (отдельный bench под
  NOVA_MAXPROCS=1).

**Рекомендация:** **directive** для existing tests (минимальная
disruption); создать отдельный `bench/micro/sleep_precision_mn.nv` для
M:N-specific precision (для будущего, не обязательно сейчас).

Новый `bench/micro/sleep_precision_mn.nv`:
```nova
// Plan 83.4.5.3 Ф.3 (2026-05-23): M:N-specific precision bench с
// relaxed budgets под scheduler jitter.
// ENV NOVA_MAXPROCS=4

bench "sleep(10ms) precision under 4-worker M:N" {
    measure {
        let start = Time.now_ms()
        Time.sleep(10)
        let elapsed = Time.now_ms() - start
        // Slack 10ms — typical M:N scheduler jitter.
        assert(elapsed >= 10 and elapsed < 25)
    }
}
```

(Optional; не блокирует closure.)

## 5. Acceptance

- `parallel_for`, `sleep_precision_bench`, `sleep_real_clock`,
  `plan40_perf_bench` PASS под active flip.
- Set-equality assertions корректны на любой execution-order.
- precision-tests с `NOVA_MAXPROCS=1` directive — tight budgets
  держатся.
- 0 регрессий single-thread пути.

## 6. Risk

- **Set-equality может скрыть real bugs** в parallel ordering
  semantics. Mitigation: отдельные ordering tests в
  `bench/micro/parallel_for_workload.nv` под REAL M:N N=4 —
  проверяют РАЗНЫЕ executions (а не специфический).
- **NOVA_MAXPROCS=1 directive для precision** — теряем real-world
  M:N precision coverage. Mitigation: optional Ф.3 M:N-specific bench.
