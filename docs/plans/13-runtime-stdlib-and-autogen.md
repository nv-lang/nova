ExternalRegistry (codegen) загружает 4 per-type файла через `include_str!`

правки реестра + emitter'а после ревью текущих сгенерированных файлов:
- `StringBuilder mut @append` возвращает `Self` (chaining).
- Все `WriteBuffer mut @write_*` возвращают `Self` (chaining).
- StringBuilder получает оператор `+` как алиас `@append` (str + char).
- `str + str` явно через `@concat` в registry (не invisible intrinsic).
- `str @char_len` → `@len` (D26 spec говорит просто `len`).
- `str @chars() -> []int` → `-> []char` (char first-class).
- ReadBuffer получает `@read_char` / `@read_str(n)` + Result-формы;
  `ReadBufferError.InvalidUtf8 { position }` для невалидного UTF-8.
- **Auto-derive `try_read_*` из `read_*` отменён** (Plan 12 Ф.4.5 ❌).
  Все формы — явно в registry. D73 From↔Into остаётся.
- Renderer: пустая строка между методами в auto-gen .nv для читаемости.
См. Ф.9 ниже.

## Цель

Сейчас `std/runtime/builtins.nv` содержит только StringBuilder /
WriteBuffer / ReadBuffer + `str.from(char)`. Чтобы builtins-driven
dispatch (Plan 12) был последовательным, **все** runtime-функции
которые знает компилятор, должны быть **спроецированы** в
`std/runtime/*.nv` как `external fn` декларации:

1. **str-runtime API** — UTF-8 операции (`s.len`, `s.slice`, `s.find`,
   `s.chars()`, `s.to_lower`, etc.). Сейчас компилятор знает их
   как special-case'ы; должны стать обычными external fn.
2. **math-runtime API** — `f64.sin/cos/sqrt/log/exp/...`,
   `f32.sin/cos/...` через D74 (instance-методы). Сейчас компилятор
   эмитит прямые `sin`/`cos` из C `<math.h>`; должны быть `external fn`
   декларации.

Файлы — **auto-generated**: ручные правки запрещены, при каждой
регенерации файлы перезаписываются. Источник — внутренний реестр
`runtime_implemented` компилятора.

## Не цели

- **Math на u64/u32** — `sin(u64)` бессмысленно. Только `f64`/`f32`
  (D74). Целочисленные операции (`abs`, `gcd`, `ipow`, `leading_zeros`,
  `popcount`) — отдельный набор, future plan, не входит в Plan 13.
- **Расширение runtime новыми функциями** — план только проецирует
  то что **уже** реализовано в codegen.
- **Замена user-facing API** — math/str API стабильны (D26/D74), Plan 13
  не меняет их сигнатуры.

## Принцип: builtins.nv как проекция компилятора

```
       ┌──────────────────────────┐
       │ Compiler runtime_         │ ← single source of truth (Rust)
       │   implemented registry    │   "что компилятор реально знает"
       └────────┬─────────────────┘
                │ generated by
                ▼
       ┌──────────────────────────┐
       │ std/runtime/*.nv         │ ← AUTO-GENERATED
       │   (read-only projection) │   "Nova-side declarations"
       └────────┬─────────────────┘
                │ consumed by
                ▼
         user Nova code
         (sb.append, x.sin, ...)
```

`std/runtime/*.nv` — **read-only artefact**, как `target/` или
скомпилированные `.o`-файлы. Хранятся в git для удобства code-review
и type-check'а без полной компиляции, но **никто их не правит вручную**.

## Структура `std/runtime/*.nv`

После Plan 13:

| Файл | Содержание |
|---|---|
| `std/runtime/builtins.nv` | StringBuilder / WriteBuffer / ReadBuffer (как сейчас) + `str.from(char)`. |
| `std/runtime/string.nv` | str API: `@len`, `@slice`, `@find`, `@chars`, `@to_lower`, etc. |
| `std/runtime/math.nv` | `f64`/`f32` math: `@sin`, `@cos`, `@sqrt`, `@atan2`, `@hypot`, `@is_finite`, плюс константы `f64.PI`, `f64.NAN`, etc. |

Все три файла:
- В namespace `std.runtime.*` → D82 whitelist разрешает `external fn`.
- Auto-generated header в первой строке:
  ```nova
  // AUTO-GENERATED by `nova-codegen --emit-runtime-stubs`.
  // Do not edit manually — changes will be overwritten.
  // Source of truth: compiler-codegen/src/codegen/runtime_registry.rs
  ```
- В prelude **не попадают сами** — но **методы и типы из них** уже
  в prelude через D26 (`str.@find` — обычный method call,
  `f64.@sin` — обычный method call). builtins.nv-driven dispatch
  (Plan 12) lookup'ит декларации в этих файлах автоматически.

## Этапы

### Ф.1 — Реестр `runtime_implemented` в компиляторе (~2ч)

Сейчас компилятор знает str/math функции через scattered special-cases
(`emit_call`, intrinsics). Plan 13 их **формализует**:

1. Создать `compiler-codegen/src/codegen/runtime_registry.rs` —
   единый структурированный реестр `Vec<RuntimeFn>` где `RuntimeFn`:
   ```rust
   struct RuntimeFn {
       module: &'static str,         // "std.runtime.string", "std.runtime.math"
       receiver: Option<RecvType>,    // Some("str"), Some("f64"), None
       is_static: bool,               // T.X(...) vs t.X(...)
       is_mut: bool,                  // mut receiver
       name: &'static str,            // "find", "sin"
       params: Vec<(&'static str, NovaType)>,
       return_ty: NovaType,
       effects: Vec<EffectId>,        // Fail[E], etc.
       c_name: &'static str,          // "Nova_str_method_find"
       doc: &'static str,             // doc comment for generated .nv
   }
   ```
2. Заполнить реестр для всех известных компилятору функций:
   - **str**: `len`, `byte_len`, `slice`, `find`, `rfind`, `contains`,
     `starts_with`, `ends_with`, `split`, `trim`, `to_lower`,
     `to_upper`, `chars`, `bytes`, `char_at`, `is_empty`, `==`, `+`
     (concat — это уже оператор, не fn).
   - **f64**: `sin`, `cos`, `tan`, `asin`, `acos`, `atan`, `atan2`,
     `sqrt`, `log`, `log2`, `log10`, `exp`, `pow`, `hypot`, `floor`,
     `ceil`, `round`, `abs`, `is_finite`, `is_nan`, `is_infinite`.
     Плюс константы `f64.PI`, `f64.E`, `f64.NAN`, `f64.INFINITY`.
   - **f32**: subset f64 (минимум для embedded use-case).
3. Источники реальных C-имён — `nova_rt/string.h`, `nova_rt/math.h`
   (создаются если нет, см. Ф.2).

**Acceptance:** реестр компилируется, содержит ≥40 функций (str ~20
+ f64 ~20). Печатается через `nova-codegen --dump-runtime` для
sanity-check'а.

### Ф.2 — Runtime C-реализация (~3-4ч)

`nova_rt/string.h` / `nova_rt/string.c` — UTF-8 операции:
- `nova_str_method_find(nova_str s, nova_str needle) -> NovaOpt_int`
- `nova_str_method_slice(nova_str s, int a, int b) -> nova_str`
- `nova_str_method_to_lower(nova_str s) -> nova_str`
- ...

**Зависимости:** UTF-8 walking utility (уже есть для `s.len` в
существующем коде — рефакторить в `nova_rt/utf8.h` если ещё не).

`nova_rt/math.h` / `nova_rt/math.c` — обёртки над C `<math.h>`:
- `nova_f64_method_sin(nova_f64 x) -> nova_f64` → `sin(x)`
- `nova_f64_method_cos(nova_f64 x) -> nova_f64` → `cos(x)`
- ...

Это в основном one-liner trampolines к libc. Зачем вообще обёртки:
- Единый mangling (`Nova_f64_method_sin` vs `sin`).
- Чёткая поверхность для D82 single source of truth.
- Возможность позже подменить (e.g., SIMD вариант).

**Acceptance:** `nova_rt/string.c` + `nova_rt/math.c` компилируются;
все функции из реестра Ф.1 имеют реализацию.

### Ф.3 — Auto-gen команда + промпт для агента (~2ч)

Добавить subcommand: `nova-codegen emit-runtime-stubs`.

**Рабочий промпт для агента/разработчика** —
[`docs/promts/regen-runtime.md`](../promts/regen-runtime.md). Это
повторно используемая инструкция: входы, шаги, acceptance, ограничения,
троблшутинг. Она версионируется вместе с кодом — при изменении
формата `RuntimeFn` или процедуры регенерации промпт обновляется
синхронно.

Pseudo-code:
```rust
fn emit_runtime_stubs() {
    let registry = runtime_registry::all();
    let mut by_module: HashMap<&str, Vec<&RuntimeFn>> = group_by_module(&registry);
    for (module, fns) in by_module {
        let path = module_to_path(module);   // "std.runtime.string" → "std/runtime/string.nv"
        let content = render_nv(&module, &fns);
        fs::write(path, content);
    }
}
```

Generator-логика для каждого `RuntimeFn` → `external fn`-декларация:

```nova
// AUTO-GENERATED by `nova-codegen emit-runtime-stubs`.
// Do not edit manually — changes will be overwritten.
// Source of truth: compiler-codegen/src/codegen/runtime_registry.rs

module std.runtime.math

// Тригонометрия (radians).
export external fn f64 @sin() -> f64
export external fn f64 @cos() -> f64
export external fn f64 @tan() -> f64

// Корни и логарифмы.
export external fn f64 @sqrt() -> f64
export external fn f64 @log()  -> f64
...
```

**Acceptance:** запуск `nova-codegen emit-runtime-stubs`:
- Создаёт/перезаписывает `std/runtime/string.nv` и `std/runtime/math.nv`.
- Файлы парсятся как валидный Nova; `nova-codegen --check std/`
  проходит без ошибок.
- Контент детерминирован (тот же реестр → байт-в-байт тот же файл).
  Stable ordering: by module → by receiver → by name.

### Ф.4 — Migration: убрать special-case emit для str/math (~2-3ч)

Сейчас в `emit_c.rs` str-операции и math-операции эмитятся
**special-case'ом** в `emit_call`. После Plan 13 — единый путь
через builtins.nv-driven dispatch (Plan 12 Ф.3 `emit_external_call`):

1. Удалить special-case branch'и для str-методов в `emit_call`.
2. Удалить special-case branch'и для math-методов.
3. После Ф.1-Ф.3 эти вызовы автоматически попадают в общий
   `external_registry.lookup` (builtins.nv-driven path).

**Edge case:** оператор-уровневые (`s1 + s2`, `a + b` для int/float)
— **остаются** intrinsics, не идут через registry. Они не fn-вызовы,
а синтаксические операторы (D-something).

**Acceptance:** все codegen-тесты проходят; диффs `emit_c.rs`
негативный на ~150-300 строк.

D26 (08-runtime.md) сейчас описывает str API частично (есть `@find`,
`@slice`, etc.) и math через D74. Plan 13 делает их **формальной
частью** prelude через single source of truth:

1. В D26 добавить раздел «Runtime stdlib проекция»:

   ```
   std/runtime/math.nv (auto-generated). Эти модули НЕ требуют
   import — методы доступны через обычный method-call синтаксис
   (s.find, x.sin), потому что str / f64 / f32 — built-in типы из
   prelude.
   ```
2. В D74 cross-link на `std/runtime/math.nv` как реализацию.
3. В D82 cross-link на Plan 13 как пример проекции реестра в
   builtins.nv-style модули.

### Ф.6 — Pre-commit/CI guard для read-only файлов (~1ч)

Чтобы предотвратить ручные правки `std/runtime/*.nv`:

1. Pre-commit hook (опциональный, в `.git/hooks/pre-commit`):
   ```bash
   if git diff --cached --name-only | grep -q '^std/runtime/.*\.nv$'; then
       echo "ERROR: std/runtime/*.nv are auto-generated."
       echo "Run 'nova-codegen emit-runtime-stubs' instead of editing manually."
       exit 1
   fi
   ```
2. CI check: после `cargo build` запускать `nova-codegen
   emit-runtime-stubs --check` (флаг сравнивает с текущим состоянием
   файлов; если diff — fail). Это защищает от случая «изменили реестр,
   забыли регенерить».

## Тесты

- **Acceptance 1 (auto-gen detrminизм):** `emit-runtime-stubs` дважды
  подряд → no diff в `std/runtime/*.nv`.
- **Acceptance 2 (round-trip):** добавить `f64.@sinh` в реестр →
  регенерить → код `let y = x.sinh()` компилируется и работает.
  Без правок Rust-codegen'а вне `runtime_registry.rs` + `nova_rt/math.c`.
- **Acceptance 3 (manual edit guard):** ручная правка
  `std/runtime/string.nv` (например, добавить fake `@@hack()`) →
  pre-commit/CI fail.
- **Regression:** все codegen-тесты (~78 nova_tests, ~42 codegen)
  проходят после Ф.4 special-case removal.
- **Negative test:** `let s = "x"; s.no_such_method()` →
  Nova-error «no method» (Plan 12 Ф.6 покрывает; Plan 13 проверяет
  что для **str** тоже работает после migration'а).

## Open questions внутри плана

1. **str API API stability.** Сейчас D26 описывает `@find`/`@slice`/etc
   как codepoint-indexed. Plan 13 не меняет — только проецирует
   уже описанное. Если найдутся расхождения D26 vs реестр — приоритет
   D26 (spec выше реализации).
2. **Math константы (`f64.PI`).** Это static-поля типа, не методы.
   В builtins.nv-стиле — как декларировать? Возможный синтаксис:
   ```nova
   export external const f64.PI: f64
   ```
   Если `external const` ещё не grammar'ом поддержан — Plan 13 Ф.1
   эмитит как static-функцию `fn f64.PI() -> f64` временно;
   future refactor закроет правильно. Q-external-const.
3. **f32 минимум.** Какой именно subset? Plan 13 Ф.1 берёт минимум
   (sin/cos/sqrt/abs) — расширяется по мере появления use-case'ов.
4. **Pre-commit vs CI guard.** Pre-commit более надёжен, но настраивается
   локально каждым разработчиком. CI — централизованно. Делать оба
   (Ф.6) или только CI? Решено: оба, pre-commit как опциональный
   convenience.

## Ф.8 — Расширение scope: f32 + полная декомпозиция builtins.nv (2026-05-08, после MVP closure)

**Контекст.** MVP закрыт: registry+emitter работают, math.nv (f64) и
string.nv сгенерированы. Но scope MVP был частичным:
- math: только f64; f32 не покрыт.
- string: основной API есть, но возможно неполный (нужен audit).
- **`std/runtime/builtins.nv` остался handwritten** — opaque types
  StringBuilder/WriteBuffer/ReadBuffer + `str.from(char)`. Это
  «исключение из правила»: остальные generated, а builtins.nv нет.

Ф.8 устраняет это исключение. После Ф.8 **в `std/runtime/` нет ни
одного handwritten файла** — всё auto-generated.

### Что меняется

1. **Файл `builtins.nv` удаляется.** Его содержимое мигрирует в
   отдельные per-type файлы (D78 «file=type», как для остальных
   типов в `std/`):

   | Был | Стал |
   |---|---|
   | `std/runtime/builtins.nv` (StringBuilder секция) | `std/runtime/string_builder.nv` (auto-gen) |
   | `std/runtime/builtins.nv` (WriteBuffer секция) | `std/runtime/write_buffer.nv` (auto-gen) |
   | `std/runtime/builtins.nv` (ReadBuffer секция) | `std/runtime/read_buffer.nv` (auto-gen) |
   | `std/runtime/builtins.nv` (`str.from(char)`) | `std/runtime/char.nv` (auto-gen) |

   `char.nv` — отдельный файл для всех char-операций, текущих
   (`str.from(c char)`) и будущих (`is_alpha`, `is_digit`, `to_upper`,
   etc., когда понадобятся).

2. **f32 math** добавляется параллельно f64 в `runtime_registry.rs`.
   Subset: те же функции что f64 — `sqrt`, `cbrt`, `abs`, `ceil`,
   `floor`, `round`, `trunc`, `sin`, `cos`, `tan`, `asin`, `acos`,
   `atan`, `atan2`, `log`, `log2`, `log10`, `exp`, `pow`, `hypot`,
   `is_finite`, `is_nan`, `is_infinite`. C-имена `nova_f32_method_*`,
   реализация в `nova_rt/math.c` через `<math.h>` `f`-suffixed
   (`sqrtf`, `sinf`, etc.).

3. **string.nv audit.** Проверить полноту покрытия str API в реестре:
   - Должно быть: `len` (или `char_len`), `byte_len`, `slice`,
     `find`, `rfind`, `contains`, `starts_with`, `ends_with`, `split`,
     `trim`, `to_lower`, `to_upper`, `chars`, `bytes`, `char_at`,
     `is_empty`.
   - Если каких-то нет в registry — добавить (вместе с C-реализацией).
   - Если в emit_c.rs есть special-case'ы для str-методов которых нет
     в registry — это «invisible runtime knowledge», нарушение D82
     single source of truth. Должно быть либо в registry, либо
     удалено.

### Этапы Ф.8

## Ф.9 — API polish + formatting (2026-05-08, после Ф.8)

**Контекст.** После Ф.8 все 6 файлов auto-gen, но при ревью registry
выявились неудачные сигнатуры/имена и недостатки форматирования.
Ф.9 — прицельные правки реестра + emitter'а.

### Что меняется

#### 1. Mutating-методы возвращают `Self` (для chaining)

**Сейчас:**
```nova
export external fn StringBuilder mut @append(s str) -> ()
export external fn WriteBuffer  mut @write_byte(v byte) -> ()
export external fn WriteBuffer  mut @write_u32_be(v u32) -> ()
// ... все @write_*
```

**Станет:**
```nova
export external fn StringBuilder mut @append(s str) -> Self
export external fn WriteBuffer  mut @write_byte(v byte) -> Self
export external fn WriteBuffer  mut @write_u32_be(v u32) -> Self
// ... все @write_*
```

Везде `Self` — единый паттерн для chaining-методов на opaque types.
Короче чем явное имя типа и единообразно через все 24 `@write_*`,
все `@append`-перегрузки и любые будущие mutating-методы.

**Use-case:** chaining без промежуточных переменных:
```nova
sb.append("hello ").append(name).append("!")
wb.write_u32_be(magic).write_u16_be(version).write_bytes(payload)
```

**C-side изменения:**
- `Nova_StringBuilder_method_append` возвращает `Nova_StringBuilder*`
  (тот же `self`, не копия). Аналогично для `Nova_WriteBuffer_method_*`
  → `Nova_WriteBuffer*`.
- В обоих случаях return — тот же pointer что receiver, без аллокации.
- `ReadBuffer @read_*` **не меняется** — там return-type это значение
  (`u16`, `byte`, etc.), а не Self. Self для read-методов был бы
  бессмыслен.

#### 2. StringBuilder: операторный алиас `+` → `@append`

**Добавить в registry** для StringBuilder:
```nova
// `sb + s` — синоним sb.append(s). Mutates sb, возвращает sb.
fn StringBuilder mut @op_add(s str)  -> Self
fn StringBuilder mut @op_add(c char) -> Self
```

Это **алиас** — codegen для оператора `+` на StringBuilder receiver
эмитит вызов того же `Nova_StringBuilder_method_append`. Отдельной
C-функции не нужно. В реестре `c_name` тот же (`Nova_StringBuilder_method_append`),
но запись новая (отдельный module + receiver lookup для оператора).

**Нюанс:** оператор `+` на StringBuilder mutates left-hand operand.
Это отступление от обычной семантики `+` (immutable, allocation), но
оправдано: StringBuilder сам по себе — mutation-builder, копирующий
`+` был бы O(n²) regression. См. Q-stringbuilder-op-add — open
question нужно ли это вообще, или оставить только `@append`. Решение
— делать, потому что AI-friendly: знакомый `+` без learning curve.

#### 3. str: алиас `+` → `@concat`

**Сейчас:** `s1 + s2` для `str` — special-case intrinsic в codegen
(прямой call `nova_str_concat`).

**После Ф.9:** добавить в registry **явное** объявление:
```nova
// Конкатенация строк. O(a+b) — новая аллокация.
fn str @concat(other str) -> str
```

И оператор `+` на str-receiver — алиас на `@concat`. Codegen
для `s1 + s2` эмитит `Nova_str_method_concat(s1, s2)`. Старое имя
`nova_str_concat` либо переименовать, либо trampoline (решает
реализатор).

**Зачем:** убрать «invisible runtime knowledge» — конкатенация
строк должна быть видна в registry, как и любой другой method.
Закрывает дыру в D82 single-source принципе для str.

#### 4. str API rename: `@char_len` → `@len`

D26 говорит «s.len — длина в codepoint'ах, базовая длина строки».
Текущее имя `@char_len` отражает реализацию (codepoint = char), но
противоречит D26 spec'у.

**Изменение:**
- `@char_len` → `@len` в registry.
- C-имя `nova_str_char_len` → `nova_str_len` (или оставить старое
  как trampoline на новое — решает реализатор).
- `@byte_len` остаётся (явное имя для O(1) byte-length, FFI use-case).

#### 5. str API уточнение: `@chars() -> []char`

**Сейчас (если в registry):** `fn str @chars() -> []int` (или
`Iter[int]`). Кодпоинты возвращаются как `int` — это bootstrap-
compromise, потому что `char` тип ещё не везде поддержан
(Q-char-literals).

**После Ф.9:** `fn str @chars() -> []char`. Тип `char` теперь
first-class:
- Переход на `[]char` для возврата codepoint sequence.
- Если внутри codegen есть assumption «codepoint = nova_int» —
  она остаётся (внутреннее представление char это nova_int 32-bit),
  но API возвращает `char`-typed элементы.

**Альтернатива:** `Iter[char]` вместо `[]char` для lazy walking.
Решает реализатор; если выбрать `[]char` — eager allocation, проще
для bootstrap; `Iter[char]` lazier, но требует state-machine на
C-уровне. Plan 13 Ф.9 предписывает **`[]char`** как minimum;
переход на `Iter[char]` — future Q.

#### 6. ReadBuffer: добавить `read_char` / `read_str(n)` + явные `try_*` формы

**Контекст.** ReadBuffer сейчас покрывает только числовые типы
(`read_byte`, `read_uN_le/be`, `read_iN_*`, `read_fN_*`). Для парсинга
текстовых форматов (HTTP headers, CSV, простые protocols) нужны
codepoint-уровневые операции:

- **`@read_char`** — один codepoint (UTF-8 1-4 байта). Throw'ит
  `ReadBufferError.UnexpectedEnd` если буфер пуст или содержит
  неполную UTF-8 sequence.
- **`@read_str(n int)`** — `n` **codepoint'ов** (variable byte length).
  Throw'ит `UnexpectedEnd` если меньше `n` полных codepoint'ов в
  remaining bytes.

**Новые декларации в registry:**

```nova
// Один codepoint (UTF-8 1-4 байта). Throw'ит UnexpectedEnd.
fn ReadBuffer mut @read_char()      Fail[ReadBufferError] -> char

// Result-форма для @read_char.
fn ReadBuffer mut @try_read_char()  -> Result[char, ReadBufferError]

// n codepoint'ов. Throw'ит UnexpectedEnd если меньше n полных
// codepoint'ов в буфере.
fn ReadBuffer mut @read_str(n int)      Fail[ReadBufferError] -> str

// Result-форма для @read_str.
fn ReadBuffer mut @try_read_str(n int)  -> Result[str, ReadBufferError]
```

**C-side новых функций:**
- `Nova_ReadBuffer_method_read_char` — UTF-8 decode одного codepoint'а
  на текущей позиции, advance position.
- `Nova_ReadBuffer_method_try_read_char` — то же, но возвращает
  `NovaResult_char_ReadBufferError`.
- `Nova_ReadBuffer_method_read_str` — UTF-8 walk на `n` codepoint'ов,
  возвращает `nova_str` (slice в виде новой `str`-структуры — копия
  байт или view, решает реализатор).
- `Nova_ReadBuffer_method_try_read_str` — то же, Result-форма.

**Ошибки UTF-8.** Невалидный UTF-8 в буфере при `read_char` /
`read_str` — это **не** `UnexpectedEnd`. Нужен дополнительный
вариант `ReadBufferError`:

```nova
type ReadBufferError
    | UnexpectedEnd { wanted int, available int }
    | InvalidUtf8 { position int }       // ← новый
```

Throw'ится когда декодер встретил байт не из valid UTF-8 sequence
до того как набрал `n` codepoint'ов. Альтернатива — replace на U+FFFD
silently, но это плохо для бинарных протоколов где невалидность —
сигнал ошибки протокола.

#### 6.5 Отмена auto-derive `try_read_*` из `read_*`

**Plan 12 Ф.4.5** ранее ввёл правило: компилятор синтезирует
`@try_read_X() -> Result[X, E]` из `@read_X() Fail[E] -> X` без явной
декларации. Это правило **отменяется в Plan 13 Ф.9**.

**Причины отмены:**

1. **Hidden magic.** Программист видит в registry / в `read_buffer.nv`
   только Fail-форму, но при autocomplete'е в IDE появляется
   `try_read_X` неоткуда. AI-генерируемому коду тоже сложнее — он
   опирается на видимые декларации, не на скрытые правила.
2. **Edge cases.** UTF-8 ошибки в `read_char`/`read_str` (см. п.6)
   делают правило хрупким: synthesized `try_read_str(n)` нужно
   корректно мапить и `UnexpectedEnd`, и `InvalidUtf8` варианты.
   Если правило универсальное — оно должно покрывать все edge cases;
   если нет — какие-то методы synthesized, какие-то нет, путаница.
3. **D82 single source of truth.** Auto-derive противоречит принципу
   «всё что компилятор умеет — видно в registry». Skip-derived формы
   были «invisible» — они не лежат в `runtime_implemented` реестре,
   только синтезируются на лету. Отмена правила восстанавливает
   симметрию.

**Что меняется:**

- `runtime_registry.rs` получает **по две записи** на каждый
  `read_*` метод: Fail-form + Result-form. Дублирование, но явное.
- C-side: **по две функции** `Nova_ReadBuffer_method_read_X` (Fail) и
  `Nova_ReadBuffer_method_try_read_X` (Result). Сейчас обе уже есть
  в runtime (Plan 04 Ф.3 — auto-derive было только на Nova-уровне),
  так что на C-стороне ничего не меняется.
- `external_registry.rs` (compiler): убрать code-path который
  синтезирует Nova-AST для `try_read_X`. Если он эмитит обёртку — она
  теперь вычитывается из registry как обычная декларация.
- `std/runtime/read_buffer.nv` (auto-gen): после Ф.9 содержит обе
  формы для каждого `read_*` метода, явно.

**Что НЕ меняется:**

- **D73 From↔Into auto-derive остаётся.** Если в registry есть
  `T.from(s S) -> Self`, компилятор синтезирует `s.into() -> T` —
  это симметричное правило, прописанное в D73, не зависит от
  Plan 12 Ф.4.5.
- **D77 как spec концепция остаётся** — Result-форма параллельна
  Fail-форме семантически. Просто декларации **обе явно**, без
  скрытого синтеза.

**Plan 12 Ф.4.5 status:** ❌ ОТМЕНЕНО. Cross-link на это решение
добавляется в spec D82 + Plan 12 (отдельным коммитом).

#### 7. Форматирование: пустая строка между методами

**Сейчас (math.nv пример):**
```nova
// Квадратный корень. NaN на отрицательном.
export external fn f64 @sqrt() -> f64
// Кубический корень.
export external fn f64 @cbrt() -> f64
// Модуль (|x|).
export external fn f64 @abs() -> f64
```

**Станет:**
```nova
// Квадратный корень. NaN на отрицательном.
export external fn f64 @sqrt() -> f64

// Кубический корень.
export external fn f64 @cbrt() -> f64

// Модуль (|x|).
export external fn f64 @abs() -> f64
```

Пустая строка между каждой группой `// doc-comment + external fn`.
Делает файлы читаемыми при code-review (методы визуально разделены).

**Edge case:** для `read_*`/`try_read_*` пар пустая строка тоже
**между парой** (после Result-формы), не внутри:
```nova
// u16 big-endian. Throw'ит ReadBufferError при недостатке байт.
export external fn ReadBuffer mut @read_u16_be() Fail[ReadBufferError] -> u16

// u16 big-endian. Result-форма (Ok(value) или Err(UnexpectedEnd)).
export external fn ReadBuffer mut @try_read_u16_be() -> Result[u16, ReadBufferError]

// u32 little-endian. Throw'ит ReadBufferError при недостатке байт.
export external fn ReadBuffer mut @read_u32_le() Fail[ReadBufferError] -> u32
```

**Реализация:** в emit_runtime_stubs renderer — `format!("{}\n\n",
external_fn_decl)` вместо `format!("{}\n", ...)`. Один тривиальный
fix, проверяется детерминизмом regen + manual eyeball'ом.

### Risks Ф.9

1. **Existing user-код использует `sb.append(s)` как statement.** После
   изменения return-type на StringBuilder, это всё ещё валидно
   (return-value просто игнорируется). Не ломает. **Но**:
   `mut`-receiver + non-`()` return может вести себя странно если
   у lhs нет `let mut sb = ...` — type-checker должен принять, но
   возможны edge cases.
2. **`+` ambiguity.** Когда `sb + s` — какой из двух registry entry
   выбрать (`StringBuilder.@op_add(str)` vs `StringBuilder.@op_add(char)`)?
   Параметр-type mangling (Plan 11 Ф.3). Без него — collision,
   нужно single overload или явное разделение в codegen.
3. **`char_len` rename ломает existing user-код.** Сейчас в std/
   используется `s.char_len`? Grep до изменения; миграция сразу в
   том же коммите.
4. **`[]char` vs `[]int`.** Если codegen эмитит boxed pointer'ы
   (Plan 13 Ф.1 inventory отметил это для `NovaArray_nova_int`), то
   `[]char` тоже unboxed-нельзя. Нужен `NovaArray_nova_char` (или
   reuse `nova_int` через alias). Реализатор решает.

### Зависимости Ф.9

- ✅ Ф.8 (per-type декомпозиция; иначе не куда добавлять `op_add`).
- ✅ Plan 11 Ф.3 (parameter-type mangling) для overloaded `op_add(str)`/`op_add(char)`.
