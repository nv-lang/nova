# Plan 91.10 — Retract D163 `needs <Cap>` syntax

> **Status:** 🟢 IN PROGRESS 2026-05-30
> **Parent:** [Plan 91 — std MVP for 0.1](91-stdlib-mvp-for-0.1.md)
> **Trigger:** При добавлении `str.from_bytes_unchecked_steal(consume bytes []u8) -> str`
> для zero-copy steal в `StringBuilder.@as_str()` (Plan 91 Ф.2.6 followup),
> D163-missing-cap fire'нул на этом external fn. Анализ выявил архитектурный
> design issue в D163.

## Зачем

D163 (Plan 100.5, 2026-05-25) ввёл `needs <Cap>` clause на `external fn` —
mandatory capability declaration когда FFI принимает/возвращает consume-типы.
Цель: capability-based security, prevent ambient authority.

**Проблема 1: conflation orthogonal concerns.**
`consume` — про ownership/linearity (контракт компилятора с типом).
`needs Cap` — про authority (кому разрешено).
D163 их жёстко связывает: "consume в external = mandatory cap". Это
дизайн-смешение двух разных свойств.

Конкретный pain: `str.from_bytes_unchecked_steal(consume []u8) -> str` —
pure memory ops, безопасная FFI, **не трогает OS**. Но имеет `consume` param →
D163 требует `needs <Cap>`. Семантически неверный gate; пришлось поставить
`needs Sys` placeholder'ом (формально неправильно).

**Проблема 2: capability — это redux эффекта.**
Nova **уже имеет** effect system (Plan 33, `Fail[E]`, `Time`, `Mem`).
Capability — structurally идентична effect-без-операций:
- annotation на signature
- propagation вверх по call graph
- statically tracked
- (effects ещё имеют operations + handlers)

Effect handlers literature (Koka, Eff): capabilities **ARE** effects.
Nova вводит параллельную redundant tracking систему. Без явного benefit.

**Проблема 3: stdlib amnesty убивает безопасность.**
`is_stdlib_runtime_module` exempt'ит весь stdlib от D163. Любой stdlib
wrapper обходит check. Security guarantee leaks. Worst of both worlds:
overhead для user code + no security gain.

**Проблема 4: coarse granularity = security theatre.**
`Fs` = "может всё с файлами". Не "может читать `/tmp/`". Реальная security
требует fine-grained (path/port/etc) или OS-level sandboxing. `needs Sys`
без fine-grain — bureaucratic overhead без teeth'ов.

## Scope

**Входит:**

1. **Parser:** удалить parsing `needs <CapName>, <CapName2>` clause после
   `external fn` signature. Заменить на hard error с migration hint
   ("D163 retracted; use effect declarations if capability tracking needed").
2. **Types:** удалить `check_external_fn_needs_caps` (D163 validation).
   Удалить use-сайты `fd.needs_caps.is_empty()` (D82 whitelist logic).
3. **AST:** `FnDecl.needs_caps: Vec<String>` — **сохранить** как always-empty
   field для минимизации churn в sum_schema_registry (~10 construction sites).
   Поле помечено deprecated в комментарии. Удаление поля — followup
   `[M-91.10-remove-needs-caps-field]`.
4. **Codegen:** удалить branching на `f.needs_caps.is_empty()` (всегда true
   после retract). Simplify forward-decl + emit paths.
5. **Fixtures:** drop `needs Fs/Sys/Net` clauses из:
   - `nova_tests/plan100_5/external_consume_*.nv` (5 файлов)
   - `nova_tests/plan100_7/file_open_read_close.nv`, `mutex_lock_release.nv`,
     `socket_listen_accept.nv` (3 файла)
   - `nova_tests/plan100_1/parser/parser_err_external_consume.nv` (1 файл)
   - Удалить файл `external_consume_err_missing_cap.nv` (тест проверял D163;
     после retract — обнолен).
6. **Spec:** retract D163 в `spec/decisions/02-types.md`. Mark как
   RETRACTED (Plan 91.10, 2026-05-30) с rationale + link на capability=effect
   insight.
7. **README spec:** удалить D163 из active D-blocks index, добавить в
   retracted/superseded section.

**НЕ входит (followup'ы):**

- Реальная имплементация Fs/Net/Sys as effects (если когда-то понадобится).
  Сейчас нет requirement'а — effect system существует, можно добавить formal
  effect declarations когда появится use case. Followup
  `[M-91.10-fs-net-effects-formal]`.
- Удаление `needs_caps` поля из FnDecl AST. Followup
  `[M-91.10-remove-needs-caps-field]`.
- Decoupling `consume` checking от capability (уже decoupled — после retract
  consume стоит сам по себе, как изначально задумано Plan 100.1).

## Принцип

Capability tracking — **не отдельная concern**, а special case of effect
tracking. Effect-system Nova уже даёт structured propagation, formal
declarations, handlers. Дублировать через `needs <Cap>` — design redux.

Если в будущем понадобится capability gating для конкретного OS-ресурса —
вводить как formal effect declaration (`type Fs effect { ... }`) с handler'ом
в каком-то корне (`with Fs = real_fs_handler { ... }`).
