# Plan 57.E.4 — Memory bandwidth measurement (design sketch, Linux-only)

> **Status:** design-sketch (not implemented в этой phase).
> **Estimate when implemented:** ~200 LOC, 1-2 dev-days.
> **External dependencies:** Linux kernel CONFIG_X86_INTEL_RDT,
> CAP_PERFMON or root, Intel CPU with MBM support (Xeon Skylake+,
> Core 10th gen+).

## Motivation

Wall-clock + CPU instructions (Plan 57.B.4 + 57.C.4) — temporal
metrics. Memory bandwidth — spatial metric: показывает сколько byte/sec
течёт между L3 / DRAM. Полезно для:

- HashMap / array workloads где cache pressure dominates.
- Allocation-heavy code (GC stress benches).
- Identifying memory-bound vs compute-bound regressions.

Real example: HashMap.@clone() regression может быть либо
"больше instructions" (codegen change), либо "выше memory bandwidth"
(extra alloc), либо обе. Different fixes — separate signal needed.

## Hardware support

### Intel

- **MBM (Memory Bandwidth Monitoring)** — Skylake+ Xeon, 10th gen
  Core (Comet Lake+). Kernel: CONFIG_INTEL_RDT.
- Events:
  - `INTEL_PT_BW_LOCAL` — local memory bandwidth.
  - `INTEL_PT_BW_REMOTE` — remote (NUMA) bandwidth.
  - `INTEL_PT_L3_OCC` — L3 cache occupancy (bytes).

### AMD

- **QoS / Memory Bandwidth Enforcement** (Zen 2+). Similar event
  exposure via `resctrl` FS.

### Other

- ARM PMU events depending on vendor (less standardized).
- macOS/Windows — нет общедоступного API.

## Architecture

```
Linux only:

bench-exe ──► perf_event_open(RAW MBM event) ──► counter fd
              ioctl RESET/ENABLE/DISABLE
              read counter per-sample

Wraps вокруг measure loop (как CPU instructions в Plan 57.C.4):
  reset → enable → measure batch → disable → read.

Output: __MBM_SAMPLE__ <bytes> markers, OR JSON extension:
  "memory_bandwidth_bytes_per_iter": [...]
```

## API sketch

```c
// nova_rt/bench.h Linux-only block:

#if defined(__linux__)
// Detected at runtime via PERF_TYPE_RAW event code probe.
static int _nova_bench_mbm_fd = -1;

static inline void nova_bench_mbm_start(void) {
    if (nova_bench_env_u64("NOVA_BENCH_MEASURE_MBM", 0) == 0) return;
    struct perf_event_attr attr;
    memset(&attr, 0, sizeof(attr));
    attr.type = PERF_TYPE_RAW;
    attr.size = sizeof(attr);
    // MBM event code (Intel uncore_imc/cas_count_read).
    // Real code probed from /sys/devices/uncore_imc_*/format/event.
    attr.config = 0x0304;  // example — actual depends on CPU.
    attr.disabled = 1;
    attr.exclude_kernel = 0;  // MBM measures total, not user-only.
    long fd = perf_event_open(&attr, 0, -1, -1, 0);
    if (fd < 0) {
        fprintf(stderr, "nova bench: MBM perf_event_open failed (need root/CAP_PERFMON)\n");
        return;
    }
    _nova_bench_mbm_fd = (int)fd;
}
```

Rust CLI: `nova bench run ... --measurement bandwidth` flag.
Detection: `nova bench mbm-check` parallel к `cpu-instr-check`.

## Implementation plan

1. **Phase 1: detection + probe (~50 LOC).**
   - Read /sys/devices/uncore_imc_*/format/event для real event codes.
   - `nova bench mbm-check` — detects support + prints CPU model + event
     codes.

2. **Phase 2: runtime integration (~80 LOC).**
   - Add к bench.h Linux block parallel к instr counter.
   - Per-sample reset+enable+disable+read.
   - Emit JSON field `memory_bandwidth_bytes_per_iter`.

3. **Phase 3: CLI + AMD adapter (~70 LOC).**
   - `--measurement bandwidth` flag.
   - AMD QoS adapter (different event codes, same perf_event_open API).
   - Result aggregation в schema + dashboard display.

## Risks

- **R1: Hardware fragmentation** — different CPU generations require
  different event codes. Mitigation: runtime probe via sysfs.
- **R2: Privilege requirement** — MBM events require CAP_PERFMON or
  root. Mitigation: graceful warn + fallback к wall-clock-only.
- **R3: Kernel config** — older kernels lack INTEL_RDT support.
  Mitigation: detect at startup; clear error message.
- **R4: macOS/Windows unsupported** — bench results lose spatial axis
  on those platforms. Acceptable since Linux dominates CI.

## Acceptance criteria (when implemented)

- [ ] `nova bench mbm-check` correctly reports MBM availability + event
      codes on Skylake+ Xeon.
- [ ] `--measurement bandwidth` adds `memory_bandwidth_bytes_per_iter`
      field в JSON output.
- [ ] HashMap.@clone bench shows distinct bandwidth signal vs wall-clock.
- [ ] Graceful fallback с no-MBM kernel/CPU.
- [ ] AMD QoS path verified on at least one Zen 3+ CPU.

## Alternative considered

**Cache miss counters** (`PERF_COUNT_HW_CACHE_MISSES`) — easier
hardware support, less direct measure of bandwidth. Useful as proxy
but doesn't directly answer "how much DRAM traffic". Could implement
parallel к MBM as Phase 1b (fewer privilege requirements).

## When to implement

После accumulated evidence that memory-bound regressions нужны
distinguishing signal. Если HashMap/GC benches already provide via
alloc counters (Plan 57.MVP + 57.C.3 heap sampler) — defer.
