# Plan 91.7 — Array methods cleanup + canonical `.new()` (D180)

> **Подплан Plan 91** ([91-stdlib-mvp-for-0.1.md](91-stdlib-mvp-for-0.1.md)).
> Статус: 🔴 OPEN (не начат).

## Мотивация

Текущий API массивов `[]T`:
- mut-методы возвращают `()` — нельзя fluent chain (`arr.push(1).push(2)` не работает)
- `@slice(from, to)` дублирует синтаксис `arr[a..b]` (Plan 96)
- sort/min/max/binary_search определены **только для `[]int`** — implementation shortcut из Plan 91.3
- Нет канонического конструктора `[]T.new()` / `[]T.with_capacity(n)` (приходится `let mut a []T = []; a.reserve(n)`)

Дополнительно — отсутствует **конвенция default-конструкторов**. Сейчас:
- `StringBuilder.new()` — explicit
- `HashMap.new()` — explicit
- `int`, `bool`, `str`, `[]T`, `Option` — нет `.new()` вообще, юзер пишет `0`, `false`, `""`, `[]`, `None` напрямую

## Scope

### Часть A — `-> @` для mut методов массива

Все мутирующие методы `[]T` должны возвращать `@` (Plan 77/D131 fluent API),
чтобы поддерживать chaining:

| Метод | Было | Стало |
|---|---|---|
| `@push(v T)` | `-> ()` | `-> @` |
| `@reserve(extra int)` | `-> ()` | `-> @` |
| `@truncate(n int)` | `-> ()` | `-> @` |
| `@fill(v T)` | `-> ()` | `-> @` |
| `@copy_from(src readonly []T)` | `-> ()` | `-> @` |
| `@extend_from(src readonly []T)` | `-> ()` | `-> @` |
| `@insert_from(i int, src readonly []T)` | `-> ()` | `-> @` |
| `@copy_within(src_from, dst_from, len)` | `-> ()` | `-> @` |

`@pop()` остаётся `-> Option[T]` (возвращает значение, не self).
`@get(i)` остаётся `-> Option[T]`.

**Реализация:** C-функции в `array.h` остаются void; на Nova-стороне можно
обёртку, либо emit_c.rs учитывает `-> @` flag для прямых array builtin-вызовов.

### Часть B — удалить `@slice`

`@slice(from, to) -> []T` дублирует `arr[a..b]` (Plan 96, zero-copy view).
Один способ — лучше. Удаление: декларации в `vec.nv` / runtime_registry + всех
вызовов в stdlib.

### Часть C — generic sort/min/max/binary_search

Текущая реализация в `std/sort.nv` — только для `[]int` (Plan 91.3 shortcut).
Распространить на `[T Ord]`:

```nova
fn[T Ord] []T mut @sort() -> @
fn[T Ord] []T @binary_search(target T) -> Option[int]
fn[T Ord] []T @min() -> Option[T]
fn[T Ord] []T @max() -> Option[T]
fn[T] []T mut @sort_by(cmp fn(T, T) -> Ordering) -> @
```

Бэкенд: monomorphization per T через Plan 101 (`fn[T Ord]` префикс). `Ord` —
protocol с `@compare(other T) -> Ordering`.

**Совместимость:** новый generic `fn[T Ord] []T @sort()` НЕ ломает existing
concrete `fn []int @sort()` — overload resolution выбирает specific over generic.
После добавления generic версии — concrete `[]int` можно удалить (или оставить
как fast-path).

### Часть D — канонические `.new()` через Nova-код (D180)

Все определения **в Nova-коде** stdlib, **без compiler magic**:

```nova
// std/runtime/numeric.nv (или новый файл)
export fn int.new()  -> Self => 0
export fn u8.new()   -> Self => 0
export fn u16.new()  -> Self => 0
export fn u32.new()  -> Self => 0
export fn u64.new()  -> Self => 0
export fn i8.new()   -> Self => 0
export fn i16.new()  -> Self => 0
export fn i32.new()  -> Self => 0
export fn i64.new()  -> Self => 0
export fn f32.new()  -> Self => 0.0
export fn f64.new()  -> Self => 0.0
export fn bool.new() -> Self => false

// std/runtime/string.nv
export fn str.new() -> Self => ""

// std/prelude/core.nv
export fn[T] Option[T].new() -> Self => None

// std/collections/vec.nv (или std/prelude/collections.nv)
export fn[T] []T.new() -> Self => []
export fn[T] []T.with_capacity(n int) -> Self {
    let mut a []T = []
    a.reserve(n)
    a
}
```

**Зависимость:** парсер должен поддерживать `[]T` как receiver в static-call.
Сейчас работает для `Option[T]`, `str`, `StringBuilder` (named types). Для `[]T`
(`TypeRef::Array`) — нужно проверить парсер; вероятно single small feature.

**Не делаем auto-generation:**
- `char`, `Result[T, E]` — нет канонического default (skip)
- user records / sum types / consume types — explicit `.new()` (forced discipline)
- protocols / fn types / external — N/A

### Часть E — D180 convention в спеке

Новый D-block в `spec/decisions/01-foundations.md` или `02-types.md`:

```markdown
## D180. Convention: canonical `.new()` constructors

**Status:** convention (not enforcement).

stdlib предоставляет `.new()` для типов с **единственным очевидным** default:
- числовые (`int`, `u8..u64`, `i8..i64`, `f32/f64`) → `0` / `0.0`
- `bool` → `false`
- `str` → `""`
- `[]T` → `[]` (empty array; works for any T including consume)
- `Option[T]` → `None`

Также `[]T.with_capacity(n)` — с pre-allocated capacity.

**Для своих типов** разработчик пишет `.new()` явно. Компилятор НЕ автогенерирует
для user records / sum types / consume types. Это design discipline:

1. Явный конструктор виден в `nova doc` и IDE.
2. Имена кодируют намерение (`User.new(name, email)` vs `User.guest()`).
3. Валидация инвариантов в момент создания.
4. Эволюция типа: добавление поля заставляет обновить конструктор → good failure.

**НЕ имеют canonical `.new()`** (compile error при попытке):
- `char` (`'\0'` сомнителен)
- `Result[T, E]` (`Ok` или `Err`? ambiguous)
- user-defined types — по конвенции этого блока
- protocols, fn types, external/opaque
```

## Acceptance criteria

- [ ] **A:** 8 mut-методов `[]T` декларированы `-> @`; ` arr.push(1).push(2).reserve(10)` компилируется и работает
- [ ] **B:** `@slice` удалён; все вызовы `.slice(a, b)` в stdlib заменены на `[a..b]`
- [ ] **C:** Generic sort/min/max/binary_search работают для `[]int`, `[]str` (через `str.compare()` из D178), `[]u64` и т.п.; концевые тесты в `nova_tests/plan91_7/`
- [ ] **D:** Все 12 канонических `.new()` declared в stdlib; работают позитивные тесты `int.new() == 0`, `[]int.new() == []`, `Option[int].new() == None` и т.д.
- [ ] **D parser support:** `[]T.new()` парсится корректно (если требуется правка парсера — отдельный коммит)
- [ ] **E:** D180 в `spec/decisions/`; ссылка из Plan 91.7
- [ ] Negative tests: `MyRecord.new()` без explicit декларации → compile error с hint на D180
- [ ] Full regression: nova test без новых failures
