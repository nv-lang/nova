## Ф.1 — Quick wins (sound + perf, ~3-4 дня)

Независимые задачи, можно делать параллельно. Закрывают самые видимые
оставшиеся gap'ы.

### Ф.1.1 — Ghost code elimination (V5 закрытие)

**Файлы:** `compiler-codegen/src/codegen/emit_c.rs`

**Проблема (V5).** `ghost let x = ...` сейчас emit'ится в C-code в debug
режиме. Должен **полностью erase'иться** в codegen — паритет с Verus
ghost types. Сейчас runtime code зависит от ghost-вычислений → нарушение
zero-cost ghost-code.

**Что нужно.**
1. В `emit_let_stmt` (или аналогичном месте) проверить `ld.is_ghost`.
2. Если true — emit ничего (skip полностью).
3. Если ghost-переменная используется в runtime code (не в `assert`/`assume`/
   `invariant`) — **compile error** «ghost binding cannot be used at runtime».
4. В `infer_expr_c_type` для ghost-only выражений — error, если попадают
   в non-ghost context.

**Acceptance.** `ghost let n = compute(); println(n)` → compile error.
`ghost let n = compute(); assert(n >= 0)` → OK, ничего не emit'ится.

**Трудоёмкость:** ~1 день.

### Ф.1.2 — Trigger patterns proper (D.1.4 V2)

**Файлы:**
- `compiler-codegen/src/verify/ir.rs` — расширить `SmtTerm::Forall`
- `compiler-codegen/src/verify/encode.rs` — заполнять pattern при кодировании
- `compiler-codegen/src/verify/backend/z3.rs` — `Z3_mk_pattern` + forall с pattern

**Проблема.** D.1.4 stub только пишет eprintln warning. SmtTerm IR
не имеет поля для триггера → Z3 квантификатор без `:pattern` →
heuristic instantiation → возможный timeout.

**Что нужно.**
1. `SmtTerm::Forall(binders, patterns: Vec<Vec<SmtTerm>>, body)` — N
   patterns (multi-trigger), каждый — список терм-выражений.
2. В `encode_expr_with_ctx` для `Forall`/`Exists`: walk body, найти
   ground subterm с App(uf, ..) содержащий bound var → emit как trigger.
3. В Z3 backend: для каждого pattern — `Z3_mk_pattern(ctx, n, terms)`,
   собрать в массив, передать в `Z3_mk_forall_ex` (с pattern-параметром).
4. `TrivialBackend` игнорирует patterns (как и сейчас).
5. Регрессия: все существующие forall-тесты должны продолжать работать.

**Acceptance.** `forall i in 0..n : xs[i] >= 0` — Z3 видит trigger
`xs[i]` (UF `_array_get(xs, i)`), instantiate'ит per i. Замер: ранее
timeout-prone тест проходит за < 100ms.

**Трудоёмкость:** ~1 день.

### Ф.1.3 — Loop decreases SMT verify

**Файлы:** `compiler-codegen/src/verify/pipeline.rs`

**Проблема.** D.0.3 хранит `decreases: Option<Box<Expr>>` на циклах,
но pipeline проверяет только fn-level decreases (D.0.4). Loop-level
`while c decreases m { body }` парсится, но SMT не верифицирует
`m_post < m_pre` на каждой итерации.

**Что нужно.** Расширить `verify_fn` после loop invariants check:
1. `collect_loop_decreases_in_body(body) -> Vec<(loop_span, decreases_expr, body_block)>`
2. Для каждого: encode `decreases_expr` как `dec_pre`, encode body to find
   мутации loop-var, encode `decreases_expr` снова как `dec_post`, assert
   `dec_post < dec_pre AND dec_pre >= 0`.
3. V1 simplification: только simple while loops с counter-decrement
   (`i = i + 1` / `i = i - 1`); complex body — Unknown.

**Acceptance.** `while i > 0 decreases i { i = i - 1 }` → Z3 proves termination.

**Трудоёмкость:** ~1-2 дня.

## Ф.2 — Loop havoc + invariant preservation (~2 дня)

**Файлы:**
- `compiler-codegen/src/verify/pipeline.rs` — havoc-based loop encoding
- `compiler-codegen/src/verify/encode.rs` — потенциально новые helpers

**Проблема.** D.0.3 entry-check доказывает что invariant выполняется при
входе в цикл (под `requires`), но НЕ доказывает preservation
(invariant survives iteration). Это — half measure: реальная Dafny/Verus
семантика инвариантов требует **havoc + preservation**.

**Алгоритм havoc + preservation (стандарт Dafny/Verus):**
1. Havoc: для всех mutable vars, потенциально модифицируемых телом
   цикла → fresh SMT vars (как будто значения произвольные).
2. Assume invariant + loop condition (на havoc'ed state).
3. Symbolic exec body.
4. Assert invariant still holds после body.
5. Если success: assume `!cond AND invariant` после loop.

**V1 scope (этот план):**
- Frame analysis: walk body для нахождения assigned vars (over-approx OK).
- Havoc только locals (не params — params не mutate в Nova 33.4 scope).
- Symbolic exec body — straight-line only (no nested loops/if).
- Body must be encodable → если нет, fall back to entry-check (current behavior).

**V2/V3 scope (будущее):**
- Nested loops, branches в body.
- Frame-aware havoc (только vars в `modifies` clause).

**Acceptance.** Классика: `while i < n invariant 0 <= i <= n { i = i + 1 }` →
Z3 proves invariant preservation + post-condition `i == n`.

**Трудоёмкость:** ~2 дня.

## Ф.3 — Purity SCC inference (~1.5 дня)

**Файлы:**
- `compiler-codegen/src/types/mod.rs` (или новый purity_infer.rs)

**Проблема (V3).** Каждая helper-функция должна быть annotation'ed
`#pure` вручную. Это boilerplate; пользователь вынужден вспоминать.
В Rust `const fn` инферится по transitive closure безопасных операций.

**Что нужно.** SCC-based inference:
1. Построить call-graph всех fn в module.
2. Найти SCC (Tarjan).
3. Для каждой SCC: проверить что все fn в ней — pure-eligible (no FFI,
   no IO, no mut params, no effect calls кроме pure_view).
4. Если вся SCC eligible → mark all as `#pure` (если уже не помечена).
5. Diagnostic: warn если user написал `#pure` явно а SCC infer бы и так
   нашёл (разрешено, просто избыточно).

**Boundary.**
- Mutual recursion → одна SCC, либо all pure либо none.
- External fn в SCC → SCC non-pure.
- User-explicit `#pure` на impure fn → compile error (раньше было).

**Acceptance.** `fn helper(x int) -> bool => x > 0` без `#pure` —
auto-detected as pure, можно использовать в `requires helper(n)`.

**Трудоёмкость:** ~1.5 дня.

## Ф.4 — Dafny-parity features (~4 дня)

Две большие фичи которые двигают Nova на уровень Dafny/Verus
по выразительности доказательств.

### Ф.4.1 — Lemma functions (~2 дня)

**Файлы:**
- `compiler-codegen/src/lexer/token.rs` — `KwLemma`
- `compiler-codegen/src/ast/mod.rs` — `Item::Lemma(LemmaDecl)` или
  `FnDecl.kind: FnKind { Fn, Lemma }`
- `compiler-codegen/src/parser/mod.rs` — `parse_lemma_decl`
- `compiler-codegen/src/verify/pipeline.rs` — `verify_lemma`
- `compiler-codegen/src/codegen/emit_c.rs` — lemma не emit'ится в runtime

**Семантика.** `lemma proof_X(x int) requires P(x) ensures Q(x) { proof_body }`:
- Тело **обязано** SMT-верифицировать (как `#verify fn` но строже —
  unknown == fail).
- НЕ emit'ится в runtime (как ghost code).
- Может быть **applied** в других контрактах через `apply proof_X(value)`.
- Body может содержать ghost let, assert, calc (см. Ф.4.2).

**Отличие от `axiom`.**
- `axiom` — unproven assumption (записан в effect-блок).
- `lemma` — proven proof term (тело верифицируется).

**Trust model.** Если lemma верифицируется → её ensures можно использовать
как assumption. Это делает proof-process модульным (как в Dafny).

**Acceptance.**
```nova
lemma sum_nonneg(n int)
    requires n >= 0
    ensures n * (n + 1) / 2 >= 0
{
    // Тривиально: Z3 видит n >= 0 → n*(n+1) >= 0 → /2 >= 0
}

fn use_sum(k int) -> int
    requires k >= 0
    ensures result >= 0
{
    apply sum_nonneg(k)  // Активирует lemma в SMT scope
    k * (k + 1) / 2
}
```

**Трудоёмкость:** ~2 дня.

### Ф.4.2 — Calc proofs (~2 дня)

**Файлы:**
- `compiler-codegen/src/lexer/token.rs` — `KwCalc`
- `compiler-codegen/src/ast/mod.rs` — `ExprKind::Calc { steps: Vec<CalcStep> }`
- `compiler-codegen/src/parser/mod.rs` — `parse_calc`
- `compiler-codegen/src/verify/pipeline.rs` — `verify_calc_steps`

**Семантика (Dafny-style).**
```nova
calc {
    x * 2
    == { lemma_double_eq(x) }
    x + x
    == { /* trivial */ }
    2 * x
}
```
Каждый шаг `== { hint }` — Z3 должен доказать что `previous == next`,
используя `hint` (опциональный lemma apply / arithmetic).

**Алгоритм verify.**
1. Парсить как chain of (term, op, hint, next_term).
2. Для каждого step: assert hint в SMT scope, прове `prev op next`.
3. Если все steps proven → result = `first op last` (transitive).

**Operator support V1:** `==`, `<=`, `<`, `>=`, `>` (нужна consistent
chain — все `==` или все `<=` без strictness conflict).

**Acceptance.** Calc-proof в lemma:
```nova
lemma triangle_eq(n int)
    requires n >= 0
    ensures sum(n) == n * (n+1) / 2
{
    calc {
        sum(n)
        == { /* def of sum */ }
        n + sum(n-1)
        == { triangle_eq(n-1) }  // Recursive lemma application
        n + (n-1) * n / 2
        == { /* arithmetic */ }
        n * (n+1) / 2
    }
}
```

**Трудоёмкость:** ~2 дня.

## Ф.5 — Liskov V1 (~2 дня)

**Файлы:**
- `compiler-codegen/src/ast/mod.rs` — `EffectMethod.contracts: Vec<Contract>`
- `compiler-codegen/src/parser/mod.rs` — parse contracts inside effect-block
- `compiler-codegen/src/verify/pipeline.rs` — extend `verify_handlers`

### Ф.5.1 — EffectMethod contracts AST + parser (~1 день)

**Проблема.** `EffectMethod` сейчас не имеет `contracts: Vec<Contract>`.
P1-4 (Liskov) blocked на этом.

**Что нужно.**
1. Добавить `pub contracts: Vec<Contract>` в `EffectMethod`.
2. Parser: после signature method'а в effect-block — accept
   `requires`/`ensures` clauses.
3. Type-check: contracts видят effect's pure_view ops, params, return type.

**Acceptance.**
```nova
type Db effect {
    SetBalance(id AccountId, amt money)
        requires amt >= 0
        ensures balance(id) == amt
}
```

### Ф.5.2 — Liskov SMT verify (~1 день, после Ф.5.1)

**Что нужно.** В `verify_handlers` (P0-1 infrastructure):
1. Для каждого `with #verify E = handler` binding:
2. Для каждого `EffectMethod` op в E с contracts:
3. Найти `handler.method` соответствующий op.
4. Verify Liskov:
   - **Contravariant pre:** `handler.requires` должен быть **слабее** чем `op.requires`
     (impl принимает шире input). SMT: `op.requires => handler.requires`.
   - **Covariant post:** `handler.ensures` должен быть **сильнее** чем `op.ensures`
     (impl даёт уже guarantee). SMT: `handler.ensures => op.ensures`.
5. Если post-violation → Disproved с counterexample.

**Acceptance.** Effect с `op requires x > 0` и handler с `op requires x >= 0`
(weaker pre) → OK (Liskov-correct refinement).

**Трудоёмкость:** ~1 день (Ф.5.1) + ~1 день (Ф.5.2).

## Ф.6 — post(Action)(view) symbolic execution V2 (~3 дня research)

**Файлы:**
- `compiler-codegen/src/verify/handler_exec.rs` — НОВЫЙ модуль
- `compiler-codegen/src/verify/pipeline.rs` — call site в `verify_handlers`

**Проблема.** P0-1 V1 (P0-1 в Plan 33.4) только для static axioms.
`post(Action(args))(view) == X` axioms — research-grade.

Это **уникальная Nova-фича**: никто из mainstream verifiers не верифицирует
side-effecting handler bodies против effect axioms. Verus трейты, Dafny
classes — не моделируют side-effects на уровне axiom.

**Алгоритм (V2 scope — linear bodies):**
1. Parse axiom formula: detect pattern `post(Action(action_args))(view(view_args)) == rhs`.
2. Найти `handler.action` body (HandlerMethodBody::Expr или Block).
3. Создать **symbolic state**: для каждого field в handler closure-capture →
   fresh SMT var (`state_field_pre`, `state_field_post`).
4. Symbolic exec body:
   - `let x = e` → SMT var declaration `x` равен encoded(e).
   - Assignment to captured var → update `_post` var.
   - Branches → reject в V2 (only linear).
   - Return → record return value.
5. Encode `view(view_args)` через post-state symbolically.
6. Assert `view_encoded == rhs_encoded`; check_sat.
7. UNSAT → Verified; SAT → Counterexample with witness.

**Boundary.**
- Linear body only (no if/match/loop) — V2.
- Handler closure captures known statically (через AST walk).
- FFI/IO body → reject (require `#trusted`).

**V3 (будущее):** branching bodies (path-sensitive analysis), nested handlers.

**Acceptance.** Bank account R4 example:
```nova
type Db effect {
    pure_view balance(id AccountId) -> money
    SetBalance(id AccountId, amt money)
    axiom after_set(id, x) =>
        post(SetBalance(id, x))(balance(id)) == x
}

with #verify Db = handler Db {
    SetBalance(id, x) => { state.set(id, x) }  // Correct
    balance(id) => state.get(id)
}
// → Verified

with #verify Db = handler Db {
    SetBalance(id, x) => { state.set(id, x + 1) }  // Wrong!
    balance(id) => state.get(id)
}
// → Counterexample: post(SetBalance(id=A1, x=100))(balance(A1)) == 101 != 100
```

**Трудоёмкость:** ~3 дня (research + implementation + tests).

## Простые правила оценки готовности (production grade)

После Plan 33.5 verifier должен удовлетворять:
1. **Soundness:** все `Proven` результаты — реально доказаны (нет silent skips).
2. **Completeness baseline:** все Dafny imperative tutorial patterns верифицируются.
3. **Performance:** quantifier-heavy fn — < 1s SMT time на typical hardware.
4. **UX:** `nova check` даёт actionable diagnostic при `Disproved` (counterexample
   с witness-значениями).
5. **Modularity:** lemma + apply работает; circular dependencies detect'ятся.
6. **Effect-aware:** `#verify` handler гарантирует семантическую корректность
   против effect axioms (включая `post(...)`).
