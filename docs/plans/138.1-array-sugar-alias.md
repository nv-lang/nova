> **Создан:** 2026-06-10.  **Статус:** 📋 PLANNED.
> **Декомпозиция:** `[M-138-array-sugar-alias]` (Plan 138 Ф.5-Ф.6 deferred).
> **Эстимат:** ~4-7 dev-day (большой compiler refactor + test surface).
> **Model:** Opus + Thinking ON (рискованный codegen).
> **Worktree:** `nova-p138-1`, branch `plan-138.1` (off `plan-138`).
> **Зависит от:** Plan 138 Ф.1-Ф.4 ✅ (Vec @index/@get/mut@index + iter-first),
> Plan 131 ✅ (Vec[T] pure-Nova impl), Plan 90.1 ✅ ([]T extend-family).

## Цель

Сделать `[]T` **чистым синтаксическим сахаром** над `Vec[T]` — тип, целиком
реализованный на Nova (`std/collections/vec_owned.nv`). Конечное состояние:

- `[]T` ≡ `Vec[T]` на уровне типов: `fn f(a []int)` принимает `Vec[int]`.
- Array-литерал `[1, 2, 3]` строит `Vec[int]` (типизированный, **без** int-erasure).
- **`NOVA_ARRAY_DECL`/`NOVA_ARRAY_IMPL` C-macro полностью удалены** — никакого
  built-in C-массива. `Vec[T]` использует только `*mut T` + `nova_alloc`.
- D239 («`[]T` as sugar over `Vec[T]`») реализован; D144 закрыт полностью.

**Принцип (ограничение пользователя):** `Vec[T]` должен быть полностью на `.nv`.
Нет C-fallback'а «оставим `NovaArray_*` как ABI-compat» — это retire.

### Что НЕ мигрируется

Поскольку `[]T` остаётся валидным **синтаксисом-сахаром**, ~1390 аннотаций
`[]T` в 319 файлах (std/tests/examples) **НЕ переписываются** — они продолжают
работать, просто резолвятся в `Vec[T]`. Это снимает основной объём риска: работа
сосредоточена в компиляторе (резолвинг + codegen), а не в bulk-sed по исходникам.

---

## Главный выигрыш: закрытие typed-storage gap

Сейчас `[]T` для не-примитивных `T` (records, sums, `Option[T]`, nested `[][]T`,
tuples) **боксит элементы в `nova_int`-слоты** (type erasure, `emit_array_lit`
emit_c.rs:25670, for-in deboxing emit_c.rs:24036-24069). `Vec[T]` хранит
`*mut T` — **типизированно, без boxing**. Миграция чинит:

```nova
mut xs []Option[int] = [Some(1), None, Some(3)]   // было: int64-erasure + ручной debox
mut grid [][]int = [[1,2],[3,4]]                   // было: pointer-stomp в nova_int
mut recs []Point = [Point{1,2}, Point{3,4}]        // было: heap-box каждого
```

## Фазы

### Ф.0 — Развязка bootstrap-циркулярности + API-parity (GATE) (~1d)

Цель: подготовить `Vec[T]`/комбинаторы к флипу, БЕЗ изменения семантики `[]T`
(он ещё `NovaArray`). Чисто prep — после фазы ничего в поведении не меняется.

**0.1 — Vec self-dependency.** `vec_owned.nv` ссылается на `[]T` в трёх местах:
- `Vec[T].from(items []T)` (стр. 88) — итерирует `[]T`.
- `Vec[T] @extend(items []T)` (стр. 346).
- `Vec[T] @as_slice() -> []T` (стр. 391).

После флипа `[]T`=`Vec[T]` эти станут `from(Vec[T])→Vec[T]` (≈ `clone`),
`extend(Vec[T])`. Переписать так, чтобы они работали и до, и после флипа:
- `from` → принимает любой `Iter[I]`-источник (generic), не конкретно `[]T`.

**0.1a — Удалить `@as_slice()` (недоделка Plan 138 Ф.2).** `as_slice` названо как
zero-copy view, но **копирует** в свежий `[]T` (стр. 391-400). Избыточен по обоим
намерениям:
- **view/чтение** → `v[..]` / `v[a..b]` через `Index[Range]` (см. 0.1b).
- **владеющая копия** → `clone()` (уже есть, стр. 468).
- `to_array()` (родительский Ф.2 хотел оставить) — **не нужен**: при `[]T`≡`Vec[T]`
  это `clone()`. Алиас делает его избыточным.

Внутренних потребителей переписать (строго лучше):
- `@equal` (стр. 462-463: `as_slice() == other.as_slice()`) → element-wise loop
  (типизированно, без копии, без зависимости от `NovaArray ==`).
- `@clone` (стр. 485: `Vec[T].from(@as_slice())` — двойная копия) → один copy-loop.
- 6 self-тестов vec_owned.nv (стр. 596-618) + 21 occ в 4 plan131-тестах:
  `v.as_slice() == [..]` → `v == [..]` (via `@equal`) или `v[..] == [..]`.

**0.1b — `@index(Range)` как настоящий Nova-метод** (а не codegen-магия
emit_c.rs:17857) — слайсы через интерфейс `Index[Range, Vec[T]]`, согласовано с
«Vec полностью на .nv». Pointer-арифметика `@data + r.start` уже легальна (`@get_mut`):
```nova
export fn Vec[T] @index(r Range) -> Vec[T] {
    if r.start < 0 || r.end < r.start || r.end > @len { panic("Vec: slice OOB") }
    ro n = r.end - r.start
    Vec { data: unsafe { @data + r.start }, len: n, cap: n }  // zero-copy view; push→detach
}
```
Codegen `v[range]` → диспатч на `@index(range)` как любой protocol-метод (Ф.2).

**0.2 — Комбинаторы auto-attach audit.** После флипа `fn []T @map(...)` станет
методом на `Vec[T]` (ресивер `[]T` резолвится в `Vec[T]`). Проверить коллизии
имён между `vec.nv`/`sort.nv` (`[]T`-методы) и `vec_owned.nv` (`Vec[T]`-методы):
`first`/`last`/`iter`/`len`/`get` могут задвоиться → E_DUPLICATE_METHOD.
Дедуп: оставить одну реализацию каждого, удалить дубль.

**0.3** — Зафиксировать список методов `[]T`, которые обязаны существовать на
`Vec[T]` (parity-матрица в плане). Acceptance Ф.0: `nova check` на vec_owned.nv +
vec.nv + sort.nv проходит; targeted vec/sort fixtures PASS (поведение `[]T`
не тронуто).

**Commit:** `refactor(plan138.1 Ф.0): break Vec→[]T circular dep + combinator parity`

### Ф.1 — Type resolution flip: `[]T` → `Vec[T]` (~1d)

`emit_c.rs:4812` `type_ref_to_c(TypeRef::Array(elem))`:
```rust
// было: вернуть "NovaArray_<elem>*"
// стало: резолвить elem → c_ty, зарегистрировать Vec[elem] mono,
//        вернуть compute_generic_type_c_name("Vec", &[elem_c]) + "*"
//        → "Nova_Vec____<elem_c>*"
```
- Зарегистрировать Vec[elem] в `generic_type_templates` worklist (как делает
  `TypeRef::Named{path:["Vec"],...}`) — переиспользовать существующий путь.
- `types/mod.rs`: `Ty::Array(t)` ↔ `Ty::Named("Vec")` с arg — выровнять для
  bound-резолва / dispatch.
- `[]fn(...)` (closure-массив) и `[N]T` (`FixedArray`) — **вне scope**, остаются
  как есть (FixedArray — отдельный built-in; closure-array — отдельный followup).

Acceptance Ф.1: проект **type-check'ается** (codegen ещё может быть сломан —
литералы/методы чинятся в Ф.2-Ф.3). Gate: `nova check` зелёный по stdlib.

**Commit:** `feat(plan138.1 Ф.1 D239): []T resolves to Vec[T] (type_ref_to_c + Ty)`

### Ф.2 — Array-литералы → Vec construction (~1d)

### Ф.3 — for-in + method dispatch на Vec (~1d)

Acceptance Ф.3: for-in + методы на `[]T` идут через Vec; targeted PASS.

## Acceptance criteria

- **B1** — `fn f(a []int)` принимает `Vec[int]` и `[1,2,3]` (A8 из Plan 138).
- **B2** — `[1,2,3]` → `Vec[int]` value; типизированное хранение (no int-box).
- **B3** — `[]Option[int]` / `[]Record` / `[][]int` хранят типизированно; `[i]` → точный T.
- **B4** — `for x in arr` идёт через VecIter; mutate видно где ожидается.
- **B5** — `arr.push/pop/len/get/map/filter/sort/...` работают на `[]T`-аннотации.
- **B6** — `NOVA_ARRAY_DECL`/`NOVA_ARRAY_IMPL` удалены из `array.h`; grep пуст.
- **B7** — 0 новых FAIL в `nova test` vs plan-138 baseline.
- **B8** — D239 в spec; D144 закрыт полностью; `[]T` = `Vec[T]` задокументирован.
- **B9** — Perf: пустой `[]` и литерал-construction без значимого регресса.
- **B10** — `as_slice`/`to_array` удалены; `@index(Range)` — Nova-body метод (не codegen);
  слайсы только через `Index` (`v[..]`/`v[a..b]`), копия — через `clone()`.
