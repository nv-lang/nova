> **Создан:** 2026-06-10.  **Статус:** 📋 PLANNED.
> **Декомпозиция:** `[M-138-array-sugar-alias]` (Plan 138 Ф.5-Ф.6 deferred).
> **Эстимат:** ~4-7 dev-day (большой compiler refactor + test surface).
> **Model:** Opus + Thinking ON (рискованный codegen).
> **Worktree:** `nova-p138-1`, branch `plan-138.1` (off `plan-138`).
> **Зависит от:** Plan 138 Ф.1-Ф.4 ✅ (Vec @index/@get/mut@index + iter-first),
> Plan 131 ✅ (Vec[T] pure-Nova impl), Plan 90.1 ✅ ([]T extend-family).

---

## Цель

Сделать `[]T` **чистым синтаксическим сахаром** над `Vec[T]` — тип, целиком
реализованный на Nova (`std/collections/vec_owned.nv`). Конечное состояние:

- `[]T` ≡ `Vec[T]` на уровне типов: `fn f(a []int)` принимает `Vec[int]`.
- Array-литерал `[1, 2, 3]` строит `Vec[int]` (типизированный, **без** int-erasure).
- **`NOVA_ARRAY_DECL`/`NOVA_ARRAY_IMPL` C-macro полностью удалены** — никакого
  built-in C-массива. `Vec[T]` использует только `*mut T` + `nova_alloc`.
- D239 («`[]T` as sugar over `Vec[T]`») реализован; D144 закрыт полностью.

**Принцип (ограничение пользователя):** `Vec[T]` должен быть полностью на `.nv`.
Нет C-fallback'а «оставим `NovaArray_*` как ABI-compat» — это retire.

### Что НЕ мигрируется

Поскольку `[]T` остаётся валидным **синтаксисом-сахаром**, ~1390 аннотаций
`[]T` в 319 файлах (std/tests/examples) **НЕ переписываются** — они продолжают
работать, просто резолвятся в `Vec[T]`. Это снимает основной объём риска: работа
сосредоточена в компиляторе (резолвинг + codegen), а не в bulk-sed по исходникам.

---

## Главный выигрыш: закрытие typed-storage gap

Сейчас `[]T` для не-примитивных `T` (records, sums, `Option[T]`, nested `[][]T`,
tuples) **боксит элементы в `nova_int`-слоты** (type erasure, `emit_array_lit`
emit_c.rs:25670, for-in deboxing emit_c.rs:24036-24069). `Vec[T]` хранит
`*mut T` — **типизированно, без boxing**. Миграция чинит:

```nova
mut xs []Option[int] = [Some(1), None, Some(3)]   // было: int64-erasure + ручной debox
mut grid [][]int = [[1,2],[3,4]]                   // было: pointer-stomp в nova_int
mut recs []Point = [Point{1,2}, Point{3,4}]        // было: heap-box каждого
```

## Фазы

### Ф.0 — Развязка bootstrap-циркулярности + API-parity (GATE) (~1d)

Цель: подготовить `Vec[T]`/комбинаторы к флипу, БЕЗ изменения семантики `[]T`
(он ещё `NovaArray`). Чисто prep — после фазы ничего в поведении не меняется.

**0.1 — Vec self-dependency.** `vec_owned.nv` ссылается на `[]T` в трёх местах:
- `Vec[T].from(items []T)` (стр. 88) — итерирует `[]T`.
- `Vec[T] @extend(items []T)` (стр. 346).
- `Vec[T] @as_slice() -> []T` (стр. 391).

После флипа `[]T`=`Vec[T]` эти станут `from(Vec[T])→Vec[T]` (≈ `clone`),
`extend(Vec[T])`. Переписать так, чтобы они работали и до, и после флипа:
- `from` → принимает любой `Iter[I]`-источник (generic), не конкретно `[]T`.

**0.1a — Удалить `@as_slice()` (недоделка Plan 138 Ф.2).** `as_slice` названо как
zero-copy view, но **копирует** в свежий `[]T` (стр. 391-400). Избыточен по обоим
намерениям:
- **view/чтение** → `v[..]` / `v[a..b]` через `Index[Range]` (см. 0.1b).
- **владеющая копия** → `clone()` (уже есть, стр. 468).
- `to_array()` (родительский Ф.2 хотел оставить) — **не нужен**: при `[]T`≡`Vec[T]`
  это `clone()`. Алиас делает его избыточным.

Внутренних потребителей переписать (строго лучше):
- `@equal` (стр. 462-463: `as_slice() == other.as_slice()`) → element-wise loop
  (типизированно, без копии, без зависимости от `NovaArray ==`).
- `@clone` (стр. 485: `Vec[T].from(@as_slice())` — двойная копия) → один copy-loop.
- 6 self-тестов vec_owned.nv (стр. 596-618) + 21 occ в 4 plan131-тестах:
  `v.as_slice() == [..]` → `v == [..]` (via `@equal`) или `v[..] == [..]`.

**0.1b — `@index(Range)` как настоящий Nova-метод** (а не codegen-магия
emit_c.rs:17857) — слайсы через интерфейс `Index[Range, Vec[T]]`, согласовано с
«Vec полностью на .nv». Pointer-арифметика `@data + r.start` уже легальна (`@get_mut`):
```nova
export fn Vec[T] @index(r Range) -> Vec[T] {
    if r.start < 0 || r.end < r.start || r.end > @len { panic("Vec: slice OOB") }
    ro n = r.end - r.start
    Vec { data: unsafe { @data + r.start }, len: n, cap: n }  // zero-copy view; push→detach
}
```
Codegen `v[range]` → диспатч на `@index(range)` как любой protocol-метод (Ф.2).

**0.2 — Комбинаторы auto-attach audit.** После флипа `fn []T @map(...)` /
`fn []int @sort()` станут методами на `Vec[T]` / `Vec[int]`. Проверенные факты:

- vec.nv комбинаторы (`@map`/`@filter`/`@fold`/`@any`/`@all`) — generic `[T]`,
  **коллизий нет** с vec_owned.nv → авто-привязка без правок.
- sort.nv (`@sort`/`@sort_by`/`@binary_search`/`@min`/`@max`/`@sum`/`@product`) —
  на `[]int` (конкретный), **коллизий нет** → станут методами `Vec[int]`.
- **Коллизия только в `@first`/`@last`**: объявлены в ОБОИХ vec_owned.nv
  (`Vec[T]`, стр. 302/305) И vec.nv (`[]T`, стр. 133/144) → после флипа
  E_DUPLICATE_METHOD. Дедуп: **удалить из vec.nv**, оставить vec_owned.nv.
- `iter`/`len`/`get`/`push`/… НЕ коллизируют: их `[]T`-версии были codegen-
  built-in'ами (`nova_array_*`), не Nova-методами.

#### 🔴 Ф.5 BLOCKED (2026-06-10) — removal impossible without finishing Ф.1/Ф.2 migration first

**Root cause (verified empirically).** The `[]T → Vec[T]` flip in `type_ref_to_c`
(Ф.1, emit_c.rs:4938) **and** the typed-Vec literal path (Ф.2,
`try_emit_typed_vec_literal` / `emit_array_lit` emit_c.rs:26101) are both **gated
behind `self.generic_type_templates.contains_key("Vec")`**. That key is present
ONLY in compilation units that (transitively) import `std/collections/vec_owned.nv`.
**`Vec` is NOT in the prelude** (`std/prelude.nv` / `std/prelude/collections.nv`
re-export StringBuilder/WriteBuffer/ReadBuffer but never Vec). Therefore:

- In a unit that pulls Vec (e.g. via `Option`-heavy code), `[]Option[int]` etc.
  correctly lower to `Nova_Vec____NovaOpt_nova_int*` (typed). ✅
- In a **Vec-free / primitive-only unit**, `mut v []int = [1,2,3]` STILL lowers to
  the legacy `NovaArray_nova_int*` via `nova_array_new_nova_int` + `nova_array_push_nova_int`,
  and `.get/.push/.len/==/[a..b]` STILL route to `nova_array_*`. **Verified:** a
  scratch fixture `mut a []int = [1,2,3]` emits `NovaArray_nova_int*` (debug trace:
  `emit_array_lit ENTER … has_vec_tmpl=false`); `nova_tests/plan138_1/t6_empty_literal.c`
  and `nova_tests/plan91/sort_basic.c` both still use `nova_array_*_nova_int`
  pervasively.

**Consequence.** Every NovaArray special-case the plan lists as "dead" is in fact
**still live** for primitive / Vec-free arrays, which is the overwhelming majority
of the test+std surface (~1390 `[]T` annotations / 319 files). Removing
`NOVA_ARRAY_DECL/IMPL(nova_int|nova_str|nova_byte|…)` or the emit_c.rs special-cases
would mass-break: `sort_basic`, `plan90_1`, the whole `[]int`/`[]str`/`[]u8` corpus.

**Additional load-bearing deps (cannot remove regardless of the gate):**
- **String layer:** `nova_str_to_bytes/to_chars/as_bytes/split` (array.h) return
  `NovaArray_nova_byte*` / `NovaArray_nova_str*` / `NovaArray_nova_int*`;
  `string_builder.h` (`from_bytes_*`, `steal_bytes`, `try_from_bytes`) consumes
  `NovaArray_nova_byte*`. These appear in *every* compiled unit (Display/format/
  StringBuilder) — see the ubiquitous `nova_array_*_nova_byte`/`_nova_str` in all
  fixtures.
- **Parallel-for** (emit_c.rs:7037/7085) materialises iter source + result as
  `NovaArray_nova_int*`.
- **Closure-array** `[]fn(...)` → `NovaArray_void_p*` — explicitly out of scope
  (`[M-138.1-closure-array]`).
- **Vec→NovaArray bridge** (emit_c.rs:21999) routes bulk-ops/FFI helpers
  (`append_zero/copy_from/copy_within/fill/compare/as_ptr`) that have no Vec
  Nova-body method onto `nova_array_*` via a layout-identical cast — still needed.

**Required unblock (belongs to Ф.1-finish + Ф.6, ~multi-day, NOT a Ф.5 removal):**
1. Make the `Vec` template available in **every** unit (prelude
   `export import std.collections.vec_owned.{Vec}` — drags in RawMem + typed-ptr
   machinery; must first break the Vec→`[]T`→Vec bootstrap cycle per Ф.0 R3, which
   was only partially done) so the Ф.1/Ф.2 flip fires for primitives too.
2. Migrate the string-layer C helpers (`nova_str_to_bytes` etc.) + `string_builder.h`
   to return/consume `Nova_Vec____<elem>*` (or add a typed bridge), so the `[]u8`/
   `[]str`/`[]char` return-type contract matches post-flip.
3. Migrate parfor to typed Vec result storage.
4. THEN run Ф.6 regression-to-green per element-class (primitives → ptr → exotic),
   and ONLY THEN are the array.h removals + emit_c.rs special-case deletions safe.

No code removed this phase (would mass-break). Baselines held (plan131 27/1,
plan138 10/10, plan138_1 8/2, plan90_1 20/0, plan91 2/0).
