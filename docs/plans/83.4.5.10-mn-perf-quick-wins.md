* **Ф.2: Smaller fiber stack default** ❌ **ИСКЛЮЧЕНО ИЗ V2 — wrong hypothesis.**

  Изначальная гипотеза (Plan 83.4.5.10 V1): "8MB slot → 1MB снизит
  MEM_COMMIT/mmap overhead → ускорит spawn". **Неверно.**

  Empirical (попытка 1MB):
  - `cancellation_test` (within[T]/race2[T] mono'd) crashed immediate
    stack overflow.
  - 256KB also failed. Reverted к 8MB.

  Анализ почему гипотеза неверна:

  | Что зависит от slot_size | Реальный effect on spawn cost |
  |---------------------------|-------------------------------|
  | mmap MAP_NORESERVE virtual reservation | ❌ Бесплатно. mmap 8MB vs 1MB одинаково ~µs — kernel просто записывает VMA. |
  | VirtualAlloc MEM_RESERVE on Windows | ❌ То же — virtual address space cheap. |
  | Physical RAM commit | ❌ Lazy, зависит от REAL stack usage (page fault triggers commit), не от slot size. |
  | GC scan range (Plan 82 GC_push_other_roots callback) | ❌ Pushes only COMMITTED pages, не reserved range. Filter MEM_COMMIT + non-PAGE_GUARD + non-PAGE_NOACCESS. |
  | TLB pressure | ❌ Negligible на 64-bit (256TB address space). |
  | Per-slot `mprotect(guard, PROT_NONE)` on Linux | ⚠️ Yes, one syscall per slot in arena init. 4096 syscalls × ~µs = ~few ms per worker startup. **One-shot, не per-spawn.** |
  | Maximum stack budget | ⚠️ Hard cap на deep recursion. Уменьшение = limitation, не выгода. |

  **Итог:** уменьшение slot size не влияет на per-spawn cost. Влияет
  только на (a) virtual reservation total (бесплатно), (b) max stack
  cap (limitation). Real spawn cost driven Boehm GC lock + mco_create +
  uv_async_send — independent of slot size.

  Сохраняем **8MB slot** (production default) — virtual reservation
  8MB × 16384 slots = 128GB virtual per Windows worker, 8MB × 4096
  slots = 32GB на Linux. Free на 64-bit.

## 1. Контекст

parallel_sum vs MAXPROCS=1 — **НЕ MET**. Empirical (2026-05-24):

| Платформа | armed default | NOVA_MAXPROCS=1 | speedup |
|-----------|---------------|-----------------|---------|
| Windows 11 (16 cores) | parallel=518ms / seq=345ms | parallel=519ms | **0.66×** (медленнее!) |
| Linux WSL2 (16 cores) | parallel=683ms / seq=441ms | parallel=755ms / seq=372ms | **0.65×** |

Проблема **не платформо-специфическая** — идентично-плохая на Windows и

(~30ms workload не амортизирует worker pool cost).

## 2. Go runtime сравнение

| Метрика | Go runtime | Nova M:N |
|---------|------------|----------|
| Spawn cost | ~50-100ns (~200 CPU cycles) | ~10-100µs (100-1000× slower) |
| Starting stack | 2 KB (grows on-demand) | ~1 MB (fixed arena slot) |
| Alloc fast path | Per-P mcache (lock-free) | Boehm GC_malloc (global lock) |
| Wake notification | futex/eventfd (direct atomic) | uv_async_send (mutex+signal) |
| GC | Concurrent, write barriers | Boehm STW |
| Goroutine struct | ~336 bytes | SpawnCtx + mco_coro + 1MB arena slot |

**Корневые причины Nova медлительности (empirical analysis 2026-05-24
после Ф.2 failed hypothesis test):**

1. **Boehm GC global lock на каждой spawn allocation.**
   - `nova_alloc_uncollectable(sizeof(NovaSpawnCtx))` → `GC_malloc_uncollectable` → global mutex.
   - 16 workers concurrent spawn'ы все сериализуются через single GC lock.
   - Go: per-P mcache — lock-free fast path для small allocations.
   - **Estimated impact: ~50-200µs per spawn под contention.**
   - **Fix:** Ф.1 per-worker SpawnCtx pool.

2. **`mco_create` cost — initialize minicoro coroutine struct.**
   - Allocates fiber stack slot из arena (cheap, lazy commit).
   - But initializes register save area, CPU context, stack pointer setup.
   - On Windows: 3× `VirtualAlloc(MEM_COMMIT, …)` per slot — header
     window (8KB) + initial stack window (16KB) + guard page (4KB).
     3 syscalls × ~µs.
   - **Estimated impact: ~10-50µs per spawn.**
   - **Fix:** mco_coro pool (similar к Ф.1). Defer V3.

3. **`uv_async_send` для cross-thread wake.**
   - libuv: pthread_mutex_lock + condition broadcast — kernel syscalls.
   - Go: direct futex (Linux) / SetEvent (Windows) — single atomic + syscall.
   - **Estimated impact: ~5-20µs per spawn.**
   - **Fix:** replace libuv async с direct OS atomic primitive. Защитная
     mark: requires libuv replacement, months work, V3.

4. **`nova_effect_snapshot_save` per spawn.**
   - Copies TLS effect handler state (per-effect pointer table).
   - Под bootstrap: trivial; under armed: per-fiber snapshot allocation.
   - **Estimated impact: ~1-5µs per spawn.**
   - **Fix:** versioned snapshots — skip copy если handlers не менялись.
     V2 followup.

5. **Не виноваты в spawn-cost (проверено empirically):**
   - ❌ **Fiber stack slot size** (8MB vs 1MB). Lazy commit делает это
     free. Plan 83.4.5.10 Ф.2 hypothesis revoked. См. §2.6 ниже.
   - ❌ **TLB pressure** от 8MB slots. Negligible на 64-bit.
   - ❌ **Boehm GC scan cost** (Plan 82 GC_push_other_roots only pushes
     committed pages — sub-µs).

**Bottleneck по убыванию impact:** GC lock (Ф.1) >> mco_create cost >>
uv_async_send >> snapshot_save >> stack slot size (free).

## 3. Quick wins (~2-3 dev-day, target ≥1× speedup — parallel НЕ медленнее sequential)

### Ф.1: Per-worker SpawnCtx free-list pool (~1-2 dev-day)

**Дизайн:** аналог Go P-mcache. Каждый worker thread держит local
free-list пустых SpawnCtx struct'ов. Spawn-path:

```c
typedef struct NovaSpawnPoolEntry {
    NovaSpawnCtxBase* ctx;
    struct NovaSpawnPoolEntry* next;
} NovaSpawnPoolEntry;

struct NovaWorker {
    ...
    NovaSpawnPoolEntry* spawn_pool_free;  /* lock-free, owner thread */
    size_t spawn_pool_count;
};
```

**Spawn-emit logic:**
```c
/* Codegen-emitted в spawn-site: */
NovaSpawnCtx* ctx;
NovaWorker* w = nova_current_worker();  /* owner thread */
if (w && w->spawn_pool_free) {
    /* Fast path — pop из P-local pool, lock-free. */
    ctx = (NovaSpawnCtx*)nova_pool_pop(w);
    memset(ctx, 0, sizeof(NovaSpawnCtx));  /* zero-init */
} else {
    /* Slow path — alloc через Boehm. */
    ctx = nova_alloc_uncollectable(sizeof(NovaSpawnCtx));
}
```

**Free-path (в worker_main после mco_destroy):**
```c
/* Push back в pool — lock-free, owner thread. */
nova_pool_push(w, ctx);
if (w->spawn_pool_count > POOL_MAX) {
    /* Excess — GC_free для очистки memory pressure. */
    nova_free_uncollectable(ctx);
}
```

**Tradeoff:** SpawnCtx structs варьируются по size (user captures
add fields). Pool entries hash'ируются по size либо размер-class
buckets (тоже как Go: 70 size classes).

**Простейшая V1:** single global pool size = max common ctx size
(~256 bytes). Larger ctx → fall back to Boehm. Покрывает 80%
spawn-sites.

### Ф.2: Smaller fiber stack default — REJECTED (wrong hypothesis)

**Status:** ❌ ИСКЛЮЧЕНО из roadmap (post-empirical analysis 2026-05-24).

**Original hypothesis (V1):** "8MB slot → 1MB снизит MEM_COMMIT/mmap
overhead на startup, ускорит spawn." Был основан на confusion между
virtual reservation cost и physical commit cost.

**Empirical test (2026-05-24):** downsize 8MB → 1MB:
- `cancellation_test` immediate stack overflow (within[T]/race2[T]
  generic monomorphized + Boehm GC stack reserves).
- 256KB also crashed.
- Reverted к 8MB.

**Почему гипотеза неверна (детальный анализ):**

Stack budget model в Nova M:N runtime:
1. **Linux POSIX arena (`fiber_arena.c`):** `mmap(MAP_NORESERVE, virtual_size)`
   на arena init per worker. Virtual address space reserved, **physical
   commit = 0 bytes**. Stack pages коммитятся on-demand через page fault
   (kernel allocates 4KB page при первом write).
2. **Windows arena (`fiber_arena_win.c`):** `VirtualAlloc(MEM_RESERVE,
   virtual_size)` — virtual address space reserved. Per-slot initial
   commit = 8KB (header) + 16KB (initial stack window) + 4KB (guard) =
   ~28KB. Stack growth via `__chkstk` touching PAGE_GUARD → VEH commits
   next page on demand.

**Слот размер влияет на:**

| Aspect | Cost as f(slot_size) | Empirical |
|--------|----------------------|-----------|
| mmap MAP_NORESERVE virtual reservation | O(1) per arena init (~µs) | 8MB vs 1MB одинаково ~µs |
| VirtualAlloc MEM_RESERVE | O(1) per arena init | Same |
| Physical RAM commit | O(actual_stack_used_bytes) | Independent of slot_size |
| Per-slot mprotect guard (Linux) | O(slot_count) one-shot syscalls per arena init | 4096 syscalls × ~µs = ~few ms (но это per WORKER startup, **не per spawn**) |
| Per-spawn `VirtualAlloc(MEM_COMMIT, …)` (Windows) | O(initial_commit_window) = 8+16+4 = 28KB | **Independent of slot_size** — initial commit window fixed |
| GC scan range | O(committed_pages) | Plan 82 GC_push_other_roots filters MEM_COMMIT only — independent of slot_size |
| TLB pressure | Negligible на 64-bit (256TB virtual) | N/A |
| Maximum recursion depth | O(slot_size) | **Single negative aspect** — meaningful ограничение, не выгода |

**Итог:** slot size **не влияет** на per-spawn cost. Влияет только на:
- Virtual reservation total (FREE на 64-bit — 8MB × 16384 slots = 128GB
  virtual per Windows worker; на современных машинах overhead noise).
- Maximum stack budget per fiber (limit, не speedup).

**Real spawn-cost drivers (см. §2 Корневые причины above):** Boehm GC
lock (Ф.1), mco_create syscalls, uv_async_send, snapshot_save —
**все independent of slot_size**.

**Сохраняем 8MB slot как production default.** Никаких изменений
к `NOVA_FIBER_STACK_SIZE` (`compiler-codegen/nova_rt/fiber_arena.h`).

**V2 followup (если нужен Go-parity):** dynamic stack growth — требует:
- Stack-check probe в каждой function prologue (codegen).
- Stack copy semantics — old → new larger arena slot, fix all
  pointers-into-stack (precise GC required).
- Write barriers на GC roots (Boehm не поддерживает; нужен переход
  на precise GC либо custom GC).

Months of work. Out of scope для Plan 83.

### Ф.3: Inline parallel-for threshold (~0.5 dev-day)

**Heuristic:** для коротких parallel-for бежать кооперативно
inline в supervised body, без spawn'ов в worker pool.

```c
/* Codegen emit_parallel_for: */
if (iter_count < NOVA_PARALLEL_INLINE_THRESHOLD) {
    /* Run inline cooperative. */
    for (i = 0; i < iter_count; i++) {
        BODY;
    }
} else {
    /* Spawn to workers (existing path). */
    ...
}
```

**Threshold:** env var `NOVA_PARALLEL_INLINE_THRESHOLD` (default e.g.
16 либо calibrated). Tunable per-workload.

**Tokio rayon parity** — `rayon::join` использует similar adaptive
splitting (`join_context` heuristics).

**Tradeoff:** loses parallelism для small batches. Но для batches < 16
overhead полностью dominates — net win.

## 5. Что НЕ включено (out of scope)

Full Go-parity требует months of work:
- **Concurrent GC** (replace Boehm либо thread-local allocation
  buffers). Major architectural change.
- **Lock-free wake notification** (eventfd Linux / IOCP Windows
  direct). Tightly coupled с libuv replacement.
- **Dynamic stack growth** (write barriers на fn prologue, stack
  copy semantics, precise GC requirements). Years of work.

Эти — отдельный roadmap, не V1 quick wins.
