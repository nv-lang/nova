## Зачем

Sync primitives Nova-side жёстко зашиты на `acq_rel`/`acquire`/`release`
(см. `compiler-codegen/nova_rt/sync_primitives.h:99/127/130/143`) —
C-слой (`sync.h`) уже имеет **все 5 ordering constants** (`__ATOMIC_RELAXED`,
`ACQUIRE`, `RELEASE`, `ACQ_REL`, `SEQ_CST` — см. `sync.h:107-113`), но
Nova-уровень не даёт пользователю выбор. Это:
1. **Перфоманс-стол:** relaxed-ordering на counter increment (telemetry,
   stats) дешевле acq_rel на ARM (saves DMB), но недоступен.
2. **AI-first контракт:** программист не видит в сигнатуре, какой
   memory-effect делает операция; LLM пишет код без understanding
   happens-before.
3. **Spec drift:** `open-questions.md §1665` явно требует «зафиксировать
   memory model между fibers в D14 production-runtime» (Q-memory-model).
   Без формального D-блока невозможно reason about correctness.

Plan 103.1 — **foundation**, дальше 103.2 (atomics) принимает `Ordering`
параметром на все ops; 103.3-103.5 опираются на тот же happens-before
contract.

## Scope

### In scope

- **`Ordering` enum** в `std/runtime/sync.nv` с 5 вариантами:
  `Relaxed | Acquire | Release | AcqRel | SeqCst`.
- **`fence(Ordering)`** module function exposing `__atomic_thread_fence`.
- **Codegen mapping** `Ordering` enum variant → `__ATOMIC_*` constant.
- **Type-checker validation:** invalid ordering на operation kind
  (compile-time error, не runtime). 2 error codes — `E_INVALID_ORDERING_LOAD`,
  `E_INVALID_ORDERING_STORE`.
- **D167 draft** в `spec/decisions/06-concurrency.md` (happens-before
  contract; final closure — в Plan 103.7).
- **Backward compat:** existing `AtomicInt`/`AtomicBool` методы без
  ordering продолжают работать как раньше (hardcoded `acq_rel`/`acquire`/
  `release`). 103.2 добавит overloads с `Ordering` параметром.

### Out of scope

- Конкретные `AtomicX.load(Ordering)` / `.store(value, Ordering)` API —
  Plan 103.2.
- Validation invalid ordering на atomic op кроме load/store — Plan 103.2
  (там валидируем все: swap, CAS, fetch_*).
- Mutex memory contract spec — Plan 103.3 (mutex.lock/unlock уже implicit
  acquire/release, документируется отдельно).
- D167 финальная редакция — Plan 103.7.
- `Ordering::Consume` (C++ rarely-used) — Nova не поддерживает (Rust тоже
  не имеет).

## API design

### `Ordering` enum

**Nova sum-type syntax verified** (см. `std/prelude/errors.nv:84-91`
`RuntimeError`): `export type Name` + variants на новых строках с
leading `|`, без braces вокруг variant-list. Payload-less variants —
просто `Variant`.

**`#stable(since = "...")` decision:** использовать **`"0.1"`** —
консистентно с baseline `std/runtime/sync.nv` (всё там `0.1`).
`"0.6"` в `std/time/duration.nv` (Plan 65) — отдельный convention для
later-shipped APIs. Не вводим `"0.2"` без global decision (избегаем
третьего version-marker).

```nova
module runtime.sync

/// Memory ordering для atomic operations и fences. Контролирует
/// happens-before отношения между операциями в разных fiber'ах.
///
/// Default ordering для simple-overload методов — `SeqCst` (M1):
/// безопасно для всех use cases, может быть переопределено для
/// perf-critical кода через `_ordered` overloads.
///
/// **Семантика** (D167):
///
/// - `Relaxed`: только атомарность операции; нет happens-before.
///   Use case: telemetry counters, statistics. Цена: dirty reads
///   возможны cross-fiber, но не torn writes.
/// - `Acquire`: для load; формирует happens-before с paired Release.
///   Use case: load shared payload after seeing flag set.
/// - `Release`: для store; формирует happens-before с paired Acquire.
///   Use case: publish payload before setting flag.
/// - `AcqRel`: для RMW (swap, CAS, fetch_*); combination Acquire+Release.
///   Use case: lock-free queue head/tail; one-shot ownership transfer.
/// - `SeqCst`: sequentially consistent — total order на всех SeqCst
///   операциях во всех fiber'ах. Use case: simple mental model,
///   correctness без deep analysis. Default.
#stable(since = "0.1")
export type Ordering
    | Relaxed
    | Acquire
    | Release
    | AcqRel
    | SeqCst
```

### `fence(Ordering)` module function

```nova
/// Memory fence — sequenced ordering point без atomic operation.
///
/// Используется для отделения unsynchronized reads/writes от synchronized
/// (publish/subscribe pattern, refcount-drop с "free данных только когда
/// counter hit 0").
///
/// **Семантика** (D167):
///
/// - `fence(Relaxed)`: no-op. Валиден синтаксически для consistency,
///   но не имеет ordering эффекта (Acquire/Release нужны).
/// - `fence(Acquire)`: все subsequent reads/writes happen-after все
///   prior Release-stores (visible-to-this-fiber).
/// - `fence(Release)`: все prior reads/writes happen-before все
///   subsequent Acquire-loads (visible-to-other-fibers).
/// - `fence(AcqRel)`: combination.
/// - `fence(SeqCst)`: total-order participation; sequenced relative
///   к всем другим SeqCst fences и SeqCst operations.
///
/// # Example: refcount-drop с deferred-free
///
/// ```nova
/// // worker fiber:
/// let prev = refcount.fetch_sub(1, Ordering.Release)
/// if prev == 1 {
///     fence(Ordering.Acquire)   // synchronize with all Release-decs
///     // safe to access data — last fiber holds it
///     destroy(data)
/// }
/// ```
#stable(since = "0.1")
export external fn fence(ord Ordering)
```

### Validation rules (compile-time)

| Operation kind | Allowed orderings | Errors на forbidden |
|---|---|---|
| `load` | `Relaxed`, `Acquire`, `SeqCst` | `E_INVALID_ORDERING_LOAD` (если `Release`/`AcqRel`) |
| `store` | `Relaxed`, `Release`, `SeqCst` | `E_INVALID_ORDERING_STORE` (если `Acquire`/`AcqRel`) |
| `swap`, `compare_exchange*`, `fetch_*` | все 5 | — (Plan 103.2 scope) |
| `fence(ord)` | все 5 (Relaxed — no-op но валиден синтаксически) | — |

Валидация при **`Ordering`-литерал известен compile-time** (типичный
случай — `Ordering.Acquire` constant). Если ordering — runtime value
(`let ord = pick()`) — fallback на runtime panic в codegen (rare path).

## D167. Memory ordering & happens-before между fiber'ами

### Что

`Ordering` enum (5 вариантов) для atomic operations и fence(). Контракт
happens-before между fiber'ами через paired Acquire/Release.

### Правило

[5 ordering вариантов с семантикой]
[fence(Ordering) semantics]
[validation: load disallows Release/AcqRel; store disallows Acquire/AcqRel]
[default Ordering = SeqCst для simple-overload methods (M1)]

### Memory model contract

Bootstrap-runtime (single-threaded fiber pool): orderings reduce to
sequenced-before — все ordering варианты semantically equivalent.

M:N runtime (Plan 23, Plan 83.4.5.x): полный memory model по C11
__atomic_* семантике; happens-before между fiber'ами через paired
Acquire/Release.

### Связь

- D14, D50 (fiber runtime) — D167 specifies memory model для D14
  production-runtime
- D79 (Channels) — channel send/recv → happens-before (Go-style)
- Plan 103.1 — реализация Ordering enum
- Plan 103.2+ — uses Ordering на atomic operations
```

Acceptance: D167 draft в спеке; `grep "^## D167"` находит. Final
text — 103.7.

## Risks + mitigations

| Риск | Mitigation |
|---|---|
| Nova enum синтаксис не совпадает с моим draft'ом (`type Ordering { A \| B \| ... }`) | Ф.1 — verify в `spec/decisions/03-syntax.md` перед commit'ом; fallback на alternate syntax. Семантика фиксирована, синтаксис — implementation detail. |
| `Ordering` как enum value → C-side switch overhead на каждый atomic call | Inline switch (5 cases) + branch predictor — negligible; compiler может inline через `const` propagation если `Ordering.X` literal. |
| Runtime-value `Ordering` (not literal) breaks Ф.4 validation | По дизайну fallback на runtime panic (rare path); document'ировано в D167. |
| `fence(Relaxed)` пользователь может call ошибочно вместо `fence(Acquire)` | `nova check` lint `W_FENCE_RELAXED_NO_OP` (warning, не error; AI-friendly suggestion). Декомпозируется отдельно если нужно. |
| Backward compat нарушится | M14 acceptance + Ф.6 backward-compat test (existing tests passes без изменений). |
