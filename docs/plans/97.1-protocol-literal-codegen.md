# Plan 97.1 — codegen для protocol-литерала (vtable struct + dispatch)

> **Создан:** 2026-05-23 (followup [Plan 97](97-protocol-effect-syntax-symmetry.md) Ф.4).
> **Статус:** 📋 proposed, не начат.
> **Приоритет:** **M** (нужен для разблокировки stdlib Plan 18:
> capability-split factory pattern — `Process.spawn -> (Stdin, Stdout, Stderr)`,
> `HttpServer.bind -> (Acceptor, ShutdownHandle)`, `Db.transaction ->
> (TxReader, TxWriter, Commit)`).
> **Оценка:** ~2–3 dev-day (~3 точечных кодовых блока + фикстуры).
> **Закрывает:** [M-protocol-literal-codegen-deferred] (`docs/simplifications.md:11124`).
> **Зависимости:** Plan 97 ✅ (parser + AST `ExprKind::ProtocolLit` +
> type-checker `check_protocol_lit`); Plan 56 D122 ✅ (`emit_protocol_box_typedef`
> + `emit_protocol_vtable_companion` инфра для hybrid dispatch).
> **Worktree:** `nova-p97-1` (новый), ветка `plan-97-1`.

## Зачем

Plan 97 Ф.4 ввёл синтаксис protocol-литерала и сделал parser + AST +
type-checker, но **runtime codegen — заглушка**: `emit_expr` для
`ExprKind::ProtocolLit { proto_name, methods }` делегирует на
`emit_handler_lit(proto_name, methods)` — тот же путь, что у эффектов.
Это piggyback'ает на effect-vtable инфре, которая для protocol-only
типов **не эмитится** (Plan 15 D53 strict: protocol — compile-time-only).

Следствие — CC-FAIL `unknown type name 'NovaVtable_<Proto>'` при
попытке скомпилировать программу с protocol-литералом:

```nova
type Locker protocol { lock() -> () }

fn make() -> Locker {
    let state = ...
    protocol Locker { lock() => state.lock() }   // ❌ CC-FAIL сейчас
}
```

Capability-split factory pattern из D142 — основной use-case — не
запускается. Plan 97.1 закрывает эту дыру минимальной инфрой,
переиспользуя существующий механизм `emit_protocol_vtable_companion`
(Plan 56 D122 hybrid dispatch).

## Архитектурное решение (Ф.0 утвердит)

Две опции:

- **(A) Synthetic concrete struct per literal.** Для каждого
  `protocol Locker { lock() => state.lock() }` codegen эмитит
  анонимный `Nova_<ProtoName>_lit_<N>` record + `Nova_<concrete>_method_<m>`
  free fn (как сейчас для effect-литералов), затем вызывает
  `emit_protocol_vtable_companion(proto, [], "Nova_<ProtoName>_lit_<N>*")`
  чтобы получить vtable+box. Метод-вызов `l.lock()` диспатчится через
  существующий vtable-companion path. **Плюс:** переиспользует Plan 72
  P3-B механизм 1:1. **Минус:** одна синтетическая структура на каждый
  литерал — больше C-кода, но bootstrap-приемлемо.

- **(B) Closure-bundle approach.** Эмитить инлайн struct из
  fn-pointers (analog handler-literal): `{ void* ctx; ret (*lock)(void*); }`,
  без registration в `protocol_method_registry`. Dispatch `l.lock()` —
  через специальный path, который **отличает** значения protocol-литерала
  от protocol-typed обычной переменной. **Плюс:** меньше C-кода на
  литерал. **Минус:** два пути dispatch'а для одного контракта (вью-
  переменная vs литерал-значение) — нарушает D40 «один очевидный путь».

**Рекомендация — A.** Один путь dispatch'а (vtable companion для всех
protocol-значений: и literal, и обычный assignment), переиспользование
существующей инфры Plan 72 P3-B. Подход B порождает два кодгенных
канала, hard для maintenance.

## Scope

**Входит:**

*Codegen vtable инфра:*
- Эмиссия `NovaVtable_<Proto>` для **каждого** объявленного
  `type X protocol {...}` (не только при использовании в литерале —
  всегда, как для эффектов). Изменение в `emit_type_decl` ветке
  `TypeDeclKind::Protocol`.
- Регистрация protocol-методов в `effect_schemas` (или зеркале
  `protocol_schemas`) чтобы `emit_handler_lit` находил сигнатуры.

*ProtocolLit codegen:*
- Заменить piggyback на `emit_handler_lit` отдельным `emit_protocol_lit`
  методом (или ветвлением внутри `emit_handler_lit`). Эмитить:
  1. Synthetic `Nova_<ProtoName>_lit_<N>` struct для capture context.
  2. Free fn `Nova_<lit_name>_method_<m>(self, args)` для каждого
     метода (с capture'ами через `self->ctx_field_X`).
  3. Вызов `emit_protocol_vtable_companion` для генерации vtable +
     box pointer.

*Method dispatch на protocol-typed value:*
- `value.method()` где `value: Locker` (named protocol-тип) —
  диспатчится через `NovaBox_<Locker>`-pattern (vtable lookup), а не
  поиском метода в `method_table` концретного типа. Расширение
  существующего `protocol_var_vtable` HashMap (Plan 72 P3-B) для
  любого protocol-typed bindings (не только generic).

**Не входит:**

- Multiple-method capture-conflict диагностика (если два метода
  захватывают `let mut x` — какие правила mutation visibility?). В
  bootstrap-trace следует closure D22/D6 правилам через managed
  heap — отдельный D-блок если возникнут edge cases.
- Hard-enforcement static↔instance matching ([M-protocol-static-enforcement-deferred]) —
  отдельный followup.
- Optimization: статически известный concrete type → прямой вызов
  без vtable. Hybrid с Plan 56 D122 уже это умеет; для литерала
  концретный тип = synthetic anon struct, прямой вызов даёт ту же
  скорость что vtable — не optimization-приоритет.
- Anonymous protocol в expression-position (`let l = protocol { method() => ... }`
  без имени): отвергнуто в Plan 97 Ф.4 (нет контракта для structural
  check'а); не возвращается.

## Фазы

### Ф.0 — Аудит + design (~0.3 д) — GATE

- **Ф.0.1** Проверить `emit_protocol_vtable_companion`: вызвать на
  probe synthetic-структуре (без существующих `Nova_<X>_method_<m>`).
  Должны быть эмитированы thunks → method calls; если вызов идёт в
  `Nova_<lit_N>_method_<m>` который ещё не определён — порядок forward
  decl'ов критичен.
- **Ф.0.2 — decision point A/B.** Утвердить A. Probe: hand-написать
  C-код для одного `protocol Locker { lock() => state.lock() }` через
  synthetic struct + companion vtable; проверить, что dispatch
  работает в standalone test.
- **Ф.0.3** Сверить, что `emit_type_decl` для protocol'ов
  (`TypeDeclKind::Protocol(_)` arm) — unconditional vtable emit не
  ломает существующих generic-bound тестов (Plan 56 D122 уже эмитит
  vtable per (proto, args) on-demand; unconditional emit может
  дублировать). Decision: эмитить **только** базовый `NovaVtable_<Proto>`
  (без args, для non-generic форм) или integrate с per-args.

**Acceptance Ф.0:** decision point A зафиксирован; probe собирается
и запускается; план Ф.1-Ф.4 детализирован под итоги probe'а.

### Ф.1 — Базовая vtable infra для protocol'ов (~0.8 д)

- **Ф.1.1** В `emit_type_decl`, ветка `TypeDeclKind::Protocol(methods)`:
  не skip, а вызвать `emit_protocol_vtable_struct(name, methods)` —
  эмитит `NovaVtable_<Proto>` (без args) и регистрирует в
  `protocol_method_registry`. Для generic-протоколов (`Iter[T]`) —
  оставить per-args path Plan 72 P3-B; non-generic — новый канал.
- **Ф.1.2** Зеркало protocol-методов в `effect_schemas` (или новый
  `protocol_schemas` с одинаковым layout): нужно для `emit_handler_lit`
  чтобы найти param-types методов literal'а.
- **Ф.1.3** Регресс: убедиться, что Plan 72 P3-B тесты (hybrid
  dispatch) не сломались — vtable emit для protocol'а не должен
  конфликтовать с companion-генерацией.

**Acceptance Ф.1:** `nova test` → 0 регрессий; для тестового
`type X protocol { run() -> () }` в эмитированном C виден
`typedef struct { void* ctx; nova_unit (*run)(void*); } NovaVtable_X;`.

### Ф.2 — ProtocolLit codegen (~0.8 д)

- **Ф.2.1** Новый `emit_protocol_lit(proto_name, methods)`:
  1. Captures: collect free idents в method bodies (как в
     `emit_handler_lit`).
  2. Synthetic ctx struct: `typedef struct { <capture_t cap;>* }
     NovaCtx_<lit_id>;`.
  3. Allocate ctx; copy captures.
  4. Synthetic method-impl free fns: `static <ret> _proto_lit_<id>_impl_<m>(void* _ctx, <args>) { ... }`.
  5. Allocate `NovaVtable_<Proto>`: `vt->ctx = ctx; vt->m = _proto_lit_<id>_impl_<m>;`.
  6. Возврат указателя на vtable.
- **Ф.2.2** Заменить piggyback в `ExprKind::ProtocolLit` arm `emit_expr`
  на новый `emit_protocol_lit`.
- **Ф.2.3** Восстановить фикстуру `nova_tests/plan97/pos_protocol_lit_basic.nv`
  (была удалена в Plan 97 Ф.4 как блок'нутая codegen'ом). Должна
  собираться и запускаться.

**Acceptance Ф.2:** `pos_protocol_lit_basic` PASS; единичный
`let r = protocol Runnable { run() => () }` собирается без CC-FAIL.

### Ф.3 — Method dispatch на protocol-typed value (~0.6 д)

- **Ф.3.1** В codegen `Member`-call: если `obj` имеет protocol-тип
  (определяется через `protocol_types` set + переменная-binding type)
  — диспатч через `((NovaVtable_<Proto>*)obj)->method(obj->ctx, args)`
  вместо `Nova_<concrete>_method_<m>(obj, args)`. Hybrid с Plan 56
  D122: если concrete known статически (через `protocol_var_vtable`
  HashMap) — прямой вызов; иначе vtable lookup.
- **Ф.3.2** Edge case: `value.method()` где value передан как
  argument в fn (тип в сигнатуре — protocol). Сейчас работает через
  `NovaBox_<proto>_<args>` (Plan 72 P3-B); расширить на non-generic
  protocol'ы без args.

**Acceptance Ф.3:** `l.lock()` на protocol-typed `l` (literal-source)
работает; capability-split factory pattern fixture проходит.

### Ф.4 — Тесты pos/neg (~0.4 д)

- **Ф.4.1** `pos_protocol_lit_basic.nv` (восстановить из Plan 97 Ф.4):
  literal создаётся + метод вызывается напрямую.
- **Ф.4.2** `pos_protocol_lit_capability_split.nv` (per Plan 97 Ф.5.13):
  capability-split factory pattern `Lock.new() -> (Locker, Unlocker)`;
  оба значения вызывают свои методы.
- **Ф.4.3** `pos_protocol_lit_closure_capture.nv`: литерал + захват
  `let mut state` через closure (D22/D6); mutation видна снаружи.
- **Ф.4.4** `pos_protocol_lit_with_anon_type.nv`: литерал, который
  потом передан в fn принимающий `protocol { ... }` (inline) —
  cross-Ф.2/Ф.4 совместимость.
- **Ф.4.5** Регресс существующих 12 фикстур `nova_tests/plan97/`
  (Plan 97) — должны остаться PASS.

**Acceptance Ф.4:** 4 новые позитивные + 12 существующих = 16/16 PASS;
полный `nova test` → 0 регрессий.

## Acceptance criteria

- [ ] `protocol Name { method-impl* }` литерал собирается в C без CC-FAIL.
- [ ] `l.method()` на protocol-typed литерал-значении работает (dispatch
      через vtable companion).
- [ ] Capability-split factory pattern `Lock.new() -> (Locker, Unlocker)`
      работает end-to-end (literal + dispatch).
- [ ] Existing Plan 72 P3-B (hybrid dispatch на generic protocol-bound)
      не сломан.
- [ ] Existing 12 фикстур Plan 97 + 4 новых = 16/16 PASS.
- [ ] Полный `nova test-all` → 0 регрессий.
- [ ] Маркер `[M-protocol-literal-codegen-deferred]` снят.

## Risks

1. **`emit_type_decl` для protocol — unconditional vtable emit**
   может конфликтовать с Plan 72 P3-B per-args path. **Митигация:**
   Ф.0.3 — probe; если конфликт — generate базовый vtable отдельно
   от per-args.
2. **Method dispatch path для protocol-typed binding** — расширение
   `protocol_var_vtable` HashMap на non-generic protocol'ы может
   нарушить hybrid-логику Plan 56 D122. **Митигация:** Ф.3 регресс
   per-test; если нарушение — split path по generic/non-generic.
3. **Capture-rules** protocol-литерала через managed heap (D22/D6) —
   при mutable capture'ах гонки между методами литерала и внешним
   кодом. **Митигация:** в bootstrap single-thread'е безопасно;
   M:N — отдельный D-блок (analog Q-mn-6 effect handler concurrent).
4. **Synthetic struct name collisions** (`Nova_<lit_id>`) при
   multiple литералах в одном модуле. **Митигация:** monotonic
   counter `protocol_lit_counter` (analog `handler_counter`).

## Non-scope (отложено)

- Optimization: статическая мономорфизация литерала под конкретный
  callee-тип (если compiler знает, что literal присваивается переменной
  типа `Locker` и сразу используется — можно избежать vtable). Plan
  56 D122 hybrid уже даёт частичный benefit.
- Полная hard-enforcement static↔instance protocol-метод matching —
  отдельный followup (`[M-protocol-static-enforcement-deferred]`).
- Anonymous protocol-литерал в expression-position (`let l = protocol { ... }`
  без имени) — отвергнут Plan 97 Ф.4.
