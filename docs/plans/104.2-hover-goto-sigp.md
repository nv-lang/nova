# Plan 104.2 — Hover + Goto-definition + Signature Help

> **Статус:** ✅ ЗАКРЫТ 2026-06-17 (Ф.7 body-walk hover добавлен)
> **Branch:** plan-104-2
> **Depends:** Plan 104.1 ✅ (diagnostics + document cache)

## Scope

Три LSP-запроса:
1. `textDocument/hover` — рендер типа + doc-comment (D104/D105) в markdown
2. `textDocument/definition` — переход к декларации (cross-file через workspace import graph)
3. `textDocument/signatureHelp` — список параметров + активный параметр + перегрузка

## Архитектурные решения

### Symbol Resolution (M-104.2-1)

Реализация symbol resolution строится поверх существующего `nova_codegen::parser::parse` +
`nova_codegen::types::check_module` без изменений в compiler-codegen (отдельный API in nova-lsp).

**Почему:** compiler-codegen API нестабилен (каждый sub-plan добавляет поля в AST/types);
создание pub API в compiler-codegen требует семантической стабилизации. V1 — LSP-side
symbol walk по AST напрямую (аналог как field_cache.rs делает это для code-lens/hover
в Plan 123). Compiler-codegen pub API планируется в Plan 104.3 (completion).

### Hover rendering (M-104.2-2)

Формат MarkedString с `nova` fenced code block для типов + plain markdown для doc-comment.
LSP spec позволяет MarkupContent (preferred by VS Code 1.70+) — используем MarkupContent
с kind=markdown.

### Goto-definition cross-file (M-104.2-3)

V1: single-file только (в контексте одного документа через `Module.items`). Cross-file
через workspace root — V2 (Plan 104.4 + dep-graph). Договор: возвращаем `null` если
символ из импортированного модуля (нет span в текущем файле).

**Обоснование:** cross-file resolve требует полного workspace graph (resolver.rs) —
это тяжелее чем hover; откладываем на Plan 104.4. В V1 cover basic use case:
ctrl+click на local variable/fn/type в том же файле → jump to decl.

### SignatureHelp (M-104.2-4)

Resolve call expression вокруг cursor позиции → extract callee name → lookup fn overloads
→ return SignatureInformation list + active parameter index.

## Детальная реализация

### Ф.1 — symbol.rs: SymbolInfo + AST walker

`SymbolInfo` enum содержит:
- `LocalVar { name, ty_text, span, doc }`
- `FnDecl { name, signature, doc, span }`
- `TypeDecl { name, kind, doc, span }`
- `MethodDecl { receiver_type, name, signature, doc, span }`
- `ProtocolDecl { name, doc, span }`
- `ImportRef { module_path, span }`

`resolve_symbol_at(src, byte_offset)` — парсит модуль, ищет самый узкий AST node
покрывающий byte_offset. Возвращает `Option<SymbolInfo>`.

### Ф.2 — hover.rs: рендер hover content

```
fn compute_hover(src: &str, pos: Position) -> Option<Hover>
```

Рендер:
- FnDecl: ` ```nova\nfn Name(params) effects -> RetTy\n ``` \n---\ndoc-comment`
- TypeDecl: ` ```nova\ntype Name { ... }\n ``` \n---\ndoc-comment`
- LocalVar: ` ```nova\nro|mut name: Ty\n ``` `
- ImportRef: ` ```nova\nimport path.to.module\n ``` `

### Ф.3 — goto_definition.rs

```
fn compute_goto_definition(src: &str, pos: Position, uri: &Url) -> Option<Location>
```

V1 — single-file. Алгоритм:
1. resolve_symbol_at → SymbolInfo
2. если SymbolInfo содержит span → Location { uri, range }
3. иначе None

### Ф.4 — signature_help.rs

```
fn compute_signature_help(src: &str, pos: Position) -> Option<SignatureHelp>
```

Алгоритм:
1. byte_offset = lsp_pos_to_byte
2. Find innermost call expression covering cursor (scan backwards for `(`)
3. Extract callee name
4. Lookup fn declarations with that name in module
5. Count commas before cursor to determine active_parameter
6. Return SignatureHelp { signatures, active_signature, active_parameter }

### Ф.5 — server.rs: wire handlers

Добавить `hover_provider: true`, `definition_provider: true`,
`signature_help_provider: Some(...)` в initialize capabilities.

### Ф.6 — Tests

Тесты через unit functions (без LSP process spawn — faster и more reliable).

## Performance budget

- hover/goto-def: ≤100ms на типичном файле (1000 строк) в release build
- signatureHelp: ≤100ms

Достигается через `run_with_large_stack` + полный parse per request (V1; кеш в V2).

### Ф.7 — Hover inside fn/test bodies (2026-06-17)

Проблема: hover работал только на **объявлениях** (top-level items), но не на
**вызовах** внутри fn/test тел (`assert(...)`, `println(...)`, методы).

**Корень:** `resolve_imports_inline` **prepend**'ит импортированные items перед
оригинальными (`new_items.append(&mut module.items)` в imports.rs ~829). После
инлайна оригинальные items файла — в **хвосте** `module.items`, а не в начале.
Параметр `span_item_limit` использовал `.take(original_len)` — захватывал только
prepended imported items; оригинальные test/fn блоки не проверялись.

**Фикс:**
- `items_start = total_after_inline - items_before_inline` — сколько items
  было prepend'нуто.
- `resolve_symbol_at_with_limit`: span-match через `.skip(items_start)` (только
  оригинальные items с корректными span'ами).
- body-walker (`find_ident_in_bodies_from`): так же `.skip(items_start)`.
- Name-lookup после body-walk ищет по ВСЕМ items (включая inlined prelude) →
  `assert`, `println` и прочие prelude-символы находятся.
- `resolve_symbol_at` (без inlining): `items_start=0` (skip ничего).

**Файлы:** `nova-lsp/src/hover.rs`, `nova-lsp/src/symbol.rs`,
`nova-lsp/src/server.rs` (pass URI to compute_hover + catch_unwind).

**Тесты:** 255 unit tests PASS; 2 новые фикстуры в `nova_tests/plan104_9/`
(`pos_hover_prelude_calls_in_test.nv`, `pos_hover_prelude_calls_in_fn.nv`).

2. 255 unit tests PASS (hover + goto + symbol + completion + …)
3. hover handler: local var/fn/type → markdown с типом + doc

8. signatureHelp: inside call → param list + active_parameter
9. Cancellation-safe (each handler checks `state.docs.get`)
10. UTF-16 positions correct (reuse `diagnostic_mapping` utilities)
11. Edge cases: whitespace/comment/EOF/multi-byte handled без panic
12. **«Без упрощений как для прода»**: hover не crash'ится при inlining (catch_unwind); prelude символы резолвятся через name-lookup по всем 450+ items после inline
