## Что включает

### Ф.1 — Spread `[...defaults, k: v]`

**Use-case:** config defaults + overrides. TS поддерживает (`{...d, x: 1}`),
Rust `extend()`, Go вручную через цикл.

**Реализация:**
1. Parser `parse_map_lit_rest`: `...expr` элемент = spread (тип должен
   быть `HashMap[K,V]` совместимый с element-K/V).
2. AST: `MapLitElem::Spread(Expr)` vs `MapLitElem::Pair(Expr, Expr)`.
3. Desugar: `[...m, k: v]` → `{ let mut _t = m.@clone(); _t.@insert(k, v); _t }`.
4. Stdlib: `HashMap.@clone()` + `HashMap.@merge_from(other)` —
   **закрыто codegen-quirk про pattern-match в while-body**, требует
   фикса Plan 48 baseline.

**Размер:** ~80 LOC после закрытия codegen-quirk.

### Ф.2 — Const map-literal (perfect-hash) — **BLOCKED**

**Status (2026-05-16):** заблокировано pre-existing codegen ordering
issue. Простейший случай:

```nova
const KEYWORDS HashMap[str, int] = ["if": 1, "else": 2]
```

CC-FAIL: `unknown type name 'Nova_HashMap____nova_str__nova_int'` —
const-decl эмитится **до** mono pass генерирует monomorphized struct.

Это не Plan 52/52.1 issue, а baseline ordering bug в emit_c.rs:
- Line 984-989: const emit ПОСЛЕ type-decl но ДО mono pass
- Mono'd structs появляются только при первом callsite в fn-body

**Зависит от:** Plan 48 followup — ordering pass для const с
generic-type'ом (либо forward-declare mono'd types, либо отложить
const emit до конца mono).

**Workaround в bootstrap:** lazy-init pattern через fn:
```nova
fn keywords() -> HashMap[str, int] = ["if": 1, "else": 2]
// usage: keywords().get("if")
```

Полная Ф.2 perfect-hash остаётся за scope (требует new codegen
infrastructure ~300 LOC + gperf-like generator).

**Use-case:** compile-time lookup tables (keyword sets, opcode tables).
Rust `phf` crate; Go `var x = map[K]V{...}` at package init.

**Реализация:**
1. AST `ExprKind::ConstMapLit` (compile-time evaluable keys/values).
2. Type-checker: всё const → generate perfect-hash table в codegen.
3. Static C array `{key_hash, value}[N]` + binary search или `gperf`-like.
4. Lazy-init НЕ нужен — embedded в const data segment.

**Размер:** ~300 LOC (включая gperf-like helper).

### Ф.3 — MIR/codegen-late lowering — **REASSESSED: NOT NEEDED**

**Original concern (Plan 52 §580):** `(4n+2)/3` load-factor формула
"баковится" в AST-emit; при swisstable-миграции потребует AST-pass
правок.

**Reassessment (2026-05-16):** проверка показала что формула **уже
полностью decoupled** от AST. В `desugar.rs::build_map_block`:
- AST эмитит `with_capacity(literal_count)` — просто `n` пар
- Сама load-factor арифметика `(min_capacity * 4 + 2) / 3` →
  `next_pow2(needed)` находится в `std/collections/hashmap.nv:105+`
  (Nova-side stdlib)

То есть **swisstable-миграция = переписать `with_capacity`/`@insert`
в hashmap.nv**, без touch'инга AST/codegen. Это уже близко к Rust-style
late-lowering benefit, без MIR infrastructure.

**Реальный remaining benefit MIR**:
- Backend-specific lowering (LLVM vs WASM vs JS) — пока не нужно
- Cross-language optimizations (inlining `with_capacity + N×insert_new`
  в одно alloca + N stores) — perf полезно, но не блокер

**Decision:** Ф.3 закрыта как **unnecessary** в текущей архитектуре.
Если в будущем появятся multiple backends (Plan 100+?) — пересмотр.
Если будут perf-critical use-cases — целевая оптимизация (Plan 60?).

Для map-literal'а конкретно: текущая архитектура (AST-desugar →
stdlib-method emit) сохраняет flexibility — менять стратегию через
Nova-side, не через compiler-source.

**Use-case:** Когда `with_capacity` formula evolves (swisstable migration,
load factor change), AST-emitted call site баковит формулу. Rust ловерит
`vec![]` в MIR — можно switch backing strategy per-target.

**Реализация:**
1. Сохранять `ExprKind::MapLit` в AST через type-check (не desugarить рано).
2. MIR-pass `lower_map_literals` в новом модуле `compiler-codegen/src/mir.rs`.
3. Backend (codegen + interp) консультируется с MIR.
4. Перевод existing tests на новую инфраструктуру.

**Размер:** ~500+ LOC; требует MIR infrastructure (отдельный модуль).

### Ф.4 — User-defined `#from_pairs` types (полный canonical relax)

**Use-case:** `BTreeMap`, `OrderedMap`, user `MyMap` — должны работать с
`[k:v]` literal без compiler-modification.

**Status сейчас:** Ф.23 заложил инфраструктуру (`from_pairs_types` set,
`inferred_target_type` AST поле, desugar использует target). Bootstrap
ограничивает: только canonical stdlib types в set. User-локальный
`#from_pairs` сейчас silently игнорируется.

**Релаксация:** добавить explicit registry для user-типов через AST-attribute
+ flag компилятора `--allow-user-from-pairs` (или `#[allow(custom-map-literal)]`
module-attribute).

**Размер:** ~50 LOC + тесты.
