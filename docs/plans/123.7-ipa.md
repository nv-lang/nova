## 1. Контекст

V1-V6 ship per-fn analysis: each fn analyzed in isolation. When a
method body calls another method (`@helper(x)`), V3 conservative
treats it как potential mutation barrier → invalidates mut caches.

**Gap:** if `@helper` is known to NOT mutate `@field`, the caller's
cache could survive the call. V7 IPA computes per-method
`field_write_set` (set of `@F` fields the method mutates), then
allows caller to preserve cache for fields ∉ callee.write_set.

---

## 2. Design

### 2.1 Field-write-set inference

Per FnDecl, compute `field_write_set: HashSet<String>`:
- Walk fn body recursively.
- For each `Assign { target: Member{SelfAccess, F}, ... }` →
  insert `F` into write_set.
- Recursive calls inferred via SCC analysis (similar to Purity
  inference Plan 33.2).
- For methods that call other methods, propagate write_set из
  callee → caller (union).

### 2.2 IPA invalidation rule (V7)

For mut field F в caller's body:
- Cache valid through `@helper(x)` call iff `F ∉ helper.field_write_set`.
- Existing V1 first-region cache extends across non-writing calls.

### 2.3 Annotation alternative

If automatic SCC inference too expensive, allow explicit
`#nofield_mut(<F1>, <F2>, ...)` annotation. Compiler verifies
annotation matches actual body (D24-style).

### 2.4 Conservative fallback

For methods without computed write_set (e.g., recursive cycles,
external calls):
- Conservative: assume mutates all fields (current V1 behavior).
- No regression от absence of IPA.

### 2.5 Scope V7

**V7 ship:** field_write_set inference + integration с D217 V1 mut
path (call-boundary now refined per IPA).

**V7.1 deferred:** integration с D218 LICM (allow mut LICM when
loop body's calls don't write F), D219 pure-cache (refine
invalidation through frame info), D217 V4 chain (per-segment
invalidation).

infrastructure investment beyond field caching scope.
