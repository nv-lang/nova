## 1. Контекст

После закрытия Plan 83.4.5.1-5 (cooperative-mode validation для 24 NEW
regressions из Ф.0 enumeration) + Plan 83.4.5.2 Ф.1-Ф.4 AsyncDetach
production-grade reimpl, **flip activation остаётся blocked** на
multi-worker double-resume race manifestируясь как:

```
nova: fiber stack overflow in slot 0 (access violation in fiber arena)
Hint: increase fiber stack size or reduce recursion
```

Это **misleading** error message — реальная причина НЕ stack overflow, а
**access violation** при concurrent mco_resume на одном fiber'е (TIB
swap conflict на Windows / minicoro context corruption на POSIX).

```c
if (q->fiber_ctx[i]) {
```

## 5. ПОЧЕМУ Plan 83.4.2 Ф.1 A2 fix НЕДОСТАТОЧЕН

A2 fix покрывает `supervised_step` skipping worker-owned fibers. Но race
происходит **НЕ** в `supervised_step`, а в одном из следующих мест:

### 5.1. Hypothesis A: Spawn-time race на scope-state initialization

Spawn путь под armed runtime:
1. Codegen на parent thread:
   ```c
   ctx->_nova_parent_scope = &q;
   ctx->_nova_init_snapshot = nova_alloc + save;  // Plan 83.4.5.4
   nova_runtime_spawn_into(&q, entry, ctx);
   ```
2. `nova_runtime_spawn_into`: `pending_remote++`, push to worker deque.
3. **Worker может pick up fiber IMMEDIATELY** (after deque push +
   uv_async_send).
4. Worker preamble (emit_c.rs lines 5260-5286) runs concurrently с
   parent's continued execution:
   ```c
   _nova_active_slot = nova_scope_alloc_slot(_nova_active_scope, _co);
   _c->_nova_worker_slot = _nova_active_slot;
   _c->_nova_fiber_scope = _nova_active_scope;
   _nova_active_scope->fiber_effect_snapshot[_nova_active_slot] = _c->_nova_init_snapshot;
   ```

**Race:** parent thread может ещё быть в processed-spawn-cleanup AND
worker thread already в fiber preamble — BOTH touch `q->fibers[]`,
`q->fiber_ctx[]`, `q->fiber_effect_snapshot[]` БЕЗ mutex/atomic
protection.

`nova_scope_alloc_slot` (fibers.h:469) использует linear search +
`q->count++` БЕЗ synchronization (assumes single-writer).

### 5.2. Hypothesis B: Concurrent worker preamble + parent's subsequent spawn

Если parent на main делает spawn → worker A picks up → parent делает
**второй** spawn → worker B picks up. Both workers call
`nova_scope_alloc_slot` concurrently → **race на `q->count`,
`q->fibers[]` writes**.

`fiber_arena` Plan 82 — per-thread storage. Если worker A allocates
fiber slot 0 with arena's coro and worker B also tries slot 0
(non-atomic `q->count` race) → both write to same slot →
**double-allocation → fiber stack overlay → access violation**.

### 5.3. Hypothesis C: supervised body execution + worker concurrent run

Parent thread в supervised body может вызвать `Time.sleep(0)` →
`_nova_time_default_sleep` (fibers.h:1709) → `nova_supervised_step`.
Этот supervised_step итерирует scope's fibers (skip worker-owned per
A2 fix). НО `q->count` читается без atomic — worker может в этот
момент инкрементить через `nova_scope_alloc_slot` из preamble.

**Reading non-atomic count concurrent with non-atomic write — UB on
x86 (in theory), и на practical level — может прочитаться partial-write
value** (если grew array reallocated в момент чтения старого pointer).

### 5.4. Какая из hypothesis правильная?

**Точно неизвестно** — не reproduced под debugger. Скорее всего, **все
три combinе** под достаточной concurrency. Plan 83.4.5.7 должен закрыть
все три atomically.

## 6. РЕКОМЕНДУЕМЫЙ DESIGN

### 6.1. Atomic fiber state machine per slot (~2-3 dev-day)

Add `nova_atomic_int fiber_state[]` array to NovaFiberQueue, parallel to
`fibers[]`. States:
- `0` (EMPTY) — slot free, no fiber.
- `1` (ALLOCATING) — worker preamble in progress (nova_scope_alloc_slot
  reserved slot, но ещё не setup'нул всё).
- `2` (RUNNABLE) — fiber suspended, ready для resume.
- `3` (RUNNING) — some thread is running mco_resume.
- `4` (PARKED) — suspended via park/park_until.
- `5` (DEAD) — terminated.

**State transitions ТОЛЬКО через atomic CAS.** Worker BEFORE
`mco_resume(co)`:
```c
int expected = NOVA_FIBER_RUNNABLE;
if (!nova_aint_cas(&q->fiber_state[i], &expected, NOVA_FIBER_RUNNING)) {
    /* Another thread is already running this fiber. Skip. */
    continue;
}
mco_resume(co);
nova_aint_store(&q->fiber_state[i], NOVA_FIBER_RUNNABLE, RELEASE);
```

Main `supervised_step` тоже атомарно guards mco_resume. Worker и main
**can NEVER simultaneously execute mco_resume** на одном fiber'е.

**Cross-runtime reference:** Go runtime `g.atomicstatus`,
tokio `task::core::Header::state` (bit-packed atomic), Kotlin
`JobSupport.state_`.

### 6.2. Atomic scope-slot allocation (~0.5-1 dev-day)

`nova_scope_alloc_slot` (fibers.h:469) сделать atomic:
```c
int slot = nova_aint_inc(&q->count) - 1;  /* atomic fetch-add */
if (slot >= q->capacity) {
    /* Grow under mutex (rare, one-time per growth). */
    nova_mutex_lock(&q->grow_mu);
    if (slot >= q->capacity) {
        nova_scope_grow(q, slot + 1);
        if (q->sched_state) nova_sched_grow_state(q, q->capacity);
    }
    nova_mutex_unlock(&q->grow_mu);
}
q->fibers[slot] = co;
/* ... rest of init ... */
```

CAS-based concurrent slot allocation БЕЗ mutex contention в hot path.
Slot reuse (через `nova_scope_free_slot` setting fibers[i]=NULL) НЕ
работает с atomic FAA pattern — нужен либо separate freelist либо
mark-only without slot reuse.

**Alternative:** keep linear-search allocation but под mutex (simpler,
но contention на каждый spawn).

## 7. ЧТО НЕ ДЕЛАТЬ — ANTI-PATTERNS

### 7.1. ❌ НЕ пытайся "просто положить mutex" в supervised_step

Mutex на supervised_step's for-loop серьёзно деградирует performance
(каждый resume под lock'ом). Производительности teardown'а будет хуже
single-thread.

### 7.2. ❌ НЕ убирай Plan 83.4.2 Ф.1 A2 fix

A2 fix (skip worker-owned в supervised_step) КОРРЕКТЕН и НУЖЕН. Это
**один из layers** защиты от race — не единственный. Don't revert.

### 7.3. ❌ НЕ полагайся на `mco_status` для синхронизации

`mco_status(co)` returns `MCO_RUNNING/MCO_SUSPENDED/MCO_DEAD` но **НЕ
атомарно**. Read-write race есть. Use atomic `fiber_state[]` slot
(раздел 6.1) вместо.

### 7.4. ❌ НЕ скрывай race через дополнительные NO_AUTOARM directives

Plan 83.4.5.x уже добавил 12 directives как escape hatch. **Не
добавляй больше** — это маскирует bug. Цель Plan 83.4.5.7 — **снять**
directives, не накопить ещё.

### 7.5. ❌ НЕ начинай с TSAN gate

TSAN gate на текущем коде будет flood'ить findings. Сначала fix race
через atomic state machine, ПОТОМ TSAN gate как validation.

### 7.6. ❌ НЕ пытайся fix через "increase fiber stack size"

Error message **misleading** — реальная причина access violation на TIB
swap conflict, не stack overflow. Increasing stack size не поможет.

### 7.7. ❌ НЕ пытайся убрать AsyncDetach обратно на SyncDetach

Plan 83.4.5.2 Ф.4 уже сделал AsyncDetach default через
`nova_runtime_spawn_orphan`. Это **production-grade** D50 semantic.
Revert на SyncDetach был бы regression. Тесты detach_test уже
migrated на explicit-sync через `runtime.drain_orphans()`.

## 8. Cross-runtime reference

### Go scheduler — `runtime.gopark` / `goready`

Go's `g` (goroutine) struct имеет `atomicstatus` атомарный field. Все
state transitions через atomic CAS — `Gpark`, `Grunning`, `Grunnable`,
`Gdead`.

Worker M подхватывает G только если atomic CAS `Grunnable → Grunning`
succeeds. Concurrent attempt fails — другой M proceeds.

Code reference: `runtime/proc.go::casgstatus`.

### tokio task scheduler — `Header.state`

tokio's `Task` struct имеет bit-packed atomic `state`:
- `RUNNING` bit — fiber currently executing.
- `NOTIFIED` bit — wake pending.
- `JOIN_INTEREST` bit — handle waiting on completion.
- `COMPLETE` bit — task finished.

Все transitions через atomic ops. `poll` (resume) только после CAS
`NOTIFIED → RUNNING` succeeds.

Code reference: `tokio/src/runtime/task/state.rs`.

### Kotlin coroutines — `JobSupport.state`

Atomic state machine. CAS `INACTIVE → ACTIVE` для start. Children
list под atomic linked-list (lock-free).

Code reference: `kotlinx.coroutines/JobSupport.kt`.

### Nova сейчас (что нужно изменить)

- `mco_status(co)` возвращает minicoro state. Но `mco_resume(co)` **НЕ
  атомарно** — это просто TIB swap + jump. Если два thread'а call
  `mco_resume` на одном `co` — double-swap → corruption.
- `q->count`, `q->fibers[]`, `q->fiber_ctx[]` — обычные **non-atomic**
  arrays. Multi-thread writes — race.

## 10. Risk register

| Риск | Mitigation |
|---|---|
| Atomic state machine adds overhead на hot path | Profile vs cooperative drain; expected ~10-20ns/transition. tokio/Go аналогично. Acceptable. |
| Slot-state inconsistency cross-thread | Memory ordering carefully: RELEASE на store, ACQUIRE на load CAS. Plan 44.1 R-audit pattern. |
| Existing tests break на CAS semantics | Bootstrap path (no CAS) preserved как fallback when `_armed == false`. |
| TSAN findings вне scope 83.4.5.7 | Document в simplifications.md как followup. |
| Slot freelist complexity для atomic FAA + reuse | V1: no slot reuse под armed (unbounded growth bounded by `nova_alloc` budget). V2 followup: per-thread freelist либо CAS-based pop. |
| Plan 82 fiber_arena multi-worker interaction | Plan 82 already supports multi-worker (per-thread arena). Validate compatibility в Ф.4 stress. |
