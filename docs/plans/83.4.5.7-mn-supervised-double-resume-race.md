## 2. Plan 83.4.2 Ф.1 A2 fix — что покрывает

Plan 83.4.2 Ф.1 (commit 7fa8e8af81c) добавил skip-логику в
`nova_supervised_step` для worker-owned fibers:

```c
if (q->fiber_ctx[i]) {
    NovaSpawnCtxBase* base = (NovaSpawnCtxBase*)q->fiber_ctx[i];
    if (base->_nova_parent_scope) {
        alive++;  /* worker owns; main skips */
        continue;
    }
}
```

Это покрывает базовый scenario: main thread в supervised_run drain
loop'е итерирует scope's fibers и SKIP'ает worker-owned (parent_scope
!= NULL) — main НЕ вызывает mco_resume на них.

## 3. Corner case symptoms (что A2 fix НЕ покрывает)

**Hypothesis:** race не в `supervised_step`'е, а в:

### 3.1. Spawn-time race: scope-state initialization

Spawn путь под armed runtime:
1. Codegen on parent thread: `ctx->_nova_parent_scope = &q;`
2. Codegen on parent thread:
   `ctx->_nova_init_snapshot = nova_alloc + save` (Plan 83.4.5.4).
3. `nova_runtime_spawn_into(&q, entry, ctx)` — `pending_remote++`,
   `nova_runtime_spawn_global(entry, ctx)` → push to worker deque.
4. Worker может pick up fiber IMMEDIATELY (deque push + uv_async_send).
5. **Worker preamble** runs concurrently with parent's continued
   execution. Allocates slot via `nova_scope_alloc_slot`.

Race: parent thread может ещё быть в processed-spawn-cleanup AND
worker thread already in fiber preamble — BOTH touch `q->fibers[]`,
`q->fiber_ctx[]`, `q->fiber_effect_snapshot[]` без mutex/atomic
protection.

`nova_scope_alloc_slot` использует
linear search + `q->count++` без synchronization (assumes
single-writer).

### 3.2. Worker preamble + parent's subsequent spawn

Если parent на main делает spawn → worker A picks up → parent делает
**второй** spawn → worker B picks up. Both workers call
`nova_scope_alloc_slot` concurrently → race on `q->count`,
`q->fibers[]` writes.

`fiber_arena` Plan 82 — per-thread storage. Если worker A allocates
fiber slot 0 with arena's coro and worker B also tries slot 0
(non-atomic `q->count` race) → both write to same slot →
double-allocation → fiber stack overlay → access violation.

### 3.3. supervised body execution + worker concurrent run

Parent thread в supervised body может вызвать `Time.sleep(0)` →
`_nova_time_default_sleep` → supervised_step. Этот supervised_step
итерирует scope's fibers (skip worker-owned per A2). НО `q->count`
читается без atomic — worker может в этот момент инкрементить через
nova_scope_alloc_slot из preamble.

Reading non-atomic count concurrent with non-atomic write — UB on x86
(in theory), и на practical level — может прочитаться partial-write
value.

## 4. Cross-runtime reference

### Go scheduler — `runtime.gopark` / `goready`

Go's `g` (goroutine) struct имеет `atomicstatus` атомарный field. Все
state transitions через atomic CAS — `Gpark`, `Grunning`, `Grunnable`,
`Gdead`.

Worker M подхватывает G только если atomic CAS `Grunnable → Grunning`
succeeds. Concurrent attempt fails — другой M proceeds.

### tokio task scheduler — `Header.state`

tokio's `Task` struct имеет bit-packed atomic `state`:
- `RUNNING` bit — fiber currently executing.
- `NOTIFIED` bit — wake pending.
- `JOIN_INTEREST` bit — handle waiting on completion.

Все trans через atomic ops. `poll` (resume) только после CAS
`NOTIFIED → RUNNING` succeeds.

### Kotlin coroutines — `JobSupport.state`

Atomic state machine. CAS `INACTIVE → ACTIVE` для start.

### Nova сейчас

`mco_status(co)` возвращает minicoro state. Но `mco_resume(co)` НЕ
атомарно — это просто TIB swap + jump. Если два thread'а call
`mco_resume` на одном `co` — double-swap → corruption.

`q->count`, `q->fibers[]`, `q->fiber_ctx[]` — обычные non-atomic
arrays. Multi-thread writes — race.

## 5. Дизайн решения

### Ф.1 — Atomic fiber state per slot (~1-1.5 dev-day)

Add `nova_atomic_int fiber_state[]` array to NovaFiberQueue,
parallel to `fibers[]`. States:
- `0` — EMPTY (slot free).
- `1` — ALLOCATING (worker preamble in progress).
- `2` — RUNNABLE (fiber suspended, ready to resume).
- `3` — RUNNING (some thread is running mco_resume).
- `4` — PARKED (suspended via park_until).
- `5` — DEAD (terminated).

State transitions via atomic CAS. Worker before `mco_resume(co)`:

```c
int expected = NOVA_FIBER_RUNNABLE;
if (!nova_aint_cas(&q->fiber_state[i], &expected, NOVA_FIBER_RUNNING)) {
    /* Another thread is already running this fiber. Skip. */
    continue;
}
mco_resume(co);
nova_aint_store(&q->fiber_state[i], NOVA_FIBER_RUNNABLE, RELEASE);
```

Main supervised_step тоже атомарно guards mco_resume. Worker и main
can NEVER simultaneously execute mco_resume on the same fiber.

### Ф.2 — Atomic scope-slot allocation (~0.5 dev-day)

`nova_scope_alloc_slot` сделать atomic:
```c
int slot = nova_aint_inc(&q->count) - 1;  /* atomic FAA */
if (slot >= q->capacity) {
    /* Grow under mutex (rare, one-time). */
    nova_mutex_lock(&q->grow_mu);
    if (slot >= q->capacity) nova_scope_grow(q, slot + 1);
    nova_mutex_unlock(&q->grow_mu);
}
q->fibers[slot] = co;
/* ... */
```

CAS-based concurrent slot allocation без mutex contention в hot path.

## 7. Risk

| Риск | Mitigation |
|---|---|
| Atomic state machine adds overhead на hot path | Profile vs cooperative drain; expected ~10-20ns/transition. tokio/Go аналогично. |
| Slot-state inconsistency cross-thread | Memory ordering carefully: RELEASE на store, ACQUIRE на load CAS. Plan 44.1 R-audit pattern. |
| Existing tests break на CAS semantics | Bootstrap path (no CAS) preserved as fallback when `_armed == false`. |
| TSAN findings | TSAN gate в Ф.4 — must be clean на concurrency/ tests. |
