# Plan 91.15 — std API tuning

> **Создан:** 2026-06-17. **Статус:** 📋 proposed.
> **Приоритет:** H.
> **Родитель:** [Plan 91](91-stdlib-mvp-for-0.1.md).

## Мотивация

Plan 91 закрыл std MVP. После закрытия накопились мелкие несоответствия и
broken-state items в std API, которые блокируют корректный `nova test` или
портят UX программиста:

1. **`[M-91.8b-precompiled-c-rebuild]` (Priority H)** — `str @eq` переименован
   в `@equal` (Plan 91.8b), но precompiled `.c` файлы (`std/collections/*.c`)
   не пересобраны. `Nova_str_method_eq` всё ещё referenced в `vec_seq.c`,
   `set.c`, `hashmap.c` → CC-FAIL при тестах `plan91_8b/pos_equal_dispatch` и
   `pos_str_equal`. Блокирует корректный regress.

2. **`[M-91.8a.2-default-body-general]`** — generalized AST-driven synthesis для
   default protocol bodies (Equal/Compare/Display) — MVP shortcut в `emit_c.rs`
   покрывает только `@equal` и `@compare`, общий механизм не дописан.
   Видно в simplifications.md как known shortcut.

3. **`[M-91.14-option-result-debug]`** — `Option[T]` и `Result[T,E]` не имеют
   `@debug` impl. `${opt:?}` и `${res:?}` падают или дают неинформативный вывод.

4. **`[M-91.14-derive-debug]`** — `#impl(Debug)` auto-derive через Plan 126.
   Сейчас каждый тип пишет `@debug` вручную. Auto-derive нужен для продуктивной
   работы.

5. **`[M-91.10-remove-needs-caps-field]`** — AST поле `needs_caps: Vec<String>`
   всегда пустое после D163 retract, но occupies ~10 construction sites.
   Минорная cleanup, но долг растёт.

6. **Мелкие API gaps** (по аудиту std/):
   - `str.split(sep str) -> []str` — отсутствует как Nova-body метод
     (есть `split` как external, но сигнатура неудобна)
   - `str.repeat(n int) -> str` — отсутствует
   - `[]T.contains(val T) -> bool [T Equal]` — отсутствует в vec_seq.nv
     (есть в plan153_3 как `@contains` но не везде доступен через prelude)
   - `int.clamp(lo int, hi int) -> int` — отсутствует
   - `f64.clamp(lo f64, hi f64) -> f64` — отсутствует

## Фазы

### Ф.1 — Fix `[M-91.8b-precompiled-c-rebuild]` (blocker H)

**Цель:** устранить CC-FAIL от `Nova_str_method_eq` в precompiled `.c`.

- Найти все precompiled `.c` в `std/collections/`: `vec_seq.c`, `set.c`,
  `hashmap.c` и др.
- Проверить каждый на `Nova_str_method_eq` — заменить на `Nova_str_method_equal`
  (или пересобрать через `nova build-std` если такая команда есть).
- Убедиться: `nova test nova_tests/plan91_8b/` — все PASS.
- Убедиться: full `nova test` — нет новых CC-FAIL.

### Ф.2 — `Option[T]` и `Result[T,E]` @debug impl

**Цель:** `${opt:?}` и `${res:?}` работают корректно для любых T.

- В `std/prelude/core.nv` (или `std/prelude/option.nv`): добавить
  `fn Option[T] @debug(sb consume StringBuilder) -> StringBuilder [T Debug]`
  — `Some(val)` → `"Some(${val:?})"`, `None` → `"None"`.
- Аналогично для `Result[T,E]`: `Ok(val)` / `Err(err)`.
- POS тест: `${Some(42):?}` → `"Some(42)"`, `${None:?}` → `"None"`,
  `${Ok(1):?}` → `"Ok(1)"`, `${Err("x"):?}` → `"Err(\"x\")"`.
- NEG тест: `${Some(my_type_without_debug):?}` → compile error (E_BOUND_MISSING).

### Ф.3 — `#impl(Debug)` auto-derive

**Цель:** убрать boilerplate `@debug` для plain record/value типов.

- В `types/mod.rs`: при обработке `#impl(Debug)` на record/value-type без
  explicit `@debug` — синтезировать тело: перебор полей → `"TypeName { field1:
  ${@field1:?}, ... }"`.
- Tuple newtype (`type T(inner)`) → `"T(${@0:?})"`.
- Sum type → синтезировать для каждого variant'а.
- Поля с `priv` — включать или исключать? **Решение:** включать (debug для
  разработчика, не для пользователя); поле можно пометить `#debug_skip` как
  followup.
- POS тест: `type Point { x int, y int } #impl(Debug)` — `${pt:?}` →
  `"Point { x: 1, y: 2 }"`.
- NEG тест: `#impl(Debug)` на type с полем без `[T Debug]` bound → compile error.

### Ф.4 — Мелкие str/numeric/vec API gaps

**Цель:** закрыть наиболее часто нужные методы.

- `str.split(sep str) -> []str` — Nova-body (через `find` + `slice`).
- `str.repeat(n int) -> str` — Nova-body (через StringBuilder).
- `[]T.contains(val T) -> bool [T Equal]` — в `std/collections/vec_seq.nv`
  если не покрыто plan153_3.
- `int.clamp(lo int, hi int) -> int` — в `std/prelude/core.nv`.
- `f64.clamp(lo f64, hi f64) -> f64` — аналогично.
- POS тесты для каждого метода.
- NEG: `"abc".split("")` (пустой sep) — паника или error? **Решение:** паника
  (аналог Python/Go), `assert(sep.len() > 0)` в теле.

### Ф.5 — `[M-91.10-remove-needs-caps-field]` cleanup

**Цель:** удалить мёртвое поле из AST.

- Найти все ~10 construction sites `needs_caps: vec![]` / `needs_caps: …`
  в `compiler-codegen/src/`.
- Удалить поле из `ExternalFnDecl` (или где живёт).
- Убедиться: `cargo build` OK, `nova test` без регрессий.

### Ф.6 — Spec/D + тесты + закрытие

- D-block amend: D132 (str API), D141 (vec API), D184 (operator dispatch — note
  про precompiled .c fix), D178 (str API amend).
- `nova_tests/plan91_15/` — сборная фикстура для всех новых методов.
- `docs/project-creation.txt`, `simplifications.md`, `backlog-followups.md`,
  `nova-private/discussion-log.md` — обновить.
- Закрыть маркеры: `[M-91.8b-precompiled-c-rebuild]`, `[M-91.14-option-result-debug]`.
