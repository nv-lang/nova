# Plan 50: Параметр с дефолтом — keyword-only на месте вызова

>
> **Зависит от:** [Plan 46](46-named-parameters.md) (базовая реализация

---

## Зачем

Ревизия D102: **параметр со значением по умолчанию передаётся только
по имени**, позиционно — compile error.

```nova
fn connect(host str, port int = 8080, tls bool = false) -> Conn

connect("localhost", port: 9000)     // ок
connect("localhost", 9000)           // ОШИБКА — port с дефолтом
connect("localhost", 9000, true)     // ОШИБКА — нечитаемые позиционные флаги
```

Причина (см. D102 §«Почему» + §«Эволюция»): позиционные `bool`/`int`-флаги
(`connect("h", false, true)`) остаются нечитаемыми и частой ошибкой
LLM-генерации даже при наличии именованных аргументов. Правило переводит
этот класс багов из «нежелательно» в **ошибку компиляции** — прямо по
AI-first миссии языка. Ментальная модель: **обязательный — позиционно,
опциональный — по имени.**

Базовая реализация D102 (Plan 46) уже разрешала позиционную привязку к
дефолтному параметру — это поведение и меняется.

---

## Архитектурное решение

Plan 46 уже даёт всю инфраструктуру:
- `argbind.rs` — `bind_call_args`, единая логика сопоставления
  аргументов с параметрами.
- type-checker `check_call_argbind` (в `BoundCtx`) — diagnostics.
- `callnorm.rs` — early-pass named→positional + defaults.

Plan 50 добавляет **одно правило** в diagnostic-фазу `check_call_argbind`:

> При сопоставлении аргументов: если **позиционный** аргумент
> связывается с параметром, у которого `default.is_some()` — это
> ошибка.

**Исключение — trailing-форма (D43).** Trailing-block / trailing-fn
заполняет последний функциональный параметр и синтаксически отлична от
позиционного аргумента в `(...)`. Если последний параметр имеет дефолт
и заполнен trailing-формой — это **не** «позиционный аргумент дефолтного
параметра», ошибки нет. Правило бьёт только по позиционным аргументам
внутри скобок вызова.

`callnorm.rs` семантически не меняется: он работает после type-check,
и к моменту нормализации запрещённых случаев уже нет. Менять normalization
не требуется.

## Фазы

### Ф.0 — Правило в `argbind` / `check_call_argbind`

- В `bind_call_args` (или `check_call_argbind`): при привязке
  позиционного аргумента к параметру отметить, если у параметра есть
  дефолт → диагностика.
- Учесть **исключение trailing-формы**: параметр, заполненный
  trailing-block / trailing-fn, не считается «позиционно привязанным»
  — ошибки нет даже при наличии дефолта.
- Покрыть единообразно: free functions, static-методы, instance-методы
  (`@`-методы), конструкторы — везде, где работает `bind_call_args`.
- `builtins.nv` / external-функции с дефолтами — то же правило.

### Ф.1 — Production-grade диагностика

Сообщение об ошибке — **actionable**, с готовым фиксом:

```
error: parameter `port` has a default value and must be passed by name
  --> app.nv:12:24
   |
12 |     connect("localhost", 9000)
   |                          ^^^^ pass as `port: 9000`
   |
   = note: parameters with defaults are keyword-only (D102)
```

- Указывать **имя параметра** и предлагать `name: <expr>` форму.
- При нескольких позиционных аргументах на дефолтных параметрах —
  отдельная диагностика на каждый (не «первая и стоп»).
- Span — на конкретный аргумент-выражение, не на весь вызов.

### Ф.2 — Миграция кодовой базы

Полный sweep (без упрощений — мигрировать **все** случаи, не
выборочно):
- `nova_tests/**` — все вызовы, где дефолтный параметр передан
  позиционно (включая тесты, добавленные Plan 46), → именованная форма.
- `std/**` — то же.
- Тесты/примеры из Plan 46, опирающиеся на старое поведение, —
  переписать либо на named-форму, либо (если проверяли именно
  позиционную привязку дефолта) превратить в negative-тесты Ф.3.
- После миграции — полная регрессия `nova test` (release-сборка)
  должна быть зелёной.

### Ф.3 — Тесты

`nova_tests/.../` — negative (`EXPECT_COMPILE_ERROR`):
- позиционный аргумент на дефолтном параметре (1 дефолт);
- позиционный аргумент на дефолтном параметре (среди нескольких);
- несколько позиционных на дефолтных — проверить, что диагностика на
  каждый;
- позиционный дефолт в instance-методе / конструкторе.

`nova_tests/.../` — positive:
- дефолтный параметр по имени — ок;
- дефолтный параметр опущен — ок;
- **trailing-форма на последнем функциональном параметре с дефолтом —
  ок** (исключение Ф.0);
- обязательный параметр позиционно + дефолтный по имени — ок;
- обязательный параметр по имени — ок (keyword-only не запрещает
  именовать обязательные).

### Ф.4 — Spec sync + docs

- D102 ревизия — уже в `03-syntax.md` (готово: §«Что», §«Правило —
  вызов», §«Почему», §«Что отвергнуто», §«Эволюция»).
- Обновить `docs/project-creation.txt` + `docs/simplifications.md`
  (bootstrap-ограничения — как `[M*]`; в частности зафиксировать, если
  treewalk-interp ([M-interp-named]) требует доработки под новое
  правило).
- Запись в discussion-log private-репы.

## Что НЕ входит

- **Исключение «один дефолт → позиционно разрешён»** — отвергнуто в
  D102 §«Что отвергнуто» (рефакторинг-ловушка, количество дефолтов не
  показатель риска).
- **Обязательные имена для всех параметров** (Swift-style) — отвергнуто;
  keyword-only применяется только к дефолтным.
- **Per-параметр opt-in в позиционность** (Swift `_`) — отложено,
  отдельное решение если понадобится.
- **Изменение `callnorm.rs` normalization** — не требуется (правило в
  diagnostic-фазе, до нормализации).

## Acceptance criteria

- [ ] Позиционный аргумент на параметре с дефолтом — compile error
      (free fn, static-метод, instance-метод, конструктор, external).
- [ ] Trailing-форма на последнем функциональном параметре с дефолтом —
      по-прежнему допустима.
- [ ] Диагностика actionable: имя параметра + предложение `name: expr`;
      отдельное сообщение на каждый нарушающий аргумент.
- [ ] Обязательные параметры по-прежнему можно передать и позиционно, и
      по имени.
- [ ] Кодовая база (`nova_tests/`, `std/`) полностью мигрирована — нет
      позиционных вызовов дефолтных параметров.
- [ ] Все negative + positive тесты Ф.3 PASS.
- [ ] Полная регрессия `nova test` без новых FAIL (release-сборка).
