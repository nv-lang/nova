# Plan 50: Параметр с дефолтом — keyword-only на месте вызова

> **Создан 2026-05-15.** Production-grade, без упрощений.
>
> **СТАТУС:** план, не начат.
>
> **Реализует:** ревизию [D102](../../spec/decisions/03-syntax.md#d102-именованные-аргументы-и-значения-параметров-по-умолчанию)
> от 2026-05-15 («параметр с дефолтом — keyword-only»).
>
> **Зависит от:** [Plan 46](46-named-parameters.md) (базовая реализация
> named args + дефолтов — закрыт). Plan 50 — **ограничение** поверх
> существующей binding-логики плюс закрытие связанного с ней долга
> ([M-interp-named]).
>
> **Приоритет:** P1 — делать сейчас, пока stdlib и пользовательский код
> не наплодили позиционных вызовов дефолтных параметров.

---

## Зачем

Ревизия D102: **параметр со значением по умолчанию передаётся только
по имени**, позиционно — compile error. Правило type-agnostic — касается
дефолтного `bool`, `int`, `string`, struct, любого типа.

```nova
fn connect(host str, port int = 8080, tls bool = false) -> Conn

connect("localhost", port: 9000)     // ок
connect("localhost", 9000)           // ОШИБКА — port с дефолтом
connect("localhost", 9000, true)     // ОШИБКА — нечитаемые позиционные флаги
```

Причина (D102 §«Почему» + §«Эволюция»): позиционные `bool`/`int`-флаги
остаются нечитаемыми и частой ошибкой LLM-генерации даже при наличии
именованных аргументов. Правило переводит этот класс багов из
«нежелательно» в **ошибку компиляции** — прямо по AI-first миссии языка.
Ментальная модель: **обязательный — позиционно, опциональный — по имени.**

Базовая реализация D102 (Plan 46) разрешала позиционную привязку к
дефолтному параметру — это поведение и меняется.

---

## Сравнение с Go / Rust / TS

| | Default-параметры | Named-аргументы | «Позиционный флаг» защищён? |
|---|---|---|---|
| **Go** | нет вообще | нет | n/a — проблемы нет, но и эргономики нет; обходят functional options (`WithTimeout(...)`) — boilerplate |
| **Rust** | нет вообще | нет | n/a — обходят builder pattern / `Default` + struct-update — boilerplate |
| **TypeScript** | да, **позиционные** (`f(a, b = 8080)`) | нет (эмулируют object-параметром) | **нет** — `f(x, true, true)` компилируется молча |
| **Nova (D102 + Plan 50)** | да | да, first-class | **да — compile error** |

Go и Rust сознательно отказались от дефолтов вообще, чтобы не иметь
«позиционного флага» — но платят boilerplate'ом (functional options,
builder). TS дал дефолты, но позиционные — и получил ровно ту
нечитаемость, которую мы предотвращаем. Nova даёт **эргономику дефолтов
TS-уровня + guardrail, которого нет ни у кого**: дефолт обязан быть
именованным. Это строго лучше всех трёх — фича есть, а её главный
failure mode невозможен.

## Архитектурное решение

Plan 46 даёт инфраструктуру:
- `argbind.rs` — `bind_call_args`, единая логика сопоставления
  аргументов с параметрами.
- type-checker `check_call_argbind` (в `BoundCtx`) — diagnostics.
- `callnorm.rs` — early-pass named→positional + defaults (двухфазный
  Block), после type-check, до codegen.
- `interp/mod.rs` — treewalk-интерпретатор (`nova run`).

### Правило

> При сопоставлении аргументов вызова: если **позиционный аргумент
> внутри скобок вызова `(...)`** связывается с параметром, у которого
> `default.is_some()` — это compile error.

Правило живёт в `bind_call_args` (производит факт «позиционная привязка
к дефолтному параметру»), диагностика эмитится в `check_call_argbind`.
Это **compile-time only** — нулевая стоимость в рантайме, сгенерированный
C не меняется.

### Все пути привязки аргументов — покрыть единообразно

`bind_call_args` — единственная точка истины, поэтому правило
автоматически покрывает все формы вызова. Явно проверить, что путь
проходит через неё:

- свободные функции;
- instance-методы (`@`-методы) — receiver на позиции 0, всегда
  позиционный, дефолта не имеет; правило к нему неприменимо;
- static-методы и конструкторы (record-литерал — отдельная история,
  правило касается **вызовов**, не литералов);
- external / `builtins.nv`-функции — дефолты допустимы в их декларации,
  правило то же;
- generic-функции (`emit_generic_fn_erased` / `emit_generic_method_erased`)
  — биндинг по той же сигнатуре, правило то же.

**Вызов через значение функционального типа.** Тип `fn(int, int) -> T`
**не несёт ни имён параметров, ни дефолтов** (они стираются при
получении fn-value). Значит вызов через fn-value — всегда чисто
позиционный, дефолтных параметров там быть не может, правило
неприменимо и это консистентно. Зафиксировать тестом, чтобы поведение
было намеренным, а не случайным.

### Исключение — trailing-форма (D43)

Trailing-block (`f(args) { block }`) и trailing-fn (`f(args) fn(p) body`)
заполняют **последний функциональный параметр** и синтаксически отличны
от позиционного аргумента в `(...)`. Правило бьёт **только по
позиционным аргументам внутри скобок вызова**. Поэтому:

- последний функциональный параметр **с дефолтом**, заполненный
  trailing-формой → **ок** (это не «позиционный аргумент в скобках»);
- тот же параметр, переданный позиционно в `(...)` → **ошибка**;
- тот же параметр, переданный именованным аргументом → ок;
- передать его и trailing-формой, и именованным аргументом → ошибка
  «связан дважды» (правило D102 №5, не Plan 50).

### Precedence диагностик

Правило Plan 50 встраивается в существующий набор argbind-диагностик.
Порядок проверок фиксируется, чтобы сообщения были осмысленными:

1. **Арность** — слишком много / слишком мало аргументов. При переборе
   позиционных аргументов сообщать «too many arguments», **не**
   «must be named» для лишних.
2. **Неизвестное имя** именованного аргумента.
3. **Двойная привязка** параметра.
4. **Позиционный после именованного.**
5. **Позиционный аргумент на дефолтном параметре** (Plan 50) — когда
   арность и порядок корректны, но позиционный лёг на дефолтный
   параметр.

### `callnorm.rs` — без изменений (обоснование)

`callnorm` нормализует named→positional + заполняет дефолты и работает
**после** type-check. К моменту нормализации запрещённые Plan 50 случаи
уже отсеяны диагностикой — `callnorm` видит только валидные вызовы
(позиционные на обязательных + именованные на любых + опущенные
дефолты). Его логика не меняется. Проверить: нет code path, где
`callnorm` запускается на не прошедшем type-check вводе.

## Фазы

### Ф.0 — Правило в `argbind` + все пути + precedence

- В `bind_call_args` пометить факт «позиционная привязка к параметру с
  дефолтом»; учесть исключение trailing-формы (параметр, заполненный
  trailing-block/trailing-fn, не считается позиционно-привязанным).
- В `check_call_argbind` — эмиссия диагностики с учётом precedence
  (см. выше): сначала арность/порядок, потом keyword-only.
- **Error recovery:** после эмиссии ошибки параметр считается
  связанным (значением переданного выражения) — продолжать проверку
  остального вызова без каскада ошибок.
- Верифицировать, что free fn / instance / static / constructor /
  external / generic — все идут через `bind_call_args`.

### Ф.1 — Production-grade диагностика

Сообщение — actionable, с **machine-applicable structured suggestion**
(готово к LSP code-action / будущему `nova fix`):

```
error: parameter `port` has a default value and must be passed by name
  --> app.nv:12:26
   |
12 |     connect("localhost", 9000)
   |                          ^^^^ help: pass as `port: 9000`
   |
   = note: parameters with defaults are keyword-only (D102)
```

- Структурированная замена: span аргумента → `"<name>: <original-text>"`,
  помечена machine-applicable (чтобы инструмент мог авто-применить).
- Имя параметра и его декларация — в diagnostic (`note: parameter
  declared here`).
- При нескольких нарушающих аргументах — **отдельная** диагностика на
  каждый (не «первая и стоп»), каждая со своим suggestion.
- Span — на конкретное выражение-аргумент, не на весь вызов.

### Ф.2 — Treewalk-интерпретатор: закрыть [M-interp-named]

Plan 50 делает named args **обязательными** для дефолтных параметров —
значит корректная раскладка named→param-order в `nova run` становится
**load-bearing**, а не edge-case. Поэтому Plan 50 закрывает
[M-interp-named] (Plan 46 оставил его как priority-L долг):

- `interp/mod.rs` `CallArg::Named` arm — перевести на полную раскладку
  через `argbind::bind_call_args` напрямую (вариант из simplifications.md
  «interp call-handling использует bind_call_args»), либо `cmd_run`
  делает `resolve_imports_inline` + полную нормализацию для всех callee
  (включая импортированные).
- Покрыть: переставленные named args + опущенные дефолты + named для
  импортированного callee — всё в `nova run`.
- Снять запись [M-interp-named] из `simplifications.md` (→ RESOLVED).

### Ф.3 — Миграция кодовой базы и документации

Полный sweep (без упрощений — мигрировать **все** случаи):

- `nova_tests/**` — все вызовы с позиционным дефолтным аргументом
  (включая тесты, добавленные Plan 46) → именованная форма; тесты,
  проверявшие именно старое позиционное поведение, → negative-тесты Ф.4.
- `std/**` — то же.
- **`spec/**` и `docs/**`** — примеры кода, показывающие позиционный
  вызов дефолтного параметра (`connect("h", 9000)`-стиль), привести в
  соответствие новому правилу.
- Процедура: собрать сборку с правилом Ф.0 → собрать все диагностики →
  применить structured suggestions Ф.1 (batch) → ручная вычитка.
- Миграция — **отдельный commit** (или несколько, если объём большой),
  чтобы поведенческое изменение и механическая правка не смешивались.
- После миграции — полная регрессия `nova test` (release-сборка)
  зелёная.

### Ф.4 — Тесты

`nova_tests/.../` — negative (`EXPECT_COMPILE_ERROR`):
- позиционный аргумент на дефолтном параметре (единственный дефолт);
- позиционный на дефолтном среди нескольких параметров;
- несколько позиционных на дефолтных — отдельная диагностика на каждый;
- позиционный дефолт в instance-методе / static-методе / конструкторе;
- позиционный дефолт в generic-функции;
- позиционный дефолт + trailing-форма для другого параметра.

`nova_tests/.../` — positive:
- дефолтный параметр по имени — ок;
- дефолтный параметр опущен — ок;
- **trailing-форма на последнем функциональном параметре с дефолтом —
  ок** (исключение Ф.0);
- обязательный позиционно + дефолтный по имени — ок;
- обязательный по имени — ок (keyword-only не запрещает именовать
  обязательные);
- вызов через fn-value — позиционный, без дефолтов, ок (закрепить
  намеренность);
- **precedence:** слишком много позиционных аргументов → диагностика
  «too many arguments», НЕ «must be named»;
- `nova run`: переставленные named + опущенный дефолт + named для
  импортированного callee дают корректный результат (регрессия на
  Ф.2 / [M-interp-named]).

### Ф.5 — Spec sync + docs

- D102 ревизия — уже в `03-syntax.md` (готово).
- `docs/project-creation.txt` — запись о реализации (фазы, файлы,

- `docs/simplifications.md` — снять [M-interp-named] (RESOLVED, Ф.2);

## Что НЕ входит

- **Исключение «один дефолт → позиционно разрешён»** — отвергнуто в
  D102 §«Что отвергнуто» (рефакторинг-ловушка; количество дефолтов не
  показатель риска).
- **Обязательные имена для всех параметров** (Swift-style) — отвергнуто;
  keyword-only применяется только к дефолтным.
- **Per-параметр opt-in в позиционность** (Swift `_`) — отложено,
  отдельное решение, если math-функции начнут раздражать.
- **`nova fix` / LSP code-action** как инструмент — вне scope; Plan 50
  лишь делает диагностику **готовой** к авто-применению (structured
  suggestion). Сам инструмент — будущая задача tooling.
- **Изменение `callnorm.rs` normalization** — не требуется (правило в
  diagnostic-фазе, до нормализации; обоснование выше).
- **Изменение default-evaluation семантики** — Plan 46 territory; Plan 50
  только ограничивает форму передачи.

## Acceptance criteria

- [ ] Позиционный аргумент на параметре с дефолтом — compile error на
      **всех** путях: free fn, instance/static-метод, конструктор,
      external/builtin, generic-функция.
- [ ] Trailing-форма на последнем функциональном параметре с дефолтом —
      по-прежнему допустима; позиционно тот же параметр — ошибка.
- [ ] Вызов через fn-value — позиционный, без дефолтов; поведение
      закреплено тестом.
- [ ] Precedence: при переборе арности сообщается «too many arguments»,
      не «must be named».
- [ ] Диагностика production-grade: имя параметра, `note: declared
      here`, machine-applicable structured suggestion `name: expr`;
      отдельное сообщение на каждый нарушающий аргумент; error recovery
      без каскада.
- [ ] Обязательные параметры по-прежнему можно передать и позиционно,
      и по имени.
- [ ] [M-interp-named] закрыт: `nova run` корректно раскладывает
      переставленные named + опущенные дефолты, в т.ч. для
      импортированных callee. Запись снята из `simplifications.md`.
- [ ] Кодовая база (`nova_tests/`, `std/`) и примеры (`spec/`, `docs/`)
      полностью мигрированы — нет позиционных вызовов дефолтных
      параметров. Миграция — отдельный commit.
- [ ] Все negative + positive тесты Ф.4 PASS.
- [ ] Полная регрессия `nova test` без новых FAIL (release-сборка).
- [ ] Каждая фаза — отдельный commit.
