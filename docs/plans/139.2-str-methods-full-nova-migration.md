## Ключевая переоценка (investigation 2026-06-12, исправляет пессимизм Plan 139.1 Ф.B)
1. **Privacy НЕ блокер.** Privacy у Nova **type-based**, не module-based (`priv_access_allowed_base`, mod.rs:4730 — проверяет только `current_recv_type == tname`, **без проверки модуля**). `current_recv_type` ставится из receiver (mod.rs:3173) для ЛЮБОЙ функции с receiver. `fn str @method` (в string.nv, модуль runtime.string) имеет receiver `str` → `current_recv_type = "str"` → **видит priv-поля str** (`@.ptr`/`@.len`) и может конструировать `str{...}`. Прошлый вывод «string.nv не может» — ошибка.
2. **View-машинерия уже есть** (D144): `Vec[T] @index(r Range) -> Self { Self { data: unsafe { @data + r.start }, len: n, cap: n } }` (vec_owned.nv:717) — чистая Nova, value-record + unsafe-арифметика. `str[Range] -> str` тоже (D144). «Конструкция view из (ptr,len)» — не новый примитив.
3. **Единственный cross-type мост:** str-метод НЕ видит priv-поля **Vec** (другой тип). Для `@as_bytes`/`from_bytes_*` нужен **публичный Vec-конструктор** `Vec[T].from_raw_parts(ptr,len,cap)` + публичный data-ptr геттер. ~10 строк в vec_owned.nv, прямой аналог `@index(Range)`.
4. **`@hash` — обоснованное C-исключение.** Это **SipHash-1-3 + per-process random seed** (nova_rt.h:265-343, getrandom/BCryptGenRandom), НЕ FNV-1a (комментарий string.nv:286 устарел). Перенос в Nova = экспонировать seed = **DoS-регрессия HashMap**. Остаётся C. 9/10 переносимы.

## Ф.0 — PROOF (gate перед массовой миграцией)
**Цель:** эмпирически подтвердить архитектуру до миграции всех методов.
- (a) Крошечный str type-метод, читающий `@.ptr` и `@.len` + конструирующий `str{ptr:…,len:…}` — компилируется БЕЗ `E_PRIV_FIELD_READ`/`E_PRIV_FIELD_INIT` (доказывает type-method privacy).
- (b) Добавить в vec_owned.nv публичный `Vec[T].from_raw_parts(ptr *T, len int, cap int) -> Self` (+ публичный data-ptr геттер, напр. `@as_raw_ptr() -> *T`) — Vec type-метод, видит priv-поля Vec.
- (c) Мигрировать ОДИН метод end-to-end — `@as_bytes()` через `Vec.from_raw_parts(@.ptr, @.len, @.len)` — build + test зелёные.
**Критерии:** A0.1 proof-фикстура компилируется+работает; A0.2 Vec.from_raw_parts + геттер landed; A0.3 @as_bytes Nova-body, 0 регрессий str-suite. **Если (a) FAILS** (privacy блокирует) → STOP, report (архитектура неверна, не продолжать).

## Ф.1 — byte/view методы ✅ ЗАКРЫТА (2026-06-12)
`@byte_at` (`unsafe { @ptr[i] }`), финализ `@as_bytes`, `@len` (решить D117 `E_SIZE_ACCESSOR_FIELD` на чтение поля `len`: переименовать поле str (`len`→напр. `nbytes`, выверить ABI-alias nova_str) ИЛИ method-carve-out ИЛИ оставить C с обоснованием).
**Критерии:** byte/view методы Nova-body; pos+neg фикстуры; 0 регрессий.

**Решение D117 — option (b) method-carve-out (наименее-рисковое):**
- Отвергнут (a) rename `len`→`nbytes`: C-typedef `nova_str` использует поле `.len`
  в ~hundreds сайтах (`s.len`, NUL-term peek `s.ptr[s.len]`, array.h/nova_rt.h/.c) —
  rename Nova-поля требовал бы либо rename C-поля (огромный blast radius), либо
  codegen-маппинг `@nbytes`→`.len` (новая инфра). Высокий риск.
- Отвергнут (c) оставить C: цель плана — миграция; carve-out тривиален и точечен.
- Принят (b): D117-guard (emit_c.rs ~17625) пропускает bare `@len`-field-read iff
  obj=`SelfAccess` ∧ `current_receiver_type==Some("str")` ∧ `name=="len"` (реальное
  поле str). Внешний `s.len` по-прежнему `E_SIZE_ACCESSOR_FIELD`. Параллель с
  generic `Vec[T] @len() => @len` (там guard не срабатывал т.к. `nova_self`-тип
  не concrete на эмите шаблона). Документировано: D117 amend (03-syntax.md).

**Реализация:**
- `str @len() -> int => @len` (Nova-body, bare priv-field-read; бывший `nova_str_byte_len`, O(1)).
- `str @byte_at(i) -> u8`: `if i<0 || i>=@len() { panic("str.byte_at: index out of bounds") } unsafe { @ptr[i] }`
  (bounds-check + raw priv-pointer index; identical OOB-panic к бывшему `nova_str_byte_at`, nova_rt.h:122).
- `@as_bytes` уже Nova-body (Ф.0). string.nv + runtime_registry.rs (entries `len`/`byte_at`: `c_name:""`, `nova_body:Some(...)`).
- module-private `external fn panic` в string.nv (#no_prelude; mirrors vec_owned.nv/cstr.nv).

**Тесты (plan139_2/):** `len_ok` (5 cases: empty/ASCII/UTF-8-bytes/byte_len-alias/is_empty),
`byte_at_ok` (5: ASCII/single/multi-byte/as_bytes-agreement/last), `byte_at_oob_high` +
`byte_at_oob_neg` (EXPECT_RUNTIME_PANIC negatives). 4/4 PASS релизным.

## Ф.3 — `@concat` / `@compare`
Nova-body (`@concat` = alloc Vec[u8]+copy+from_bytes; `@compare` = byte-loop). Решить operator-lowering: операторы `+`/`<` сейчас лоуэрятся напрямую в `nova_str_concat`/`nova_str_cmp` (emit_c.rs:17137-17152). Либо (a) роутить операторы через методы (retire C-fn, риск perf), либо (b) оставить C-fn для операторов + Nova-body метод (дубль). Выбрать обоснованно, задокументировать perf.
**Критерии:** методы Nova-body; решение по lowering зафиксировано; pos+neg; 0 регрессий; reframe `[M-139.1-operator-lowered-methods]`.
