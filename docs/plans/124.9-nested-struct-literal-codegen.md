## 1. Проблема (как сообщено Plan 128.2 Ф.2)

Вложенный record-литерал с явными типами на каждом уровне:

```nova
type D { mut value int }
type C { mut d D }
type B { mut c C }
type A { mut b B }

mut a = A { b: B { c: C { d: D { value: 0 } } } }
```

Сообщено, что codegen field-value type inference брал **внешний** тип
структуры (`A`) вместо declared типа поля, аллоцируя `Nova_A` четыре раза
вместо `Nova_A`/`Nova_B`/`Nova_C`/`Nova_D`. Workaround в Plan 128.2 был
explicit `.new()` конструкторы вместо литералов.

Релевантный codegen: `compiler-codegen/src/codegen/emit_c.rs::emit_record_lit`
(plain-record path, ~line 24540): field values эмитятся через
`emit_expr_with_target_type(v, &field_ty)`, где `field_ty` = тип поля из
`record_schemas[struct_name][f.name]`. Вложенный `RecordLit` с собственным
`type_name` (`B { ... }`) должен резолвиться к своему типу.

## 2. Ф.0 — bug repro + baseline (THIS PHASE)

### Результат репродукции (2026-06-06, post-Plan-126.2 baseline)

Репро-файл `nova_tests/plan124_9/repro_nested_literal.nv` (4-level nested,
explicit type names на каждом уровне) **PASS** через `nova test-build`, и
сгенерированный C — **КОРРЕКТНЫЙ**:

```c
/* SRC: mut a = A { b: B { c: C { d: D { value: 0 } } } } */
Nova_A* _nv_tmp_297 = (Nova_A*)nova_alloc(sizeof(Nova_A));
Nova_B* _nv_tmp_298 = (Nova_B*)nova_alloc(sizeof(Nova_B));
Nova_C* _nv_tmp_299 = (Nova_C*)nova_alloc(sizeof(Nova_C));
Nova_D* _nv_tmp_300 = (Nova_D*)nova_alloc(sizeof(Nova_D));
_nv_tmp_300->value = ((nova_int)0LL);
_nv_tmp_299->d = _nv_tmp_300;
_nv_tmp_298->c = _nv_tmp_299;
_nv_tmp_297->b = _nv_tmp_298;
Nova_A* a = _nv_tmp_297;
```

Каждый уровень аллоцирует свой declared field-type — `Nova_A` НЕ
дублируется. **Bug как описан Plan 128.2 НЕ воспроизводится** на baseline с
влитым Plan 126.2 (merge содержит +351 строк в `types/mod.rs`). Вероятная
гипотеза: bug был побочно исправлен Plan 126.2 (или другим pipeline-fix между
Plan 128.2 Ф.2 и merge main 2026-06-06), либо оригинальный trigger был
anonymous-record вариант (`A { b: { c: ... } }` без inner-типов) — который
сейчас даёт честный `CODEGEN-FAIL: anonymous record literal without spread
not supported in codegen` (отдельная нереализованная фича, не wrong-type bug).

- **Ф.1 — Corner-case audit.** Проверить варианты, которые МОГЛИ бы триггерить
  wrong-type inference: (a) одинаковое имя поля на разных уровнях; (b) поле,

  return position; (d) value-record (`type X value`) nested literals;

  зафиксировать сгенерированный C.

  already-fixed** (если все corner-cases корректны). Если fix нужен —
  скорректировать `emit_record_lit` field-value path так, чтобы вложенный
  `RecordLit` с собственным `type_name` всегда резолвился к своему типу,
  игнорируя `expected_record_type` родителя.

## 4. Acceptance criteria

- **A124.9.1** — 4-level nested record literal `A { b: B { c: C { d: D { value:
  N } } } }` компилируется и каждый уровень аллоцирует свой declared
  field-type (`Nova_A`/`Nova_B`/`Nova_C`/`Nova_D`), НЕ внешний тип N раз.
- **A124.9.2** — Runtime: `a.b.c.d.value` читается и пишется корректно
  (round-trip read/write через 4-level chain).
- **A124.9.3** — Corner-case: nested literal с полем, имя которого совпадает с
  внешним типом, аллоцирует корректные типы (нет leak `expected_record_type`
  родителя во вложенный typed литерал).
- **A124.9.4** — Nested literal в fn-arg и return-position резолвит типы
  полей корректно.
- **A124.9.5** — Generic nested literal (`Box { value: Box { value: 0 } }`)
  мономорфизируется корректно на каждом уровне.
- **A124.9.6** — POS/NEG fixtures `nova_tests/plan124_9/` PASS; 0 regressions
  на full `nova test`; `[M-124.9-nested-record-literal-codegen]` закрыт
  (как fix либо как already-fixed-by-126.2 с зафиксированным baseline-доказательством).
