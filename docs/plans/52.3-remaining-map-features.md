## Что осталось

### Ф.1 — Empty array literal в map-position (`let m HashMap[K,V] = []`)

**Описание:** Codegen treats `[]` как пустой array, не пустую мапу.
Workaround: `HashMap[K,V].new()`.

**Подход:**
1. В `emit_array_lit` (или предшествующем pass) проверить
   `expected_record_type` или contextual hint — если expected это
   HashMap (или any #from_pairs тип), эмитить как `Target.new()`.
2. Альтернатива: в desugar.rs обнаружить `let x T = []` где T помечен
   #from_pairs и заменить на `T.new()` сразу.

### Ф.2 — Mono-pass quirk для generic helper-methods

**Investigation needed:** `HashMap.@clone()` / `@merge_from()` с
pattern-match в while-body падают с **разными** ошибками в mono pass
(`if condition must be bool` etc).

Шаги:
1. Reproducer внутри test (не через stdlib)
2. Найти где mono pass конкретизирует pattern-match path  
3. Понять что именно генерирует bad condition
4. Fix

После закрытия — реализовать Ф.3 spread.

### Ф.3 — Spread `[...defaults, k: v]`

После Ф.2 (clone/merge_from работают):

1. AST: `MapLitElem::Pair(Expr, Expr)` | `MapLitElem::Spread(Expr)`
   (новый enum, миграция из текущего Vec<(Expr,Expr)>)
2. Parser: распознать `...expr` в map-литерале
3. Type-check: validate что spread-expr типизирован совместимо
4. Desugar: `[...m, k: v]` → `{ let _t = m.@clone(); _t.@insert(k, v); _t }`

### Ф.4 — Type inference no annotation edge cases

**Описание:** `let m = ["a": 1]` без аннотации может фейлиться в
test-блоках (M-52-type-inference-no-annotation).

Investigation: что именно происходит, root cause через trace.
