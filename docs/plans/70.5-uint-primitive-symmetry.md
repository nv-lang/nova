# Plan 70.5: `uint` symmetric primitive type

> **Трудоёмкость:** ~1 dev-day (actual).
> **Родительский план:** [Plan 70](70-no-silent-nova-int-fallback.md)

---

## Зачем

Nova has `int` (alias `i64`) but no symmetric `uint`. Сейчас:
- Indexing `arr[i int]` — int может быть negative → runtime check needed
- Sizes `len() -> int` — declared return может быть negative semantically
- API asymmetric: `i64` имеет typed alias `int`, `u64` не имеет typed alias

Modern languages provide symmetric pair:

| Язык | signed | unsigned | Width |
|---|---|---|---|
| **Rust** | `isize` | `usize` | platform-pointer (64-bit on x64) |
| **Go** | `int` | `uint` | platform-pointer |
| **Swift** | `Int` | `UInt` | platform-pointer |
| **C** | `int`/`long`/etc | `unsigned`/`size_t` | platform-dependent |
| **Nova (текущее)** | `int` (= `i64`) | (нет) | fixed 64-bit |

## Open Questions (с recommended answers)

### Q1: `uint` alias of `u64` или distinct primitive?

**Recommended: ALIAS of `u64`** (mirror текущий `int` = `i64` pattern).

**Why:**
- Bootstrap Nova не имеет multi-arch story — `int` уже just-alias `i64`.
  Asymmetry было бы strange (signed alias, unsigned distinct).
- Modern parallels:
  - Rust `usize` distinct potentially (platform width) — Nova bootstrap
    pre-multi-arch, не релевантно
  - Go `uint` distinct (platform width) — same
  - Swift `UInt` distinct (platform width) — same
- Future evolution: когда Nova добавляет multi-arch support (Plan 58+),
  `int`/`uint` могут стать distinct platform-pointer types. Сейчас —
  bootstrap-grade alias.

**Codegen:** `"uint" => "uint64_t"` (mirror `"int" => "nova_int"`).
В фьюче можно add typedef `nova_uint = uint64_t` parallel `nova_int =
int64_t`.

---

### Q2: `int → uint` cast — explicit only? saturation? wrap?

**Recommended: explicit `as uint` cast с saturation** (negative → 0).

**Why:**
- Nova already has D54 `as`-cast saturation для `float → int`
  (Plan 07). Apply same pattern: out-of-range → boundary value.
- Modern parallels:
  - **Rust** `i64 as u64` — bit-cast (wraps; -1 → u64::MAX). User-hostile
    surprise. Often source of bugs.
  - **Swift** `UInt(value)` initializer — **traps** on negative. Safer
    but verbose. Has separate `UInt(bitPattern:)` for explicit wrap.
  - **Go** `uint(x)` где `x int` — bit-cast wraps. Same Rust pitfall.
  - **Recommended Nova:** saturation (`-5 as uint → 0`, matches
    `f64 as int` D54 invariant). Closer к Swift safety без runtime trap.

**Bit-cast variant:** `(x as i64).bit_cast[u64]()` или explicit method —
для FFI cases где wrap semantically correct. Don't make это default.

---

### Q3: Index types — `arr[i int]` или `arr[i uint]`?

**Recommended: keep `int`** для indexing API.

**Why:**
- **Rust** uses `usize` everywhere для indexing — compile-time
  guarantee non-negative. Но: ergonomics suffer (constant `as usize`
  casts при index math с `i32` results).
- **Swift / Go / Kotlin** use `Int` для indexing — runtime check для
  negative. Ergonomic, slight overhead.
- Nova precedent: `arr.len() -> int`, current Range/Iter `-> Option[int]`.
  Switching к uint = breaking change для 100+ APIs.

**Decision:** `arr[i int]` остаётся. Type-checker может add lint
warning «consider non-negative» где доступна constant-fold proof.

**Когда `uint` нужен:**
- Explicit non-negative contracts: `fn parse_age(s str) -> Option[uint]`
- Bitfield/flags-like APIs: `Flags { type Mask = uint }`
- FFI к C `size_t`/`uint*_t` parameters

---

### Q4: Default literal type `42` — int или uint?

**Recommended: `int` (keep current behavior)**.

**Why:**
- Backward compat: all existing Nova code assumes `42: int`.
- Modern parallels:
  - Rust: integer literal type is inferred (default `i32`); no `uint`
    default.
  - Go: untyped constant; defaults к `int` when used in expression.
  - Swift: literal default `Int`.
- Nova inference: literal `42` defaults `int`, promotes к context
  type. User opt-into uint via `42 as uint` или suffix `42u`.

**Suffix syntax (optional):** `42u` или `42u64` для uint literals,
mirror Rust `42u64` / Go `uint(42)`. Decision: add suffix syntax
если/когда uint use grows. Sometimes annotation enough (`let x uint = 42`).

## Сводка рекомендованного дизайна

| Aspect | Recommendation | Rationale |
|---|---|---|
| **Q1: alias или distinct?** | Alias of `u64` (= `uint64_t`) | Mirror `int` = `i64` alias pattern |
| **Q2: int→uint cast** | `as uint` с saturation (neg → 0) | D54 precedent, safer чем wrap |
| **Q3: Indexing type** | Keep `int` (no breaking change) | Swift/Go pattern, ergonomic |
| **Q4: Literal default** | `int` (keep current) | Backward compat |
| **Suffix syntax** | Defer (`42u` если usage grows) | YAGNI до first concrete need |

## Risks

| Risk | Mitigation |
|---|---|
| **API churn** — users add `uint` everywhere → migration noise | Don't force; uint opt-in только для new APIs |
| **Confusion `int` vs `i64` vs `uint` vs `u64`** | Spec D-block clarifies — `int` alias `i64`, `uint` alias `u64` (bootstrap) |
| **Saturation surprises** | Test fixtures + diagnostic explaining behavior |
| **Future multi-arch break** | Document evolution path — может stать platform-pointer когда Nova adds 32-bit arches |
