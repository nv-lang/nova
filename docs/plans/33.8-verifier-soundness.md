# Plan 33.8: Verifier soundness hardening (D24)

> Подплан семейства **Plan 33** (контракты). Создан по результатам
> аудита «с чистого листа» при закрытии Plan 33.7 (2026-05-21). Аудит
> нашёл **3 SOUNDNESS-CRITICAL дыры** — места, где верификатор объявляет
> контракт «доказан», хотя в рантайме он может быть ложным.

## Зачем

Контракт, помеченный «доказан», стирается из release-сборки (D24). Если
«доказан» ≠ «истинно в рантайме» — баг уходит в прод вообще без проверки.
Хуже того, компоновочные доказательства (вызывающий код доверяет `ensures`
вызываемого) распространяют ложь дальше. Пока эти дыры открыты, Plan 33
нельзя называть production-grade.

## Находки аудита (file:line)

1. **Переполнение `int`.** `int` в рантайме — 64-бит; `emit_c.rs:10949`
   эмитит голый C `+`/`-`/`*` (UB при переполнении, на практике wrap).
   Верификатор кодирует `int` как **безграничный** Z3 Int (`z3.rs:120`),
   без аксиом границ. Спека противоречит сама себе: `04-effects.md:1067`
   (overflow = `panic`) vs `04-effects.md:3204` (`a+b` checked →
   `RuntimeError.Overflow`). `#nooverflow` (Plan 33.7) на `int` — no-op.
2. **Сохранение инварианта цикла.** `pipeline.rs:2112`
   `extract_body_assignments` собирает только присваивания первого уровня,
   пропускает `*=`/`/=` и присваивания во вложенных `if`/блоках. Havoc-набор
   берётся из того же списка → инвариант «доказывается» на устаревшем
   значении, молча `Proven`.
3. **`assume`.** `verify_fn` не ассертит `Stmt::Assume` в backend (фича
   мёртвая — `assume x>0` не помогает доказательству); обещанный в
   AST-комментарии линт `trust-introduced` не реализован.

## Цель

Закрыть 3 дыры так, чтобы **«доказано» строго влекло «истинно в рантайме»**,
сохранив градуальную модель (доказал → стёр, не доказал → рантайм-проверка).

## Принципы

- **Fail-safe.** Если анализатор чего-то не умеет смоделировать —
  возвращать `Unknown` (рантайм-проверка остаётся), НИКОГДА не `Proven`.
- **Соответствие рантайму.** Семантика, в которой рассуждает SMT, обязана
  совпадать с тем, что делает codegen.
- Минимальный sound-фикс в первую очередь, оптимизации (стирание проверок) —
  отдельной фазой.
- Каждая фаза — позитивные И негативные тесты.

## Ф.1 — Переполнение `int` (P0)

**Ключевой инсайт.** Если codegen **паникует** при переполнении `int`, то
безграничное рассуждение Z3 становится sound: функция либо вернёт
истинный матем. результат (переполнения не было), либо умрёт паникой
(значение не вернётся). Постусловие выполняется для всех фактических
возвратов. То есть основной soundness-фикс — в **codegen**, а не в
верификаторе.

- **Ф.1.1 — Решение по семантике + спека.** Зафиксировать: переполнение
  `int` = `panic` (согласуется с `04-effects.md:1067`). Удалить
  противоречащую строку `04-effects.md:3204`. Уточнить/добавить D-блок:
  «int arithmetic overflow → panic; sized BitVec-типы → wrap (Plan 33.7)».
  ⚠️ Развилка `panic` vs catchable-`throw` — подтвердить с пользователем.
- **Ф.1.2 — Codegen: checked `int` арифметика.** `emit_c.rs`: `+`/`-`/`*`
  для `int` → `__builtin_add_overflow`/`_sub_`/`_mul_` → при overflow
  `nova_panic("integer overflow")`. **Это закрывает дыру #1 целиком**
  (sound baseline). Цена — проверка на каждой операции; убирается в Ф.1.3.
- **Ф.1.3 — Верификатор: overflow-VC для `int` (completeness + оптимизация).**
  Для каждой `int` `+`/`-`/`*` в `#verify`-функции генерировать VC
  «нет переполнения» (линейно: `INT_MIN ≤ a+b ≤ INT_MAX`). Доказано →
  panic-check можно стереть; не доказано → W-warning «возможна паника при
  переполнении». Переиспользует машинерию `collect_bv_arith_ops_in_body`
  из Plan 33.7, обобщённую на `int`.
- **Ф.1.4 — `nat`.** Аксиома `nat ≥ 0` для каждого `nat`-параметра/var
  (сейчас `nat` → безграничный знаковый Int без ограничения).
- **Ф.1.5 — Тесты.** Позитив (доказано no-overflow → стёрто), негатив
  (недоказуемое переполнение → warning / проверка сохранена).

## Ф.2 — Сохранение инварианта цикла (P0)

- **Ф.2.1 — Fail-safe (быстро, закрывает дыру немедленно).** Если тело
  цикла содержит присваивание, которое `extract_body_assignments` не
  моделирует (Mul/Div compound, присваивание во вложенном `if`/блоке) —
  `verify_loop_preservation` возвращает `Unknown`, не `Proven`.
- **Ф.2.2 — Полноценно.** Рекурсия сбора присваиваний в `if`/вложенные
  блоки/`match`; поддержка `*=`/`/=`; havoc по полному набору.
- **Ф.2.3 — Тесты.** Негатив: условное присваивание ломает инвариант →
  результат НЕ `Proven`. Позитив: корректный цикл с присваиванием в `if`.

## Ф.3 — `assume` escape hatch (P0/P1)

- **Ф.3.1 — Линт `trust-introduced`.** `assume` вне `#trusted`-функции →
  W-warning (как обещано AST-комментарием `ast/mod.rs`).
- **Ф.3.2 — Решение по фиче.** Либо подключить `assume` к backend как
  аксиому **с гейтом `#trusted`** (вне `#trusted` — warning + всё равно
  ассертим), либо честно оставить drop + задокументировать. Рекоменд.:
  подключить — полезный escape-hatch для FFI/внешнего знания.
- **Ф.3.3 — Тесты.** Негатив: `assume false` вне `#trusted` → warning, не
  должен молча всё доказывать.
