# Plan 33.8: Verifier soundness hardening (D24)

> Подплан семейства **Plan 33** (контракты). Создан по результатам
> аудита «с чистого листа» при закрытии Plan 33.7 (2026-05-21). Аудит
> нашёл **3 SOUNDNESS-CRITICAL дыры** — места, где верификатор объявляет
> контракт «доказан», хотя в рантайме он может быть ложным.

## Зачем

Контракт, помеченный «доказан», стирается из release-сборки (D24). Если
«доказан» ≠ «истинно в рантайме» — баг уходит в прод вообще без проверки.
Хуже того, компоновочные доказательства (вызывающий код доверяет `ensures`
вызываемого) распространяют ложь дальше. Пока эти дыры открыты, Plan 33
нельзя называть production-grade.

## Находки аудита (file:line)

1. **Переполнение `int`.** `int` в рантайме — 64-бит; `emit_c.rs:10949`
   эмитит голый C `+`/`-`/`*` (UB при переполнении, на практике wrap).
   Верификатор кодирует `int` как **безграничный** Z3 Int (`z3.rs:120`),
   без аксиом границ. Спека противоречит сама себе: `04-effects.md:1067`
   (overflow = `panic`) vs `04-effects.md:3204` (`a+b` checked →
   `RuntimeError.Overflow`). `#nooverflow` (Plan 33.7) на `int` — no-op.
2. **Сохранение инварианта цикла.** `pipeline.rs:2112`
   `extract_body_assignments` собирает только присваивания первого уровня,
   пропускает `*=`/`/=` и присваивания во вложенных `if`/блоках. Havoc-набор
   берётся из того же списка → инвариант «доказывается» на устаревшем
   значении, молча `Proven`.
3. **`assume`.** `verify_fn` не ассертит `Stmt::Assume` в backend (фича
   мёртвая — `assume x>0` не помогает доказательству); обещанный в
   AST-комментарии линт `trust-introduced` не реализован.

## Цель

Закрыть 3 дыры так, чтобы **«доказано» строго влекло «истинно в рантайме»**,
сохранив градуальную модель (доказал → стёр, не доказал → рантайм-проверка).

## Принципы

- **Fail-safe.** Если анализатор чего-то не умеет смоделировать —
  возвращать `Unknown` (рантайм-проверка остаётся), НИКОГДА не `Proven`.
- **Соответствие рантайму.** Семантика, в которой рассуждает SMT, обязана
  совпадать с тем, что делает codegen.
- Минимальный sound-фикс в первую очередь, оптимизации (стирание проверок) —
  отдельной фазой.
- Каждая фаза — позитивные И негативные тесты.

## Ф.1 — Переполнение `int` (P0)

**Ключевой инсайт.** Если codegen **паникует** при переполнении `int`, то
безграничное рассуждение Z3 становится sound: функция либо вернёт
истинный матем. результат (переполнения не было), либо умрёт паникой
(значение не вернётся). Постусловие выполняется для всех фактических
возвратов. То есть основной soundness-фикс — в **codegen**, а не в
верификаторе.

**СТАТУС Ф.1: 🟢 soundness закрыт (2026-05-21).** Ф.1.1+Ф.1.2+Ф.1.4+Ф.1.5
сделаны; Ф.1.3 — оптимизация → V2.

- **Ф.1.1 — Спека ✅.** Зафиксировано (решение пользователя): переполнение
  `int` = `panic`. Из `04-effects.md` убрана противоречащая строка
  `a + b (checked, overflow) → RuntimeError.Overflow`; добавлен явный
  абзац (int overflow = panic, sized-типы = wrap). Исправлен ложный
  комментарий `ir.rs` про «Int bounded».
- **Ф.1.2 — Codegen: checked `int` арифметика ✅.** `effects.h`:
  `nova_int_checked_add/sub/mul` — `__builtin_*_overflow` → `nv_panic`
  при переполнении. `emit_c.rs`: оба emit-сайта (`emit_expr` и
  `emit_expr_with_target_type`) для `nova_int` Add/Sub/Mul эмитят
  checked-helper. Sized-типы (wrap, Plan 33.7) не затронуты.
  **Это закрывает дыру #1**: паника делает безграничную Z3-кодировку
  `int` sound. Полный `nova_tests` suite: 931 PASS / 0 FAIL — нулевой
  fallout (никакой код не полагался на молчаливый wrap).
- **Ф.1.3 — overflow-VC для `int` (ОПТИМИЗАЦИЯ, отложено в V2).**
  Генерировать VC «нет переполнения» для каждой `int`-арифм. операции,
  чтобы (а) предупреждать «возможна паника» и (б) стирать panic-check
  где доказано. НЕ soundness-фикс — Ф.1.2 уже закрыл дыру (паника).
  Чистая оптимизация производительности + completeness. → V2.
- **Ф.1.4 — `nat ≥ 0` ✅.** `verify_fn` ассертит `nat >= 0` для каждого
  `nat`-параметра и его `_old_`. Helper `type_ref_is_nat`.
- **Ф.1.5 — Тесты ✅.** `int_overflow_add_panic.nv` +
  `int_overflow_mul_panic.nv` (EXPECT_RUNTIME_PANIC) +
  `int_arith_no_overflow_positive.nv`. Suite: 291 PASS / 0 FAIL.

## Ф.2 — Сохранение инварианта цикла (P0) — ✅ soundness закрыт (2026-05-21)

- **Ф.2.1 — Fail-safe ✅ (commit a3f… в plan33-4).** `loop_body_model_incomplete()`
  распознаёт тело вне sound-envelope havoc-модели (составное `*=`/`/=`,
  присваивание во вложенном if/блоке/цикле/match, повторное присваивание).
  В этих случаях `verify_loop_preservation` возвращает `Warning` (W2402),
  а не ложный `Proven`. Walker `expr_has_assignment`/`stmt_has_assignment`/
  `block_has_assignment` обходит всё поддерево тела.
- **Ф.2.3 — Тесты ✅.** `loop_cond_assign_w2402.nv` (условное присваивание),
  `loop_compound_assign_w2402.nv` (составное `*=`). Оба ловят регрессию —
  без fail-safe цикл получал ложный `Proven` и 0 warnings. Контрактный
  suite: 287 PASS / 0 FAIL.
- **Ф.2.2 — Полное моделирование (PRECISION, отложено в V2).** Чтобы циклы
  с условным/составным присваиванием можно было ДОКАЗАТЬ (а не честно
  warning'нуть), нужно моделировать `if c { x = e }` как `ite` и `*=`/`/=`
  как соответствующую операцию, с корректной последовательной композицией.
  Это расширение полноты, НЕ фикс soundness (Ф.2.1 уже закрыл дыру —
  верификатор честно говорит «не могу», а не лжёт). Делается отдельно
  при наличии реального спроса.

## Ф.3 — `assume` escape hatch (P0/P1) — ✅ закрыт (2026-05-21)

- **Ф.3.1 — Линт `trust-introduced` ✅.** `lints.rs::check_assume_trust` —
  `assume` вне `#trusted`-функции → lint-warning `trust-introduced`
  (walker `collect_assume_spans_*` обходит всё тело). Внутри `#trusted` —
  молча (граница доверия объявлена явно).
- **Ф.3.2 — Решение: НЕ подключать `assume` к SMT в V1.** Наивная
  проводка (ассертить все `assume` глобально) — **unsound**: текстуально
  поздний `assume` стал бы виден более ранней проверке (например
  preservation вложенного цикла), модель `verify_fn` не flow-sensitive.
  Корректная интеграция требует flow-sensitive верификации — крупная
  фича вне scope 33.8. Текущее поведение (`assume` не влияет на SMT)
  **soundness-safe**. Исправлен лживый AST-комментарий `Stmt::Assume` —
  честно: «V1 не интегрирован в SMT; flow-sensitive интеграция → V2».
- **Ф.3.3 — Тесты ✅.** `assume_trust_introduced_warn.nv`
  (EXPECT_COMPILE_WARNING) + 2 unit-теста `lints.rs`. Suite: 288 PASS.

> Попутно: исправлено pre-existing breakage — 3 `FnDecl`-литерала в
> `sum_schema_registry.rs` (test-код) не имели поля `no_overflow` из
> Plan 33.7 → `cargo test` не компилировался.
