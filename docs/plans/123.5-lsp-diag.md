## 1. Контекст

D217 §6 + D218 §8 + D219 §8 mention debug-info preservation. V5
exposes cache decisions to user developer tooling:

1. **CLI flag `--explain-cache`** (`nova compile`/`nova check`):
   emit human-readable report of cache decisions per method.
2. **LSP code-lens** над method header: "N caches inserted".
3. **Hover enhancement** на `@field`: показывает cache info если
   field cached.

## 2. Design

### 2.1 Analysis API (new)

`field_cache::analyze_module(&Module, &Config) -> ExplainReport`
runs cache analysis WITHOUT mutation, returns:

```rust
pub struct ExplainReport {
    pub per_fn: Vec<FnCacheInfo>,
}

pub struct FnCacheInfo {
    pub type_name: String,
    pub fn_name: String,
    pub span: Span,
    pub ro_caches: Vec<String>,    // field names
    pub mut_caches: Vec<String>,
    pub licm_hoists: Vec<String>,
    pub pure_caches: Vec<String>,
    pub chain_caches: Vec<Vec<String>>,
    pub total: usize,
}
```

### 2.2 CLI flag

`nova check --explain-cache <file>.nv` → text output:
```
fn ReadBuffer @try_read_u32_le() — 4 caches:
  ro field cache: data, pos
  pure call cache: len
  chain cache: header.signature
```

### 2.3 LSP integration

Per Plan 104.x infrastructure:
- `state.rs` extended с `explain_report: Option<ExplainReport>`.
- `compiler.rs` runs analyze_module on incremental rope.
- `server.rs` provides `textDocument/codeLens` handler emitting
  one lens per fn.

### 2.4 V5 scope

LSP code-lens — optional extension if time permits.
