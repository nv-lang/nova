# Plan 118.7 — `raw &x` syntax для сырого стек-адреса

> **Создан:** 2026-06-18. **Статус:** ✅ CLOSED. **Приоритет:** P1.
> **Закрыт:** 2026-06-18 (main). **Model:** Sonnet 4.6.
> **Родитель:** [Plan 118](118-typed-pointers-and-unsafe.md).
> **Владеет:** D216 §4 amend (второй раз — после Plan 118.6).

---

## Проблема

Plan 118.6 ввёл две семантики для одного синтаксиса `&x`:

| контекст | поведение |
|---|---|
| `&x` | safe: escape analysis + auto-promote в heap если нужно |
| `unsafe { &x }` | сырой стек-адрес, без промоута — может быть dangling |

Это нарушает принцип `unsafe {}` в Nova: блок **разрешает** опасную операцию,
но не **меняет её семантику**. Программист видит `&x` в обоих случаях и должен
помнить контекст чтобы понять что происходит с памятью.

---

## Решение

Новый унарный оператор `raw &x` — синтаксически отличимый от `&x`:

```nova
// safe: escape analysis + auto-promote при необходимости
ro p = &x

// raw: сырой стек-адрес, без промоута — требует unsafe {}
unsafe {
    ro p = raw &x   // raw = программист берёт ответственность
    c_fn(p)         // передать T* в C-функцию, FFI
}
```

`raw` — keyword-модификатор перед `&`. Читается как «сырой адрес x».

### Инвариант после плана

- `&x` — **всегда** safe + escape analysis + auto-promote. Работает везде,
  без `unsafe {}`.
- `raw &x` — **всегда** сырой стек-адрес, без промоута. Требует `unsafe {}`
  (иначе `E_UNSAFE_REQUIRED`). Тип — `*T` (или `*mut T` если биндинг `mut`).
- `unsafe { &x }` — теперь эквивалентен `&x` (unsafe-контекст не влияет на
  `&`, только на `*p` deref). Escape analysis работает как обычно.

## Фазы

### Ф.1 — Lexer + AST: новый токен и узел

- `lexer`: `raw` — **контекстное ключевое слово** (не зарезервировано в lexer,
  аналог `bench`/`measure`). Парсер распознаёт `Ident("raw") + Amp` контекстно
  в `parse_unary()`. `raw` как идентификатор в std/ и nova_tests/ — не ломается.
- `ast/mod.rs`: `UnOp::RawAddrOf` (рядом с `UnOp::AddrOf`).
- `parser/mod.rs`: в `parse_unary()` — перед обработкой `&` проверить
  `TokenKind::KwRaw` → bump → expect `&` → parse operand →
  `ExprKind::Unary { op: UnOp::RawAddrOf, operand }`.
  Применить те же lvalue-проверки что и для `AddrOf`
  (`E_AMP_LITERAL`, `E_ARRAY_INDEX_PTR_BANNED`, `E_ADDR_OF_NON_LVALUE`).

### Ф.2 — Type-checker: `raw &x` требует `unsafe {}`

- `types/mod.rs` `check_unsafe_context_in_module` walker:
  - `UnOp::RawAddrOf` при `depth == 0` → `E_UNSAFE_REQUIRED`
    (аналогично нынешнему `UnOp::Deref`).
  - `UnOp::AddrOf` в unsafe-блоке — убрать любую специальную логику
    «suppress promote». Теперь `unsafe { &x }` = обычный `&x`.
- `types/mod.rs` type-check для `UnOp::RawAddrOf`: тип = `*T` (ro биндинг)
  или `*mut T` (mut биндинг) — идентично `AddrOf`.

### Ф.3 — Escape analysis: `raw &x` не промоутит

- `escape_analyze.rs`:
  - `walk_expr` + `walk_expr_in_escape`: для `UnOp::RawAddrOf` — **не** вызывать
    `mark_promoted`. Просто рекурсить в operand.
  - `UnOp::AddrOf` в escape position — поведение не меняется (промоутит как раньше).

### Ф.4 — Codegen: `raw &x` → `&stack_var`

- `emit_c.rs`: новая ветка для `UnOp::RawAddrOf` — эмитировать `&(operand_c)`
  напрямую, без heap-allocation. Аналог нынешнего `UnOp::AddrOf` без учёта
  `promoted_value_record_locals` / `promoted_primitive_locals`.
- `UnOp::AddrOf` в codegen — без изменений (escape analysis уже решил
  нужен ли промоут).

### Ф.5 — Миграция: `unsafe { &x }` → `unsafe { raw &x }` где нужен сырой адрес

Sweep по `nova_tests/` и `std/`:
- Найти все `unsafe { ... &x ... }` где намерение — сырой адрес (FFI, raw pointer).
- Заменить `&x` → `raw &x` внутри unsafe-блока.
- Остальные `unsafe { &x }` где промоут нужен — убрать `unsafe {}` или оставить
  (теперь поведение `&x` внутри unsafe не отличается от внешнего).
- Обновить `.c` снимки затронутых тестов.

### Ф.6 — Spec + закрытие

- `spec/decisions/02-types.md` D216 §4: amend:
  - `&x` — safe, всегда с escape analysis + auto-promote
  - `raw &x` — unsafe-only, сырой стек-адрес, без промоута; требует `unsafe {}`
  - `unsafe { &x }` — эквивалентен `&x` (unsafe-контекст не влияет на `&`)
  - Обновить таблицу операторов

## Тесты

**POS:**
- `raw &x` внутри `unsafe {}` → сырой стек-адрес, C compile OK
- `raw &x` на mut биндинг → `*mut T`
- `&x` внутри `unsafe {}` → escape analysis работает как обычно (промоут если нужен)
- `&x` вне unsafe → промоут при escape (не регрессия)

**NEG:**
- `raw &x` вне `unsafe {}` → `E_UNSAFE_REQUIRED`
- `raw &literal` → `E_AMP_LITERAL` (те же проверки что у `&`)
- `*p` вне unsafe → `E_UNSAFE_REQUIRED` (не регрессия)
