## Цель в одной фразе

Ввести **declaration-driven schema registry** для sum-types, через который codegen смотрит variant info / value-type ABI / method routing **до** падения на hardcoded `sum_schemas` baseline — чтобы декларации в `std/prelude/core.nv` (Option, Result, Error) и `std/prelude/errors.nv` (RuntimeError) могли расширять и переопределять hardcoded compromise **без удаления** последнего, разблокировав 17 deferred Option/Result methods и RuntimeError file-based миграцию.

## Проблема (verified state, 2026-05-18)

### 1. Hardcoded `sum_schemas` pre-population — 4 sum-types

`compiler-codegen/src/codegen/emit_c.rs:958-1023` пре-наполняет `self.sum_schemas` ДО любого `emit_type_decl`:

- **Option** (lines 960-964): `{Some("nova_int"), None}` + alias `NovaOpt_nova_int → ditto`.
- **Result** (lines 966-969): `{Ok("nova_int"), Err("nova_str")}` — **monomorphization compromise** на конкретные `(int, str)`.
- **Error** (lines 977-985): record schema `{msg: nova_str}` + method_receiver `Error.new`.
- **RuntimeError** (lines 1004-1023): 6 variants — `DivByZero, Overflow, IndexOutOfBounds{index, length}, TypeMismatch(str), AssertFailed(str), NoHandler(str)`.

Эти insertions выполняются в `new()` / `init_sum_schemas` constructor codegen'а **до** того, как resolve_imports_inline переносит `std/prelude/core.nv` items в `module.items`. Поэтому даже если декларация будет добавлена в файл — codegen увидит её, но **существующий entry в `sum_schemas` уже занят** (Option/Result/Error/RuntimeError → hardcoded compromise wins).

### 3. Две параллельные infrastructure (key issue)

После анализа codegen + runtime headers выявлены **две независимые** механики работы с sum-types:

**Infra (a) — Value-type runtime helpers** (`compiler-codegen/nova_rt/array.h`):
- `NOVA_DECLARE_OPTION_T(T)` macro generates per-T `NovaOpt_<T>` value struct (lines 41-100).
- `Nova_Result` — single non-generic struct с hardcoded `Ok(nova_int) / Err(nova_str)` storage (lines 469-507).
- Runtime helpers `Nova_Option_method_is_some_<T>`, `Nova_Result_method_unwrap_or`, etc. (lines 207-527).
- `nova_make_Option_Some(int)`, `nova_make_Result_Ok(int)`, `nova_make_Result_Err(str)` — non-templated constructors.

**Infra (b) — Generic mono path** (`emit_c.rs:7286-...`):
- `drain_generic_type_worklist` (line 7286) lazily emits per-instance `Nova_<Name>____<T1>__<T2>` structs.
- `generic_type_templates: HashMap<String, TypeDecl>` (line 468) хранит шаблон.
- `generic_type_worklist: RefCell<Vec<(String, Vec<String>, String)>>` (line 472) — pending instances.
- Result of mono: full per-(T,E) `Nova_<sumName>____<T>__<E>` struct + per-variant `nova_make_Nova_<sumName>____..._<Variant>` constructors.

**Conflict pattern (this is the "62.A agent finding"):**

При попытке добавить `external fn Option[T] @unwrap_or(x T) -> T` в `std/prelude/core.nv`:

1. Type-checker регистрирует `Option[T] @unwrap_or` с signature `T → T` через standard method-table mechanism (`types/mod.rs:357-368`).
2. Codegen видит method-call `opt.unwrap_or(default)` где `opt_ty = NovaOpt_nova_str`.
3. **Hardcoded branch wins** (line 11583): `return Ok(format!("Nova_Option_method_unwrap_or_{}({}, {})", elem_ty, obj_c, v));` — это infra (a) trampoline, expects `NovaOpt_<T>` value-type + same-T `default`.
4. Но если type-checker считает `default` имеет тип `T` (erased) — он эмитится как `void*` через infra (b) erasure path.
5. **Type mismatch:** `Nova_Option_method_unwrap_or_nova_str(opt, (void*)default)` — C compiler reject'ит.
6. Symptom: «BinOp `==` on `Nova_Option_method_unwrap_or_*` return value fails because cannot compare void* vs nova_str».

Дополнительно: для `Result[T, E]` infra (b) НЕ срабатывает потому что `Result` есть в `generic_type_templates` **только если** declaration в user code, а в hardcoded baseline declaration отсутствует (только pre-populated schema). Поэтому `Result[int, str]` использует hardcoded infra (a); `Result[(K,V), Err]` (custom) НЕ имеет infrastructure (Plan 59 Phase 7.5 deferred).

**Key insight:** ни одна из infrastructure не «правильнее». (a) даёт zero-cost stack-allocated Option для primitive T; (b) даёт generic instantiation для arbitrary T. Registry должен **выбирать** правильную infra per call-site, основываясь на типе T и наличии declared method-body.

### 4. RuntimeError migration block (Plan 62.C)

После переноса `RuntimeError` в `std/prelude/errors.nv`:
- Type-checker увидит declaration → попытается зарегистрировать.
- Codegen `emit_type_decl` для `RuntimeError` (`emit_c.rs:4977`) попытается эмитить `typedef struct Nova_RuntimeError { ... }` + `nova_make_RuntimeError_DivByZero()` constructor.
- **Конфликт:** в `nova_rt/array.h` уже определены `Nova_RuntimeError` struct + `nova_make_RuntimeError_*` constructors (см. comment в `emit_c.rs:1000-1003`). C compiler ошибётся на redeclaration.
- Текущий fix: `BUILTIN_RUNTIME_TYPES` skip list (`emit_c.rs:1170-1176`) исключает `RuntimeError` из forward-decl emission. Но это **silent** — `emit_type_decl` всё ещё будет эмитить полную декларацию.
- Plan 62 line 248 ставит: «удалить pre-populated `sum_schemas["RuntimeError"]` (797-820) — заменить на registry lookup». Без 62.A.bis это требует ещё одного skip-flag в `emit_type_decl` + ручной transfer фиксированной schema, что fragile.

### 5. `find_variant` ambiguity (`emit_c.rs:16796-16813`)

Текущая логика:
```rust
fn find_variant(&self, variant_name: &str) -> Option<(String, Vec<String>)> {
    let mut result: Option<(String, Vec<String>)> = None;
    for (type_name, variants) in &self.sum_schemas {
        if let Some(fields) = variants.get(variant_name) {
            match result {
                None => result = Some((type_name.clone(), fields.clone())),
                Some((ref existing, _)) => {
                    if type_name.len() < existing.len()
                        || existing.starts_with("NovaOpt_")
                        || existing.starts_with("Nova_Result")
                    {
                        result = Some((type_name.clone(), fields.clone()));
                    }
                }
            }
        }
    }
    result
}
```

Lookup linear по всем sum_schemas, prefer shorter / non-mangled. **Issue:** если user declared `type Color | Red | Green | Blue`, и в каком-то prelude items потенциально есть `Some/None/Ok/Err`, lookup может дать wrong type for ambiguous name `Some`. Currently работает потому что `find_variant("Some")` уникально возвращает Option (нет user-types с variant Some). Но **lookup не учитывает declaration source** — нельзя prefer'нуть file-based prelude декларацию.

## Что НЕ делаем (rejected alternatives)

| Alternative | Почему отвергнут |
|---|---|
| **Полная замена `sum_schemas` на registry** | Breaking change для всех 12 hardcoded code points одновременно → high regression risk. Layered lookup (registry first, hardcoded fallback) — incremental migration. |
| **Удаление hardcoded baseline в 62.A.bis** | Требует Plan 59 sum-mono extension (Phase 7.5) для полного `Result[T,E]` mono. Plan 59 §«Out-of-scope» явно говорит «deferred до dedicated planning sessions». |
| **Registry хранит только variant info (без method routing)** | Не решает «17 deferred methods» проблему. Method routing — основная часть value-add. Без него Plan 62.A не unfreeze'нется. |
| **Lazy registry population** | Усложняет mental model + race conditions с `drain_generic_type_worklist`. Eager population при `init_prelude_decls` (после `resolve_imports_inline`, до `emit_type_decl`) — single deterministic phase. |
| **Generic schema через type-checker `Ty` (а не codegen)** | Type-checker уже имеет sum_decls (через `BoundCtx::sum_variant_names`); но codegen needs C-type info которой type-checker не хранит. Дублирование — корректное разделение слоёв. |
| **Registry в отдельном file (separate module)** | Premature. ~200 LOC scaffolding internal к `emit_c.rs`, прямо рядом с `sum_schemas`. Extract в отдельный module — refactor для будущего после 62.A.bis stable. |
| **Rewrite hardcoded methods (11567-11763) одним PR** | 12 method-routing branches × pattern emit logic = ~500 LOC change. Высокий regression risk. Заменять по одному в 62.A (после 62.A.bis ready). |
| **Полностью declarative (registry читает все decisions из decl)** | Bootstrap-pattern Nova: codegen знает C ABI (intptr_t casting, pointer/value distinction, NovaOpt_* mangling). Hide эти знания за declaration metadata — premature abstraction. Registry хранит C ABI metadata explicitly. |

## Решение — архитектурные правила

### Core abstraction: `SumSchemaEntry`

Новая структура в `emit_c.rs` рядом с `sum_schemas`:

```rust
/// Plan 62.A.bis: declaration-driven schema entry для sum-type.
/// Зарегистрирован при `init_prelude_decls` если `std/prelude/core.nv`
/// или `std/prelude/errors.nv` содержат declaration; иначе — при
/// `emit_type_decl` для user-defined sum'а. Pre-populated hardcoded
/// entries (Option/Result/RuntimeError) — `source = HardcodedBaseline`,
/// перебиваются при registration declared'а.
#[derive(Debug, Clone)]
struct SumSchemaEntry {
    /// Nova-level name: "Option", "Result", "RuntimeError", "MyEnum".
    nova_name: String,
    /// Variants: variant_name → ordered list of field C-types
    /// (positional Tuple variant) or single-element with mangled
    /// "<sum>::<variant>::<field>" key registered в record_variant_field_types
    /// (Record variant).
    variants: HashMap<String, Vec<String>>,
    /// Variant kind: Unit | Tuple | Record. Нужно чтобы pattern_cond
    /// знал accessor form (`payload.<V>._0` vs `payload.<V>.<field>`).
    variant_kinds: HashMap<String, SumVariantKind>,
    /// ABI: value-type (Option NovaOpt_<T> on stack) vs reference-type
    /// (Nova_<Name>* via Boehm GC). Определяет `.tag` vs `->tag` accessor.
    abi: SumAbi,
    /// C-type mangling template (Some Option) vs full mono (Result[T,E]).
    /// Determines how codegen resolves type_ref_to_c for `Option[X]`.
    mangling: SumMangling,
    /// Method routing: для каждого declared method — где взять trampoline.
    /// Hardcoded для existing infra (a) methods; DeclaredBody для
    /// user file-based declarations.
    methods: HashMap<String, MethodRouting>,
    /// Provenance: куда `find_variant` отсортирует при overlapping variants.
    source: SchemaSource,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
enum SumAbi {
    /// NovaOpt_<T> on stack, accessor ".tag" / ".value". Used by Option only.
    ValueOptionLike,
    /// Nova_<Name>* via nova_alloc, accessor "->tag" / "->payload.<V>._<i>".
    /// Used by Result, RuntimeError, user sum-types.
    HeapPointerLike,
}

#[derive(Debug, Clone)]
enum SumMangling {
    /// Single canonical C-type, parameterized via NovaOpt_<T> templating
    /// в register_novaopt_decl. Used by Option.
    NovaOptTemplate,
    /// Single canonical Nova_<Name> struct, hardcoded payload types.
    /// Used by Result (Ok=nova_int, Err=nova_str compromise) + Error.
    SingleHardcodedStruct { c_name: String },
    /// Full mono per-instantiation: Nova_<Name>____<T1>__<T2>.
    /// Drives drain_generic_type_worklist для variant constructors.
    /// Used by user-declared generic sums (after Plan 59 Phase 7.5: also Result).
    GenericMono,
    /// Non-generic sum (no type params): Nova_<Name> single struct, no mono.
    /// Used by RuntimeError, user non-generic sums.
    NonGeneric,
}

#[derive(Debug, Clone)]
enum MethodRouting {
    /// Existing trampoline в array.h: `Nova_<Name>_method_<m>` или
    /// `Nova_Option_method_<m>_<T_sani>`. c_name — фактический symbol.
    /// is_per_t: true если symbol mangled per-T (Option methods).
    HardcodedRuntimeFn { c_name: String, is_per_t: bool },
    /// Nova-side declaration с body — codegen emit'ит method body как
    /// regular method overload через emit_method_overload. Sig сохранён.
    DeclaredBody { sig: MethodSig, recv_is_mut: bool },
    /// External fn с trampoline в runtime (no Nova body, runtime impl).
    /// Aналог runtime_registry.rs entries.
    ExternalFn { c_name: String, sig: MethodSig },
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
enum SchemaSource {
    /// Built-in pre-populated (Option/Result/Error/RuntimeError). Lowest
    /// priority. Перебивается DeclaredFromPrelude / DeclaredFromUser.
    HardcodedBaseline,
    /// Из `std/prelude/core.nv` / `std/prelude/errors.nv`. Higher priority
    /// чем Hardcoded. Trumps user declarations с same name (W_PRELUDE_SHADOW).
    DeclaredFromPrelude,
    /// User-defined в текущей compilation unit. Lowest priority vs
    /// prelude (shadowing warning).
    DeclaredFromUser,
}
```

### State и lifecycle

**Storage** в `CEmitter`:
```rust
struct CEmitter {
    // ... existing fields ...

    /// Plan 62.A.bis: layered schema registry. Lookup precedence:
    /// DeclaredFromPrelude > DeclaredFromUser > HardcodedBaseline.
    /// При collision higher priority WIN, lower → preserved для fallback
    /// в `lookup_sum_schema_with_fallback`.
    sum_schema_registry: HashMap<String, Vec<SumSchemaEntry>>,

    /// Reverse index: variant_name → set of (sum_name, source) для
    /// find_variant speedup. Updated parallel к sum_schema_registry.
    variant_to_sum: HashMap<String, Vec<(String, SchemaSource)>>,
}
```

**Lifecycle stages:**

1. **`init_hardcoded_baseline()`** — called in `CEmitter::new()`. Mirrors current pre-population (Option/Result/Error/RuntimeError) **но** через `register_schema(SchemaSource::HardcodedBaseline)` API. Заменяет direct `sum_schemas.insert(...)` calls на line 958-1023.
2. **`init_prelude_decls(module: &Module)`** — called **после** `resolve_imports_inline`, **до** `emit_type_decl` loop (новый pass между existing pre-passes 1a-1a2 в `emit_module`). Scan'ит `module.items` for type-decls помеченных `#prelude` attribute или originating from path `std.prelude.*`; регистрирует через `register_schema(SchemaSource::DeclaredFromPrelude)`. **Это инвалидирует pre-populated entry**, но сохраняет его в Vec для fallback.
3. **`emit_type_decl(t)`** — для user-defined types вызывает `register_schema(SchemaSource::DeclaredFromUser)`. Existing collision detection остаётся (uniqueness check на user level).

**Lookup API:**

```rust
impl CEmitter {
    /// Primary lookup: most authoritative entry для given sum name.
    /// Используется в большинстве call-sites вместо `sum_schemas.get()`.
    /// Returns highest-priority entry (Prelude > User > Hardcoded).
    fn lookup_sum_schema(&self, name: &str) -> Option<&SumSchemaEntry> {
        let entries = self.sum_schema_registry.get(name)?;
        // Vec sorted DESCENDING by source priority при insertion.
        entries.first()
    }

    /// Lookup with explicit fallback: returns (primary, fallback_iter).
    /// Используется в migration steps где hardcoded baseline нужен как
    /// safety-net (e.g. Result method routing — if DeclaredFromPrelude entry
    /// существует но без method `unwrap_or`, fallback к HardcodedRuntimeFn).
    fn lookup_sum_schema_layered(&self, name: &str) -> impl Iterator<Item = &SumSchemaEntry> {
        self.sum_schema_registry.get(name)
            .map(|v| v.iter())
            .into_iter()
            .flatten()
    }

    /// Backward-compat shim для existing `sum_schemas.get(name)` calls.
    /// Returns variant→fields map в legacy format. Internal-only — все
    /// new code должен использовать lookup_sum_schema.
    fn legacy_sum_schema_view(&self, name: &str) -> Option<HashMap<String, Vec<String>>> {
        self.lookup_sum_schema(name)
            .map(|e| e.variants.clone())
    }

    /// find_variant replacement. Возвращает (sum_name, fields, source).
    /// Sort: Prelude entries first → user → hardcoded. Внутри одного source
    /// — shorter name preferred (existing behavior).
    fn find_variant_v2(&self, variant_name: &str) -> Option<(String, Vec<String>, SchemaSource)> {
        let candidates = self.variant_to_sum.get(variant_name)?;
        // Sorted by SchemaSource at registration time.
        candidates.first().and_then(|(sum_name, src)| {
            self.sum_schema_registry.get(sum_name)?
                .iter()
                .find(|e| e.source == *src)
                .and_then(|e| e.variants.get(variant_name).cloned()
                    .map(|fields| (sum_name.clone(), fields, *src)))
        })
    }
}
```

### Layered lookup precedence — concrete dispatch points

Каждая existing hardcoded точка трансформируется так:

```rust
// BEFORE (current):
if name == "Option" || name == "NovaOpt_nova_int" {
    // typed compound literal logic ...
}

// AFTER (layered):
let abi = self.lookup_sum_schema("Option").map(|e| e.abi);
if abi == Some(SumAbi::ValueOptionLike) {
    // typed compound literal logic (same as before) ...
}
```

**Critical:** sequence preservation. Behavior **identical** baseline-wise (всё ещё `Option` → `ValueOptionLike` → same compound literal). Изменение: если в `std/prelude/core.nv` появится **shadowing** declaration с разным ABI — registry это вернёт и codegen разветвится на правильный path.

### Решение конфликта двух infrastructure

**Decision matrix** для `unwrap_or` example:

| Sum type | T concrete? | Has DeclaredBody method? | Path |

fn emit_sum_method_call(&mut self, obj_ty: &str, method: &str, args: &[CallArg]) -> Option<Result<String, String>> {
    let nova_name = self.c_ty_to_sum_name(obj_ty)?;
    let entry = self.lookup_sum_schema(&nova_name)?;
    let routing = entry.methods.get(method)?;
    Some(match routing {
        MethodRouting::HardcodedRuntimeFn { c_name, is_per_t } => {
            self.emit_hardcoded_trampoline(c_name, *is_per_t, obj_ty, args)
        }
        MethodRouting::DeclaredBody { sig, recv_is_mut } => {
            self.emit_declared_method_via_overload(&nova_name, method, sig, *recv_is_mut, obj_ty, args)
        }
        MethodRouting::ExternalFn { c_name, sig } => {
            self.emit_external_method_trampoline(c_name, sig, obj_ty, args)
        }
    })
}

### Edge cases & invariants

1. **Hardcoded baseline всегда зарегистрирован первым** в `Vec<SumSchemaEntry>` order. После `init_prelude_decls` он остаётся в Vec на indexed `[1..]` (или дальше); index `[0]` — highest priority entry.
2. **Shadowing warning:** если `register_schema(DeclaredFromUser, "Option")` срабатывает, и уже есть `DeclaredFromPrelude` entry — emit `W_PRELUDE_SHADOW` diagnostic (link to Plan 62 §«User shadowing»).
3. **Variant rename detection:** если declared Option имеет variants `{Some, Maybe}` (user changed `None` → `Maybe`) — registry хранит declared variants, hardcoded baseline в fallback Vec. Pattern-match `match opt { Some(v) => ... None => ... }` — `None` ресолвится через variant_to_sum (HardcodedBaseline entry), но codegen эмитит `NOVA_TAG_Option_None` который **не существует** в emitted Nova type. **Decision:** Phase 4 acceptance test that user shadow MUST match variant set; otherwise compile error «sum declared in prelude has different variant set».
4. **Mono'd Result interaction:** existing `Nova_Result____<T>__<E>` keys в `sum_schemas` (registered by `drain_generic_type_worklist`, line 5346, 7504, 4973) — продолжают быть registered as `SchemaSource::DeclaredFromUser` (since they come from generic_type_templates user-or-implicitly-provided). Lookup precedence: mono'd entry per concrete name (`Nova_Result____nova_int__nova_str`) finds itself; bare `Result` name finds DeclaredFromPrelude (после Phase 2) или HardcodedBaseline.
5. **Effect type / non-sum type collision:** `Fail` effect и `Fail` sum — currently separated by `effect_schemas` vs `sum_schemas`. Registry **не трогает** effect_schemas (out of 62.A.bis scope; Plan 62.F).

## Open questions с recommendations

1. **`origin_module: Option<Vec<String>>` storage — `Item` or `TypeDecl`?**
   - Discussion: `Item` is the variant wrapper, `TypeDecl` is the actual struct. Putting it на `Item` requires match every place items are constructed; on `TypeDecl` — only TypeDecl constructors. But other Item variants (Fn methods on prelude types) also need origin tracking (Phase 4).
   - **Recommendation:** put `origin_module` on **both** `TypeDecl` AND `FnDecl` (the two Item kinds 62.A.bis cares about). `Item::origin_module()` extension method dispatches. Other Item kinds — `None` (acceptable; out of scope).

2. **Discovery trigger — module path `std.prelude.*` или `#prelude` attribute?**
   - Discussion: path-based discovery couples to filesystem layout; attribute-based decouples but requires spec amendment + parser change.
   - **Recommendation:** path-based for 62.A.bis (matches Plan 62 splittable structure decision, line 122-129). Attribute-based as future enhancement если/когда prelude items могут жить вне `std.prelude.*` (currently not planned).

3. **`SumAbi::ValueOptionLike` — heuristic by name == "Option" или declarative marker?**
   - Discussion: heuristic is brittle (что если user declares non-Option ValueOption-like sum?); declarative requires syntax (`#abi(value)`).
   - **Recommendation:** heuristic for 62.A.bis (matches bootstrap pragmatism; only Option uses this ABI currently — see array.h `NOVA_DECLARE_OPTION_T`). Add `#abi(value)` attribute syntax in Plan 62.D if/when StringBuilder/WriteBuffer need similar opt-in.

4. **Legacy `sum_schemas: HashMap<..., ...>` field — remove или keep as cache?**
   - Discussion: removing requires deeper migration (many call sites use `self.sum_schemas.contains_key(x)` / `.get(x)`). Keeping as cache adds invalidation burden.
   - **Recommendation:** keep as **read-only auto-derived cache** populated в end of `init_prelude_decls`. Migration to direct `lookup_sum_schema()` happens incrementally over multiple plans. Phase 2 acceptance allows shim usage.

5. **Method routing for `Option @unwrap` (Fail-effected) — how to wire через registry?**
   - Discussion: `unwrap` requires Fail handler dispatch (Plan 61 typed-error infrastructure). Registry's `MethodRouting::DeclaredBody` would need to know about effects.
   - **Recommendation:** out of scope for 62.A.bis. `unwrap` stays hardcoded в emit_c.rs:11587-11597 (inline Fail dispatch) — `MethodRouting::HardcodedRuntimeFn { c_name: "<inline>", is_per_t: false }` placeholder. Phase 4 routing for Fail-effected methods documented as Plan 62.A follow-up.

6. **Bootstrap order: `init_prelude_decls` BEFORE or AFTER `emit_type_decl` loop?**
   - Discussion: BEFORE — declarations available when type-decls emit (codegen sees prelude Option exists). AFTER — type-decls emit first, registry registration latent.
   - **Recommendation:** BEFORE. Place hook между 1a2 (line 1212) и 1 (line 1227) в `emit_module`. Rationale: emit_type_decl for **prelude items themselves** must register с DeclaredFromPrelude source (not DeclaredFromUser default). Hook before loop ensures source classification consistent.

7. **`variant_to_sum` ordering для tie-breaking — sort at insertion или at lookup?**
   - Discussion: insertion order makes lookup O(1) per variant; sort at lookup is O(log N) but simpler invalidation.
   - **Recommendation:** sort at insertion. `Vec<(String, SchemaSource)>` maintained in DESCENDING priority order. `find_variant_v2` reads `.first()` — O(1). Re-sort happens только при `register_schema()` (uncommon).

8. **Variant set check — strict equality или subset allowed?**
   - Discussion: subset (prelude declares Option | Some | None | Pending) allows extension but breaks pattern exhaustiveness for hardcoded match statements.
   - **Recommendation:** strict equality в 62.A.bis. Extension requires edition bump (Plan 62.F). Diagnostic guides пользователя.

9. **Should registry track `RecordSchemaEntry` symmetrically?**
   - Discussion: same problem exists for `Error` (record schema hardcoded), `ChannelPair`, etc. Registry для records would be symmetric.
   - **Recommendation:** out of scope for 62.A.bis. Sum-type complexity is higher (12 dispatch points vs ~3 для records); records can wait для Plan 62.A.ter если/когда needed. `Error` migration in 62.A handled via existing `record_schemas` HashMap.

10. **`MethodRouting::DeclaredBody` для declared method with non-trivial Nova body — does codegen emit body or skip?**
    - Discussion: `external fn Option[T] @unwrap_or(x T) -> T` — body NOT defined в Nova (external). But user could write `fn Option[T] @custom_helper() -> bool => @is_some()` with Nova body.
    - **Recommendation:** for 62.A.bis, support **only `external fn`** declarations on prelude types — body-less, dispatch to existing C trampolines. Nova-body methods routed как `MethodRouting::DeclaredBody { has_nova_body: true }` — Phase 4 emits через `emit_method_overload`. **But** for Option/Result в particular, all 17 D26 methods are external (implementations in `array.h`); no Nova-body case until user adds custom methods (which works via existing infrastructure orthogonally).
