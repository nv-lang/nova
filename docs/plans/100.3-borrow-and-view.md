# Plan 100.3: implicit view default + closure capture + `match consume`

> **Создан 2026-05-23 как sub-plan [Plan 100 umbrella](100-linear-must-consume.md).**
> **Статус:** 📋 proposed (Ред. 2 2026-05-24: drop `view T` keyword,
> default-view везде). **GATED на [100.1](100.1-core-must-consume.md)**.
> **Оценка:** ~3 dev-day (упростилось после Ред. 2).
> **D-блок:** **D157**.

---

## Цель

Финализировать модель borrow/view для consume-типов:

1. **Implicit view default** (D133 finalized): param / for / match /
   if-let / let-alias БЕЗ `consume` keyword'а = view-mode (read-only
   borrow). С `mut` — mut-view. С `consume` — transfer.
2. **`view T` keyword отвергнут** (Ред. 2) — default-view не нуждается
   в keyword'е.
3. **Closure capture analysis** — automatic mode-detection (view /
   mut-view / consume) по operations в body.
4. **`match consume @expr`** explicit-consume pattern matching
   (D133/D156 sibling).

Закрывает 2 hole'а D133 bootstrap'а:

1. **Deep peek в `Option[ConsumeType]`** — `match @file` (view-default)
   peek'ает без consume; `match consume @file` — explicit ownership.
2. **Closure capture** consume-var — D157 вводит automatic mode-
   detection: consume-closure (FnOnce) vs mut-view-closure (FnMut)
   vs view-closure (Fn).

Дизайн — **scope-only borrow** (без `view` keyword): время жизни
borrow'а ограничено лексическим scope'ом. Без lifetime-параметров, без
runtime cost, sound через статический запрет на escape.

## Сравнение

| Capability | Rust | TS | Kotlin | Nova target |
|---|---|---|---|---|
| Read-only borrow | ✅ `&T` | ❌ | ❌ | ✅ **`view T`** |
| Mutable borrow | ✅ `&mut T` | n/a | n/a | ❌ **не вводим** (mut методы D131 покрывают) |
| Borrow в pattern (`match &opt`) | ✅ | n/a | n/a | ✅ **`match @file`** |
| Borrow в closure | ✅ | n/a | n/a | ✅ **`view`-capture** |
| Lifetime annotations | ❌ требуются | n/a | n/a | ✅ **не требуются** (scope-only) |
| Borrow checker cognitive cost | ❌ высокий | n/a | n/a | ✅ **низкий** (scope-only, без lifetime'ов) |
| Compile-time soundness | ✅ | n/a | n/a | ✅ |

Nova **превосходит Rust** на одной оси — отсутствие lifetime
annotations. Цена — borrow не может пересечь scope-boundary (return,
store в struct). Для resource-management use-cases (peek в Option,
read-helper, closure-callback) этого достаточно.

## Дизайн-решения

### D1 — keyword `view T`

```nova
fn print_file_info(f File) {              // ← view T — borrow читать-только
    println(f.fd)                               // ✅ читать поля
    f.close()                                   // ❌ consume через view — error
    f.write("...")                              // ❌ mut method через view — error (см. D2)
}
```

Альтернативы — отвергнуты:
- `&T` (Rust-style) — путает с обычным «pointer» из C, конфликт с
  D6 «нет указателей в user-facing» Nova.
- `ref T` (Kotlin/C++) — `ref` не существует как keyword в Nova,
  inconsistent с другими qualifier-keyword'ами (`mut`/`consume`).
- `borrow T` — лишний syllable, длиннее `view`.

**`view`** — новое keyword, чёткая семантика «read-only снимок объекта,
не trogаю».

### D2 — что разрешено через `view`

| Действие | `view T` | regular `T` (read-only-no-consume per 100.1 D7) |
|---|---|---|
| `t.field` (read) | ✅ | ✅ |
| `t.regular_method()` (без mut/consume) | ✅ | ✅ |
| `t.mut_method()` | ❌ error E (D157-mut-via-view) | ❌ (D7) |
| `t.consume_method()` | ❌ error E (D157-consume-via-view) | ❌ (D7) |
| передача в `view`-param другой fn | ✅ | ✅ |
| передача в `consume`-param | ❌ error | ❌ (D7) |
| store в поле | ❌ error E (D157-view-escape-store) | ❌ (D7) |
| return | ❌ error E (D157-view-escape-return) | ❌ (D7) |
| capture в closure, который returns | ❌ error E (D157-view-escape-closure) | ❌ (D7) |
| capture в closure, который invoked в-scope | ✅ | ✅ |

**view = строже чем regular read-only** (D7 100.1): запрещает даже
mut-методы (которые D7 разрешает на read-only-param). Логика: view —
borrow на source-объект, source может быть consume-typed (Live), mut
изменил бы invariant'ы консьюм-обязательства.

### D3 — `view` propagation через scope

`view`-binding живёт в scope'е binding-сайта (function/closure/arm).
Source `Live` пока scope view'а активен; на scope exit — source
снова свободен (просто scope-mechanic, без runtime).

```nova
fn outer() {
    consume tx = begin()
    print_info(tx)                          // ✅ view-передача
    tx.commit()                                  // ✅ tx Live после
}                                                // scope-exit OK

fn outer_bad() {
    consume tx = begin()
    ro saved = view tx                          // ❌ привязать к локальной — escape
    tx.commit()                                  
}                                                // (если bind to local разрешим — saved может
                                                 //  outlive tx; нужно scope-tracking)
```

Чтобы избежать сложности lifetime — **`view` нельзя bind к локальной
переменной**. View живёт только в выражении / pattern-arm / closure-
body. Это accepts мелкое ограничение ради zero-lifetime overhead.

**Альтернатива:** разрешить `let v = view tx` если scope `v` ⊆ scope
`tx` (статический check). Bootstrap consider — может быть, см. Ф.0
GATE.

### D4 — `match view` для deep peek

```nova
type Service consume { consume file Option[File] }

fn Service @file_id() -> Option[int] {
    match @file {                           // ← view-match
        Some(f) => Some(f.fd),                   // f: view File, read-only
        None => None,
    }
}
```

`match <expr>` — destructive по форме, но binding'и в arm'ах —
**view, не linear**. После match — `@file` остаётся Live (не Consumed).

Это **закрывает deep-peek hole** bootstrap'а.

### D5 — `view`-capture в closure

```nova
fn outer() {
    consume tx = begin()
    ro f = || println(tx.id)                    // closure capture; должно быть view
    f()                                           // OK, tx остаётся Live
    f()                                           // OK, permits multiple invokes
    tx.commit()                                   // ✅ tx Live после closure-scope
}
```

Closure-body анализируется как функция:
- consume операции над captured tx → требуют captured как `consume`
  (closure становится FnOnce-equivalent — invoke только раз).
- view операции (read-only) → captured как `view` (FnMut/Fn analog —
  multiple invokes).

**Closure escape check** (D157-view-escape-closure): closure capturing
view не может быть returned / stored — выйдет за scope source'а.

### D6 — consume-closure («FnOnce»)

Параллельно — если closure consume'ит captured var, она помечается
**consume-closure** (analog Rust `FnOnce`). Можно вызвать **ровно
один раз**, после вызова — closure сама Consumed.

```nova
fn outer() {
    consume tx = begin()
    ro commit_it = || tx.commit()              // consume-closure (FnOnce)
    commit_it()                                  // ✅ tx Consumed, closure Consumed
    commit_it()                                  // ❌ use-after-consume на closure
    tx.commit()                                  // ❌ tx уже Consumed
}
```

Закрывает 4-й hole bootstrap'а — «closure без consume в теле — silent
forget». Теперь closure явно `consume`-typed; если не вызван — он
Live, на scope exit обязан consumed (выполнен) или передан / consume'd
обычным механизмом.

### D7 — `mut`-borrow НЕ вводим

Соблазн добавить `mut view T` (Rust `&mut T`). Отвергнуто:
- `mut`-методы D131 + field-aware flow 100.1 D5 покрывают большинство
  use-case'ов через `consume` или regular `mut @method` на receiver'е.
- `mut view` усложнил бы aliasing rules (exclusive vs shared) — это
  именно borrow-checker, который мы избегаем.
- Если кому-то нужна mutable-borrow семантика — она выражается через
  обычный mut method на consume-record'е (D5).

### D8 — runtime cost = 0

`view` — type-level only marker. Codegen эмитит обычный pointer/value
(без extra tag). Все проверки compile-time через `check_consume`.

### D9 — consume-rvalue в arg-position (без binding)

Прямой `f(make_tx())` где `make_tx() -> Tx consume` — fresh
consume-owner передаётся как arg **без сохранения через `consume name = …`**.
Правила по qualifier'у callee-param:

| Callee param | OK? | Почему |
|---|---|---|
| `f(consume t Tx)` — consume-param | ✅ | ownership передаётся напрямую; callee обязан consume внутри |
| `f(t Tx)` — view-param (default) | ❌ E (D133-consume-rvalue-in-view) | callee не consume'нет; rvalue → ресурс утечёт молча |
| `f(mut t Tx)` — mut-view-param | ❌ E (D133-consume-rvalue-in-mut-view) | то же — mut-view не consume'нет |

**Логика:** view/mut-view не consume'нят. После возврата rvalue
остаётся не consumed и не bound → flow-checker не имеет slot'а в
`ConsumeCtx` → must-consume gate его не увидит → leak. Запрет —
единственное безопасное правило.

**Diagnostic hint:** «привяжи через `consume name = make_tx()`,
затем `f(name)`».

**Цепочки** `g(f(make_tx()))` — рекурсивно по правилу для соответствующего
param'а каждого call'а.

Cross-ref: spec D133 §«Consume-rvalue в arg-position».

## Фазы

### Ф.0 — GATE: design probe (~0.5 dev-day)

D1–D8 locked. Decision points для probe:
- **let-bind view к локальной**: разрешить с scope-check (b) или
  только в expression-position (a)? **Рекомендация: (a)** —
  bootstrap consistency, без сложного scope-graphic'а.
- **view в pattern**: `Some(f)` (explicit) vs `match @file
  { Some(f) }` (implicit, всё arm-binding'и — view)? **Рекомендация:
  match-level (implicit)** — single point of control.

### Ф.2 — Lexer + parser (~0.5 dev-day)

- Lexer: новый `KwView`.
- AST: `TypeRef::View(Box<TypeRef>)` (но это **не отдельный тип** —
  это qualifier на usage); `Param { view: bool, consume: bool, ... }`;
  `match <expr>` → `Expr::MatchView { ... }`; closure capture
  qualifier через type-inference (не explicit на capture-site в
  bootstrap).
- Parser: `view T` в param-type position; `match <expr> { ... }`
  expression; `view <expr>` в call-site (`f(tx)`).
- Parse error: `view` + `consume` на одном binding/param.
- Parse error: `view` + `mut` на одном binding.

**Acceptance Ф.2:** 4 parser-фикстуры в `nova_tests/plan100.3/parser/`.

### Ф.3 — Type-checker view propagation (~1 dev-day)

- `view`-typed param — internal mode "view" в `ConsumeCtx`.
- Operations через view: read-fields ✅; mut-method ❌; consume-method
  ❌; передача в view-param ✅; передача в consume-param ❌; store ❌;
  return ❌.
- `view <expr>` в call-site: source остаётся Live в caller-scope (не
  меняет state на `Consumed`).
- Escape detection: closure capturing view не может escape scope (Ф.4).
- E (D157-mut-via-view), E (D157-consume-via-view), E (D157-view-escape-*).

### Ф.4 — `match view` + closure analysis (~1 dev-day)

**Match view** (D4):
- `match <expr>` — все binding'и в arm'ах — view-mode.
- Source `<expr>` остаётся Live после match.
- Patterns: `Some(f)` где f — view; `None` без binding'ов.

**Closure analysis** (D5/D6):
- Анализ closure-body на consume / view operations над captured vars.
- Если консьюмит captured X — closure становится consume-closure
  (FnOnce-like).
- Если только view-операции — closure становится view-closure
  (multi-invoke OK).
- Consume-closure: invoke = consume; повторный invoke → error;
  closure-binding Live → обязан invoked / passed dalее.
- View-closure: invoke = OK; closure может escape если все captured
  view-source'ы outlive closure (scope check).

## Acceptance criteria

- [ ] `view T` parses в param-position; в comb с consume/mut — parse
      error.
- [ ] `view`-param разрешает чтение полей + non-consume не-mut methods
      + передача в view-param.
- [ ] `view`-param запрещает consume-method, mut-method, store, return,
      escape-through-closure.
- [ ] `match <expr> { ... }` peek'ает без consume — source
      Live после match.
- [ ] View-closure поддерживает multi-invoke; не может escape source-
      scope.
- [ ] Consume-closure (FnOnce) — single-invoke; повторный invoke →
      error; forget closure → error.
- [ ] Deep peek `Option[ConsumeType]` работает через `match view`.
- [ ] D9: rvalue → consume-param ✅; rvalue → view/mut-view-param ❌

- [ ] 8 pos + 6 neg фикстур PASS.
- [ ] Полный `nova test` → 0 регрессий.
- [ ] Spec D157 published; cross-ref D131/D133/D156.
- [ ] Bootstrap hole'ы `[M-deep-peek-impossible]` и

## Границы

- **Mutable borrow (`mut view T`)** — не вводится; mut методы D131
  + field-aware D5 покрывают. Future plan если возникнет давление.
- **`view` bind к локальной переменной** (`let v = view tx`) —
  отложено в Ф.0 GATE (рекомендация: только expression/arm-position).
- **`view` через FFI** — extern не может принимать `view` (compile
  error). Future plan если будет реальный FFI use-case.

## Risks

1. **Closure escape detection** — сложная часть. **Митигация:** запрет
   return / store closure-value, если closure capturing view. Это
   жёстко но sound.
2. **Lifetime creep.** Соблазн добавить explicit lifetime'ы для гибкости.
   **Митигация:** жёсткая позиция «view только в scope, без bind к local
   var» в bootstrap; future plans могут расширить с lifetime'ами если
   реально нужно.
3. **`view`-param и mut-методы**: rule «view запрещает mut» может быть
   surprising. **Митигация:** диагностика подробная — «`view` denies
   mutation; use regular param without `view` for shared-write».
