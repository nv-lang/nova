## 1. Контекст

Plan 123.1 (D217) implemented basic CSE: ro field — full-body
cache; mut field — first-region cache до first write/call boundary.

**Gap:** loop body. Внутри `for i in 0..N { ... @data[i] ... }`,
`@data` accessed N раз — но Plan 123.1 pass не hoist'ит load из
loop, потому что Loop body — отдельный Block scope, и cache let
emit'ится в method-body prefix (вне loop), что bag'ит nothing если
loop body не имеет local prefix.

Wait — actually Plan 123.1 pass walks recursively into loops; ro
field reads inside loops are counted в total reads of fn. Cache
let emit'ится в fn-body prefix; reads inside loop replaced с
cache ident. Loop iteration не invalidates ro cache (ro fields
frozen).

**Real gap:** для **mut fields**, Plan 123.1 doesn't cache при
наличии loops because loop body contains potential mutations
(though typically loop body iterates over collection, не mutate
@field). И когда loop body нечитает mut @field, но loop iteration
не mutate field — V1 pass не caches because of conservative
mut-call-invalidation.

**Real LICM contribution:** explicit detection of loop-invariant
field reads inside loops (both ro and mut where loop body doesn't
mutate). Emit hoisted cache **directly before loop**, scoped к
loop's containing block.

Example real-world impact (array iteration):

```nova
// Before LICM (Plan 123.1 emits ro cache for entire method):
fn Image @sum_pixels() -> int {
    ro _at_pixels = @pixels   // hoisted by Plan 123.1
    mut total = 0
    for i in 0..@pixels.len() {       // ← loop bound читает @pixels
        total += _at_pixels[i] as int  // ← здесь уже cached
    }
    total
}
```

`@pixels` already cached, тут Plan 123.1 достаточно. LICM добавляет
value когда:
1. **Method has mutation** before loop → Plan 123.1 не caches mut
   field, но loop body never mutates it → LICM hoists immediately
   before loop.
2. **Pure compute methods на `@field.<method>()`** — `@buf.len()`
   called every iteration; LICM (V2) hoists call result.
3. **Chain access** `@a.b.c[i]` — Plan 123.4 chain cache + LICM
   compose.

V2 focuses на (1): mut field LICM when loop body doesn't mutate.
(2) and (3) are Plan 123.3 / 123.4 territory.

## 2. Design

### 2.1 LICM scope (DECISION-A.2)

**V2 scope:**
- `for pat in iter { body }` (regular + parallel).
- `while cond { body }`.
- `loop { body }`.
- `while let pat = expr { body }`.

**Targeted reads:**
- `Member { obj: SelfAccess, name: F }` reads inside loop body.

**Hoisting eligibility:**
- Field `F` must NOT be modified anywhere in loop body (no `@F = ...`
  assignment, no compound assign).
- Field `F` must NOT be referenced by any closure inside loop body
  (conservative aliasing).
- Loop body must NOT contain `Spawn` / `Supervised` / `Detach` —
  concurrent access invalidates (V2 conservative; V3 may refine).
- Loop body **may** contain `Call` expressions but only ro fields
  hoisted (call could mutate mut field via alias).
- For mut fields: loop body must NOT contain any `Call` (full V2
  conservative).

### 2.2 Hoisting placement (DECISION-B.2)

Hoisted `ro _at_<F>_loop = @<F>` inserted **immediately before the
loop** in the enclosing block. Cache local name = `_at_<F>_loop`
(distinct from Plan 123.1 `_at_<F>` to avoid confusion — debug-info
shows different cache origin). Collision-avoidance: suffix `_N`.

If loop is in trailing-expr position (not in Block.stmts), wrap loop
in a Block to insert hoist. Edge case.

### 2.3 Composition with Plan 123.1

LICM runs **before** Plan 123.1 (in pass-pipeline sense — actually
they run together in `cache_module`, LICM phase first). Rationale:
- LICM creates new bindings inside method body.
- Plan 123.1 sees those as locals (not @field anymore) — won't
  duplicate cache.
- If Plan 123.1 ran first, it would cache `@F` at method-body
  prefix; LICM then has nothing to hoist (already cached).

**Composition result:** for `ro @F` accessed inside a loop:
- Plan 123.1 caches at method-body prefix.
- LICM no longer hoists (already cached above).
- Net: one cache let at method-body prefix, replacing all reads
  including loop body.

For `mut @F` accessed only inside loop:
- Plan 123.1 mut path: if loop is preceded by Call or write → no
  prefix cache. If method-body prefix region has no @F reads → no
  cache.
- LICM step: detects @F invariant в loop → hoists immediately
  before loop.
- Net: hoisted cache before loop.

### 2.4 Threshold + max-per-fn (DECISION-C.2)

- LICM threshold: ≥2 reads inside the loop. (Single read inside loop
  не worth hoisting.)
- Max hoists per loop: 4 (separate from `max_per_fn` cap).
- Total caches per fn (Plan 123.1 + LICM combined): still bounded by
  `max_per_fn` (default 8). LICM hoists count toward this.

### 2.5 Edge cases (DECISION-F.2)

- **Nested loops:** hoist to the nearest enclosing loop boundary
  where field is invariant (i.e. NOT mutated in any nested loop
  iteration). Conservative: hoist to outermost loop body block.
- **`break` / `continue` / `return`:** correctness preserved — hoisted
  load executed before loop body iterations; control-flow exits
  don't matter.
- **Loop never iterates (e.g. `0..0`):** hoisted load executed in
  vain. Semantics preserved — ro field load is side-effect-free.
  Slight overhead acceptable for zero-iteration loops.
- **Loop with `perform Effect.X`:** ro fields safe (frozen). Mut
  fields NOT hoisted in V2 (perform may invoke handler что mutates).
- **Closure inside loop:** if closure body refs `@F` → skip hoisting
  F.

  invoked **before** existing per-fn caching.
- For each fn body, walk recursively:
  - On encountering Loop (For/While/Loop/WhileLet/ParallelFor):
    - Collect all `@F` reads in body (recursive into nested but
      tracking depth).

    before loop.
  - Replace reads of `@F` inside loop body with cache ident.

## 5. Risk register (V2-specific)

Per Plan 123 umbrella §7.1:

- **R-2.1: loop-carried mutation invisible.** Method call внутри loop
  body может indirectly mutate `@field`. Mitigation: V2 — skip LICM
  for any call in body для mut fields. Ro unaffected.
- **R-2.2: early-return interaction.** Hoisted load executed even
  if loop never runs. Mitigation: only hoist read-only loads
  (side-effect-free).
- **R-2.3: composition with Plan 123.1.** Risk: double-cache or
  shadowing. Mitigation: LICM runs first; Plan 123.1 sees hoisted
  locals и доcaches at method-body prefix only if accesses outside
  loop also ≥ threshold.
