# Plan 83.3: `Blocking`-эффект → libuv threadpool offload

## 1. Проблема

`NOVA_MAXPROCS=N` обещает «N CPU параллелизма». Это правда только если
worker'ы **не залипают**. Nova-worker — это слитые Go-`P`+`M` (Plan 83
§3 П1): OS-поток + libuv-loop + deque. Если fiber на worker'е делает
**genuinely-blocking** работу инлайн, весь worker встаёт — теряется один
`P`, фактический параллелизм проседает ниже `NOVA_MAXPROCS`.

Вся обычная I/O в Nova уже async через libuv (Plan 22 — sleep, timers,
channels — не блокируют). Остаётся genuinely-blocking, помеченное
эффектом **`Blocking`** (D50): FFI в блокирующие C-библиотеки,
блокирующие syscall'ы вне `uv_fs`, тяжёлый CPU-bound leaf.

Как делают другие:
- **Go** — `M` залипает в syscall'е, но `P` отвязывается и подхватывается
  другим `M` → параллелизм сохраняется.
- **tokio** — `spawn_blocking` уводит блокирующую задачу в отдельный пул
  (default 512 потоков), async-worker'ы свободны.
- **Nova сейчас** — `Blocking`-операция выполняется инлайн на worker'е →
  пинит его.

## 2. Решение

**`Blocking`-эффект-операция уходит в libuv threadpool** (`uv_queue_work`),
fiber паркуется, worker свободен. Тогда `NOVA_MAXPROCS` = устойчивый
параллелизм = семантика `GOMAXPROCS`.

Поток (переиспользует park/wake D93 — Plan 22 Ф.3, уже работает для
`Time.sleep`):
1. Fiber вызывает `Blocking`-операцию.
2. Runtime пакует работу в `uv_work_t`, `uv_queue_work(loop, work,
   work_cb, after_work_cb)`.
3. Fiber **паркуется** (D93) — worker свободен (берёт другой fiber /
   work-steal).
4. `work_cb` исполняется на потоке libuv threadpool — делает блокирующую
   работу.
5. `after_work_cb` исполняется обратно на loop'е worker'а-владельца —
   **будит** запаркованный fiber с результатом.
6. Fiber резюмится с результатом.

libuv threadpool **процесс-глобален** (один на все loop'ы) — `after_work_cb`
приходит на тот loop, что submit'ил → fiber просыпается на своём home-
worker'е. Консистентно с существующим park/wake.

## 3. Фазы

### Ф.0 — Аудит текущей обработки `Blocking` (~0.5 дня) ✅ ЗАКРЫТА

Установить, как codegen/рантайм трактуют эффект `Blocking` сейчас
(inline? no-op marker? есть ли вообще dispatch). Зафиксировать точку
интеграции. (Аналогично: `Detach` сейчас — bootstrap `SyncDetach` inline.)

**Результат аудита (2026-05-22):**

- `Blocking` — известное имя эффекта в type-checker'е (`types/mod.rs` —
  список suspend-эффектов `Net|Fs|Db|Time|Blocking`; запрещён внутри
  `realtime`-блока, D64). Парсится в сигнатурах функций.
- **Codegen-dispatch'а НЕТ.** `emit_c.rs` не содержит обработки
  `Blocking` — операции под этим эффектом исполняются инлайн, как
  обычные вызовы. Рантайм-обёртки offload'а нет.
- **`uv_queue_work`/`uv_work_t` нигде не используются** —
  threadpool-offload-механизм отсутствует целиком.
- **Ни одна функция в `std`/`examples`/`nova_tests` не помечена
  `Blocking`** — эффект объявлен в D50, но не имеет потребителей.
- **Точка интеграции:**
  - рантайм — новый `nova_blocking_offload(...)` в `nova_rt`, по
    образцу park/wake `Time.sleep` (D93; impl в `fibers.h`/`channels.h`,
    `NOVA_STOP_ASYNC`-контракт);
  - codegen (Ф.4) — маршрутизация операций под эффектом `Blocking` в
    эту обёртку;
  - тесты (Ф.5) — `Blocking`-фикстуры создаются с нуля.
- **Следствие для оценки:** 83.3 — greenfield-фича целиком, включая
  первых потребителей эффекта. Объём Ф.1-Ф.5 подтверждён (~6-8
  dev-days); это отдельная сфокусированная работа, не tail-end-задача.

### Ф.1 — `uv_queue_work` offload + park/wake (~2-3 дня)

- `uv_work_t`-обёртка: park-on-work, wake-on-`after_work_cb` через
  существующий D93-механизм (`nova_sched_park`/`nova_sched_wake`,
  `stop_cb` контракт `NOVA_STOP_ASYNC` — как `Time.sleep`).
- Передача результата `work_cb → after_work_cb → fiber` (box по типу,
  как interrupt-value в D93/Plan 39).
- **`UV_THREADPOOL_SIZE`** — default libuv = 4, мало (tokio = 512).
  Поднять до разумного default'а (ориентир — `max(4, NumCPU×?)` либо
  фикс ~64-128) + env `NOVA_BLOCKING_THREADS`. Важно: переменная
  читается libuv'ом **до** создания первого loop'а → выставлять в
  runtime-прологе до `nova_evloop()`.

### Ф.2 — GC-safety потоков threadpool (~1-2 дня)

`work_cb` исполняется на потоке libuv threadpool — он **не**
Boehm-registered и **не** fiber.
- **V1 (scope-ограничение):** `Blocking`-работа — **leaf**: FFI/syscall
  без аллокации GC-памяти и без вызовов обратно в Nova-рантайм. Покрывает
  основной use-case (блокирующий FFI). Простая, корректная.
- **V2 (followup):** GC-регистрация потоков threadpool
  (`GC_register_my_thread` once-per-thread через TLS-флаг) → разрешает
  произвольный Nova-код под `Blocking`. Отложено — V1 достаточно для
  паритета цели.
Решение V1 зафиксировать в spec D50 как контракт эффекта `Blocking`.

### Ф.3 — Cancellation (~1 день)

- Не стартовавший `uv_work_t` — отменяется `uv_cancel()` → fiber
  будится с cancel.
- **In-flight** блокирующая работа **не прерывается** (C-вызов
  непрозрачен) — fiber дожидается завершения, результат отбрасывается,
  бросается cancel. Это **industry-standard**: Go не прерывает
  блокирующий cgo-вызов, tokio не отменяет running `spawn_blocking`.
  Зафиксировать как documented-ограничение, не упрощение.
- Интеграция с `CancelToken` (D75) / supervised-cancel через `stop_cb`.

### Ф.4 — Codegen: маршрутизация `Blocking`-операций (~1-2 дня)

- codegen эмитит для операций под эффектом `Blocking` обёртку offload'а
  вместо инлайн-вызова.
- Граница: только то, что реально помечено `Blocking` — обычные функции
  не затрагиваются.

### Ф.5 — Тесты + spec (~1 день)

- Фикстуры `nova_tests/`: `Blocking`-операция не пинит worker (параллельно
  идут другие fiber'ы); пул-насыщение (> `NOVA_BLOCKING_THREADS`
  одновременных — корректная очередь); cancel не-стартовавшей и
  in-flight работы.
- spec D50 amend: `Blocking` → threadpool-offload + V1-контракт (leaf) +
  cancellation-семантика.

## 4. Acceptance

- Fiber под `Blocking` не блокирует свой worker — соседние fiber'ы
  прогрессируют (измеримо: N `Blocking`-операций + параллельный
  CPU-fiber → CPU-fiber не стоит).
- `NOVA_MAXPROCS=N` под блокирующей нагрузкой даёт ≈N CPU throughput
  (не проседает к 1).
- `UV_THREADPOOL_SIZE`/`NOVA_BLOCKING_THREADS` — настраиваемы; пул-
  насыщение деградирует graceful (очередь, не дедлок).
- Cancel: не-стартовавшая работа отменяется; in-flight — documented.
- 0 регрессий.

## 5. Риск

- Низкий-средний. Переиспользует обкатанный park/wake D93 (тот же путь,
  что `Time.sleep`).
- V1 leaf-ограничение `Blocking` — осознанное; полный Nova-код под
  `Blocking` (V2) отложен честно.
- libuv threadpool-default 4 — легко упустить; Ф.1 явно поднимает.
