## 1. Проблема

`NOVA_MAXPROCS=N` обещает «N CPU параллелизма». Это правда только если
worker'ы **не залипают**. Nova-worker — это слитые Go-`P`+`M` (Plan 83
§3 П1): OS-поток + libuv-loop + deque. Если fiber на worker'е делает
**genuinely-blocking** работу инлайн, весь worker встаёт — теряется один
`P`, фактический параллелизм проседает ниже `NOVA_MAXPROCS`.

Вся обычная I/O в Nova уже async через libuv (Plan 22 — sleep, timers,
channels — не блокируют). Остаётся genuinely-blocking, помеченное
эффектом **`Blocking`** (D50): FFI в блокирующие C-библиотеки,
блокирующие syscall'ы вне `uv_fs`, тяжёлый CPU-bound leaf.

Как делают другие:
- **Go** — `M` залипает в syscall'е, но `P` отвязывается и подхватывается
  другим `M` → параллелизм сохраняется.
- **tokio** — `spawn_blocking` уводит блокирующую задачу в отдельный пул
  (default 512 потоков), async-worker'ы свободны.
- **Nova сейчас** — `Blocking`-операция выполняется инлайн на worker'е →
  пинит его.

## 2. Решение

**`Blocking`-эффект-операция уходит в libuv threadpool** (`uv_queue_work`),
fiber паркуется, worker свободен. Тогда `NOVA_MAXPROCS` = устойчивый
параллелизм = семантика `GOMAXPROCS`.

Поток (переиспользует park/wake D93 — Plan 22 Ф.3, уже работает для
`Time.sleep`):
1. Fiber вызывает `Blocking`-операцию.
2. Runtime пакует работу в `uv_work_t`, `uv_queue_work(loop, work,
   work_cb, after_work_cb)`.
3. Fiber **паркуется** (D93) — worker свободен (берёт другой fiber /
   work-steal).
4. `work_cb` исполняется на потоке libuv threadpool — делает блокирующую
   работу.
5. `after_work_cb` исполняется обратно на loop'е worker'а-владельца —
   **будит** запаркованный fiber с результатом.
6. Fiber резюмится с результатом.

libuv threadpool **процесс-глобален** (один на все loop'ы) — `after_work_cb`
приходит на тот loop, что submit'ил → fiber просыпается на своём home-
worker'е. Консистентно с существующим park/wake.

## 3. Фазы

### Ф.0 — Аудит текущей обработки `Blocking` (~0.5 дня) ✅ ЗАКРЫТА

Установить, как codegen/рантайм трактуют эффект `Blocking` сейчас
(inline? no-op marker? есть ли вообще dispatch). Зафиксировать точку
интеграции. (Аналогично: `Detach` сейчас — bootstrap `SyncDetach` inline.)

**Результат аудита (2026-05-22):**

- `Blocking` — известное имя эффекта в type-checker'е (`types/mod.rs` —
  список suspend-эффектов `Net|Fs|Db|Time|Blocking`; запрещён внутри

- **Codegen-dispatch'а НЕТ.** `emit_c.rs` не содержит обработки

- **`uv_queue_work`/`uv_work_t` нигде не используются** —
  threadpool-offload-механизм отсутствует целиком.
- **Ни одна функция в `std`/`examples`/`nova_tests` не помечена
  `Blocking`** — эффект объявлен в D50, но не имеет потребителей.
- **Точка интеграции:**
  - рантайм — новый `nova_blocking_offload(...)` в `nova_rt`, по
    образцу park/wake `Time.sleep` (D93; impl в `fibers.h`/`channels.h`,
    `NOVA_STOP_ASYNC`-контракт);
  - codegen (Ф.4) — маршрутизация операций под эффектом `Blocking` в
    эту обёртку;
  - тесты (Ф.5) — `Blocking`-фикстуры создаются с нуля.
- **Следствие для оценки:** 83.3 — greenfield-фича целиком, включая
  первых потребителей эффекта. Объём Ф.1-Ф.5 подтверждён (~6-8
  dev-days); это отдельная сфокусированная работа, не tail-end-задача.

### Ф.1 — `uv_queue_work` offload + park/wake (~2-3 дня)

**Runtime-механизм ✅ ГОТОВ (2026-05-22).** `nova_blocking_offload` +
callbacks в `compiler-codegen/nova_rt/fibers.h` — рядом с
`_nova_sleep_via_libuv`:

- `uv_work_t`-обёртка: `uv_queue_work` → `work_cb` (threadpool-поток
  делает блокирующую работу) → `after_work_cb` (на loop'е owner'а:
  wake parked fiber). Park/wake — существующий D93-механизм
  (`nova_sched_park`/`nova_sched_wake`, `register_pending`/
  `unregister_pending`). Проще sleep'а: `uv_work_t` — request, не
  handle → нет `uv_close`-dance.
- cancel: `stop_cb` → `uv_cancel` (отменяет ТОЛЬКО не-стартовавшую
  работу; in-flight C-вызов доводится — industry-standard), возвращает
  `NOVA_STOP_ASYNC`. `after_work_cb` отрабатывает в любом случае → wake.
- Build-verified (`static inline` — компилируется начисто, 0 регрессий
  в полном прогоне). Пока **не подключён** — нужен Ф.4 (codegen).

**Ф.1 leftover — threadpool sizing ✅ ЗАКРЫТ (2026-05-22):**
`_nova_init_blocking_threadpool` в `eventloop.c` (вызывается из
`nova_evloop_init`, до первого `uv_queue_work`): читает
`NOVA_BLOCKING_THREADS` (default 64) → пробрасывает в
`UV_THREADPOOL_SIZE`; явный пользовательский `UV_THREADPOOL_SIZE`
уважается; невалидное значение → warning + default. libuv клампит
к [1, 1024].

Передача результата (`work_cb → fiber`) реализована в Ф.4.2 —
`_nova_result` поле ctx-struct (см. §7).

### Ф.2 — GC-safety потоков threadpool ✅ ЗАКРЫТА (2026-05-22)

`work_cb` исполняется на потоке libuv threadpool — он **не**
Boehm-registered и **не** fiber.
- **V1 (scope-ограничение):** `Blocking`-работа — **leaf**: FFI/syscall
  без аллокации GC-памяти, без вызовов обратно в Nova-рантайм и без
  control-flow-escape наружу. Покрывает основной use-case (блокирующий
  FFI). Простая, корректная.
- **V2 (followup):** GC-регистрация потоков threadpool
  (`GC_register_my_thread` once-per-thread через TLS-флаг) → разрешает
  произвольный Nova-код под `Blocking`. Отложено — V1 достаточно для
  паритета цели.

**Результат:** V1-контракт зафиксирован в spec D50 §4 («V1 leaf-
контракт») как контракт эффекта `Blocking`. Codegen его не enforce'ит
(V2); facet'ы — `[M-83.3-blocking-leaf-contract]` + control-flow-escape
(C-компилятор ловит `return <value>`/`break`/`continue`). Проверено:
`nova_preempt_check` на threadpool-потоке — no-op (`_nova_preempt_ptr`
это TLS, NULL вне fiber'а).

### Ф.3 — Cancellation ✅ ЗАКРЫТА (2026-05-22)

- Не стартовавший `uv_work_t` — отменяется `uv_cancel()` → fiber
  будится с cancel.
- **In-flight** блокирующая работа **не прерывается** (C-вызов
  непрозрачен) — fiber дожидается завершения, результат отбрасывается,
  бросается cancel. Это **industry-standard**: Go не прерывает
  блокирующий cgo-вызов, tokio не отменяет running `spawn_blocking`.
  Documented-ограничение, не упрощение.
- Интеграция с `CancelToken` (D75) / supervised-cancel через `stop_cb`.

**Результат:** механизм реализован в Ф.1 (`_nova_blocking_stop_cb` +
`uv_cancel` + `nova_sched_register_pending`). `after_work_cb`
отрабатывает в любом случае → fiber гарантированно будится, дедлока
нет. Семантика зафиксирована в spec D50 §4 («Cancellation»). Тест —
`blocking_cancel_test.nv` (offload в `supervised(cancel:)` без cancel /
с cancel / cancel после scope'а).

### Ф.4 — Примитив `blocking { }` (декомпозирован 2026-05-22)

Аудит (Ф.0) показал: примитива `blocking { }` нет в языке вообще —
нужен полный greenfield по всем слоям компилятора, не просто «codegen-
маршрутизация». Поэтому Ф.4 разбит на две закрываемые единицы.
Пофайловая карта реализации обеих — в §7.

#### Ф.4.1 — Синтаксис + type-checker + bootstrap-codegen ✅ ЗАКРЫТА (2026-05-22)

Примитив существует и проверяется, но пока исполняется инлайн (offload
— Ф.4.2). Аналог bootstrap'а `detach` (SyncDetach — inline).

**Результат (2026-05-22):**
- Лексер: `KwBlocking` + keyword `"blocking"`.
- AST: `ExprKind::Blocking(Block)`.
- Парсер: `parse_blocking` (зеркало `parse_detach`) + dispatch +
  name-table.
- Type-checker: новое поле `CapState.declared_effects` (эффекты
  сигнатуры enclosing-fn, заполняется в `walk_fn_body`); `blocking { }`
  → ошибка, если `Blocking` не объявлен; → ошибка внутри `realtime`.
  `"Blocking"` добавлен в builtin-type-set (резолвится в сигнатурах).
- Codegen: `emit_blocking` — тело инлайн (как `emit_detach`); ветки
  `ExprKind::Blocking` во всех обходах AST (scan/collect/emit/infer).
- Прочие проходы: `interp`/`lints`/`callnorm`/`desugar`/`verify` —
  ветки `Blocking` добавлены.
- Тесты: `nova_tests/concurrency/blocking_test.nv` (12 позитивных
  кейсов) + `negative_capability/blocking_no_effect.nv` +
  `blocking_in_realtime.nv`. Все PASS.

- **Лексер:** токен `KwBlocking` + keyword `"blocking"` (§7).
- **AST:** `ExprKind::Blocking(Block)`.
- **Парсер:** `parse_blocking` + dispatch (зеркало `parse_detach`).
- **Type-checker:** `blocking { }` требует эффект `Blocking` в сигнатуре
  enclosing-функции; запрещён внутри `realtime { }` (D64).
- **Codegen (bootstrap):** `emit_blocking` — тело инлайн (как
  `emit_detach`). Семантически корректно, но worker НЕ освобождается —
  явный, plan-tracked промежуток до Ф.4.2.
- **Тесты:** позитивный — `blocking { }` с эффектом `Blocking`
  компилируется и выполняется; негативные — `blocking { }` без эффекта
  `Blocking` → compile error; `blocking { }` внутри `realtime` → error.
- **Acceptance:** примитив парсится, type-check ловит обе ошибки;
  0 регрессий.

#### Ф.4.2 — Codegen offload: `emit_blocking` → `nova_blocking_offload` ✅ ЗАКРЫТА (2026-05-22)

Замена inline-bootstrap на реальный увод работы в threadpool.

**Результат (2026-05-22):**
- `emit_blocking` переписан: block-extraction по образцу `emit_spawn`,
  но проще (тело не fiber и не scheduled — обычная leaf-функция):
  - capture-анализ свободных переменных тела (`collect_idents_block` +
    `collect_bound_names_block`) → ctx-struct `NovaBlkCtx_nova_blk_N`
    (захваты by-value/by-pointer как у spawn + поле `_nova_result`);
  - сгенерированная `static void _nova_blk_N(void* _blk_arg)` —
    распаковывает ctx, выполняет тело с capture-rewriting'ом
    (`current_spawn_captures`), пишет результат в `_c->_nova_result`;
  - на месте `blocking { }`: stack-local ctx (offload синхронен с
    точки зрения fiber'а — стек-кадр переживает park) + заполнение
    захватов + context-sensitive ветка.
- **Forward-decl** `_nova_blk_N` + ctx-struct typedef эмитятся прямо в
  `lambda_forward_decls` из `emit_blocking` — без правки scan-фазы
  (counter `blocking_counter` чисто emit-side, textual fwd-decl
  counter-sync'а не требует, в т.ч. mono-context).
- **Context-sensitivity:** `if (mco_running())` → `nova_blocking_offload`
  (fiber: worker свободен); `else` → `_nova_blk_N(&ctx)` inline
  (main-поток — нет worker'а пинить).
- **Result-passing:** `blocking { }` отдаёт значение trailing expr тела
  (`infer_expr_c_type` обновлён — `Blocking` ≠ `Detach`); читается из
  `ctx._nova_result` после wake.
- **Тесты:** `concurrency/blocking_offload_test.nv` — offload в
  fiber-контексте (result-passing), **worker не запинен** (sibling-fiber
  выставляет marker пока A запаркован на offload'е — A видит 1, не 0),
  inline на main-потоке, несколько fiber'ов с offload'ом параллельно.
- **Acceptance:** worker свободен под `blocking`-нагрузкой
  (`a_saw == 1` доказывает); 0 регрессий.

### Ф.6 — Type-checker enforcement тела `blocking { }` ✅ ЗАКРЫТА (2026-05-22)

Амендмент по review-замечанию: V1 leaf-контракт был документирован, но
codegen'ом не проверялся. Тело `blocking { }` исполняется на
libuv-threadpool-потоке (не fiber, не GC-registered) — то есть среда
inherently `nogc` + no-suspend. Компилятор это знает → должен проверять.

**Результат:**
- Новый флаг `CapState.blocking_body_active` — отдельный от
  `realtime_active` (иначе вложенный `blocking` отвергался бы как
  «`blocking` внутри `realtime`»).
- Тело `blocking { }` walk'ается с `blocking_body_active = true` +
  `realtime_nogc = true` (реюз nogc-механизма).
- **nogc:** alloc-вызовы (`[]T.new`, `HashMap.new`, `StringBuilder`,
  `str.from`, ...) внутри `blocking { }` → compile error
  (context-aware сообщение, отличное от `realtime nogc`).
- **Suspend-эффекты:** вызов функции/эффект-операции с эффектом
  `Net`/`Fs`/`Db`/`Time` внутри `blocking { }` → compile error
  (`blocking_body_forbidden_effect` — `Blocking` НЕ включён: вложенный
  `blocking` на threadpool-потоке исполняется inline, безвреден).
- **Тесты:** `negative_capability/blocking_body_alloc_rejected.nv` +
  `blocking_body_suspend_rejected.nv`.
- **Граница (осознанно вне scope, см. AskUser-решение):** `Fail`/
  `throw`/`?` НЕ запрещены — спековый пример `blocking { c_read() }`
  с `Fail`-FFI остаётся компилируемым; `throw` на threadpool-потоке —
  V1-риск, документирован в D50 §4. `nogc`-проверка — консервативный
  whitelist (не ловит user-record-литералы) — приближение, не 100%.

## 4. Acceptance

- Fiber под `Blocking` не блокирует свой worker — соседние fiber'ы
  прогрессируют (измеримо: N `Blocking`-операций + параллельный
  CPU-fiber → CPU-fiber не стоит).
- `NOVA_MAXPROCS=N` под блокирующей нагрузкой даёт ≈N CPU throughput
  (не проседает к 1).
- `UV_THREADPOOL_SIZE`/`NOVA_BLOCKING_THREADS` — настраиваемы; пул-
  насыщение деградирует graceful (очередь, не дедлок).
- Cancel: не-стартовавшая работа отменяется; in-flight — documented.
- 0 регрессий.

## 5. Риск

- Низкий-средний. Переиспользует обкатанный park/wake D93 (тот же путь,
  что `Time.sleep`).
- V1 leaf-ограничение `Blocking` — осознанное; полный Nova-код под
  `Blocking` (V2) отложен честно.
- libuv threadpool-default 4 — легко упустить; Ф.1 явно поднимает.

## 7. Карта реализации Ф.2-Ф.5 (для следующей сессии)

Ф.1 runtime-механизм готов (`nova_blocking_offload` в `fibers.h`).
Остаётся обвязка примитива `blocking { }` — greenfield по слоям.
Прецедент синтаксиса — `detach { }` (block-примитив с эффектом
`Detach`); но codegen `blocking` ближе к `spawn` (нужна экстракция
тела в отдельную функцию, не inline).

**Лексер** (`compiler-codegen/src/lexer/`):
- `token.rs`: `KwBlocking` в `enum TokenKind` (рядом с `KwDetach`,
  ~стр. 95) + ветка display (~стр. 232).
- `mod.rs`: `"blocking" => TokenKind::KwBlocking` в keyword-таблице
  (~стр. 636, рядом с `"detach"`).

**AST** (`compiler-codegen/src/ast/mod.rs`):
- `ExprKind::Blocking(Block)` рядом с `Detach(Block)` (~стр. 1389).

**Парсер** (`compiler-codegen/src/parser/mod.rs`):

- name-table `KwBlocking => "blocking"` (~стр. 4743);
- `parse_blocking()` — копия `parse_detach()` (~стр. 5831).

**Type-checker** (`compiler-codegen/src/types/mod.rs`):
- `blocking { }` требует эффект `Blocking` в сигнатуре enclosing-функции
  (как `detach` → `Detach`);
- запрещён внутри `realtime { }` (D64) — `Blocking` уже в списке
  suspend-эффектов (~стр. 3073);
- негативные тесты: `blocking{}` без эффекта `Blocking`; `blocking{}`
  в `realtime`.

**Codegen** (`compiler-codegen/src/codegen/emit_c.rs`) — Ф.4, главная часть:
- `emit_blocking(body)` — НЕ inline (в отличие от `emit_detach`,
  ~стр. 5146). Экстракция тела по образцу `emit_spawn` (~стр. 4506):
  capture-анализ свободных переменных → ctx-struct (захваты + результат)
  → сгенерированная `void _nova_blk_N(void* ctx)` (распаковать, выполнить
  тело, записать результат) → на месте `blocking{}`:
  `nova_blocking_offload(_nova_active_scope, _nova_active_slot,
  _nova_blk_N, &ctx)` + чтение результата.
- forward-decl `_nova_blk_N` — ветка в `scan_expr_fwd` (~стр. 5210,
  рядом со `Spawn`).
- ветки `ExprKind::Blocking` в: `emit_expr` (~стр. 12658, рядом с
  `Detach`), `collect_bound_names_block` (~стр. 5380),
  `collect_idents_block` (~стр. 5568), `scan_block_fwd`.

**Открытые дизайн-вопросы:**
1. **Context-sensitivity.** `nova_blocking_offload` требует fiber-
   контекст (scope/slot). `blocking{}` на main-потоке (вне fiber'а) —
   нет worker'а пинить → выполнять тело inline. Codegen — ветка по
   контексту (как context-sensitive default impl `Time.sleep`).
2. **Result-passing.** Тело `blocking{}` может давать значение
   (`blocking { c_read() }` → `[]u8`). Box по типу в ctx-struct,
   читать после wake (как interrupt-value D93/Plan 39).
3. **V1 leaf-контракт.** Тело `blocking{}` выполняется на не-Boehm-
   registered threadpool-потоке → V1: тело — leaf (FFI без GC-
   аллокации и без вызовов в Nova-рантайм). Codegen'ом не enforced;
   зафиксировать в D50. V2 (Ф.2) — GC-регистрация threadpool-потоков.

**Тесты Ф.5** (позитивные + негативные): `blocking`-операция не пинит
worker (параллельный CPU-fiber прогрессирует); пул-насыщение; cancel
не-стартовавшей и in-flight работы; негативные — `blocking` без
эффекта / в `realtime`.

**Spec Ф.5:** D50 amend — `blocking { }` → threadpool-offload + V1 leaf-
контракт + cancellation-семантика.
