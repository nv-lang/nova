> **Spec:** [D145](../../spec/decisions/02-types.md#d145).

## 1. Зачем

`std/collections/vec.nv` написан pattern `fn []T @method[U]`, где T —
именованный тип (не дженерик). Парсер silent-принимает, codegen
падает → vec.nv не компилируется в exe → Plan 91 (std MVP) blocked.

**Lesson learned:** автор vec.nv assumed «T в `[]T` автоматически
становится generic». Это **wrong assumption** — D145 финализирует:
**никакого implicit T**, explicit `fn[T]` prefix обязателен.

## 2. Что меняется

### Grammar

```ebnf
FnDecl        ::= "fn" GenericPrefix? ReceiverSpec "@" MethodName MethodGenerics?
                  "(" Params ")" Effects? ("->" ReturnType)? Body
GenericPrefix ::= "[" GenericParam ("," GenericParam)* "]"
GenericParam  ::= identifier (Bound)?     // D72 syntax reused
Bound         ::= type                    // protocol-type (D72)
```

### Правила

| Receiver-shape | `fn[T]` нужен? | Пример |
|---|---|---|
| `Option[T]`, `HashMap[K, V]` (named-generic carrier) | ❌ | `fn Option[T] @map` |
| `[]T` | ✅ | `fn[T] []T @map` |
| `T` bare | ✅ | `fn[T] T @identity` |
| `(T, U)` tuple | ✅ | `fn[T, U] (T, U) @swap` |
| `(T, Option[U])` mix | T нужен | `fn[T] (T, Option[U]) @pair` |
| `[]Option[T]` composite | T через Option | ❌ |

### Error codes (4)

| Code | Когда | Suggestion |
|---|---|---|
| `E_BARE_TYPEVAR_NEEDS_PREFIX` | `fn T @method` без `fn[T]` без named T в scope | Add `fn[T]` OR define `type T` |
| `E_PREFIX_SHADOWS_NAMED_TYPE` | `fn[T] T @method` + есть `type T` в scope | Rename prefix var OR use full path |
| `E_DUPLICATE_GENERIC_DECL` | `fn[K, V] HashMap[K, V] @method` (carrier уже декларирует) | Remove `fn[…]` или удалить из `HashMap[…]` |
| `E_UNDECLARED_TYPEVAR_IN_RECEIVER` | `fn []T @method` без `fn[T]` префикса И без `type T` в scope | Add `fn[T]` prefix |

### Ф.1 — Parser: grammar extension

**Acceptance:** парсер принимает `fn[T] ReceiverType @method` грамматику.

**Actions:**
1. Add `parse_fn_prefix_generics()` между `fn` keyword и receiver-spec
   в `compiler-codegen/src/parser/...`.
2. Reuse `parse_generic_params` от D72 (bound syntax уже работает).
3. AST: extend `FnDecl` если нужно (новое поле `prefix_generics` или
   reuse `generics`).
4. Tests parser-level:
   - `fn[T] T @method` — parses → generics=[T].
   - `fn[T Hashable] []T @method` — parses → generics=[(T, Some(Hashable))].
   - `fn[ ] T @method` — parse error (empty list).
   - `fn[T, T] T @method` — parse error (duplicate).

### Ф.2 — Type-checker: disambiguation enforcement

**Acceptance:** все 4 error codes из §2 enforced + 8-row matrix covered.

**Actions в** `compiler-codegen/src/check/...`:

1. **Сбор fn-generics:**
   - От `fn[…]` prefix → pre-decl set.
   - От carrier-brackets в receiver (`Option[T]`, `HashMap[K, V]`) → carrier-decl set (existing D119).
2. **Disambiguation walk** receiver-type:
   - Single-upper identifier (T, U, K, V, E, F):
     - Если в pre-decl или carrier-decl set → typevar.
     - Если в scope как `type X` → named type ref.
     - Если в обоих → error `E_PREFIX_SHADOWS_NAMED_TYPE` или `E_DUPLICATE_GENERIC_DECL`.
     - Если ни в одном → error `E_BARE_TYPEVAR_NEEDS_PREFIX` /
       `E_UNDECLARED_TYPEVAR_IN_RECEIVER`.
   - Multi-char identifier: scope-lookup, не trigger typevar.
3. **Usage validation:** каждый pre-decl typevar **используется** в
   receiver/params/return; иначе warning `W_UNUSED_PREFIX_TYPEVAR`.

### Ф.3 — Codegen: dispatch на mono'd `fn[T]` methods

**Acceptance:** `vec.nv map[U]/filter/fold[Acc]/any/all/first/last` компилируются с
mono per (T) или (T, U).

**Actions в** `compiler-codegen/src/codegen/emit_c.rs`:

1. **Receiver-c-type resolution для array-T:** при receiver-c-type
   `NovaArray_<X>*`, extract T = <X>, push в receiver_subst (analog
   Plan 99 logic для Option[T]).
2. **Mono-name generation:** `Nova___Array____<T>_method_<m>` (+ `____<U>` для method-level).
3. **Bare-T receiver mono:** `Nova_<concrete_T>_method_<m>` где T = arg type.
4. **Tuple receiver:** `Nova__NovaTuple_<T1>_<T2>_method_<m>`.
5. Register mono-instance в worklist + emit body с substituted typevars.

### Ф.4 — vec.nv migration

**Acceptance:** vec.nv переписан с `fn[T]` prefix; все 7 методов type-check + codegen + run.

**Actions:**
1. Edit `std/collections/vec.nv`:
   ```nova
   fn []T @map[U](f fn(T) -> U) -> []U      →  fn[T] []T @map[U](f fn(T) -> U) -> []U
   fn []T @filter(pred fn(T) -> bool) -> []T →  fn[T] []T @filter(pred fn(T) -> bool) -> []T
   fn []T @fold[Acc](init Acc, f ...)        →  fn[T] []T @fold[Acc](init Acc, f ...)
   fn []T @any(pred ...) -> bool              →  fn[T] []T @any(pred ...) -> bool
   fn []T @all(pred ...) -> bool              →  fn[T] []T @all(pred ...) -> bool
   fn []T @first() -> Option[T]               →  fn[T] []T @first() -> Option[T]
   fn []T @last() -> Option[T]                →  fn[T] []T @last() -> Option[T]
   ```
2. Verify vec.nv tests (line 152+) PASS.

- `duplicate_name_parse_neg.nv` — `fn[T, T] T @method` → parse error
