## Зачем

Сейчас нет способа re-export'ить item из чужого модуля под собственным
namespace без wrapper-функции. Это приведёт к boilerplate при росте
stdlib.

**Сценарий:**
```
std/encoding/json/
├── parse.nv      // export fn parse_str(s) -> Json
├── format.nv     // export fn to_str(j Json) -> str
└── types.nv      // export type Json
```

Public API хочется иметь `std.encoding.json.{parse_str, to_str, Json}`,
но в folder-module это автоматически. **Но если у нас:**

```
std/encoding/json/
├── _public.nv         // facade — собирает API
├── internal/
│   ├── parser.nv
│   ├── lexer.nv
│   └── formatter.nv
```

Тогда `_public.nv` хочет re-export из `internal/parser.{parse_str}`,
не дублируя сигнатуру.

---

## Архитектурное решение

**Rust `pub use`-style** (не TS barrel):

```nova
export import std.encoding.json.internal.parser.{parse_str}
export import std.encoding.json.internal.formatter.{to_str}
```

Уже частично есть в Plan 35.A R26 — `export import` поддержано. **Что
нужно дополнительно:**

1. **Rename при re-export:** `export import X.{Foo as Bar}`.
2. **Selective re-export:** `export import X.{A, B}` (есть) vs
   `export import X.*` (decide: yes/no).
3. **Dedup logic:** если 2 peers делают `export import X.{A}` — A
   видим snover externally один раз (no error).
4. **Type-checker visibility enforcement** для re-exported items (R26 todo).

## Что НЕ входит

  hard reject.
