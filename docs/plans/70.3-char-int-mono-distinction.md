## Зачем

`char` и `int` оба mapping'ятся в один C тип `nova_int` (= `long long`):

```rust
// compiler-codegen/src/codegen/emit_c.rs:2663,2680
"int"  => Ok("nova_int".into()),
"char" => Ok("nova_int".into()),   // SAME!
```

Это даёт **silent type bugs** в generic instantiation: `Option[char]`
и `Option[int]` mangle'ятся в одинаковое `NovaOpt_nova_int`,
structurally interchangeable → type-checker не различает их в slot
positions.

**Concrete observed bug** (триггер плана): функция declared с
`-> Option[int]` фактически возвращала `Option[char]` (комментарий в
nova_rt/array.h явно «Returns Option[char] = NovaOpt_nova_int»), ~50
callers в std/* использовали как char (`Some('/')`, `unwrap_or('.')`,
передача в `Option[char]`-параметры). Type-checker пропустил —
declaration → wrong, runtime → правильно из-за collapse'а.

## Inventory collapse sites

Все примитивы Nova → C маппинги (per `type_ref_to_c` emit_c.rs:2663+):

| Nova | C-тип | Collision с |
|---|---|---|
| `int` | `nova_int` (`long long`) | **`char`** ← collision |
| `char` | `nova_int` ← **CAUSES** Plan 70.3 bug | `int` |
| `i64` | `nova_int` | `int`, `char` |
| `i32` | `int32_t` | distinct ✅ |
| `i16` | `int16_t` | distinct ✅ |
| `i8` | `int8_t` | distinct ✅ |
| `u8`/`byte` | `uint8_t`/`nova_byte` | distinct ✅ (Plan 68 cleanup) |
| `u16`-`u64` | `uint*_t` | distinct ✅ |
| `bool` | `nova_bool` | distinct ✅ |
| `f64` | `nova_f64` (`double`) | `f32`? |
| `f32` | `nova_f64` ← collapses to f64 в bootstrap | `f64` |

**Two collapse pairs:**
1. **`char` ↔ `int` ↔ `i64`** — all → `nova_int`
2. **`f32` ↔ `f64`** — both → `nova_f64` (separate concern, не в scope Plan 70.3)

Plan 70.3 фокусируется на (1) — char/int — поскольку char семантически
**отдельный** kind (Unicode codepoint vs integer math).

## Manifestation в generic mono

`compute_generic_type_c_name` использует `sanitize_c_for_ident(c_ty)`
для name составления:

```rust
// "HashMap"+["nova_str", "nova_int"] → "Nova_HashMap____nova_str__nova_int"
```

Поскольку и `int` и `char` mapping'ятся в `nova_int`, для них generates
**идентичные** mangled-имена:

```
Option[int]   → NovaOpt_nova_int
Option[char]  → NovaOpt_nova_int    (SAME!)
[]int          → NovaArray_nova_int
[]char         → NovaArray_nova_int  (SAME!)
Map[int, V]    → Nova_HashMap____nova_int__<V>
Map[char, V]   → Nova_HashMap____nova_int__<V>  (SAME!)
```

Pattern-matched arms switching on type discriminant cannot distinguish.

## Архитектурные опции

### Option A: distinct C typedef `nova_char`

```c
// nova_rt/types.h (existing)
typedef long long nova_int;
typedef long long nova_char;  // NEW — same underlying, different name
```

Plus update `type_ref_to_c`:
```rust
"char" => Ok("nova_char".into()),   // not "nova_int"
```

**Pros:**
- Minimal codegen change (1 line + 1 typedef)
- C-level still single underlying type — zero ABI cost
- Mangling distinguishes: `NovaOpt_nova_char` vs `NovaOpt_nova_int`

**Cons:**
- Existing 50+ callers с `Some('/')` или `Some(42)` (int literal) where
  char expected — type-checker now rejects coercion. Migration required.
- Runtime helpers (e.g. NOVA_ARRAY_DECL для nova_int) — нужны
  параллельные `nova_char` versions OR runtime-side reuse через
  alias-typedef detection.

### Option B: разные mangled-only имена, C type unchanged

```rust
fn sanitize_c_for_ident(c_ty: &str, nova_ty_hint: Option<&str>) -> String {
    match (c_ty, nova_ty_hint) {
        ("nova_int", Some("char")) => "nova_int_char_mark".into(),
        _ => default_sanitize(c_ty)
    }
}
```

**Pros:**
- C-level zero change — `nova_int` everywhere
- Generic mono names distinguish

**Cons:**
- Threads `nova_ty_hint` через много вызовов `compute_generic_type_c_name`
- Magic suffix `_char_mark` confusing — leak abstraction в C names

### Option C: type-checker-only enforcement (no codegen change)

Stricter type-checker rule: `int` literal in `Option[char]` slot →
explicit cast required. Generic mono продолжает collapse в C, но Nova
code более strict.

**Pros:** Minimum churn в codegen.
**Cons:** Doesn't fix the underlying mangling collision; only catches
some user errors. Если type-checker has bugs (как сейчас!) — silent
collapse continues.

---

### Рекомендация: **Option A** (distinct `nova_char` typedef)

Cleanest architecturally. Migration cost — well-bounded:
- 50 callers (per user audit) need explicit `'/' as char` или fix
  declared type
- nova_rt helpers — automated parallel emit (similar к nova_str / nova_int
  pattern)
