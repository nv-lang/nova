## 1. Scope

Generic types with priv fields:
```nova
export type Stack[T] {
    priv mut items []T
    ro capacity int
}
```

Verify:
1. Parser принимает `priv` modifier на generic record field (T-substitution
   не нужна на parse/check phase).
2. Mono'd instance (`Stack[int]`, `Stack[str]`) enforces priv same as
   non-generic — priv check pre-mono (AST level), regardless of T.
3. Generic method calling another generic method — recv tracking
   корректен через T-parameterized `fn Stack[T] @push(x T)`.
4. Mono'd anonymous tuple D216 (Plan 59.1) — priv preserved через
   monomorphization (если tuple field marked priv — D222 NEW в 124.4
   covers this, но verifies foundation here для record case).
5. `Option[Account]` где Account имеет priv fields — outer Option
   access OK, inner Account-field access обычные правила.

Out of scope:
- Named tuple priv (Plan 124.4 D222)
- Protocol method impl boundary (Plan 124.4)
- Edition flip (Plan 124.7)

## 2. Design

### 2.1 Architecture decision: AST-level enforcement = generics work «for free»

Plan 124.1/124.2 enforcement читает `fdecl.priv_field` из
`RecordField` (AST). Mono'd struct codegen — отдельный layer
(post-typecheck). При expansion `Stack[int]` AST не дублируется —
field decls остаются те же. → priv check inherently uniform.

**Acceptance**: positive + negative fixtures для Generic Record
демонстрируют поведение identical к non-generic; никаких code-changes
не требуется.

### 2.2 Receiver tracking for generic methods

`fn Stack[T] @push(item T)` — receiver type = `Stack[T]`.
`current_recv_type` field в TypeCheckCtx — name-based (`Some("Stack")`).
Generic parameter `[T]` не влияет на name comparison. Works.

### 2.3 Mono'd anonymous tuple (Plan 59.1 D216 cross-check)

Plan 59.1 анонимные tuples мономорфизируются per instantiation как
`_NovaTuple_<arity>_<L_T>_<N_T>...`. Named tuples (D215) имеют
TypeDecl с RecordField list. Priv check работает identically:
- Named tuple `type Vec3(priv x f64, ...)` — Plan 124.4 D222 (tuple
  form syntactic + access rules).
- Anonymous tuple `(a, b)` — no field naming, no priv concept
  (positional access controlled separately в 124.4 D222 §2).

Этот sub-plan фокусируется на **generic record case**; 124.4 берёт
on tuple form.

## 5. D-block amend

D220 (02-types.md) — add **§G1 Generic types** clause:

```
### §G1 Generic types

Per-field `priv` modifier applies uniformly к generic types: 
`type Stack[T] { priv mut items []T }`. Field visibility is 
determined pre-monomorphization (AST level); mono'd instances
(`Stack[int]`, `Stack[str]`) inherit identical enforcement
regardless of T substitution.

Receiver-type tracking (TypeCheckCtx.current_recv_type) uses
type-name only — generic parameters do not factor into match.
`fn Stack[T] @push(...)` body has recv = "Stack" → priv access ok.

Inside generic methods, accessing private fields on values of the
same generic type (any T instantiation) is allowed. Cross-type
access (e.g., `Stack[int].@field` from `Queue[T].method`) — blocked
(receiver = Queue, not Stack).

For anonymous tuples and named tuples — see D222 (Plan 124.4).
```
