# Plan 33.9: Contracts — Opaque / Reveal / Fuel

> Продолжение [Plan 33.6](33.6-contracts-production-hardening.md). Решает
> matching-loops и proof-context explosion для recursive `#pure` функций
> при scaling до 50+ контрактов.

## Контекст

После Plan 33.6 (50+ спринтов production hardening, 227 PASS) verification
infrastructure стабилен. При попытке масштабировать на больший корпус
contracts обнаруживается **matching-loops problem**: Z3 пытается развернуть
тело recursive `#pure` fn бесконечно, превышая proof budget.

Pattern (Dafny/Verus наследие):

```nova
#pure fn list_len[T](l: List[T]) -> int =>
    match l {
        Nil => 0
        Cons(_, tail) => 1 + list_len(tail)
    }

#verify
fn reverse_preserves_len[T](xs: List[T]) -> List[T]
    requires xs.len() > 0
    ensures list_len(result) == list_len(xs)
{ /* ... */ }
```

Z3 quantifier `list_len(xs) == list_len(reverse(xs))` пытается развернуть
`list_len` рекурсивно на каждом уровне — matching loop. Текущий timeout
2000ms hits.

Решение: **opaque/reveal/fuel** — стандартный Dafny/Verus подход:

- `#opaque fn` — body НЕ раскрывается в SMT scope по умолчанию.
  Эффект: `list_len(xs)` обрабатывается как abstract uninterpreted function.
- `reveal name` — раскрывает opaque body в текущем fn-body scope.
  Эффект: внутри одной fn после `reveal` Z3 видит body, в других fn — нет.
- `#fuel(n)` — раскрывает recursive opaque fn до глубины `n` (default 0).
  Эффект: matching loop ограничен n уровнями.

**Nova-уникальные преимущества (защищать):**
- `#opaque` как первоклассный attribute (не magic comment как Dafny `{:opaque}`).
- `reveal` как statement (не #-attribute) — выразительно для пошаговой
  proof-construction.
- `#fuel(n)` композируется с `#proof_budget(timeout_ms=N)` (Plan 33.6 Ф.3.4)
  для fine-grained control.

**Расхождение с ближайшими аналогами:**
- **Dafny:** `{:opaque}` annotation, `reveal X();` statement, `{:fuel X, n}` —
  Nova синтаксис чище (нет magic comment'ов), композируется с другими attrs.
- **Verus:** `#[verifier::opaque]`, `reveal!(fn_name)` macro — Nova
  читабельнее (без macros).
- **F*:** `unfold` / `opaque_to_smt` — Nova attribute model более consistent
  с rest of language.

## Цели

1. **Соundness:** opaque fn должна вести себя как uninterpreted function
   в SMT scope (Z3 не должна "случайно" увидеть body).
2. **Decidability:** reveal/fuel дают предсказуемый proof budget — нет
   matching loop divergence.
3. **Modularity:** один fn может reveal opaque body другой fn без
   affect на verification других fn's в module.
4. **Zero-cost runtime:** opaque/reveal/fuel — чисто spec-level attrs,
   не emit'ятся в C-code.

## Не входит в Plan 33.9

- Cross-module reveal (V3 — требует module import tracking).
- Fuel inference (auto-determined depth based on call-graph) — V3.
- Opaque на `lemma` (lemma уже proof term, opaque concept не applicable) —
  скорее всего semantic error.
- Interaction с `pure_view`/`axiom` — TBD, обсудить в Ф.5.
- IDE support (LSP reveal hints) — Plan 33.16 scope.

## Ф.1 — Parser + AST для `#opaque` attribute

**Трудоёмкость:** ~0.5 дня

**Файлы:**
- `compiler-codegen/src/lexer/token.rs` — `#opaque` обычно парсится как
  attribute (KwHash + Ident), keyword не нужен.
- `compiler-codegen/src/ast/mod.rs` — `FnDecl.is_opaque: bool`.
- `compiler-codegen/src/parser/mod.rs` — `parse_attr` принимает `opaque`.

**Подход:**
1. В `parse_fn_attrs` (или wherever existing attrs parsed) — добавить
   case for `opaque`.
2. Default `is_opaque = false`.
3. Validation: `#opaque` совместим с `#pure` (only pure fn opaqueable);
   conflict с `#verify` (#verify fn должна быть полностью видна) — error.

**Acceptance:**
- `#opaque #pure fn foo => ...` — parses OK.
- `#opaque #verify fn foo() ...` — compile error `#opaque conflicts with
  #verify [E2410]`.
- `#opaque fn foo => ...` без `#pure` — compile error `#opaque требует
  #pure [E2410]`.

**Тесты:**
- `nova_tests/contracts/opaque_parse_positive.nv`
- `nova_tests/contracts/opaque_with_verify_fail.nv` (EXPECT_COMPILE_ERROR E2410)
- `nova_tests/contracts/opaque_without_pure_fail.nv` (EXPECT_COMPILE_ERROR E2410)

## Ф.3 — Parser + AST для `#fuel(n)` attribute

**Трудоёмкость:** ~0.3 дня

**Файлы:**
- `compiler-codegen/src/ast/mod.rs` — `FnDecl.fuel: Option<u32>`.
- `compiler-codegen/src/parser/mod.rs` — `parse_attr` принимает
  `fuel(N)` argument.

**Синтаксис:**
```nova

```

**Семантика:**
- `#fuel(0)` (default для opaque) — body не раскрывается.
- `#fuel(N)` — рекурсивный unfold до глубины N.
- `#fuel` без `#opaque` — warning W2403 (fuel meaningful только с opaque).

**Acceptance:**
- `#fuel(3)` парсится.
- `#fuel(0)` эквивалентно отсутствию `#fuel`.
- `#fuel` без аргумента → parse error.
- `#fuel(2)` без `#opaque` → W2403.

**Тесты:**
- `nova_tests/contracts/fuel_parse_positive.nv`
- `nova_tests/contracts/fuel_without_opaque_w2403.nv`

## Ф.4 — TrivialBackend integration

**Трудоёмкость:** ~0.5 дня

**Подход:**
TrivialBackend не делает SMT unfolding (это работа Z3). Для opaque/reveal
интеграция тривиальна:

- `#opaque` fn body просто **игнорируется** при substitution в caller
  ensures/requires. Caller видит fn-call как opaque term (как сейчас
  обрабатываются non-pure calls).
- `reveal name` statement — silent passthrough (нет эффекта на TrivialBackend
  reasoning).
- `#fuel(n)` — silent passthrough.

**Файлы:**
- `compiler-codegen/src/verify/encode.rs` — в `encode_expr_with_ctx` для
  `ExprKind::Call`: если `fd.is_opaque` — emit UF (uninterpreted function
  application), не inline body.
- `compiler-codegen/src/verify/pipeline.rs` — `reveal` statement в fn body:
  для TrivialBackend skip (no-op), для Z3 backend это будет hook.

**Acceptance:**
- `#opaque #pure fn h(x int) -> int => x * 2` + caller `requires h(5) > 0` →
  TrivialBackend returns Unknown (не пытается inline body). Раньше было
  silent через composition (Ф.2 Plan 33.4) — теперь explicit Unknown.
- `reveal h` в caller body — TrivialBackend ignores, behavior как до Ф.4.

**Тесты:**
- `nova_tests/contracts/opaque_trivial_unknown_w2402.nv` — opaque fn
  под TrivialBackend выдаёт W2402 «cannot prove without reveal или Z3 backend».

## Ф.5 — Z3 backend integration (axiomatic encoding)

**Трудоёмкость:** ~2 дня (research-grade)

**Подход:**
Standard approach (Dafny/Verus):

1. **Opaque fn declaration** — emit Z3 UF declaration:
   ```smt
   (declare-fun foo (Int Int) Int)
   ```
   Body НЕ axiomatized.

2. **Reveal в fn body** — emit Z3 axiom:
   ```smt
   (assert (forall ((x Int) (y Int))
     (! (= (foo x y) <encoded_body>)
        :pattern ((foo x y)))))
   ```
   Patterns (Plan 33.5 Ф.1.2) guard matching loop.

3. **Fuel(n) для recursive** — emit n-step unfolding:
   ```smt
   ; #fuel(2) для list_len
   (assert (= (list_len Nil) 0))
   (assert (forall ((h T) (t (List T)))
     (! (= (list_len (Cons h t)) (+ 1 (list_len_fuel1 t)))
        :pattern ((list_len (Cons h t))))))
   (assert (forall ((h T) (t (List T)))
     (! (= (list_len_fuel1 (Cons h t)) (+ 1 (list_len_fuel0 t)))
        :pattern ((list_len_fuel1 (Cons h t))))))
   ; list_len_fuel0 — UF без axiom (cutoff)
   ```

**Файлы:**
- `compiler-codegen/src/verify/backend/z3.rs` — `declare_opaque_fn`,
  `assert_opaque_body` (для reveal), `assert_fuel_chain`.
- `compiler-codegen/src/verify/pipeline.rs` — emit reveal/fuel axioms
  в правильном scope (push/pop boundaries).

**Acceptance:**
- list_len pattern: `#opaque #pure #fuel(2) fn list_len ...` + caller
  с `reveal list_len` — Z3 верифицирует без timeout.
- Без reveal — Z3 не доказывает (как и должен).

**Тесты:**
- `nova_tests/contracts/opaque_reveal_z3_positive.nv` (REQUIRES_SMT_BACKEND z3)
- `nova_tests/contracts/opaque_no_reveal_z3_fail.nv` (REQUIRES_SMT_BACKEND z3,
  EXPECT_COMPILE_ERROR — proves отсутствие reveal означает opaque не
  доступен в SMT).
- `nova_tests/contracts/fuel_recursive_z3_positive.nv` (REQUIRES_SMT_BACKEND z3) —
  Dafny pattern reverse-preserves-length с fuel(2).

## Не входит в Plan 33.9 (отложено)

- **Cross-module reveal** (`reveal other_module.foo`) — V3 (требует module
  import tracking).
- **Auto-fuel inference** — Plan 33.17 (AI-loop) или manual для V3.
- **Opaque generic instantiation** (`#opaque #pure fn id[T](x: T) -> T => x`) —
  V2 (требует mono-aware UF declarations).
- **reveal в lemma body** — V2 (lemma уже proof term, reveal внутри может
  быть полезен для пошаговой construction).
- **Pattern hints** (`reveal foo {:pattern ...}`) — V3 (require pattern parser).

## Open questions

1. Должно ли `#opaque` быть default для `#pure` recursive fns? **No** —
   breaking change. Default opt-out по дизайну (как Dafny v3).
2. `reveal` в lemma body — должен ли это быть error или silent passthrough?
   **TBD в Ф.5** (зависит от лemma model).
3. Composition с `#trusted external fn` — `#trusted` уже implies opaque?
   **No** — orthogonal. `#trusted` означает "доверяй axioms", `#opaque`
   означает "не раскрывай body". External fns не имеют body чтобы
   раскрывать; `#opaque` на них noise.
