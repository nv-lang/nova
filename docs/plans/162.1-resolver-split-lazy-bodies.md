# Plan 162.1 — Полный split резолвера: collect-signatures → lazy-bodies

> **Создан:** 2026-06-16. **Статус:** ✅ CLOSED (2026-06-16). P2.
> **Дочерний план:** [Plan 162](162-rust-model-module-resolution.md) Ф.1 — foundational two-pass resolver infrastructure.
> **Worktree:** `D:\Sources\nv-lang\nova-p162b`, ветка `plan-162b-resolver-split`.

## Что уже сделано в Plan 162 Ф.1

- `visited: HashMap<Vec<String>, Vec<String>>` — dedup с кэшом exported_names
- `in_progress: HashSet<Vec<String>>` — cycle guard: при цикле `return Ok(())` вместо stack overflow
- Циклы между модулями **не падают** ✅

## Что НЕ сделано (эта задача)

`resolve_imports_inline` по-прежнему **жадный**: при встрече `import X` сразу загружает и inline-merge'ит X рекурсивно, включая тела функций. «Ленивость» существует только в смысле «не бесконечная рекурсия», но не в смысле «тела резолвятся по обращению».

Настоящая Rust-модель: сначала **собрать все сигнатуры** (типы, имена функций, их сигнатуры), потом **лениво** проверять тела при обращении.

## Цель

Разделить `resolve_imports_inline` на два чётких прохода:

1. **collect_signatures(path, ...)** → `ModuleSigTable`  
   Парсит все in-build модули транзитивно, собирает только:
   - `Item::Type` (объявления типов, без методов-тел)
   - `Item::Fn` сигнатуры (имя, параметры, возврат, эффекты) — без тела `body`
   - `Item::Const` (значения констант)
   Строит `HashMap<ModuleKey, ModuleSignatures>`. Никакого typecheck тел.

2. **resolve_bodies(module, sig_table)** → typecheck тел  
   Используя готовый `sig_table`, резолвит тела функций лениво:
   - При typecheck вызова `foo()` — находит `foo` в `sig_table`, не перечитывает файл
   - Тело `foo` резолвится при первом обращении к нему (`ensure_body_checked`)

## Декомпозиция (4 шага)

### Шаг 1 — Новая структура `ModuleSigTable`
Добавить в `imports.rs` / новый файл `sig_table.rs`:
```rust
pub struct FnSig { pub name: String, pub params: Vec<ParamSig>, pub ret: TypeRef, pub effects: Vec<String> }
pub struct ModuleSignatures { pub types: Vec<TypeDecl>, pub fns: Vec<FnSig>, pub consts: Vec<ConstDecl>, pub module_name: Vec<String> }
pub struct ModuleSigTable { pub modules: HashMap<Vec<String>, ModuleSignatures> }
```

### Шаг 2 — `collect_all_signatures(entry_path, ...) → ModuleSigTable`
Новая функция: рекурсивно обходит все транзитивные импорты entry, парсит каждый файл, собирает только сигнатуры (не тела). Использует тот же visited/in_progress guard что уже есть.

### Шаг 3 — Lazy body resolution
Модифицировать typecheck в `types/mod.rs`:
- `TypeCheckCtx` получает `sig_table: &ModuleSigTable`
- При резолве вызова функции из другого модуля — смотрит в `sig_table` для типов параметров/возврата
- Тело не читается пока не нужно для inline/специализации

### Шаг 4 — Интеграция + backward compat
- `resolve_imports_inline` остаётся публичным API, внутри вызывает collect_all_signatures + resolve_bodies
- Семантика для пользователя не меняется — это внутренний рефакторинг
- Существующие тесты должны пройти без изменений

## Тесты

**POS:**
- `f1_cycle_lazy_pos.nv` — цикл A→B→A где тело A использует тип из B, тело B — тип из A → компилируется
- `f2_cross_module_fn_call.nv` — вызов функции из другого модуля работает через sig_table
- `f3_sig_table_populated.nv` — все типы/функции из транзитивных импортов видны

**NEG:**
- `f4_unknown_fn_neg.nv` — вызов несуществующей функции → ошибка (sig_table не находит)

**Регресс:** полный `nova test` — zero NEW FAIL. Замер компайл-тайма до/после.

## Критерии приёмки

- **A1.** Тела функций НЕ парсятся/резолвятся до обращения к ним.
- **A2.** Цикл где тело A зависит от тела B (не только сигнатуры) — компилируется без stack overflow.
- **A3.** Все существующие тесты PASS (zero regression).
- **A4.** Компайл-тайм замерен: стоимость collect-all-signatures не деградирует > 10% vs baseline.
- **A5.** `ModuleSigTable` — источник истины для cross-module типов; дублирования с inline-merge нет.
- **G0 (обязательный, «без упрощений как для прода»):** полный split реализован, не имитирован; источник истины — sig_table, не hardcode; backward compat 100%.

## Риски

- **Сложность:** `resolve_imports_inline` — ~700 строк с историей Plan 35/42/81/107/162. Изменение затрагивает всю цепочку.
- **Backward compat:** typecheck в `types/mod.rs` ожидает `module.items` уже merge'нутыми. Нужно либо сохранить inline-merge для items, либо рефакторить typecheck тоже.
- **Митигация:** делать incremental — сначала collect_signatures параллельно с существующим inline-merge (два прохода), потом постепенно убирать дублирование.

## Реализация (CLOSED 2026-06-16)

### Шаги и результат

**Шаг 1 — `ModuleSigTable` + новые структуры (`imports.rs`)**

Добавлены `FnSig`, `ModuleSignatures`, `ModuleSigTable` в `compiler-codegen/src/imports.rs`. `ModuleSigTable` хранит `HashMap<Vec<String>, ModuleSignatures>` — по ключу пути модуля сигнатуры типов + функций.

**Шаг 2 — `collect_all_signatures` + `TypeCheckCtx.sig_table`**

Функция `collect_all_signatures` рекурсивно обходит транзитивные импорты entry, парсит каждый файл, собирает только декларации (не тела). Использует тот же `visited/in_progress` guard из Plan 162 Ф.1. `TypeCheckCtx` получил поле `sig_table: ModuleSigTable`. Добавлен `build_with_sig_table()` конструктор. Дефолтный `build()` инициализирует пустую таблицу (`ModuleSigTable::new()`) — backward compat, нет I/O.

**Шаг 3 — Lazy helpers: `is_known_type` + `is_known_fn`**

`is_known_type(&self, name: &str) -> bool` и `is_known_fn(&self, name: &str) -> bool` на `TypeCheckCtx` — проверяют как локальные таблицы (`self.types` / `self.fn_decls`), так и `sig_table` через `find_type_modules` / `find_fn_modules`. `is_known_type` используется в `verify_impl_protocols` (убран `#[allow(dead_code)]`). `is_known_fn` размечен `#[allow(dead_code)]` для будущих вызовов (Plan 162.1 Ф.3+).

**Шаг 4 — Тесты 4/4 PASS, компайл-тайм overhead 1.5%**

- `f1_sig_table_known_pos` — тип из `sig_table` виден в `verify_impl_protocols`
- `f2_cycle_body_dep_pos` — тело A зависит от типа из B, тело B от типа из A — компилируется
- `f3_unknown_type_neg` — неизвестный тип → ошибка (sig_table не находит)
- `f4_transitive_sig_pos` — транзитивные сигнатуры видны

Overhead collect_all_signatures vs baseline: **~1.5%** (в рамках критерия A4 ≤10%).
