# Plan 83.7: `runnext` LIFO slot — cache-warm handler chains (Go/Tokio parity)

> **Создан:** 2026-05-24 (out of Plan 83 audit, [83-audit-2026-05-24.md](83-audit-2026-05-24.md) §6.3).
> **Статус:** 📋 proposed.
> **Приоритет:** P1.
> **Оценка:** 0.5 dev-day.
> **Родитель:** [Plan 83](83-mn-default-on-gomaxprocs.md).
> **Зависимости:** Plan 83.6 рекомендуется (но независим).

## 1. Проблема

Текущая Nova: каждый `spawn` идёт через Chase-Lev deque push. Для
hot-loop pattern:

```nv
loop {
    let msg = chan.recv()       // <- yields, current fiber parks
    let resp = process(msg)
    next_chan.send(resp)         // <- wakes another fiber on same worker
}
```

woken fiber пушится в deque tail; следующий `pop` (LIFO owner) возьмёт
**самый недавний** ctx, но если между wake и pop успели прийти другие
spawn'ы — отрабатываются они первыми, **разрушая cache-warmth** для
handler chain.

**Go runtime** имеет single-slot `runnext` priority queue per P:
- `newproc()` пишет goroutine в `runnext`, displacing previous → local
  runq.
- `runqget()` сначала проверяет `runnext`, потом local runq.
- Net effect: для chained patterns (`request → handler → response`)
  каждый next-step CPU-кэш warm.

**Tokio** делает то же через **LIFO slot** в worker.

**Эффект (industry-measured):** на server-loop workloads — ~30-50% latency
reduction. На non-chained workloads — neutral.

## 2. Дизайн

### 2.1 NovaWorker extension

```c
struct NovaWorker {
    /* ... existing fields ... */
    mco_coro* runnext;  /* atomic ptr — single-slot priority queue */
};
```

`runnext` — atomic pointer (CAS для thread-safety; main pattern same-thread
read-write, но steal может attempt CAS-grab).

### 2.2 Push priority

В `nova_runtime_dispatch_ready` (runtime.c:302-321) либо в `wake` path:

```c
static void _worker_dispatch_ready(void* ctx, mco_coro* co) {
    NovaWorker* w = (NovaWorker*)ctx;
    /* Plan 83.7: same-thread wake → runnext priority slot. */
    if (w == nova_current_worker()) {
        mco_coro* prev = nova_aptr_load(&w->runnext);
        if (nova_aptr_cas(&w->runnext, &prev, co)) {
            /* prev displaced — push в deque если был. */
            if (prev) nova_deque_push(&w->deque, prev);
            return;  /* no wake — owner already running */
        }
    }
    /* Slow path: push deque + async wake. */
    nova_mutex_lock(&w->wake_mu);
    nova_array_push(w->wake_pending, co);
    nova_mutex_unlock(&w->wake_mu);
    uv_async_send(&w->wake_handle);
}
```

### 2.3 Pop priority

В `_worker_main` step (2) (runtime.c:367+):

```c
/* Step 2: Owner LIFO pop — priority order:
 *   1. runnext (most recent wake, hot)
 *   2. local deque (FIFO/LIFO depending on push side)
 *   3. yielded queue (preempted CPU-fibers)
 */
mco_coro* co = NULL;

/* (2a) Plan 83.7 runnext priority. */
co = nova_aptr_exchange(&w->runnext, NULL);
if (!co) {
    /* (2b) Normal deque. */
    co = nova_deque_pop(&w->deque);
}
if (!co) {
    /* (2.5) yielded FIFO. */
    co = nova_array_pop_front(w->yielded);
}
```

### 2.4 Steal interaction (DESIGN DECISION)

Two options:

**Option A (Go-style):** steal-thief может **steal runnext** через CAS.
- Pros: balanced load on imbalanced workloads.
- Cons: defeats locality benefit.

**Option B (Tokio-style):** runnext **NOT stealable**, steal only из
local deque.
- Pros: max cache-warmth для owner.
- Cons: imbalanced worker может starve other workers.

**Nova: choose Option B** — simpler invariant + Nova-уникальная strength
= structured concurrency (cancellation handles imbalance via wake-all).
Document explicitly в D-block.

### 2.5 Shutdown drain

В `_worker_main` cleanup loop (runtime.c:623-649) — drain runnext тоже:
```c
mco_coro* co = nova_aptr_exchange(&w->runnext, NULL);
if (co) { ... drain ... }
```

## 3. Тесты

### Positive (новые):

**`bench/m_n/handler_chain_pingpong.nv`** — synthetic 2-fiber ping-pong:
```nv
module bench_m_n.handler_chain_pingpong

bench "10⁴ ping-pong same-worker latency" {
    use std::channels::*

    let (tx_a, rx_a) = chan[int].make()
    let (tx_b, rx_b) = chan[int].make()
    let n = 10_000

    measure {
        supervised {
            spawn {
                let mut i = 0
                while i < n {
                    tx_a.send(i)
                    let _v = rx_b.recv()
                    i = i + 1
                }
            }
            spawn {
                let mut i = 0
                while i < n {
                    let v = rx_a.recv()
                    tx_b.send(v)
                    i = i + 1
                }
            }
        }
    }
}
```

Acceptance: 30%+ latency reduction vs pre-83.7 baseline.

### Negative (новые):

**`nova_tests/plan83_7/runnext_displaced_to_deque.nv`** — verify second
wake displaces previous runnext в deque (no loss):
```nv
module plan83_7.runnext_displaced_to_deque

test "two consecutive wakes — both fibers eventually run" {
    use std::channels::*

    let (tx1, rx1) = chan[int].make()
    let (tx2, rx2) = chan[int].make()
    let mut counter = 0

    supervised {
        spawn {
            let _v = rx1.recv()
            counter += 1
        }
        spawn {
            let _v = rx2.recv()
            counter += 1
        }
        // Wake both back-to-back — second must NOT displace first into oblivion.
        tx1.send(1)
        tx2.send(2)
    }
    assert(counter == 2)
}
```

**`nova_tests/plan83_7/runnext_not_stealable.nv`** — verify Option B
semantic enforced (другой worker не steal'ит runnext):

## 4. Acceptance

### Core:
- ✅ runnext slot added в NovaWorker + dispatch_ready uses CAS push.
- ✅ Pop priority order: runnext → deque → yielded.
- ✅ Shutdown drain runnext.
- ✅ `handler_chain_pingpong.nv` — ≥30% latency reduction.
- ✅ Bootstrap + armed PASS не регрессирует.
- ✅ TSAN 0 races.

### Stretch:
- ≥50% latency reduction.

## 5. Риски

- **Imbalanced workloads** под Option B — workers с активным runnext
  могут starve idle peers. Mitigation: documentation + benchmark
  имбаланса (separate measurement).
- **runnext race** на CAS — single producer/consumer pattern (same-thread)
  если steal disabled. Verify CAS used правильно.
