# Plan 100.4.3: `okdefer` + reason-aware `defer |result|` form

> **Создан 2026-05-23 как sub-sub-plan [Plan 100.4 umbrella](100.4-cleanup-on-failure.md).**
> **Статус:** 📋 proposed. **GATED на [100.1](100.1-core-must-consume.md)**.
> **Оценка:** ~2 dev-day.
> **D-блок:** **D149**.

---

## Цель

Симметризовать defer-family — добавить `okdefer` (выполняется только
на success-path; complement к `errdefer`) + optional reason-aware
форму `defer |result| { ... }` для cleanup, который меняет поведение
в зависимости от exit-reason.

Текущая семья defer'ов:
- `defer { ... }` — все exit-paths.
- `errdefer { ... }` — только error-paths (throw/panic/interrupt).
  (+ Plan 100.4.5 расширяет на interrupt.)

Дыра:
- Нет complement-keyword'а «success-only». Для symmetric pattern
  `commit on success, rollback on error` нужен либо explicit
  `tx.commit()` в success-path, либо awkward inversion. **okdefer
  закрывает.**

Использование `okdefer`:

```nova
consume tx = begin()
errdefer { tx.rollback() }                  // на error → rollback
okdefer  { tx.commit() }                    // на success → commit
do_work()?                                   // если Err — errdefer
                                             // если OK — okdefer
                                             // tx covered на обоих paths
```

Чистая симметрия. Без okdefer:

```nova
consume tx = begin()
errdefer { tx.rollback() }
do_work()?
tx.commit()                                  // explicit success-path consume
```

Эквивалентно по поведению, но **asymmetric**: success-path использует
explicit-call, error-path — defer-family. okdefer выравнивает.

## Сравнение

| Capability | Go | Rust | TS (ES2024) | Kotlin | Nova target |
|---|---|---|---|---|---|
| Success-only cleanup distinction | ❌ | ❌ | ❌ | ❌ | ✅ **`okdefer`** (unique) |
| Reason-aware cleanup (доступ к Ok/Err в body) | ❌ | ❌ | ❌ | ⚠️ `try { } finally { val r = ... }` manual | ✅ **`defer \|result\| { ... }`** |
| Symmetric defer-family | ❌ | ❌ | ❌ | ❌ | ✅ defer + errdefer + okdefer |
| LIFO consistency | ⚠️ defer-only | n/a | n/a | n/a | ✅ all three composable |

Plan 100.4.3 даёт Nova **unique capabilities** среди GC-языков:
exhaustive cleanup-path distinction + reason-aware cleanup. Это
**уверенно лучше всех остальных** для resource-management ergonomics.

## Дизайн-решения

### D1 — `okdefer` keyword

Новый scope-level statement, mirror к `errdefer`:

```
statement = ...
          | 'defer'    body
          | 'errdefer' body
          | 'okdefer'  body                    // ← NEW
```

`okdefer body` — выполнится **только** при normal exit (end-of-scope
или `return expr` без error).

### D2 — Триггерные правила okdefer

Аналогично D90 §7-§8 для errdefer:

| Exit-path | `defer` | `errdefer` | `okdefer` |
|---|---|---|---|
| Normal end-of-scope | ✅ | ❌ | ✅ |
| `return expr` (без error) | ✅ | ❌ | ✅ |
| `throw err` / `expr?` / `expr!!` | ✅ | ✅ | ❌ |
| `panic(msg)` | ✅ | ✅ | ❌ |
| `interrupt v` (после 100.4.5 amend) | ✅ | ✅ | ❌ |
| `exit(code)` | ❌ | ❌ | ❌ |

okdefer + errdefer — exhaustive (один и только один из них срабатывает
при non-exit() exit'е). defer — orthogonal (всегда).

### D3 — LIFO consistency

Multi-defer-family LIFO:

```nova
defer A
okdefer B
errdefer C
okdefer D
defer E
```

Normal exit LIFO: `E → D → B → A` (defer + okdefer; errdefer skipped).
Error exit LIFO: `E → C → A` (defer + errdefer; okdefer skipped).

Каждое запускается ровно в порядке declaration (reversed).

### D4 — okdefer body — те же ограничения что defer/errdefer

okdefer body — D90 §4 (после 100.4.1 amend: Fail allowed + composition),
§5 (после 100.4.2 amend: suspend allowed), §6 (no top-level return/
throw/break — unchanged).

okdefer cleanup-fail на normal exit composes как обычная Fail в Plan
100.4.1.

### D5 — `defer |result| { ... }` — reason-aware form (optional)

Расширенная форма defer (только `defer`, не errdefer/okdefer — они и
так reason-narrow):

```nova
defer |result| {
    match result {
        Ok(value) => Log.info("success: ${value}"),
        Err(e)    => Log.error("failed: ${e}"),
    }
}
```

`result` — bind exit-reason:
- `Ok(value)` — normal exit с value (для function body — return value;
  для inner-scope — `()`).
- `Err(e)` — exit через `throw e` / `expr?`.
- `Panic(msg)` — exit через `panic(msg)` (option, если различение
  panic vs throw нужно).

`result` — read-only borrow на actual exit-reason; не модифицируется.

### D6 — Когда использовать что

| Need | Solution |
|---|---|
| Cleanup на любом exit'е (logging, metric) | `defer { ... }` |
| Rollback только на error | `errdefer { ... }` |
| Commit только на success | `okdefer { ... }` (после Plan 100.4.3) |
| Cleanup с разной логикой success vs error | `defer \|result\| { match result { ... } }` |

Programmer выбирает наиболее narrow form для clarity.

### D7 — Backward-compat

`defer` / `errdefer` поведение unchanged. `okdefer` — новое keyword,
existing код не затронут.

`defer |result| { ... }` — новая syntax-form, optional. Existing
`defer { ... }` (без `|result|`) — работает как раньше.

### D8 — Check_consume interaction (preview)

100.4.5 будет распознавать:
- `okdefer { tx.commit() }` cover'ит tx на success-path.
- `errdefer { tx.rollback() }` cover'ит tx на error-path.
- Если оба есть — tx covered на всех (non-exit) paths.

### Ф.2 — Lexer + parser + AST (~0.4 dev-day)

- Lexer: `KwOkdefer`.
- AST:
  - `Stmt::OkDefer(body)` — mirror к `Stmt::ErrDefer`.
  - `Stmt::DeferWithResult { result_pattern, body }` — для reason-aware
    form.
- Parser:
  - `okdefer body` parse mirror к `errdefer`.
  - `defer |pattern| body` parse — extends existing `defer`.
- Parse-time:
  - `okdefer` + `defer` на одной строке — отдельные statements (LIFO
    independent).
  - `errdefer` + `okdefer` для одного var — допустимо (exhaustive
    coverage).

### Ф.3 — Runtime: okdefer trigger (~0.4 dev-day)

- Runtime tracking «last exit-path» — error / success / panic.
- При scope-exit:
  - Walk defer-stack LIFO.
  - Для каждого entry: проверить тип (defer / errdefer / okdefer),
    matched ли с actual exit-path.
  - Execute matched.
- `okdefer` skipped on error-path.
- `errdefer` skipped on success-path.
- `defer` executed always.

### Ф.4 — `defer |result|` runtime + binding (~0.3 dev-day)

- Runtime: предоставить `result` value перед запуском body.
- Pattern-match in body — standard match arm-cases.
- `result` — value-type, immutable (read-only borrow по сути).

## Acceptance criteria

- [ ] `okdefer body` парсится; trigger только на normal exit/return.
- [ ] Mixed LIFO с defer/errdefer/okdefer — корректный порядок.
- [ ] `defer |result| { ... }` форма — result доступен внутри body.
- [ ] result patterns — `Ok(v)`, `Err(e)`, `Panic(m)` exhaustive.
- [ ] Backward-compat: existing defer/errdefer работает как раньше.
- [ ] okdefer body может Fail (Plan 100.4.1 integration).
- [ ] okdefer body может suspend (Plan 100.4.2 integration).
- [ ] 8 pos + 4 neg фикстур PASS.
- [ ] Полный `nova test` → 0 регрессий.
- [ ] Spec D149 published; cross-ref D90 family.

## Границы

- **Pattern-match exhaustiveness check** для `defer |result|` body —
  bootstrap: warning если non-exhaustive; future enhancement —
  compile-error.
- **`asyncdefer` / `tryDefer` другие variants** — не вводим; defer +
  errdefer + okdefer — exhaustive.

## Risks

1. **Keyword burden.** Ещё одно keyword — больше syntax для
   запоминания. **Митигация:** symmetric с errdefer; запоминается как
   ok/err family.
2. **Reason-aware form complexity** — pattern-match в defer body.
   **Митигация:** optional form; programmer не обязан использовать.
3. **LIFO с mixed family** — может удивлять программистов. **Митигация:**
   docs + spec D149 §«Mixed LIFO».
