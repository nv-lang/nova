## M1 — sizeof/align_of для Named user types

**Scope:** Extend `compiler-codegen/src/const_fn_eval.rs::type_size_or_align`
к user-defined records и sums:
- **Records** `type Point { x int, y int }` — C struct layout с natural
  alignment + tail-pad to max-field-align.
- **Tagged sums** `type Color { Red, Green, Blue }` — tag (i32 = 4 bytes
  align 4) + max variant size, alignment = max(4, max_variant_align).
- **Variant constructors** `type Result[T, E] { Ok(T), Err(E) }` — for
  concrete instantiations, same as sum но variants have fields.

**TypeDecl lookup:** Need access to module's TypeDecl registry. Add
parameter `type_decls: &HashMap<String, &TypeDecl>` to type_size_or_align
(или wrap в struct context). Caller sites updated to thread map through.

**Tests:** sizeof_named_record_ok / sizeof_named_sum_ok /
sizeof_recursive_record_ok (record-of-records).

## M2 — Parser disambiguation `|x|` after `Stmt::Const`

**Scope:** Parser amendment в `compiler-codegen/src/parser/mod.rs` —
после `Stmt::Const(name = expr)`, lookahead определяет: если next token
is `|` followed by Ident-list and `|`, parse as ClosureLight expression.
Иначе binary OR.

**Heuristic:** `|` at expression start position (Stmt-trailing) с pattern
`|<ident-list>|` — ClosureLight. Otherwise binary OR.

**Tests:** closure_outer_const_pipe_ok (uses `|x| body` syntax now works).

## M3 — HOF inference для closure-generic

**Scope:** Currently V4.5 Ф.4 requires explicit TurboFish: `make_id[int]()`.
M3 extends к HOF call sites без TurboFish:

```nova
fn make_id[T]() -> const fn(T) -> T => fn(x T) -> T => x
fn apply_i(f fn(int) -> int, x int) -> int => f(x)

test {
    ro r = apply_i(make_id(), 7)   // infer T=int from apply_i's f param
}
```

**Implementation:** Reuse Ф.3 `infer_generic_subst` + `GenericMutateCtx`
в const_fn_closure.rs — but specialized к closure-returning fns. Inference
matches host's `-> const fn(T) -> T` return type против callee's expected
arg type `fn(int) -> int`.

**Tests:** closure_generic_hof_inferred_ok.

## M4 — Nested generic concrete types

**Scope:** Currently V4.5 Ф.3 only accepts simple Named (e.g., `int`,
`str`) as concrete для generic param. M4 extends к:
- Named c generics: `Option[int]`, `Result[int, str]`
- Tuples: `(int, str)`
- Arrays: `[]int`, `[4]int`

**Mangling:** Extend mangle scheme от plain `int` к serialized form:
`Option_int`, `Result_int_str`, `tup_int_str`, `arr_int`, `fixarr4_int`.

**Tests:** trampoline_generic_tuple_concrete_ok / _option_concrete_ok.
