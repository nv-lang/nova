## Зачем

Plan 114.4.2 V1 закрыл базовый `const fn` (literals + arithmetic + as +
direct calls на другие const fn — without control flow, recursion, mixed
params, generic, first-class).

Plan 114.4.3 расширяет V1 surface до production-grade comptime fn
matching capabilities Rust `const fn` (with recursion) + Zig comptime
(mixed const/runtime params) + generic-aware:

**Без расширений:**
- Нельзя написать `const fn fact(const n int) -> const int =>
  if n <= 1 { 1 } else { n * fact(n - 1) }` — V1 запрещает if + recursion.
- Нельзя `fn alloc(const size int) -> []u8` (runtime return с const
  param — `[M-114.4.2-runtime-return]` который mixed-args покрывает).
- Нельзя `fn[T] sizeof[T]() -> const int` (generic const fn).
- Нельзя first-class refs: `ro f = my_const_fn` — V1 reject'ит.

**С Plan 114.4.3:**
```nova
// Recursion + control flow:
fn fact(const n int) -> const int =>
    if n <= 1 { 1 } else { n * fact(n - 1) }

// Mixed args (const + runtime):
fn make_buf(const size int, fill u8) -> []u8 {
    ro buf [size]u8 = ...
    for i in 0..size { buf[i] = fill }
    buf
}

// Generic const fn:
fn[T] sizeof() -> const int => @intrinsic_sizeof_of_T()

// First-class:
ro adder = make_adder(5)        // make_adder — const fn, возвращает closure
ro r = adder(3)                  // r == 8
```

## Дизайн

### Ф.1 — Control flow в const fn body

Расширяет body checker whitelist:
- `if` / `else` expressions: каждая ветвь проходит recursive whitelist.
- `match` expressions: scrutinee + каждый arm body — constexpr.
- Block expressions с разрешёнными stmt forms.
- `for`/`while`/`loop` — V2.1 (deferred, отдельный safety hatch);
  V2.0 only conditionals + match (без loops для простоты evaluator).

Evaluator extension:
- `eval_expr` для If: evaluate cond → bool; evaluate then/else branch.
- `eval_expr` для Match: evaluate scrutinee; match patterns; evaluate
  matching arm body.
- Pattern matching V2.0 subset:
  - Literal patterns (`42`, `"hello"`, `true`, `'a'`).
  - Wildcard `_`.
  - Ident-bind (`x` — binds к local env).
  - Tuple-deconstruct (`(a, b)`) — для tuple const values если когда-нибудь
    добавятся в ConstValue.

Errors:
- `E_CONST_FN_MATCH_EXHAUSTIVE` — match без exhaustive coverage.
- `E_CONST_FN_PATTERN_NOT_SUPPORTED` — unsupported pattern (V2.0
  only literals + wildcard + ident; record/sum patterns — V2.1).

**Loops (V2.1 deferred):** `for`/`while`/`loop` требуют bounded
termination proof — иначе evaluator может уйти в infinite loop.
V2.0 reject с suggested workaround «use recursion».

### Ф.2 — Recursion

Allowed: direct и mutual recursion с termination check.
- Body checker: убрать E_CONST_FN_RECURSION ошибку. Call-graph cycle
  detection остаётся (информационный warning) но не error.
- Evaluator extension:
  - Add recursion-depth threshold (default 256, configurable через
    `#fn_eval_max_depth(N)` attribute per fn).
  - Memoization уже есть (Plan 114.4.2 V1) — recursion с identical args
    кэширует mid-execution через memo.
  - При reaching depth-limit → `E_CONST_FN_EVAL_DEPTH_EXCEEDED` с
    suggestion increase limit или add memoization.
- Termination check (V2.0 best-effort):
  - Static check: at least одна conditional ветвь не recursive
    (если if-branch true → recurse, else-branch must return literal).
  - Без формального well-founded measure (out-of-scope; V3 если нужно).

Recursion stress-test fixture:
- `fact(20)` = 2432902008176640000 — verifies precise i64 arithmetic.
- `fib(30)` — verifies memoization speedup.

### Ф.3 — Mixed args (const + runtime)

Parser: убрать all-or-nothing check. Allow any mix:
```
fn f(const a int, b int, const c char) -> int    // mixed const + runtime
fn f(const a int) -> []u8                        // const-param + runtime-return
fn f(a int, const b int) -> const int            // mixed params, const return
```

Body checker:
- Const params: same V1 rules — used в const expressions.
- Runtime params: standard rules — used в runtime expressions.
- Return:
  - `-> const T` — return value evaluable at compile time. Body must
    produce constexpr expression based ONLY on const params (no runtime
    param refs в return).
  - `-> T` (без const) — runtime return. Mixed params + runtime body.

Monomorphization (codegen):
- Const-param values trigger specialisation: каждый unique const-arg
  combination → отдельная C fn `Nova_fn_{name}_{const_a}_{const_c}_...`.
- Runtime params pass через как обычно.
- Naming convention: `Nova_fn_{name}_cst_{hash(const_args)}`.
- Dead specialisations dropped per Plan 114.4.2 V1 codegen drop logic.

Errors:
- `E_CONST_FN_RUNTIME_REF_IN_CONST_RETURN` — if `-> const T` body
  references runtime parameter.
- `E_CONST_FN_RUNTIME_ARG_TO_CONST_PARAM` — call-site passes runtime
  value to const param (already covered by Plan 114.4.2 V1
  E_CONST_FN_NON_CONST_ARG; keep code).

### Ф.4 — Generic const fn

Allow `fn[T] foo(const a int) -> const int { ... }`:

Body checker:
- Generic param `T` visible в const fn body как opaque type.
- `sizeof[T]()` / `align_of[T]()` / `T.const_field` — intrinsics, V2.0
  reject (V3 follow-up).
- Body must NOT depend on T's structure (V2.0 — type-erased const fn).

Reality check: most generic const fn use cases involve `sizeof(T)` /
type metadata, which V2.0 doesn't provide. Useful fixed cases:
- `fn[T] id(const x int) -> const int => x` — trivially generic.
- `fn[T] pack(const a int, const b int) -> const int => a * 1000 + b`
  — generic for caller code reuse, doesn't depend on T.

Codegen: per-T monomorphization key (Plan 70.5 pipeline). Each `Box[int]`
+ `Box[str]` call site → separate specialisation if RHS depends on T,
single specialisation if T-independent.

Errors:
- `E_CONST_FN_GENERIC_NEEDS_T_REFLECTION` — body tries `sizeof[T]` /
  similar; V2.0 not supported. Suggest factor out non-generic const fn.

### Ф.5 — First-class const fn

V2.0 — limited first-class: const fn refs allowed в **const contexts only**:
- `const FN_REF = my_const_fn` — module-level const = fn ref.
- Pass to HOF expecting `fn(int, int) -> int` — V2.0 reject (would
  need runtime trampoline; complex).
- `ro f = my_const_fn; f(5)` — runtime call site. V2.0:
  - If callsite has const args known statically → evaluate, replace.
  - Else → require runtime fn (currently const fn без runtime body
    → reject).
- Treat `ro f = const_fn` as **deferred evaluation** — `f` is logical
  alias for const fn, calls через f equivalent direct call (compiler
  alias resolution).

V2.0 ABI decision: const fn НЕ имеют runtime symbol (Plan 114.4.2 V1
codegen drop). First-class V2.0 — name aliasing, не function pointer.
Real first-class через runtime symbol generation — V3 follow-up.

Errors:
- `E_CONST_FN_FIRST_CLASS_RUNTIME_HOF` — pass const fn to runtime HOF
  (`map(arr, my_const_fn)` где `map` ожидает runtime fn) — V2.0 reject.
