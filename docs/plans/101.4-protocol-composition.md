# Plan 101.4 — Protocol composition `use A, B` (closes D53 open question)

> **Sub-plan [Plan 101](101-receiver-generic-prefix.md) (master).**
> Введение `use A, B` composition syntax **внутри protocol body**.
> Single-protocol composition `use Reader` работает естественно без
> двусмысленности. **Literal-composition (anonymous) НЕ вводится**.
> Закрывает open question в [D53 §«Открытые вопросы»](../../spec/decisions/02-types.md#d53).
> **Status:** 📋 proposed 2026-05-24 (Ред. 5), не начат.
> **Eval:** ~1 dev-day.
> **GATED на:** ничего (orthogonal к 101.1/2/3).

## 1. Зачем

Сейчас protocol'ы не могут composeиться. Дублирование method-signatures:

```nova
type Reader protocol { read(buf []u8) -> int }
type Writer protocol { write(buf []u8) -> int }

// СЕЙЧАС нужно дублировать:
type ReadWriter protocol {
    read(buf []u8) -> int     // duplicate of Reader
    write(buf []u8) -> int    // duplicate of Writer
    close() -> ()
}
```

D53 open question: «Composition protocol'ов» — открыт. Решение через
`use A, B` keyword внутри `protocol { ... }` body.

## 2. Дизайн (Ред. 5)

### Synтакс — `use A, B` внутри блока

```nova
type Reader protocol { read(buf []u8) -> int }
type Writer protocol { write(buf []u8) -> int }

// Multi-composition:
type ReadWriter protocol {
    use Reader, Writer
    close() -> ()              // own method
}

// Single-composition (естественно, без ambiguity):
type ReadExt protocol {
    use Reader
    job() -> ()
}

// Pure composition без own methods:
type Streamable protocol {
    use Reader, Writer, Closeable
}

// Mix anywhere в block — own methods и composition в любом порядке:
type Complex protocol {
    init() -> ()
    use Reader
    helper() -> int
    use Writer
}

// Anonymous-composition в type-position (extension D53):
fn process(rw protocol { use Reader, Writer }) { ... }
fn process(rw protocol { use Reader, Writer; close() -> () }) { ... }
```

### Семантика

- `use A, B, C` внутри `protocol { ... }` — **flattens** все
  method-signatures из A, B, C в этот protocol.
- Resulting protocol method-set = union(A.methods, B.methods, C.methods, own_methods).
- Order независим — `use Reader, Writer` ≡ `use Writer, Reader`.
- Multiple `use`-statements аккумулируются:
  ```nova
  use A, B
  use C            // same effect как use A, B, C
  ```
- T satisfies `ReadWriter` ⟺ T satisfies все methods из union.

### Disambiguation

| Контекст | `protocol Name { ... }` означает |
|---|---|
| `type X protocol { ... }` | type-decl; body может содержать `use Foo, Bar` (composition) + signatures (`->`) |
| `protocol { ... }` в type-position | anonymous type; body может содержать `use Foo` + signatures |
| `let v = protocol Y { ... }` | literal; body содержит impls (`=>`); **`use Foo` запрещён** (см. ниже) |

### Literal-composition — **ОТВЕРГНУТО**

```nova
// ❌ ОТВЕРГНУТО:
let v = protocol Foo {
    use Reader              // error: E_LITERAL_COMPOSITION_NOT_ALLOWED
    read(buf) => impl1
    close() => impl2
}
// «`use` in protocol literal not allowed; declare a named protocol
//  composition first via `type Name protocol { use ... }` and instantiate it»
```

**Почему:** literal — это **value-construction** (impls). Composition —
**type-level operation**. Смешивать слои когнитивно нагружено.
Industry-aligned (Rust/Go/Java/Kotlin/Scala не разрешают anonymous-
composition в literals).

Workflow для use case'а:
```nova
type MyRW protocol { use Reader, Writer }
let v = protocol MyRW {
    read(buf)  => impl1
    write(buf) => impl2
}
```

### Differences vs D39 (record-embed)

| Aspect | D39 record `use name Type` | D53+ protocol `use Type` |
|---|---|---|
| Form | `use name Type` (field-form) | `use Type` (нет field) |
| Field-name | обязателен (`use account Account`) | НЕТ — protocol не runtime entity |
| Delegation | auto-proxy methods через `@account.method` | НЕТ — просто union sigs |
| Sub-typing | ❌ (D39 не subtype) | ✅ — implementor ReadWriter автоматически Reader, Writer |
| Storage | runtime field added | compile-time only |
| Multiple-embed | per-line | `use A, B, C` allowed (новое для 101.4) |

Same keyword (`use`), разная семантика — parser распознаёт по контексту
(record-body vs protocol-body) и по syntax (`use name Type` vs `use Type[, ...]`).

### Cycle detection

```nova
type A protocol { use B }
type B protocol { use A }     // ERROR: cyclic composition
type X protocol { use X }     // ERROR: self-composition
```

Compiler tracks composition-graph, error при cycle.

### Duplicate method handling

```nova
type Reader protocol { read() -> int; close() -> () }
type Writer protocol { write() -> int; close() -> () }   // оба имеют close()

type ReadWriter protocol {
    use Reader, Writer
    // ERROR: E_CONFLICTING_COMPOSITION
    // «method `close` declared in both Reader and Writer;
    //  override in body OR rename one»
}
```

Override syntax (override `close()` в ReadWriter body) — отдельная
фича, defer if нужно.

### Чем Nova лучше альтернатив

| Lang | Composition syntax | Где | Anonymous-in-literal? |
|---|---|---|---|
| Rust | `trait T: A + B` | before block | ❌ |
| Java | `interface T extends A, B` | extends-list | ❌ |
| Kotlin | `interface T : A, B` | colon-list | ❌ |
| Scala 3 | `trait T extends A, B` | before block | ❌ (через `new A with B`) |
| Go | `interface { A; B }` | embed inside | ❌ |
| TS | `interface T extends A, B` | extends | ✅ (`A & B` intersection types, но без literals) |
| **Nova post-101.4** | `protocol { use A, B }` | embed inside | ❌ (rejected) |

**Nova edge:**
1. **Single-protocol composition естественна** — `use Reader` clear,
   нет ambiguity (Rust/Java требуют `extends Reader` even с single).
2. **Comma-list compact** — `use A, B, C` vs Go `A; B; C`.
3. **Mix anywhere в block** — methods и `use` в любом порядке (Rust
   требует `: A + B` перед body, нельзя interleave).
4. **D39 keyword reuse** — same `use` для embed (record + protocol),
   semantic intuition consistent («include this stuff»).
5. **Literal-composition rejected** — снижает cognitive load.

## 3. Phases

### Ф.0 — Probe

`nova_tests/plan101_4/probe/`:
- `protocol_use_single_pre.nv` — `type X protocol { use Reader; method() -> () }` — current parser behavior.
- `protocol_use_multi_pre.nv` — `type X protocol { use Reader, Writer; ... }` — baseline.
- `protocol_literal_use_pre.nv` — `let v = protocol Y { use Reader; ... }` — baseline (likely accepts or weird).

### Ф.1 — Parser

**Acceptance:** parser принимает `use Type [, Type]*` statements внутри
`protocol { ... }` body. Также в anonymous protocol type-position.
**Rejects** в literal-position с loud error.

**Actions:**
1. Внутри `parse_protocol_body` — add hook на `use` keyword:
   - `use TypeName [, TypeName]*` → composition list.
   - `name(args) -> ret` → method signature (existing).
2. Literal-detector → error `E_LITERAL_COMPOSITION_NOT_ALLOWED` если
   `use` встречается в literal-body.
3. AST: `ProtocolMember = MethodSig | Compose(Vec<TypeRef>)`.

### Ф.2 — Type-checker

**Acceptance:**
- Composition resolved at type-check — flatten method-sets.
- Duplicate signature → error `E_CONFLICTING_COMPOSITION`.
- Cycle detection: build composition-graph, error на cycle.
- Self-composition → error.
- T satisfies composed-protocol ⟺ T has все methods из union.

### Ф.3 — Codegen

No runtime impact — composition compile-time only. Method-dispatch
routes через existing D72 dispatch.

### Ф.4 — Tests positive + negative

#### Positive (7)

- `protocol_use_single.nv` — `type ReadExt protocol { use Reader; job() -> () }`.
- `protocol_use_two_comma.nv` — `use Reader, Writer` + own method.
- `protocol_use_pure.nv` — pure composition без own methods.
- `protocol_use_mix_order.nv` — methods и `use` в любом порядке в block.
- `protocol_use_anon_type_pos.nv` — `fn process(rw protocol { use Reader, Writer })`.
- `protocol_use_satisfaction.nv` — type с read+write impl auto-satisfies composed protocol.
- `protocol_use_as_bound.nv` — `fn[T ReadWriter] T @process()` (101.2 integration).

#### Negative (5)

- `protocol_use_self_neg.nv` — `type A protocol { use A }` → error.
- `protocol_use_cycle_neg.nv` — A→B→A → error «cyclic composition».
- `protocol_use_duplicate_method_neg.nv` — `use Reader, Writer`
  оба имеют `close()` → error `E_CONFLICTING_COMPOSITION`.
- `protocol_use_record_neg.nv` — `use SomeRecord` где SomeRecord
  не protocol → error «embed in protocol requires protocol-type».
- `protocol_literal_use_neg.nv` — `let v = protocol Y { use Reader; ... }`
  → error `E_LITERAL_COMPOSITION_NOT_ALLOWED`.

## 4. Risks

| Risk | Mitigation |
|---|---|
| Method-name conflicts (Reader+Writer оба `close()`) | Loud error + suggest rename или override (override syntax — отдельная фича) |
| Cycle detection пропускает indirect (A→B→C→A) | Topological sort composition-graph |
| `use` keyword двойная семантика (D39 records vs 101.4 protocols) | Parser разлогает по контексту тривиально; doc cross-references |
| Literal-composition reject confuses users | D145 doc explicitly: «composition только в type-decl/type-pos, literals — named» |

## 6. Не входит

- Method override syntax в composition — отдельный план если нужно.
- Diamond composition (A embeds B, A embeds C, B and C embed D) —
  closed под cycle detection + union, никаких specials.
- Literal-anonymous-composition — отвергнуто (rejected by design, см. §2).
