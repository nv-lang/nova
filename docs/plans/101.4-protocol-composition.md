# Plan 101.4 — Protocol composition `protocol A + B` (closes D53 open question)

> **Sub-plan [Plan 101](101-receiver-generic-prefix.md) (master).**
> Введение **`+` composition syntax** в `type X protocol A + B { ... }`
> (type-decl) и `fn f(x protocol A + B)` (type-annotation).
> **Литерал-composition (`let v = protocol A + B { ... }`) НЕ
> вводится** — литералы создают instance только named protocol.
> Закрывает open question в [D53 §«Открытые вопросы»](../../spec/decisions/02-types.md#d53).
> Symmetric с multi-bound `[T A + B]` (Plan 101.3) — один `+` оператор.
> **Status:** 📋 proposed 2026-05-24 (Ред. 4), не начат.
> **Eval:** ~1 dev-day.
> **GATED на:** ничего (orthogonal к 101.1/2/3).

## 1. Зачем

Сейчас protocol'ы не могут composeиться. Дублирование method-signatures:

```nova
type Reader protocol { read(buf []u8) -> int }
type Writer protocol { write(buf []u8) -> int }

// СЕЙЧАС нужно дублировать:
type ReadWriter protocol {
    read(buf []u8) -> int     // duplicate of Reader
    write(buf []u8) -> int    // duplicate of Writer
    close() -> ()
}
```

D53 open question: «Composition protocol'ов» — открыт. Решение через
`+` syntax — параллель Rust `trait T: A + B`, Java `interface T extends A, B`,
Kotlin `interface T : A, B`, и symmetry с Nova D72 multi-bound `[T A + B]`.

## 2. Дизайн

### Synтакс — `+` композиция

```nova
type Reader protocol { read(buf []u8) -> int }
type Writer protocol { write(buf []u8) -> int }

// Named-composition (declaration):
type ReadWriter protocol Reader + Writer {
    close() -> ()   // own method
}

// Pure composition без own methods:
type Streamable protocol Reader + Writer + Closeable

// One-liner:
type Hash protocol Hashable + Eq + Ord
```

### Anonymous-composition в type-position

Существующий D53 §628 разрешает anonymous protocol в type-pos:
```nova
fn close_all(items []protocol { close() -> () }) { ... }
```

Plan 101.4 расширяет на composition:
```nova
fn process(rw protocol Reader + Writer) { ... }
fn process(rw protocol Reader + Writer { close() -> () }) { ... }   // composition + extra method
```

### Literal-composition — **ОТВЕРГНУТО**

```nova
// НЕ работает:
let v = protocol Reader + Writer {
    read(buf) => impl1
    write(buf) => impl2
}
// ERROR: E_LITERAL_ANONYMOUS_COMPOSITION
// «anonymous protocol composition not allowed in literal;
//  declare a named protocol first via `type Name protocol A + B`»
```

**Почему отвергнуто:** literal-composition смешивает type-construction
(`A + B` — intersection of protocols) и value-construction (`{ ... }` —
impl body). Когнитивно нагружено. Industry-aligned (Rust/Go/Java/Kotlin/Scala
не разрешают anonymous-composition в literals). Use case покрывается
extract'ом в named type:

```nova
// Workflow: декларируем named composition, используем literal:
type MyRW protocol Reader + Writer
let v = protocol MyRW {
    read(buf) => impl1
    write(buf) => impl2
}
process(v)
```

### Семантика composition

- `protocol A + B + C` = intersection-protocol с union method-set из A, B, C.
- T satisfies `protocol A + B` ⟺ T satisfies A AND B.
- Resulting protocol = union всех signatures + own methods (в block body).
- Duplicate signature (Reader и Writer оба декларируют `read()`) → error
  «conflicting composition: `read` declared in both A and B».

### Cycle detection

```nova
type A protocol B + ... { ... }
type B protocol A + ... { ... }     // ERROR: cyclic composition
type X protocol X { ... }            // ERROR: self-composition
```

Compiler tracks composition-graph, error при cycle.

### Disambiguation: `+` vs `{ ... }`

Parser: после `type Name protocol`:
- `{ ... }` immediately → no composition, body содержит signatures.
- `IDENT (+ IDENT)*` then `{ ... }` или EOL → composition (+ optional body).

### Связь с 101.3 multi-bound

`[T ReadWriter]` ≡ `[T Reader + Writer]` (101.3). Два path к той же цели:
- 101.3 — **ad-hoc** intersection в use-site (`fn f[T Reader + Writer]`).
- 101.4 — **named** composition (`type ReadWriter protocol Reader + Writer`).

Оба полезны: ad-hoc когда reuse не нужен; named — когда reuse часто.
**Symmetric `+`** — single operator для intersection в обоих contexts.

### Differences vs D39 (record-embed)

| Aspect | D39 record `use name Type` | D53+ protocol `+` composition |
|---|---|---|
| Syntax | `use name Type` (field-form) | `protocol A + B` (operator form) |
| Field-name | обязателен | НЕТ field — protocol не runtime entity |
| Delegation | auto-proxy via `@name.method` | НЕТ delegation — просто union sigs |
| Sub-typing | ❌ (D39 не subtype) | ✅ — implementor ReadWriter автоматически Reader, Writer |
| Storage | runtime field added | compile-time only |
| Keyword reuse | `use` (existing) | `+` (existing в D72/101.3) |

Разный keyword — разная семантика. Не смешиваются.

### Чем Nova лучше альтернатив

| Lang | Composition syntax | Где | Anonymous-in-type? |
|---|---|---|---|
| Rust | `trait T: A + B` | before block | ❌ (`dyn A + B` существует, но decl нужен named) |
| Java | `interface T extends A, B` | before block | ❌ |
| Kotlin | `interface T : A, B` | colon-list | ❌ |
| Scala 3 | `trait T extends A, B` | before block | ✅ (`A with B`) |
| Go | `interface { A; B }` | embed inside | ❌ |
| TS | `interface T extends A, B` | extends | ✅ (`A & B` intersection types) |
| **Nova post-101.4** | `protocol A + B` | before block (decl) + type-pos | ✅ (type-pos) |

**Nova edge:**
1. **Symmetric `+` с multi-bound** (`[T A + B]` и `protocol A + B`) — один оператор, единая семантика.
2. **Industry-standard «composition before block»** — familiar Rust/Java/TS/Kotlin/Scala.
3. **Anonymous composition в type-position** — `fn f(x protocol A + B)` — parallel TS `A & B`.
4. **Literal-composition отвергнут** — снижает cognitive load (literals только named).

## 3. Phases

### Ф.0 — Probe

`nova_tests/plan101_4/probe/`:
- `protocol_composition_decl_pre.nv` — `type RW protocol Reader + Writer { close() }` — current parser behavior (likely parse error).
- `protocol_composition_anon_pre.nv` — `fn f(x protocol Reader + Writer)` — likely parse error.
- `protocol_composition_literal_pre.nv` — `let v = protocol Reader + Writer { ... }` — current behavior baseline.

### Ф.1 — Parser

**Acceptance:** parser принимает `protocol A + B + C { ... }`-форму в:
- type-decl: `type X protocol A + B { ... }`.
- type-pos (annotations): `fn f(x protocol A + B) { ... }`.

И **rejects** в literal-pos с loud error.

**Actions:**
1. Extend `parse_protocol_kind_tail()` — после `protocol` keyword optionally parse `Type (+ Type)*` chain перед `{`.
2. Literal-detector (по context — `let x = protocol ...` is literal) → error
   `E_LITERAL_ANONYMOUS_COMPOSITION` если composition присутствует.
3. AST: `ProtocolDecl { composition: Vec<TypeRef>, body: Vec<ProtocolMember> }`.

### Ф.2 — Type-checker

**Acceptance:**
- Composition resolved at type-check — Reader's methods flattened в ReadWriter.
- Duplicate signature → error «conflicting composition: `read` declared in both A and B».
- Cycle detection: build composition-graph, error на cycle.
- Self-composition → error.
- T satisfies `protocol A + B` ⟺ T has все methods из union.

### Ф.3 — Codegen

No runtime impact — composition compile-time only. Method-dispatch на
T:ReadWriter routes через existing D72 dispatch.

## 4. Risks

| Risk | Mitigation |
|---|---|
| Method-name conflicts при composition (Reader и Writer оба имеют `close()`) | Loud error + suggest rename. Override syntax — отдельная фича, defer |
| Cycle detection пропускает indirect (A→B→C→A) | Topological sort composition-graph |
| Literal-composition reject confuses users expecting symmetry | D142/D145 doc explicitly: «composition только в type-context» |
| `+` collision с future arithmetic в type-context | Невозможна — type-context не has number literals |

## 6. Не входит

- Method override syntax в composition — отдельный план если нужно.
- Diamond composition (A composes B, A composes C, B and C compose D) — closed под cycle detection + union, никаких specials.
- Literal-anonymous-composition — отвергнуто (rejected by design, см. §2).
