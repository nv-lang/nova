# Plan 101.4 — Protocol composition `use Foo` (closes D53 open question)

> **Sub-plan [Plan 101](101-receiver-generic-prefix.md) (master).**
> Введение embed protocols через `use TypeName` keyword внутри
> `protocol { ... }` body. Параллель [D39](../../spec/decisions/02-types.md#d39)
> record-embed. Закрывает open question в
> [D53 §«Открытые вопросы»](../../spec/decisions/02-types.md#d53)
> («Composition protocol'ов»).
> **Status:** 📋 proposed 2026-05-24, не начат.
> **Eval:** ~1 dev-day.
> **GATED на:** ничего (independent — orthogonal к 101.1/2/3).

## 1. Зачем

Сейчас protocol'ы не могут embed'ить друг друга:

```nova
type Reader protocol { read(buf []u8) -> int }
type Writer protocol { write(buf []u8) -> int }

// СЕЙЧАС: нет способа объявить ReadWriter as composition
// Приходится дублировать:
type ReadWriter protocol {
    read(buf []u8) -> int     // duplicate of Reader
    write(buf []u8) -> int    // duplicate of Writer
}
```

D53 open question: «Composition protocol'ов (`Foo : Bar`) — открытый
вопрос». Параллель — Go interface composition:

```go
type ReadWriter interface {
    Reader    // embed
    Writer    // embed
}
```

В Nova D39 уже есть `use Type` для record-embed. Логично переиспользовать
тот же keyword в protocol context.

## 2. Дизайн

### Synтакс

```nova
type ReadWriter protocol {
    use Reader        // embed Reader (all methods included)
    use Writer        // embed Writer
    close() -> ()     // own method
}
```

### Семантика

- `use Reader` внутри `protocol { ... }` — **flattens** Reader's
  method-signatures в ReadWriter.
- Resulting protocol = union всех methods из embedded + own methods.
- Дубликат метода в embed'ах → error (требует override or rename).
- Type satisfies ReadWriter ⟺ satisfies все methods union'а
  (auto-implies Reader + Writer).

### Differences vs D39 (record-embed)

| Aspect | D39 record `use name Type` | D53+ protocol `use Type` |
|---|---|---|
| Field-name | обязателен (`use account Account`) | НЕТ field — protocol не runtime entity |
| Delegation | auto-proxy methods через `@account.method` | НЕТ delegation — просто union sigs |
| Sub-typing | ❌ (D39 explicitly не subtype) | ✅ — implementor ReadWriter автоматически Reader, Writer |
| Storage | runtime field added | compile-time only |

Disambiguation в protocol body parser:
- `use TypeName` → embed (parallel D39).
- `name(args) -> ret` → method signature (БЕЗ `@` per D53).
- Conflict (single line с обоими токенами) — error.

### Связь с 101.3 multi-bound

`[T ReadWriter]` ≡ `[T Reader + Writer]` (если 101.3 done). Два пути
to same goal:
- 101.3 — **ad-hoc** intersection в use-site (`[T A + B]`).
- 101.4 — **named** composition (`type ReadWriter protocol { use A; use B }`).

Оба полезны: ad-hoc когда reuse не нужен; named — когда reuse часто.

### Cycle detection

```nova
type A protocol { use B }
type B protocol { use A }   // ERROR: cyclic embed
```

Compiler tracks embed-graph, error при cycle.

### Self-embed

```nova
type A protocol { use A }   // ERROR: self-embed
```

## 3. Phases

### Ф.0 — Probe

`nova_tests/plan101_4/probe/`:
- `protocol_use_pre.nv` — `type RW protocol { use Reader; use Writer }`
  — current parser behavior (likely parse error or silent treat as
  method-decl wrong).

### Ф.1 — Parser

**Acceptance:** parser distinguishes `use TypeName` (embed) от
`name(...) -> ret` (method) внутри `protocol { ... }`.

**Actions:**
1. Внутри `parse_protocol_body` — add hook на `use` keyword.
2. AST: `ProtocolBody = Vec<ProtocolMember>` где `ProtocolMember =
   MethodSig | Embed(TypeRef)`.
3. Parse errors:
   - `use field_name Type` (field-form) — error «protocol embed doesn't take field-name».
   - `use 42` (non-type after `use`) — parse error.

### Ф.2 — Type-checker

**Acceptance:**
- Embed resolved at type-check — Reader's methods flattened in ReadWriter.
- Duplicate signature (Reader and Writer оба have `read()`) → error
  «conflicting embed: `read` declared в both A and B; override or rename».
- Cycle detection: build embed-graph, error на cycle.
- Self-embed → error.
- T satisfies ReadWriter ⟺ T has все методы из union — used in
  bound checking.

### Ф.3 — Codegen

No runtime impact — protocol composition compile-time only. Method-
dispatch на T:ReadWriter routes как для T:Reader OR T:Writer per
called method (existing D72 dispatch).

### Ф.4 — Tests positive + negative

#### Positive (6)

- `protocol_single_embed.nv` — `type RW protocol { use Reader }` (degenerate, single embed).
- `protocol_two_embed.nv` — classic Reader + Writer → ReadWriter.
- `protocol_embed_plus_own.nv` — embed Reader + own `close()` method.
- `protocol_nested_embed.nv` — `A` embeds `B`, `B` embeds `C` (transitivity).
- `protocol_embed_used_as_bound.nv` — `fn[T ReadWriter] T @process()` (101.2 integration).
- `protocol_embed_satisfaction.nv` — type with `read` and `write` impl auto-satisfies ReadWriter.

#### Negative (4)

- `protocol_self_embed_neg.nv` — `type A protocol { use A }` → error.
- `protocol_cycle_embed_neg.nv` — A→B→A → error «cyclic embed».
- `protocol_duplicate_method_neg.nv` — Reader and Writer оба декларируют `read()` → error.
- `protocol_embed_record_neg.nv` — `type A protocol { use SomeRecord }` (SomeRecord не protocol) → error «embed in protocol requires another protocol type».

#### Acceptance Ф.4

- [ ] 6/6 positive PASS.
- [ ] 4/4 negative → expected error.

## 4. Risks

| Risk | Mitigation |
|---|---|
| Method-name conflicts при embed (Reader и Writer оба имеют `close()`) | Loud error + suggest rename или override (override syntax — отдельная фича, defer if нет) |
| Cycle detection пропускает indirect cycles (A→B→C→A) | Topological sort embed-graph, comprehensive cycle detection |
| Programmers ожидают что embed = subtype в Go sense (interface satisfies super) | D-block explicitly: «protocol embed = method union, structural satisfaction» |

## 5. Acceptance overall

- [ ] 6 pos + 4 neg PASS.
- [ ] D53 open question closed.
- [ ] D-block D147 (или D53 amend).
- [ ] Полный nova test 0 регрессий.

## 6. Не входит

- Method override syntax в protocol composition — отдельный план если нужно.
- Diamond inheritance (A embeds B, A embeds C, B and C both embed D) — closed под cycle detection + union, никаких specials.
