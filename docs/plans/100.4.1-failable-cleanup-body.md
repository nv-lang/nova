# Plan 100.4.1: failable cleanup body — defer/errdefer с `Fail` effect

> **Создан 2026-05-23 как sub-sub-plan [Plan 100.4 umbrella](100.4-cleanup-on-failure.md).**
> **Статус:** 📋 proposed. **GATED на [Plan 100.1](100.1-core-must-consume.md)**.
> **Оценка:** ~3 dev-day.
> **D-блок:** **D147** (amend D90 §4).

---

## Цель

Снять ограничение D90 §4 «defer body INFALLIBLE» — production cleanup
для consume-типов (commit/rollback/close/release) часто fail'абельный.
Реализовать **production-grade composition cleanup-error'а с
propagating error'ом** через Plan 49 multi-error infra.

Текущее (D90 §4):
```nova
defer { tx.commit() }                       // ❌ parse error если commit имеет Fail effect
defer {
    with Fail = handler { fail(e) { Log.error(e); interrupt () } } {
        tx.commit()                          // ✅ verbose 6-line handler-wrap
    }
}
```

Целевое (после Plan 100.4.1):
```nova
defer { tx.commit() }                       // ✅ Fail допустим
                                            //    cleanup-fail → composite-error с propagating
```

## Сравнение с другими языками

| Capability | Go | Rust | TS (ES2024) | Kotlin | Java | Nova target |
|---|---|---|---|---|---|---|
| Cleanup body может fail | ✅ (return err) | ❌ panic-in-Drop = abort | ✅ Symbol.dispose throws → SuppressedError | ⚠️ throws | ✅ AutoCloseable.close throws → addSuppressed | ✅ **Plan 49 composition** |
| Error composition при cleanup-fail-mid-error | ⚠️ manual | ❌ abort | ✅ SuppressedError chain | ⚠️ manual | ✅ Throwable.addSuppressed | ✅ **Plan 49 multi-error tree** |
| Visibility cleanup-fail в сигнатуре | ⚠️ method-by-method | n/a (drop ↛ Result) | ⚠️ via TypeScript types | ⚠️ checked exceptions | ✅ throws-list | ✅ **`Fail[E]` effect explicit** |
| Compile-time enforcement cleanup-fail handling | ❌ | n/a | ❌ runtime | ❌ checked-exc only-some | ✅ checked exceptions | ✅ **Fail effect always tracked** |

Nova **matches Java/TS** на composition; **превосходит** Rust (нет
double-panic-abort); **превосходит** Java/TS на «cleanup-fail видно из
сигнатуры всегда» (effect-typed везде, не optional).

## Дизайн-решения

### D1 — Fail effect в defer body допустим

D90 §4 amended:

```
БЫЛО: defer body не должно иметь Fail effect.
СТАЛО: defer body может иметь Fail effect; ошибка композируется
       с propagating error через Plan 49 multi-error.
```

### D2 — Composition rules (Plan 49 multi-error tree)

3 сценария:

**Сценарий A: defer-fail на normal exit (нет propagating)**

```nova
fn process() Fail[Err] -> () {
    consume tx = begin()
    defer { tx.commit() }       // commit может Fail
    do_work()                    // success
}
// На exit: defer срабатывает; если commit fail'ит — throw как обычно.
// Caller получает Fail из defer'а.
```

**Сценарий B: defer-fail во время error-propagation**

```nova
fn process() Fail[Err] -> () {
    consume tx = begin()
    defer { tx.commit() }       // commit may fail
    do_work_that_fails()?        // throws Err1
    // defer fires during unwinding:
    //   commit fails with CommitErr → composite error
    //   Plan 49 multi-error: { primary: Err1, suppressed: [CommitErr] }
}
// Caller получает primary=Err1 + suppressed=[CommitErr]
```

**Сценарий C: multiple defers, each can fail**

```nova
fn process() Fail[Err] -> () {
    consume tx1 = begin()
    consume tx2 = begin()
    defer { tx2.commit() }      // если tx2 fail
    defer { tx1.commit() }      // если tx1 fail
    do_work()?                    // throws Err1
    // LIFO unwinding:
    //   tx2.commit() runs — может fail Err2
    //   tx1.commit() runs — может fail Err3 (продолжает несмотря на Err2)
    //   composite: { primary: Err1, suppressed: [Err2, Err3] }
}
```

Multi-defer error accumulation — **Plan 100.4.4** (LIFO продолжает
несмотря на partial-failure).

### D3 — Composition primitive: Plan 49 `MultiError` (или эквивалент)

Plan 49 определяет:
- `nova_throw_typed_<E>` — типизированный throw.
- `error_user_payload` + per-E vtable для composition.
- **(новое в 100.4.1)** функция `nova_compose_suppressed(primary,
  suppressed)` — добавляет `suppressed` в primary's suppressed-chain.

API уровня языка:
```nova
type MultiError {
    primary: Err,
    suppressed: []Err,            // chain, LIFO order suppression
}

fn MultiError @primary() -> Err
fn MultiError @suppressed() -> []Err
```

User не пишет MultiError напрямую — это происходит автоматически при
defer-fail-during-propagation. Caller получает thrown value, может
inspect через `.primary()` / `.suppressed()`.

### D4 — defer-fail-on-normal-exit семантика

Если defer fail'ит при normal exit (Scenario A), что caller получает?

- Function declared `Fail[Err]` — caller получает `Fail` от defer
  cleanup'а.
- Function NOT declared `Fail[E]` — **compile error**: «defer body
  has Fail effect; enclosing function must declare Fail[E]».

Это force'ит programmer'а явно объявлять Fail в сигнатуре функции,
если defer может fail. **Compile-time visibility**.

### D5 — Type-checker: defer body Fail propagation

Effect-checker (Plan 49 infrastructure):
- Walking defer body — собирает Fail-effects всех expression'ов.
- Если есть Fail — enclosing function-sig обязан иметь `Fail[E]`.
- Иначе error E (D147-defer-fail-not-in-sig).

### D6 — Async-friendly composition

Plan 49 уже работает с async-fiber'ами (cancel-throw-routing). 100.4.1
никаких новых async-mechanism'ов не вводит; defer-fail composes
одинаково в sync и async контекстах.

### D7 — Diagnostic / AI-first ergonomics

Сообщение при composite-error:

```
error: composite error during scope exit
  primary error:    Err1 ("operation failed")
    at do_work_that_fails (line 12)
  suppressed during cleanup:
    [1] CommitErr ("network timeout")
        at tx.commit() in defer (line 14)
    [2] Err3 ("disk full")
        at tx1.commit() in defer (line 13)
```

Чётко: primary видно сразу, suppressed — chain ниже. Programmer
понимает, что primary — root cause, suppressed — cascaded cleanup-
failures.

- Defer-firing logic в runtime:

## Risks

1. **Backward-compat** existing handler-wrap code — продолжает
   работать (нет migration); просто становится opt-in shorthand.
2. **Plan 49 infrastructure** должен поддерживать composition primitive.
   **Митигация:** Ф.3 проверяет; добавляет `nv_compose_suppressed`
   если нет.
3. **Programmer не ожидает Fail из defer**. **Митигация:** compile
   error D147-defer-fail-not-in-sig force'ит explicit Fail в fn-sig;
   диагностика подробная.
4. **MultiError API ergonomics** — chain inspection может быть длинной.
   **Митигация:** AI-first diagnostic (Ф.7) выдаёт structured listing.
