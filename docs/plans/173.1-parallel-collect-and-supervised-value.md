# Plan 173.1 — `parallel for → []T` (сбор через канал+consume) + `supervised` как значение

> **Под-план [Plan 173](173-error-system-unify-harden.md)** (Ф.3 → семейство 173.0–173.3). **Создан:** 2026-06-21.
> **Статус:** 📋 PROPOSED (дизайн зафиксирован в диалоге 2026-06-21).
> **Запуск:** «**выполни план 173.1**». **Зависит от:** [173.0](173.0-concurrency-runtime-substrate.md) (рантайм-субстрат),
> [173.3](173.3-data-race-freedom-share.md) (consume-в-spawn + `#share` для безопасности захвата).
> **Сквозной критерий:** «без упрощений, как для прода».

## 1. Зачем

Сейчас `parallel for → []T` (D71) работает только для `T ∈ {int,bool,f64,str}` и `iter ∈ {Range, array-lit, array-ident}`;
запись-результат тайпчекается как `[]T`, но codegen молча деградирует в `unit` → утечка сырого C-error
(`[M-parfor-record-result-miscompile]`, README `check_all → []HealthStatus` не собирается). И `supervised { … }`
возвращает `unit` (bootstrap-заглушка) — trailing-выражение отбрасывается.

## 2. Решения (sign-off 2026-06-21)

1. **`supervised { … }` — value-expression** (снятие bootstrap-заглушки «возвращает unit»): возвращает своё
   trailing-выражение после join всех детей. `parallel for` = сахар над ним.
2. **`parallel for x in xs { f(x) }` → `[]T`** для **ЛЮБОГО** `T` — примитив · heap-record · value/stack-record
   (`type X value`, D228) · **анонимный** tuple · **именованный** tuple (D215) · sum-тип · вложенный `[]T` — и
   **любого итератора** (без `len()`). **Обязательно, без исключений** (закрывает `[M-parfor-record-result-miscompile]`;
   это **НЕ отложенный followup**, а ядро Ф.2 и критерий приёмки). Без keyword'а `collect` — поведение по умолчанию.
   Номер — через `xs.enumerate()`. Канал несёт `T` единообразно (value-типы — копией, heap — ссылкой); codegen один на все виды.
3. **Сбор — через канал, дренаж ВНУТРИ supervised, completion order, плотный** (никаких индексных слотов,
   никакого concurrent-push в общий буфер). Десугар через **`consume`** (ядро Plan 173):

```nova
ro pages = parallel for url in urls { fetch(url) }
≡
supervised {                                  // ← возвращает значение
    consume (tx, rx) = Channel.new(K)         // K — небольшой буфер (back-pressure), НЕ N
    consume tx {                              // adopt: родительский tx закрыт в конце ЭТОГО блока
        for url in urls {
            spawn consume c = tx.clone() {    // ЯВНО: клон создан в родителе (момент spawn → refcount++)
                c.send(fetch(url))            //   и MOVE'нут в ребёнка; ребёнок владеет c, закрывает на своём выходе
            }                                 // (зеркало `consume c = e { … }`, scope владения = ребёнок; см. 173.3)
        }
    }                                         // родительский tx закрыт → refcount = N (клоны детей)
    consume rx {
        mut acc = []
        while Some(p) = rx.recv() { acc.push(p) }   // None, когда последний ребёнок закрыл клон
        acc                                          // ← значение supervised
    }
}
```

   - **Канал закрывается по refcount:** клон создаётся в родителе в момент `spawn` (учтён в счётчике **до**
     закрытия родительского tx), но **владение у ребёнка** → закрывается на выходе ребёнка. Последний закрытый
     клон → канал закрыт → дренаж завершается. Нет дедлока (дренаж конкурентен с детьми внутри scope).
   - **`Stop`** (супервизор, 173.2) → ребёнок не послал → результата нет → плотный, без дыр.
   - **`Escalate`** → отмена siblings + throw наверх; `acc` не возвращается.
   - **`tx.clone()` API:** новый owned-`Sender` на тот же канал, sender-refcount +1; close (consume-cleanup) −1;
     0 → канал закрыт → `recv()` → `None` (модель Rust `mpsc::Sender`). Новый хэндл обязателен — каждый ребёнок
     независимо владеет/закрывает свой.
   - **Размер буфера `K` (tuning):** дефолт — небольшая константа (back-pressure; дренаж конкурентен и дёшев →
     блокировки редки, пик памяти O(K)+результат). **Опц.-оптимизация:** если источник НЕ ленивый итератор и даёт
     `@len()` (массив/Vec) — взять буфер = `len` (нет блокировок; буфер O(N), но и результат O(N) — тот же порядок).
     Выигрыш маргинальный (дренаж дёшев) → деталь codegen/runtime-tuning, не блокер.
   - **API создания канала (🔲 открыто):** `Channel.new(cap)` vs зеркалить `[]T.with_capacity(n)`. **Не зеркалить
     вслепую:** у массива `with_capacity` — *мягкий* хинт пре-аллокации (длина растёт сверх `n`), у канала `cap` —
     *жёсткая* граница (`send` блокируется на full = back-pressure). Разная семантика → `with_capacity` на канале
     вводит в заблуждение. Рекомендация: явный bounded-API (`Channel.new(cap)` / `Channel.bounded(n)` /
     `Channel.unbounded()`, как Rust `sync_channel`/Go `make(chan,n)`). TBD.
4. **Альтернатива закрытия** (деталь реализации): вместо refcount — `supervised` сам закрывает канал по
   завершении последнего ребёнка (без клонов). Выбор refcount-vs-supervised-close — в реализации; семантика та же.

## 3. Фазы

- **Ф.1 — `supervised` value-expression:** parser/AST/codegen — trailing пробрасывается caller'у (вместо `(void)`).
  Тесты: `supervised { … v }` → v; вложенные; void-форма (нет trailing) → unit.
- **Ф.2 — channel+consume десугар `parallel for`:** клон-в-родителе + move-в-ребёнка (173.3 capability) + дренаж
  внутри + refcount-close. Снимает 4-примитивное ограничение и ограничение итераторов (закрывает
  `[M-parfor-record-result-miscompile]`).
- **Ф.3 — bounded fan-out:** буфер `K` + (опц.) cap живых детей через Semaphore (Plan 103.4) — память O(N) на
  результат, живых fiber'ов O(workers).
- **Ф.4 — Stop/Escalate-проводка** (совместно с 173.2): Stop → не шлёт; Escalate → отмена+throw.

## 5. Тесты (pos + neg)

- **pos** `nova_tests/err173_1/`: `parallel for` → `[]T` для **КАЖДОГО вида элемента** — примитив, heap-record,
  value/stack-record, анонимный tuple, именованный tuple, sum-тип, вложенный `[]T` (матрица типов, не один кейс);
  над **итератором** (без len); пустой вход; один элемент; `supervised { … v }` возвращает v; completion-order
  (все результаты присутствуют, порядок не гарантирован); Stop → элемент отсутствует (плотный); большой вход +
  bounded fan-out (живых fiber'ов ≤ cap).
- **neg:** бесконечный итератор без `.take(n)` → диагностик (нельзя собрать неограниченный поток в `[]T`);
  (через 173.3) небезопасный захват в теле → `E_CONCURRENT_MUT_CAPTURE`.

## 6. Критерии приёмки

1. `parallel for` собирает `[]T` для любого `T` и любого итератора; **дедлока нет** (дренаж внутри scope).
2. `[M-parfor-record-result-miscompile]` закрыт **на ВСЕХ видах элемента** (примитив / heap-record / value-record /
   анон-tuple / именованный-tuple / sum / вложенный `[]T`) — собирается, не C-error. **Обязательный критерий, без упрощений и исключений.**
3. `supervised` возвращает значение; старые parallel-for/supervised зелёные.
4. **Без упрощений.**
