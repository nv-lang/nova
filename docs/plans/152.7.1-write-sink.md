## Контекст и мотивация

До Plan 152.7.1 методы `Display` и `Debug` протоколов имели сигнатуры:

```nova
fn T @display(mut sb StringBuilder) -> ()
fn T @debug(mut sb StringBuilder) -> ()
```

Это создавало жёсткую зависимость: каждый тип, реализующий `Display`, должен был зависеть
от `StringBuilder` (конкретного типа из `std/runtime/string_builder.nv`). Следствия:

1. **Перекрёстные ссылки:** примитивные типы `int`/`f64`/`bool`/`str` в prelude тянут
   `StringBuilder` → лишний import-граф-вес.
2. **Нет direct-to-sink:** `print(x)` требует промежуточную str-аллокацию через
   `StringBuilder`, даже когда можно писать прямо в stdout/сеть/буфер.
3. **Расширяемость ограничена:** нельзя направить форматирование в произвольный sink.

Прецеденты:
- **Rust:** `fmt::Write` (`write_str` + `write_fmt`), `fmt::Formatter` инкапсулирует sink;
  `Display::fmt(&self, f: &mut Formatter)` — не в `String`.
- **Go:** `io.Writer` (`Write([]byte)`) — всё форматирование (`Fprintf`, `Fprintln`)
  принимает `io.Writer`.
- **Java:** `Appendable` (`append(CharSequence)`) — базовый sink для `StringBuilder`,
  `Writer`, `StringBuffer`.

Nova теперь идёт по тому же пути: вводится протокол `Write` как тонкий sink-интерфейс.

## D258 AMEND — Write-sink протокол

### Write-протокол (core)

```nova
protocol Write {
    mut @write_str(s str) -> ()
}
```

**Семантика:**
- `write_str(s)` — записать строку `s` в sink (без возврата значения, `()`).
- Реализующий тип может аккумулировать (`StringBuilder`), делегировать в буфер
  (`WriteBuffer`) или писать прямо в поток (future: stdout, file, network).
- `mut` ресивер — sink обновляет внутреннее состояние.

### Display и Debug (обновлённые сигнатуры)

```nova
protocol Display {
    @display(mut w Write) -> ()
}

protocol Debug {
    @debug(mut w Write) -> ()
}
```

Параметр `sb StringBuilder` заменён на `w Write` во всех реализациях.

### StringBuilder.write_str

```nova
fn StringBuilder @write_str(s str) -> () {
    @buf.append(s)
}
```

`StringBuilder` реализует `Write` делегируя в `@buf.append(s)` (внутренний буфер).
Таким образом, все существующие `x.display(sb)` продолжают работать без изменений
на call-site: `sb StringBuilder` → `w Write` (StaticDispatch через codegen).

### Кодогенерация

В `compiler-codegen/src/codegen/emit_c.rs`:
- `type_ref_to_c`: `"Write"` → `"Nova_StringBuilder*"` (static monomorphization:
  `Write` в параметре `@display`/`@debug` всегда реализован `StringBuilder` на этапе
  интерполяции — `"${x}"` строит `StringBuilder` и передаёт его).
- `extract_protocol_type_name`: bypass для `Write` → не ошибается `E7201` при
  `w.write_str(...)` внутри тела Display/Debug метода.

**Статическая мономорфизация (ключевое решение):** протокол `Write` не использует
динамическую диспетчеризацию (vtable). Интерполяция `"${x}"` всегда монтирует
`StringBuilder` как единственный sink-тип — компилятор видит конкретный тип, а `Write`
в C-кодогене маппится прямо в `Nova_StringBuilder*`. Это даёт **нулевую overhead**
по сравнению с прямым `StringBuilder`-параметром — vtable отсутствует.

### auto_derive обновлён

`compiler-codegen/src/protocols/auto_derive.rs`: синтезированные `@display`/`@debug`
(авто-derive для struct/record) используют `w Write` параметр + `w.write_str(...)`.

## Критерии приёмки

**G0 (без упрощений, production-grade):** полная Write-sink миграция, не подмножество;
`@display`/`@debug` принимают `Write` — не `StringBuilder`-конкретный тип.
Прогон **релизными** `nova` + `nova-codegen`.

**G1 (Write-протокол определён):** `protocol Write { mut @write_str(s str) -> () }`
в `std/prelude/protocols.nv`. `StringBuilder` реализует `Write` через `@write_str`.

**G2 (Display/Debug мигрированы полностью):** все примитивные impl (`int`, `f64`,
`f32`, `bool`, `char`, `str`) + `Vec` + auto-derive синтезируемые методы используют
`w Write` параметр. Ни одного `sb StringBuilder` в protocol-impl файлах (кроме
внутренней реализации самого `StringBuilder.write_str`).

**G3 (Кодогенерация корректна):** `Write` в param-позиции C-маппится в
`Nova_StringBuilder*`; `w.write_str(s)` внутри Display/Debug тел не даёт CC-FAIL;
авто-derive `@display`/`@debug` компилируется.

**G4 (Интерполяция не сломана):** `"${x}"` для int/f64/bool/str/char → корректная
строка; `"${p}"` для пользовательского типа с `@display(mut w Write)` → то же что
`p.display(sb); sb.as_str()`. T8 (8 интерполяционных кейсов) PASS.

**G5 (Пользовательские типы работают):** тип с явным `fn T @display(mut w Write)`
корректно принимает `StringBuilder` как `w`, пишет через `w.write_str(...)`. T7 PASS.

**G6 (0 регрессий):** plan154_1 9/0, plan126_2 10/0, plan126 21/0, plan137 16/0,
plan91_14 14/0; plan152_7 t6/t7/t8 все PASS.
