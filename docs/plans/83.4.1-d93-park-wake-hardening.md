## 1. Bug

**Симптом** (`sleep_bench` / `sleep_precision_bench` / `sleep_real_clock`,
все под активным M:N-флипом):
```
nova: FATAL sleep wake before close_cb (stage=0) — D93 protocol bug
```

**Локация:** `compiler-codegen/nova_rt/fibers.h:1552-1557` (sleep-path)
+ `:1640+` (blocking-path, аналогичная защита).

**Контракт D93** (Plan 93 park/wake): wake может прийти ТОЛЬКО когда
`close_cb` уже отработал и handle полностью release'нут (`stage ==
CLOSED`). До этого момента fiber parked даже если `timer_cb` fired
(timer лишь инициирует close, сам wake не делает).

**Реальное поведение под M:N:** `timer_cb` запускает `close_cb`
**асинхронно**; main thread в drain-loop wake'нет fiber'а через
`pending_remote==0` ДО завершения `close_cb` (stage всё ещё `OPEN`).
Wake приходит «раньше» — fatal-check срабатывает, abort.

## 2. Root cause

`nova_sched_park(scope, slot)` возвращается, когда ЛИБО:
1. Получен явный wake через `nova_sched_wake`.
2. `pending_remote` достигла нуля И scope drained.

(2) под M:N может произойти ПОСЛЕ того как `timer_cb` уведомил
scheduler о готовности (через counter dec) но ДО того как `close_cb`
завершил unwind handle'а. Park возвращает control, fiber resume'ится,
check `stage == CLOSED` падает.

## 3. Подходы

### A. Loop park до явного `stage == CLOSED`

В `_nova_sleep_park` обернуть `nova_sched_park` в while:
```c
while (st.stage != NOVA_SLEEP_CLOSED) {
    nova_sched_park(scope, slot);
}
```
Каждый spurious-wake — re-park. Простая защита; цена — лишние
park/wake-циклы при misordered wake.

### B. Park-condition в API

Расширить `nova_sched_park` сигнатурой `(scope, slot, condition_fn,
ctx)` — park возвращается ТОЛЬКО когда condition_fn вернёт true.
Атомарный check внутри scheduler'а под лок. Чище, но требует API-
изменения и review других park-сайтов.

**Рекомендация:** **A** для быстрой защиты + B как followup-cleanup
если spurious-wake'ов окажется много в профиле.

## 4. План реализации

1. Targeted-тест: skeleton `sleep_minimal` (один fiber, `Time.sleep(1)`,
   M:N через явный `runtime.init(2)`) — baseline.
2. Применить подход A к `_nova_sleep_park` (`fibers.h:1546`) и
   `_nova_blocking_park` (`:~1640`).
3. Verify: `sleep_bench`/`sleep_precision_bench`/`sleep_real_clock`
   проходят под explicit `runtime.init(0)`.
4. Stress: `sleep_bench` × 10 runs — нет flake.

## 5. Acceptance

- `sleep_bench`, `sleep_precision_bench`, `sleep_real_clock` проходят
  под explicit `runtime.init(0)` и активным M:N — 10 запусков без
  flake.
- Никакого `FATAL sleep wake before close_cb` в логе.
- 0 регрессий полного `nova test` clang.
