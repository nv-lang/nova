# Plan 33.4: Contracts — Production-Parity Gap Closure

> Продолжение [Plan 33.3](33.3-contracts-advanced.md). Закрывает
> критические расхождения с Go/Rust/TS верификаторами, выявленные
> в production-аудите 2026-05-14.

## Контекст

После завершения Plan 33.3 Ф.9 (pure_view + axiom + #verify/#trusted gate,
Z3 backend через собственные FFI-биндинги) был проведён сравнительный
аудит Nova contracts vs Go (нет formal verification), Rust (Prusti/
Creusot/Verus), TypeScript (нет), Dafny, F*/KeY.

**Nova-уникальные преимущества (защищать, документировать):**
- effect-aware contracts: `#pure <op>` + `axiom` прямо в `effect` / `protocol`
  — нет аналога ни в одном mainstream языке (Dafny/Verus не знают об эффектах)
- `axiom` как keyword внутри effect-блока (не атрибут, не внешняя lemma)
- built-in `nova check` без внешней тулзы (Z3 встроен в nova binary)
- собственные Z3 FFI без crate-dependency — полный контроль над версией

**Критические расхождения (этот план):** 5 gap'ов P0, 6 gap'ов P1.

## P0 — Критические soundness/correctness gap'ы

### P0-1. `#verify` handler — gate без реальной верификации

**Файл:** `compiler-codegen/src/types/mod.rs:2922-2925`

Комментарий в коде (verbatim):
```
/// - `#verify_handler` — symbolic verification handler.action body
///   против axiom'ов эффекта (Ф.9.7). Bootstrap V1: атрибут принимается
///   но реальной верификации нет — placeholder для Ф.9.7.
```

**Проблема.** Программист пишет `with #verify Db = my_handler` и думает,
что Nova проверяет что `my_handler.SetBalance` реализует `non_negative`
axiom. На деле gate принимает `#verify` как синтаксис, но **не делает
ни одного SMT-вызова** для verification.

**Расхождение с ближайшим аналогом.** В Verus: `#[verifier::external_body]`
vs обычный verified body — distinction enforced с реальным proof check.
В Dafny: `method M() {...}` — тело обязательно верифицируется если нет
`{:verify false}`. Nova заявляет верификацию через keyword `#verify`
но не доставляет — это soundness gap.

**Что нужно реализовать (Ф.9.7).** Для каждого axiom вида
`post(action)(view) == X`:
1. Найти handler.action body.
2. Symbolic execution: environment для action body + pre-state.
3. После symbolic exec: assert `view_impl == X` где `view_impl` —
   результат symbolic-вычисления `#pure view` через handler's state.
4. Push в Z3; `unsat` = verified; `sat` = counterexample с witness.

**Уточнения:**
- Только для axiom'ов с `post(action)(view)` constructor; static axioms
  (`balance(id) >= 0` без `post`) — verification через handler body
  нетривиальна, V2.
- Handler body с FFI / side effects → автоматически reject, требует `#trusted`.
- Scope: handler'ы зарегистрированные через `with #verify E = h`.

**Acceptance.**
Пример bank account: `ensures Db.balance(to) == old(Db.balance(to)) + amount`
доказывается Z3 при правильном axiom и handler. Неправильный handler
(SetBalance хранит wrong amount) → counterexample с witness-значениями.

### P0-2. Composition в контрактах → EncodingError::Unsupported

**Файл:** `compiler-codegen/src/verify/encode.rs:110`

```rust
Err(EncodingError::Unsupported(format!(
    "function call in contract expression not yet supported \
     (Plan 33.2 composition with `#pure` functions)"
)))
```

**Проблема.** `#pure fn helper(x int) -> bool => x > 0` объявлена,
`requires helper(n)` — compile-time `#pure` check проходит, но SMT
encoding возвращает `Unsupported`. Контракт молча пропускается или
помечается `unknown` без верификации.

**Что нужно.** В `encode_expr` для `ExprKind::Call`:
1. Найти callee fn в module (по имени).
2. Если `fn.is_pure` — emit UF application `(callee_uf arg0 arg1 ...)`.
3. Если fn имеет `ensures` clause — добавить axiom связи:
   `∀ args. callee_uf(args) == encoded(body)` (или упростить через
   inline-substitute тело если оно достаточно простое).
4. Рекурсивно для вложенных вызовов (с depth limit).

**Boundary.** Взаимно-рекурсивные `#pure` fn (SCC) → encoding через
shared uninterpreted function; Ф.9.7 добавит SCC-correctness constraint.

**Acceptance.** `fn is_positive(x int) -> bool => x > 0`; контракт
`requires is_positive(n)` в другой fn → Z3 доказывает что вызывающая
fn получает `n > 0` (через unfolding или UF axiom).

### P0-3. Loop invariants — runtime assert, не SMT

**Файлы:**
- `compiler-codegen/src/parser/mod.rs` (`skip_loop_clauses`)
- `compiler-codegen/src/ast/mod.rs` (`ExprKind::For/While/Loop`)
- `compiler-codegen/src/verify/encode.rs` (нет loop handling)

**Проблема V-part 1: AST.** `invariant` и `decreases` между loop-header
и body парсятся как `skip_loop_clauses` — т.е. полностью игнорируются.
`ExprKind::For`/`While`/`Loop` не имеют полей `invariants`/`decreases`.

**Проблема V-part 2: SMT.** Без AST-полей encode.rs не видит loop contracts
→ не может применить havoc + invariant-preservation pattern.

**Проблема V-part 3: runtime fallback.** Plan 33.2 добавил `invariant`
как runtime assert (debug-only). Это **не** то же самое что формальное
доказательство — runtime может не cover все пути (e.g. если invariant
нарушается через внешний state, runtime никогда не поймает до первого
iteration где state проверяется).

**Что нужно:**

AST: добавить поля к loop вариантам:
```rust
pub enum ExprKind {
    For {
        // ... existing ...
        invariants: Vec<Contract>,
        decreases:  Option<Expr>,
    },
    While {
        // ... existing ...
        invariants: Vec<Contract>,
        decreases:  Option<Expr>,
    },
    Loop {
        // ... existing ...
        invariants: Vec<Contract>,
        decreases:  Option<Expr>,
    },
}
```

SMT encode: havoc + invariant-preservation pattern:
1. Havoc all vars potentially modified in body (frame analysis, или
   over-approximate: all mutable vars in scope).
2. Assert `invariant` (pre-loop).
3. Assert loop condition.
4. Encode body symbolically.
5. Assert `invariant` still holds (post-iteration).
6. For `decreases <expr>`: assert `decreases_val_new < decreases_val_pre`
   AND `decreases_val_pre >= 0` (well-founded ordinal).
7. After loop: assert `!condition` AND `invariant`.

**Breaking change.** Всем interp/codegen/types match-case'ам придётся
добавить обработку новых полей (новые поля опциональны, `Vec::new()` /
`None` default не ломает existing logic).

**Acceptance.** Классика: `forall i in 0..n : xs[i] >= 0` через
loop invariant на `fill` loop → Z3 proves.

### P0-4. `decreases` — счётчик глубины, не well-founded ordinal

**Файл:** `compiler-codegen/src/codegen/emit_c.rs:5183-5197`

```rust
let depth_var = if f.decreases.is_some() {
    // ...
    let var = format!("_nova_decreases_depth_{}", san);
    // ...
    self.line(&format!(
        "if ({}++ > 10000) nova_contract_violation(..., \"decreases recursion depth exceeded 10000\", ...);",
        var
    ));
```

**Проблема.** Текущая реализация — счётчик вызовов. Это ловит только
infinite recursion с depth > 10000. **Не ловит:**
- Рекурсию с нелинейным убыванием (шаги 1000, 999, ..., 500 — всё ещё
  принимается, 10000 шагов ≥ длина убывания)
- Lexicographic termination (`(a, b) → (a-1, ∞)` — счётчик не захватит
  что второй компонент растёт пока первый убывает)
- Well-founded set (custom `<` relation)

**Что нужно — минимум (V2):**
- SMT-side: в encode_fn добавить constraint что `decreases_val_post < decreases_val_pre`
  для каждого рекурсивного вызова (через uninterpreted fn или concrete value).
- Runtime: убрать depth-counter или поднять до 1M (с `--decreases-limit` флагом).
- Well-founded ordinals (Nat, lex pairs) — V3.

**Acceptance.** `fn fib(n int) -> int decreases n` → Z3 верифицирует
что рекурсивные вызовы имеют `n-1 < n` и `n-2 < n`.

## P1 — Важные gap'ы (не критические)

### P1-1. Frame conditions — compile-time type check, нет SMT axiom

**Файл:** `compiler-codegen/src/verify/encode.rs` (нет frame axiom)

`modifies x, y` объявляется и type-check'ится (compile error если
переменная не в scope), но SMT encoding не добавляет frame axiom
«всё что не в modifies — не изменяется». Без этого:
`fn f() modifies x { x = 1; y = 2 }` — y-mutation не reject'ится SMT
(только runtime assert в debug если есть `ensures y == old(y)`).

**Что нужно.** В `encode_fn` после body:
```smt
; frame axiom: forall var not in modifies-list: var_post == var_pre
(assert (= y_post y_pre))
```

### P1-2. Trigger patterns для Z3 quantifiers

Z3 quantifiers без `:pattern` annotation могут не инстанцировать trigger
и зависнуть (timeout). Plan 33.3 Ф.10 описал pattern analysis через AST,
но не реализовал.

**Что нужно.** В `encode_expr(Forall/Exists)`:
1. Парсить body для нахождения «ground trigger» — subterm содержащий
   quantified variable + приложение UF/array-select.
2. Эмитить `:pattern ((select xs i))` для bound-quantifier.
3. Если trigger не найден → compile warning «SMT quantifier may timeout;
   add explicit trigger pattern or reduce scope».

### P1-3. Fail-fn contracts

`fn f() Fail -> T requires ... ensures ...` — Fail-raising functions
имеют контракты, но SMT encoding не моделирует Fail-path:
- `ensures result > 0` только для happy path (non-Fail return)
- Нет `ensures_fail` для Fail path
- Caller не получает знание «если f бросит Fail, то ...»

**Расхождение:** в F\* `exn` эффект тоже в сигнатуре, контракты имеют
`ensures_exn` clause.

**Что нужно.** Добавить `ensures_fail <bool>` clause в контракт (Plan
33.2+ scope). SMT encoding: два пути — happy `ensures` + fail `ensures_fail`.

### P1-4. Liskov substitutability для protocol

`with #verify P = impl` проверяет axioms эффекта/protocol против gate,
но **не доказывает** что `impl` является корректным refinement согласно
Liskov Substitution Principle: если `P` требует `pre` и обещает `post`,
impl обязан принимать более широкий `pre'` и давать более узкий `post'`.

В Verus: `trait Foo` + `impl Foo for T` — Verus проверяет что impl
satisfies trait postcondition.

**Что нужно.** При `with #verify P = impl`:
1. Для каждой operation `op` в `P` с контрактами: assert что
   `impl.op` satisfies same pre/post (contravariant pre, covariant post).
2. Это требует P1-1 (frame axioms) и P0-2 (composition) как pré-requis.

### P1-5. Axiom binder — `Option<TypeRef>` вместо `BinderType` enum

**Файл:** `compiler-codegen/src/ast/mod.rs`

`EffectAxiom.binders: Vec<(String, Option<TypeRef>)>` использует `None`
для untyped binder. Семантически три состояния: `Untyped`, `Typed(TypeRef)`,
`Generic` (from `[T]` params) — но тип не различает их явно. Записано в
`Q-axiom-binder-type` в `spec/open-questions.md`.

**Что нужно:**
```rust
pub enum BinderType {
    Untyped,
    Typed(TypeRef),
    Generic(String),          // generic param name, e.g. "T"
}
pub struct BinderDef {
    pub name:  String,
    pub kind:  BinderType,
    pub span:  Span,
}
```
Обновить: ast/mod.rs + parser/mod.rs + types/mod.rs + verify/pipeline.rs
(4 файла, механический рефактор).

## Реализация

### Ф.1 — P0-1: Symbolic handler verification (Ф.9.7)

**Трудоёмкость:** ~3 дня (research-grade)

**Файлы:**
- `verify/pipeline.rs` — `verify_handler_binding(binding, effect_axioms)`
- `verify/encode.rs` — `encode_handler_body(op_body, pre_state) -> SortedExpr`
- `types/mod.rs` — убрать placeholder comment; подключить `verify_handler_binding`

**Подход:**
1. Собрать все `with #verify E = impl` binding'и из module.
2. Для каждого axiom вида `post(action)(view) == X`:
   a. Найти `impl.action` body.
   b. Создать symbolic state (fresh variables для каждого параметра).
   c. Symbolic exec body: assign/read operations → SMT expressions.
   d. После body: encode `view` через symbolic state.
   e. Assert `view_encoded == X_encoded`; `check_sat`.
   f. UNSAT → verified; SAT → counterexample witness.
3. Reject если body содержит FFI-call, IO, или channel-op (→ требует `#trusted`).

**Упрощения V1 (документировать в simplifications.md):**
- Static axioms (`balance(id) >= 0` без `post`) — skip V1.
- Handler body с branching: только linear path (no-if V1).
- Верифицируем только `SetBalance`-style operations (один return path).

### Ф.2 — P0-2: Composition SMT encoding

**Трудоёмкость:** ~1 день

**Файлы:**
- `verify/encode.rs` — `encode_expr(Call)` case

**Подход:**
1. При `ExprKind::Call { func, args }`:
   - Look up `func` в module.items как `FnDecl`.
   - Если `fn.is_pure`:
     a. Declare UF `(declare-fun fname (arg_sorts) ret_sort)` если не объявлена.
     b. Inline-substitute body если достаточно простое (no recursion, no branches).
     c. Иначе emit `(fname arg0_encoded arg1_encoded ...)` + axiom связи.
2. Non-`#pure` fn в контракте → `EncodingError::UserError` (было Unsupported).
3. Depth limit 10 для рекурсивного inline.

### Ф.3 — P0-3: Loop invariants в AST + SMT

**Трудоёмкость:** ~2 дня

**Файлы:**
- `ast/mod.rs` — добавить `invariants`/`decreases` поля к For/While/Loop
- `parser/mod.rs` — `parse_while`/`parse_for`/`parse_loop`: заменить
  `skip_loop_clauses` на `parse_loop_clauses -> (invariants, decreases)`
- `verify/encode.rs` — `encode_expr(While/For/Loop)` с havoc + invariant pattern
- `codegen/emit_c.rs` — For/While/Loop match'и: добавить игнор новых полей
  (уже есть `f.decreases` pattern — расширить)
- `interp/*.rs` — For/While/Loop match'и: добавить игнор

**Осторожность:** Это breaking-change по AST. Все downstream match'и
обязаны получить обновление (используй `cargo check` для находки).

### Ф.4 — P0-4: Well-founded decreases (V2)

**Трудоёмкость:** ~1 день

**Файлы:**
- `verify/encode.rs` — `encode_fn` для рекурсивных fn
- `codegen/emit_c.rs` — убрать/повысить depth-counter

**Подход:**
1. В encode_fn: при нахождении рекурсивного вызова fn в body:
   a. Encode `decreases_expr_post < decreases_expr_pre` (для Nat-decreases).
   b. Assert в Z3.
2. Для lex-pair: `(a_post, b_post) ≤_lex (a_pre, b_pre)` через
   `(a_post < a_pre) || (a_post == a_pre && b_post < b_pre)`.
3. Runtime: убрать `if (depth++ > 10000)` или заменить на
   `--decreases-limit N` (default: 1_000_000).

## Acceptance criteria

1. **P0-1 (Ф.1):** Bank-account R4 example: handler с wrong `SetBalance`
   → counterexample при `#verify`. Handler с correct impl → UNSAT (verified).
2. **P0-2 (Ф.2):** `#pure fn is_positive(x int) -> bool => x > 0` →
   `requires is_positive(n)` → Z3 proves `n > 0` в caller.
3. **P0-3 (Ф.3):** Fill-array loop с `invariant forall i in 0..k : xs[i] == 0` → Z3 proves.
4. **P0-4 (Ф.4):** `fn fib(n int) decreases n` → Z3 verifies `n-1 < n` AND `n-2 < n`.
5. **P1-1 (Ф.5):** `fn f() modifies x` → Z3 catches `y = 2` mutation
   через frame violation (не только runtime assert).
6. **Регрессия:** Все существующие contract тесты (≥ 47 PASS) остаются green.
7. **Spec sync (Ф.8):** 8 D-decisions задокументированы в spec/decisions/.
