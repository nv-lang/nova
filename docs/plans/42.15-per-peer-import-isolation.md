# Plan 42.15: Per-peer / per-import isolation (NameResCtx refactor)

> **Создан 2026-05-14** — sub-plan Plan 42 (правило C). Выделен из
> Plan 42.14 Ф.4 + Ф.6 (re-defer). Объединяет два связанных gap'а:
> [M10] Rule C Path-form enforcement + [M12] selective import
> visible-scope. Оба требуют **одного и того же** архитектурного
> изменения — per-peer/per-import visible scope в type-checker'е.
>
> **Приоритет:** MEDIUM. Не блокер — оба gap'а это «UX-промис не строгий»,
> реальных нарушений нет. Важно для production library API hygiene.

---

## Два gap'а — один корень

### [M10] Rule C Path-form enforcement

Plan 42 правило C: peers share **declarations**, но **не imports**.
Сейчас:
- Bare `Ident` проверяется per-peer (Plan 42.04 — `peer_module_names`).
- **Path-form** (`alias.func()`) — `ExprKind::Path` полностью
  пропускается name-check'ом.
- `module.items` flat → imported items одного peer видны codegen'у
  как top-level, "протекают" в namespace других peers.

### [M12] Selective import visible-scope

`import X.{A}` — selective list парсится, rename (`A as B`) работает,
но items НЕ в `{...}` всё равно merge'атся в visible scope. `import X.{A}`
не прячет `B`.

### Общий корень

Оба — следствие **flat `module.items`** + **NameResCtx без per-import
visible tracking**. Resolver inline-expansion сваливает всё в один
namespace; type-checker не различает «видно потому что я импортировал»
vs «видно потому что транзитивная зависимость».

---

## Архитектурное решение

**Разделить codegen-scope и visible-scope:**

1. **Codegen-scope** = всё (для completeness — `A`'s body ссылается на
   `B` из того же модуля X). `module.items` flat остаётся для codegen.

2. **Visible-scope** (per-peer, per-import) — что type-checker считает
   «доступным по имени» в данном peer:
   - declarations всех peers этого folder-module (Rule C: shared decls)
   - + items явно импортированные **этим** peer'ом
   - + selective list если `import X.{A}` — только `A`, не весь X
   - НЕ: imports других peers, НЕ: транзитивные не-selective items

**Реализация:**
- `PeerFile` получает `visible_names: HashSet<String>` — что видно
  в этом peer (вычисляется в resolver из его imports + selective lists).
- `NameResCtx`: `is_known(name, file_id, scope)` проверяет
  `peer_files[file_id].visible_names` вместо общего flat top_level.
- `ExprKind::Path` — head-segment тоже проверяется (lowercase alias →
  должен быть в peer's visible module names).
- Codegen — не трогается, читает flat `module.items`.

## Риски

- **False positives.** Path-form head может быть тип / effect /
  builtin namespace (`gc`, `fibers`). Нужен careful whitelist +
  Capitalized-skip (уже есть).
- **Транзитивные зависимости.** `import X.{A}` где `A` использует `B`
  — `B` в codegen-scope (ok), но если user пишет `B()` напрямую →
  должна быть ошибка. Edge: `B` может совпасть с builtin.
- **Backward compat.** Существующие тесты используют folder-modules
  без strict isolation — могут начать падать. Mitigation: аккуратная
  миграция, возможно warning-mode сначала.
