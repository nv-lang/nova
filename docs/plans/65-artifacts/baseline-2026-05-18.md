## Boehm GC finalizer infra (AD7 prerequisite)

**Status: UNCONFIRMED / likely deferred.**

Search results:
- `grep GC_REGISTER_FINALIZER` in worktree → **no hits** in runtime sources.
- `alloc_boehm.c` line 17 explicitly: *"exact live count requires finalizer
  cooperation which Boehm does not provide"*.
- `alloc_boehm.c` line 113: similar wording — finalizers not currently used.
- `gc_test_helpers.c:78` mentions finalizers conceptually (second-pass collect
  guard) but does not register any.

Implication for Ф.2 (R13 — drop semantics test `f9_drop_no_leak.nv`):
the underlying `NovaAfterState` is **not** GC-managed today — it is `malloc`'d
and owned by libuv (see channels.h:1071-1084, "raw malloc, NOT nova_alloc").
The lifecycle is: timer fires or `on_select_lost` cancels → `uv_close` →
`_nova_after_close_cb` → `free`.

This means **finalizer-based cleanup as described in AD7 cannot be implemented
without first wiring `GC_REGISTER_FINALIZER` end-to-end**. The current design
relies on either timer-fire or explicit cancellation via `on_select_lost`.

Honest-defer for Ф.2: AD7 finalizer registration deferred behind
`[M-chanreader-gc-finalizer]` marker — see Plan 65 evolution note below. f9 test
becomes positive-only check that timer fires OR is cancelled by scope exit (not
by GC). True drop semantics arrive after Boehm finalizer wiring (out-of-scope
for Plan 65; suggested future work).
