## Зачем

Текущий baseline `std/runtime/sync.nv` — только `AtomicInt`/`AtomicBool`
с 4-6 операциями каждый, hardcoded ordering. **Industry-standard**:
12 sized types × ~15 операций × 5 orderings.

Конкретные gaps:

| Gap | Impact |
|---|---|
| **Sized atomic типы** (`AtomicI8/I16/I32/U64/Usize/...`) отсутствуют — есть только `AtomicInt` (`i64`) | Нет precise atomics для byte flags, counter overflows, sized buffer indexes; force conversion + waste cache lines |
| **`fetch_or`/`and`/`xor`** отсутствуют | Нет lock-free bit flags (bitmap operations через CAS-loop — медленно) |
| **`fetch_max`/`fetch_min`** отсутствуют | Нет lock-free max/min trackers (tail-pointer, high-watermark) |
| **`compare_exchange_weak`** в C-слое (`sync.h:163`) есть, не exposed | CAS-loop'ы платят DMB на ARM unnecessarily |
| **`MemOrdering`-параметризованные методы** отсутствуют | Все ops через hardcoded `acq_rel`/`acquire`/`release` — нет relaxed для perf-critical counters |
| **`AtomicPtr`** в C-слое (`sync.h:188`) есть для runtime internals, не exposed | Нет lock-free first-writer-wins pattern на user-side |

Plan 103.2 — **core atomic suite** на production уровне. Использует
`MemOrdering` enum из 103.1.

## Scope

### In scope

**12 sized atomic типов** в `std/runtime/sync.nv`:

| Тип | Wraps | Use case |
|---|---|---|
| `AtomicI8` | `int8` | byte-sized signed flags |
| `AtomicI16` | `int16` | sized counters |
| `AtomicI32` | `int32` | most common int atomics |
| `AtomicI64` | `int64` | wide counters, timestamps |
| `AtomicU8` | `byte` (Plan 69) | byte-sized unsigned flags, bitmap |
| `AtomicU16` | `u16` | sized unsigned counters |
| `AtomicU32` | `u32` | bitset slots, refcount |
| `AtomicU64` | `u64` | wide unsigned counters |
| `AtomicIsize` | `int` (= `i64` Plan 70.4 D129) | platform-word signed |
| `AtomicUsize` | `uint` (= `u64` Plan 70.5) | platform-word unsigned, indexes |
| `AtomicBool` | `bool` | one-shot flags, cancel sentinels |
| `AtomicPtr` | managed ref `Ptr[T]` | first-writer-wins, lock-free queues |

**Каждый integer тип** (10 шт.) поддерживает:

- `load(ord MemOrdering) -> T` + `load() -> T` (default SeqCst)
- `store(v T, ord MemOrdering)` + `store(v T)`
- `swap(v T, ord MemOrdering) -> T` + `swap(v T) -> T`
- `compare_exchange(expected T, desired T, success MemOrdering, failure MemOrdering) -> bool` + `compare_exchange(expected T, desired T) -> bool` (default SeqCst)
- `compare_exchange_weak(expected T, desired T, success MemOrdering, failure MemOrdering) -> bool` + weak default-ordering overload
- `fetch_add(v T, ord MemOrdering) -> T` + `fetch_add(v T) -> T` (returns previous)
- `fetch_sub(v T, ord MemOrdering) -> T` + `fetch_sub(v T) -> T`
- `fetch_or(v T, ord MemOrdering) -> T` + `fetch_or(v T) -> T`
- `fetch_and(v T, ord MemOrdering) -> T` + `fetch_and(v T) -> T`
- `fetch_xor(v T, ord MemOrdering) -> T` + `fetch_xor(v T) -> T`
- `fetch_max(v T, ord MemOrdering) -> T` + `fetch_max(v T) -> T`
- `fetch_min(v T, ord MemOrdering) -> T` + `fetch_min(v T) -> T`
- `fetch_nand(v T, ord MemOrdering) -> T` + `fetch_nand(v T) -> T` (для consistency с Rust; AI-rare)

**AtomicBool** (не integer): load/store/swap/compare_exchange[_weak]/
fetch_or/fetch_and/fetch_xor. (Нет add/sub/min/max — бессмысленно
для bool.)

**AtomicPtr[T]** (для GC-managed reference):
- `load(ord)` / `store(v, ord)` / `swap(v, ord)` / `compare_exchange[_weak]`
- **No** fetch_add/sub/or/and/xor (pointer arithmetic — `unsafe` paradigm, не Nova)
- Static-constructor `AtomicPtr.null() -> AtomicPtr[T]` + `AtomicPtr.new(v Ptr[T])`

**Backward compat (M14):**
- `AtomicInt` → deprecated alias на `AtomicI64`; existing `AtomicInt.new(v)`, `.load()`, `.store(v)`, `.fetch_add(d)`, `.fetch_sub(d)`, `.compare_exchange(e, d)` сохраняют те же сигнатуры (default SeqCst через AtomicI64 overloads).
- `AtomicBool` сохраняет существующий API; новые ordering-aware overloads добавляются без замены.
- Deprecation warning `W_ATOMIC_INT_DEPRECATED` через `compiler-codegen/src/diag.rs` (`Diagnostic` + `Suggestion` с `Applicability::MachineApplicable`) — suggest `AtomicI64`.

**Wraparound contract** (для unsigned overflow + signed overflow):
- `fetch_add` на signed-overflow → **wrapping** (consistent с Rust default, Java).
- `fetch_add` на unsigned-overflow → wrapping (modular arithmetic).
- Документировано в D168 (Plan 103.7); НЕ panic. **Если пользователь нуждается в checked-overflow** — отдельный `try_fetch_add` overload, отложен в 103.x future (low priority).

**GC integration для `AtomicPtr[T]`:**
- `Ptr[T]` — managed reference (Nova GC marks); `AtomicPtr` хранит refcell, не raw pointer.
- GC scan pass должен видеть `AtomicPtr[T].value` — добавляется в `GC_set_push_other_roots` callback (precedent: Plan 82 fiber arena).

### Out of scope

- `AtomicF32`/`AtomicF64` — отдельный low-priority future plan (через bits transmutation).
- Generic `Atomic[T]` для user-defined records — `[M-fn-prefix-int-only-mono]` block, M2.
- Atomic arrays (Rust `[AtomicI32; N]`) — Nova arrays GC-managed, отдельный design.
- consume-form `MutexGuard`-style для atomic — M16 (atomics не resource).

## API design (verbatim, готов к Ф.1 копированию)

```nova
module runtime.sync

// ─── AtomicI64 (canonical pattern) ──────────────────────────────

/// Lock-free 64-bit signed atomic integer.
///
/// Wraparound: signed overflow в fetch_add/sub — modular arithmetic
/// (consistent с Rust default, Java AtomicLong).
///
/// # Default ordering
///
/// Methods без `MemOrdering` параметра используют `SeqCst` (M1) — safest.
/// Для perf-critical использовать `_ordered` варианты с explicit
/// ordering из [`MemOrdering`].
#stable(since = "0.1")
export external fn AtomicI64.new(v int64) -> Self

// load
#stable(since = "0.1")
export external fn AtomicI64 @load(ord MemOrdering) -> int64
#stable(since = "0.1")
export external fn AtomicI64 @load() -> int64    // default SeqCst

// store
#stable(since = "0.1")
export external fn AtomicI64 mut @store(v int64, ord MemOrdering)
#stable(since = "0.1")
export external fn AtomicI64 mut @store(v int64)

// swap
#stable(since = "0.1")
export external fn AtomicI64 mut @swap(v int64, ord MemOrdering) -> int64
#stable(since = "0.1")
export external fn AtomicI64 mut @swap(v int64) -> int64

// CAS strong
#stable(since = "0.1")
export external fn AtomicI64 mut @compare_exchange(expected int64, desired int64, success MemOrdering, failure MemOrdering) -> bool
#stable(since = "0.1")
export external fn AtomicI64 mut @compare_exchange(expected int64, desired int64) -> bool

// CAS weak (may fail spuriously, cheaper on ARM)
#stable(since = "0.1")
export external fn AtomicI64 mut @compare_exchange_weak(expected int64, desired int64, success MemOrdering, failure MemOrdering) -> bool
#stable(since = "0.1")
export external fn AtomicI64 mut @compare_exchange_weak(expected int64, desired int64) -> bool

// fetch_* — returns previous value
#stable(since = "0.1")
export external fn AtomicI64 mut @fetch_add(v int64, ord MemOrdering) -> int64
#stable(since = "0.1")
export external fn AtomicI64 mut @fetch_add(v int64) -> int64
// ... аналогично fetch_sub/or/and/xor/max/min/nand
```

**Остальные 9 integer типов** (`AtomicI8/I16/I32/U8/U16/U32/U64/Isize/
Usize`) — same shape, заменён parameter/return type. Файл `sync.nv`
расширяется на ~600 строк (12 типов × ~50 строк каждый).

**`AtomicBool`** + **`AtomicPtr[T]`** — specialised форма (см. In scope
описание операций выше).

### Backward compat (M14)

```nova
/// Deprecated alias на `AtomicI64`. Migration: `AtomicI64.new(v)`.
#deprecated(since = "0.1", suggest = "AtomicI64")
export type AtomicInt = AtomicI64
```

`type-alias` mechanism + deprecation warning. Existing user code
компилируется без изменений; nova check выдаёт `W_ATOMIC_INT_DEPRECATED`.

## Tests breakdown

### Positive (8 файлов в `nova_tests/plan103_2/`)

1. **`atomic_i64_smoke.nv`** — load/store/swap/CAS/fetch_add/sub/or/and/
   xor/max/min/nand на одном `AtomicI64` — round-trip single-fiber.
2. **`atomic_each_int_size.nv`** — coverage всех 10 integer types
   (`AtomicI8`/`I16`/`I32`/`I64`/`U8`/`U16`/`U32`/`U64`/`Usize`/`Isize`):
   `new(42)` + `load()` + `fetch_add(1)` + `load()` — verify correct
   width.
3. **`atomic_bool_full.nv`** — `AtomicBool` все ops (load/store/swap/
   CAS/fetch_or/and/xor) + backward compat (no-ordering API).
4. **`atomic_ptr_first_writer.nv`** — 4 fibers race для CAS на
   `AtomicPtr[T]`; exactly one wins, остальные see winner's value.
   GC-test: AtomicPtr value protected from collection.
5. **`atomic_ordering_relaxed_fast_path.nv`** — counter с
   `fetch_add(1, MemOrdering.Relaxed)` × 1000 × 4 fibers = exactly 4000
   (под M:N с 4 workers).
6. **`atomic_fetch_bitmask.nv`** — bitset через `AtomicU32.fetch_or` /
   `fetch_and` (set/clear bit) — verify bit ops correctness.
7. **`atomic_fetch_max_min.nv`** — high-watermark tracker: N fibers
   record max через `fetch_max(value, MemOrdering.Relaxed)` — final value
   = max всех.
8. **`atomic_wraparound_semantics.nv`** — `AtomicI8.new(127).fetch_add(1)`
   → returns 127, new value = -128 (wraparound, NOT panic).
   `AtomicU8.new(255).fetch_add(1)` → returns 255, new = 0.

### Negative (6 файлов; покрывают type-checker hooks из 103.1)

1. **`atomic_load_with_release_neg.nv`** — `a.load(MemOrdering.Release)` →
   `EXPECT_COMPILE_ERROR E_INVALID_ORDERING_LOAD`.
2. **`atomic_load_with_acqrel_neg.nv`** — то же с AcqRel.
3. **`atomic_store_with_acquire_neg.nv`** — `a.store(v, MemOrdering.Acquire)`
   → `E_INVALID_ORDERING_STORE`.
4. **`atomic_store_with_acqrel_neg.nv`** — то же с AcqRel.
5. **`atomic_ptr_unsafe_arith_neg.nv`** — `AtomicPtr` без fetch_add
   (нет такого метода) → `E_UNKNOWN_METHOD`.
6. **`atomic_int_deprecated_warn.nv`** — `AtomicInt.new(1)` →
   `EXPECT_WARNING W_ATOMIC_INT_DEPRECATED` (existing code продолжает
   компилироваться).

### Property (4 файла)

1. **`cas_weak_convergence_prop.nv`** — N fibers CAS loop через
   `compare_exchange_weak`, eventual convergence (counter = N×iters
   ± spurious-fails-retried) — property: never lose updates.
2. **`atomic_relaxed_no_torn_writes.nv`** — single 64-bit `AtomicI64`
   stress: 1 writer × 4 readers с `Relaxed` ordering; readers видят
   только полные значения (никогда half-written), даже если happens-
   before не enforced.
3. **`atomic_seqcst_total_order.nv`** — два `AtomicBool` (X, Y), два
   writers (X.store; Y.store) + два readers — under SeqCst observable
   total order (litmus test, classic IRIW).
4. **`atomic_ptr_gc_safety.nv`** — `AtomicPtr` holds large allocation,
   writer replaces через CAS, reader holds stale pointer — GC не
   collect'ит stale value пока reader его читает (managed-ref
   tracking).

### Stress (2 файла, частично перекрываются с 103.8)

1. **`atomic_stress_10k_fetch_add.nv`** — 4 fibers × 10000 fetch_add(1)
   = exactly 40000 (M:N с 4 workers); 100 итераций — stable.
2. **`atomic_stress_mixed_ops.nv`** — 4 fibers: fetch_add / fetch_or /
   compare_exchange — observable invariant сохраняется.

## Risks + mitigations

| Риск | Mitigation |
|---|---|
| 12 types × 13 ops = 156 method declarations — велик файл sync.nv | Acceptable; copy-paste pattern; future refactoring (generator) — отдельный план если maintenance pain |
| GC integration AtomicPtr — потенциально tricky | Precedent есть (Plan 82 fiber arena registered roots); falls back на raw ptr с manual GC barriers как escape |
| Default-SeqCst overload misuse: AI пишет hot-path counter без Relaxed | Plan 103.7 D173 AI-first guidance: «counters/stats → Relaxed; publish/subscribe → Acquire/Release; default → SeqCst» |
| AtomicI8/U8 overflow surprises пользователя | D168 явно документирует wraparound; lint `W_NARROW_ATOMIC_OVERFLOW_RISK` (optional, low priority) |
| `type-alias` mechanism (AtomicInt = AtomicI64) может не поддерживаться | Verify в Ф.7; fallback на embed pattern (`type AtomicInt { use AtomicI64 }`) или explicit re-export |
| CAS_weak на x86 = same as strong → property test ARM-specific | Документировать в D168; ARM CI — отложено до Linux ARM CI готов |
