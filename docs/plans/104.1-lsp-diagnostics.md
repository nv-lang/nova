## Цель

LSP `textDocument/publishDiagnostics` для Nova-файлов с production-grade
характеристиками:

- **Latency** ≤500ms от edit до подсветки на типичных файлах (≤2k LOC).
- **Incremental** sync (не full Rope rebuild) — TextDocumentSyncKind::Incremental.
- **Cancellation-safe**: новый didChange отменяет pending recompile корректно.
- **Multi-file workspace**: при изменении файла A пересчёт всех модулей,
  которые его импортируют (full workspace recheck V1; per-module dep-graph — V2).
- **Все диагностики компилятора** mapped в LSP Diagnostic (severity, range,
  code, source, related_information). Никаких потерь данных.
- **UTF-16 positions** (LSP requires) — конверсия из byte offsets через
  Rope с правильным учётом surrogate pairs.

**Production criteria** (acceptance):
1. VSCode connected → подсветка ошибок ≤500ms после edit на nova_tests/.
2. Edge cases: multi-byte UTF-8 в идентификаторах, surrogate pairs в строках,
   очень большие файлы (10k LOC), сотни одновременных didChange.
3. Memory: при закрытии редактора — нет утечек (cache cleanup, task cancellation).
4. Stress: 100 didChange/s на один файл → сервер не падает, latency не растёт
   (debounce работает).
5. Все existing nova-lsp/tests/ продолжают PASS.

## Архитектура

```
didChange ──┐
            ├─► debouncer (200ms tokio sleep + cancel-token)
didSave  ───┘                        │
                                     ▼
                            recompile_task (tokio::spawn)
                                     │
                                     ▼
                            nova_codegen::check_workspace
                                     │
                                     ▼
                            Diagnostic → lsp_types::Diagnostic
                                     │
                                     ▼
                            client.publish_diagnostics()
```

### Stack additions

- **nova_codegen** как path-dep в `nova-lsp/Cargo.toml` (`{ path = "../compiler-codegen" }`).
- **tokio_util** для CancellationToken.
- **lsp-types** transitively через tower-lsp.

### Cancellation model

Каждый файл имеет `Mutex<Option<CancellationToken>>` в WorkspaceState.
Новый didChange:
1. Берёт lock.
2. Если есть pending token → cancel().
3. Создаёт новый token.
4. Spawn задачу с этим token; задача проверяет `is_cancelled()` перед
   тяжёлыми этапами (parse, type-check, emit).

### Incremental sync

LSP сообщает изменения как range + new_text. Применяется к Rope через
`rope.remove(byte_range)` + `rope.insert(start, new_text)`.

Range приходит в UTF-16 positions. Конверсия UTF-16 → byte через
`ropey::Rope::utf16_cu_to_char` + `char_to_byte`.

### Multi-file recheck strategy V1

При didChange файла A:
1. Recompile сам A.
2. Если файл — peer-module (Plan 42) или импортируется другими (через
   `nova_codegen::import_graph`) — recompile зависимых.

V1 strategy: **full workspace recheck** (как gopls V1). Достаточно для файлов
до ~50 модулей. Per-module incremental — V2 (separate sub-plan 104.1.X если
не дойдём по budget).

### Ф.3 — Debouncer + CancellationToken

**Out:**
- `nova-lsp/src/debouncer.rs`:
  ```rust
  pub struct Debouncer {
      pending: Arc<Mutex<HashMap<Url, CancellationToken>>>,
      delay: Duration,
  }
  impl Debouncer {
      pub fn schedule<F, Fut>(&self, uri: Url, work: F)
          where F: FnOnce(CancellationToken) -> Fut + Send + 'static,
                Fut: Future<Output = ()> + Send;
  }
  ```
- При schedule: если есть pending token для uri → cancel().
- Spawn новой задачи: `tokio::sleep(delay)` → если не cancelled → run work.
- Work получает token, может проверять `is_cancelled()` в hot loop.

**Tests (nova-lsp/tests/debouncer.rs):**
- pos1: schedule → через delay работа выполняется
- pos2: schedule + immediate schedule (same uri) → первая cancelled, вторая выполняется
- pos3: schedule разные uri → выполняются параллельно
- pos4: 100 rapid schedules одного uri → только последняя выполняется (debounce effectiveness)
- neg1: work, который panic'ит → не падает debouncer, лог + продолжает
- edge1: cancel сразу после schedule → work не выполняется
- edge2: schedule после shutdown — graceful (нет panic)
- edge3: work чекает is_cancelled в loop → корректно прерывается

Commit: `feat(nova-lsp): Plan 104.1.Ф.3 debouncer + cancellation`

### Ф.5 — publishDiagnostics + per-file workflow

**Out:**
- В `Backend::did_change` / `Backend::did_save`:
  1. apply_changes к Rope в state.
  2. debouncer.schedule(uri, |token| async { ... })
  3. В work: spawn_blocking { check_file(uri, text) } → map diagnostics →
     check token.is_cancelled() → client.publish_diagnostics(uri, diags, version).
- При `did_open`: immediate check (no debounce), publish.
- При `did_close`: publish empty diagnostics (clear редактору).
- Version tracking: передавать version в publish_diagnostics для outdated suppression.

**Tests (nova-lsp/tests/publish_workflow.rs, e2e through process):**

## Production constraints (нельзя сокращать)

1. **Incremental sync** — TextDocumentSyncKind::Incremental, не Full.
   Full rebuild Rope на каждый edit неприемлемо для prod.
2. **Cancellation** — каждая pending compile cancel'ится новым didChange.
   Без этого: dropped edits + wasted CPU.
3. **Catch panics** в compiler invocation — server не должен падать
   на pre-existing compiler bug. Возвращать InternalError diagnostic.
4. **UTF-16 positions** — LSP spec requires. Не отсылать byte offsets.
5. **Version tracking** — outdated publishDiagnostics (после нового didChange)
   suppressed клиентом, но всё равно отправляем правильный version.
6. **Memory cleanup** на shutdown / didClose — нет утечек.
7. **Tracing instrumentation** — для debug в production.

## Decision points (lock'нуть до старта)

| ID | Decision | Решение |
|---|---|---|
| D104.1-1 | Sync kind: Full vs Incremental | **Incremental** (prod) |
| D104.1-2 | Multi-file: full workspace vs per-module dep-graph | **Full workspace V1**; per-module — отдельный V2 sub-plan если нужно |
| D104.1-3 | Cancellation: per-uri token vs single global queue | **Per-uri token** (parallelism for unrelated files) |
| D104.1-4 | Panic handling: catch_unwind vs let crash | **catch_unwind** (server must survive) |
| D104.1-5 | File watching: didChangeWatchedFiles vs polling | **Both**: respect didChangeWatchedFiles if client sends; mtime poll on workspace scan для filesystem-edits-outside-editor |
| D104.1-6 | Debounce delay | **200ms** (gopls / rust-analyzer default) |
| D104.1-7 | Worker thread for compile: spawn_blocking vs dedicated thread pool | **spawn_blocking** (default tokio threadpool; size auto) |

## Risks

| Risk | Mitigation |
|---|---|
| nova_codegen API меняется (Plan 91/100 в работе) | Локальный `compiler.rs` adapter абстрагирует — изменения API локализованы в нём |
| UTF-16 conversion bugs (off-by-one на surrogate pairs) | Edge-tests с emoji + cyrillic в фикстурах |
| Race: didChange во время текущего publish_diagnostics | Version check — клиент игнорирует outdated; на сервере explicit cancel |
| Memory leak в дашмап cache | Test pos2 в Ф.8 — 100 cycles → stable RSS |
| Compiler panic'ит на edge input | catch_unwind + InternalError diagnostic; не отвалить сервер |
