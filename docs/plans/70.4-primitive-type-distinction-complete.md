После [Plan 70.3](70.3-char-int-mono-distinction.md) (`char/int`)
аудит выявил **4 оставшихся collapse pattern** в codegen mono'd generics.
Все — codegen-only mechanical migration в тех же файлах
(`emit_c.rs` + `external_registry.rs` + `nova_rt/array.h`).

Объединяю в один plan потому что:
- Same files affected — atomic refactor cheaper than 3 separate sessions
- Same architectural pattern — parallel `NOVA_ARRAY_DECL` instances + match arm additions
- Same migration cost — caller code already uses correct primitive names

## Inventory (Ф.0 — confirmed via emit_c.rs:2820-2835)

### Ф.1 — f32/f64 array distinction (P2, was original 70.4 scope)

Scalar level — already distinct (`nova_f32`/`nova_f64` distinct C types).
Array path collapses:

```rust
"f64" | "f32" => "NovaArray_nova_f64*".into(),  // BOTH f32 AND f64 → f64 array
```

ABI mismatch (4-byte vs 8-byte элементы) → silent data corruption если
`[]f32` обрабатывается как `[]f64` через FFI или unsafe cast.

### Ф.2 — Sized-int array distinction (P1, ABI-real, **highest impact**)

Scalar level distinct (`int32_t`, `int16_t`, etc), but array fallback
collapses ALL sized-int variants:

```rust
_ => "NovaArray_nova_int*".into(),  // catch-all
```

Affected: `[]i8`, `[]i16`, `[]i32`, `[]i64`, `[]u16`, `[]u32`, `[]u64`
— **8 types**, все silently emit `NovaArray<int64_t>*` вместо packed
sized array.

Concrete impact:
- `[]i32` буфер для GPU/audio/network → 2x memory overhead + alignment shift
- `[]u32` для color/pixel arrays → wrong wire format
- FFI к C-libraries ожидающим `int32_t buf[N]` → silent failure

**SAME bug-class как Plan 70.3 char/int** — discovered отдельно потому
что Plan 70.3 audit fokus'нулся на char-specific.

### Ф.3 — int/i64 cosmetic alias

Scalar level both → `nova_int`. Не silent miscompilation (semantically
same int64 type), но Nova-level `Map[int, V]` ≡ `Map[i64, V]` mangle
identical → cannot type-checker reject swap.

**Decision (proposed):** Document `int` as **alias** `i64` в spec D-block.
Don't introduce distinct mangling — это бы заставило user code
explicitly choose `int` или `i64` для каждого generic instantiation,
что hostile. Modern parallels:
- Rust: `isize` distinct from `i64` (platform-pointer width)
- Go: `int` distinct from `int64` (platform-pointer width)
- Nova bootstrap: `int` = `i64` (no platform variation — keep alias)

### Ф.4 — byte/u8 final cleanup (Plan 69 closure leftover)

Plan 68/69 declared «canonical name — u8», но codegen всё ещё имеет
два **разных** mappings:

```rust
"byte" => Ok("nova_byte".into()),  // 8-byte struct typedef (legacy)
"u8"   => Ok("uint8_t".into()),    // canonical
```

`nova_byte` and `uint8_t` are different C type names (both 1-byte
underlying). Generic mono mangle `[]byte` ≠ `[]u8` — collapse legacy.

**Decision:** Remove `"byte"` mapping, route к `"u8"` via type-alias
resolution. Existing user code using `byte` (если есть) — automated
migration tool.

## Архитектурное решение (Option A — extend нынешний pattern)

For ALL 4 phases: same approach as Plan 70.3:

1. **Add NOVA_ARRAY_DECL** instances в `array.h` для каждого distinct
   sized-int + nova_f32 (f64 уже есть).
2. **Update `emit_c.rs:2820-2835` match arms** — add explicit cases
   для каждого type, не `_` catch-all.
3. **Parallel в `external_registry.rs:243-256`** — second copy of
   type_ref_to_c needs sync.
4. **Backward-compat** через accept-both pattern (mirror Plan 70.3
   StringBuilder + char).

## Risks

| Risk | Mitigation |
|---|---|
| **Runtime header explosion** — 8 sized-int × (Array+Opt+eq) = ~24 inline fns | Macro `NOVA_ARRAY_DECL` уже параметризован, just add instances |
| **Existing test corpus uses `[]int` heavily** — migration cost | `int` остаётся `nova_int` (alias), no change для `[]int` callers |
| **byte→u8 user code breakage** | Migration tool + type-alias resolution в parser |
| **Bench regression** от array helper duplication | Inline fns — zero runtime cost, only binary size impact (negligible) |
| **Plan 71 / 72 spec D-block collision** | D129 reserved сейчас (D128 = 70.3); coordinate если другие активные plans claim |
