# Plan 118.5 — universal right-binding rule + `unsafe T` first-class migration

> **Создан:** 2026-06-04. **Статус:** 🆕 PLANNED.
>
> **Расширяет:** D216 V2 amend (spec/decisions/02-types.md). **Breaking
> change** для `*ro T` / `*mut T` / `*unsafe T` syntax.
>
> **Зависит от:** Plan 118 V1 (foundational FFI), Plan 118 Ф.4 V1 (typed
> read/write).

---

## Что

Унификация type-level modifier syntax под единое правило «right-binding»:
modifier применяется ко всему справа до конца type-expression или до
следующего modifier. Параллельно — `unsafe T` становится first-class type
wrapper с MaybeUninit semantic.

См. D216 V2 amend §V2.1-V2.7 для полной спецификации.

## Почему

Текущая грамматика типов имеет **два правила** одновременно:
- `ro T` parses как right-binding (`Readonly(T)` через recursive `parse_type`)
- `*ro T` / `*mut T` / `*unsafe T` parses как inline-modifier (modifier внутри
  pointer constructor)

Inconsistency дискомфортна для users (intuition диктует одно правило для
всех modifiers) и закрывает orthogonal expressiveness:
- `unsafe * T` (unsafe pointer) ↔ `* unsafe T` (valid ptr to uninit T) —
  невозможно express под current inline-modifier scheme

Migration cost линейно с FFI footprint — fix НОВЫЕ deferred plans (118.1
Ф.2-Ф.4 / 118.2 / 118.3) под новый syntax до того как accumulate'нутся
fixtures под старый.

## 2026-06-04 (late evening) — V3 design rules + 4 new constraints

User refined design via 4 successive clarification rounds:
1. Modifier propagates auto-right (per D33 amend pattern); `ro * ro T` redundant
2. `unsafe` orthogonal к `ro/mut` (additional axis, can stack)
3. `unsafe * T == unsafe * unsafe T` (full propagation; redundant inner)
4. `ro mut T` storage-class-dependent ban; `ro unsafe T` ordering ban; `safe` keyword

  - If T's storage class is value → emit `E_MUTABILITY_CONFLICT_VALUE_TYPE`
  - If T is ref-type AND at binding context (let/param/field) → OK

### Ф.3 — V3 Rule 3: modifier ordering (safety-outer)

  Readonly/Mut/Pointer; stop at Named/Array/Tuple/Func)
- Parser KwRo/KwMut arms: after recursive parse_type returns inner, check
  `inner.contains_unsafe_in_chain()` — if found, emit `E_MODIFIER_ORDER`

### V3 user-confirmed clarifications (2026-06-04)

**Value type list — final (per user):**
- Primitives: `int` (alias `isize`), `uint` (alias `usize`),
  `i8`/`i16`/`i32`/`i64`, `u8`/`u16`/`u32`/`u64`, `f32`, `f64`,
  `bool`, `char`, `byte`, `str`, `ptr`
- Value records (`type X value { ... }`) — Plan 124.8 D228
- Named tuples (`type Point(x f64, y f64)`) — Plan 120 D215
- Anonymous tuples `(A, B, C)` literal type syntax
- Unit `()`

**Reference type list:**
- Records (default heap)
- Arrays `[]T` / FixedArray `[N]T`
- Pointer (any modifier wrapping)
- Func, Protocol

**int/uint aliasing:**
- `int` = alias for `isize` (platform-pointer-width signed)
- `uint` = alias for `usize` (platform-pointer-width unsigned)
- Both forms recognized identically by V3 storage-class check
- On 64-bit target: int ↔ i64, uint ↔ u64

**`safe` reserved keyword migration:**
- Pre-V3 code using `safe` as identifier (fn/let/var name) requires rename
- Audit found 1 occurrence in codebase: `nova_tests/plan118_5/t4_neg_unsafe_arg_to_safe_param.nv` (`fn safe` → `fn accept_safe`)
- Stdlib + examples + other tests UNCHANGED (`safe` appears only в comments как English word, not identifier)
- Documentation impact: any future docs/examples using `safe` as identifier must rename
