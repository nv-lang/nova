# Plan 83.4: M:N supervised-drain hardening — gate-unblock для Plan 83.2

> **Создан:** 2026-05-23 (вырос из Plan 83.2 honest partial).
> **Статус:** 📋 proposed, не начат.
> **Приоритет:** P2 — gate для Plan 83.2 (флип M:N в дефолт).
> **Родитель:** [Plan 83](83-mn-default-on-gomaxprocs.md) / sibling
> [Plan 83.2](83.2-mn-default-flip.md).

## 1. Контекст

[Plan 83.2 Ф.1](83.2-mn-default-flip.md) (2026-05-23) подготовил
инфраструктуру default-on M:N: public API `nova_runtime_auto_arm()` +
защитный auto-arm в `nova_runtime_spawn_global` / `nova_runtime_spawn_into`
(коммиты `b72ce59b475` + `0af6e6ba482`). Активация флипа — одна
строка `nova_runtime_auto_arm()` в `emit_c.rs::emit_main_wrapper`
(сейчас закомментирована).

Прямая активация вскрыла **9 pre-existing M:N багов** в supervised-
drain layer, проявлявшихся ДО 83.2 только при explicit `runtime.init`.
До закрытия этих багов флип ломает **19 RUN-FAIL** из 75 concurrency-
тестов под clang.

Plan 83.4 — целевая серия фиксов этих 9 багов. После — флип
выполняется одной строкой codegen-emit'а.

## 2. Список багов (из `[M-83.2-supervised-mn-bugs]`)

### 🔴 Категория A — runtime M:N race conditions (1-2 dev-day каждый)

#### A1. D93 sleep-wake protocol race
**Симптом:** `nova: FATAL sleep wake before close_cb (stage=0) — D93 protocol bug`.
**Тесты:** `sleep_bench`, `sleep_precision_bench`, `sleep_real_clock`.
**Локация:** `compiler-codegen/nova_rt/fibers.h:1552-1557`.
**Корень:** `timer_cb` запускает `close_cb` асинхронно; под M:N main-
thread drain wake'нет fiber'а ДО завершения `close_cb`; `stage`
остался `OPEN`. Park/wake state machine (Plan 93) под M:N имеет окно
гонки между worker'ом, owning timer'а и main'ом, drain'ящим scope.
**Подход:** проверка `stage == CLOSED` должна быть в loop'е через
`nova_sched_park` либо park-condition должен включать `stage`-state.

#### A2. supervised-drain double-resume
**Симптом:** `fiber stack overflow in slot 0 (access violation in fiber arena)`.
**Тесты:** `supervised_errors`, `supervised_cancel_stress_test`.
**Корень:** main thread в `nova_supervised_step` пытается `mco_resume`
fiber'а, которого уже стащил worker через work-stealing. mco_coro
state становится `MCO_RUNNING` на worker'е, main'овский resume даёт
re-entrance → stack overflow.
**Подход:** drain должен проверять статус через `mco_status` И
`fiber.parent_scope == &main_scope` ДО resume; cross-worker fiber'ы —
ждать через `pending_remote` без попытки resume на main.

#### A3. per-fiber handlers под M:N
**Симптом:** `inner with в spawn перекрывает outer для своего fiber — outer_seen == 111`.
**Тест:** `per_fiber_handlers`.
**Корень:** handler-scope-snapshot save/restore в `nova_supervised_step`
не учитывает worker-context-switch. Когда fiber мигрирует A→B,
handler-state TLS остаётся на A's стороне.
**Подход:** handler-snapshot должен жить в `mco_coro` user_data, не
в TLS; restore — в worker'е перед resume.

### 🔴 Категория B — API/semantic mismatches (0.5-1 dev-day каждый)

#### B1. `fiber_arena_stats` main vs worker aggregation
**Симптом:** `spawn увеличивает slots_active под arena — now_active > before_active` FAIL.
**Тест:** `fiber_arena_stats`.
**Корень:** API возвращает stats per-thread (main thread's arena);
spawn под M:N идёт на worker'a → worker's arena. Main query видит 0.
**Подход:** либо global aggregation (обход `_nova_fw_arena_list`),
либо честно вернуть «not on worker» + добавить `fibers.global_slots_active()`.

#### B2. `time_handler` storage swap
**Симптом:** Time-effect handler не виден из spawn'ed fiber.
**Тест:** `time_handler`.
**Корень:** handler-storage swap в `nova_supervised_step` не
синхронизирован с worker'ами. Аналог A3 для Time-effect.
**Подход:** общий с A3 — handler-storage в user_data.

#### B3. `parallel_for` ordering
**Симптом:** 9 из 14 sub-тестов падают на encoded-log assert'ах.
**Тест:** `parallel_for`.
**Корень:** тесты опираются на single-thread порядок исполнения
(encoded log). Под M:N порядок не специфицирован.
**Подход:** переписать тесты на set-equality (порядок-независимо)
либо честно пометить `NOVA_MAXPROCS=1` для теста ordering.

#### B4. `main_yield` семантика
**Симптом:** `runtime.yield()` на main теряет fiber когда runtime armed.
**Тест:** `main_yield`.
**Корень:** yield на main thread'е роутится в worker scheduler →
no-op либо потеря.
**Подход:** `runtime.yield()` на main = `uv_run(NOWAIT)` (cooperative
turn в main loop'е), без worker-route.

#### B5. `cancel_semantics` propagation
**Симптом:** cancellation через worker boundary с race window.
**Тест:** `cancel_semantics_test`.
**Корень:** `nova_cancel_request` ставит флаг в scope, но worker
fiber может проверять stale snapshot.
**Подход:** atomic-load флага в park-condition + проверка перед
yield.

### 🟢 Категория C — тестовые ассерты под новый контракт (0.1 dev-day)

#### C1. `mn_runtime_smoke` test 1
**Симптом:** assert `!runtime.is_initialized()` initially.
**Корень:** auto-arm в main() меняет контракт — runtime armed с start.
**Подход:** обновить assert на `runtime.is_initialized()` (true) либо
переименовать тест-кейс.

#### C2. `mn_maxprocs_getter` 2/3
**Симптом:** аналог C1.
**Подход:** аналог C1.

## 3. Фазы

### Ф.1 — Категория C (тестовые ассерты, ~0.5 дня)

Лёгкий старт: переписать ассерты в `mn_runtime_smoke.nv` и
`mn_maxprocs_getter.nv` под новый контракт «runtime armed по умолчанию».
Эти fixes — НЕ блокеры (тесты не падают пока флип не активирован),
но без них активация даст немедленные нерелевантные fails.

### Ф.2 — Категория A (runtime races, ~3-4 дня)

A1+A2+A3 — самые тонкие. Каждый требует careful runtime архитектурного
решения. Делать последовательно, с targeted-тестом на каждый. После
каждого фикса — полный `nova test` без флипа (чтобы убедиться, что
существующий M:N путь не сломался).

### Ф.3 — Категория B (API/semantic, ~3 дня)

B1-B5 — функциональные баги M:N edge cases. Многие переиспользуют
решения из A3 (handler-storage).

### Ф.4 — Активация флипа (~0.5 дня)

Раскомментировать `nova_runtime_auto_arm()` в `emit_c.rs::emit_main_wrapper`.
Запустить полный `nova test`. Ожидание: 1071/0/56 (как сейчас) либо
лучше (новые M:N-возможности активны для всех программ).

После Ф.4 unblock'аются:
- [Plan 83.2 Ф.2](83.2-mn-default-flip.md#ф2-миграция--escape-hatch-1-2-дня) — sweep + escape hatch docs.
- [Plan 83.2 Ф.3](83.2-mn-default-flip.md#ф3-spec--docs--benchmark-1-день) — spec D-block + speedup-бенчмарк.

### Ф.5 — closure ([Plan 83.2 Ф.2+Ф.3], ~2-3 дня)

Завершить Plan 83.2 целиком.

## 4. Acceptance

- 9 регрессирующих concurrency-тестов из `[M-83.2-supervised-mn-bugs]`
  проходят под активным флипом (без `runtime.init` в теле теста).
- Полный clang `nova test` после Ф.4: ≥ 1071 PASS / 0 FAIL.
- Plan 83.2 §4 acceptance выполнен: «Compiled-программа без единого
  `runtime.*` вызова использует все CPU при fiber-нагрузке».
- 0 регрессий MSVC (от Plan 82 followup baseline).

## 5. Риск

- **Park/wake state machine (A1, B5)** — наиболее тонкая часть. Может
  потребоваться расширенный аудит D93/Plan 93 protocol invariants.
- **Handler-storage refactor (A3, B2)** — затрагивает фундаментальный
  механизм algebraic effects; ошибка регрессирует handler-aware
  пользовательский код.
- **A2 supervised drain** — требует строгого ownership-протокола
  (кто resume'ит fiber: main или worker). Возможно нужен явный
  spawn-flag «pin-to-main» для fiber'ов, которые НЕ должны мигрировать.
