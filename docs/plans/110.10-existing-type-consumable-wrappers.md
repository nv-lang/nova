## Scope per sub-task

### 110.10.1 — Channel[T] Consumable wrapper

**Why:** `consume ch = Channel.new() { body }` гарантирует close-on-exit
+ drain pending sends при cancel.

**Implementation:**
- `std/concurrency/channel.nv`: add `fn Channel[T] consume @on_exit(outcome ScopeOutcome) -> ()` impl.
- `on_exit` body: call existing `close()` method + drain semantics
  (already implemented в Plan 21 V2).
- Spec: D170 amend — note Channel implements Consumable.

**Tests:**
- POS: `nova_tests/plan110_10/channel_consume_ok.nv` — normal scope completion,
  channel closed.
- POS: `nova_tests/plan110_10/channel_consume_throw_ok.nv` — body throws,
  on_exit fires, channel closed.
- POS: `nova_tests/plan110_10/channel_consume_cancel_ok.nv` — outer cancel,
  on_exit fires within shield budget.

**Acceptance:**
- A110.10.1.a: Channel.on_exit closes channel + drains buffer.
- A110.10.1.b: Three outcomes (success/failure/cancel) all close.
- A110.10.1.c: Zero regression на existing channel tests.

### 110.10.2 — TCP/UDP socket Consumable wrappers

**Why:** `consume socket = TcpStream.connect(addr) { body }` гарантирует
shutdown(rw) + close on exit. Plan 83.12 уже ship'ал raw sockets, но
не Consumable.

**Implementation:**
- `std/net/tcp.nv`: add `fn TcpListener consume @on_exit(...)` + same для
  `TcpStream`. Body calls existing `close()`.
- `std/net/udp.nv`: same для `UdpSocket`.
- Spec: existing Plan 83.12 D-block amend — Consumable status.

**Tests:**
- POS per type: tcp_listener_consume_ok / tcp_stream_consume_ok /
  udp_socket_consume_ok. Each verifies close on success path.
- POS: tcp_stream_consume_panic_ok — panic в body → on_exit fires.

**Acceptance:**
- A110.10.2.a: All three socket types implement Consumable.
- A110.10.2.b: Close idempotent (consume + manual close OK).
- A110.10.2.c: Zero regression на Plan 83.12 fixtures.

### 110.10.3 — JoinHandle Consumable wrapper

**Why:** `consume h = spawn { task() } { body }` гарантирует join (or
explicit drop) before scope exit. Currently `JoinHandle` requires manual
`.join()` или leak. Plan 83.4.2 supervised drain ownership — gate.

**Pre-implementation check:** Plan 83.4.2 status.
- If closed → proceed.
- If open → этот sub-task blocked, document blocker + defer.

**Implementation (if unblocked):**
- `std/concurrency/spawn.nv`: add `fn JoinHandle[T] consume @on_exit(...)`
  impl. Body calls `.join()` ignoring result, OR `.cancel()` if outcome
  = Failure/Cancel.
- Outcome-aware: Success → wait для completion (join), Failure/Cancel →
  cancel child.

**Tests:**
- POS: join_handle_consume_join_ok — normal scope, child joined.
- POS: join_handle_consume_cancel_ok — body throws, child cancelled.

**Acceptance:**
- A110.10.3.a: JoinHandle implements Consumable.
- A110.10.3.b: Outcome-aware (join vs cancel).
- A110.10.3.c: Zero regression на Plan 83.4 supervised tests.

## Status

### V1 partial closure (2026-06-05): scope deferred

**Branch:** `plan-110.10-existing-type-consumable` (off main `fb81c46d8fd`).

**Что landed:**

- 🟢 **Critical bug fix [M-110.x-cleanup-shield-deadline-underflow]**
  (commit `af4e7d96a62`) — discovered durante 110.10.1 implementation
  attempt. Nested consume{} shields were not restoring outer's deadline
  on inner leave; `nv_consume_leave_shield()` only cleared deadline when
  mask reached 0 → outer body resuming saw stale inner deadline →
  inflated/bogus CleanupTimeoutError fires.
  - Runtime: `nv_consume_enter_shield(int) → int64_t` (returns prev);
    `nv_consume_leave_shield(int64_t prev)` (restores).
  - Codegen: threads prev_deadline через fresh local var per consume-block.
  - Spec D196 R4 amend.
  - New regression fixture `nested_shield_deadline_restore_v1_1.nv`.
  - 5/5 key Plan 110 fixtures PASS на release nova-cli.
  - Closes external agent's marker `[M-110.x-cleanup-shield-deadline-underflow]`.

**Что НЕ landed:**

- 🔴 **110.10.1 Channel Consumable wrapper** — implementation attempt
  revealed architectural blocker. Original plan-doc assumed adding
  `external fn ChanWriter[T] consume @on_exit(...)` к existing built-in
  ChanWriter/ChanReader types would work. Exemplar `MutexGuard` requires
  type be declared `consume`:
  ```nova
  export type MutexGuard consume { ptr int }
  export external fn MutexGuard consume @on_exit(...) -> ()
  ```
  No precedent для non-consume type + @on_exit method добавления.
  Making ChanWriter/ChanReader consume types — **breaking change** for
  existing users (Plan 21 V2 ship'нул capability-split BEZ consume modifier).

- 🔴 **110.10.2 Tcp/Udp Consumable wrappers** — same architectural blocker.
  TcpListener/TcpStream/UdpSocket currently не consume types.

- 🔴 **110.10.3 JoinHandle Consumable wrapper** — depends на Plan 83.4.2 +
  same architectural blocker.

**Design questions deferred к V2 plan revision:**

1. **Migration strategy** для making Channel/socket types consume:
   - Option A: opt-in via separate `Channel.new_consume()` constructor returning
     consume-typed variant.
   - Option B: hard-flip к consume + auto-fix tool migrating existing code.
   - Option C: enhance type system к support non-consume types implementing
     Consumable (D194 hot-path uses this pattern, но без full protocol impl).

2. **Pre-implementation type-check** для existing Plan 21 / Plan 83.12 fixtures —
   how many break per option?

3. **Coordination с Plan 91** (std MVP) — std/concurrency module organization.
