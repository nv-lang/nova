> **Создан:** 2026-06-10.  **Статус:** 📋 PLANNED.

## Зачем (recap блокера)

Plan 138.1 приземлился **частично**: `[]T`→`Vec[T]` флип в `type_ref_to_c`
**гейтится** через `generic_type_templates.contains_key("Vec")` —
[emit_c.rs:2055](../../compiler-codegen/src/codegen/emit_c.rs) +
[:4938](../../compiler-codegen/src/codegen/emit_c.rs). `Vec` **не в prelude**,
поэтому юниты, не упоминающие Vec, всё ещё лоуэрят `[]T` в `NovaArray`. Полное
удаление `NOVA_ARRAY_DECL/IMPL` (138.1 Ф.5) заблокировано пятью load-bearing
зависимостями:

1. **Не-универсальный флип** (gate выше) — primitive/Vec-free юниты → NovaArray.
2. **string-слой** `[]T`-producers (`external fn`, C-backed → NovaArray).
3. **parfor** — NovaArray usage.
4. **closure-array** `[]fn` → `NovaArray_void_p*` ([emit_c.rs:4918](../../compiler-codegen/src/codegen/emit_c.rs) explicit exclusion).
5. **Vec→NovaArray bulk-op bridge** — Vec-методы частично маршрутят через `nova_array_*`.

Цель 138.2: снять все пять, завершить retire NovaArray, закрыть D239/D144 **полностью**.

## Фазы

### Ф.0 — Универсальная доступность `Vec` (ядро `[M-138.1-vec-in-prelude]`) (~1-2d)

> **🔴 ИСХОД (2026-06-10): BLOCKED — попытка приземления, откат по HIGH-RISK протоколу.**
> Механика **доказана и работает** для узкого acceptance-критерия, но универсальный
> флип **архитектурно неотделим** от Ф.2-Ф.4 (NovaArray-API reconciliation): включение
> флипа в изоляции каскадит в **40+ регрессий** по корпусу. Дерево оставлено GREEN
> (F0 откатан, 0 tracked-file изменений). **C1 не достигнут на landed-ветке** (флип OFF).
>
> **Доказательство (флип OFF, текущее состояние ветки):** Vec-free юнит
> (`mut x []int = [1,2,3]`) компилируется в `NovaArray_nova_int*`
> (`Nova_Vec____nova_int` = 0 вхождений, `NovaArray` = 112). Это **ожидаемое**
> reverted-состояние.
>
> **Доказательство (флип ON, до отката — acceptance MET):** тот же Vec-free юнит
> лоуэрился в `Nova_Vec____nova_int*` (32 вхождения), residual NovaArray = 2
> (намеренный `Vec.from([]T)` literal-bridge param). 4 gate-сайта проходили
> универсально для prelude-юнитов и корректно degrade-или для `#no_prelude`.
>
> **Механизм (Option A — ЕДИНСТВЕННЫЙ жизнеспособный):**
> - `std/prelude.nv`: `export import std.collections.vec_owned.{Vec, VecIter}`
>   сразу ПОСЛЕ Range re-export (строка 114; порядок важен — vec_owned зависит от Range).
> - `std/collections/vec_owned.nv`: директива `#prelude(core, runtime, collections,
>   protocols)` перед `module` + явный `import std.collections.range.{Range}`
>   (Range живёт только в facade, не в prelude-submodule).
> - 4 gate-сайта `contains_key("Vec")` (emit_c.rs:2055/4938/26100/31637) **НЕ удалять** —
>   они проходят универсально для prelude-юнитов и graceful-degrade для `#no_prelude`.
> - Rust-rebuild НЕ нужен (std .nv читаются с диска в runtime, не include_str).
>
> **Option B (codegen-seed) — ОТКЛОНЁН после spike.** Seeding только template в
> `generic_type_templates` НЕ достаточно: 4 gate-сайта зависят от полного STRUCT
> DEFINITION + всех method-bodies (`generic_type_methods["Vec"]`), которые приходят
> только через import-merge pipeline. Seed только template → dangling
> `Nova_Vec____<T>` + undefined `Nova_Vec____<T>_method_push` → CC-FAIL. Spec-фрейминг
> «B легче» был основан на ложной предпосылке, что template один достаточен.
>
> **5 классов регрессий при флипе ON (требуют Ф.2-Ф.4, НЕ codegen-фикса в Ф.0):**
> 1. **insert/append API divergence** (plan90_1 20→10): []T extend-family (D141)
>    `dst.insert(i, src_ARRAY)` = bulk-insert, но Vec `insert(i, v T)` = single-elem.
>    Одно имя, несовместимая семантика → нужен либо bulk-overload на Vec, либо переписка тестов.
> 2. **[]u8 literal int64-stride** (plan118_1 11→6, buffers) = известный
>    `[M-138.2-vec-u8-literal-stride]` (literal sized-int element coercion).
> 3. **Vec-of-record write-index `_p` typedef bug** (plan138_1 10→9): `v[i]=Point{...}`
>    эмитит `(Nova_Point_p)(tmp)` — typedef не существует, должно быть `Nova_Point*`.
>    Narrow codegen-баг в MutIndex write-path.
> 4. **map_literals каскад** (28→9): map-литералы строят []T-backed bucket-arrays.
> 5. **SHADOW collision** (plan123_3_1 E7310): user `type Vec { x, y }` (non-generic)
>    коллизит с prelude `Vec[T]`; prelude-методы выживают и arity-check'аются против
>    user-Vec. Расширить W_PRELUDE_SHADOW methods-skip на user-redeclared receiver type.
>
> **РЕКОМЕНДАЦИЯ:** приземлять Ф.0 + Ф.2 + Ф.3 + Ф.4 как ОДИН атомарный multi-day
> unit; сначала фиксить `[M-138.2-vec-u8-literal-stride]` + Vec-of-record `_p` typedef
> bug, потом флип. Re-create `nova_tests/plan138_2/t3_universal_novec_unit.nv` когда
> флип реально приземлится.

Цель: `[]T` лоуэрится в `Vec[T]` в **КАЖДОМ** юните, не только где Vec упомянут.
Снять gate `contains_key("Vec")` (emit_c.rs:2055 + 4938).

**Две механики (выбрать после spike):**

- **Option A — Prelude integration.** Добавить в `std/prelude.nv`:
  `export import std.collections.vec_owned.{Vec, VecIter}`. Дать `vec_owned.nv`
  директиву `#prelude(core, runtime, errors)` для разрыва цикла
  prelude→vec_owned→prelude (зеркало `range.nv`). **Риск:** как с Range
  (Plan 62.D bis-1) может вскрыть latent codegen-баги (match-inference,
  closure-capture, W_PRELUDE_SHADOW). **Бонус:** `Vec` становится видимым именем
  без import (DX-плюс).
- **Option B — Codegen-seed (рекомендуется первым).** Безусловно регистрировать
  Vec[T] template на codegen-init (грузить шаблон из vec_owned в каждый юнит),
  чтобы gate проходил универсально **без** prelude-auto-import семантики. Легче по
  cycle-риску; `Vec` как имя не авто-видим (юзеры пишут `[]T` — что и нужно для
  сахара). Точечно решает реальную потребность (lowering `[]T`), не трогая
  prelude-видимость.

**Рекомендация:** spike Option B (легче, целит ровно в потребность); fallback на A
если codegen-seed не дотягивается до шаблона чисто. В обоих случаях — убрать gate.

**Acceptance Ф.0:** Vec-free юнит (`fn main(){ mut x []int = [1]; x.push(2) }` без
`import ...Vec`) генерит `Nova_Vec____nova_int*`, не `NovaArray`. Targeted suites
строятся.

**Commit:** `feat(plan138.2 Ф.0): universal []T->Vec[T] (remove contains_key gate)`

### Ф.1 — String-слой → **SUBSUMED by Plan 139** (решение 2026-06-10: 139-first)

> **Развилка решена (139-first, без двойной работы).** Вместо lightweight-переписки
> `[]T`-producers с сохранением примитива `nova_str`, string-слой закрывается
> **полностью** через **Plan 139** (`str` → Nova value-record `{ ptr *ro u8, len int }`).
> Plan 139 Ф.2 переписывает `@to_bytes/@to_chars/@split/@as_bytes/from_bytes_*` на
> Nova → `Vec` как часть превращения str в value-тип. `@as_bytes` zero-copy при этом
> **не требует** нового `as_ptr`-примитива — поле `@ptr` уже доступно методам str.
>
> Поэтому эта фаза в 138.2 **не выполняется** — см. `docs/plans/139-str-as-nova-value-type.md`.
> Канонический порядок: **138.2 Ф.0 → 139 Ф.0-Ф.2 → 138.2 Ф.2-Ф.4**.

## Followups

- `[M-138.2-closure-array-vec]` — `[]fn` → `Vec[fn]` если в Ф.2 отложено.
- `[M-138.2-codepoint-cursor]` — O(n) codepoint-iterator примитив для `to_chars` (вместо O(n²) `@char_at`).
- `[M-138.2-fixed-array]` — `[N]T` FixedArray (наследие 138.1, вне scope).
- ✅ `[M-138-rawmem-bulk-ops]` — DONE (commit f0bec0e6717): `@realloc_to`/`@append`/`@clone` →
  `RawMem.copy_nonoverlapping`; `@insert`/`@remove` shift → `RawMem.copy` (memmove). GC-safe casts.
- ✅ `[M-138-vec-bulk-parity]` — DONE (commit f0bec0e6717): `@copy_from`/`@copy_within` (RawMem.copy),
  `@fill`/`@append_zero` (RawMem.fill), `@compare` (RawMem.compare, normalized -1/0/+1) добавлены на Vec.
  plan138_2 t1 2/2. **PREREQUISITE для Ф.3 закрыт.**
- ✅ `[M-138-rawmem-dedup-fill]` — DONE (commit 43fb6a736ea): `write_bytes` удалён, `fill` остался.
- 🟡 `[M-138-mutindex-rename]` — PARTIAL: spec D240 desugaring-prose исправлен на `a.@index(key,val)`
  (commit 94d39606d6a), но Vec-метод **остался** `@index_set` — агент задокументировал, что `v[i]=val`
  codegen-инлайнится (ExprKind::Index, не диспатчит на именованный метод), потому 3-й `@index`-overload
  косметика. Реальный rename метода — открыт (для D240-конформности декларации).
- 🆕 `[M-138.2-vec-u8-literal-stride]` — `[]u8 = [1,2,3]` литерал хранит int64-страйд, не 1-байтовый u8
  (typed-storage gap для sized-int литералов, 138.1 Ф.2). `Vec[u8]` через `push(_ as u8)` хранит корректно;
  баг только в array-литерале. Чинить в literal-codegen (sized-int element coercion).
- 🆕 `[M-138-getmut-rename]` — `@get_mut(i) -> Option[*mut T]` → `mut @get(i)` (user request; mut-receiver
  overload поддержан Plan 135). Различие: возвраты разные (`Option[T]` vs `Option[*mut T]`) — нужен тест,
  что `infer_expr_c_type` tiebreak'ает возврат по recv_mutable. Рекомендация была «оставить @get_mut»
  (Rust precedent, no footgun), но user подтвердил mut-overload — применить с differing-return-type тестом.
