# Plan 70.4: f32↔f64 distinction в codegen mono'd generics

> **Создан:** 2026-05-19. **Статус:** 📋 proposed, не начат.
> **Приоритет:** P2 (lower than 70.3 — partial distinction уже есть).
> **Трудоёмкость:** ~1-2 dev-days. **Родительский план:** [Plan 70](70-no-silent-nova-int-fallback.md)

---

## Зачем

`f32` и `f64` имеют **distinct scalar C типы**:

```c
// nova_rt/nova_rt.h
typedef double   nova_f64;
typedef float    nova_f32;
```

Scalar codegen path correct: `type_ref_to_c("f32") → "nova_f32"`,
`type_ref_to_c("f64") → "nova_f64"` (emit_c.rs:2672-2673).

**НО** generic-container mangling collapse'ит обе в `nova_f64`:

```rust
// compiler-codegen/src/codegen/emit_c.rs:2826
"f64" | "f32" => "NovaArray_nova_f64*".into(),    // hardcoded f64-only
```

Result: `[]f32` и `[]f64` оба → `NovaArray_nova_f64*` — структурно
identical. ABI mismatch (f32=4byte, f64=8byte) → silent data corruption
если `[]f32` literal обрабатывается как `[]f64`.

## Scope analysis

Different scope from 70.3 потому что:

1. **Scalar level — already distinct ✅** (`nova_f32` vs `nova_f64` C types
   exist в runtime headers).
2. **Container/generic level — partial collapse:**
   - `[]f32` → `NovaArray_nova_f64*` (HARDCODE, не через
     `compute_generic_type_c_name`)
   - `Option[f32]` — likely correct via `NovaOpt_nova_f32` mangling
     (uses type_ref_to_c scalar path)
   - `Map[f32, V]` — likely correct (same)
3. **Hot trigger:** array-of-f32 для GPU buffers, audio samples, ML
   tensors — performance-critical paths где silent f64 promotion даёт
   2× memory cost.

## Архитектурное решение

### Option A: Remove hardcode, use proper mangling

`emit_c.rs:2826` change:

```rust
// Before:
"f64" | "f32" => "NovaArray_nova_f64*".into(),

// After (use mangled name via type_ref_to_c → compute_generic_type_c_name):
"f64" => "NovaArray_nova_f64*".into(),
"f32" => "NovaArray_nova_f32*".into(),
```

Plus ensure `NovaArray_DECL(nova_f32)` macro instantiated в runtime.

**Pros:**
- Minimal change (2 lines)
- Same architectural pattern как user-types

**Cons:**
- Runtime helper instances для nova_f32 (array push/pop/etc) — needs
  to be generated параллельно nova_f64 versions

### Option B: Audit type_ref_to_c bypass paths

Find ALL spots где `f32`/`f64` collapsed в hardcoded paths beyond line
2826. Likely a few similar patterns в other generic emit.

**Pros:** Comprehensive
**Cons:** Larger scope (~5-10 sites? unaudited)

**Рекомендация: Option A** + audit pass для других hardcoded f32→f64
collapse paths.

## Risks

| Risk | Mitigation |
|---|---|
| **Runtime helpers explosion** — separate NovaArray macros для f32 | Use share-implementation pattern (similar к int/char Plan 70.3) |
| **Performance regression** для existing f64-heavy code | Distinct mangling — zero-cost (different C name, same operations) |
| **Other hardcoded paths missed** | Ф.0 audit script comprehensive grep |
