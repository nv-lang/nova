## Ф.1 — size_of/align_of для composite types

**Status:** 🟢 V1 LANDED 2026-06-02 (commit `2d552ea2742`).

**Scope:** Extend `compiler-codegen/src/const_fn_eval.rs::type_size_or_align`
от primitives-only до composite types БЕЗ TypeDecl lookup:
- Tuples — C struct layout с natural alignment + tail-pad.
- FixedArray `[N]T` — `N * size_of(T)`, align = elem align.
- Array `[]T` — slice ABI: 16 bytes (pointer + length), align 8.
- Unit `()` — size 0, align 1.
- Readonly — transparent wrapper.
- Primitive table now (size, align) tuples — `str = (16, 8)`.

**Tests (release nova-cli):**
- POS: `size_of_tuple_ok` (4 tuple cases), `size_of_fixed_array_ok`
  (FixedArray/Array/Unit), `size_of_nested_composite_ok` (nested layouts),
  `size_of_padding_ok` (7 padding/alignment edge cases — added 2026-06-02
  post-rename для documenting layout semantics).
- NEG: `size_of_named_record_neg` (Named user-defined records still V2).

**Acceptance — A36 + A38:**
- A36: size_of/align_of returns correct layout values для
  tuples/arrays/Unit/Readonly.
- A38: Padding/alignment edge cases verified — inner-pad/tail-pad/big-gaps/
  no-pad/multi-step/unit/mixed-sizes layouts всё correct.

**User-facing concept doc:** [docs/size-of-align-of.md](../size-of-align-of.md) —
explanation что эти intrinsics возвращают, зачем нужно, layout semantics
композитных типов с padding examples, Rust comparison.

**Still V2 (требует TypeDecl access):**
- Named records `Type { f1, f2 }` — followup `[M-114.4.4-trampoline-named-types]`.
- Named sum-types (tagged unions) — same.
- Generic instantiations `Option[int]` — same.

## Ф.2 — closure-returning const fn captures outer consts

**Status:** 🟢 V1 LANDED 2026-06-02 (commit `879ebc11ba6`).

**Scope:** Расширить closure-from-const-fn V4.3 для body форма
`FnBody::Block { stmts: all Stmt::Const, trailing: closure_literal }`.

```nova
fn make_thing(const base int) -> const fn(int) -> int {
    const offset = base + 10
    fn(x int) -> int => x + offset
}

ro t5 = make_thing(5)   // offset = 15, spec body: x + 15
ro t100 = make_thing(100) // offset = 110, spec body: x + 110
```

**Implementation:**
- `const_fn_closure.rs::has_closure_returning_body` допускает Block form.
- `extract_outer_consts_and_closure` извлекает Stmt::Const decls + trailing closure.
- `generate_closure_spec` eval'ит outer consts в порядке с running subst
  map (host params + prior outer consts), результат добавляет в map.
- `subst_then_eval` reuses `try_literal_to_value`.
- `types/mod.rs::check_const_fn_decl` extended для Block body с outer
  Stmt::Const + trailing closure — scope = host params + outer consts +
  closure params.

**Tests:**
- POS: `closure_outer_const_ok` (single outer const), `closure_outer_chained_ok`
  (3 chained outer consts referencing each other).
- NEG: `closure_outer_not_const_neg` (`ro X = ...` runtime — rejected).

**Acceptance — A37:** closure body captures outer Stmt::Const decls;
each const evaluated с host params substituted; chained references work.

**Parser limitation (NOT a simplification):**
- ClosureLight `|x| body` после `const X = ...` Stmt парсится как
  binary OR. Workaround: explicit ClosureFull `fn(x T) -> R => body`.
- Followup `[M-114.4.4-closure-light-after-const-stmt-parser]` для
  parser disambiguation (e.g., require `|x|` only as expression start
  token, not after Stmt boundary).

## Ф.3 — generic const fn first-class use (DEFERRED)

**Status:** 🟡 DESIGN PENDING — not shipped V4.4.

**Goal:**
```nova
fn id[T](const x T) -> const T => x
fn apply_i(f fn(int) -> int, x int) -> int => f(x)

test {
    ro r = apply_i(id, 7)   // Want: id__trampoline_int generated
    assert(r == 7)
}
```

**Blocker — design question requires user input:**

Three viable approaches, each requires substantial cross-cutting work:

1. **Parser support for TurboFish-as-fn-value.** `id[int]` (без postfix
   continuation) currently parses as Index expression. To use as fn
   value: parser amendment to allow `id[int]` in expression position
   when context indicates fn-typed slot. Requires parser changes +
   ambiguity resolution с array indexing.

2. **HOF context type inference.** At trampoline pass site, look up
   callee's expected fn-type from registry, infer generic T from there.
   Requires:
   - Fn signature registry built before trampoline pass.
   - Type inference from HOF arg position → expected `fn(T1,..) -> R`
     → solve for generic T.
   - Substitute T in TypeRef positions (param types, return) + body
     intrinsics (`size_of[T]`).

3. **Explicit type annotation.** `ro f fn(int) -> int = id` —
   annotation provides target type, trampoline pass uses it. Requires
   type-checker integration to expose annotation at pass time.

**Recommended path:** #2 (HOF type inference) — most natural UX,
no parser/syntax changes. Requires ~300 LOC trampoline pass extension.

Tracked: `[M-114.4.4-trampoline-generics]`.
