# Plan 91.11 — StringBuilder cleanup + rename family + zero-copy steal + parser multi-line chain

> **Статус:** 🟢 IN PROGRESS 2026-05-30
> **Parent:** [Plan 91 — std MVP for 0.1](91-stdlib-mvp-for-0.1.md)
> **Sibling:** [Plan 91.10 — D163 retract](91.10-d163-retract-capability-syntax.md)
> **Trigger:** User-driven cleanup: «optimize StringBuilder using all latest updates».
> При работе surfaced 5 ortho-issues — все включены в этот sub-plan.

## Зачем

StringBuilder API накопил debt: дублирующие методы (`@append_bytes` vs `@append`),
misnamed verb (`@to_str` для consume-finalize вместо `@as_str`), missing
optimisations (copy вместо view, multi-call вместо chain, no pre-reserve в loop).

Параллельно: `[]T` API family `extend_from`/`insert_from`/`copy_from` —
семейство по dispatch-детали (`_from` суффикс = "takes source array"), а не
по semantic class. Verbs не обнажали что делает метод (append? overwrite?
insert?). Naming gap снижал API discoverability.

Параллельно: Plan 91.7 D181 `-> @` fluent return design подразумевал
multi-line chain (`arr.method()\n .method2()`), но parser терминировал
expression на newline → парс-ошибка. Half-baked feature.

Параллельно: zero-copy опт для `StringBuilder.@as_str()` blocked отсутствием
`from_bytes_unchecked_steal(consume []u8) -> str` primitive — был только
copy-вариант `from_bytes_unchecked(readonly []u8)`.

Параллельно: bug `[M-mono-method-var-boxed-leak]` surfaced при попытке
использовать `[]T.count_of(pred)` (Plan 91.8c) в StringBuilder — mono'd
method body inherit'ил caller's heap-promotion map → CC-FAIL «undeclared
_box_count».

## Scope

**Входит (5 cohesive issues):**

### (1) StringBuilder API cleanup (`std/runtime/string_builder.nv`)

- **`@append_bytes(readonly []u8)` removed** — merged into `@append([]u8)` overload.
  After Plan 91.10 D141 amend `[]T.append(src)`, `@append_bytes` redundant.
  Single API: `@append(str | char | []u8)` — three implementations dispatched
  by arg type. Cleaner mental model.

- **`@plus(s)`/`@plus(c)` removed** — `+` operator on StringBuilder через D46
  desugaring был ergonomic shortcut. Removed because: (a) mutable-receiver
  value-overload visually misleading (`sb + "x"` looks pure), (b) `.append(x)`
  shorter+explicit after Plan 91.7 fluent-return. `str + str` оператор
  сохраняется (str-side `@plus` остаётся, pure concat).

- **`@to_str() → @as_str()` rename** — semantic precision. `to_X` обычно
  означает "convert/copy"; `as_X` — "view/cast/reinterpret". После zero-copy
  steal (Plan 91.11 part 2), `@as_str()` reuses buf data ptr — semantically
  view, не copy. Rename signals "no allocation" (when cap > len).
  Propagation: 22+ файлов в std + nova_tests + string interp codegen
  (`Nova_StringBuilder_consume_to_str → Nova_StringBuilder_consume_as_str`).

- **Performance optimisations:**
  - `.new() => with_capacity(16)` — DRY one-liner
  - `.with_capacity(n) => { buf: []u8.with_capacity(n) }` — direct intrinsic
  - `.from(c char) => with_capacity(4).append(c)` — 4× меньше pre-alloc
  - `@clone => with_capacity(len).append(@buf)` — chain
  - `@starts_with/@ends_with`: `as_bytes()` view + `@buf[a..b].compare(...) == 0`
    (memmove memcmp вместо byte loop)
  - `@append(s str)`: `s.as_bytes()` (view) вместо `s.to_bytes()` (copy)
  - `@append_repeat`: pre-reserve(total) → 1 realloc вместо O(log n)
  - `@append(c char)`: multi-line push chain (D181 fluent) для UTF-8 byte
    sequence emit
  - `@as_str` использует zero-copy steal (см. part 2)

### (2) Zero-copy steal API

- **New external `str.from_bytes_unchecked_steal(consume []u8) -> str`**
  (`std/runtime/string.nv`).
- **Runtime: `nova_str_steal_bytes`** (`compiler-codegen/nova_rt/string_builder.h`):
  - Happy path: `cap > len` → write `'\0'` at `data[len]` in-place + return
    nova_str wrapping существующий pointer (zero-copy)
  - Fallback: `cap == len` → alloc(len+1) + memcpy + null-term
- **`@as_str()` использует steal** — Plan 91 stdmvp Ф.2.6 followup.
- **Compiler dispatch** в `emit_c.rs` (4 sites): method routing, type infer.

### (3) Rename family (D141 amend, 2026-05-30)

- `[]T.extend_from(src) → []T.append(src)` (bulk append, grows)
- `[]T.insert_from(i, src) → []T.insert(i, src)` (bulk insert, grows +
  memmove tail)
- `copy_from` остаётся (Rust-style `copy_from_slice`; теперь isolated имя)
- `copy_within`, `fill`, `reserve`, `truncate`, `compare` — без изменений

**Naming rationale (в `simplifications.md`):**
`_from`-family склеивал ops по dispatch-детали ("берёт source array"), не
по semantic class. После rename:
- append-family: `push(v)` + `append(src)` — единый verb для add-to-end
- insert-family: `insert(i, src)` — overload с будущим `insert(i, v T)`
- overwrite-family: `copy_from(src)` + `fill(v)` — equal-len mutation
- move-family: `copy_within(...)` — internal copy

**Migration:** 33 файла затронуто. Compiler (parser/types/codegen/lints/ast +
array.h runtime macros), std (string_builder.nv), 12 test fixtures (10
`git mv` + 2 lint + 1 plan91_7).

### (4) Parser multi-line chain continuation

- `parse_postfix` теперь lookahead через `Newline`/`Semicolon`: если next
  значимый token — `.` (member access), skip whitespace + continue chain.
  Иначе terminate (statement-end).
- **Enables:**
  ```nova
  @buf.push((cp >> 6) as u8)
      .push((cp & 0x3F) as u8)
  ```
- **Conservative scope:** только `.` postfix. `?.`/`!.`/wrapping call args
  multi-line continuation — separate followup
  `[M-parser-multiline-chain-extended]`.
- **Rationale:** Plan 91.7 D181 fluent-return design implied multi-line
  chains (idiomatic style для `-> @` API), grammar не закреплял — gap.
  Cross-language matches Swift/Kotlin/Scala (leading-dot continuation).

### (5) `[M-mono-method-var-boxed-leak]` closed

- `emit_monomorphized_method` добавил `save/restore var_boxed` map
  (`std::mem::take` save at entry + restore before body emit — mirror
  existing `emit_lambda_body` pattern).
- **Root cause:** mono'd method bodies emitted via worklist drain inherited
  caller's `var_boxed` (heap-promotion для closure captures) → identifier
  `count` в mono'd `count_of` body resolved через caller's `(*_box_count)`
  → CC-FAIL «undeclared _box_count».
- **Discovered при:** попытка использовать `@buf.count_of(pred)` в
  `StringBuilder.@char_len` (Plan 91.8c generic API).

**НЕ входит (followup'ы):**

- `@char_len` через `@count_of` (требует `import std.sort` который ломает
  import cycle в string_builder). Followup
  `[M-stringbuilder-count_of-after-sort-no-prelude]`.
- `[M-91.11-builder-chaining-stale-into]` — `nova_tests/runtime/builder_chaining`
  использует stale `.into()` (pre-existing brittleness, не каскад от моих
  changes — Plan 109 cleanup gap).
- `[M-parser-multiline-chain-extended]` — multi-line continuation для `?.` /
  `!.` / wrapping call args.
