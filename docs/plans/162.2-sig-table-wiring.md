# Plan 162.2 — Wiring sig_table в compile path + lazy cross-module fn lookup

> **Создан:** 2026-06-16. **Статус:** 🟡 IN PROGRESS. P2.
> **Дочерний план:** [Plan 162](162-rust-model-module-resolution.md) Ф.1 — полный two-pass.
> **Зависит от:** Plan 162.1 (ModuleSigTable infrastructure — ✅ в main).
> **Worktree:** `D:\Sources\nv-lang\nova-p162c`, ветка `plan-162c-lazy-typecheck`.

## Текущее состояние (после Plan 162.1)

`ModuleSigTable` + `collect_all_signatures()` + `is_known_type()` + `is_known_fn()` — в main.
Но:
- `TypeCheckCtx` создаётся через `build()` → **пустая sig_table** (`ModuleSigTable::new()`).
- `collect_all_signatures()` нигде не вызывается в продакшн-пути.
- `is_known_fn()` помечена `#[allow(dead_code)]` — **не используется**.
- Два D-block конфликта исправлены: D285→D291 (cycle guard) + D290→D292 (ModuleSigTable) в 07-modules.md (Ф.0 DONE).

## Цель

Замкнуть цепочку: sig_table реально используется при typecheck, а не только лежит в структуре.

### Ф.1 — Аудит compile pipeline
Найти точно:
- где вызывается `TypeCheckCtx::build()` — это точка вставки `collect_all_signatures`
- как `resolve_imports_inline` вызывается от entry point
- где в `types/mod.rs` резолвится вызов функции (ExprKind::Call или аналог)
- где объявлена `is_known_fn` и как она сейчас не используется

### Ф.2 — Wire collect_all_signatures в compile path
В точке где сейчас `TypeCheckCtx::build()`:
1. Вызвать `collect_all_signatures(entry_path, lib_paths, ...)` → `sig_table`
2. Передать в `TypeCheckCtx::build_with_sig_table(sig_table)`
3. Fallback: если `collect_all_signatures` возвращает `Err` — `ModuleSigTable::new()` (backward compat)
4. `cargo check` — zero new errors

### Ф.3 — Wire is_known_fn в function call resolution
В `types/mod.rs` в месте где резолвится fn call:
- Когда local lookup не находит имя — добавить: `self.is_known_fn(name)` как fallback
- Убрать `#[allow(dead_code)]` с `is_known_fn`
- Смысл: если sig_table знает эту fn (из транзитивных импортов) — не ошибка
