# Plan 104.10 — LSP V2: production gap closure

> **Статус:** 📋 proposed 2026-06-19
> **Приоритет:** P1 (критично для пользователей) / P2 (важно, но терпимо)
> **Depends:** Plan 104 ✅ (все sub-plans закрыты)

## Контекст

Plan 104 (V1) закрыл базовый feature set (diagnostics/hover/goto/completion/rename/format/quick-fixes).
Этот план закрывает gap'ы между V1 и уровнем rust-analyzer / gopls — то, что пользователь
замечает при ежедневной работе.

## Фазы по приоритету

### P1 — Критично (ощущается сырым без этого)

#### Ф.1 — Cross-file goto-definition
**Маркер:** `[M-104.2-cross-file-goto]`

Сейчас: hover и goto-def работают только в рамках одного файла. Ctrl+Click на функцию из
другого модуля → ничего.

**Что нужно:**
- После `resolve_imports_inline` импортированные items попадают в `module.items` с оригинальными
  span'ами из других файлов. Нужно хранить маппинг `(item_span → source_file_path)`.
- `compute_goto_definition`: если span принадлежит другому файлу → вернуть `Location` с тем URI.
- `resolve_imports_inline` уже даёт нам `PeerFile.path` для каждого файла — нужно пробросить эту
  информацию в результат резолвера.

**Оценка:** ~2 dev-day

#### Ф.2 — Symbol cache (parse + import-inline per URI)
**Маркер:** `[M-104.2-symbol-cache]`

Сейчас: каждый hover заново парсит + запускает `resolve_imports_inline` (тяжёлая операция —
инлайнит 450+ prelude items). На большом файле ~50–100ms задержка.

**Что нужно:**
- В `WorkspaceState` добавить `symbol_cache: DashMap<Url, CachedModule>` где
  `CachedModule { module: Module, items_start: usize, env: ModuleEnv, version: i32 }`.
- Инвалидировать при `didChange` (по version).
- `compute_hover` и `compute_goto_definition` берут из кеша если version совпадает.

**Оценка:** ~1 dev-day

#### Ф.3 — Completion через compiler API (пользовательские типы и методы)
**Маркер:** `[M-104.9-dynamic-method-completion]`

Сейчас: методы в completion — статические таблицы в `completion.rs`. Пользовательские типы
и их методы не предлагаются после `.`.

**Что нужно:**
- После `resolve_imports_inline` + `check_module` у нас есть `ModuleEnv.fns` и `ModuleEnv.types`.
- В `compute_completions` при контексте `MethodDot`: резолвить тип объекта перед `.` через
  `infer_rhs_type` / `find_local_var_at` → имя типа → найти все методы с matching receiver
  в `module.items` → вернуть как CompletionItems.
- Статические таблицы остаются как fallback для stdlib-типов.

**Оценка:** ~2 dev-day

### P2 — Важно, но терпимо

#### Ф.4 — Hover на member access (r.start → тип поля)
**Маркер:** `[M-104.2-local-var-type-inference]` Variant A followup

Сейчас: `ro r = 0..=5; r.start` — hover на `start` ничего не показывает. Вариант B
(infer_rhs_type) даёт тип самой переменной (`Range`), но member access не резолвится.

**Что нужно:**
- Расширить `ModuleEnv` полем `expr_types: HashMap<Span, Ty>` (Variant A).
- В type-checker заполнять при инференции каждого выражения.
- В `find_ident_in_bodies_from`: если cursor на `obj.field` — взять тип `obj` из
  `expr_types[obj.span]` → найти поле/метод в соответствующем type decl.
- Это крупный лифт в compiler-codegen.

**Оценка:** ~4 dev-day (крупнейшая задача в этом плане)

#### Ф.5 — Rename через symbol table (не regex)
**Маркер:** `[M-104.6-symbol-table-rename]`

Сейчас: rename по word-boundary regex — переименует `foo` во всех файлах, даже в другом scope.

**Что нужно:**
- `compute_rename`: сначала resolve symbol at cursor (`resolve_symbol_at`) → получить
  `(name, decl_span, decl_file)`.
- Find-references: только те вхождения, которые резолвируются к той же `decl_span`.
- Требует Ф.1 (cross-file resolution) как prerequisite.

**Оценка:** ~1.5 dev-day (после Ф.1)

#### Ф.6 — organize imports
**Маркер:** `[M-104.5-organize-imports]`

Сейчас: `source.organizeImports` заявлен в capabilities, но тело — no-op.

**Что нужно:**
- Собрать все `import` statements в файле.
- Удалить неиспользуемые (те, чьи символы не встречаются в body).
- Отсортировать оставшиеся по алфавиту.
- Вернуть TextEdit заменяющий блок импортов.

**Оценка:** ~1 dev-day

### P3 — Полировка

#### Ф.7 — Inlay hints (типы переменных без аннотации)
Маркер: нет (новый)

`ro x = foo()` → показывать `: int` inline как серый текст справа от имени (как в rust-analyzer).
Требует Ф.4 (Variant A expr_types) или хотя бы Variant B для простых случаев.

**Оценка:** ~1.5 dev-day (после Ф.4)

#### Ф.8 — Semantic tokens
Маркер: нет (новый)

Более точная подсветка через LSP semantic tokens (различать локальную переменную от типа от
функции даже без tree-sitter). Требует symbol resolution.

**Оценка:** ~2 dev-day

#### Ф.9 — Signature help по типу receiver
**Маркер:** `[M-104.2-signature-type-dispatch]`

Сейчас: при `x.foo(│)` выбирается первый overload с именем `foo`, независимо от типа `x`.

**Оценка:** ~1 dev-day (после Ф.4)

#### Ф.10 — Incremental references index
**Маркер:** `[M-104.4-refs-incremental-index]`

Сейчас: find-references сканирует все `.nv` файлы целиком per-request.
V2: инкрементальный индекс обновляемый при `didChange`.

**Оценка:** ~2 dev-day
