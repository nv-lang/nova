### 0.2 Ключевые факты, определяющие архитектуру

**F1. `Span` УЖЕ содержит `file_id: FileId`** (`diag.rs:27-33`):
```rust
pub struct Span { pub start: usize, pub end: usize, pub file_id: FileId }
```
`FileId` = u32, `MAIN_FILE_ID = 0`. Есть `SourceMap` (`diag.rs:78-136`): `FileId → (path, source)`,
методы `register()`, `get()`, `path_for()`.

**F2. `PeerFile` хранит провенанс** (`ast/mod.rs:71-104`): `path: PathBuf`, `file_id: FileId`,
`items_here: Vec<Item>`, `is_entry_module: bool`, `module_name: Vec<String>`. После
`resolve_imports_inline` поле `module.peer_files` **populated** — у каждого файла есть path+file_id.

**F3. НО nova-lsp теряет провенанс.** hover/goto используют `nova_codegen::parser::parse(src)`
(single-string), который зовёт `parse_with_file_id(src, MAIN_FILE_ID=0)` (`parser/mod.rs:10359`).
Все спаны входного файла → `file_id=0`. При `resolve_imports_inline` инлайненные items парсятся
с **distinct** `file_id` (`imports.rs:685`), но nova-lsp нигде не строит `SourceMap` (grep пуст)
→ не может отобразить `file_id → path`. `resolve_imports_inline` возвращает `Result<()>`, не SourceMap.

**F4. `ModuleEnv` НЕ содержит per-expression типов** (`types/mod.rs:62-84`): только `types`, `fns`
(с overload-Vec), `consts`, proven-сайты. Checker вычисляет типы выражений при обходе, но **не
сохраняет** карту `Span → Ty`. `Ty` enum — `types/mod.rs:18-53` (Int/Float/Str/Bool/Named/Array/Tuple/Func/…).

**F5. Нет workspace `Cargo.toml`** в корне. Сборка: `cd nova-lsp && cargo build --release`.
Тесты nova-lsp: `cargo test` (НЕ требует `NOVA_GC_LIB_DIR` — это только для nova_tests C-codegen).
Unit-тесты = прямой вызов функций; integration = spawn бинаря + JSON-RPC по stdio.

**F6. `references` УЖЕ cross-file** (`symbols.rs:189`, `collect_nv_files` `symbols.rs:761`). Gap не
в «single-file», а в (a) full-FS-скан per-request (O(n)) и (b) regex без semantic-фильтра.

### 0.3 Корневая зависимость → порядок фаз

Многие фичи (completion типов, hover на `r.start`, inlay hints, signature dispatch) упираются в
**одну** недостающую вещь: **карту типов выражений** (`expr_types: Map<Span, Ty>`). Поэтому план
делится так: сначала фундамент (provenance + expr_types), затем фичи поверх него.

## BLOCK A — ФУНДАМЕНТ

### Ф.0 — Source provenance в nova-lsp (file_id → path map)

**Проблема (F3):** nova-lsp не знает, из какого файла пришёл инлайненный item → cross-file
goto/hover невозможны.

**Дизайн (production, без SourceMap-рефакторинга компилятора):**
Использовать УЖЕ заполняемое `module.peer_files` (F2). После `resolve_imports_inline` каждый
`PeerFile` несёт `(file_id, path)`. Построить в nova-lsp `FileIdMap = HashMap<FileId, PathBuf>`
обходом `module.peer_files`. Этого достаточно — НЕ требует менять компилятор.

**НО** есть тонкость: входной файл парсится через `parse(src)` (file_id=0) ДО инлайна, а
`resolve_imports_inline` ре-парсит peers с distinct id. Нужно убедиться, что spans
оригинальных items входного файла консистентны с тем, что в `peer_files` для entry-модуля.

**Шаги:**
1. **Новый модуль `nova-lsp/src/provenance.rs`:**
   ```rust
   pub struct ResolvedModule {
       pub module: Module,
       pub items_start: usize,
       pub file_map: HashMap<FileId, PathBuf>,  // из peer_files
       pub env: Option<ModuleEnv>,              // заполняется Ф.1/Ф.2
   }
   pub fn resolve_module_for(path: &Path, src: &str) -> ResolvedModule
   ```
   Внутри: parse → `resolve_imports_inline` (catch_unwind как сейчас) → собрать `file_map` из
   `module.peer_files` (`pf.file_id → pf.path`). Entry-file → его реальный path.
2. **Хелпер `span_to_location(span, file_map, fallback_uri) -> Location`:** если `span.file_id`
   есть в `file_map` → `Url::from_file_path(path)`; иначе `fallback_uri` (текущий файл).
3. **Проверить consistency entry-file spans.** Если `parse(src)` даёт file_id=0, а `peer_files`
   для entry даёт file_id=N≠0, добавить в `file_map` маппинг `0 → entry_path` И `N → entry_path`.
   (Оба указывают на тот же файл — безопасно.)
4. Рефактор `hover.rs` и `goto_definition.rs` на использование `resolve_module_for` вместо
   inline-парсинга (Ф.3/Ф.4 завершат переключение).

**Спека/доки:** D-блок D304 (см. Ф.14) — «LSP source provenance via peer_files».

**Тесты (`nova-lsp/src/provenance.rs` unit + integration):**
- POS: `resolve_module_for` на файле с `import` → `file_map` содержит ≥2 записи (entry + prelude).
- POS: `span_to_location` для спана из другого файла → Location с ДРУГИМ URI.
- POS: `span_to_location` для спана entry-файла → Location с тем же URI.
- NEG: спан с file_id, которого нет в map → fallback на текущий URI (не паника).
- NEG: файл без импортов → `file_map` ≥1 запись (только entry), goto работает single-file.
- EDGE: битый импорт (resolve паникует) → catch_unwind, `file_map` хотя бы с entry, не паника.

**Критерии приёмки Ф.0:**
1. `resolve_module_for` возвращает корректный `file_map` (verified обходом `peer_files`).
2. cross-file спан → правильный URI; entry-спан → текущий URI.
3. 0 паник на битых импортах (catch_unwind).
4. Все новые тесты PASS; 0 регрессий в существующих nova-lsp тестах.
5. **Без упрощений:** провенанс через реальные `peer_files`, НЕ через повторный текстовый поиск
   определения по имени в файлах.

### Ф.1 — Symbol cache (parsed Module + ModuleEnv per URI)

**Проблема:** каждый hover/goto/completion заново парсит + `resolve_imports_inline` (инлайнит
450+ prelude items) + (после Ф.2) `check_module`. На большом файле ~50–100ms.

**Дизайн:**
1. В `WorkspaceState` (`state.rs`) добавить:
   ```rust
   pub resolved_cache: DashMap<Url, CachedResolved>,
   ```
   где `CachedResolved { resolved: Arc<ResolvedModule>, version: i32 }`.
2. `get_or_build_resolved(uri, version, src) -> Arc<ResolvedModule>`: если cache hit по version →
   вернуть; иначе построить через `resolve_module_for` (+ `check_module` для env, Ф.2) и закешировать.
3. Инвалидация: `didChange` обновляет version → следующий запрос перестроит. `didClose` → удалить.
4. **Потокобезопасность:** `Arc` для shared-read без клонирования Module; DashMap для concurrent.
5. **Бюджет памяти:** кешировать только открытые документы (на `didClose` чистить). Не кешировать
   весь workspace.

**Тесты:**
- POS: два hover-запроса подряд на одной version → второй берёт из кеша (проверить через
  счётчик построений — добавить `AtomicU64 build_count` в тест-режиме или через timing).
- POS: после `didChange` (новая version) → перестроение (build_count++).
- POS: `didClose` → запись удалена из cache.
- NEG: запрос на незакешированный URI → построение без паники.
- EDGE: конкурентные запросы на один URI (spawn 2 потока) → без data-race (DashMap), оба
  получают валидный результат.
- PERF: hover на файле 1000 строк из кеша ≤10ms (vs ≥50ms холодный) — `perf.rs` measure.

**Критерии приёмки Ф.1:**
1. Cache hit по совпадающей version (не перестраивает).
2. Инвалидация на `didChange`/`didClose` корректна.
3. Concurrent-safe (DashMap + Arc), тест на 2 потока PASS.
4. Hover из кеша ≤10ms (perf-тест).
5. **Без упрощений:** кешируется полный `ResolvedModule` (Module+file_map+env), не частично.

### Ф.2 — `expr_types: HashMap<Span, Ty>` в ModuleEnv (COMPILER-CODEGEN)

**Это крупнейшая и самая фундаментальная фаза.** Корень для Ф.5/Ф.6/Ф.8/Ф.9/Ф.10.

**Проблема (F4):** checker вычисляет типы выражений, но выбрасывает. IDE не может узнать тип
произвольного выражения (`r.start`, `foo().bar`, receiver в `x.method()`).

**Дизайн (production, opt-in чтобы не замедлять обычный compile):**
1. **Добавить поле в `ModuleEnv`** (`types/mod.rs`):
   ```rust
   pub expr_types: HashMap<Span, Ty>,   // заполняется только при record_expr_types=true
   ```
2. **Флаг сбора.** Добавить в `TypeCheckCtx` булев `record_expr_types`. Обычный `check_module`
   → `false` (zero-overhead, как сейчас). Новый `check_module_with_expr_types(module) -> ModuleEnv`
   → `true`. Это критично: НЕ замедлять `nova check`/`build`/`test`.
3. **Точки записи.** В `infer_expr` (или эквивалент — найти центральную функцию вывода типа
   выражения в `types/`), после вычисления `ty` выражения `e`: если `record_expr_types`, то
   `env.expr_types.insert(e.span, ty.clone())`. Покрыть: Ident, Member, Index, Call, Binary,
   литералы, Range, As, TupleLit, ArrayLit, RecordLit. Для member-access — записать тип и
   объекта, и результата (нужно для Ф.6).
4. **Span как ключ.** `Span` уже `Hash+Eq` (`diag.rs:27`). Коллизии маловероятны (start/end/file_id
   уникальны на выражение). Если выражение синтезировано без span (span=default) — НЕ записывать.
5. **Граница ответственности.** Если тип невыводим (generic-параметр, ошибка) → не записывать
   (отсутствие в map = «неизвестно», IDE деградирует gracefully).

**Где искать центральную функцию инференции:** `compiler-codegen/src/types/` — grep `fn infer_expr`,
`fn check_expr`, `fn expr_type`, `Ty::` возвраты. Если инференции выражений как единой функции нет
(bootstrap может проверять структурно) — оценить объём: возможно потребуется добавить
expression-type walker. **Это исследовать ПЕРВЫМ шагом Ф.2** и зафиксировать реальный объём;
если walker отсутствует целиком — выделить под-фазу Ф.2a (построить минимальный expr-type walker
для IDE: literals + local vars + fn-call-return + member-on-record + index — покрывает 90% IDE-кейсов)
и Ф.2b (полное покрытие) с маркером на остаток.

**Спека:** D305 (Ф.14) — «ModuleEnv.expr_types — opt-in per-expression type map для IDE».
Зафиксировать: opt-in (zero-cost для compile), semantics «отсутствие = неизвестно».

**Тесты (compiler-codegen, `nova_tests/plan104_10/` + Rust unit в types/):**
- POS (Rust unit): `check_module_with_expr_types` на `ro x = 5` → `expr_types` содержит span литерала `5` → `Ty::Int`.
- POS: `ro r = 0..=5` → span Range → `Ty::Named("Range")` (или как Range представлен).
- POS: `foo()` где `foo -> str` → span Call → `Ty::Str`.
- POS: `rec.field` где field: int → span Member → `Ty::Int`; span obj → `Ty::Named("Rec")`.
- POS: `arr[i]` где arr: []int → span Index → `Ty::Int`.
- POS: zero-overhead — `check_module` (флаг false) → `expr_types` пуст.
- NEG: невыводимое выражение (bare generic T) → НЕ в map (graceful).
- NEG: синтезированный span (default) → НЕ записан.
- EDGE: вложенные `a.b.c` → все три уровня в map с правильными типами.

**Критерии приёмки Ф.2:**
1. `check_module_with_expr_types` заполняет map; `check_module` оставляет пустым (zero-overhead доказан тестом).
2. Покрыты: литералы, Ident, Member, Index, Call, Binary, Range, As, Tuple/Array/Record lit.
3. Невыводимые/синтетические → не в map (без мусора).
4. 0 регрессий в полном compile-pipeline (`expr_types` не влияет на codegen/diagnostics).
5. **Без упрощений:** реальная инференция из checker, НЕ текстовая эвристика. Остаток покрытия
   (если Ф.2a/b split) — явный маркер `[M-104.10-expr-types-coverage]` с перечнем непокрытого.

## BLOCK B — ФИЧИ

### Ф.3 — Cross-file goto-definition

**Нужен:** Ф.0.

**Дизайн:** `compute_goto_definition` использует `resolve_module_for` (кеш Ф.1) → `resolve_symbol_at`
→ `symbol.span()` → `span_to_location(span, file_map, uri)`. Если символ объявлен в другом файле →
Location с тем URI; range через byte_offset в ИСХОДНИКЕ того файла (прочитать с диска или из
открытых docs).

**Тонкость:** range вычисляется в координатах файла-цели, не входного. Нужен исходник цели для
UTF-16 конверсии. Брать из `WorkspaceState.docs` если открыт, иначе читать с диска.

**Тесты:**
- POS: goto на fn из другого модуля → Location с URI того модуля, корректный range.
- POS: goto на тип из импорта → правильный файл.
- POS: goto на локальную fn в том же файле → тот же URI (регресс single-file).
- POS: goto на prelude-символ (`assert`) → Location в `std/prelude/*.nv`.
- NEG: goto на неизвестный символ → None.
- NEG: goto на whitespace → None.
- EDGE: goto на символ из файла, который не открыт в редакторе → читается с диска, range корректен.
- Integration (spawn): полный JSON-RPC `textDocument/definition` cross-file.

**Критерии приёмки Ф.3:**
1. Cross-file goto возвращает правильный URI + range.
2. Single-file goto не сломан (регресс-тесты).
3. Range корректен в UTF-16 для файла-цели (multi-byte тест).
4. `definition_provider` capability уже `true` — теперь честно cross-file.
5. **Без упрощений:** резолв через span.file_id+file_map, НЕ повторный текстовый grep по имени.

### Ф.5 — Type-driven completion методов

**Нужен:** Ф.2 (expr_types) + Ф.0.

**Проблема:** completion методов после `.` использует текстовую эвристику типа ресивера
(`infer_type_of_expr`, completion.rs:1072 — наивный scan аннотаций + naming-convention) + статические
таблицы. Пользовательские типы и их методы не предлагаются корректно.

**Дизайн:**
1. При completion-контексте `MethodDot`: найти выражение-ресивер перед `.`, взять его span, найти
   тип в `expr_types` (Ф.2). Если найден `Ty::Named("Foo")` → искать все методы с receiver `Foo` в
   `module.items` (включая cross-file через inline) → CompletionItems с сигнатурами.
2. Статические таблицы (`int_methods`/`vec_methods`/…) ОСТАЮТСЯ как fallback для встроенных типов
   (int/str/Vec/Option/Result) — они быстрее и всегда корректны для stdlib.
3. Для пользовательских record/sum типов — динамический поиск методов из AST.
4. Текстовую эвристику `infer_type_of_expr` оставить как последний fallback (когда expr_types
   не покрыл выражение), но пометить, что primary path — expr_types.

**Тесты:**
- POS: `ro u = User{...}; u.` → предлагает методы User (пользовательский тип).
- POS: `ro v = Vec[int].new(); v.` → методы Vec (статические таблицы, регресс).
- POS: `foo().` где `foo -> Bar` → методы Bar (тип из возврата функции).
- POS: cross-file тип → его методы предлагаются.
- NEG: `unknown_var.` → пустой/graceful (не паника).
- EDGE: `a.b.` (chained) → методы типа `a.b`.

**Критерии приёмки Ф.5:**
1. Методы пользовательских типов предлагаются (через expr_types + AST-поиск).
2. Stdlib-методы не сломаны (статические таблицы fallback).
3. Cross-file типы покрыты.
4. **Без упрощений:** primary path — реальный тип из expr_types, эвристика только last-resort
   с маркером `[M-104.10-completion-text-fallback]` если останется.

### Ф.6 — Hover на member-access (r.start)

**Нужен:** Ф.2.

**Проблема:** `ro r = 0..=5; r.start` — hover на `start` ничего не показывает.

**Дизайн:** в `find_ident_in_bodies_from` / resolver — если cursor на `obj.field`: взять тип `obj`
из `expr_types[obj.span]` → `Ty::Named("Range")` → найти поле/метод `start` в decl `Range` →
вернуть SymbolInfo с типом поля.

**Тесты:**
- POS: `r.start` где r: Range → hover показывает `start int` (или фактический тип поля).
- POS: `rec.field` пользовательского record → тип поля.
- POS: `obj.method` → сигнатура метода.
- NEG: `r.nonexistent` → None (поля нет).
- EDGE: `a.b.c` → hover на `c` → тип через цепочку.

**Критерии приёмки Ф.6:**
1. Hover на поле через тип объекта из expr_types работает.
2. Методы тоже резолвятся.
3. Несуществующее поле → None.
4. **Без упрощений:** тип объекта из expr_types, поле из реального decl.

### Ф.7 — Rename через symbol-table

**Нужен:** Ф.0 + references.

**Проблема:** rename = regex word-boundary → переименует `foo` во всех scope с тем же именем.

**Дизайн:**
1. `compute_rename`: сначала `resolve_symbol_at(cursor)` → `(name, decl_span, decl_file_id)`.
2. Для каждого кандидата-вхождения (из текущего regex-скана) — резолвить его символ и сравнивать
   `decl_span`. Переименовывать только совпадающие с тем же объявлением.
3. **Реализм bootstrap:** полный resolve каждого вхождения дорог и требует точного name-resolution,
   которого в bootstrap-checker может не быть. Поэтому V2-rename использует **scope-aware фильтр**:
   - резолвить символ под курсором → определить его kind (local var / fn / type / field).
   - для **local var**: ограничить rename телом объявляющей функции (scope известен из AST).
   - для **top-level (fn/type)**: cross-file rename как сейчас, но с проверкой, что вхождение не
     затенено локальным биндингом того же имени (shadow-check через AST scope-walk).
4. Сохранить атомарный post-rename type-check (D296) — уже есть.

**Тесты:**
- POS: rename local var → меняется только в её scope, не трогает одноимённую в другой функции.
- POS: rename top-level fn → все cross-file вхождения.
- POS: shadow — `fn a(){ ro x=1 } fn b(){ ro x=2 }` rename x в a → b.x не тронут.
- NEG: rename в conflict (имя занято) → ошибка через atomic check (D296).
- NEG: rename keyword/literal → отклонён prepareRename.
- EDGE: rename типа, используемого cross-file → все файлы.

**Критерии приёмки Ф.7:**
1. Local var rename ограничен scope (shadow-safe).
2. Top-level rename cross-file + shadow-check.
3. Atomic conflict-check (D296) сохранён.
4. **Без упрощений:** scope из AST, НЕ просто regex. Остаток (если полный per-occurrence resolve
   не сделан) → маркер `[M-104.10-rename-full-resolve]` с обоснованием.

### Ф.8 — Signature help по типу receiver

**Нужен:** Ф.2.

**Проблема:** `x.foo(│)` берёт первый overload `foo` по имени, игнорируя тип `x`.

**Дизайн:** в `signature_help` — если вызов это `recv.method(...)`: взять тип `recv` из expr_types →
выбрать overload с matching receiver-типом. Для свободных fn с overload по аргументам — выбирать по
числу/типам уже введённых аргументов.

**Тесты:**
- POS: `vec.push(│)` → сигнатура push именно Vec, не другого типа с push.
- POS: overload по числу аргументов → правильный по позиции курсора.
- NEG: неизвестный receiver → fallback на первый (graceful).
- EDGE: вложенный вызов `f(g(│))` → сигнатура g.

**Критерии приёмки Ф.8:**
1. Overload выбирается по типу receiver (expr_types).
2. Fallback graceful при неизвестном типе.
3. **Без упрощений:** dispatch по реальному типу.

### Ф.9 — Inlay hints (типы + имена параметров)

**Нужен:** Ф.2. tower-lsp 0.20 поддерживает `InlayHint` (F5).

**Дизайн:** advertise `inlay_hint_provider`. Хендлер `textDocument/inlayHint(range)`:

**Критерии приёмки:** lazy resolve работает; initial list легче; без упрощений.

## 4. Методология исполнения (фоновые агенты / workflow)

> Эти правила ОБЯЗАТЕЛЬНЫ при использовании фоновых агентов или Workflow.

- **НЕ `git stash`** — repo-global, конкурентные worktree делят `.git`. Для baseline-сравнения:
  temp-worktree ИЛИ commit-reset (закоммить → измерь → reset). Никогда не stash при параллельных агентах.
- **`git add` только конкретные файлы**, никогда `git add -A`/`.` — рядом могут работать другие агенты.
- **Перед commit:** `git diff --cached --stat` (в индексе могут быть чужие pre-staged изменения).
- **НЕ добавлять `Co-Authored-By: Claude`** в commit messages.
- **Отдельный commit на каждую фазу** (Ф.0, Ф.1, … — отдельные коммиты).
- **Rate-limit фоновых агентов/workflow:** серверный rate-limit иногда роняет агента. Стратегия:
  (a) дробить на фазы, не запускать >3–4 агентов параллельно; (b) при падении агента —
  retry той же под-задачи (фазы атомарны и идемпотентны по дизайну); (c) Workflow поддерживает
  resume через `resumeFromRunId` — закешированные фазы не перевыполняются; (d) если фаза падает
  повторно из-за rate-limit — выполнить её inline (не в фоне).
- **Тесты nova-lsp:** `cd nova-lsp && cargo test --release` (НЕ нужен NOVA_GC_LIB_DIR).
- **Тесты compiler-codegen (Ф.2):** `cd compiler-codegen && cargo test --release` + при изменении
  поведения — sample `nova test` через nova-cli для регресс-проверки (НЕ полный — он 60–90 мин;
  дробить батчами <10 мин по `bash-timeout-10min-max`).
- **При изменении `.nv` в std/prelude** — пересобрать nova-cli (include_str!).
- **Baseline для perf (Ф.1):** kill-switch на ТОМ ЖЕ бинаре, не sibling.
- **Последовательная работа над многофазным планом** — в постоянном worktree `nova-p104-10`,
  не плодить sub-worktree (`feedback-sequential-no-sub-worktree`); naming `nova-pNN` (`feedback-worktree-naming`).
