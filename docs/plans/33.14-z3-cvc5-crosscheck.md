# Plan 33.14: Z3 ↔ CVC5 cross-check

> Подплан семейства **Plan 33** (контракты). Вторая линия защиты
> soundness — в дополнение к Plan 33.8 Ф.7 (regression-suite).
>
> **Реанимирует обещание Plan 33.3:** cross-check Z3↔CVC5 был в исходном
> scope Plan 33.3 (с требованием CI-merge-гейта), но не доставлен и тихо
> снят. Plan 33.6 зарезервировал под него слот 33.14.

## Зачем

Два аудита Plan 33.8 нашли 6 SOUNDNESS-CRITICAL дыр, и **второй аудит
нашёл дыру в фиксе первого**. Code-review недостаточен. Защита строится
в два слоя:

1. **Regression-suite** (Plan 33.8 Ф.7) — ловит ИЗВЕСТНЫЕ классы багов
   при их повторном появлении.
2. **Cross-check** (этот план) — ловит НОВЫЕ/неизвестные баги:
   - **баги кодирования** Nova → SMT (например Ф.6.2: Z3-FFI-трансляция
     молча теряла формулу → ложный `Proven`);
   - **баги решателя** (редко, но бывают).

Принцип: два независимых решателя почти никогда не имеют одинаковый баг.
Оба сказали `Proven` — доверие высокое. Разошлись — красный флаг.

## Цель

Прогонять каждую verification condition (VC) через **два независимых
пути** — Z3 (текущий FFI-backend) и CVC5 (через текстовый SMT-LIB) — и
падать в CI при расхождении ОПРЕДЕЛЁННЫХ ответов (`Proven` vs `Disproved`).

## Принципы

- **CI-only, не в обычной компиляции.** `nova build` / `nova check`
  остаются с одним решателем — время компиляции разработчика НЕ растёт.
  Cross-check — отдельный режim, гоняется в CI на корпусе.
- **Cross-check проверяет верификатор, не пользовательский код.** Поэтому
  его место — корпус тестов + контракты stdlib, а не каждая сборка.
- **Дёшево интегрировать.** НЕ линковать CVC5 через FFI. У Nova уже есть
  SMT-IR (`SmtTerm`) — допечатать текст SMT-LIB v2 и запускать `cvc5`
  подпроцессом. Текстовый путь — это ещё и второй независимый путь
  кодирования (поймал бы Ф.6.2 даже без CVC5).
- **Gate только на definite-disagreement.** `Unknown`/timeout с любой
  стороны — норма (решатели имеют разные перф-профили), не ошибка.

## Ф.1 — SMT-LIB v2 эмиттер

- Принтер `SmtTerm` / `Assertion` / declared vars → текст SMT-LIB v2:
  `(set-logic ...)`, `(declare-fun ...)`, `(declare-sort ...)`,
  `(assert ...)`, `(check-sat)`.
- Покрыть все sorts/ops: Int, Bool, Str (Seq), FP, BitVec, `forall`/
  `exists` с patterns, UF.
- Тесты: golden-snapshot на репрезентативных формулах.

## Ф.2 — CVC5 runner

- Запуск `cvc5` подпроцессом: скормить SMT-LIB текст на stdin, распарсить
  `sat` / `unsat` / `unknown` из stdout.
- Обнаружение бинарника (env `NOVA_CVC5` / PATH); если CVC5 не найден —
  graceful skip с понятным сообщением (не падать).
- Извлечение модели/контрпримера (`(get-model)`) при `sat` — для diff.

## Ф.3 — Cross-check режим

- Режим включается флагом CLI (`nova verify --cross-check`) или env
  (`NOVA_CROSSCHECK=1`). По умолчанию — выключен.
- Для каждой VC: прогнать Z3 (FFI) И CVC5 (text), собрать пару ответов.
- Backend-обёртка `CrossCheckBackend` поверх существующего `SmtBackend`
  trait, либо отдельный verify-проход.

## Ф.4 — Сравнение + diff-репорт

- Классификация пары `(z3_result, cvc5_result)`:
  - оба `Proven` / оба `Disproved` / оба `Unknown` → OK;
  - один `Unknown` — OK (записать в info-лог);
  - `Proven` vs `Disproved` → **DISAGREEMENT** — ошибка.
- Diff-репорт при расхождении: имя fn, контракт, span, текст VC,
  оба ответа, контрпример CVC5.

## Ф.5 — CI-job + gate

- CI-job `contracts-crosscheck`: `nova verify --cross-check` на корпусе
  (`nova_tests/contracts/` + контракты stdlib).
- Gate: 0 definite-disagreements обязательно для merge.
- Если per-PR ×2-время дорого — вариант nightly + per-PR только на
  изменённых файлах.

## Ф.6 — Тесты, спека, доки

- Тесты эмиттера (Ф.1 golden) + логики сравнения (Ф.4).
- Спека: восстановить запись про cross-check в Plan 33.3 / D24 как
  доставленную.
- Doc: `docs/contracts.md` — секция про cross-check (как запустить).

## Объём / риски

- Средний по объёму (несколько дней). Главная работа — Ф.1 (эмиттер) и
  Ф.2 (runner); остальное — склейка.
- Риск: CVC5 и Z3 могут расходиться легитимно на `Unknown` (разные
  таймауты/эвристики) — поэтому gate ТОЛЬКО на definite-disagreement.
- Зависимость: бинарник `cvc5` в CI-окружении (как Z3 сейчас).

## Итог реализации

Главное отклонение от исходного наброска (Plan 33.3 Ф.12) — **намеренное**:
CVC5 подключён НЕ через FFI, а через текстовый SMT-LIB v2 и подпроцесс.
Это даёт второй независимый путь *кодирования* (`verify::smtlib` не
разделяет код с Z3-FFI-трансляцией), а не только второй решатель —
именно текстовый путь поймал бы баг Plan 33.8 Ф.6.2.

**Инварианты soundness, заложенные в реализацию:**

- Эмиттер — точное зеркало `Z3Backend` по набору операторов. При любой
  неуверенности — `EmitError`, а не приблизительный SMT-LIB; иначе
  cross-check давал бы ложные disagreement'ы.
- Любой сбой cvc5 (parse-error, crash, не найден) → `Unknown`, никогда
  не definite-ответ. `Unknown` с любой стороны классифицируется как OK.
- `_field_*` кодируются свежими константами на структурно-различные
  аргументы — точное зеркало `z3::legacy_uninterpreted_app` (без
  congruence-аксиомы UF), чтобы разная сила кодирования не давала
  ложных расхождений.
- BV overflow-предикаты кодируются в чистой BV-теории с семантикой,
  совпадающей с `Z3_mk_bv*_no_*flow`.

**Gate (Ф.5):** `contracts-crosscheck` workflow + airtight файл-лог
`NOVA_CROSSCHECK_LOG` — корпус компилируется процесс-на-файл, файл
аккумулирует расхождения всего прогона независимо от per-test pass/fail.

**Известное ограничение:** cvc5-бинарник нужен в окружении. На dev-машине
без cvc5 cross-check gracefully вырождается в «только Z3» с warning'ом;
в CI наличие cvc5 проверяется явно (`cvc5 --version`), без него job
падает — «тихая деградация» gate не обходит.
