# Plan 154 (umbrella) — No silent no-op dispatch

> **Создан:** 2026-06-13. **Статус:** 🟡 IN PROGRESS (154.0 ✅ CLOSED; 154.1 в работе).
> **P1** (correctness/footgun). **Тема:** ни один вызов метода не должен **молча**
> резолвиться в no-op / чужую реализацию — каждый такой случай — **громкая ошибка**
> или явный механизм.

---

## Идея

Зонтик над классом багов «тихий no-op/мис-диспатч в method-resolution» — когда вызов
метода компилируется, но **молча** делает не то (no-op, чужую реализацию), без
диагностики. Каждый такой случай ловится громко (compile-error) либо закрывается
явным языковым механизмом.

Триггеры (репортнуты пользователем):
1. `fn str @to_lower()` в user-модуле — молча игнорировался (codegen берёт std,
   тело юзера мёртвое). → **154.0**.
2. `Vec[int].from([10,20,30]).debug(sb)` — element-метод примитива молча роутится в
   erased-стаб контейнера (мусор/no-op). → **154.1**.

## Принцип (инвариант)

- **I1. Никакого silent no-op dispatch.** Вызов метода либо резолвится в корректное
  тело, либо — compile-error. Никаких молча-no-op / молча-чужих реализаций.
- **I2. Extension да, override чужого — нет** (154.0/D267): добавлять метод можно
  (нет orphan rule), переопределять существующий из другого модуля — ошибка.
- **I3. Конформность протокола — структурная ИЛИ явная** (154.1/D268): `#impl(P)` —
  opt-in декларация намерения + checker-enforcement.
