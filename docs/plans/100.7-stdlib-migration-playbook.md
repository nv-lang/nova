## Migration inventory — stdlib типов с consume-potential

### High-priority (миграция в pilot)

| Тип | Текущий статус | Consume? | Pilot? | Зачем |
|---|---|---|---|---|
| `File` (Plan 18 P0) | не реализован | ✅ yes | ✅ pilot 1 | file descriptors must close |
| `TcpSocket` (Plan 18 P0) | не реализован | ✅ yes | ✅ pilot 2 | sockets must close |
| `Mutex` lock-guard | не реализован | ✅ yes | ✅ pilot 3 | release on scope exit |
| `Transaction` (Plan 18 / DB) | не реализован | ✅ yes | ✅ pilot 4 | commit/rollback distinction |

Эти 4 — **canonical resource types**, ради которых вся Plan 100 и
делалась. Pilot — **realistic** в отличие от mock-only в 100.1.

### Medium-priority (migration LATER, не в Plan 100.7)

| Тип | Текущий статус | Consume? | Зачем |
|---|---|---|---|
| `StringBuilder` (existing) | affine D131 | ⚠️ optional | `into()` forgets — buffer leaks. Possibly. Decision in Ф.0 audit. |
| `ReadBuffer` / `WriteBuffer` | affine D131 | ❌ no | Buffers могут быть забыты (no resource leak — just GC). |
| `ChanWriter` / `ChanReader` (Plan 21) | regular | ⚠️ optional | Close-on-drop convention exists; consume markers improve. |
| `CancelToken` (Plan 49) | regular | ❌ no | Bind-check via Plan 47, not consume. |
| `Iter[T]` instances | regular | ❌ no | Iterators are cheap; forgetting OK. |

Решение audit'ом Ф.0 (с rationale per type).

### Low-priority / out-of-scope

| Тип | Зачем НЕ migrate |
|---|---|
| Primitive types (int, bool, char, ...) | Value-types, no resources. |
| `Result[T, E]` / `Option[T]` | Already generic-propagation aware (100.2). |
| `HashMap` / `Vec` / `HashSet` | Owned data, no external resource. |

## Дизайн-решения

### D1 — Migration tooling: `nova consume-migrate <path>`

CLI subcommand для auto-migration:

```
$ nova consume-migrate std/io/file.nv

Analyzing std/io/file.nv...
Suggestions:
  Line 5: type File { ... }
    → Suggestion: type File consume { ... }
    Reason: file_open returns File, file_close consumes; resource type.

  Line 12: fn File @close() -> Result[(), IoErr]
    → Suggestion: fn File consume @close() -> Result[(), IoErr]
    Reason: close is consume-method (terminal).

Apply (y/n/manual)?
```

Implementation reuses existing `nova check` + new analyzer suggesting
migrations.

### D2 — Edition-versioning support (Plan 62.F.bis)

Migration breaks existing user-code (file.nv currently без consume —
после migration требует consume tx = ...).

`nova.toml` edition:
```toml
[package]
edition = "2026"                              // pre-consume era; uses old API
# vs
edition = "2027"                              // post-consume migration
```

Plan 62.F.bis edition resolver picks correct prelude/stdlib version.

### D3 — Deprecation cycle

Migration ≠ removal. Old API gets `#[deprecated_since("2027.1")]`:

```nova
// Edition 2026:
fn File.open(path str) -> Result[File, IoErr]     // legacy

// Edition 2027:
external consume fn nova_file_open(path str) -> File
fn File consume @close() -> Result[(), IoErr]

#[deprecated_since("2027.1", replacement = "use `consume f = File.open()` + defer close")]
fn File.open_legacy(path str) -> File              // back-compat shim
```

User code на edition 2026 продолжает работать; 2027 migration warning.

### D4 — Pilot migration scope (4 типа)

Plan 100.7 включает **полную migration** 4 high-priority типов:
- `std/io/file.nv` — File type с consume.
- `std/net/tcp.nv` — TcpSocket с consume.
- `std/sync/mutex.nv` — Mutex + Lock guard с consume.
- `std/db/transaction.nv` (mock; real DB — Plan 18) — Transaction.

Каждый pilot — **production-grade**:
- `external consume fn` declarations (Plan 100.5).
- consume-methods (open / accept / lock / commit / close / release /
  rollback).
- defer/errdefer pattern documentation (Plan 100.4.5).
- generic integration (Result wrapping, Option lazy-init).
- tests (positive: canonical use; negative: forget; integration).

### D5 — Migration ordering (dependency graph)

```
File (independent)
  ↓
TcpSocket (independent)
  ↓
Mutex (independent)
  ↓
Transaction (depends on DB stack — mock first, real later)
```

Каждый — параллельный (independent), но pilot order suggested:
File → Mutex → TcpSocket → Transaction (по сложности cleanup).

### D6 — Breaking change management

Major version bumps (Plan 100.6 D153):
- Adding consume to existing type = **breaking** (caller code must
  add `consume` binding-form).
- Edition-protected; users on old edition не затронуты.

Plan 03 `nova audit` ловит cross-package consume-additions без
version bump.

### D7 — Migration verification protocol

Каждая migration follows:

1. **Probe** — written 5 representative use-cases (open-read-close;
   parallel access; error-path; supervised cancel; cross-fiber).
2. **Implement** — actual migration on stdlib type.
3. **Test** — all probes pass; existing fixtures regression-tested.
4. **Document** — `docs/idiom/<type>-resource.md` canonical pattern.
5. **Audit** — `nova audit` confirms no cross-package break.

Standardized across all 4 pilots.

### D8 — Non-pilot types — defer policy

`StringBuilder`/`ReadBuffer`/`WriteBuffer` etc. — bootstrap **остаются
affine** (D131 only). Migration позже если возникнет давление. Plan
100.7 explicitly documents rationale per non-pilot type.

## Границы

- **Migration ВСЕХ stdlib типов** — out of scope; 4 pilots only.
  Medium-priority types — отдельным plan'ом по мере давления.
- **Real DB layer** — Transaction migration через mock; real DB
  integration в Plan 18 stdlib roadmap.
- **Migration historical data** — пользовательский код с persisted
  state (serialized non-consume types) — out of scope.

## Risks

1. **Edition split** — codebase с mixed editions может быть сложным
   debug. **Митигация:** Plan 62.F.bis edition isolation; explicit
   marker per file.
2. **Migration tool false-positives** — может неправильно suggest
   consume. **Митигация:** `--interactive` mode + manual review.
3. **Plan 18 stdlib readiness** — File/Socket нужны Plan 18 P0
   implementation. **Митигация:** Plan 100.7 pilots — **самостоятельная
   stdlib**, не зависит от Plan 18 deployment; Plan 18 будет адаптироваться
   к Plan 100 API.
