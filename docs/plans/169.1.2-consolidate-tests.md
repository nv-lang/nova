## Цель

Сократить число compile-units (CU) тест-набора, склеивая отдельные тест-файлы
в **folder-module по ТЕМЕ** (atomics, sync, generics, syntax, effects, …), а не
по номеру плана. Базовый потолок: ~1314 → ~900 CU.

## Доказанный метод (plan103_2, 2026-06-19)

1. **Форма модуля** (из `scripts/catb_convert.py`): folder-module = `module nova_tests.<тема>`;
   negative = `module neg.<stem>` в `neg/`.
2. **Валидация — ТОЛЬКО канонически:** `nova test . --filter <тема>` из корня репо.
   Standalone `nova test nova_tests/<папка>` folder-module НЕ обрабатывает (root
   модуля считается иначе) — не использовать для проверки.
3. **Связь «тест ↔ план» через имя файла** (договорённость): при слиянии планов в
   `module nova_tests.<тема>` файлы именуются с префиксом плана —
   `plan103_2_atomic_i64.nv`, `plan103_1_ordering.nv`. Происхождение видно по
   имени файла; раннер печатает путь при падении → навигация цела. Имена файлов
   произвольны, на module-форму не влияют.
4. **Коллизии top-level имён между планами** (граница Plan 170):
   - `fn`/`const`/тип-без-методов → `priv(file)` (ноль rename, читаемо);
   - тип-С-методами → ordinal-rename (`Counter`→`Counter1`), т.к. метод-символ
     `Nova_<T>_static_<m>` не file-дискриминирован;
   - `fn main()` не уживаются (один entry/модуль) → main-based тесты сначала
     `main`→`test "…"`, иначе не сливать.
5. **Чинить fallout по ходу** (ось 169.2): свежий compiler-enforcement ломает
   позитивы (`E_LOCAL_NOT_MUT`: счётчики/флаги без `mut` → добавить `mut`, не
   выхолащивая). Склейка и починка идут вместе.

## Фазы

### Ф.1 — Позитивы по темам (основная, −239+ CU)
Таксономия тем (какой план в какую группу) + слияние позитивных тестов одной
темы в `module nova_tests.<тема>` (файлы с префиксом плана). priv(file)/rename
коллизий. Починка fallout (mut и т.п.). Валидация `nova test . --filter`.
Категории A (pos-only) + смешанные: negative выносится в `neg/` (`module neg.<stem>`),
позитивный остаток сливается в тему.

### Ф.2 — `panics`-клаузула + panic-тесты (−78 CU, компиляторная фича, рантайм-риск)
Контекстное KW `panics` (инверсия PASS/FAIL): `test "…" panics "паттерн" { … }`.
`nv_panic` (`effects.h:566`) при `_nova_test_frame` делает `longjmp` (не abort) —
panic уже ловится setjmp (`emit_c.rs:17218-17253`); вводим ожидание паники →
114 runtime-panic тестов в folder-module.
- parser+AST: `TestDecl { …, panics: Option<String> }`.
- codegen: при `panics.is_some()` инвертировать setjmp-ветки + `strstr(msg, pattern)`.
- миграция: `fn main`+`// EXPECT_RUNTIME_PANIC <pat>` → `test "<stem>" panics "<pat>"`.
- **ГЛАВНЫЙ РИСК — GC/fail-frame consistency между паниками в одном процессе**
  (`longjmp` пропускает деструкторы): `_nova_fail_top`, `_nova_current_handler_iframe`
  (чинился Plan 169.2 §16), GC-arena, scheduler. Стратегия: сначала ОДНА простая
  группа (contract/assert-паники, без concurrency) → если 5-10 паник сосуществуют,
  катить группами, concurrency-паники последними; если рассыпается → нужен
  `nova_runtime_reset()` между тестами (отд. инфра, тогда отложить). `EXPECT_RUNTIME_PANIC`
  остаётся для legacy `fn main` + селектора `--panic` (169.1.1).
