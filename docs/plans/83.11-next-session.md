### Step-by-step plan (8 hours total)

#### Step 1 — Capture TIMEOUT cases with auto-dump (1h)

Запустить `stress_iso_3e.exe` в loop с подавлением stdout/stderr перенаправлением в файлы.
Watchdog в supervised_run_impl сам напишет state-dump в stderr на >5s wait.

```bash
mkdir -p /tmp/p83-11-diag
for i in $(seq 1 30); do
  timeout 12 "$exe" > "/tmp/p83-11-diag/run-$i.stdout" 2> "/tmp/p83-11-diag/run-$i.stderr"
  echo "run $i exit=$?"
done

# Найти TIMEOUT runs (те где watchdog dump fired):
grep -l "NOVA_RUNTIME_DUMP" /tmp/p83-11-diag/run-*.stderr | head -10
```

**Deliverable:** ≥3 stderr-файлов с state-dump'ами для analysis.

#### Step 2 — Parse + classify dumps (2h)

State-dump формат (см. `nova_runtime_dump_state` в runtime.c):

```
=== NOVA_RUNTIME_DUMP === time=Xs reason=watchdog
[main] state=waiting_supervised pending_remote=N
[driver] thread_running=Y armed_list_size=Z diag_close_cb=A diag_wake_called=B diag_cas_won=C diag_drained=D
[worker 0] runnext=<ptr> deque_size=N wake_pending=M
  scope.count=K parked=[01010...] pending_wake=[00100...]
  fiber[slot]=<ptr> mco_status=SUSPENDED/RUNNING/DEAD state=PARKED/IDLE/...
  ...
[worker 1] ...
[cancel-token] requested=true reason=<...>
=== END DUMP ===
```

**Classification (по dump'ам):**

| Pattern A | parked[s]=1 && pending_wake[s]=1 | Race в post-barrier recheck OR pending_wake leak |
| Pattern B | parked[s]=1 && pending_wake[s]=0 + fiber MCO_SUSPENDED + не в deque + armed=0 | Wake потерян: close_cb прошёл но dispatch не вызван |
| Pattern C | parked[s]=0 + fiber MCO_SUSPENDED + не в deque | Fiber парк'ен через legacy path, минуя driver |
| Pattern D | armed_list_size > 0 после cancel timeout | Driver thread залип / CANCEL_SCOPE job не обработан |
| Pattern E | other (новый паттерн) | Hypothesize from raw data |

**Deliverable:** таблица "run N → pattern X, slot Y, worker Z".

#### Step 3 — Driver thread state (1h)

Из dump'а: `[driver] thread_running` — поток жив? `armed_list_size` — есть ли неотменённые таймеры? Counter delta:

- `diag_close_cb` vs `diag_wake_called` — должны совпадать
- `diag_wake_called` vs `diag_cas_won` — wake CAS все выиграли?
- `diag_cas_won` vs `diag_drained` — все wake'и дошли до worker'ов?

Если cas_won < wake_called — race в CAS parked (наш known case, Option A не закрыл).
Если drained < cas_won — wake_pending не drained worker'ом (worker stuck в uv_run без async fire?).

**Deliverable:** counter delta анализ + hypothesis.

#### Step 4 — Cross-check routing (1h)

Из dump'а: для каждого parked fiber slot, какой `_nova_active_scope`? Совпадает ли с worker scope или указывает на supervised scope?

Если scope mismatch → wake идёт в один scope, fiber парк'ен в другом → wake потерян by-design.

**Deliverable:** или confirmation что routing правильный, или указание на bug.

#### Step 5 — Reproduce + iterate (1.5h)

Если pattern E (новый) или гипотеза неясна — расширить инструментацию:
1. Edit `nova_runtime_dump_state` — добавить новые поля (например, MCO внутренности, libuv loop state)
2. Rebuild (~2 мин)
3. Re-run stress (~3 мин)
4. Analyze new dumps

Up to 3 итераций инструментации.

**Deliverable:** обновлённый infrastructure + новые dumps для финальной hypothesis.

#### Step 6 — Hypothesis + decision (1.5h)

С данными steps 1-5 формулировать **конкретную** гипотезу: "race в строке X файла Y, потому что Z". Не "наверное где-то wake-before-park".

**Решение:**
- Если гипотеза локализована → fix-patch + verify (часы) → **Phase A done, Phase B skipped**
- Если гипотеза подтверждает архитектурный rewrite → **переход к Phase B (Option B)** с точным mandate
- Если гипотеза неясна → документ "что пробовали, не зашло", оценка альтернатив

**Deliverable:** документ `docs/plans/83.11-diagnostics-report.md` с findings + recommendation.

## Phase B — Option B (2-4 days, Opus 4.7, **ТОЛЬКО если Phase A подтвердит**)

**Model:** Opus 4.7
**Effort:** Maximum
**Thinking:** ON (mandatory — high-stakes design + rewrite)

### Trigger

Phase A Step 6 deliverable = "архитектурный rewrite неизбежен, гипотеза такая-то".

### Scope (fixed)

Заменить `parked[]` + `pending_wake[]` + `dispatch_ready` callback на единый `NovaWakerState` атомик per slot.

**State machine (port из Tokio task::state):**
```c
typedef struct {
    atomic_uint64_t state;  /* packed bits:
        bit 0-7:   FiberState (NEW/RUNNING/PARKED/WOKEN/DEAD)
        bit 8-15:  WakeCount (saturating counter)
        bit 16-23: refcount (для cancel-during-resume guard)
        bit 24-63: reserved (slot generation, ABA guard)
    */
} NovaWakerState;
```

**API замены:**
- `nova_sched_park(scope, slot)` → CAS state RUNNING→PARKED, yield
- `nova_sched_wake(scope, slot)` → CAS state PARKED→WOKEN, push на deque если won
- Все 10+ callers `parked[]` переписаны:
  - `nova_runtime_cancel_worker_fibers`
  - `nova_sched_count_parked`
  - `nova_supervised_run_impl`
  - `nova_scope_cancel_wake_all`
  - `nova_sched_is_parked`
  - Plan 103.x sync primitives (channels, mutex, semaphore, condvar, barrier, CountDownLatch)

### Acceptance criteria (firm)

- `stress_iso_3e`: **30/30 PASS** (target 100%)
- `supervised_cancel_stress_test` (3 tests): **30/30 each**
- Plan 103.x conformance suite: **no regression**
- M:1 stress: **30/30 PASS** (M:N N=1 надёжен)
