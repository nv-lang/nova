### Step-by-step plan (8 hours total)

#### Step 1 — Capture TIMEOUT process (1h)

Запустить `stress_iso_3e.exe` в loop. Когда runtime > 5 сек — это TIMEOUT (PASS должны быть <2с).
**Не убивать** процесс таймаутом — оставить висеть. Получить PID.

```bash
# В одном терминале:
while true; do
  $exe > /tmp/stress.log 2>&1 &
  PID=$!
  sleep 5
  if kill -0 $PID 2>/dev/null; then
    echo "TIMEOUT pid=$PID"
    break  # оставляем висеть
  fi
  wait $PID
done
echo "PID=$PID для gdb attach"
```

**Deliverable:** живой PID зависшего процесса.

#### Step 2 — gdb attach + thread dump (2h)

```bash
gdb -p $PID
(gdb) info threads
(gdb) thread apply all bt full > /tmp/all-threads.txt
(gdb) thread apply all py-bt 2>/dev/null  # если есть python bindings
```

**Сохранить:** `/tmp/all-threads.txt` — stack trace всех потоков.

**Что искать:**
- Main thread: где? Скорее всего в `uv_run UV_RUN_ONCE` (нормально, ждёт signal)
- Driver thread: где? Если в `uv_run` — нормально (нечего делать). Если в `nova_sched_wake` / `dispatch_ready` — подозрительно.
- 16 worker threads: где каждый? Все ли в idle `uv_run UV_RUN_ONCE`? Кто-то застрял в `mco_resume`?
- Sysmon thread: должен быть в `uv_sleep(10)` — игнорируем.

**Deliverable:** список threads + classification (idle/active/stuck).

#### Step 3 — Inspect atomics state (2h)

Найти `NovaSchedState` для каждого worker scope и dump:
- `parked[]` — какие slot'ы помечены парк'ом
- `pending_wake[]` — какие имеют pending wake
- `armed_sleeps_head` в supervised scope — есть ли ещё неотменённые timer entries

```gdb
(gdb) print _workers[0].scope.sched_state->parked@64
(gdb) print _workers[0].scope.sched_state->pending_wake@64
(gdb) print _workers[0].deque
(gdb) print _workers[0].wake_pending_count
(gdb) print _workers[0].runnext
```

Для каждого worker'а (0-15) — собрать снимок.

**Что искать (4 паттерна):**

| Паттерн | Что означает | Action |
|---------|-------------|--------|
| `parked[s]=true && pending_wake[s]=1` для некоторых s | wake delivered но park не consumed | Race в park's post-barrier recheck OR fiber resumed но pending_wake не cleared |
| `parked[s]=true && pending_wake[s]=0`, fiber не в deque/runnext, и нет armed_sleeps entry | wake потерян: было ARMED → CANCELLED → close_cb, но wake не дошёл | Driver thread проблема ИЛИ dispatch_ready не вызван |
| `parked[s]=false`, fiber в `MCO_SUSPENDED`, не в deque | fiber парк'ил себя в legacy парке, не в driver-park | mismatch routing (sleep пошёл legacy путём вместо driver) |
| `armed_sleeps_head != NULL` после cancel | CANCEL_SCOPE job не выполнен driver'ом | Driver thread залип ДО обработки CANCEL_SCOPE |

**Deliverable:** snapshot + классификация какой паттерн (или новый).

#### Step 4 — Trace driver thread state (1.5h)

Если driver thread не в idle `uv_run` — где?
```gdb
(gdb) thread <driver-tid>
(gdb) bt full
(gdb) print _nova_driver.job_head
(gdb) print _nova_driver.job_tail
```

Сколько jobs в очереди, какие? `_nova_diag_drv_*` counters — что показывают?

```gdb
(gdb) print _nova_diag_drv_close_cb
(gdb) print _nova_diag_drv_wake_called
(gdb) print _nova_diag_drv_wake_cas_won
(gdb) print _nova_diag_drv_dispatch_cross
(gdb) print _nova_diag_drv_worker_drained
(gdb) print _nova_diag_drv_sleep_returned
```

Сравнить: должно быть `close_cb == wake_called`, `wake_called == sleep_returned + still_parked`.

**Deliverable:** counter delta анализ + hypothesis формулировка.

#### Step 5 — Cross-check with `parked[]` field в нашем понимании (1h)

Параллельно — посмотреть, может быть проблема **не в park/wake**, а в **routing**: fiber парк'ится в один scope, а wake идёт в другой.

```gdb
(gdb) print _workers[0].scope == _workers[1].scope  # должно быть false
(gdb) # для каждого fiber'а supervised scope: какой его _nova_active_scope?
```

Также проверить `dispatch_ready` ptr во всех worker scopes — не NULL ли где-то?

**Deliverable:** или подтверждение что routing корректен, или указатель на bug в routing.

**Deliverable:** документ `docs/plans/83.11-diagnostics-report.md` с findings + recommendation.

## Phase B — Option B (2-4 days, Opus 4.7, **ТОЛЬКО если Phase A подтвердит**)

**Model:** Opus 4.7
**Effort:** Maximum
**Thinking:** ON (mandatory — high-stakes design + rewrite)

### Trigger

Phase A Step 6 deliverable = "архитектурный rewrite неизбежен, гипотеза такая-то".

### Scope (fixed)

Заменить `parked[]` + `pending_wake[]` + `dispatch_ready` callback на единый `NovaWakerState` атомик per slot.

**State machine (port из Tokio task::state):**
```c
typedef struct {
    atomic_uint64_t state;  /* packed bits:
        bit 0-7:   FiberState (NEW/RUNNING/PARKED/WOKEN/DEAD)
        bit 8-15:  WakeCount (saturating counter)
        bit 16-23: refcount (для cancel-during-resume guard)
        bit 24-63: reserved (slot generation, ABA guard)
    */
} NovaWakerState;
```

**API замены:**
- `nova_sched_park(scope, slot)` → CAS state RUNNING→PARKED, yield
- `nova_sched_wake(scope, slot)` → CAS state PARKED→WOKEN, push на deque если won
- Все 10+ callers `parked[]` переписаны:
  - `nova_runtime_cancel_worker_fibers`
  - `nova_sched_count_parked`
  - `nova_supervised_run_impl`
  - `nova_scope_cancel_wake_all`
  - `nova_sched_is_parked`
  - Plan 103.x sync primitives (channels, mutex, semaphore, condvar, barrier, CountDownLatch)

### Acceptance criteria (firm)

- `stress_iso_3e`: **30/30 PASS** (target 100%)
- `supervised_cancel_stress_test` (3 tests): **30/30 each**
- Plan 103.x conformance suite: **no regression**
