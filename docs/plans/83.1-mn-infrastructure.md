# Plan 83.1: M:N-инфраструктура (pre-flip)

> **Создан:** 2026-05-22 (декомпозиция [Plan 83](83-mn-default-on-gomaxprocs.md)).
> **Статус:** 📋 proposed, не начат.
> **Приоритет:** P2.
> **Зависимость:** нет — вся работа безопасна при текущем opt-in дефолте.
> **Закрывается независимо** от Plan 82 (в отличие от [83.2](83.2-mn-default-flip.md)).
> **Родитель:** [Plan 83](83-mn-default-on-gomaxprocs.md) (roadmap-индекс,
> §3 — общие требования паритета П1-П6).

---

## 1. Scope

Вся инфраструктура GOMAXPROCS-конфигурации **без** изменения дефолта.
После 83.1 M:N остаётся **opt-in** — но: число worker'ов резолвится
по-взрослому, есть `NOVA_MAXPROCS`, `runtime.init` стал опциональным
тюнером, worker-пул ленивый, `nova test`/`bench` не страдают от
oversubscription. Это делает флип ([83.2](83.2-mn-default-flip.md)) —
одним маленьким обозримым изменением.

Полезно само по себе даже без флипа: контейнерные деплои получают
cgroup-correct число worker'ов при `runtime.init(0)`, появляется
`NOVA_MAXPROCS` и lazy-spawn.

## 2. Что уже есть (перепроверено)

- `nova_runtime_init(int)` (`runtime.c:522`) — при `n<=0` зовёт
  `uv_available_parallelism()`. libuv 1.52 — уже cgroup+affinity-aware
  (ChangeLog: «linux: fix uv_available_parallelism using cgroup»).
- Worker-пул **eager** (`runtime.c:578-623` — цикл `uv_thread_create`).
- Авто-shutdown'а нет — `nova_runtime_shutdown` зовётся только явно.
- `runtime.nv` API: `init`/`shutdown`/`worker_count`/`is_initialized`/
  `current_worker_id`/`yield`. Геттера «целевой maxprocs» нет.

## 3. Фазы

### Ф.1 — Worker-count resolution (~1 день)

- Источник `NumCPU` — **`uv_available_parallelism()`** (уже cgroup+
  affinity-aware; **не** переизобретать через `sysconf`/`GetSystemInfo` —
  это была бы регрессия по cgroup-корректности).
- Порядок разрешения (Plan 83 §3 П6): `explicit runtime.init(n) > ENV
  NOVA_MAXPROCS > uv_available_parallelism()`. Единая функция-резолвер.
- Клэмп `[1, 1024]`; значение > 1024 — только через явный env, с
  диагностическим warning.
- Динамический re-read cgroup-квоты (Go 1.25) — **followup**, в spec
  отметить как известную дельту vs Go.

### Ф.2 — ENV `NOVA_MAXPROCS` (~0.5 дня)

- Парсинг/валидация `NOVA_MAXPROCS` (аналог `GOMAXPROCS`; имя в стиле
  существующих `NOVA_*`: `NOVA_TARGET_OS`, `NOVA_FEATURES`, …).
- `NOVA_MAXPROCS=1` — явный single-worker.
- Невалидное значение (не число, ≤0, мусор) → понятная диагностика на
  stderr + fallback на default (`uv_available_parallelism`), не abort.

### Ф.3 — API reshape: `runtime.init` → опциональный тюнер (~2 дня)

- Рантайм авто-конфигурируется на resolved `maxprocs` без вызова `init`
  (число вычислено, но потоки ещё не подняты — см. Ф.4).
- `runtime.init(n)` — **опциональный** override, валиден только **до
  первого `spawn`** (до материализации пула).
- Новый getter `runtime.maxprocs()` — целевое число worker'ов (аналог
  `runtime.GOMAXPROCS(-1)`). Отличается от `worker_count()`:
  `maxprocs()` = цель, `worker_count()` = фактически поднятые (при
  lazy-spawn до первого `spawn` = 0).
- Семантика `init` **после** старта пула — **ошибка с диагностикой**
  (не молчаливый no-op: тихий no-op маскировал бы баг конфигурации).
- Обновить шапку `std/runtime/runtime.nv` — убрать «opt-in proof of
  concept / не для production».

### Ф.4 — Lazy worker-пул + auto-shutdown (~2-3 дня)

- **Lazy spawn (Plan 83 §3 П4, V1):** на старте процесса — 0 worker-
  потоков; resolved `maxprocs` лишь вычислен. На первом worker-bound
  `spawn` поднимается весь пул `maxprocs`. V2 (инкрементальный рост
  пула, полный Go-`M`-паритет) — followup.
- **Sysmon-поток (Plan 44.7) — тоже ленивый:** поднимается **вместе с
  пулом** (он нужен только для preemption worker'ов), не на старте
  процесса. Итог: **программа без `spawn` (hello-world) исполняется на
  одном — главном — потоке процесса.** 0 worker-потоков, 0 sysmon;
  libuv-threadpool тоже ленив (создаётся при первом `uv_queue_work` —
  `print` его не триггерит). Паритет с Go или лучше — у Go-рантайма
  sysmon живёт всегда.
- **Auto-shutdown:** codegen эмитит `nova_runtime_shutdown()` в эпилоге
  `main` (детерминированно); `atexit` — fallback для `exit()`-путей.
  Учесть: join worker'а, залипшего в блокирующем вызове (зависит от
  [83.3](83.3-blocking-effect-threadpool.md) — до её закрытия join
  может затянуться на FFI-блокировке; documented-ограничение).
- Семантика `detach`-работы (D50 `Detach`) при shutdown — drain либо
  явно documented-kill; решить в Ф.4, зафиксировать.

### Ф.5 — Thread-budget для `nova test` / `bench` (~1-2 дня)

Закрыть П3-oversubscription **до** флипа (иначе 83.2 регрессирует
CI-время):
- `nova test` гоняет тест-файлы как параллельные subprocess'ы
  (`--jobs num_cpus`). Test-runner выставляет subprocess'ам бюджет
  `NOVA_MAXPROCS`, чтобы суммарно worker-потоков ≈ `NumCPU`, не
  `NumCPU²`. lazy-spawn (Ф.4) уже спасает тесты без `spawn`
  (большинство) — бюджет нужен для concurrency-тестов.
- `bench`: микро-бенчи по умолчанию `NOVA_MAXPROCS=1` (иначе
  M:N-шум искажает single-thread-замеры); параллельные бенч-workload'ы —
  явный opt-in.
- Проверка: время полного `nova test` не растёт после Ф.5 (замер до/после).

## 4. Acceptance

- `runtime.init(0)` поднимает cgroup-correct число worker'ов в
  контейнере (а не «железные» CPU).
- `NOVA_MAXPROCS` override'ит; порядок разрешения (П6) соблюдён;
  невалидное значение → диагностика + fallback.
- `runtime.maxprocs()` возвращает целевое число; `init` после старта
  пула → диагностируемая ошибка.
- Hello-world (без `spawn`) создаёт **0** worker-потоков.
- `nova test` не регрессирует по времени.
- Дефолт **остаётся opt-in** — 83.1 ничего не «раздаёт»; 0 регрессий
  в полном прогоне.

## 5. Риск

- Низкий: вся работа аддитивна при сохранённом opt-in дефолте.
- Lazy-spawn V1 (весь пул на первом `spawn`) — программа с одним `spawn`
  получает `NumCPU` потоков (Go создал бы ~1-2). Приемлемо для V1;
  инкрементальный рост — followup, отмечен честно.
- auto-shutdown join при FFI-блокировке worker'а может затянуться —
  смягчается [83.3](83.3-blocking-effect-threadpool.md); до неё —
  documented.
