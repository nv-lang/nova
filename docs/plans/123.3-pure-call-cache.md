## 1. Контекст

D24 `Purity::Pure` infrastructure (Plan 33.1 + 33.2 SCC inference):
methods могут быть annotated `@pure` (или inferred Pure via SCC).
Pure methods have no effects, no side-effects, no mutation of state.

**Common pattern:** hot-path methods вызывают `@<pure_method>()`
многократно (например `@len()`, `@is_empty()`, `@capacity()`).
Codegen эмитит full call sequence per invocation, even though
result is deterministic.

Example:
```nova
fn ReadBuffer @advance(n int) -> Result[(), BufError] {
    if @pos + n > @len() { return Err(...) }   // 1st @len() call
    @pos = @pos + n
    Ok(())
}

fn ReadBuffer @available() -> int {
    @len() - @pos                              // 1 @len() call
}

fn ReadBuffer @double_check() -> bool {
    @len() > 0 && @len() < 1_000_000           // 2 @len() calls
}
```

V3 caches `@<pure_method>()` results within scope where:
- Method is `Purity::Pure` (annotated or inferred).
- Accessed ≥ threshold times.
- No `@<F> = ...` mutation anywhere в body (conservative — V3.1
  may refine с region-based caching).

## 2. Design

### 2.1 V3 scope

**Targeted:** `Call { func: Member { obj: SelfAccess, name: M }, args:
[] }` where:
- Method `M` registered as Pure (FnDecl.purity == Purity::Pure +
  same receiver type).
- Empty args list (или V3.1 — simple literal args).

**Excluded в V3:**
- `@field.<pure_method>()` chains — Plan 123.4 territory.
- Pure calls with non-trivial args.
- Pure calls на parameters / locals (только self).
- Effectful methods (Purity::Effectful / Unknown).

### 2.2 Eligibility

Per method body:
1. Build pure_methods registry: `HashMap<TypeName, HashSet<MethodName>>`
   из FnDecl с Purity::Pure.
2. For each fn body, scan calls matching V3 pattern.
3. Group by method name. Count occurrences per group.
4. For each group: if count ≥ threshold AND no @field write anywhere
   in body AND no closure-capture of any field used by method AND
   no concurrent body — emit cache.

### 2.3 Cache placement + naming

- Cache let inserted at body start (prefix).
- Naming: `_at_<method>_call` (distinct от D217 `_at_<F>` и D218
  `_at_<F>_loop`).
- Collision avoidance: numeric suffix `_N`.

### 2.4 Composition с D217 + D218

V3 phase runs **between** D218 LICM and D217 per-fn caching:
- D218 first — hoist loop-invariant field reads.
- V3 — cache pure-call results.
- D217 last — per-fn ro/mut field caching, sees V3 cache locals as
  regular locals (no @field interference).

### 2.5 Invalidation semantics

**V3 conservative:** if **ANY** `@<F> = ...` assignment exists в method
body, skip all pure-call caching (entire body). Rationale: without
knowing which fields `M` reads (frame-tracking via D24 §reads
infrastructure exists but complex), assume mutation invalidates.

**V3.1 refinement (followup):** use `f.reads` frame information to
determine which fields the pure method reads. Cache valid until any
of those fields written.

### 2.6 Edge cases

- **Recursive pure methods:** cache still valid (recursion-result
  identical per invocation в same scope).
- **Generic pure methods:** receiver type generic — pure_methods
  registry keyed by base type name (generic-agnostic).
- **Override:** в Nova нет dynamic dispatch для receiver-based pure
  methods (static dispatch always). Safe.

## 5. Risk register

- **R-3.1:** `@pure` annotation drift. Mitigation: type-checker
  enforces (existing D24 infrastructure).
- **R-3.2:** Pure method с allocation — caching reduces alloc count.
  Mitigation: distinguish observably-pure из contract-pure
  (V3.1 followup).
- **R-3.3:** Composition с D218 LICM — LICM may hoist `@<F>` whose
  pure method depends on; V3 must see post-LICM AST. Mitigation:
  D218 LICM → V3 pure → D217 order respects this.
