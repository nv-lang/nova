> **Зависимости:** [Plan 95](95-builtin-sum-method-mono.md) ✅ (его

## Зачем

Plan 95 перенёс на Nova-body 4 чистых тег-предиката (`is_some`/
`is_none`/`is_ok`/`is_err`). Анализ остальных 14 методов показал
**5 ещё переносимых без новой инфры** — это методы, которым:
- не нужен Fail-handler (не `unwrap`);
- не нужен closure-applying (не `unwrap_or_else`/`map`/`ok_or` без
  user-кода);
- не нужно method-level generic `[U]`/`[E]`/`[F]` (не `map`/`map_err`/
  `ok_or` от UserT);
- не нужно Plan 98 inference (return type — receiver type-params
  напрямую).

| Метод | Текущая C-магия | Тело на Nova |
|---|---|---|
| `Option.unwrap_or(d)` | `Nova_Option_method_unwrap_or_<T>` per-T | `=> match @ { Some(v) => v, None => d }` |
| `Option.or(o)` | `Nova_Option_method_or_<T>` per-T | `=> match @ { Some(v) => Some(v), None => o }` |
| `Result.unwrap_or(d)` | `Nova_Result_method_unwrap_or_<n>` non-per-T | `=> match @ { Ok(v) => v, Err(_) => d }` |
| `Result.ok()` | `Nova_Result_method_ok_<n>` non-per-T | `=> match @ { Ok(v) => Some(v), Err(_) => None }` |
| `Result.err()` | inline emit `emit_c.rs:11903+` | `=> match @ { Ok(_) => None, Err(e) => Some(e) }` |

## Не входит в Plan 95.bis (требует доп. инфры)

- **`unwrap`** (Option, Result) — нужна работающая `fail()` в обычном
  Nova (handler-dispatch, Plan 61 lineage). Отдельный план.
- **`unwrap_or_else`** (Option, Result) — closure-applying (D75 closure
  ABI, currently fragile).
- **`map`** (Option, Result), **`map_err`**, **`ok_or`** — closure-
  applying + method-level generic (`[U]`/`[F]`/`[E]`) + Plan 98 для
  return-type inference (`Option[U]`, `Result[U,E]`).

Эти 8 методов остаются C-routed; Plan 95.bis их не трогает.

## Архитектурное уточнение (vs Plan 95)

Plan 95 Ф.5 для `Result.is_ok`/`is_err` использует
`result_method_c_name(obj_ty, base)` для имени mono'd функции — это
даёт **суффиксированное** имя (`Nova_Result_method_is_ok_<n>`) для
mono'd `NovaRes_<n>*` и **короткое** (`Nova_Result_method_is_ok`) для
legacy `Nova_Result*`. После удаления `#define`-алиасов из `array.h`
(Plan 95 Ф.5.2) legacy-callsites сломались бы, **но их не оказалось**
(полный test PASS, Plan 59 mono'd всё).

В Plan 95.bis для `unwrap_or`/`ok` `array.h` всё ещё содержит
**активные** `#define`-алиасы (`Nova_Result_method_unwrap_or →
_nova_int_nova_str`, `Nova_Result_method_ok → _nova_int_nova_str`).
Если оставить короткое имя в dispatch'е — конфликт: my Nova-body
эмитится коротким именем + #define подменяет в его теле любые
references → возможна C-redefinition / undefined-symbol.

**Решение Plan 95.bis:** изменить Result `DeclaredBody`-dispatch чтобы
mono-имя **всегда суффиксировалось** (даже для legacy `Nova_Result*`).
Тогда:
- legacy `Nova_Result*` callsite → my dispatch emits `Nova_Result_method_
  unwrap_or_nova_int_nova_str(obj_c, ...)` — даже если obj_c имеет тип
  `Nova_Result*`, cast не нужен (C-funcptr-call с неявной адресацией?
  → нет, это явная type-mismatch). Поэтому **проверить, что legacy
  call-sites не существуют** (как Plan 95 Ф.5).
- mono'd `NovaRes_<n>*` callsite — то же суффиксированное имя.

Если test покажет legacy-проблему — добавить cast в dispatch (`(NovaRes_
<n>*)obj_c`). Решение в Ф.0.1.
