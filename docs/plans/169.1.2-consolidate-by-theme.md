# Plan 169.1.2 — Консолидация positive-тестов (по плану / по группе сути)

> **Под-план [Plan 169.1](169.1-test-suite-profiling-and-speedup.md)** (ось «скорость»).
> **Создан:** 2026-06-19. **Статус:** 📋 proposed.
> **Приоритет:** P1 — лучший ROI по CU/час (механика, низкий рантайм-риск).
> **Выигрыш (базовый):** −239 CU (39 pos-only мульти-папок → folder-module).
> **Выигрыш (тематический):** потенциально больше (см. Уровень 2).

## Контекст

Перепись 169.1: 72 мульти-module папки. Категория (A) — 39 pos-only,
не cross-resolve → консолидируемы. Метод базовый — ordinal-suffix rename
(как Cat-B, `scripts/catb_convert.py`).

## Стратегия — два уровня

### Уровень 1 (базовый): папка плана → один folder-module

Каждая `plan_NN/` с несколькими positive-файлами (каждый свой `module`) →
объединить в `module nova_tests.plan_NN`. Конфликты имён разрешать
ordinal-suffix rename. Крупнейшие: `plan91_12` (21), `plan67` (14),
`plan103_2`/`plan91_15` (11), `plan144_0` (10).
- **−239 CU**, механически, готовым скриптом.

### Уровень 2 (агрессивный): группа сути → один модуль

**Замысел (запрос 2026-06-19):** не всегда есть смысл держать модуль на каждый
номер плана. Если несколько planNN относятся к ОДНОЙ тематической группе
(синтаксис, файберы/concurrency, протоколы, контракты, …) — их можно слить в
ОДИН модуль по сути теста, а не по номеру плана:
- `plan55`/`plan56`/syntax-planNN → `module nova_tests.syntax`
- concurrency-planNN (plan83*/plan103*) → `module nova_tests.concurrency`
- protocol-planNN → `module nova_tests.protocols`

→ десятки plan-папок одной темы схлопываются в единицы модулей. Выигрыш по CU
существенно больше Уровня 1.

**Цена / риски Уровня 2:**
- **Конфликты имён растут** — чем больше файлов в одном модуле, тем больше
  одноимённых top-level (нужен агрессивный ordinal-rename, имена менее читаемы).
- **Теряется связь «тест ↔ план»** — труднее найти, к какому плану относится
  упавший тест (сейчас путь = `plan91_12/...`).
- **cross-resolve и negative** в группу НЕ включать (категории B/C/D из 169.1).

**Решение по Уровню 2 — нужна таксономия тем** (какие planNN в какую группу).
Это отдельное design-решение. Уровень 1 от него не зависит → делать Уровень 1
первым, Уровень 2 — опционально после, если ROI оправдан.

## План работ

1. **Ф.1 (Уровень 1):** прогнать `scripts/catb_convert.py` (или расширенный) по
   39 pos-only папкам категории (A). Skip: cross-resolve (B), negative-heavy (C),
   partial (D — отдельно в [169.1.3](169.1.3-consolidate-partial-d.md)).
   Регресс затронутых папок (0 новых FAIL). **−239 CU.**
2. **Ф.2 (Уровень 2, опц.):** составить таксономию тем; слить plan-папки одной
   темы в `module nova_tests.<theme>`. Замерить дополнительный −CU vs цену
   навигации. Решение go/no-go по данным.

## Замечания

- Уровень 1 — чистая выгода, низкий риск, рекомендуется безусловно.
- Уровень 2 — экспериментальный; делать только если −CU существенно перевешивает
  потерю co-location «тест ↔ план». Возможно частично (только крупные группы:
  syntax, concurrency).
- Не трогать: 674 compile-error (жёсткий пол), cross-resolve группы (B).
