# Plan 57.E.2 — Distributed bench coordination (design sketch)

> **Status:** design-sketch (not implemented в этой phase).
> **Estimate when implemented:** ~500 LOC, 2-3 dev-days.
> **External dependencies:** SSH client + key setup per remote.

## Motivation

Single-machine bench results вводят bias по hardware. Production
Nova users могут запускать на mix of:
- Linux Intel Xeon (CI)
- Linux ARM (cloud)
- macOS M-series (developer machines)
- Windows x86_64 (Windows developers)

Per-machine baselines (закрыто Plan 57.D.4) разделяют results, но не
позволяют parallel measurement многих hardware configurations за один
PR cycle.

**Distributed bench**: один command запускает идентичный bench suite
параллельно на N remote machines, агрегирует results в single dashboard.

## Architecture

```
                        ┌──────────────┐
                        │ orchestrator │  (local dev/CI)
                        │   nova bench │
                        │   remote run │
                        └──────┬───────┘
                               │
                ┌──────────────┼──────────────┐
                │              │              │
            SSH/repo     SSH/repo        SSH/repo
                │              │              │
        ┌───────▼──┐    ┌──────▼───┐    ┌─────▼────┐
        │ runner-1 │    │ runner-2 │    │ runner-3 │
        │ x86 Lin  │    │ ARM Lin  │    │ macOS M3 │
        └──────────┘    └──────────┘    └──────────┘
              │              │              │
              │ rsync result.json back to orchestrator
              ▼              ▼              ▼
          orchestrator: aggregate → dashboard
```

## API sketch

```
# Config: ~/.nova-bench-remotes.toml
[remote.linux-xeon]
host = "perf-1.example.com"
user = "bench"
repo = "/home/bench/nova-lang"
runner_id = "linux-xeon-perf"
ssh_key = "~/.ssh/nova_bench"

[remote.arm-cloud]
host = "192.0.2.5"
user = "bench"
repo = "/srv/nova-lang"
runner_id = "linux-arm-perf"

# Command (orchestrator-side):
nova bench remote run <bench-file> \
    --remotes linux-xeon,arm-cloud \
    --gather-into ./remote-results/

# Под капотом:
# 1. ssh user@host "cd repo && git fetch && git checkout <sha>"
# 2. ssh user@host "cd repo && nova-cli/target/release/nova bench run \
#                    <bench-file> --out /tmp/r.json"
# 3. rsync user@host:/tmp/r.json ./remote-results/<runner_id>.json
# 4. После всех — nova bench dashboard with --multi-runner mode.

# Helpers:
nova bench remote setup <name>     # ssh-keygen + copy-id helper.
nova bench remote ping              # health check всех configured.
nova bench remote sync-repo         # git pull on all remotes.
```

## Implementation plan

1. **Phase 1: config + ping (~100 LOC).**
   - `~/.nova-bench-remotes.toml` parsing через minimal TOML (как
     `bench.toml`).
   - `nova bench remote list` / `nova bench remote ping <name>`.
   - SSH spawn через `std::process::Command("ssh", ...)`.

2. **Phase 2: run orchestration (~200 LOC).**
   - `nova bench remote run` spawn'ит N parallel ssh sessions.
   - Captures stdout/stderr per-remote → temp files.
   - rsync gathered results.
   - Optional `--fail-fast` flag.

3. **Phase 3: aggregation (~150 LOC).**
   - dashboard.rs extended to read multi-runner data dir.
   - Per-runner overlay в trend chart (separate series per runner).
   - "Compare runners" view (side-by-side table).

4. **Phase 4: testing + docs (~50 LOC + docs).**
   - E2E test с localhost SSH (если CI supports).
   - perf-conventions.md additions для multi-machine setup.

## Risks

- **R1: SSH key management** — нужен setup workflow + secure storage
  recommendations. Out-of-scope for bench tool; document in
  perf-conventions.md.
- **R2: Repo state drift** — orchestrator checkout'ит specific SHA
  чтобы avoid mismatched code. Failure mode: ssh exec errors → log
  + skip that runner.
- **R3: Network noise** — slow ssh может biased wall-clock comparison.
  Mitigation: bench measurement is local; only metadata transferred.
- **R4: Permission model** — bench machine runs `nova bench` which
  spawns child processes. Need restricted user (non-root) с limited
  sudo для governor setup.

## Acceptance criteria (when implemented)

- [ ] Config file parses + `remote list` validates each entry.
- [ ] `remote ping` succeeds for всех reachable remotes.
- [ ] `remote run <bench>` спавнит N parallel ssh sessions, gathers
      results в local dir.
- [ ] Dashboard generates с per-runner overlay charts.
- [ ] E2E test (localhost SSH) verifies end-to-end flow.

## Alternative considered

**Centralized job queue** (Kubernetes Job, Nomad task) — more
production-grade но требует cluster infra. SSH simpler для dev/small
team, fully self-contained.

## When to implement

После accumulation 1+ year history showing ROI of multi-runner data
для actual regression detection.
