# Plan 48.1 — Cross-module generic template registration ordering

> **Статус:** 🟡 IN PROGRESS 2026-06-03
> **Parent:** [Plan 48 — Mono pass](48-monomorphization.md)
> **Trigger:** Plan 91.13 (Ф.3 JSON conformance) wedged на cross-module
> mono'd generic — `JsonValue.@as_object() -> Option[HashMap[str, JsonValue]]`
> emit'ит forward-decl с `NovaOpt_nova_int` (erased), body с `NovaOpt_Nova_HashMap_p`
> (mono'd) → CC-FAIL type mismatch.

## Зачем

Текущий emit pipeline:

```
1. emit_module() → for each type_decl in module.items:
     generic_type_templates.insert(t.name)        // ← registers HashMap, Vec, etc.

2. emit_fn_forward_decl() for each fn:
     type_ref_to_c(return_type)                   // ← reads generic_type_templates
       └─ if Option[HashMap[K,V]]:
            recurse → type_ref_to_c(HashMap[K,V])
                       └─ checks generic_type_templates["HashMap"]
                          - PRESENT (same module) → mono mangle ✓
                          - ABSENT (cross-module) → fallback Nova_HashMap* (erased) ✗
                       returns "Nova_HashMap*"
            Option case: inner_c = Nova_HashMap* (not mono'd)
            check is_concrete_type for "HashMap" → false (mono name expected)
            returns "NovaOpt_nova_int" (erased)
       returns "NovaOpt_nova_int"

3. emit_fn body (later, after worklist drain registers HashMap mono instance):
     match @ { Object(m) => Some(m), _ => None }
       infer match arms → result_ty = "NovaOpt_Nova_HashMap_p" ✓
       _nv_match_31 declared as NovaOpt_Nova_HashMap_p
       return _nv_match_31 → returns NovaOpt_Nova_HashMap_p
       but fn signature said NovaOpt_nova_int → C type mismatch
```

Корневая причина: `generic_type_templates` registration происходит per-module
(step 1 — только items текущего module). Cross-module imports (`import HashMap
from std.collections.hashmap`) НЕ регистрируют templates в потребляющем модуле.

Plan 48 (original) предполагал single-module compilation. Cross-module mono
работает для **direct method calls** (через emit_call dispatch), но для **type
references в return/param signatures** registration gap проявляется.

## Принцип решения

**Eager registration:** перед forward-decl pass'ом собрать ВСЕ generic type
templates из:
1. Текущего module (как сейчас)
2. **Всех импортированных модулей** (resolve через manifest + import graph)
3. ExternalRegistry (как сейчас)

Это даёт type_ref_to_c знание о всех generic templates до того, как он
понадобится для signatures.

## Risks

- **High**: pass ordering — самая хрупкая часть codegen. ~1555 fixtures
  могут потерять баланс. Mitigation: incremental commits + per-phase
  test run.
- **Medium**: eager registration triggers premature mono → emit'ит unused
  mono'd structs → bloat. Mitigation: register template definitions only,
  не enqueue instances. Instances остаются on-demand.
- **Low**: cross-module name collision. Mitigation: ExternalRegistry
  pattern already handles это (Plan 103.5).
