# Plan 33.3: Контракты — Advanced (`#pure` views + ghost + quantifiers + FP/strings + perf + Dafny-parity)

> Часть [Plan 33](33-contracts-implementation.md). Поднимает [33.2](33.2-contracts-imperative.md)
> до уровня **«не хуже Dafny / Creusot»**: handler-state контракты,
> ghost state, bounded quantifiers, IEEE 754 FP, string/set/map теории,
> incremental cache, parallel verification, cross-check Z3↔CVC5,
> module strict mode, `#trusted` external, AI-friendly JSON diag,
> Dafny-tutorial port (20 примеров).

## Scope

**Включено:**
- `#pure <op>` в effect/protocol-декларации + `axiom <name>(...) => <formula>`.
- `#verify` / `#trusted` атрибут на `with`-binding (обязательный для
  handler'ов эффектов с axiom'ами).
- `ghost let` / `ghost var` / `ghost <stmt>` — spec-only state.
  Никогда не emit в codegen (ни debug, ни release).
- `assume <bool>` — escape hatch для FFI / external knowledge.
  Warning категории `trust-introduced` вне `#trusted` функции.
- Bounded quantifiers: `forall x in xs : P(x)`, `exists x in xs : P(x)`,
  `forall i in lo..hi : P(i)`.
- **IEEE 754 FP** (`f32`/`f64`) — теория FloatingPoint Z3/CVC5 по умолчанию.
- Strings beyond length: `(Seq Int)` теория полноценно (substring,
  contains, index_of, concat).
- `Set[T]` / `Map[K, V]` через arrays + cardinality UF.
- Incremental SMT cache: `target/contracts-cache/<hash>.json`.
- Parallel verification через rayon thread pool.
- Cross-check Z3↔CVC5 (CVC5 — production impl, не stub).
- `#must_verify_module` strict mode на module-декларации.
- `#trusted` external fn с контрактами (контракты регистрируются как
  axioms без proof).
- AI-friendly JSON diagnostic format + CLI hooks (`nova contracts ...`).
- Dafny-tutorial port: ≥ 20 классических примеров.

**НЕ включено (вне Plan 33):**
- Полный LSP server (CLI hooks — да, full LSP — отдельный план).
- Inductive predicates (research, отдельный D-decision).
- Refinement types.
- Higher-order contracts (closure-args).
- Hardware/timing/probabilistic contracts.
- Distributed contracts через spawn/suspend.
- LLM-генерация контрактов из тестов.
- Unbounded quantifiers (запрещены по дизайну).

## Фазы 33.3

### Ф.9 — `#pure` views + axioms + `#verify`/`#trusted` handlers

**AST расширения:**
```rust
pub struct EffectDecl {
    // ... existing ...
    pub axioms: Vec<EffectAxiom>,
}

pub struct EffectOp {
    pub name: String,
    pub kind: OpKind,                       // Operation | PureView
    pub params: Vec<Param>,
    pub ret: Option<TypeRef>,
}

pub struct EffectAxiom {
    pub name: String,
    pub formula: Expr,
    pub span: Span,
}
```

**Парсер.** В `effect { ... }` или `protocol { ... }` после operations:
```
type Db effect {
    SetBalance(id AccountId, amount money)
    #pure balance(id AccountId) -> money

    axiom non_negative(id) =>
        balance(id) >= 0

    axiom balance_after_set(id, x) =>
        post(exec(SetBalance(id, x)))(balance(id)) == x
}
```

Здесь `post(action)(view)` — встроенный SMT-IR constructor «значение
`view` после выполнения `action`». Парсер принимает как built-in.

**Type-check.**
- `#pure fn(args) -> R` объявляется как обычная op, но без body.
- `axiom name(binders) => formula` — `formula` обязана быть `bool`,
  `#pure`, видны binders + все `#pure` ops эффекта.
- В контрактах функции: `#pure` op разрешена, если эффект в
  сигнатуре enclosing-fn ИЛИ в `reads` clause.
- Unique-name checks: дубликаты operations / axioms / коллизии имён
  axiom↔op — все compile errors.

**SMT encoding.**
- `#pure fn(args) -> R` → UF `Effect.view : (args) -> R`.
- `axiom <formula>` → assert global в любом scope, где эффект импортирован.
- **Axiom inconsistency check**: после загрузки всех axioms эффекта —
  `assert true; check_sat`. Unsat → compile error «axioms inconsistent».

**`#verify` / `#trusted`.** Обязательный атрибут на `with`-binding для
handler'ов эффекта, у которого объявлены axioms.

```nova
with #verify Db = postgres_handler { ... }
with #trusted Db = ffi_handler { ... }
```

`#verify` (Ф.9.7) проверяет: каждый axiom `post(action)(view) == X`
соответствует implementation handler'а:
- Для каждой `action` в axiom — symbolically analyze handler.action body.
- Доказать что `view` после возврата = `X`.

Если handler использует FFI / DB / нелинейку и symbolic analysis
fails → обязан явно пометить `#trusted`. Без `#verify` или `#trusted`
использование handler'а для эффекта с axioms — **compile error**.

**Negative tests:**
- Effect с inconsistent axioms — error.
- Handler без `#verify`/`#trusted` для эффекта с axiom — error.
- `#pure` op с body — error.
- `axiom` ссылается на non-`#pure` op — error.

**Acceptance:** R4-пример с `ensures Db.balance(to) == old(Db.balance(to)) + amount`
доказывается Z3 за <500ms при наличии правильных axioms.

### Ф.9.6 — handler-gate + naming refactor (V1 closed)

`#verify` / `#trusted` атрибут + gate при использовании handler'а в
коде, где enclosing effect объявляет axioms.

**Refactor V1 (сессия 2026-05-14):**
| Старое имя | Новое имя | Где |
|------------|-----------|-----|
| `pure_view <op>` keyword | `#pure <op>` атрибут | в effect/protocol |
| `#must_verify` | `#verify` | на `fn` |
| `#verify_handler` | `#verify` | на `with`-binding |
| `#trusted_handler` | `#trusted` | на `with`-binding |

**Trigger gate:** срабатывает только если effect объявляет хотя бы
один `axiom`. Раньше фиксировался на `pure_view` ops, но это не давало
семантики «верифицировать»: без axiom утверждать о handler'е нечего.

**Уровневая сложность для программиста (complexity ladder):**
- **Уровень 0** — обычные effect'ы и handler'ы, никакого SMT.
  Нужно знать только `effect`, `handler`, `with`. Стартовая сложность
  как в Koka / OCaml effects.
- **Уровень 1** — добавить `#verify fn` с `requires`/`ensures` на
  чистых функциях. Нужно знать только что Z3 проверит булеву формулу
  при компиляции. Как Verus level 1.
- **Уровень 2** — `#pure <op>` + `axiom` в effect'ах для verifiable
  state observation. Нужно понимать UF и что axiom — глобальная
  истина, доказываемая handler'ом. Только этот уровень требует чтения
  Dafny/Verus литературы.

В Plan 33.4 split — Уровень 0/1 для всех; Уровень 2 — opt-in для
financial / safety-critical модулей.

**Дополнительные правила V1:**
- protocol тоже разрешает `#pure` + `axiom` (symmetry с effect).
  В V1 — trusted-by-default; verification impl'а через `#verify_impl`
  отложена.
- Unique-name checks внутри effect/protocol: дубликаты operations,
  дубликаты axioms, axiom name = operation name — все compile errors.
- axiom как декларация (НЕ `#axiom` атрибут): keyword внутри
  effect/protocol-блока, расположен после operations.

### Ф.10 — Ghost state + `assume` + bounded quantifiers

**Ghost state.**

**AST расширения:**
```rust
pub enum Stmt {
    // ... existing ...
    Ghost(Box<Stmt>),                       // ghost let / ghost var / ghost block
    Assume(Expr, Span),
}
```

Парсер: `ghost let x = ...`, `ghost var y = ...`, `ghost { ... }` блок.

**Type-check ghost.**
- Видим в:
  - других `ghost`-stmts;
  - `requires`/`ensures`/`invariant`;
  - body `#pure` функций.
- Использование в non-ghost emit-code → error.

**Codegen ghost.** **Никогда не emit в C** (как Dafny). Ни в debug,
ни в release. Это значит:
- Invariant'ы, использующие ghost-данные, в debug **не проверяются
  runtime** — только SMT.
- Это документировано как design choice (паритет с Dafny).
- В тестах: если invariant с ghost'ом нарушен — найдёт только SMT,
  не runtime.

**SMT encoding ghost.** Fresh sort-vars, участвуют в encoding как обычные.

**`assume`.**

**Type-check:** `expr` обязан быть `bool`. Вне `#trusted` функции →
warning категории `trust-introduced`.

**SMT:** `(assert <expr>)` на точке вставки.

**Codegen debug:** `if (!expr) violation(NOVA_ASSUME)`.
**Release:** стирается.

**Bounded quantifiers.**

**Парсер** (новые keywords `forall`, `exists`):
```
forall <ident> in <expr> : <bool>
forall <ident> in <range-expr> : <bool>
exists <ident> in <expr> : <bool>
```

При виде unbounded `forall <ident> : <type> : <bool>` — error
«unbounded quantifier not supported; use `forall x in collection : P(x)`».

**Type-check:** `expr` после `in` — `Iter[T]` (array, range, set, map);
`x` биндится к `T`; body `bool` и `#pure`.

**SMT encoding.**

Для concrete-size `xs` (literal или len-constant) → конъюнкция
`P(xs[0]) ∧ P(xs[1]) ∧ ...`.

Для symbolic xs:
```smt
(assert (forall ((i Int))
  (=> (and (>= i 0) (< i (len xs)))
      (P (select xs i)))
  :pattern ((select xs i))))
```

Pattern annotation — детерминированный (через `select`). Без адекватного
pattern (анализ AST) → error «pattern required for symbolic quantifier».

**Acceptance:**
- Binary search **полностью** верифицируется с ghost + quantifiers + loop invariants:
```nova
fn binary_search(xs []int, target int) -> int
    requires sorted(xs)
    ensures (result == -1 && forall i in 0..xs.len() : xs[i] != target)
         || (0 <= result && xs[result] == target)
```
- `assume` outside `#trusted` — warning `trust-introduced`.
- Ghost не emit в release `.o` (verify-zero-cost CI).

### Ф.11 — IEEE 754 FP + strings + sets/maps

**FP (IEEE 754 first-class).**

Кодировка:
- `f32` → `(_ FloatingPoint 8 24)`.
- `f64` → `(_ FloatingPoint 11 53)`.

Операции: `+`, `-`, `*`, `/`, `sqrt`, `abs`, `is_nan`, `is_inf`,
сравнения — через SMT FP-теорию.

**Default — точная семантика.** Атрибут `#verify_fp_fast` опционально
переключает на Real-approximation (быстрее, но unsound для NaN/Inf —
программист принимает unsoundness явно, warning при компиляции).

Тест: `fn abs(x f64) -> f64 ensures result >= 0.0` — **НЕ** доказывается
тривиально (NaN); требуется `requires !x.is_nan()`.

**Strings (`(Seq Int)` полноценно).**

Операции: `len`, `concat`, `substring`, `contains`, `index_of`,
`starts_with`, `ends_with`. Все через SMT Seq-теорию.

Атрибут `#verify_strings_off` для opt-out (медленнее с включёнными
строками — для кода без строковых контрактов).

**Sets / Maps.**

| Nova | SMT |
|---|---|
| `Set[T]` | `(Array T Bool)` + UF `card : Set -> Int` + axioms |
| `Map[K, V]` | `(Array K (Option V))` + UF `card` + axioms |

Axioms в `axiomatize.rs` для каждой builtin-операции.

**Acceptance:**
- FP-тест с NaN edge case проходит.
- Substring/concat/contains теоремы доказываются.
- Set-union / Map-update тесты.

### Ф.12 — Incremental cache + parallel verification + Z3↔CVC5 cross-check

**Incremental cache.**

`target/contracts-cache/<hash>.json`:
```json
{
  "fn_id": "module/path::fn_name",
  "input_hash": "sha256:<AST + deps + contracts>",
  "smt_query_hash": "sha256:<encoded SMT>",
  "result": "proven" | "unknown" | "counterexample",
  "solver": "z3-4.13.0",
  "duration_ms": 142,
  "timestamp": "2026-..."
}
```

Pipeline:
1. Compute `input_hash` (Salsa-style incremental на AST + transitive
   contract-deps).
2. Lookup cache.
3. Hit + `proven` → пропустить SMT-call.
4. Hit + `counterexample` → переиграть для актуальных diagnostics.
5. Miss → verify + save.

Инвалидация по change любой dependency. Тесты:
- cache-hit: повторный build без изменений → 0 SMT calls.
- cache-invalidate: смена одного контракта → invalidate только её и
  зависящих, не остальные.

**Parallel verification.**

`worker.rs` через `rayon::ThreadPool` с N=num_cpus workers.

Каждый `verify_fn` — independent job (своя SMT-context). Coordination —
только финальный diagnostics-merge.

Z3 thread-safe: `Z3_global_param_set("parallel.enable", "true")`.
CVC5 thread-safe natively.

**Acceptance speedup ≥ 6×** на 8 cores для 100-fn corpus.

**Cross-check Z3↔CVC5.**

CVC5 — production impl, не stub. Driver через FFI (CVC5 C++ API через
Rust binding).

CI флаг `--cross-check`: каждая SMT-задача прогоняется через **оба**
solver'а. Disagreement (Z3=unsat, CVC5=sat) → compile error
«solver disagreement — possible bug, please report».

В CI dedicated job `contracts-cross-check`. На production-tests
(50 наиболее сложных) — selective cross-check (не на всех 150 чтобы
не утроить CI time).

**Acceptance:**
- Incremental: rebuild без изменений на 100-fn corpus < 2sec.
- Parallel: speedup ≥ 6× на 8 cores.
- Cross-check без disagreements на test corpus.

### Ф.13 — Module strict mode + `#trusted` external + AI-friendly JSON diag

**`#must_verify_module`.**

```nova
#must_verify_module
module banking.core {
    ...
}
```

Все функции внутри автоматически `#must_verify`. Любой unproven
контракт → compile error.

**`#trusted` external fn.**

```nova
#trusted
external fn libc_strlen(s str) -> int
    requires s.is_valid_cstring()
    ensures result >= 0
```

Без `#trusted` — `external fn` с контрактами = compile error.
С `#trusted` — контракты регистрируются как axioms (без proof).

При вызове такой fn в коде с контрактами — её ensures встраиваются как
предположения в caller'е.

**AI-friendly JSON diag.**

```bash
nova check --diag-format=json my_file.nv > diag.json
nova contracts list <file.nv>
nova contracts suggest <file.nv> <fn-name>
nova contracts verify <file.nv>
nova contracts counterexample <file.nv> <fn> <contract-id>
```

JSON schema → `docs/contracts-diag-schema.json`.

Полный LSP server — не в этом плане.
