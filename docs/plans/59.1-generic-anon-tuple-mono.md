### 1.1 Что работает сейчас

- ✅ `fn make() -> (int, str)` — anonymous positional tuple **без**

  через универсальный mechanism.
- ✅ `fn make_result[T]() -> Result[T, str]` — Plan 59 Ф.7.5
  специфический case для `Result` (mono через `NovaRes_<T>_<E>`).
- ✅ `Channel.new(cap)` для всех текущих use-cases — работает через
  3 ad-hoc special-cases:
  - `compiler-codegen/src/codegen/emit_c.rs:18435`
    `if name == "Channel" && method == "new" { ... }` ветка в
    emit_method_call route.
  - `compiler-codegen/src/codegen/emit_c.rs:20159`

  - `compiler-codegen/src/codegen/emit_c.rs:22694`
    `is_channel_new` route в `emit_tuple_destructure` для `(tx, rx) =
    Channel.new(...)` pattern.
  
  Эти ветки эмитят `Nova_ChannelPair { tx: ChanWriter*, rx:
  ChanReader* }` heap struct вместо общего generic tuple mono pipeline.

### 1.2 Что НЕ работает (bug surface)

- ❌ `fn[T] dup(v T) -> (T, T)` — generic anonymous tuple **с**
  generic param. CC-FAIL: «initializing '_NovaTuple2' with an
  expression of incompatible type». Codegen эмитит `_NovaTuple2`
  (generic placeholder) вместо инстанциированного типа.
- ❌ Spec D91 ([06-concurrency.md:2121](../../spec/decisions/06-concurrency.md#L2121))
  декларирует `fn Channel[T].new(cap) -> (ChanWriter[T], ChanReader[T])`
  но это **аспирационная нотация** — реальный return type =
  `Nova_ChannelPair*` heap record, не tuple.
- ❌ Любой user-side `fn[T] f(v T) -> (A[T], B[T])` pattern — невозможен.
  Это блокирует, например, custom pair-returning factories.

### 1.3 Plan 59 / Plan 120 разница

- **Plan 59 (закрыт):** anonymous positional tuple mono для базовых
  cases + Result-специфический mono Ф.7.5. Ф.7.4 (named tuple) и

- **Plan 59.1 (этот):** закрывает **generic anonymous tuple mono** в

## 2. Design

### 2.1 Корень проблемы (hypothesis pre-investigation)

Codegen для anonymous tuple type использует `_NovaTuple<arity>`
naming convention (например `_NovaTuple2`, `_NovaTuple3`). Эти
typedef'ы — **monomorphic** (`_NovaTuple2 { void* t0; void* t1; }` или
similar erased form). Когда generic context inferences элемент типа —
эта монотипизация не подставляется в element slots.

Гипотезы (уточнить в Ф.0 investigation):
1. Codegen эмитит `_NovaTuple<N>` placeholder type без learning что
   element types — generic params, требующие mono.
2. Type-checker fills inferred T в Type::Tuple но codegen pipeline
   не consumes этого:
   - либо AST tuple type не carry generic substitution
   - либо emit_c.rs::resolve_tuple_type не consults substitution
     env
3. Mangling: для Plan 59 Result — есть `NovaRes_<T>_<E>` per
   instantiation. Для tuple должен быть аналогичный
   `NovaTuple<arity>_<T1>_<T2>_..._<TN>` schema.

### 2.2 Architectural approach (Production-grade)

Опираясь на Plan 59 Ф.7.5 pattern (Result mono):
1. **Type-checker:** при возврате generic tuple — инстанциирует
   element types в substitution env, emit substituted Type::Tuple
   с конкретными monomorphized element types.
2. **Mangling:** schema `NovaTuple<arity>_<elem1_c_ty>_<elem2_c_ty>_..._<elemN_c_ty>`
   per instantiation (как `NovaRes_<T>_<E>`). Cache в
   `tuple_instantiations: HashSet<Vec<String>>` чтобы не emit
   duplicate typedef'ов.
3. **Codegen — emit typedef:** на первом encounter monomorphized
   tuple — emit `typedef struct { <elem1_c_ty> t0;
   <elem2_c_ty> t1; ... } NovaTuple<arity>_<elem1>_..._<elemN>;` в
   pre-fn-decl prologue (после type decls, перед fn decls — параллель
   Plan 59 Result post-typedef ordering).
4. **Codegen — emit constructor:** `(a, b)` expression в context'е
   generic tuple → emit `((NovaTuple<arity>_<...>) { .t0 = a, .t1 = b })`
   compound literal, с substituted type name.
5. **Codegen — destructure:** existing `emit_tuple_destructure` для
   general anonymous tuple должно работать через substituted type
   — нужно убедиться что `_NovaTuple<arity>` legacy path заменяется
   на substituted.
6. **Channel.new clean-up:** удалить 3 ad-hoc branches; убедиться
   что без них Channel.new корректно идёт через generic mono pipeline.

### 2.3 Backward compatibility

- Existing `fn make() -> (int, str)` без generics — продолжает
  работать. `_NovaTuple<arity>` legacy schema может остаться как
  fallback **только** для anonymous tuples без generic params.
  Альтернатива: всегда mono'ить через `NovaTuple<arity>_<elems>`
  schema (для non-generic cases substitution тривиален — emit
  `NovaTuple2_nova_int_nova_str` вместо `_NovaTuple2`). Cleaner
  long-term — единая schema.
  
  Decision Ф.0: единая schema (DECISION-A) — выкорчевать legacy
  `_NovaTuple<arity>` чтобы избежать code path divergence.
  Migration: replace all uses of `_NovaTuple<arity>` на
  `NovaTuple<arity>_<...>` через codegen rewrite + удаление legacy
  emit_anonymous_tuple_typedef.

### 2.4 Edge cases (учесть в Ф.0+)

- **Nested generic tuple:** `fn[T] f() -> (A[T], (B[T], C[T]))` —
  вложенный mono. Внутренний tuple должен инстанциироваться
  первым (post-order).
- **Tuple as element of array/option:** `fn[T] f() -> []((T, T))`
  или `fn[T] f() -> Option[(T, T)]`. Recursive mono visit.
- **Tuple as field в record:** `type Pair[T] { fields (T, T) }`.
- **Self-referential через generic:** edge case — нужны safety
  hatches (visited set в visitor).

## 5. Risk register

- **R-1: hidden Channel.new dependencies.** Beyond 3 known special-cases
  могут быть references на `Nova_ChannelPair` в runtime C headers,
  std/runtime/, или интерпретаторе. Mitigation: grep verification +
  full concurrency suite regression.
- **R-2: generic mono visitor — infinite recursion.** Self-referential
  generic tuple (`type X(T, X)` — не валидно в Nova но через type
  alias может возникнуть) → bare recursion в mono visitor.
  Mitigation: visited set в monomorphization visitor.
- **R-3: backward compat regression — non-generic tuples.** Если
  schema unification (DECISION-A) ломает existing non-generic tuple
  cases. Mitigation: pre-Ф.1 baseline, all existing tests должны
  PASS после Ф.1 implementation.
- **R-4: record-form destructure `ro { tx, rx } = Channel.new(4)`.**
  Anonymous tuple positional — record-form может не работать. 4-5
  tests могут потребовать migration на positional. Mitigation:
  decision Ф.3 — migrate или add `ro {tx,rx}` syntax sugar для
  positional tuples (низкий приоритет, можно followup).
