## 1. Контекст (полное знание из investigation)

### 1.1 Проблема

Тест `nova_tests/plan83_10/panic_in_nested_supervised.nv` TIMEOUT под
armed M:N. Pattern:

```nova
let _ = with Fail[int] = |e| { inner_caught = true; 0 } {
    supervised {
        spawn {
            supervised {
                spawn { throw 99 }
            }
        }
    }
    42
}
```

### 1.2 Root cause (точно установлено investigation)

**Файл:** `compiler-codegen/nova_rt/fibers.h:1264-1385` (`nova_supervised_step`).

Сценарий хэнга:

1. Outer supervised создаёт `q_outer` на main stack. Main = active scope.
2. Outer spawn (под armed M:N) пушит fiber `F_outer` в `q_outer` через
   `nova_runtime_spawn_into` → fiber попадает в worker W's deque, его
   `_nova_parent_scope = &q_outer`.
3. Main вызывает `nova_supervised_run_impl(q_outer)` → drain loop.
4. `nova_supervised_step(q_outer)` itera fibers:
   - `F_outer.fiber_ctx[0].base->_nova_parent_scope != NULL` (он worker-owned)
   - Строки 1297-1302: **SKIP** worker-owned fibers, `alive++; continue;`.
   - Main не резюмит `F_outer`.
5. `alive > 0`, `parked == 0`, `parked != alive` → loop крутит без yield.
   Wait, actually: `if (alive == 0) break; else if (parked > 0 &&
   parked == alive) uv_run(...);` иначе просто next iteration.
6. **Hot loop на main без yield.** Worker W теперь должен подхватить
   `F_outer` из deque.
7. Worker W pop'ает `F_outer`, mco_resume, fiber выполняет outer-spawn body:
   - Enter inner supervised — создаёт `q_inner` на worker's mco stack.
   - `_nova_active_scope = &q_inner` (TLS на worker thread W).
   - Inner spawn — `nova_runtime_spawn_into(q_inner, ...)` → пушит
     `F_inner` в w'_deque (some worker, may be W via dispatch_ready
     runnext slot). `F_inner._nova_parent_scope = &q_inner`.
8. Worker W вызывает `nova_supervised_run_impl(q_inner)` — **на worker
   thread**.
9. `nova_supervised_step(q_inner)`:
   - `F_inner.base->_nova_parent_scope != NULL` → SKIP. `alive++`.
   - Worker W теперь застрял в supervised_run_impl loop'е. Он держит
     thread. Он НЕ возвращается в его main loop (`_worker_main` runtime.c:519)
     где есть `uv_run(UV_RUN_NOWAIT)` + deque pop + runnext drain.
10. Если `F_inner` оказался в W's runnext или deque — W не достанет его,
    потому что W в supervised_run_impl loop.
11. Если `F_inner` ушёл на другого worker (steal) — он может закончиться,
    decrement `pending_remote`. Тогда `q_inner.pending_remote == 0`, drain
    выходит. OK case. Но если W's runnext был пушен `F_inner` (via
    dispatch_ready same-thread branch), F_inner застрял в W's runnext.

**Конкретный hang:** W's local runnext/deque holding F_inner, W busy
in `supervised_run_impl(q_inner)` busy-loop, F_inner never resumed.

### 1.3 Why Plan 83.10 fix didn't cover this

Plan 83.10 fix (`first_error_atomic_payload` + `nova_throw_typed`)
handles **простой случай**: `supervised { spawn { throw } }`. Там
main drives drain, worker resumes spawn, throw propagates via
`nova_fiber_report_atomic_kinded` → `q_outer.first_error_atomic`,
main detects, re-throws. Works because main and worker are different
threads.

Nested case — оба supervised drain'а на одном thread (worker), и worker
не может крутить drain + свой local loop одновременно.

### 1.4 Existing infrastructure relevant

- `_current_worker_id` (TLS) — runtime.c:489. Knows current thread's
  worker id (или -1 для main).
- `_workers[i]` — global pool.
- `nova_deque_pop` / `nova_deque_steal` — deque API.
- `w->runnext` — same-thread priority slot (Plan 83.7).
- `nova_fiber_state_cas` — Plan 83.4.5.7 double-resume guard.
- `_nova_active_scope`, `_nova_active_slot` — TLS scope tracking.

## 2. Scope

**Цель:** When `nova_supervised_run_impl` runs on a worker thread,
allow it to "pump" the worker's local fiber-pickup logic so that
fibers in the current scope (worker-owned) can be resumed inline.

**В scope:**
- Detect "worker thread" context in supervised_run_impl.
- Allow worker to pop its own deque/runnext for fibers in current scope.
- Atomic state guard (IDLE→RUNNING CAS) to avoid double-resume race
  with potential stealers.
- Verify acceptance on `panic_in_nested_supervised` test + at least
  3 other nested test patterns (write 2-3 new fixtures).

**НЕ в scope:**
- Cross-thread cancel close ([M-83.10.1-armed-cancel-timer-hang] — Plan 83.10.2).
- TLS handler race ([M-83.10.1-per-fiber-handler-tls-race]).
- Fail handler longjmp ([M-83.10.1-fail-handler-cross-mco-longjmp]).
- Performance optimization (cooperative drain on worker may be slower
  than dispatch; acceptable trade-off for correctness).

## 3. Design

### 3.1 Detect "running on a worker thread"

`_current_worker_id` is set in `_worker_main` preamble (runtime.c:520).
Value -1 (или sentinel) on main thread. Use:

```c
static inline bool _nova_on_worker_thread(void) {
    return _current_worker_id >= 0 && _current_worker_id < _n_workers;
}
```

(May already exist — search runtime.c. Otherwise add as static inline
in runtime.c with extern в нужном месте.)

### 3.2 Modify nova_supervised_step

**Файл:** `compiler-codegen/nova_rt/fibers.h:1264-1385`.

Current skip logic (line 1297-1302):
```c
if (q->fiber_ctx[i]) {
    NovaSpawnCtxBase* base = (NovaSpawnCtxBase*)q->fiber_ctx[i];
    if (base->_nova_parent_scope) {
        alive++;
        continue;
    }
}
```

New logic:
```c
if (q->fiber_ctx[i]) {
    NovaSpawnCtxBase* base = (NovaSpawnCtxBase*)q->fiber_ctx[i];
    if (base->_nova_parent_scope) {
        /* Plan 83.10.3 (2026-05-26): if we're on a worker thread and
         * the fiber's not currently RUNNING on another worker, resume
         * it cooperatively here. Required for nested supervised on
         * worker thread — worker can't return to its main pickup
         * loop while inside supervised_run_impl. */
        if (_nova_on_worker_thread() &&
            nova_fiber_state_cas(co, NOVA_FIBER_STATE_IDLE,
                                 NOVA_FIBER_STATE_RUNNING)) {
            /* We claimed it. Fall through to resume below. State
             * machine will be updated after mco_resume. */
        } else {
            alive++;
            continue;
        }
    }
}
```

Where "fall through" means the normal resume path (line 1310+) executes.
Note: this also handles parked check — parked fibers won't satisfy
IDLE→RUNNING CAS (their state would be PARKED), so they're correctly
skipped.

### 3.3 State machine integrity

Plan 83.4.5.7 invariant: only one thread can transition IDLE→RUNNING.
Our cooperative resume uses the same CAS. If another worker steals
F_inner concurrently and CAS succeeds first, our CAS fails, we skip
(alive++). No double-resume.

After resume:
- If fiber parked → state = PARKED (set by park internals).
- If fiber done (MCO_DEAD) → state = DEAD.
- Else → state = IDLE (already handled by existing post-resume code).

### 3.4 Drain other workers' deques in nested case?

Currently steal happens только в worker's main loop (`_worker_main`).
Inner `supervised_run_impl` on worker shouldn't steal — focus on its
own scope's fibers. CAS guard ensures correctness; if F_inner is on
another worker's deque (work-stealing migration), that worker resumes
it. We just spin-wait via the loop. Acceptable.

Однако if all workers tied up (each in nested supervised), and F_inner
on some other worker's runnext: deadlock-ish. But empirically: each
nested supervised owns only its scope; F_inner is on the worker that
pushed it (most likely current W's runnext via dispatch_ready
same-thread). Our cooperative resume handles this case.

### 3.5 Pending_remote semantics

Worker fiber decrements `pending_remote` on completion. If we resume
F_inner inline on worker, F_inner's completion path still decrements
`q_inner.pending_remote` (codegen emit_spawn does this in fiber
epilogue, regardless of which thread runs it). Drain loop:

```c
if (alive == 0) {
    int remote = nova_aint_load(&q->pending_remote);
    if (remote == 0) break;
    uv_run(...);
    continue;
}
```

Так и работает. `pending_remote` decrements happens inside fiber's
own emission, both for worker and our cooperative resume.

### 3.6 Effect snapshot

Existing code (line 1310+) restores `q->fiber_effect_snapshot[i]`
before mco_resume. For F_inner spawned on worker, the snapshot was
saved by `nova_fiber_spawn_into` или `nova_runtime_spawn_into` at
spawn time. Restore is correct regardless of which thread resumes.

### 3.7 Fail-frame top

Same: `q->fiber_fail_top[i]` saved/restored per fiber. No thread
dependency.

### 3.8 GC roots

Existing `_nova_gc_add_fiber_roots` / `_nova_gc_remove_fiber_roots`
work per-co regardless of thread. No change.

## 7. Risks

- **Double-resume race:** CAS guard уже Plan 83.4.5.7 invariant.
  Cooperative resume использует тот же CAS — корректно.
- **Stack growth:** Inner supervised drain runs on worker's mco
  stack (~16K default). Если deeply nested + heavy local state →
  overflow. Plan 82 Windows fiber arena (closed) addresses
  baseline stack capacity. Mitigation: новые тесты не должны быть
  deeply recursive.
- **Performance:** одна и та же thread runs inner drain + inner
  fiber → no parallelism in nested case. Acceptable trade-off (nested
  supervised is rare pattern).
- **Other workers idle:** other workers могут steal F_inner from W's
  deque. CAS guard prevents double-resume. Both workers may attempt
  simultaneously; only one succeeds. Loser doesn't decrement
  pending_remote (since никогда не запустил fiber). Pending_remote
  decrements ровно один раз from completing fiber.
