# Plan 83.14 — ReentrantMutex owner-tracking под M:N

> **Создан:** 2026-06-19 (обнаружено в ходе [Plan 169.2](169.2-nova-tests-fix-sweep.md) fix-sweep).
> **Статус:** 📋 proposed — execution-ready для агента.
> **Приоритет:** P1 — два падающих теста в concurrency-suite, реальный баг (не тест).
> **Источник:** [Plan 83](83-mn-runtime-roadmap.md) M:N runtime. Отдельно от
> уже закрытых armed-races (83.10.x) — это owner-identity баг, не race timing.

## Симптомы (2 теста, pre-existing с f4e8bdf8)

**1. POSITIVE тест ложно паникует** — `nova_tests/plan103_4/barrier_all_or_none_prop`:
```
RUN-FAIL: nova: unhandled Fail: ReentrantMutex.unlock() called by
          non-owner fiber or mutex not locked
```
Барьер-тест НЕ должен паниковать. Значит ReentrantMutex считает, что
`unlock()` вызывает не-владелец, **хотя это тот же логический владелец**.

**2. NEGATIVE тест зависает вместо паники** —
`nova_tests/plan103_3/neg/reentrant_unlock_from_wrong_fiber_neg` (TIMEOUT >10s):
```nova
mut rm = ReentrantMutex.new()
rm.lock()                        // main захватывает
supervised {
    spawn { rm.unlock() }        // B — не владелец → должен RUNTIME_PANIC
}
```
Ожидается `EXPECT_RUNTIME_PANIC`, но процесс зависает (panic из spawned fiber
внутри `supervised` не пропагируется ИЛИ owner-проверка не срабатывает).

## Гипотеза причины

ReentrantMutex отслеживает владельца. Под **M:N** fiber мигрирует между
OS-потоками (work-stealing scheduler). Если owner запоминается как **OS-thread
id** (`pthread_self`/`GetCurrentThreadId`), а не как **fiber id**, то:
- (тест 1) тот же fiber после миграции на другой поток видит «чужой» thread-id
  → ложный non-owner panic;
- (тест 2) либо обратное — другой fiber на том же потоке проходит owner-check.

Проверить в C-runtime: чем идентифицируется owner ReentrantMutex.

## Что сделать (агенту)

1. Найти реализацию ReentrantMutex owner-tracking (C runtime + `std/runtime/sync.nv`):
   `grep -rn "ReentrantMutex\|owner\|non-owner" compiler-codegen/nova_rt/ std/`.
2. Установить, чем хранится owner: thread-id vs fiber-id (`mco_coro*` / fiber
   handle). Если thread-id — **это и есть баг под M:N**.
3. Fix: owner = **fiber-identity** (стабильна при миграции между потоками),
   не OS-thread. Сверить с тем, как owner трекают Mutex/RwLock семейства
   (Plan 103.3) — возможно там та же или уже верная схема.
4. Отдельно проверить тест 2: доходит ли panic из `spawn` внутри `supervised`
   до main (re-throw). Если owner-fix чинит тест 1, но тест 2 всё ещё висит —
   это вторая, supervised-panic-propagation проблема (смежная, см. 83.10.x).
5. Прогон: `nova test nova_tests/plan103_3` (7/8→8/8) +
   `nova test nova_tests/plan103_4` (5/6→6/6). Регресс concurrency-suite.
