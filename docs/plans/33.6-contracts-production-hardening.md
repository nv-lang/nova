# Ф.1 — КРИТИКАЛ: Soundness honesty (~3-4 дня)

Эта фаза превращает «proven» из маркетингового слогана в честный SMT-гарант.
Без неё пользователь не может доверять верификации.

## Ф.1.1 — Ghost erasure в codegen (закрытие V5, Plan 33.5 Ф.1.1)

**Проблема.** `ghost let n = compute()`:
- Сейчас в [emit_c.rs:5841](../../compiler-codegen/src/codegen/emit_c.rs#L5841)
  есть `if decl.is_ghost { skip }` — но это **частичный** skip только в
  одной ветке. В других путях codegen ghost эмитится как обычный `let`
  → `compute()` вызывается в runtime, занимает memory, дёргает effects.
- Type-check НЕ запрещает читать ghost-var из non-ghost кода
  (есть `check_ghost_usage` в `types/mod.rs`, но покрытие неполное).
- В release-сборке ghost-код остаётся → нарушение zero-cost ghost
  (Verus/Dafny parity).

**Что нужно сделать (шаги):**

1. **Найти все места emit для `LetDecl`/`VarDecl`/`Stmt::Let`** в
   `codegen/emit_c.rs`. Используй `Grep` на `is_ghost` + ручной обход
   `emit_stmt` / `emit_let_stmt` / `emit_var_decl` / `emit_block`.
2. В каждом месте проверить `decl.is_ghost`. Если true — **полностью
   skip** emit (ни declaration, ни initialiser). Это касается **debug
   и release** одинаково (паритет с Verus).
3. **`ghost { stmt; stmt; }` блок** — skip целиком в codegen. Найти
   через `Grep ExprKind::Ghost`.
4. **Type-check rule:** расширить `check_ghost_usage` в
   [types/mod.rs](../../compiler-codegen/src/types/mod.rs) (поиск по `ghost_set`).
   Должен покрывать:
   - чтение ghost-var в right-hand side non-ghost assignment;
   - передача ghost-var в non-ghost fn-call (включая println, метод-вызовы);
   - return ghost-var из non-ghost fn;
   - использование в `if`/`while` condition non-ghost блока (НО: разрешено
     в `assert`/`assume`/`invariant`/`ensures`/`requires`).
   Error message: `"ghost binding '{name}' cannot be used at runtime; allowed only in requires/ensures/invariant/assert/assume/ghost-блок"`.
5. **Spec ops в ghost-блоке разрешены** — внутри `ghost { ... }` можно
   читать ghost-var; внутри `assert`/`assume` — тоже OK.
6. **`#pure` fn body** — можно читать ghost-var, т.к. `#pure` функции
   spec-only по контракту (не emit'ятся в runtime если они только в
   контрактах).

**Файлы:**
- `compiler-codegen/src/codegen/emit_c.rs` — emit_let_stmt, emit_var_decl,
  emit_block (всё что эмитит `let`/`var`/`{...}`).
- `compiler-codegen/src/types/mod.rs` — `check_ghost_usage`.
- `compiler-codegen/src/ast/mod.rs` — убедиться что `LetDecl.is_ghost`,
  `Stmt::Ghost(Box<Stmt>)`, `ExprKind::Ghost` существуют (по плану — да).

**Acceptance:**
- `ghost let n = compute(); println(n)` → compile-error
  `"ghost binding 'n' cannot be used at runtime"`.
- `ghost let n = compute(); assert(n >= 0)` → OK, в сгенерированном
  `.c` нет ни `compute()`, ни хранения `n`. Проверка:
  `grep -c "compute" target/debug/build/.../file.c` = 0.
- В release-сборке `nm file.o | grep compute` пусто (если `compute` ghost-only).

**Тесты (создать):**
- `nova_tests/contracts/ghost_runtime_use_fail.nv` (EXPECT_COMPILE_ERROR) —
  `ghost let x = 5; let y = x + 1` (read ghost в non-ghost).
- `nova_tests/contracts/ghost_pass_to_fn_fail.nv` (EXPECT_COMPILE_ERROR) —
  `ghost let x = 5; println(x)`.
- `nova_tests/contracts/ghost_return_fail.nv` (EXPECT_COMPILE_ERROR) —
  `fn f() -> int { ghost let x = 5; x }`.
- `nova_tests/contracts/ghost_in_assert_positive.nv` — `ghost let x = 5; assert(x >= 0)`.
- `nova_tests/contracts/ghost_in_invariant_positive.nv` — ghost в loop invariant.
- `nova_tests/contracts/ghost_in_ghost_block_positive.nv` —
  `ghost { let a = 1; let b = a + 1; }`.
- `nova_tests/expected_runtime/ghost_zero_cost.nv` — после compile проверить
  через `nova-codegen check --dump-c` отсутствие ghost-call'ов в выводе.

**Трудоёмкость:** 1 день.

## Ф.1.2 — Не-silent `EncodingError::Unsupported`

**Проблема.** [encode.rs:23,119-211](../../compiler-codegen/src/verify/encode.rs#L23)
возвращает `EncodingError::Unsupported(msg)` для:
- field access (`x.balance`);
- method calls (`xs.len()`);
- pattern match;
- генерик-вызовы с не-int/bool/str аргументами;
- сложные арифметические комбинации (Mul/Div с переменными).

[pipeline.rs:327,374,633](../../compiler-codegen/src/verify/pipeline.rs#L327)
ловит `Unsupported`, пишет в stderr `eprintln!("verify: skipped ...")`
и **продолжает компиляцию** с runtime fallback. Результат: пользователь
видит `// proven` в одних местах и `// runtime-checked` в других без
явной диагностики — silent downgrade.

**Что нужно сделать:**

1. **Ввести явный атрибут `#unverified`** (новый, parallel `#verify`)
   на fn — означает «программист осознанно отказывается от SMT
   доказательства, runtime fallback OK».
2. **По умолчанию** (без `#unverified`): любой `EncodingError::Unsupported`
   во время verify-pipeline → **compile error**, не silent skip. Format:
   ```
   error[E2401]: contract uses unsupported construct
     --> file.nv:12:5
   12 |     requires x.balance > 0
      |              ^^^^^^^^^ field access in contract not yet supported by SMT encoder
      = note: SMT encoder currently supports: int/bool/str literals, arithmetic, comparisons, &&/||/!, if-then-else, #pure function calls, forall/exists in bounded range
      = help: mark `fn` as `#unverified` to opt out of SMT verification (runtime check only), or extract logic into a `#pure` helper
   ```
3. **С `#unverified`** — silent skip OK, только warning categorisation:
   `warning[W2401]: fn 'transfer' opted out of SMT verification`.
4. **`#must_verify` (существующий)** + `Unsupported` → error даже строже,
   с pointer на конкретное место в формуле.
5. **Удалить все `eprintln!("verify: skipped ...")`** silent-fallback'и
   из `pipeline.rs` — должны стать структурированными `Diagnostic`-ами
   через `compiler-codegen/src/diagnostic.rs`.
6. **Сохранить TrivialBackend-категорию** — для случаев когда Z3
   недоступен (build без feature) и формула слишком сложная для
   trivial: должна выдавать `Unknown` с `UnknownReason::SmtBackendLimited`,
   но это **отдельная категория** от `Unsupported`. `Unknown` →
   compile error в `#must_verify` контексте, warning иначе (как сейчас).

**Файлы:**
- `compiler-codegen/src/lexer/token.rs` — добавить `KwUnverified` если
  attr-парсинг этого требует (`#unverified` парсится как attribute,
  не keyword — проверить).
- `compiler-codegen/src/ast/mod.rs` — `FnDecl.is_unverified: bool`.
- `compiler-codegen/src/parser/mod.rs` — парс `#unverified` атрибута
  по аналогии с `#verify`.
- `compiler-codegen/src/types/mod.rs` — gate: `#unverified` +
  `#verify`/`#must_verify` на одной fn → conflict error.
- `compiler-codegen/src/verify/pipeline.rs` — заменить все
  `eprintln!("verify: skipped")` на structured diagnostic emission;
  при `Unsupported` без `is_unverified` → push diagnostic, fail compile.
- `compiler-codegen/src/diagnostic.rs` — категория `E2401 UnsupportedInContract`.

**Acceptance:**
- `requires x.balance > 0` без `#unverified` → compile-error E2401.
- Тот же код с `#[unverified]` → warning W2401, компилируется,
  runtime check работает.
- `#must_verify` + `Unsupported` → error даже строже (не позволяется
  `#unverified` совместить).
- Regression: все существующие `nova_tests/contracts/*.nv` помечены либо
  `#verify`/`#must_verify` (если encoder может) либо `#unverified`
  (если не может). Скан: запустить `cargo run --release -- test
  nova_tests/contracts/` — все тесты явно классифицированы.

**Тесты:**
- `nova_tests/contracts/unsupported_field_access_fail.nv` (EXPECT_COMPILE_ERROR)
  с `requires x.balance > 0`.
- `nova_tests/contracts/unverified_attribute_positive.nv` — то же с
  `#unverified` → компилируется.
- `nova_tests/contracts/unverified_conflict_with_verify_fail.nv`
  (EXPECT_COMPILE_ERROR) — `#verify` + `#unverified`.
- `nova_tests/contracts/must_verify_unsupported_fail.nv`
  (EXPECT_COMPILE_ERROR) — `#must_verify` + field access.

**Трудоёмкость:** 1.5 дня.

## Ф.1.3 — `post(Action)(view)` Unknown — должен быть error по умолчанию

**Проблема.** [pipeline.rs:2083-2089](../../compiler-codegen/src/verify/pipeline.rs#L2083)
возвращает `VerifyResult::Unknown(UnknownReason::Unsupported {
detail: "post-axiom verification requires V2" })` для `post(...)`
axioms за пределами V2 scope (branching, nested handlers, FFI).
По умолчанию это silent-pass — компиляция продолжается, диагностики

**Что нужно:**

1. **Сделать `Unknown` от handler-verification fail = compile error**
   когда handler помечен `#verify` (не `#trusted`). Логика: «программист

      |                       ^^^^^^^^^^ axiom `post(SetBalance)(balance) == x` requires V2 symbolic exec
      = note: V1 supports linear handler bodies only (no if/match/loop/FFI)
      = note: detected branching at line 47 in my_handler.SetBalance

**Файлы:**

**Acceptance:**

**Тесты:**

## Ф.3.1 — Counterexample minimisation (delta-debugging)

**Проблема.** Сейчас Z3 возвращает model со всеми bindings (для каждой
declared variable). Программисту/LLM невозможно понять какие bindings
**критичны** для counterexample, а какие случайные. Особенно плохо для
формул с >5 переменных.

**Что нужно:** delta-debugging на model.

**Алгоритм:**
1. Получили `Sat(model)` от Z3, model = `{x: 5, y: 0, z: -3, w: 100}`.
2. Для каждой bindings `(k, v)`:
   a. Build new assertion: `(assert (= k v))` (зафиксировать значение).
   b. Build complement: оригинальные assertions БЕЗ исходной violated
      clause + новое equality.
   c. `check_sat` → если **ещё SAT** значит binding критичный (counterexample
      работает с этим значением); если **UNSAT** — binding случаен,
      убираем из minimal model.
3. Повторить N итераций или пока model стабильна.
4. Вернуть **minimal model** в diagnostic.

**Файлы:**
- `compiler-codegen/src/verify/backend/z3.rs` — добавить
  `pub fn minimise_model(&mut self, model: Model, assertions: &[SmtTerm]) -> Model`.
- `compiler-codegen/src/verify/pipeline.rs` — вызвать после `Sat`.
- `compiler-codegen/src/diagnostic.rs` — формат counterexample.

**Acceptance:**
- Тест: `fn f(x int, y int, z int) requires x > 0 ensures x + y > 0`.
  Counterexample должен показать `x = 1, y = -2` (минимум), не

Эта фаза добавлена после аудита соответствия плана и реализации,

### Ф.6.1 — V19 close: encoding gaps в контрактах (~2-3 дня, PROD-CRITICAL)

**Проблема.** В `verify/encode.rs` несколько AST-конструкций НЕ
попадают в pattern match вообще (catch-all без `EncodingError::Unsupported`)
или возвращают silent `Unknown` вместо явного `[CONTRACT_UNSUPPORTED]`:

- **Tuple access** (`pair.0`, `pair.1`) — нет ветки в match
- **Field access** на record типах — нет ветки в match
- **Method calls** на non-pure types — silent skip
- **Pattern matching** в контрактах (e.g. `match opt { Some(x) => ... }`) — нет ветки
- **Closures / lambda / block expressions** — catch-all без диагностики

**Soundness risk:** контракт может использовать unsupported конструкцию,
encoding молча возвращает `EncodingError::Other` или Unknown → проверка
проходит как "tautology" при `Default` mode. Пользователь видит
"verified" маркер, на деле SMT не валидировал.

**Что сделать:**

1. **Exhaustive match по `ExprKind`** в `encode_expr` —
   каждая ветка либо кодирует, либо возвращает
   `EncodingError::Unsupported(reason)` с `[CONTRACT_UNSUPPORTED]` prefix.
2. **Snapshot test** проверяет, что добавление новой `ExprKind`-варианты
   в AST требует обновления `encode_expr` (через `#[non_exhaustive]` или
   compile-time assertion на полноту).
3. **Negative-tests:**
   - `e2401_tuple_in_ensures_fail.nv` — `ensures (a, b) == (1, 2)` → E2401
   - `e2401_field_access_in_requires_fail.nv` — `requires p.x > 0` → E2401
   - `e2401_method_in_invariant_fail.nv` — `invariant list.len() == n` → E2401
   - `e2401_match_in_ensures_fail.nv` — `ensures match opt { Some(x) => ..., None => false }` → E2401
4. **Documentation:** в `nova-doc.md` чётко перечислить supported
   expression kinds для контрактов; ссылки на E2401 с примерами fix.

**Acceptance:**
- Каждая unsupported конструкция в контракте → явная E2401 с pointing к токену
- `Grep "EncodingError::Other"` в `encode.rs` ≤ 1 (только catch-all с явным reason)
- 4+ новых negative-теста PASS

**Закрывает V19.**

### Ф.6.2 — Silent fallback elimination (~1-1.5 дня, PROD-CRITICAL)

**Проблема.** В `pipeline.rs` найдены 5 мест, где `VerifyMode::Default`
+ encoding/Unknown → молчаливый skip без warning:

1. [pipeline.rs:~2125](../../compiler-codegen/src/verify/pipeline.rs#L2125) —
   Default + Unknown = runtime fallback без diagnostic
2. [pipeline.rs:~488](../../compiler-codegen/src/verify/pipeline.rs#L488) —
   loop invariant не encodable = silent skip
3. [pipeline.rs:~706](../../compiler-codegen/src/verify/pipeline.rs#L706) —
   invariant instantiation не encodable = silent skip
4. [pipeline.rs:~2157](../../compiler-codegen/src/verify/pipeline.rs#L2157) —
   body limitation в Default = silent
5. [handler_exec.rs:~1157](../../compiler-codegen/src/verify/handler_exec.rs#L1157) —
   Unknown для static axiom + handler без diagnostic

**Что сделать:**

1. **Новый warning code W2402** — "verification skipped, body uses
   unsupported construct; mark `#unverified` to acknowledge".
2. В каждом из 5 мест: вместо silent skip — `emit_warning(W2402, ...)`
   с указанием конкретной конструкции и suggest fix (`#unverified` или
   `#trusted`).
3. **Атрибут `#silent` или `#trusted_body`** для подавления W2402
   (по аналогии с `#[allow(...)]` в Rust) — для случаев, когда
   программист явно знает, что body не верифицируется.
4. **Acceptance test:** скомпилировать `nova_tests/contracts/`
   с `--warnings-as-errors` → должно падать на любой silent fallback
   без явной аннотации.

**Acceptance:**
- 5 silent locations либо emit W2402, либо имеют suppression mechanism
- Все existing тесты PASS (с правильной аннотацией где нужно)
- `nova_tests/contracts/w2402_silent_body_fallback_positive.nv`,
  `w2402_silent_loop_invariant_positive.nv`

### Ф.6.3 — TrivialBackend coverage extension (~2 дня, M)

**Проблема.** TrivialBackend сейчас умеет:
- constant folding (1+2=3)
- reflexivity (x==x)
- boolean simplifications (true && x → x)
- linear int comparisons между литералами

Этого недостаточно даже для тривиальных контрактов вроде
`ensures result == x + 1` где `x` — symbolic var (после symbolic exec
выражение `x+1 == x+1` тривиально, но текущий TrivialBackend не
видит структурного равенства).

**Что сделать:**

1. **Structural equality** — `E1 == E1` → true для любых E1
   (после α-renaming binders).
2. **Simple congruence** — `f(a, b) == f(a, b)` → true.
3. **Trivial implication** — `A ==> A` → true; `false ==> A` → true; `A ==> true` → true.
4. **Linear arithmetic chains** — `a < b && b < c ==> a < c` (transitivity).
5. **Quantifier instantiation для одного binder** — `forall x. P(x)`
   при заданном concrete `a` → проверка `P(a)`.

**Что НЕ делать (defer):** non-linear arithmetic, array theory,
quantifier alternation (forall-exists) — это работа Z3, не trivial.

**Acceptance:**
- 80% тестов из `nova_tests/contracts/`, что сейчас REQUIRES_SMT_BACKEND z3,
  должны проходить без Z3
- 10 unit-тестов в `trivial.rs` на новые шаги
- Сравнить pass-rate trivial vs z3: gap должен сократиться

### Ф.6.4 — V12 close: post-axioms V2 symbolic execution (~3-4 дня, M)

**Проблема.** Handler `post(Action)(view)` axioms сейчас:
- Static axioms (без post()) — кодируются как UF declarations + body axioms (V1, работает)
- post(Action) axioms — `Unknown("post-axiom verification requires V2")` ([handler_exec.rs:~109](../../compiler-codegen/src/verify/handler_exec.rs#L109))

Программист, который хочет верифицировать handler с pre/post-conditions,
не может — только `#trusted` обход.

**Что сделать (high-level, требует доразработки спеки):**

1. **Symbolic execution тела handler-метода** — каждое присваивание
   `state.field = expr` → SMT assertion на new value of pure-view `state.field()`.
2. **Branch encoding** — `if cond { A } else { B }` → ITE в SMT
   (already есть для контрактов, нужно расширить для handler bodies).
3. **Loops в handlers** — defer (требует loop-invariant inference
   на body, который сам не contract → Plan 33.7+).
4. **Sound boundary:** если body содержит конструкцию вне supported set
   (FFI, complex loops, recursive call) → E2402 явный, не silent.

**Risks:**
- Это самая сложная часть Plan 33.* — требует design proposal перед
  implementation.
- Может быть откатано в отдельный Plan 33.7-handler-symbolic-exec.

**Acceptance (если фаза идёт целиком):**
- `nova_tests/contracts/handler_post_axiom_v2_positive.nv` — простой
  set/get с post() axioms проходит #verify без #trusted
- 5+ позитивных + 3+ негативных тестов на edge cases

## Ф.7.1 — Fix f14_* test gating (Plan 33.3 Ф.14 cleanup)

## Ф.7.2 — V4 entry-snapshot для `old()` (substitute_old soundness)

**Проблема.** [`pipeline.rs:2027-2038`](file:///d:/Sources/nova-lang-p33/compiler-codegen/src/verify/pipeline.rs#L2027)
сейчас делает `_old_x → x` (тривиальная подстановка). Для mut params
это **unsound**: `ensures result == old(x) + 1` где `x` потом mutated в
body — Z3 видит `result == x + 1` с current `x`, а должен видеть с
entry-state `x`.

**Что нужно:**
1. При encode body: создать fresh SMT vars `_entry_<param>` для каждого
   mut param.
2. На входе fn: assert `_entry_x = x`.
3. В `substitute_old`: `_old_x → _entry_x` (не `x`).
4. После mutation в body: encode update как assign (`x := x + 1`) к
   рабочему `x`, `_entry_x` остаётся frozen.

**Файлы:**
- `compiler-codegen/src/verify/pipeline.rs::substitute_old` — fix.
- `compiler-codegen/src/verify/pipeline.rs::verify_fn` — добавить
  entry-snapshot declarations.

**Acceptance:**
- Negative test: `fn f(mut x int) ensures result == old(x) + 1 { x := x*2; x }`
  → counterexample (current substitute_old говорит «proven» ошибочно).
- Positive test: тот же с `{ x + 1 }` body → proven.

**Трудоёмкость:** 0.5 дня.

**Проблема.** `handler_exec.rs:246-295` (verify_liskov_method) содержит 4 silent skips:

Этот код возвращает `Vec<Diagnostic>` который пушится в `report.errors`. Нужно

## Ф.3.1 — Counterexample minimisation (GA-blocker)

**Проблема.** Z3 model сейчас передаётся в `format_counterexample` как есть.

## Ф.3.3 — Pattern-aware suggested fixes

**Проблема.** Failed verify даёт «contract violation», но не подсказывает
**как** fix'ить (даже когда pattern очевиден: `x / k` без `requires k != 0`).

**Что нужно — 5 pattern matchers:**
1. `x / k` или `x % k` без `requires k != 0` → suggest «add `requires k != 0`».
2. `arr[i]` без `requires i >= 0 && i < arr.len()` → suggest bounds.
3. `n + 1` в `decreases` (вместо `n - 1`) → suggest «decreases должно убывать».
4. `result == old(x)` для mut param без frame axiom — suggest add `modifies`.
5. `ensures result.is_some()` для match-returning fn без Some-arm → suggest
   exhaustive arms.

**Файлы:**
- `verify/suggest.rs` (новый, ~200 LOC) — AST inspection + match patterns.
- `verify/pipeline.rs::format_counterexample` — call suggest.

**Acceptance:**
- 5 negative tests: каждый с одним pattern → suggest появляется в diagnostic.

**Трудоёмкость:** 2 дня.

## Ф.5.1 — In-memory subsumption cache

**Проблема.** Каждая verify-session — full re-verify. Module с 100+ fn с
одинаковыми мелкими VC (`x >= 0 || x < 0`, `a+b == b+a`) тратит CPU впустую.

**Что нужно:**
1. `HashMap<u64, CachedSat>` в `Z3Backend` (или общий cross-call).
2. Canonical form формулы: alpha-rename vars, sort terms in `and`/`or`,
   xxhash64.
3. Lookup: hit → return cached SatResult; miss → call Z3, store.
4. Bypassing через `#proof_budget(cache = false)` opt-out.
5. Stats: `nova check --verify-stats` показывает hit-rate.

**Файлы:**
- `verify/subsumption_cache.rs` (новый, ~150 LOC).
- `verify/backend/z3.rs` — `check_sat_cached(...)`.

**Acceptance:**
- 100 fn с 50 идентичных VC → wall-clock < 60% baseline.
- Cache hit-rate > 30% на std/* (если применимо).

**Трудоёмкость:** 2 дня.

## Ф.6.3 — TrivialBackend extension (congruence + transitivity)

**Проблема.** TrivialBackend handles только reflexivity + constant folding.
Промежуточные ensures `result == x + 0` или `result == 1 * x` дают Unknown.

**Что нужно (V1, low-hanging):**
1. **Congruence**: если `a == b` и `b == c` известно (через requires) → `a == c`.
2. **Identity elim**: `+ 0`, `* 1`, `&& true`, `|| false`, `not(not(x))`.
3. **Commutativity normalization**: `a + b == b + a` → true.

Не полная theory, но снимает 20-30% TrivialBackend Unknown'ов на простых VC.

**Файлы:** `verify/backend/trivial.rs`.

**Acceptance:**
- Тесты `f14_*` (которые сейчас REQUIRES_SMT_BACKEND z3): минимум 5 из 19
  становятся PASS под Trivial → можно убрать gate.

**Трудоёмкость:** 1 день.

первое строже. Программист либо забыл удалить, либо ошибочно скопировал.

**Решение:** в `verify_module` — после сбора lower/upper bounds per Var из всех
requires, если есть 2+ literal bounds для одной Var → W2402 для нестрожайших.

**Acceptance:** `redundant_requires_tighter_w2402.nv` EXPECT_COMPILE_WARNING W2402.

**Трудоёмкость:** 0.5 дня.

## Ф.22.3 — De Morgan simplification

**Проблема.** `(not (and X Y))` структурно отличается от `(or (not X) (not Y))`
хотя semantically equivalent. Может блокировать excluded middle detection.

**Решение:** в `simplify_app` для `not` — если внутри `(and ...)`/`(or ...)`,
применить De Morgan: `(not (and X Y))` → `(or (not X) (not Y))`.

**Защитный подход:** только для 2-arg cases, чтобы не raise complexity.

**Трудоёмкость:** 0.5 дня.
