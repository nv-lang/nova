# Ф.1 — КРИТИКАЛ: Soundness honesty (~3-4 дня)

Эта фаза превращает «proven» из маркетингового слогана в честный SMT-гарант.

## Ф.1.1 — Ghost erasure в codegen (закрытие V5, Plan 33.5 Ф.1.1)

**Проблема.** `ghost let n = compute()`:
- Сейчас в [emit_c.rs:5841](../../compiler-codegen/src/codegen/emit_c.rs#L5841)
  есть `if decl.is_ghost { skip }` — но это **частичный** skip только в
  одной ветке. В других путях codegen ghost эмитится как обычный `let`
  → `compute()` вызывается в runtime, занимает memory, дёргает effects.
- Type-check НЕ запрещает читать ghost-var из non-ghost кода
  (есть `check_ghost_usage` в `types/mod.rs`, но покрытие неполное).
- В release-сборке ghost-код остаётся → нарушение zero-cost ghost
  (Verus/Dafny parity).

**Что нужно сделать (шаги):**

1. **Найти все места emit для `LetDecl`/`VarDecl`/`Stmt::Let`** в
   `codegen/emit_c.rs`. Используй `Grep` на `is_ghost` + ручной обход
   `emit_stmt` / `emit_let_stmt` / `emit_var_decl` / `emit_block`.
2. В каждом месте проверить `decl.is_ghost`. Если true — **полностью
   skip** emit (ни declaration, ни initialiser). Это касается **debug
   и release** одинаково (паритет с Verus).
3. **`ghost { stmt; stmt; }` блок** — skip целиком в codegen. Найти
   через `Grep ExprKind::Ghost`.
4. **Type-check rule:** расширить `check_ghost_usage` в
   [types/mod.rs](../../compiler-codegen/src/types/mod.rs) (поиск по `ghost_set`).
   Должен покрывать:
   - чтение ghost-var в right-hand side non-ghost assignment;
   - передача ghost-var в non-ghost fn-call (включая println, метод-вызовы);
   - return ghost-var из non-ghost fn;
   - использование в `if`/`while` condition non-ghost блока (НО: разрешено
     в `assert`/`assume`/`invariant`/`ensures`/`requires`).
   Error message: `"ghost binding '{name}' cannot be used at runtime; allowed only in requires/ensures/invariant/assert/assume/ghost-блок"`.
5. **Spec ops в ghost-блоке разрешены** — внутри `ghost { ... }` можно
   читать ghost-var; внутри `assert`/`assume` — тоже OK.
6. **`#pure` fn body** — можно читать ghost-var, т.к. `#pure` функции
   spec-only по контракту (не emit'ятся в runtime если они только в
   контрактах).

**Файлы:**
- `compiler-codegen/src/codegen/emit_c.rs` — emit_let_stmt, emit_var_decl,
  emit_block (всё что эмитит `let`/`var`/`{...}`).
- `compiler-codegen/src/types/mod.rs` — `check_ghost_usage`.
- `compiler-codegen/src/ast/mod.rs` — убедиться что `LetDecl.is_ghost`,
  `Stmt::Ghost(Box<Stmt>)`, `ExprKind::Ghost` существуют (по плану — да).

**Acceptance:**
- `ghost let n = compute(); println(n)` → compile-error
  `"ghost binding 'n' cannot be used at runtime"`.
- `ghost let n = compute(); assert(n >= 0)` → OK, в сгенерированном
  `.c` нет ни `compute()`, ни хранения `n`. Проверка:
  `grep -c "compute" target/debug/build/.../file.c` = 0.
- В release-сборке `nm file.o | grep compute` пусто (если `compute` ghost-only).

**Тесты (создать):**
- `nova_tests/contracts/ghost_runtime_use_fail.nv` (EXPECT_COMPILE_ERROR) —
  `ghost let x = 5; let y = x + 1` (read ghost в non-ghost).
- `nova_tests/contracts/ghost_pass_to_fn_fail.nv` (EXPECT_COMPILE_ERROR) —
  `ghost let x = 5; println(x)`.
- `nova_tests/contracts/ghost_return_fail.nv` (EXPECT_COMPILE_ERROR) —
  `fn f() -> int { ghost let x = 5; x }`.
- `nova_tests/contracts/ghost_in_assert_positive.nv` — `ghost let x = 5; assert(x >= 0)`.
- `nova_tests/contracts/ghost_in_invariant_positive.nv` — ghost в loop invariant.
- `nova_tests/contracts/ghost_in_ghost_block_positive.nv` —
  `ghost { let a = 1; let b = a + 1; }`.
- `nova_tests/expected_runtime/ghost_zero_cost.nv` — после compile проверить
  через `nova-codegen check --dump-c` отсутствие ghost-call'ов в выводе.

**Трудоёмкость:** 1 день.

## Ф.1.2 — Не-silent `EncodingError::Unsupported`

**Проблема.** [encode.rs:23,119-211](../../compiler-codegen/src/verify/encode.rs#L23)
возвращает `EncodingError::Unsupported(msg)` для:
- field access (`x.balance`);
- method calls (`xs.len()`);
- pattern match;
- генерик-вызовы с не-int/bool/str аргументами;
- сложные арифметические комбинации (Mul/Div с переменными).

[pipeline.rs:327,374,633](../../compiler-codegen/src/verify/pipeline.rs#L327)
ловит `Unsupported`, пишет в stderr `eprintln!("verify: skipped ...")`
и **продолжает компиляцию** с runtime fallback. Результат: пользователь
видит `// proven` в одних местах и `// runtime-checked` в других без
явной диагностики — silent downgrade.

**Что нужно сделать:**

1. **Ввести явный атрибут `#unverified`** (новый, parallel `#verify`)
   на fn — означает «программист осознанно отказывается от SMT
   доказательства, runtime fallback OK».
2. **По умолчанию** (без `#unverified`): любой `EncodingError::Unsupported`
   во время verify-pipeline → **compile error**, не silent skip. Format:
   ```
   error[E2401]: contract uses unsupported construct
     --> file.nv:12:5
   12 |     requires x.balance > 0
      |              ^^^^^^^^^ field access in contract not yet supported by SMT encoder
      = note: SMT encoder currently supports: int/bool/str literals, arithmetic, comparisons, &&/||/!, if-then-else, #pure function calls, forall/exists in bounded range
      = help: mark `fn` as `#unverified` to opt out of SMT verification (runtime check only), or extract logic into a `#pure` helper
   ```
3. **С `#unverified`** — silent skip OK, только warning categorisation:
   `warning[W2401]: fn 'transfer' opted out of SMT verification`.
4. **`#must_verify` (существующий)** + `Unsupported` → error даже строже,
   с pointer на конкретное место в формуле.
5. **Удалить все `eprintln!("verify: skipped ...")`** silent-fallback'и
   из `pipeline.rs` — должны стать структурированными `Diagnostic`-ами
   через `compiler-codegen/src/diagnostic.rs`.
6. **Сохранить TrivialBackend-категорию** — для случаев когда Z3
   недоступен (build без feature) и формула слишком сложная для
   trivial: должна выдавать `Unknown` с `UnknownReason::SmtBackendLimited`,
   но это **отдельная категория** от `Unsupported`. `Unknown` →
   compile error в `#must_verify` контексте, warning иначе (как сейчас).

**Файлы:**
- `compiler-codegen/src/lexer/token.rs` — добавить `KwUnverified` если
  attr-парсинг этого требует (`#unverified` парсится как attribute,
  не keyword — проверить).
- `compiler-codegen/src/ast/mod.rs` — `FnDecl.is_unverified: bool`.
- `compiler-codegen/src/parser/mod.rs` — парс `#unverified` атрибута
  по аналогии с `#verify`.
- `compiler-codegen/src/types/mod.rs` — gate: `#unverified` +
  `#verify`/`#must_verify` на одной fn → conflict error.
- `compiler-codegen/src/verify/pipeline.rs` — заменить все
  `eprintln!("verify: skipped")` на structured diagnostic emission;
  при `Unsupported` без `is_unverified` → push diagnostic, fail compile.
- `compiler-codegen/src/diagnostic.rs` — категория `E2401 UnsupportedInContract`.

**Acceptance:**
- `requires x.balance > 0` без `#unverified` → compile-error E2401.
- Тот же код с `#[unverified]` → warning W2401, компилируется,
  runtime check работает.
- `#must_verify` + `Unsupported` → error даже строже (не позволяется
  `#unverified` совместить).
- Regression: все существующие `nova_tests/contracts/*.nv` помечены либо
  `#verify`/`#must_verify` (если encoder может) либо `#unverified`
  (если не может). Скан: запустить `cargo run --release -- test
  nova_tests/contracts/` — все тесты явно классифицированы.

**Тесты:**
- `nova_tests/contracts/unsupported_field_access_fail.nv` (EXPECT_COMPILE_ERROR)
  с `requires x.balance > 0`.
- `nova_tests/contracts/unverified_attribute_positive.nv` — то же с
  `#unverified` → компилируется.
- `nova_tests/contracts/unverified_conflict_with_verify_fail.nv`
  (EXPECT_COMPILE_ERROR) — `#verify` + `#unverified`.
- `nova_tests/contracts/must_verify_unsupported_fail.nv`
  (EXPECT_COMPILE_ERROR) — `#must_verify` + field access.

**Трудоёмкость:** 1.5 дня.

## Ф.1.3 — `post(Action)(view)` Unknown — должен быть error по умолчанию

**Проблема.** [pipeline.rs:2083-2089](../../compiler-codegen/src/verify/pipeline.rs#L2083)
возвращает `VerifyResult::Unknown(UnknownReason::Unsupported {
detail: "post-axiom verification requires V2" })` для `post(...)`
axioms за пределами V2 scope (branching, nested handlers, FFI).
По умолчанию это silent-pass — компиляция продолжается, диагностики
нет.

**Что нужно:**

1. **Сделать `Unknown` от handler-verification fail = compile error**
   когда handler помечен `#verify` (не `#trusted`). Логика: «программист
   запросил verification → если verifier не может — это явный bug».
2. **С `#trusted`** — silent OK, как сейчас.
3. **Категория `UnknownReason::HandlerExecLimited`** (новая,
   отдельная от Unsupported) — для `post()` axioms с branching/loops/FFI.
   Diagnostic:
   ```
   error[E2402]: handler body verification incomplete
     --> file.nv:42:5
   42 |     with #verify Db = my_handler { ... }
      |                       ^^^^^^^^^^ axiom `post(SetBalance)(balance) == x` requires V2 symbolic exec
      = note: V1 supports linear handler bodies only (no if/match/loop/FFI)
      = note: detected branching at line 47 in my_handler.SetBalance
      = help: simplify handler body to single-expression form, or use #trusted to skip verification
   ```

**Файлы:**
- `compiler-codegen/src/verify/pipeline.rs` (handler exec call site).
- `compiler-codegen/src/diagnostic.rs` — E2402, `UnknownReason::HandlerExecLimited`.

**Acceptance:**
- `with #verify Db = h` где `h.SetBalance` имеет `if`-branch → E2402.
- `with #trusted Db = h` тот же handler → OK.

**Тесты:**
- `nova_tests/contracts/handler_verify_branching_fail.nv` (EXPECT_COMPILE_ERROR).
- `nova_tests/contracts/handler_trusted_branching_positive.nv`.

# Ф.3 — ВЫСОКИЙ: UX + AI-first diagnostics (~5-7 дней)

## Ф.3.1 — Counterexample minimisation (delta-debugging)

**Проблема.** Сейчас Z3 возвращает model со всеми bindings (для каждой
declared variable). Программисту/LLM невозможно понять какие bindings
**критичны** для counterexample, а какие случайные. Особенно плохо для
формул с >5 переменных.

**Что нужно:** delta-debugging на model.

**Алгоритм:**
1. Получили `Sat(model)` от Z3, model = `{x: 5, y: 0, z: -3, w: 100}`.
2. Для каждой bindings `(k, v)`:
   a. Build new assertion: `(assert (= k v))` (зафиксировать значение).
   b. Build complement: оригинальные assertions БЕЗ исходной violated
      clause + новое equality.
   c. `check_sat` → если **ещё SAT** значит binding критичный (counterexample
      работает с этим значением); если **UNSAT** — binding случаен,
      убираем из minimal model.
3. Повторить N итераций или пока model стабильна.
4. Вернуть **minimal model** в diagnostic.

**Файлы:**
- `compiler-codegen/src/verify/backend/z3.rs` — добавить
  `pub fn minimise_model(&mut self, model: Model, assertions: &[SmtTerm]) -> Model`.
- `compiler-codegen/src/verify/pipeline.rs` — вызвать после `Sat`.
- `compiler-codegen/src/diagnostic.rs` — формат counterexample.

**Acceptance:**
- Тест: `fn f(x int, y int, z int) requires x > 0 ensures x + y > 0`.
  Counterexample должен показать `x = 1, y = -2` (минимум), не
  `x = 1, y = -2, z = <random>`.
- Время minimisation < 100ms на counterexample c 10 vars.

**Тесты:**
- `nova_tests/contracts/counterexample_minimal.nv` (REQUIRES_SMT_BACKEND z3,
  EXPECT_COMPILE_ERROR с проверкой что diagnostic содержит только
  минимальный set vars). Test runner может grepать stderr на отсутствие
  unrelated vars.

**Трудоёмкость:** 1.5 дня.

## Ф.3.2 — Witness extraction для `exists`

**Проблема.** При неудаче `exists x in 0..n : P(x)` сейчас выводится
только «cannot prove existence». Полезного hint нет.
При **удаче** Z3 знает witness (значение x для которого P(x) истинно),
но мы его не показываем.

**Что нужно:**

1. **При SAT result** для `exists`-формулы — extract `x = v` из model,
   показать в info-diagnostic:
   ```
   info: existence witness found
     --> file.nv:12:5
   12 |     ensures exists i in 0..n : xs[i] == target
      |             ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
      = note: witness: i = 3, xs[3] = target
   ```
2. **При UNSAT** для positive `exists` (failed proof) — показать
   counterexample model где `forall i: ¬P(i)` (отрицание existence):
   «не существует, потому что: при xs = [1,2,3], target = 5».

**Файлы:**
- `compiler-codegen/src/verify/pipeline.rs` — extract witness после
  encoding existence.
- `compiler-codegen/src/diagnostic.rs` — `Diagnostic::info_witness(...)`.

**Acceptance:**
- Positive existence proof → info-line с witness в `nova check --verbose`.

## Инструкции для агента-исполнителя

1. **Работать в worktree** `d:\Sources\nova-lang-p33` (Plan 33 ветка).
2. **Перед началом каждой Ф.X:**
   - перечитать релевантные `compiler-codegen/src/verify/*.rs`.
   - `cd nova-cli && cargo build --release` для baseline.
   - `target/release/nova test nova_tests/contracts/` для baseline pass-count.
3. **После каждой Ф.X:**
   - regression suite PASS в обоих backend'ах
     (trivial — default, Z3 — `cargo build --release --features z3-backend`).
   - обновить `docs/simplifications.md`.
   - добавить запись в `docs/project-creation.txt`.
   - сделать commit (`git add` явно, не `-A`).
4. **Если фаза blocked** (зависит от не-implemented фичи или
   архитектуры) — задокументировать в plan-файле и переходить к
   следующей независимой фазе. Не молчать.
5. **Файл-структура:**
   - новые модули в `compiler-codegen/src/verify/`;
   - новые тесты в `nova_tests/contracts/`;
   - negative-тесты с маркером `// EXPECT_COMPILE_ERROR`;
   - Z3-only тесты с маркером `// REQUIRES_SMT_BACKEND z3`.
6. **Не использовать**:
   - `Co-Authored-By` или `Generated-with` trailers (правило feedback_commit_trailer);
   - `--jobs N` для `nova test` (правило feedback_no_jobs_flag);
   - изменения в `minicoro.h` / `bdwgc` (правило feedback_third_party_libs).
7. **Сложные решения** (например выбор формата diagnostic, имя
   атрибута) — задокументировать в `discussion-log` (nova-lang-private)
   ИЛИ в комментарии плана с rationale.
