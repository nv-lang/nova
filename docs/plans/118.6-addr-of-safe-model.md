# Plan 118.6 — Safe `&x` model + убрать `addr_of`/`addr_of_mut`

> **Родитель:** [Plan 118](118-typed-pointers-and-unsafe.md).
> **Владеет:** D216 §4 amend.

---

## Идея

Сейчас `&x` требует `unsafe { }` для **всех** типов. Это избыточно: если GC
+ escape analysis гарантируют валидность указателя — unsafe не нужен.

**Принцип:** `&x` safe ↔ компилятор гарантирует валидность. Unsafe только
когда программист берёт ответственность на себя (сырой стек-указатель без
промоута).

---

## Что такое heap-промоут

`x` и `pt` (value-record) **изначально всегда на стеке** — обычные локальные
переменные в стек-фрейме. Heap-промоут — это не дефолтное поведение, а
**оптимизация безопасности**, применяемая только когда нужна.

**Когда промоут НЕ нужен:** `&x` взят и использован внутри той же функции, не
уходит за её scope — `x` остаётся на стеке, указатель валиден пока функция
жива, лишней аллокации нет.

**Когда промоут нужен:** escape analysis видит что адрес `x` **уходит за scope**
(возвращается из функции, кладётся в поле другой записи, передаётся в замыкание
и т.д.) → codegen аллоцирует `x` через `nova_alloc` вместо стека → GC владеет
→ указатель валиден произвольно долго.

```
без escape:              с escape:
+----------+            +-----------+     +---------+
| стек     |            | стек      |     | heap    |
| x = 42   | <-- &x     | ptr_to_x  |---> | x = 42  | <-- &x
+----------+            +-----------+     +---------+
  работает пока          после return:      живёт пока
  функция жива           стек мёртв         GC не соберёт
```

## Модель (D216 §4 amend)

### `&x` — safe, без `unsafe { }`

Указатель **всегда валиден** — компилятор гарантирует через escape analysis
+ auto-promote:

- **value-record** (`type X value { ... }`) → escape analysis (Plan 127) →
  если указатель уходит за scope → auto-promote в heap → GC владеет →
  валиден всегда. **Уже работает.**
- **примитив** (`int`, `f64`, `bool`, `char`, `u8`, ...) → то же самое: если
  указатель уходит за scope → auto-promote в heap. Примитив ничем
  принципиально не отличается от value-record — оба на стеке, оба живут в
  пределах scope. Требует **расширения escape analysis на примитивы.**

```nova
ro x = 42                      // x на стеке
ro p = &x                      // p используется внутри fn → x остаётся на стеке
                               // p передаётся наружу → x промоутится в heap

type Point value { x int, y int }
ro pt = Point { x: 1, y: 2 }  // pt на стеке
ro p = &pt                     // если p уходит за scope → pt промоутится целиком
ro px = &pt.x                  // корень chain = pt → если уходит → pt в heap целиком
```

**`&x.field` — промоутится весь биндинг, не поле отдельно.**
Escape analysis отслеживает корень chain (`pt` в `pt.x`) → если корень на стеке
→ весь record/tuple размещается в heap → поле получает стабильный адрес.
Если record/tuple уже в heap (heap-record, не value-record) — адрес поля
стабилен сразу, без промоута.

### `unsafe { &x }` — сырой указатель, без промоута

Программист берёт ответственность: указатель на стек, без escape analysis,
без auto-promote. Может быть dangling если используется после выхода из scope.

```nova
unsafe {
    ro p = &x      // сырой стек-указатель, без промоута
    c_fn(p)        // передать int* в C-функцию — типичный FFI
}
```

Разрешён для **всех типов** (value-record и примитив) — программист знает
что делает.

### `mut` определяется биндингом автоматически

```nova
ro x = 42
ro p = &x          // *int   — ro биндинг → readonly pointer

mut y = 42
ro p = &y          // *mut int — mut биндинг → mutable pointer
```

## Убрать `addr_of` и `addr_of_mut`

`addr_of(x)` и `addr_of_mut(x)` — исторический workaround. Сейчас:
- `addr_of(x)` десугарится в `&x` (UnOp::AddrOf) — полный дубль
- `addr_of_mut(x)` проверяет mut-биндинг, но это уже делает binding-система
  автоматически — `mut x` → `*mut T`, избыточная проверка

**Замена:**
```nova
// было:
unsafe { addr_of(x) }       →  unsafe { &x }
unsafe { addr_of_mut(x) }   →  unsafe { &x }   // x должен быть mut

// safe:
addr_of(x)                  →  &x   // (после этого плана — safe)
```

Миграция через `E_ADDR_OF_REMOVED` с fix-it подсказкой.

## Фазы

### Ф.1 — Расширить escape analysis на примитивы

- `escape_analyze.rs`: добавить примитивные типы (`int`, `f64`, `bool`,
  `char`, все `u8`/`u16`/`i8`/`i16`/`i32`/`i64`/`u32`/`u64`/`f32`) в
  детектор escaped locals
- Добавить `promoted_primitive_locals` в codegen (аналог
  `promoted_value_record_locals`)
- Codegen: при `&x` на promoted примитиве → эмитить heap-allocated копию
  (`nova_alloc` + store value + return pointer), не `&stack_var`
- Тест: `ro p = &x; return_ptr(p)` → примитив промоутится, не dangling

### Ф.2 — `&x` safe (убрать `E_UNSAFE_REQUIRED` для `AddrOf`)

- `types/mod.rs` `check_unsafe_context_in_module`: убрать `UnOp::AddrOf` из
  unsafe-walker (оставить только `UnOp::Deref` и pointer arithmetic)
- `types/mod.rs`: убрать `addr_of`/`addr_of_mut` из unsafe Call-arm
- Тест pos: `ro p = &x` без `unsafe { }` компилируется
- Тест neg: `ro p = *ptr` вне unsafe → `E_UNSAFE_REQUIRED` (deref остаётся unsafe)

### Ф.3 — Убрать `addr_of` и `addr_of_mut`

- `const_fn_eval.rs`: убрать desugar-ветку для `addr_of`/`addr_of_mut`
- `types/mod.rs`: убрать из known-intrinsics списка (строки 11227–11228),
  убрать `E_ADDR_OF_MUT_REQUIRES_MUT_BINDING` check, убрать из unsafe Call-arm
- Добавить `E_ADDR_OF_REMOVED` с fix-it: `addr_of(x)` → `&x`,
  `addr_of_mut(x)` → `&x` (mut биндинг)
- Миграция: sweep по `nova_tests/` и `std/` — заменить все вхождения
- Тест neg: `addr_of(x)` → `E_ADDR_OF_REMOVED`

### Ф.4 — D216 §4 amend + spec

- `spec/decisions/02-types.md` D216 §4: обновить семантику `&x`:
  - safe без `unsafe { }` — escape analysis гарантирует валидность
  - `unsafe { &x }` — сырой стек-указатель, без промоута
  - примитив и value-record — одинаковая модель (escape analysis для обоих)
  - `addr_of`/`addr_of_mut` — retired, `E_ADDR_OF_REMOVED`
- Обновить таблицу error-кодов D216 §8

### Ф.5 — Тесты + закрытие

- POS: `&x` на int без unsafe → компилируется и корректен
- POS: `&x` на value-record без unsafe → компилируется (уже работало)

- **A1.** `&x` без `unsafe { }` компилируется для всех типов (value-record + примитив)
