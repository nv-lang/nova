## 1. Контекст

### 1.1 Что landed в Plan 127 V1

Plan 127 V1 (merge 2026-06-05) ввёл `AllocKind` tri-state
`{ Heap, Value, ValueHeapPromoted }` для value-records. Когда
`escape_analyze` walker определяет что `&v` escape'ит (return, closure
capture, heap-field store, global save, fn-arg sink), binding `v`
auto-promote'ится со stack-allocated `NovaValue_X` на heap-allocated
`Nova_X*` через `emit_record_lit` path:

```c
// AllocKind::Value (stack):
NovaValue_Point v;
v.x = 1.0;
v.y = 2.0;

// AllocKind::ValueHeapPromoted (heap):
Nova_Point* v = (Nova_Point*) nova_alloc(sizeof(Nova_Point));
v->x = 1.0;
v->y = 2.0;
```

### 1.2 Где остался bug

**Field access codegen** для `ValueHeapPromoted` binding всё ещё
повторяет stack-mode emit: на месте `ptr->field` генерируется
`ptr_field` (underscore concatenation) — pattern зарезервирован для
`NovaValue_X` value-mode где struct поля раскладываются flat в C-emit
shape.

**Симптом** (из Plan 127 Ф.5 honest findings):

```nova
// nova_tests/plan127/t3_field_access_after_promote_ok.nv
fn make_point() -> *Point {
    ro p = Point { x: 10.0, y: 20.0 }
    unsafe { &p }                       // auto-promote → Nova_Point*
}

test "field access through promoted heap pointer" {
    ro ptr = make_point()
    assert(unsafe { ptr.x } == 10.0)    // emit'ит `ptr_x` вместо `ptr->x`
}
```

C compile failure:
```
error: no member named 'ptr_x' in 'struct Nova_Point'
```

### 1.3 Affected fixtures

| Fixture | Symptom |
|---|---|
| `t3_field_access_after_promote_ok.nv` | `ptr.x` / `ptr.y` access после auto-promote |
| `t8_multiple_field_types_promote_ok.nv` | `p.f` / `p.i` / `p.b` mixed-type fields after promote |
| `t9_repeat_addr_of_same_local_ok.nv` | `ptr.x` / `ptr.y` через double-`&v` (idempotent promote) |

Каждый из этих 3 fixtures **должен PASS** после landing'а — fix
ограничивается codegen `emit_c.rs::emit_member_access`.

## 2. Design

### 2.1 Root cause

`emit_member_access` (или эквивалентный path в `emit_c.rs`)
определяет shape доступа к полю по **type kind** (record vs value-record
vs primitive) **без учёта** `AllocKind` per-binding:

- `AllocKind::Heap` (`Nova_X*`) → emit `obj->field` (pointer deref).
- `AllocKind::Value` (`NovaValue_X`) → emit `obj.field` или
  `obj_field` (depending on lowered shape).
- `AllocKind::ValueHeapPromoted` (`Nova_X*` под капотом, но **тип
  binding'а** = `*X` user-visible) → **должен** emit `obj->field`.

Текущий код trips на shared codepath: всё что не `AllocKind::Heap` →
treats as stack value-mode → `_field` concat.

### 2.2 Fix shape

Identify Member-access path в `emit_c.rs`. Resolve binding
`AllocKind` по `obj`-expression's resolved-binding lookup (binding
registry already tracks `AllocKind` per Plan 127 Ф.2). Branch:

```rust
match resolved_alloc_kind {
    AllocKind::Heap | AllocKind::ValueHeapPromoted => {
        // Pointer dereference path.
        emit!("{}->{}", obj_c, field_name);
    }
    AllocKind::Value => {
        // Stack value-mode emit (existing path).
        emit_value_field_access(obj_c, field_name);
    }
}
```

**Key invariant:** `ValueHeapPromoted` binding в runtime = `Nova_X*`
identical к `Heap` allocation; единственная разница — origin
(user wrote `value` modifier на typedef, не `heap`). User-visible
binding type — `*X` per Plan 118 binding-type rule. Codegen
shape для field access **identical** между двумя modes.

### 2.3 Codegen invariants preserved

- `prepare_method_recv` (extended Plan 127 Ф.3) уже handles
  `ValueHeapPromoted` mode — fix не трогает method-receiver path.
- `emit_record_lit` (Ф.3) уже handles heap-init pattern — fix не трогает
  allocation path.
- Symmetric: `t1_local_only_no_escape_ok.nv` (stack-mode `NovaValue_X*`)
  должен оставаться PASS — fix не должен break value-mode emit.
- Cross-ref D228 §«heap-promoted binding access shape» — добавить
  явное правило в Ф.2 spec touch-up (если необходимо после verification).

### 2.4 Out of scope

- **Precise escape mode** (Plan 127 V2 `[M-127-precise-escape]`) — Plan
  127.1 не меняет escape-detection logic, только codegen lowering.
- **Array-element promote** (`[M-127-array-element-promote]`) — V3
  followup, separate concern.
- **Mut-method receiver через promoted binding** — Plan 128 + Plan
  128.1 cover это в `prepare_method_recv`; Plan 127.1 ограничен read
  field-access path.
