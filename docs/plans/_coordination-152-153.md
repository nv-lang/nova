# Координация Plan 152 ↔ Plan 153 (handoff)

> Точки сходимости/координации строкового слоя (152) и Vec/Slice-слоя (153).
> Источник — handoff от агента Plan 153 (2026-06-13). Агент 152 обязан учесть
> пункт 1 при дизайне линз (152.1); остальное — гигиена/координация.

---

## 1. 🔴 str.as_bytes() ↔ `Slice[T]` (153.4) — главный пункт

153 вводит zero-copy **`Slice[T]` / `SliceMut[T]`** — честный read-only срез-вью над
буфером `Vec`. Это **тот же концепт**, что байтовая линза `str`.

- **Сейчас (152/main):** `str.@as_bytes() -> ro []u8` = owned `Vec[u8]` через
  `Vec[u8].from_raw_parts(ptr, len, len)` с `cap == len` — **«фейково-readonly»**
  (это полноценный `Vec`, просто помеченный `ro` на binding'е).
- **Цель:** `str.@as_bytes() -> Slice[u8]` (честный вью), чтобы НЕ плодить два разных
  «read-only slice»-типа в языке.
- **Директива:** агент 152 **НЕ хардкодит** семантику `ro []u8` так, чтобы она
  конфликтовала с будущим `Slice[T]`. Тип байт-линзы (`as_bytes` возврат) —
  **финализируется при сходимости с 153.4**. До тех пор `ro []u8` остаётся (не ломать
  существующее), но D250 (152.1) фиксирует open-point «byte-lens type → Slice[u8] при
  готовности 153.4».
- **Влияние на 152.0:** нет (152.0 — рефактор; `as_bytes` остаётся `ro []u8`). Влияет
  на **152.1 D250** (дизайн линз) — там пометить.

## 2. Accessor-гигиена — D117 AMEND владеет **Plan 153**

- D117 AMEND (size-accessor field-read внутри методов) — **собственник Plan 153**. Агент
  152 **НЕ пишет** D117 AMEND-спеку независимо; ссылается на 153.
- **Применение:** внутри str-методов читать поле **`@len` напрямую** (field), не getter
  `@len()`/`@byte_len()`. Внутренний field-read size-аккумулятора — **разрешён**.
- **Противоречие в string.nv** (резолвнуть в 152.1 Ф.5.1): комментарий line 34-37
  («D117 НЕ срабатывает на `@len` через SelfAccess») ↔ line 379-381 («`@len()` spelled,
  bare `@len` trips D117»). **Истина:** internal SelfAccess field-read `@len` разрешён →
  предпочитать `@len`.
- **Влияние на 152.0:** в Ф.2 (pure move) тела **НЕ менять** (golden-эквивалентность).
  Замену `@len()`→`@len` делать в 152.1 Ф.5.1, не в 152.0.

## 3. Минимализм API (главный урок 153)

Предпочитать один общий примитив набору частных (в 153 выпали
`reserve_exact`/`shrink_to_fit`/`shrink_to` = `cap(...)`). Прогнать str-API той же
линзой (в 152.2): нет ли дублей-удобств, выразимых через базовый метод.
**Намеренные пары — оставить:** `as_*` vs `to_*` (вью vs owned), `@index`-panic vs
`@get`-Option (D238). **Влияние:** ревью-чеклист 152.2.

## 4. Общий codegen-фикс `[M-138.2-vec-self-return]` (fluent `-> @`)

153 чинит `@`-chaining (мис-типизация receiver в `void*` для value-record/generic-метода).
От этого выигрывает **StringBuilder.@append**-цепочка в 152 (152.0 Ф.4). **НЕ дублировать
фикс** — координировать, кто чинит (вероятно 153). Если 152.0 Ф.4 упрётся в это —
свериться со статусом 153, не патчить codegen параллельно.

## 5. Скалярные `int.min`/`int.max` отсутствуют (`[M-153-scalar-min-max]`)

Math в Nova — method-style (`a.max(b)`, как `a.sin()`). `str.@compare` сейчас делает
ручной min (`if an < bn {…}`). Когда 153 Ф.0 добавит примитив — упростить; **пока —
explicit if/else**, НЕ выдумывать свободный `max(a,b)`.
