# Plan 100.6: cross-module + visibility + mangling — consume через границы пакетов

> **Создан 2026-05-23 как sub-plan [Plan 100 umbrella](100-linear-must-consume.md).**
> **Статус:** 📋 proposed. **GATED на [100.1](100.1-core-must-consume.md)**.
> **Оценка:** ~3 dev-day.
> **D-блок:** **D153**.

---

## Цель

Закрыть критическую production-дыру: **consume-типы должны корректно
работать через границы модулей и пакетов**. Без этого Plan 03 package
ecosystem становится бесполезным для resource-management — каждый
пакет был бы изолированным mini-Plan-100, без cross-package guarantees.

Конкретные дыры в Plan 100.1 (bootstrap):
1. Module A экспортирует `consume`-тип; Module B импортирует — должны
   ли visibility-rules (Plan 35 R26) пускать consume-маркер?
2. Mangling схема (Plan 81 D134 symbol mangling v0) должна включать
   consume в decorated names — иначе двa modul'я с `Transaction` (один
   consume, другой нет) дадут одинаковый mangled symbol → silent ABI
   mismatch.
3. Package boundary (Plan 03 manifest) — может ли package1 объявить
   `consume`-тип, реэкспортировать через package2, который его не знает?
4. Plan 84 relative imports — `./` / `../` consume-types работают?
5. Plan 42 folder-modules — `internal/` consume-типы; module-level
   `_module.nv` consume-конвенции.

## Сравнение

| Capability | Rust | Kotlin/Java | Go | TS | Nova target |
|---|---|---|---|---|---|
| Pub visibility consume-маркера | ✅ pub Drop visible | ⚠️ через AutoCloseable interface | ⚠️ exported method | ⚠️ via TypeScript types | ✅ **D153 propagation** |
| ABI mangling включает ownership-info | ✅ через type | ⚠️ via signature | ❌ | n/a | ✅ **mangling с consume-bit** |
| Cross-package consume-types в registry | ✅ Cargo + Rust types | ⚠️ Maven coordinates | ⚠️ go modules | ⚠️ npm types | ✅ **Plan 03 manifest declares consume** |
| Re-export consume через package | ✅ через `pub use` | ⚠️ implementation detail | ⚠️ alias | ⚠️ re-export | ✅ **export import** (Plan 42.09) preserves consume |
| Visibility violation detection | ✅ через pub/priv | ✅ access modifiers | ✅ exported/unexported | ⚠️ TypeScript only | ✅ **Plan 35 R26 + D153** |

Nova **matches Rust** на всех осях; **превосходит** на одной —
mangling включает consume-bit, ловит silent ABI mismatch (Rust имеет
type-id but consume через `Drop` impl runtime-checked).

## Дизайн-решения

### D1 — consume-маркер part of type signature, visible через export

```nova
// package A, module a/types.nv
export type Transaction consume {
    id int,
}
```

```nova
// package B, module b/main.nv
import a.types.Transaction

fn main() {
    consume tx = Transaction { id: 1 }
    tx.commit()                                // ✅ consume-marker visible
}
```

`consume` — **part of exported type signature**. Plan 35 visibility
(R26) пропускает marker как обычный атрибут типа. Если в A type не
помечен `consume`, в B нельзя использовать `consume tx = ...` binding-
form — error.

### D2 — Mangling включает consume-bit (extends Plan 81 D134)

Plan 81 D134 определил symbol-mangling v0 для cross-file resolution.
Plan 100.6 расширяет:

```
nova_fn_<pkg>_<mod>_<name>_<consume-bit>_<param-types>_<return-type>
```

`<consume-bit>` = `c` если consume-маркер на type-decl, `_` иначе.

Это ловит ABI mismatch:

```nova
// package A v1.0
export type Resource consume { ... }
export fn Resource consume @close() -> ()
// → nova_fn_a_resource_close_c_...

// package A v2.0 (breaking change!)
export type Resource { ... }                  // ⚠️ убрали consume
export fn Resource @close() -> ()
// → nova_fn_a_resource_close__...           // другой mangled symbol

// B imports a@v1; linker ловит mismatch на load.
```

Plan 03 manifest могут lock'ать consume-marker version через semver
(major-bump при изменении).

### D3 — Re-export через `export import` (Plan 42.09)

```nova
// package B re-exports A.Transaction
export import a.types.{Transaction}
```

Re-export **preserves** consume-marker. Plan 42.09 уже работает; Plan
100.6 verifies через тесты.

### D4 — Folder-modules (Plan 42) integration

```
my_pkg/
  resources/                                  // folder-module
    _module.nv                                // module-level config
    file.nv                                   // type File consume {...}
    socket.nv                                 // type Socket consume {...}
```

`_module.nv` может содержать module-level `consume`-conventions (e.g.,
default-consume-bound для generics — нет, это per-type). Folder-module
consume types работают как regular cross-module.

### D5 — Relative imports (Plan 84) + consume

```nova
import ./types.{Transaction}                  // relative
consume tx = Transaction { ... }              // consume-marker propagates
```

No special rules — relative imports treat consume identically to
absolute.

### D6 — Package version contracts (Plan 03)

`nova.toml` могут declare consume contracts:

```toml
[package]
name = "my_lib"
version = "1.0.0"

[exports.consume_types]
Transaction = "1.0"                           // consume contract v1
File = "1.0"
```

Cross-version compat:
- v1.0 → v1.x: consume-status unchanged.
- v1.x → v2.0: consume-status may change (major-bump required).

`nova audit` (Plan 03.4) verifies consume-contracts.

### D7 — Private consume-types — internal use only

```nova
type InternalCache consume { ... }            // no `export`
// usable только в этом package; cross-package — invisible
```

Plan 35 visibility R26: private types не leak. consume на private —
no effect cross-package.

### D8 — Cross-module diagnostic clarity

При cross-package consume-violation diagnostic должен указать source
package:

```
error: consume value `tx` (type a::Transaction) not consumed
  note: type defined in package 'a' v1.0 at a/types.nv:5
  note: consume via .commit() or .rollback() (declared in 'a')
```

Includes package origin, version, hint про consume-методы.

## Границы

- **Dynamic library loading** (dlopen-style) — out of scope; bootstrap
  Nova не поддерживает dynamic loading.
- **ABI freeze** для stable API — Plan 03 ecosystem-wide concern;
  Plan 100.6 enforce'ит при upgrade'ах.
- **Cross-version migration assistance tool** — future plan (`nova
  consume-upgrade` command).

## Risks

1. **Mangling-bit collision** с existing schemes — Plan 81 D134
   определил v0; consume-bit добавится как extra-field. **Митигация:**
   semver-bump на mangling (v0 → v1) если breaks.
2. **Plan 03 manifest schema growth** — `[exports.consume_types]` —
   ещё одна секция. **Митигация:** optional, default-derived из type-decls.
3. **Cross-package diagnostic cost** — package origin lookup может
   быть дорогим. **Митигация:** cache в LinearityRegistry.
