## Зачем

После Plan 108.1/108.2/108.3 (default-ro для params/locals/loops/patterns),
**protocol method declarations остались единственной позицией с implicit
default-mut**. Это нарушает consistency contract:

**Текущее состояние** ([std/prelude/collections.nv:107-114](../../std/prelude/collections.nv#L107-L114)):
```nova
export type Iterable[T] protocol {
    next() -> Option[T]               // ← НЕТ @, нет mut, parser не поддерживает
}

// Implementers вводят @ и mut в free-form, type-checker не enforce'ит match:
fn Counter mut @next() -> Option[int] { @cur += 1; ... }       // mut receiver
fn RangeIter mut @next() -> Option[int] { ... }                // mut receiver
fn HashMapIter[K,V] mut @next() -> Option[(K, V)] { ... }      // mut receiver
```

Spec comment в `collections.nv:107`: «**bootstrap parser НЕ поддерживает
`@`-префикс и `mut`-модификатор перед именем метода в protocol-declaration**.
Receiver mutability подразумевается use-case'ом и объявляется на стороне implementer'а».

**3 проблемы текущего «honour system»:**

1. **Type-checker не enforce'ит mut consistency между protocol и impl.**
   Protocol объявляет `next() -> Option[T]` (без `@`, без mut). Implementer
   пишет `fn Counter mut @next()` (mut receiver). **Mismatch silently accepted**.

2. **`#impl(Iterable)` annotation (Plan 91.9 D186) impotent.** Если user
   пишет `#impl(Iterable) type Counter { ... }` + `fn Counter @next() ->
   Option[int]` (ro receiver), iterator не работает, но type-checker accepts.
   Bug surfaces only at runtime via silent infinite loop / non-progress.

3. **Inconsistency с остальными default-ro positions + нет `@` в protocol.**
   После Plan 108.x params/locals/loops/patterns — всё default-ro. И в impl
   `fn T @method()` — `@` обязателен. А в protocol declaration — ни `@`,
   ни modifier. Это двойная inconsistency.

   Кроме того: **effects** (`effect { job(str) -> int }`) — набор функций
   (нет receiver), нет `@`. **Protocols** — набор методов (есть receiver),
   должны иметь `@`. Без `@` protocol и effect выглядят одинаково — семантика
   теряется.

**Plan 108.4 closes both gaps:**
- `@` обязателен перед именем метода в protocol declaration.
- `mut`/`ro`/`consume` prefix перед `@` для receiver mutability. Default = `ro`.
- Type-checker: enforce impl receiver matches protocol declaration.
- Stdlib migration: все existing protocol declarations получают `@` + explicit mut.

## Дизайн

### Grammar — protocol method `@` + receiver prefix

```nova
type Iterable[T] protocol {
    mut @next() -> Option[T]                      // mut receiver
    @len() -> int                                  // ro receiver (default, no prefix)
    @is_empty() -> bool                            // ro
}

type Closeable protocol {
    consume @close() -> Result[(), Error]         // consume receiver
}

type Comparable[T] protocol {
    @compare(other ro T) -> int                   // ro receiver, ro param
}

type Writer protocol {
    mut @write(data ro []u8) -> Result[int, IoError]
    mut @flush() -> Result[(), IoError]
    consume @close() -> Result[(), IoError]       // consume — закрывает permanently
}

type Hashable protocol {
    @hash() -> u64                                 // ro — детерминированный hash
}
```

**Почему `@` обязателен в protocol:**
- Mirrors impl syntax: `fn T @method()` ↔ `@method()` в protocol declaration.
- Отличает protocol (методы с receiver) от effect (функции без receiver):
  `effect { job(str) -> int }` — нет `@`; `protocol { @job(str) -> int }` — есть.
- Static methods в protocol — `.method()` (leading dot, без `@`), как и раньше.

**Правила:**

1. **`@` обязателен** перед именем instance-метода. Отсутствие → `E_PROTO_METHOD_NEEDS_AT`.
2. **Default = `ro`** (consistent с param/local/loop-var default per Plan 108.x).
   Prefix перед `@` определяет receiver mutability для всех implementers.
3. **`ro` redundant** (default), но allowed для documentation clarity:
   `ro @len() -> int` ≡ `@len() -> int`.
4. **`mut`/`consume` ≠ `ro`** — implementer должен match.
5. **Modifier-conflicts** (parse error):
   - `mut ro @foo()` — `E_PROTO_METHOD_MOD_CONFLICT`
   - `mut consume @foo()` — `E_PROTO_METHOD_MOD_CONFLICT`
   - `consume ro @foo()` — `E_PROTO_METHOD_MOD_CONFLICT`

### Implementer enforcement

```nova
type Counter { mut cur int, end int }

// ✓ MATCH: protocol `mut @next()` → impl mut
fn Counter mut @next() -> Option[int] {
    if @cur >= @end { None } else { ro v = @cur; @cur += 1; Some(v) }
}

// ✗ MISMATCH: protocol mut → impl ro
fn Counter @next() -> Option[int] { ... }
// → E_PROTO_IMPL_RO_FOR_MUT:
//   "Protocol Iterable declares `mut @next()`; implementation must
//    also use mut receiver. Change to `fn Counter mut @next()`."

// ✓ MATCH: protocol `@len()` (ro default) → impl ro
fn Counter @len() -> int => @end - @cur

// ✗ MISMATCH: protocol ro → impl mut
fn Counter mut @len() -> int { ... }
// → E_PROTO_IMPL_MUT_FOR_RO:
//   "Protocol Iterable declares `@len()` (ro); implementation cannot
//    use mut. Change to `fn Counter @len()`."

// ✓ MATCH: protocol `consume @close()` → impl consume
fn FileHandle consume @close() -> Result[(), IoError] { ... }

// ✗ MISMATCH: protocol consume → impl mut
fn FileHandle mut @close() -> Result[(), IoError] { ... }
// → E_PROTO_IMPL_MUT_FOR_CONSUME
```

### Default-ro «structural conformance»

После Plan 108.4 protocol-method `@method()` **без prefix** = ro receiver.
Implementer `fn T @method()` **без prefix** = ro receiver. **Match implicit'ом — OK**:

```nova
type Hashable protocol {
    @hash() -> u64                                 // ro implicit (no prefix)
}

type MyId { id u64 }

fn MyId @hash() -> u64 => @id                     // ro implicit — matches
```

Для backwards-compat (pre-108.x когда default был mut implicit'ом) — Plan
108.4 включает stdlib migration sweep (Ф.3) — все existing protocol decls
получают explicit `mut`/`ro` где applicable.

### Cross-modifier с `#impl(P)` (Plan 91.9 D186)

`#impl(P)` annotation forces explicit protocol-conformance check. Plan 108.4
делает этот check **stricter** — теперь enforce'ит и receiver mutability:

```nova
#impl(Iterable[int])
type Counter { mut cur int, end int }

fn Counter @next() -> Option[int] { ... }         // ← E_PROTO_IMPL_RO_FOR_MUT
                                                   // (Iterable declares `mut @next()`)
```

Без `#impl(P)` — structural duck-typing (D58) — Plan 108.4 enforce'ит при
**use-site** (for-in loop, generic bound `[T Iterable[U]]`).

### Сравнение с mainstream

| Язык | Receiver mutability в protocol/trait/interface |
|---|---|
| Rust | `trait Iterator { fn next(&mut self) -> Option<Self::Item> }` — explicit `&mut self` |
| Go | `interface { Next() *T }` — implicit pointer mut (no shared ref); receiver type decides |
| Swift | `protocol IteratorProtocol { mutating func next() -> Element? }` — `mutating` keyword |
| Kotlin | `interface Iterator<T> { fun next(): T }` — Kotlin permits any side effect (no mut tracking) |
| Java | `interface Iterator<T> { T next(); }` — same as Kotlin, no static mutability |
| Haskell | type classes — pure (immutable); state-mut via monads |
| **Nova V2 (Plan 108.4)** | `protocol { mut @next() -> Option[T] }` — `@` обязателен + `mut`/`ro`/`consume`, **enforced** | **best-in-class** (Swift-tier + consume + явный receiver) |

Nova matches Swift `mutating` (explicit + enforced) и расширяет на `consume`
(linear-types) — unique combination.

## Грамматика

Текущая (фрагмент `compiler-codegen/src/parser/mod.rs:2244+` `parse_effect_methods`):

```ebnf
proto_method_decl ::= IDENT "(" param_list? ")" effect_list? ("->" type)?
effect_method_decl ::= IDENT "(" param_list? ")" effect_list? ("->" type)?
```

Новая (после Plan 108.4):

```ebnf
// Protocol — @ обязателен для instance; . для static (без изменений)
proto_method_decl  ::= ("mut" | "ro" | "consume")? "@" IDENT "(" param_list? ")"
                       effect_list? ("->" type)?
proto_static_decl  ::= "." IDENT "(" param_list? ")" effect_list? ("->" type)?

// Effect — без изменений, нет @ (функции без receiver)
effect_method_decl ::= IDENT "(" param_list? ")" effect_list? ("->" type)?
```

**Tokenizer:** без изменений (`KW_MUT`/`KW_RO`/`KW_CONSUME`/`AT`/`DOT` уже tokens).

**Parser:** разделить `parse_effect_methods` на два:
- `parse_effect_methods` — оставить как есть (для `effect { }` блоков).
- `parse_protocol_methods` — новый, требует `@` для instance-методов;
  `E_PROTO_METHOD_NEEDS_AT` если встретили `IDENT "("` без `@`/`.`.

**AST:** `EffectMethod` получает field `receiver_mut: ReceiverMutability`
(enum `Ro | Mut | Consume`); default = `Ro`. Поле `is_static: bool` остаётся
(`.method()` = static).

**Errors:**
- `E_PROTO_METHOD_NEEDS_AT` — instance-метод в protocol без `@`.
- `E_PROTO_METHOD_MOD_CONFLICT` — multiple modifiers (`mut ro @foo()`).
- `E_PROTO_IMPL_RO_FOR_MUT` — protocol `mut @m()`, impl ro.
- `E_PROTO_IMPL_MUT_FOR_RO` — protocol `@m()` (ro), impl mut.
- `E_PROTO_IMPL_MUT_FOR_CONSUME` — protocol `consume @m()`, impl mut/ro.
- `E_PROTO_IMPL_CONSUME_FOR_MUT` — protocol `mut @m()`, impl consume.

## D-block changes

### D209 (NEW) — Protocol method `@` syntax + receiver mutability

**Локация:** `spec/decisions/04-effects.md` (рядом с protocol-related blocks).

**Что.** Protocol instance-methods требуют `@` перед именем. Receiver
mutability prefix (`mut`/`ro`/`consume`) опционален перед `@`. Default = `ro`.
Implementers обязаны match. Effects (`effect { }`) без изменений — нет `@`.

**Визуальная разница protocols vs effects:**
```nova
// Effect — набор функций (нет receiver):
type Logger effect {
    log(msg str) -> ()
}

// Protocol — набор методов (есть receiver = @):
type Closeable protocol {
    consume @close() -> Result[(), Error]
}
```

**Grammar:**
```ebnf
proto_method_decl ::= ("mut" | "ro" | "consume")? "@" IDENT "(" param_list? ")"
                      effect_list? ("->" type)?
proto_static_decl ::= "." IDENT "(" param_list? ")" effect_list? ("->" type)?
```

**Default receiver:** `ro` (без prefix'а).

**Enforcement:**
- `#impl(P)` annotation (D186) — declares-conformance: type-checker matches
  every method's receiver_mut.
- Structural conformance (D58) — at use-site (for-in / generic bound) —
  same check.

**Errors:**
- `E_PROTO_METHOD_NEEDS_AT` — parse-time (instance-method без `@`).
- `E_PROTO_METHOD_MOD_CONFLICT` — parse-time (multiple modifiers).
- `E_PROTO_IMPL_RO_FOR_MUT` — protocol `mut @m()`, impl ro.
- `E_PROTO_IMPL_MUT_FOR_RO` — protocol `@m()` (ro), impl mut.
- `E_PROTO_IMPL_MUT_FOR_CONSUME` / `E_PROTO_IMPL_CONSUME_FOR_MUT` —
  consume mismatches.

**Migration:** stdlib protocols (Iterable, Comparable, Hashable, Closeable,
etc) — Ф.3 sweep добавляет explicit prefixes. Existing implementers
match-able since pre-108.4 «honour system» convention уже использовал
correct mut на side implementer'а (per `collections.nv:107-114` note).

**Cross-ref:**
- D58 amend (Iterable signature stay `mut next()`).
- D72 amend (`T Iterable[X]` bound — mut consistency).
- D186 amend (`#impl(P)` annotation checks receiver_mut).
- Plan 108.1/108.2/108.3 — consistency story (default-ro everywhere).

### D58 amend — Iterable structural protocol signature

**Локация:** `spec/decisions/03-syntax.md`.

Old: «`Iterable[T]` = duck-typing на `next() -> Option[T]`».
New: «`Iterable[T]` = duck-typing на `mut next() -> Option[T]`». Use-site
checker enforce'ит mut consistency.

### D72 amend — `T Iter` bound usage

**Локация:** `spec/decisions/03-syntax.md` (где определён D72).

Add note: «когда type участвует как bound `[T Iterable[U]]`, type-checker
проверяет receiver_mut consistency для всех protocol methods. Mismatch →
`E_PROTO_IMPL_*` errors».

### D186 amend — `#impl(P)` annotation enforcement

**Локация:** `spec/decisions/02-types.md` (где определён D186).

Add note: «`#impl(P)` annotation теперь checks receiver_mut в дополнение
к method signature. Несовпадение receiver mut между protocol и impl →
`E_PROTO_IMPL_*` errors».

- **T1.6** `protocol { .from(x int) -> Self }` — static, `is_static = true`,
  без `receiver_mut`.

### T2 — Parser negative

- **NEG-T2.1** `protocol { mut ro @foo() }` → `E_PROTO_METHOD_MOD_CONFLICT`.
- **NEG-T2.2** `protocol { mut consume @bar() }` → same.
- **NEG-T2.3** `protocol { ro consume @baz() }` → same.
- **NEG-T2.4** `protocol { mut mut @qux() }` → same (duplicate).
- **NEG-T2.5** `protocol { next() -> int }` (no `@`) → `E_PROTO_METHOD_NEEDS_AT`
  hint: «add `@` before method name: `@next()`».

### T3 — Type-checker positive (impl match)

- **T3.1** Protocol `mut @next()` + impl `fn T mut @next()` → OK.
- **T3.2** Protocol `ro @hash()` + impl `fn T @hash()` → OK (both ro).
- **T3.3** Protocol `consume @close()` + impl `fn T consume @close()` → OK.
- **T3.4** Default `@len()` + impl `fn T @len()` (no prefix → ro) → OK.

### T4 — Type-checker negative (mismatches)

- **NEG-T4.1** Protocol `mut @next()` + impl `fn T @next()` (ro) →
  `E_PROTO_IMPL_RO_FOR_MUT` с pointer на оба sites.
- **NEG-T4.2** Protocol `@hash()` (ro) + impl `fn T mut @hash()` →
  `E_PROTO_IMPL_MUT_FOR_RO`.
- **NEG-T4.3** Protocol `consume @close()` + impl `fn T mut @close()` →
  `E_PROTO_IMPL_MUT_FOR_CONSUME`.
- **NEG-T4.4** Protocol `mut @foo()` + impl `fn T consume @foo()` →
  `E_PROTO_IMPL_CONSUME_FOR_MUT`.

### T5 — `#impl(P)` annotation (D186)

- **T5.1** `#impl(Iterable[int]) type Counter { ... }` + `fn Counter mut
  @next()` → OK (annotation + mut match).
- **NEG-T5.2** `#impl(Iterable[int]) type Counter { ... }` + `fn Counter
  @next()` (ro) → `E_PROTO_IMPL_RO_FOR_MUT` (annotation triggers explicit
  check).

### T6 — Structural conformance (D58 for-in)

- **T6.1** `for x in counter` where Counter has `mut @next()` — OK.
- **NEG-T6.2** `for x in bad_counter` where BadCounter has `@next()` (ro) →
  use-site error «Iterable requires `mut @next()`; BadCounter.next is ro».

### T7 — Stdlib migration

- **T7.1** `Iterable[T] { next() }` → `{ mut @next() -> Option[T] }`;
  все existing implementers (`Counter`, `RangeIter`, `HashMapIter`, etc) — pass.
- **T7.2** `Comparable[T] { compare(other) }` → `{ @compare(other ro T) -> int }`.
- **T7.3** `Hashable { hash() }` → `{ @hash() -> u64 }`.
- **T7.4** `Default { default() }` → `{ .default() -> Self }` (static, точка).

### T8 — Property tests

- **T8.1** `prop_proto_decl_modifiers_parse` — random valid combinations
  parse OK.
- **T8.2** `prop_impl_mut_match_required` — random protocol+impl pairs,
  mismatch always caught.

- **R2** Cross-platform Windows + Linux × clang + MSVC.
- **R3** `for x in xs` patterns везде в существующем коде — pass
  (Iterable.next remains compatible).
