## 1. Problem — in two words: «вырезать доказанные overflow-чеки»

Plan 33.8 Ф.1.2 made every **unbounded `int`** (i64) `+`/`-`/`*` emit a *checked* form in
generated C — `nova_int_checked_{add,sub,mul}` → `__builtin_*_overflow` → `nv_panic("integer
overflow: …")`. This is a **soundness guard** (spec 04-effects.md / D13: `int` overflow = panic,
never silent wrap/UB), and it is *always on* — in debug **and** release.

Most of those checks are provably unnecessary: `for i in 0..10 { … i + 1 … }` can never
overflow. Paying a branch on every arithmetic op in hot loops is the cost Plan 33.8 Ф.1.3
flagged for later removal. This plan removes it **when, and only when, an SMT proof shows the
op cannot overflow** — exactly mirroring Plan 140.2's elision of provably-in-range Vec bounds
checks.

**Out of scope (nothing to elide):**
- *Sized* ints (`u8`/`i32`/…) — **wrap** semantics (Plan 33.7), no check emitted.
- `#nooverflow` BitVec VCs (Plan 33.7) — an *opt-in correctness* check for sized ints,
  orthogonal to the always-on `int` panic.
- `/`, `%`, unary `-` — no checked helper emitted today (no runtime check to elide).

## 2. Key design insight

The existing overflow-VC machinery (`bvadd_no_overflow_s/u`, …) is **BitVec-only** and does not
apply to unbounded `int` (modelled in SMT as the unbounded `Int` sort — no overflow concept).
So the proof is **a range-fit in unbounded-Int arithmetic**: for `a OP b`, prove

```
INT64_MIN ≤ (a OP b)  ∧  (a OP b) ≤ INT64_MAX        (i64::MIN = −9223372036854775808,
                                                       i64::MAX =  9223372036854775807)
```

under the facts available at the site (literals, `requires`, loop bounds `for i in lo..hi`,
`nat ≥ 0`). The runtime check panics **iff** the true mathematical `a OP b` leaves the i64
range, so proving the result stays in-range is exactly the elision condition. `int` operands
are modelled as *unbounded* (no artificial range axiom) → the goal is provable only when real
facts bound the result, never spuriously → **sound by construction**. `*` is non-linear → Z3
may return `Unknown` → **conservatively keep the check** (never elide on a non-proof).

## 3. Architecture — mirror Plan 140.2 exactly

Two proven-sets per the 140.2 / D257 soundness model:

| set | proved from | codegen elides |
|-----|-------------|----------------|
| **always-safe** | loop/code/literals only (Pass A, no `requires`) | **always** — even under `--contracts=off` / `#unchecked` (proof independent of contracts) |
| **contract-based** | needs fn-`requires` (Pass B ∖ Pass A) | **only when contracts enforced** for `requires` (under `--contracts=off` / `#unchecked(requires)` the check stays — `requires` not enforced ⇒ elision would be unsound) |

This is the **soundness boundary**: an `int`-overflow panic is a real safety guard and is
**never** elided by `#unchecked` alone — *only* by proof. The always-safe set is contract-
independent so it may be elided under `#unchecked`; the contract-based set may not.

### Verifier (`compiler-codegen/src/verify/pipeline.rs`)
- New `prove_int_overflow_sites(&self, module, fd) -> (Vec<Span>, Vec<Span>)` — clone of
  `prove_vec_index_sites`: pre-scan for `int` `+`/`-`/`*` sites (skip setup if none); shared
  `EncodeCtx`; Pass A (no requires) → always-safe; Pass B (assert requires) ∖ A → contract-based.
- `ovf_walk_{block,stmt,expr}` — clone of `idx_walk_*`: pushes `for i in lo..hi` bounds into the
  Z3 scope (`push`/`assert lo≤i<hi`/`pop`); at each `Binary{Add|Sub|Mul}` calls `ovf_try_prove`,
  records `e.span` on success. (No frame-safety — overflow is length-agnostic.)
- `ovf_try_prove(op, left, right, ctx, backend)` — encodes both operands, builds the range goal,
  `try_prove` → `Unsat` = proven. The `IntLit`-typed goal naturally excludes BitVec/FP ops (sort
  mismatch ⇒ not proven), so no explicit sort filter is needed (and a mis-recorded non-`int`
  span is harmless — codegen has no check there to elide).
- `ModuleVerifyReport`: `+ proven_overflow_sites: Vec<Span>` `+ proven_overflow_sites_contract: Vec<Span>`.
- `verify_module`: under the existing `!Trivial` gate, also run `prove_int_overflow_sites` per fn.

### Codegen (`compiler-codegen/src/codegen/emit_c.rs`)
- Fields `proven_overflow_sites` / `proven_overflow_sites_contract` (`HashSet<usize>`); setter
  `set_proven_overflow_sites(always, contract)`; predicate `overflow_site_elided(span_start)` =
  `proven_overflow_sites ∋ s || (!contracts_elided_here() && proven_overflow_sites_contract ∋ s)`
  — identical shape to `index_site_elided`.
- Gate the two **binary-expression** emission sites (`emit_expr` ~18398, `emit_expr_with_target_type`
  ~17621): when `overflow_site_elided(expr.span.start)` → fall through to the plain C operator
  instead of `nova_int_checked_*`.

### Wiring (`compiler-codegen/src/main.rs`)
- `emitter.set_proven_overflow_sites(&report.proven_overflow_sites, &report.proven_overflow_sites_contract)`
  alongside the existing `set_proven_index_sites*` calls (all codegen paths).

## 4. Scope boundary (documented, not a hidden simplification)

**V1 covers binary-expression `int` `+`/`-`/`*`** (sites 18398 + 17621 — the same Binary node,
one walker collection). This is the high-value 95 %: loop-index arithmetic and bounded-input
arithmetic.

**Compound-assign (`x += y`, site ~16840) is deferred** → followup
`[M-140.4-compound-assign-overflow-elision]`. Rationale: compound targets are typically
*unbounded accumulators* (`sum += a[i]`) that grow without a provable bound, so elision there is
near-never possible; the check stays (sound). Separate `Stmt::Assign` AST path with its own span
linkage. High effort, ~zero realisable elision → out of V1.

## 6. Acceptance criteria

- **OE1** — always-safe elision: `for i in 0..N { … i+1 … }` (constant `N`) → generated C has
  **no** `nova_int_checked_add` for that op; still elided under `--contracts=off`.
- **OE2** — contract-based elision: `fn f(a int) requires 0<=a && a<=1000 => a + 5` → elided
  under `--contracts=enforce`.
- **OE3** — contract-off keeps check: OE2's site under `// CONTRACTS off` → `nova_int_checked_add`
  **present** (contract-based ⇒ not elided when `requires` unenforced).
- **OE4** — soundness neg: unbounded `a + b` (no facts) → check **kept** → `EXPECT_RUNTIME_PANIC
  integer overflow` at `int.MAX + 1`.
- **OE5** — non-linear `*` un-proven stays checked (no false elision); proven `*` (constant
  bounds) elided.
- **OE6** — zero regressions: full `contracts/` + `basics` + `plan33_7`/`plan140*` green; the
  `int_overflow_add_panic` soundness test still panics.
- **OE7** — Trivial backend (default, no z3): no elision, no crash (gate respected).
