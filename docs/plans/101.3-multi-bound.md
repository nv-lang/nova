> Закрывает [Q-multi-bound](../../spec/open-questions.md#q-multi-bound).
> **Status:** 📋 proposed 2026-05-24, не начат.

## 1. Зачем

D72 currently требует anonymous-protocol или named-protocol для
multi-bound:

```nova
// Сейчас (verbose)
fn cache[T protocol { hash() -> u64, eq(other Self) -> bool }](xs []T) -> ...

// Или вынести в named:
type HashableEq protocol { hash() -> u64, eq(other Self) -> bool }
fn cache[T HashableEq](xs []T) -> ...
```

Хочется compact inline syntax:

```nova
fn cache[T Hashable + Eq](xs []T) -> ...
fn[T Hashable + Ord] []T @sort_dedup() -> []T => ...
```

Парallel Rust `<T: A + B>`. Закрывает open question, который висит
с D72 (active с давнего времени).

## 2. Дизайн

### Synтакс

```ebnf
Bound      ::= type ("+" type)*   // multi-bound через +
```

### Семантика

`[T A + B + C]` = T satisfies **все** protocols A, B, C (intersection).

### Disambiguation

- `+` в bound-position (внутри `[…]` после имени typevar) — multi-bound.
- `+` в expr-position — arithmetic. Контексты не пересекаются (bound
  inside type-decl brackets).
- Trailing `+`: `[T A +]` — parse error (нужен type после).
- Single bound `[T A]` (без `+`) — existing D72, без изменений.

### Чем Nova лучше альтернатив

| Lang | Multi-bound | Pros / cons |
|---|---|---|
| Rust | `<T: A + B>` | `+` familiar, но `:` в decl context |
| TS | `T extends A & B` | `&` intersection, но `extends` verbose |
| Kotlin | `<T : A>` + `where T : B` | дубликат T, verbose |
| Go | `interface { A; B }` (composition) | `;` separator, отдельный type-decl |
| Scala 3 | `T <: A & B` | `<:` subtype-оператор странный |
| Java | `<T extends A & B>` | `&` ок, но `extends` verbose |
| **Nova** | `[T A + B]` | no `:`, no `extends`, no `where` — самый compact |

## 3. Phases

### Ф.0 — Probe

`nova_tests/plan101_3/probe/`:
- `multi_bound_pre.nv` — `fn cache[T Hashable + Eq](xs []T) ...` —
  parser current behavior (likely parse error).
- `bound_plus_at_end_neg_pre.nv` — `fn[T A +] T @method` — likely
  parse error already.

### Ф.1 — Parser

**Acceptance:** parser accepts `Bound + Bound + ...` chains в bound-position.

**Actions:**
1. Modify `parse_generic_bound` в parser — allow `+`-separated chains.
2. AST: `Bound` либо `Type` либо `Intersection[Type, Type, ...]`.
3. Parse errors:
   - `[T +]` — leading `+` → error.
   - `[T A +]` — trailing `+` → error.
   - `[T A B]` (no `+`) — error «expected `+` or `]`».

### Ф.2 — Type-checker

**Acceptance:** T satisfies **all** bounds в intersection chain; loud
error если хоть один не satisfied.

**Actions:**
1. Extend bound-satisfies check для intersection: T must satisfy
   each Bi для всех i.
2. Error message: «T does not satisfy `A + B` — missing methods from
   `B`: foo, bar».

### Ф.3 — Codegen

Mono-name conversion: bounds НЕ влияют на mono-name (как сейчас в D72).
Intersection bound — это compile-time check, не runtime. Codegen
identical к single-bound case.

## 4. Risks

| Risk | Mitigation |
|---|---|
| `+` conflict с future arithmetic-in-types | `[…]` brackets — pure type context; identifier'ы only; no number literals → unambiguous |
| Programmers ожидают `&` (TS) | `+` familiar Rust audience (target); doc cross-references |
| Duplicate bounds `[T A + A]` — error or normalize? | Recommend warning + auto-dedup; design call в Ф.0 review |
