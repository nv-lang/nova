# Plan 118.1.6 — `#unsafe` как часть типа function pointer (`*fn(...)`)

## Что

Сейчас Plan 118.1.5 enforces `#unsafe` через имя callee — `unsafe_fns: HashSet<String>` lookup в типа-чекере. Это работает для прямых вызовов `foo()` или `obj.foo()`, но НЕ работает для вызовов через function pointer `*fn(...)`.

Plan 118.1.6 делает `#unsafe` частью **типа** function pointer (как `unsafe fn()` в Rust):
- `*fn() -> int` — safe function pointer
- `*unsafe fn() -> int` — unsafe function pointer
- Два разных типа, разные правила coercion + call-site enforcement.

## Почему

1. **Гap в Plan 118.1.5:** call через function pointer (`(some_fn_ptr)()`) обходит `unsafe_fns` HashSet lookup — checker не emit'ит E_UNSAFE_CALL_REQUIRES_WRAP. Если у пользователя function pointer на unsafe extern fn, он может вызвать его без `unsafe { }` блока.
2. **FFI callback patterns:** C-libraries часто принимают callback-функции (`qsort`, `signal`, etc.). При передаче Nova fn в C-callback мы обязаны указать safety contract — это делается через тип pointer'а.
3. **Rust precedent:** Rust'ы тип function pointer encode's safety и ABI. `fn()` ≠ `unsafe fn()` ≠ `extern "C" fn()`. Это proven pattern + миграция Rust-разработчиков становится intuitive.

## Дизайн (rules)

### Coercion правила (per Rust precedent, user-confirmed)

```
`*fn(...)` → `*unsafe fn(...)`:   ✅ allowed (safe — это «подмножество» unsafe; covariant)
`*unsafe fn(...)` → `*fn(...)`:   ❌ E_UNSAFE_FN_PTR_COERCION (нельзя «забыть» unsafe)
Same-shape `*fn` ↔ `*fn`:         ✅ identity (struct equality по signature)
```

### Call-site enforcement

```
Вызов `(safe_fn_ptr)()`:           OK везде
Вызов `(unsafe_fn_ptr)()` без unsafe {}: ❌ E_UNSAFE_CALL_REQUIRES_WRAP
Вызов `(unsafe_fn_ptr)()` внутри unsafe {}: ✅ OK
```

### Получение pointer

```nova
fn safe() -> int { 42 }
#unsafe fn risky() -> int { 42 }

ro p1 *fn() -> int        = safe          // OK
ro p2 *unsafe fn() -> int = safe          // OK (coerce safe → unsafe)
ro p3 *unsafe fn() -> int = risky         // OK
ro p4 *fn() -> int        = risky         // ❌ E_UNSAFE_FN_PTR_COERCION

p1()                                       // OK
unsafe { p3() }                            // OK — explicit unsafe block
p3()                                       // ❌ E_UNSAFE_CALL_REQUIRES_WRAP
```

## Calling convention заметка (informational)

Rust также encodes ABI в function pointer type (`extern "C" fn()` ≠ `fn()`). Nova V1 предполагает C calling convention implicitly для *fn(...) — единый convention. Multi-ABI support (если когда-либо нужен): отдельный followup `[M-118.1.x-fn-ptr-abi-tag]`.

## ABI tag design (informational)

- **Current:** `*fn(...)` implies C calling convention (System V on Unix, MS x64 on Windows). Codegen lowers `TypeRef::Func` к `void*`, без ABI-emit logic. Только 4 фикстуры используют `*fn(`; в std/ и examples/ — ноль usage.
- **Question:** заменить ли syntax на `*extern "C" fn(...)` для explicit multi-ABI?
- **Decision:** **REJECTED — keep `*fn(...)` as C-ABI shorthand (Option B / Option A в rust-precedent rubric).** Three converging research outputs unanimously recommend сохранять current syntax:
  1. **fn-ptr-syntax audit:** verbose default `*extern "C" fn(...)` punishes >99% C-ABI case ежедневно для feature (`[M-118-stdcall-fn-ptr]`) explicitly marked «niche, add when needed». Contradicts Nova's «common-case unmarked» philosophy (D36 let-default, D52 value-default, D82 implicit-C external fn).
  2. **rust-precedent:** Rust treats C ABI как единственный stable cross-platform target — другие ABI (stdcall = Win32-only; sysv64/win64 = single-arch) niche. Zig migration cautionary tale: kept `extern fn` sugar для C common case даже после callconv refactor.
  3. **Grammar extends additively:** `*fn(...)` is `Pointer(Func)`; добавление `*extern "stdcall" fn(...)` — pure addition: reserve `KwExtern` token + accept `extern "STR"` prefix в fn-type production + add `abi: Option<AbiTag>` field на `TypeRef::Func` (default `None ≡ C`). Existing parse path unchanged.
- **Implementation hook when needed:** при активации `[M-118-stdcall-fn-ptr]`: (a) lexer reserves `extern` keyword (currently free — collision-checked); (b) `AbiTag { C, Stdcall, Vectorcall, System, CUnwind, ... }` enum + optional field; (c) codegen emits `__stdcall` / `__attribute__((stdcall))` based on tag; (d) ABI string `"C"` = canonical explicit form для default; (e) mirror на `external fn` declarations для symmetry.
- **Spec note (added к D216 §10):** «`*fn(Args) -> Ret` canonically equivalent to future `extern "C" *fn(Args) -> Ret`; alternative ABIs reserved via explicit `extern "STR"` prefix syntax. Followup `[M-118.1.x-fn-ptr-abi-tag]`.»
- **Followup marker (deferred):** `[M-118.1.x-fn-ptr-abi-tag]` — multi-ABI support, активируется при первом non-C ABI need (Win32 COM/stdcall ИЛИ Rust 2024 `extern "C-unwind"` throwing FFI).

## Forward declaration question

- **Current:** `external fn` работает ИСКЛЮЧИТЕЛЬНО для D82 nova_rt projection (runtime-implemented в nova_rt/), НЕ как general forward decl. После Plan 115/118 — также для user-level C-FFI bindings. Parser enforces strict dichotomy: `external fn` обязательно без body; non-external `fn` обязательно с body (`=>` или `{}`). Bare `fn foo() -> int` без body и без `external` → hard parse error «expected `=>` or `{` for function body». Нет third branch.
- **Question:** поддерживать ли Nova-body forward declaration (declare signature → define body позже в том же module)?
- **Decision:** **REJECTED — НЕ добавлять Nova-body forward declaration.** Forward-decl audit shows net-negative для design surface:
  1. **Все классические use cases уже решены лучшими механизмами:**
     - Same-file mutual recursion: type-checker делает two-pass resolution (collect все fn signatures сначала, потом проверяет bodies — `types/mod.rs:300+`). `fn a() { b() }` + `fn b() { a() }` работает без forward decl.
     - Cross-file mutual recursion: peer-files model (spec/decisions/07-modules.md:160-183) — все .nv в одной папке делят `module X.Y` namespace, видны друг другу без import.
     - Header/impl split: Nova module-based, не C/C++. Нет concept'а header'а; split создал бы двойственность «где смотреть signature» (нарушает D10 AI-first locality).
     - Public API без body exposure: `export` modifier (D5/D47) + opaque `external type` (D126) уже покрывают это.
  2. **Conflict с D82 single-source принципом:** forward decl создаёт третью категорию (`fn` без body, ожидающая match с `fn` с body) рядом с existing dichotomy `external fn` (runtime-implemented) vs `fn` с body (Nova-implemented). Parser не сможет determine в момент парсинга `fn foo()` что это — forward decl ИЛИ typo (developer forgot body). Силент acceptance скрывает ошибки; current error «expected `=>` or `{`» actionable.
  3. **Rust precedent:** Rust forward decls живут ТОЛЬКО в `unsafe extern "C" { fn foo(x: i32); }` блоках для FFI — pure Rust fns CANNOT быть forward-declared, body required at declaration site. Nova `external fn` уже mirrors этот pattern для FFI use case.
  4. **D10 AI-first locality + Go precedent:** forward decl scatters definition across files; AI-агент + человек предпочитают атомарную definition. У Go (highest LLM-success rate per spec) нет forward decl — same package = transitive visibility.
- **Followup marker:** **NONE.** Решение final, не deferred. Если когда-нибудь Plan 118.x или другой план попросит forward decl — pushback с этим audit'ом. Если понадобится C-FFI prototype binding — distinct syntax (`extern "C"` или `#abi`), НЕ overloading bare `fn` без body.
- **Test gap (optional):** добавить fixture в `nova_tests/` для negative case `fn foo() -> int` без body → verify E_PARSER error stable. Закроет gap в test coverage (сейчас 0 fixtures проверяют этот error path).
