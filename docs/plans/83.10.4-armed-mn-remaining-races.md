## 2. Architecture invariants (must hold throughout)

Эти инварианты — **non-negotiable**. Любое решение, нарушающее их, отвергается
без compromise.

**I-1. libuv handle thread-affinity.** `uv_*` операции на handle вызываются
только из потока, владеющего `uv_loop_t`, к которому handle привязан.
Cross-thread close → через `NovaDeferredCloseQueue` (Plan 83.10.2 infrastructure).

**I-2. TLS handler snapshot consistency.** При входе в `mco_resume(coro)` TLS
handler stack должен отражать handlers, которые fiber видел при последнем
`mco_yield`. Worker switch (W1 → W2) обязан save/restore handler stack атомарно
вокруг resume.

**I-3. setjmp/longjmp single-thread invariant.** `longjmp` к `jmp_buf`,
установленному на потоке T1, **никогда** не вызывается с потока T2. Любой
fail/throw путь, требующий cross-thread унвинда → реализуется через сигнал
к owning thread + локальный `longjmp` там.

**I-4. Cancel propagation idempotency.** `tok.cancel()` вызванный из main или
worker — одинаковая семантика. Никаких отдельных code path по `is_main_thread()`.

**I-5. CAS-based state machines всегда recover-able.** Любая `nova_aint_cas`
последовательность должна иметь четкий fallback на спорные исходы (CAS lose).
Нет dead-end states.

**I-6. Никаких `NOVA_AUTOARM=0` workaround'ов в фиксах.** Если fix требует
disable armed M:N — это не fix, а sidestep. Возвращаемся к RCA.

### Ф.4 — RCA + fix longjmp cross-mco (Sonnet 4.6 + High + **Thinking ON**)

**Длительность:** ~1 dev-day.

**Маркер:** `[M-83.10.1-fail-handler-cross-mco-longjmp]`.
**Тест:** `effects/fail_handler.nv`.

**Шаги:**

1. **RCA:**
   - `with Fail = handler { ... } { ... }` устанавливает `jmp_buf` через
     `setjmp` на главном потоке (или потоке handler-frame).
   - Под armed M:N `throw` происходит на worker thread'е, где fiber
     резюмируется → `longjmp` пытается прыгнуть к `jmp_buf` другого потока.
   - Это нарушает I-3 (single-thread setjmp/longjmp).
2. **Solution direction (similar to 83.10-armed-user-throw-routing):**
   - Worker thread обнаруживает USER_THROW → **не longjmp'ает локально**,
     а **сигналит** через `nova_runtime_report_throw_to_scope(scope, err)`.
   - Owning thread (где `setjmp` был сделан) на следующем uv_run pass
     dequeue'ит throw → локальный `longjmp`.
   - Reuse infrastructure из 83.10 fix (USER_TYPED routing).
3. **Документ:** обновить `spec/decisions/06-concurrency.md` D-блок про
   handler routing (если уже есть D-блок 83.10 — extend; иначе new D).
4. **Тесты:**
   - **Positive:** `nova_tests/plan83_10_4/fail_handler_armed.nv` — proof
     of fix.
   - **Negative regression guard:** `nova_tests/plan83_10_4/fail_handler_cross_worker_race.nv`
     — 50 fiber'ов throw'ают одновременно с разных worker'ов, handler ловит
     все (count == 50).
   - **Existing:** `effects/fail_handler.nv` — удалить AUTOARM=0, PASS armed.
5. **Ф.4 commit:** `fix(plan83.10.4 Ф.4): longjmp cross-mco — cross-thread throw routing`.

**Acceptance:**

- 3 теста (1 existing + 2 new) PASS armed 10/10.
- I-3 invariant verified (longjmp всегда на owning thread).

**Гейт:** ✅ если 10/10 PASS.

## 3.5 Coordination model — single-agent vs parallel

**Sonnet 4.6 является минимальной моделью для этой работы (§0).** План
поддерживает два режима исполнения. Агент **выбирает режим** по инструкции,
полученной при старте — режим **не должен** меняться середине.

### 3.5.1 Single-agent sequential (M-SEQ)

**Один агент** выполняет Ф.0 → Ф.1 → Ф.2 → Ф.3 → Ф.4 → Ф.5 → Ф.6 → Ф.7 → Ф.8
в одном worktree `nova-p83-10-4`, одной ветке
`plan-83.10.4-remaining-races`. Каждая фаза — отдельный commit (или
коммиты, если требует §3 каждой фазы).

**Ответственность:** агент сам управляет всеми фазами. Никаких внешних
координаторов.

**Длительность:** ~3.5 dev-day с Thinking ON.

### 3.5.2 Multi-agent parallel (M-PAR)

**5 агентов**, каждый в своём worktree. Координация через явный merge
protocol (§3.6).

| Agent | Worktree | Ветка | Фазы | Когда стартует |
|---|---|---|---|---|
| A1 | `nova-p83-10-4-audit` | `plan-83.10.4-audit` | Ф.0 + Ф.1 | Сразу |
| A2 | `nova-p83-10-4-tls` | `plan-83.10.4-tls` | Ф.2 + Ф.3 | После A1 merge в main |
| A3 | `nova-p83-10-4-longjmp` | `plan-83.10.4-longjmp` | Ф.4 | После A1 merge в main, параллельно A2 |
| A4 | `nova-p83-10-4-cancel` | `plan-83.10.4-cancel` | Ф.5 | После A1 merge в main, параллельно A2/A3 |
| A5 | `nova-p83-10-4-closure` | `plan-83.10.4-closure` | Ф.6 + Ф.7 + Ф.8 | После A2/A3/A4 merge в main |

**Branch base:** каждая ветка отслеживает свежий `github/main` (после
предыдущей merge gate).

### 3.5.3 Worktree assignment rules

- **Worktree convention:** `nova-p83-10-4-<role>` (нарушение конвенции
  `nova-pNN` допустимо для роли-суффикса — иначе 5 worktree неотличимы).
- **Libuv submodule:** копировать из main + удалить `.git`
  (memory `project-worktree-nova-test-setup`).
- **Env vars:** `NOVA_GC_LIB_DIR`, `NOVA_GC_INCLUDE_DIR` указывают на
  main repo `vcpkg_installed/x64-windows-static/`.
- **CLI build:** `cargo build --release --manifest-path nova-cli/Cargo.toml`
  в самом worktree (отдельная target/ на worktree).

### 3.5.4 Branch convention

- `plan-83.10.4-audit` — Ф.0+Ф.1 commit'ы + audit-report.md.
- `plan-83.10.4-tls` — Ф.2+Ф.3 commit'ы.
- `plan-83.10.4-longjmp` — Ф.4 commit'ы.
- `plan-83.10.4-cancel` — Ф.5 commit'ы.
- `plan-83.10.4-closure` — Ф.6+Ф.7+Ф.8 commit'ы.

Все 5 веток отдельно push'аются на `github` + `gitverse`. Финальный merge
в main делается **человеком-координатором** (не агентом).

## 3.6 Parallel merge protocol (M-PAR mode only)

**Между фазами parallel-mode'а нужен человек-координатор** для merge.
Агенты **не** делают merge между собой автоматически.

### 3.6.1 Gate 1: после A1 (audit)

A1 finishes → coordinator:
1. Reviews `audit-report.md`.
2. Merges `plan-83.10.4-audit` → `main` (no-ff).
3. Push'ит `main` на github + gitverse.
4. Стартует A2, A3, A4 параллельно (после fetch fresh main).

### 3.6.2 Gate 2: после A2/A3/A4 (fix-фазы)

Каждый из A2/A3/A4 finishes на своей ветке. Coordinator:

1. **Sequential merge** в `main` (не одновременно — merge conflicts на
   `fibers.h`/`runtime.c`/`nova_sched.h` гарантированы):
   - merge A4 (`cancel`) → main → push (выбран первым как самый изолированный).
   - merge A3 (`longjmp`) → main → resolve conflicts → push.
   - merge A2 (`tls`) → main → resolve conflicts → push.
2. **После каждого merge** — full nova test verify, не regress.
3. Стартует A5.

### 3.6.3 Gate 3: после A5 (closure)

A5 finishes → coordinator:
1. Reviews simplifications/project-creation/discussion-log updates.
2. Merge `plan-83.10.4-closure` → `main` (no-ff).
3. Push.
4. **Закрывает план** в memory.

### 3.6.4 Conflict resolution rules

При conflict в Gate 2:
- **`fibers.h`/`runtime.c`/`nova_sched.h`:** keep **both** changes, manually
  resolve по invariants §2 (I-1..I-6). Если непонятно — откатить parallel-mode,
  передать в single-agent.
- **Тесты (`nova_tests/plan83_10_4/*`):** оба набора держим, никогда не
  удаляем тесты из других веток.
- **Docs:** оба набора пометок маркеров держим.

### 3.6.5 Fallback to single-agent

Если на любом Gate conflicts невозможно разрешить безопасно (≥1 invariant
§2 нарушается) → **откат parallel-mode'а**:
1. Hard-reset `main` до состояния до Gate 2.
2. Удалить worktree A2/A3/A4 (но **не** удалять ветки в remote, как archive).
3. Стартовать единого agent в режиме M-SEQ от текущего main.
