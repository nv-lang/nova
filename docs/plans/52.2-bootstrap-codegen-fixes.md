## Зачем

Plan 52.1 Ф.1 (spread `[...defaults, k:v]`) и Ф.2 (const map-literal)
не были закрыты из-за двух **pre-existing baseline issues**:

1. **Codegen quirk (Ф.1 блокер):** `match { Some(v) => { ... } }` в
   expression-position внутри helper-method генерирует "anonymous
   record literal without spread not supported in codegen".
2. **Const ordering bug (Ф.2 блокер):** `const X HashMap[K,V] = [...]`
   падает с `unknown type name 'Nova_HashMap____nova_str__nova_int'`
   — const эмитится до mono pass.

Оба ограничения **не Plan 52 регрессии** — они существовали в baseline.
Закрытие в этом плане разблокирует Ф.1 + Ф.2.

## Ф.1 — Codegen: emit_record_lit fallback для wrap-in-block context

### Investigation

Минимальный repro:
```nova
test "x" {
    let mut i = 0
    while i < 1 {
        match Some(42) {
            Some(v) => { let _ = v }
            None => {}
        }
        i += 1
    }
}
```

→ CODEGEN-FAIL `anonymous record literal without spread not supported`.

Та же конструкция в top-level fn-body работает (см. `@get` / `@insert`
в `std/collections/hashmap.nv`).

### Root cause hypothesis

`emit_record_lit` (emit_c.rs:12490+) при anonymous record literal
fallback'нет на `current_fn_return_ty` для infer'инга struct-имени.
Если current_fn_return_ty не подходит (test-fn возвращает `nova_unit`)
— падает.

Но `match` arm-body `{ }` это **Block**, не record-литерал. Значит
где-то AST-pass конвертит Block в anonymous RecordLit. Возможные
кандидаты:
- `desugar.rs` для `{}` map-coercion (но это для `{field: v}`,
  не для empty block)
- type-checker может оборачивать в record для type-inference
- emit_match wrapper

Investigation steps:
1. Добавить trace в emit_record_lit fallback path — log AST + context
2. Найти узел AST который "anonymous record" — какой именно был source
3. Локализовать pass который генерирует — может быть пропущенный
   match arm wrapping

### Fix

После локализации:
- Если pass нелегитимно эмитит RecordLit для Block — fix в этом pass
- Если emit_record_lit fallback слишком узок — расширить (например,
  для test-fn body использовать inferred struct из контекста match-arm)

### Acceptance Ф.1

- [x] **Repro test PASS** (match arm-body block в while-loop). Fix:
      `looks_like_record_lit` для empty `{}` возвращает `false`,
      empty `{}` парсится как Block.
- [x] **Регрессия закрыла +27 чужих тестов** (485 → 512 PASS).
- [ ] ⛔ DEFERRED: `HashMap.@clone()` / `@merge_from` — упирается
      во второй слой mono-pass для generic helper-methods. Codegen
      падает с `if condition must be bool, got nova_int` на
      generated/desugared mono'd код. Это **отдельная** issue.
- [ ] ⛔ DEFERRED: spread parser/desugar — зависит от helpers.

### Reassessment

Plan 52.2 Ф.1 закрыл **первый слой** (empty `{}` parser quirk) с
massive collateral wins (+27 чужих тестов проходят). Но spread
требует ещё **второго слоя** — mono-pass для generic helpers с
pattern-match, который оказался отдельной проблемой выявленной только
после прохождения первого слоя.

Spread сам по себе декомпозируется на 2 части:
1. **Parser/desugar для `[...expr, k: v]`** — ~50 LOC, тривиально
2. **Working `HashMap.@clone()`/`@merge_from()`** — нужно для desugar
   target. Заблокировано mono-pass quirk.

Решение для bootstrap: использовать `HashMap.from(array_of_pairs)` +
концатенацию массивов. Реализация полного spread — Plan 52.3
после фикса mono-pass.

## Ф.2 — Codegen: const-decl ordering после mono pass

### Investigation

`const KEYWORDS HashMap[str, int] = ["if": 1]` генерирует:
```c
// Line 96: const decl (early)
static Nova_HashMap____nova_str__nova_int* _nova_const_KEYWORDS_value;
// Line 99: struct definition (later, через mono)
typedef struct Nova_HashMap____nova_str__nova_int ...
```

→ C-compiler: line 96 references type unknown until line 99.

### Fix options

**Option A — Forward-declare mono'd types перед const-decl:**
- В `emit_module` сначала собрать все типы используемые в const-decl'ах
- Эмитить `typedef struct X X;` forward-declarations
- Затем const-decl, затем mono pass с full struct definitions

**Option B — Отложить const emit до конца mono pass:**
- `emit_module` order: types → mono types → fns → const-decls (last)
- Но: const используется в fn'ах, fn-forward-decls должны быть раньше
- Усложнение ordering

**Option C — Use ptr-typed `void*` для const-value до использования:**
- `static void* _nova_const_KEYWORDS_value;` (works without forward-decl)
- Cast at use-site
- Минимально инвазивно, но теряет type safety на C-side

**Решение:** Option A — forward-declare mono'd types. Аккуратно
fits с existing mono pass.

### Fix

В `emit_c.rs::emit_module`:
1. Pre-scan const-decls + fns с generic-возвратами → выясняем какие
   mono'd struct-имена нужны
2. После type-decl emit, **перед** const-decl emit — эмитим
   forward-declarations этих struct-ов:
   ```c
   typedef struct Nova_HashMap____nova_str__nova_int Nova_HashMap____nova_str__nova_int;
   ```
3. Const-decl emit referencing'ит type через alias — компилируется
4. Mono pass дает полную struct-definition позже

### Acceptance Ф.2

- [ ] `const M HashMap[str, int] = ["if": 1]` компилируется + работает
- [ ] `const N HashMap[int, str] = [1: "x"]` (другой mono) тоже работает
- [ ] Const-evaluation correct (lazy init на первом use → cached)
- [ ] Регрессия по non-map const'ам без новых FAIL
- [ ] Type-check error если const-литерал runtime-зависимый
