# Plan 42.12: Conditional compilation (`#cfg`)

> **Создан 2026-05-14** — приоритет **HIGH**, **блокер для Plan 18 P0**
> (networking, crypto cross-platform).
>
> **Зависит от:** Plan 42 stable (folder-modules + `PeerFile`).

---

## Зачем

При росте stdlib появятся платформенно-специфичные backend'ы:

| Stdlib | Windows | Linux | macOS |
|---|---|---|---|
| TLS | Schannel | OpenSSL/rustls | Secure Transport |
| Filesystem | Win32 API | POSIX | POSIX+darwin |
| Crypto entropy | BCryptGenRandom | getrandom | getentropy |
| Networking | Winsock | epoll/io_uring | kqueue |

**Без compile-time switch:**
- Runtime branching → dead code в release binary (Linux-only код в .exe)
- Security risk (включён код для другой ОС)
- Bloat: каждый бинарь содержит N платформенных runtime'ов

Plan 18 P0 (networking, crypto) **не написать без `#cfg`**. Это первый
блокер.

---

## Архитектурное решение

**Два механизма** (как в Go), но выбираем **оба**:

### Mechanism A: Filename suffix convention (Go-style)

```
std/net/
├── tls.nv               // shared API (только signatures)
├── tls_windows.nv       // peer, active только на Windows
├── tls_linux.nv         // peer, active только на Linux
└── tls_macos.nv         // peer, active только на macOS
```

Compiler выводит target из filename suffix:
- `_windows.nv` → active при `target=windows`
- `_linux.nv` → active при `target=linux`
- `_macos.nv` → active при `target=macos`
- `_unix.nv` → active при `target ∈ {linux, macos, bsd}`
- `_posix.nv` → синоним `_unix`
- Без suffix → active всегда

**Pros:** просто, no AST changes, peer filter в imports.rs.
**Cons:** только OS-level, не feature-flags.

### Mechanism B: `#cfg` attribute (Rust-style minimal)

```nova
#cfg(feature = "experimental_io_uring")
module std.net.tls

export fn connect(host str) -> Connection { ... }
```

Только **простые predicates**:
- `#cfg(feature = "X")` — feature flag из nova.toml
- `#cfg(target_os = "Y")` — OS check (комплементарно filename suffix)

**Никаких `any(...)`, `all(...)`, `not(...)` композиций** — strict
minimal. Если нужна композиция — два attribute блока ИЛИ filename.

**Финальный выбор:** **A** (OS routing) + **B** minimal (features).

## Phases

### Ф.1 — Filename suffix convention

- **Recognized suffixes:** `_windows`, `_linux`, `_macos`, `_unix`, `_posix`
- **Detection:** `compiler-codegen/src/imports.rs::resolve_module_paths`
  filter peers by current target.
- **Target detection:** CLI flag `--target=windows|linux|macos`,
  default — host (Rust `cfg!(target_os = ...)`).
- **`_test.nv` interaction:** `_windows_test.nv` valid — оба suffix
  применяются (active на windows в test mode).
- **AST:** `PeerFile.target_filter: Option<TargetFilter>`.

**Acceptance:**
- `std/net/tls_windows.nv` собирается только на Windows.
- 278 PASS regression untouched.

### Ф.2 — `#cfg(feature = "X")` attribute (minimal)

- **Syntax:** `#cfg(feature = "name")` перед module declaration или
  перед top-level item.
- **Evaluator:** binary on/off. No `any/all/not`. Feature names — string
  identifiers из `nova.toml [features]`.
- **CLI:** `nova build --features=experimental,unstable_tls`.
- **AST:** `ModuleAttr::CfgFeature(String)`, `Item.attrs` для items.

**Acceptance:**
- `nova test --features=foo` активирует `#cfg(feature = "foo")` peers.
- Без `--features` — items с `#cfg` skip'аются.

### Ф.3 — Item-level `#cfg` ⏳ ОТЛОЖЕНО

- **Status (2026-05-14):** module-level `#cfg(target_os/feature)` уже
  реализован в Ф.2. Item-level routing — extension: одна функция в
  peer OS-specific. Требует AST extension для `Item.attrs`, парсер
  для item-level attrs, codegen filter.
- **Решение:** отложить. Filename suffix (Ф.1) + module-level `#cfg`
  (Ф.2) покрывают 95% production cases. Item-level — edge case,
  можно work-around через split peer.
- **Когда возвращаться:** когда stdlib встретит item-level OS-need
  без возможности split peer.

## Risks

1. **Cross-compile complexity.** `--target=linux` на Windows-host
   требует Linux libc / cross C-toolchain. **Mitigation:** Ф.1 покрывает
   только AST-level filter; cross-compile toolchain — отдельный план.
   Initial scope: same-target only.
2. **Feature combinatorics.** Если 5 features, 32 combos для testing.
   **Mitigation:** binary on/off only, no predicates. Тестируем каждую
   feature отдельно + 1-2 combo.
3. **Spec drift.** `#cfg` syntax близок к Rust — но Rust имеет тяжёлые
   `cfg-attr`, `cfg-expr in expr position`. **Mitigation:** Strict
   minimal — только attribute, никаких predicate expressions.

## Что НЕ входит

- `any/all/not` predicates — future revisit.
- Conditional code в **expression position** (`if #cfg(target_os = ...)`)
  — not needed.
- Cross-compile toolchain setup — separate plan.
- Build-time code generation.
- Macro system based on cfg.

## Сравнение с Go / Rust / TS

| Feature | Go | Rust | TS | Nova (this plan) |
|---|---|---|---|---|
| Filename suffix | `_linux.go` | n/a | n/a | `_linux.nv` ✅ |
| Build constraints | `//go:build linux` | `#[cfg(target_os = "linux")]` | n/a | `#cfg(target_os = ...)` ✅ minimal |
| Feature flags | build tags | `#[cfg(feature = "x")]` | tsconfig.paths hack | `#cfg(feature = "x")` ✅ |
| Predicates | `&&`, `\|\|`, `!` | `any/all/not` | n/a | ❌ (minimal scope) |
| cfg-on-expression | n/a | ✅ | n/a | ❌ (not needed) |
| cfg-on-item | n/a | ✅ | n/a | ✅ Ф.3 |

**Nova подход** — комбинация Go simplicity (filename suffix) + Rust
expressiveness (`#cfg(feature)`) **минус** complexity (no predicates,
no cfg-expr).
