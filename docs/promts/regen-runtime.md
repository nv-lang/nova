# Промпт: регенерировать `std/runtime/*.nv`

**Связь:** [Plan 13](../plans/13-runtime-stdlib-and-autogen.md),
[D82](../../spec/decisions/08-runtime.md#d82).

## Цель

Перегенерировать read-only Nova-side projection реестра runtime-
функций компилятора. Файлы `std/runtime/*.nv` (`builtins.nv`,
`string.nv`, `math.nv`) — auto-generated; источник истины —
`compiler-codegen/src/codegen/runtime_registry.rs`.

После прогона все `external fn`-декларации в `std/runtime/*.nv`
точно соответствуют тому что компилятор знает в bundled runtime.

## Когда запускать

- После любого изменения `runtime_registry.rs` (добавили/убрали
  функцию, поменяли сигнатуру).
- После любого изменения C-стороны (`nova_rt/string.c`,
  `nova_rt/math.c`) — если поменяли C-сигнатуру.
- В CI как regression check (Plan 13 Ф.6).

## Входы

- `compiler-codegen/src/codegen/runtime_registry.rs` — структурированный
  список `RuntimeFn` со всеми сигнатурами.
- `compiler-codegen/nova_rt/*.c` / `*.h` — реальные C-реализации.

## Выходы

Перезаписываются (single-writer). Точный набор файлов зависит от
содержимого `runtime_registry.rs` — генератор пишет один файл на
каждый уникальный module-prefix:

- `std/runtime/string.nv` — UTF-8 ops для `str`.
- `std/runtime/math.nv` — `f64`/`f32` math (sqrt/sin/cos/...).
- `std/runtime/string_builder.nv` — StringBuilder API (после Ф.8).
- `std/runtime/write_buffer.nv` — WriteBuffer API (после Ф.8).
- `std/runtime/read_buffer.nv` — ReadBuffer API (после Ф.8).
- `std/runtime/char.nv` — char-операции (`str.from(c char)`, etc., после Ф.8).

(До завершения Ф.8 `std/runtime/builtins.nv` может ещё
существовать как handwritten — после Ф.8 его не должно быть.)

Все файлы начинаются с шапки:
```nova
// AUTO-GENERATED by `nova-codegen emit-runtime-stubs`.
// Do not edit manually — changes will be overwritten.
// Source of truth: compiler-codegen/src/codegen/runtime_registry.rs
```

## Шаги

1. **Sanity-check состояния.** Убедись что:
   - `cargo build` в `compiler-codegen/` проходит чисто.
   - В `runtime_registry.rs` нет TODO/FIXME без owner'а — каждая
     `RuntimeFn` должна указывать реальную C-функцию.
2. **Запустить генератор:**
   ```
   cd compiler-codegen
   cargo run -- emit-runtime-stubs
   ```
   Это перезаписывает `std/runtime/builtins.nv`, `string.nv`, `math.nv`.
3. **Проверить детерминизм.** Запусти команду повторно — diff должен
   быть пустым:
   ```
   cargo run -- emit-runtime-stubs
   git diff std/runtime/
   ```
   Если есть diff — bug в генераторе (нестабильный порядок,
   нестабильное форматирование). **Не коммитить** до фикса.
4. **Проверить что всё собирается.** В корне:
   ```
   pwsh ./run_tests.ps1 -IncludeStdlib
   ```
   Все codegen-тесты + nova_tests должны проходить. Расхождение
   означает что .nv-проекция не матчит ожиданиям type-check'а или
   codegen'а.
5. **Проверить C-сторону.** Для каждой новой функции из `runtime_registry`
   убедись что есть реальный `Nova_X_method_Y` в `nova_rt/*.c`.
   Если генератор сообщил «not implemented in runtime» — добавь
   C-реализацию **до** коммита `.nv`-файлов, иначе валидация D82
   Diagnostics упадёт.
6. **Закоммитить.** В одном коммите три рукотворных + три auto-gen
   файла:
   - `runtime_registry.rs` (изменение реестра).
   - `nova_rt/<module>.c` / `.h` (C-реализация).
   - `std/runtime/builtins.nv` / `string.nv` / `math.nv` (auto-gen).

   Commit-message формат: `runtime: add f64.@cbrt (cube root)`. В теле
   упомянуть «std/runtime/*.nv regenerated by `emit-runtime-stubs`».

## Ограничения

- **НЕ редактируй `std/runtime/*.nv` вручную.** Изменения теряются
  при следующей регенерации. Если нужно изменить декларацию —
  правь `runtime_registry.rs` и регенерируй.
- **НЕ убирай AUTO-GENERATED шапку.** Pre-commit/CI guard её
  проверяет.
- **НЕ добавляй декларацию для функции, у которой нет C-реализации
  в `nova_rt/*.c`.** Компилятор откажется загружать (D82 Diagnostics
  «not implemented in runtime»).
- **НЕ добавляй auto-derivable формы** (например, `@try_read_X`
  если есть `@read_X` Fail-form). Компилятор синтезирует их сам;
  явная декларация — compile error по правилу Plan 12 Ф.4.5.

## Acceptance

- [ ] `cargo run -- emit-runtime-stubs` завершается без ошибок.
- [ ] Повторный прогон даёт пустой diff (детерминизм).
- [ ] `run_tests.ps1 -IncludeStdlib` проходит полностью.
- [ ] Каждая декларация из обновлённых `.nv`-файлов имеет реализацию
      в `nova_rt/*.c`.
- [ ] Шапка AUTO-GENERATED присутствует во всех трёх файлах.
- [ ] Коммит включает и source (`runtime_registry.rs` + `nova_rt/*.c`),
      и auto-gen (`std/runtime/*.nv`).

## Tроблшутинг

| Симптом | Причина / решение |
|---|---|
| `emit-runtime-stubs` падает с «duplicate C-name» | Два `RuntimeFn` сгенерили одинаковый mangled name. Проверь параметры overload — возможно нужен parameter-type mangling (Plan 11 Ф.3). |
| Повторный прогон даёт diff | Нестабильный порядок в генераторе. Должна быть sort by (module, receiver, name). Bug в `runtime_registry::all()` или в renderer. |
| `run_tests.ps1` падает после регенерации с «no method» | Декларация удалена из реестра, но user-код всё ещё вызывает функцию. Либо вернуть в реестр, либо обновить user-код. |
| Линкер (`cc`) ругается на undefined reference | Декларация добавлена в реестр, но C-реализация не написана. См. шаг 5. |
| Компилятор ругается «signature mismatch» при загрузке `std/runtime/string.nv` | Сигнатура в реестре расходится с C-реализацией. Сверь `RuntimeFn::params/return_ty/effects` с `Nova_<name>` в `.c`-файле. |
