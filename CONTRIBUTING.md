# Участие в Nova

Спасибо за интерес к проекту. Nova сейчас — дизайн-документ, и контрибуции
ценны как в обсуждение идей, так и в текст спецификации.

## Как внести вклад

1. Откройте issue с описанием предложения или проблемы — для нетривиальных
   изменений лучше согласовать направление до PR.
2. Сделайте форк, создайте ветку от `main`.
3. Оформите PR. Каждый коммит должен быть подписан DCO (см. ниже).

## Лицензия

Проект распространяется под двойной лицензией `MIT OR Apache-2.0`, документация —
под `CC-BY-4.0`. Полные тексты: [LICENSE-MIT](LICENSE-MIT),
[LICENSE-APACHE](LICENSE-APACHE).

Любой вклад, намеренно отправленный для включения в проект, лицензируется на тех
же условиях, без дополнительных условий — в соответствии с разделом 5 Apache
License 2.0.

## Developer Certificate of Origin (DCO)

Вместо CLA проект использует [DCO](https://developercertificate.org/) — лёгкий
способ подтвердить, что вы имеете право вносить изменения и согласны с лицензией
проекта.

Каждый коммит должен содержать строку `Signed-off-by` с вашим настоящим именем и
email:

```
Signed-off-by: Имя Фамилия <email@example.com>
```

Добавляется автоматически флагом `-s`:

```
git commit -s -m "сообщение"
```

Если забыли — можно дополнить последний коммит:

```
git commit --amend -s --no-edit
```

Или несколько коммитов сразу через rebase:

```
git rebase --signoff main
```

PR без подписей не пройдёт проверку CI.

## SPDX-хедеры (на будущее)

Когда в репозитории появятся исходники компилятора и стандартной библиотеки,
каждый файл должен начинаться со строки SPDX:

```
// SPDX-License-Identifier: MIT OR Apache-2.0
```

Для документации (`.md`):

```
<!-- SPDX-License-Identifier: CC-BY-4.0 -->
```

Сейчас это не требуется — действует общая лицензия из корня репозитория.

## Soundness-regression-suite (Plan 33.8 Ф.7)

Верификатор контрактов (Plan 33) обязан быть **sound**: вердикт «доказано»
должен строго влечь «истинно в рантайме». Два аудита нашли 6 soundness-дыр —
поэтому действует правило защиты от регрессий:

- **Нашли soundness-баг** (верификатор объявил контракт «доказан», хотя он
  ложен; или проверка молча пропущена) → **обязательно** добавьте тест,
  который ловит именно эту регрессию.
- Пометьте такой тест маркером-комментарием `// SOUNDNESS_REGRESSION`
  в первых строках файла (`.nv`-тест) или над `#[test]` (unit-тест).
- CI-job `soundness-regression-guard` следит, что число guard-тестов не
  убывает. Добавили новый — поднимите порог `MIN` в
  `.github/workflows/contracts-z3.yml`.
- НЕ ослабляйте и не удаляйте `SOUNDNESS_REGRESSION`-тесты без явного
  обоснования в PR — каждый стережёт конкретную закрытую дыру.
