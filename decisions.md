# Nova — журнал дизайн-решений

Здесь фиксируются принятые решения с причинами и **отвергнутые альтернативы**,
чтобы при доработке не терять контекст. Каждое решение — отдельный пункт.

---

## D1. Парадигма: protocols + data, без классов

**Решение.** Никаких классов и наследования. Вместо них:

- `type` — данные (record, sum-type, alias). См. [D17](#d17).
- `fn Type.name(...)` — static-функции и конструкторы. См. [D35](#d35).
- `fn Type @method(...)` — методы инстанса. См. [D35](#d35).
- `protocol` — структурный контракт (роль интерфейса/trait). См. [D9](#d9), [D42](#d42).
- `use Parent` внутри `type` — delegation, не наследование. См. [D39](#d39).

**Почему.** Наследование — известный антипаттерн (fragile base class,
diamond problem, божественные классы). Go доказал, что без классов жить
можно; Rust — что без GC и наследования можно писать сложные системы;
OCaml — что sum-types убивают половину паттернов GoF.

**Структурно, не номинально.** Любой тип со совпадающими методами
автоматически удовлетворяет `protocol`-контракту. Никаких `impl Trait for
Type` блоков (как в Rust). См. [D9](#d9), [D42](#d42).

**Замена наследования.** `use Parent` внутри `type` — delegation,
компилятор генерирует прокси-методы. См. [D39](#d39).

> **Историческая заметка.** В ранних черновиках использовались Rust-style
> `trait` / `impl Trait for Type`. Они были заменены на структурный
> `protocol` ([D9](#d9), [D42](#d42)) — без явного `impl`-блока, по
> совпадению методов. `paradigm.md` описывает старую модель и ожидает
> переписки.

## D2. Эффекты вместо ключевых слов `async`/`throws`/`unsafe`

**Решение.** Единая система эффектов в PascalCase как обычные типы:
`Throws[E]`, `Io`, `Async`, `Db`, `Net`, `Log`, `Alloc[R]` и т.п.
Эффекты выводятся компилятором, в публичных сигнатурах указываются
явно между списком параметров и `->`.

```nova
fn parse(s str) Throws -> int
fn fetch(url str) Net Async Throws -> Response
fn save(u User) Throws Db Log -> ()
```

**Почему.** В Java/Python/JS любая функция может бросить что угодно — это
не видно по сигнатуре. Checked exceptions Java получились плохо
(не комбинируются с дженериками и лямбдами). Go-стиль `if err != nil` —
много шума и легко забыть. Эффекты — средний путь: ошибка/IO/async видны
в типе, но синтаксис дешёвый.

> **Историческая заметка.** В первых черновиках имена эффектов были
> lowercase (`throws io async`) — пересмотрено в [D11](#d11) и [D30](#d30):
> эффекты — обычные типы, всегда PascalCase. `Throws` без параметров =
> `Throws[Error]`.

Подробнее — [effects.md](spec/effects.md), [D3](#d3), [D11](#d11), [D28](#d28).

## D3. Синтаксис эффектов: типы между `)` и `->`

**Решение.**

```nova
fn save(u User) Throws Io Async -> ()
fn fetch(url str) Net Async Throws -> Response
fn process(o Order) Db Log -> Receipt
```

Эффекты — обычные **типы в PascalCase**, перечислены через пробел между
закрывающей скобкой параметров `)` и стрелкой возврата `->`.

**Почему.**

1. Граница задана структурой (`)` слева, `->` справа), парсер однозначен.
2. Эффекты — это **типы** ([D2](#d2), [D11](#d11)), а не keyword'ы. То же
   PascalCase правило, что для всех типов в Nova ([D30](#d30)).
3. Читается слева направо как фраза.

> **Историческая заметка.** В черновике D3 эффекты были lowercase
> (`throws io async`), параметры с `:` (`u: User`). Современный синтаксис
> (PascalCase, без `:`) зафиксирован в [D11](#d11), [D17](#d17), [D30](#d30).

### Эволюция этого решения (важно)

Сначала был выбран синтаксис `!throws io async`:

```nova
fn save(u: User) !throws io async   // ОТВЕРГНУТО
```

Идея — `!` как «маркер эффектов», вдохновлено Swift force-unwrap (`x!`)
и Ruby/Scheme (мутирующие методы `sort!`).

**Почему отвергнуто (две независимые причины):**

**Причина 1 — `!` похож на `not`.**
В подавляющем большинстве языков `!` — логическое НЕ (`!x`, `!is_empty`).
Глаз на доли секунды читает `!throws` как «**не** throws» — ровно
противоположный смысл. Один зоркий глаз поломал дизайн в обсуждении —
для языка, который должен оптимизироваться под чтение, это плохо.

**Причина 2 — непоследовательность с несколькими эффектами.**
В `!throws io async` символ `!` стоит только перед первым эффектом,
а `io` и `async` идут «голыми». Возникает вопрос: где кончается список
эффектов? Открывающая скобка должна закрываться, иначе глаз не понимает
границы.

**Рассмотренные альтернативы и почему отвергнуты:**

- `!throws !io !async` — каждому эффекту свой `!`. Шумно, и проблема
  «`!` похож на not» остаётся.
- `!{throws, io, async}` — `!` + явный блок. Фигурные скобки заняты
  телом функции, путается.
- `~throws io async` — `~` нигде не значит «не», но неудобно набирать
  на части клавиатур, и визуально слабее слов.
- `<throws, io, async>` (Koka-style) — академически чисто, но угловые
  скобки нужны для дженериков, и читается тяжелее.
- Атрибуты `@throws @io @async` — четыре символа лишних, и `@`
  ассоциируется с метаданными, а не с типом.
- Без маркера, всё выводить молча — опасно, эффекты должны быть **видны
  на глаз** в публичном API.

**Главный урок.** Символьная пунктуация дёшева на одном месте и дорожает
экспоненциально с количеством мест. Слова масштабируются линейно.

## D4. `?` для пробрасывания ошибки

**Решение.** Оператор `?` после выражения — «если ошибка, верни её выше».
Работает только в функциях с эффектом `throws`.

**Почему.** Заимствовано из Rust/Swift, проверено годами использования.
Дешевле try/catch, безопаснее `if err != nil`.

## D5. Видимость: только `export` или приватно

**Решение.** Два уровня: `export` (экспортируется из модуля) или ничего
(приватно для модуля). Никаких `protected`, `internal`, `package-private`.

**Почему.** В Go видимость через CamelCase мешает называть вещи
естественно. В Java/C# четыре уровня — overkill, никто не помнит правила.
Два уровня — достаточно для 99% случаев, остальное решается на уровне
модулей.

### Эволюция: `pub` → `export`

В ранних черновиках использовалось ключевое слово `pub` (как в Rust).
Заменено на `export` потому что:

- **Симметрия с `import`** — `export X`/`import X` точные противоположности.
  Прецедент TS/JS, AI-friendly через естественный английский.
- **Освобождает `use`** для embed/delegation внутри `type` (см. paradigm).
  В Rust `pub` + `use` оба про модули; в Nova `use` уже занят embed'ом,
  и пара `pub`/`use` создавала бы конфликт смыслов.
- **AI-first.** Слово длиннее, но смысл прозрачнее без знания
  Rust-сокращений.

Цена — ~3 символа дополнительно перед декларацией. Приемлемо.

См. [D29](#d29-модули-и-импорты) для полного синтаксиса import/export.

## D6. Память: managed по умолчанию, regions opt-in для real-time

**Решение.** Современный concurrent GC по умолчанию. Программист пишет
код **не думая о памяти**, как в Go/Java. Циклы освобождаются автоматически.
Для real-time зон (звук, торговля, embedded) — эффект `Realtime` в
сигнатуре функции. Тело автоматически оборачивается в region (GC
внутри выключен), явный `region { ... }` нужен только для контроля
над несколькими аренами (см. ниже).

> **Историческая заметка.** В первой версии D6 региональный блок был
> **обязательно явным** — `Realtime` в сигнатуре + ручной `region { ... }`
> вокруг тела. Это дублировало контракт и нарушало AI-friendly принцип
> (LLM должна была угадывать, что нужен `region`). Пересмотрено:
> implicit region для тела по умолчанию, явный `region` только для
> сложных кейсов с несколькими аренами.

### Два уровня памяти

```
┌─────────────────────────────────────────────────────────┐
│ Managed heap (default)                                  │
│   - Concurrent GC (паузы <1ms)                          │
│   - Generational, non-moving для FFI                    │
│   - Escape analysis: что не утекает — на стеке/в арене  │
│   - Никаких префиксов в коде                            │
│   - Циклы освобождаются автоматически                   │
└─────────────────────────────────────────────────────────┘
              │
              │ opt-in для real-time
              ▼
┌─────────────────────────────────────────────────────────┐
│ region { ... } блок с эффектом Realtime                 │
│   - GC выключен внутри блока                            │
│   - Аллокации в арену, освобождение en-masse на выходе  │
│   - Гарантированно нет GC pauses                        │
│   - Для звука, торговли, embedded                       │
└─────────────────────────────────────────────────────────┘
```

**Escape analysis** делает большую часть работы для perf: значения,
не утекающие за пределы вызова, остаются **на стеке** или в **арене
вызова** — без аллокаций в managed heap, без GC pressure. Программист
пишет обычный код, компилятор сам решает.

Для случаев, где escape analysis не справляется (объект пересекает
границу fiber'а, сохраняется в долгоживущее место, возвращается из
функции), объект попадает в managed heap — и **это нормально для
99% случаев**. Современный GC справляется без видимых последствий
для backend-кода.

### Базовое использование — программист не пишет ничего особого

```nova
type Tree {
    value int
    children []Tree         // обычная ссылка, GC управляет
    parent Tree              // никаких ~weak — циклы освобождаются автоматом
}

let root = Tree { value: 1, children: [], parent: ... }
// ... работа с root
// освобождается автоматически когда становится недостижим
```

**Никаких `~T`, `~&T`, `~weak` префиксов.** Никаких compile-time ошибок
о циклах. Программа работает как в Go/Java/C# — пишешь логику, GC делает
остальное.

### Region — opt-in для real-time

Эффект `Realtime` объявляет «функция выполняется без GC pauses».
Компилятор проверяет: внутри realtime-кода запрещены аллокации в managed
heap, разрешены только region-allocations и стек.

**Implicit region для тела функции.** Если тело Realtime-функции **не
содержит** явного `region { ... }` блока верхнего уровня, компилятор
оборачивает всё тело в implicit region автоматически. Программисту не
нужно дублировать контракт `Realtime` ручным `region`-блоком:

```nova
// Простой случай — implicit region, программист не пишет region {}
fn map_audio(samples []f32, gain f32) Realtime -> []f32 =>
    samples.map(|x| x * gain)
```

Возвращаемое значение из такой функции автоматически копируется в
managed heap на границе функции (как `to_owned()` на выходе из явного
region — но компилятор делает это сам).

**Явный `region { ... }`** нужен, когда программисту требуется контроль:
несколько фаз с разными аренами, явное копирование промежуточных
результатов между ними, или вложенные функции, переиспользующие арену
вызывающего:

```nova
fn process_audio_block(samples []f32) Realtime -> []f32 => {
    let scratch = region {
        let buf = []f32.with_capacity(1024)
        // ... первая фаза, временные данные
        buf.to_owned()
    }
    region {
        // вторая фаза с другой ареной
        finalize(scratch)
    }
}
```

Правило: **если в теле верхнего уровня есть `region { ... }`, компилятор
не вмешивается** — верит, что программист управляет аренами явно. Если
явного `region` нет — обёртывает всё тело.

`region { ... }` — примитив языка, как `parallel for`/`race`/`with_timeout`
([D14](#d14-async-как-эффект--fiber-runtime)).

**Эволюция реализации:**

- **MVP (v0.5):** implicit region создаётся **всегда** для тела
  Realtime-функции без явного region. Стоимость — одна арена на вызов.
- **v0.7+:** escape analysis убирает арену там, где она не нужна
  (функция работает только на стеке, ничего не аллоцируется в куче).
- **v1.0+:** дальнейшие оптимизации — переиспользование арены
  вызывающего, стирание неиспользуемых регионов.

Это согласовано с принципом «программист пишет, компилятор делает» —
тот же путь, по которому идут escape analysis в Java HotSpot и Go.

### Современный GC — целевые характеристики

Конкретный движок — выбор реализации (как SMT-движок в [D24](#d24-стратегия-smt-проверки-контрактов)).
Дизайн фиксирует **класс**:

- **Concurrent** (параллельно с приложением) — паузы <1ms p99
- **Generational** (большинство объектов умирает молодыми)
- **Non-moving для FFI** или с pinning — указатели стабильны
- **Throughput overhead** — целевой ~5-10% (как ZGC, Shenandoah)
- **Memory overhead** — целевой ~1.5x

Кандидаты реализации: MMTk (фреймворк современных GC, используется
Java/Ruby/Julia), собственный concurrent collector, или адаптация
существующего. Выбор — на этапе реализации.

### Почему пересмотрели

Ранее [D6](#) и [D21](#d21-отменено-opt-in-cycle-collection-снято-в-пользу-managed-memory)
фиксировали «opt-in cycle collection» — программист выбирал `~T` или
`~&T` на уровне типа. Это давало real-time гарантии **по выбору
программиста**.

Пересмотр в пользу managed memory:

1. **Целевая ниша Nova — backend + AI-кодинг**, не embedded/real-time.
   Kubernetes, Docker, etcd, Prometheus, CockroachDB на Go доказали,
   что современный GC **не мешает** инфраструктуре интернета.
2. **AI-first**: LLM, читающая код, **не должна выбирать** `~T`/`~&T`/
   `~weak` для каждой структуры. Это трение, увеличивающее ошибки.
3. **Когнитивный налог** на программиста: ~80% случаев программист
   **не знает**, нужен ли real-time. Опт-ин по дефолту = угадывание.
4. **Прецедент антипаттерна**: Java/Swift/C++ сообщества жалуются на
   misuse weak-ссылок. Nova повторяла бы ту же ошибку.
5. **«Простота + огромные возможности»** как фишка языка: убрав
   префиксы памяти, упрощаем грамматику, освобождаем ментальный
   бюджет на effects/handlers/контракты.

### Что отменено

- **Префиксы `~T`, `~&T`, `~weak`** — нет в языке
- **`&T` / `mut &T` borrow** — нет в языке (см. ниже, отменено
  отдельным шагом дискуссии)
- **Cycle collector Bacon-Rajan** ([D21](#d21-отменено-opt-in-cycle-collection-снято-в-пользу-managed-memory)) —
  заменён на единый concurrent GC
- **Эффект `Alloc[Cycle]`** — снят, аллокации в managed heap не отдельный
  эффект
- **Compile-time анализ циклов через `~T`** — не нужен, GC справляется
- **Тип `Weak[T]` в stdlib** — НЕ вводится. Use cases решаются иначе:
  - Кеш с auto-cleanup → `Cache[K, V]` с TTL/LRU из stdlib
  - Observer pattern → handler-механизм Nova ([D10](#d10-революционная-ставка-всё--эффект--ai-first))
  - GC-cycle оптимизация → не нужна для backend

### Что сохранилось

- **Стек / escape analysis** — компилятор держит на стеке всё, что не
  утекает; для не-утекающих значений — без GC overhead
- **Регионы** — явная opt-in фича через `region { }` для real-time

### Почему отменили `&T` borrow

После пересмотра памяти на managed GC я (агент) предложил оставить
`&T` как «opt-in borrow для hot path». Пользователь спросил: «нам
это точно надо? на стеке принудительно нельзя передать?»

Перевзвесил. Аргументы:

1. **`&T` рефлекторно скопирован из Rust.** В Rust borrow нужен
   потому что нет GC — единственная альтернатива move/clone. В Nova
   с GC передача объекта = передача указателя, никакого move/clone не
   требуется.
2. **Escape analysis закрывает большинство perf-кейсов.** Не утекающие
   значения остаются на стеке без аллокаций. Это работает в Go, Java
   HotSpot, .NET — встроено в современные runtime.
3. **Slice уже передаётся эффективно.** `data []f64` — это
   `(ptr, len, cap)` структура, передача дешёвая. Не нужен отдельный
   `&[]T` borrow.
4. **Lifetime checker — research-уровень.** Стоит дорого реализовать,
   для прикладного языка с GC выгода низкая.
5. **Прецедент Go** — нет borrow, и язык успешно работает в backend-
   инфраструктуре.

Для real-time hot path остаётся `region { ... }` блок — это
**достаточный** escape hatch.

### Цена

1. **Потеря дифференциации.** «Opt-in cycle collection» был кандидатом
   в третью уникальную заявку Nova ([D9](#d9-честная-оценка-новизны)).
   Теперь Nova — «backend-язык с GC, как Go», что **слабее**, но
   честнее.
2. **Memory overhead ~1.5x** — цена GC.
3. **Tail-latency для p99.99** — современные concurrent GC дают
   pauses <1ms, для backend это **не проблема**. Если кто-то столкнётся
   с GC pauses на high-load (как Discord Read States) — решается через
   `region` для критичных частей или профилирование allocation patterns.

### Связь с другими решениями

- **Заменяет [D21](#d21-отменено-opt-in-cycle-collection-снято-в-пользу-managed-memory)** —
  D21 теперь помечен «отменено».
- **Согласовано с [D14](#d14-async-как-эффект--fiber-runtime)** — fiber
  stacks остаются как описано, GC работает поверх.
- **Согласовано с [D10](#d10-революционная-ставка-всё--эффект--ai-first)** —
  AI-first выигрывает: LLM пишет код без выбора префиксов.
- **Согласовано с принципом «простота + возможности»** — минимум
  концепций, escape hatch через `region` для real-time.

### Главный тезис о памяти

> **Программист пишет, GC работает.** В обычном коде нет ни одного
> префикса памяти. Escape analysis оставляет на стеке всё, что не
> утекает. `region { ... }` — opt-in для real-time зон. Циклы
> освобождаются автоматически. Современный concurrent GC даёт
> pauses <1ms.

## D7. Один язык — три режима компиляции

**Решение.** AOT (бинарь, как Go), JIT (как .NET), интерпретатор
(как Python). Один и тот же исходник.

**Почему.** Скрипт за 1 строку и сервер на 100k строк — это **разные
режимы запуска одного языка**, а не разные языки. Julia доказала, что
это работает.

## D9. Честная оценка новизны

**Решение.** Nova не изобретает ни одной фичи. Каждая идея уже где-то есть:
effect system — Koka, регионы — Zig/Odin, traits + sum-types — Rust,
структурная типизация — Go, три режима компиляции — Julia,
embed-delegation — Go, мономорфизация + `dyn` — Rust.

Единственная заявка на «такого нет нигде» — **сочетание** этих идей
в одном прикладном языке. Это слабая заявка, и её нужно держать
честно перед глазами, чтобы дизайн не превращался в «всего понемногу».

### Почему это слабая заявка

Языки-комбинаторы проваливаются чаще, чем выживают: Scala, D, Nim,
Crystal — все технически сильные, ни один не доминирует. Языки
выигрывают не фичами, а **экосистемой + killer use-case**:

- Go — серверы, потому что Google + Docker/Kubernetes
- Rust — системное программирование, потому что Mozilla + Linux kernel
- Python — ML, потому что NumPy/PyTorch
- TypeScript — фронт, потому что Microsoft + VS Code

У гипотетической Nova ни одного из этих рычагов нет.

### Две вещи, которые могли бы быть реально новыми

Если выбирать **одно предложение** на вопрос «зачем переходить с того,
что есть» — это:

**1. Effect system в прикладном языке с понятными ошибками.**
Koka/Effekt/Eff — академические, ими никто не пишет в продакшне.
Нет ни одного прикладного языка, где effects заменили
`async`/`throws`/`unsafe` одновременно и были бы понятны программисту
с Java-фоном за день. Решающая инженерная задача — **сообщения
компилятора про эффекты**. В Koka они академически точны и человечески
непонятны. Никто пока эту задачу не решил.

**2. Три режима компиляции (AOT/JIT/интерпретатор) в строго
типизированном языке без жертв.** Julia умеет JIT+AOT, но типизация
слабая. C# умеет AOT+JIT, но REPL ограниченный. Go умеет AOT, REPL
нет. TypeScript = только интерпретация. Nova могла бы быть **первым
строго типизированным языком, где `nova run script.nv` работает как
Python, а `nova build` даёт бинарь как Go**, без переписывания.

### Что **не** является заявкой

`traits + data`, регионы памяти, structured concurrency, sum-types,
borrow по запросу — это **повторение лучших практик**. Нужно, чтобы
язык вообще был хорошим, но **не отличает** Nova от Rust/Swift/Kotlin.
Заявлять «у нас sum-types» в 2026 — как заявлять «у нас замыкания»
в 2010. Базовый санитар.

### Главный вывод для дизайна

Если очередная идея не усиливает одну из двух уникальных заявок выше,
а просто «делает язык хорошим» — её добавлять можно, но **не надо
выдавать за отличие**. И до написания компилятора нужно ответить
письменно: «зачем мне переходить с X на Nova одним предложением».
Большинство языков-неудачников начали с компилятора, а не с этого
вопроса.

## D10. Революционная ставка: «всё — эффект» + AI-first

**Решение.** Nova переходит от «хорошего языка с набором фич» к языку
с **одной центральной абстракцией** и **одним killer use-case**.

**Центральная абстракция: всё — эффект.**
Не просто `throws/io/async`. **Любое нечистое вычисление — эффект**:
аллокация (`alloc`), мутация (`mut`), время (`time`), случайность
(`random`), сеть (`net`), БД (`db`), файлы (`fs`), параллелизм (`par`),
лог (`log`), и пользовательские эффекты, которые программист объявляет
сам. Эффекты алгебраичны (Koka/Effekt-style): у каждого есть **handler**
— функция, перехватывающая эффект и решающая, что с ним делать.

Из этой одной идеи следует всё:
- Тестирование без моков = подмена handler'а
- Транзакции = handler эффекта `db`
- Undo/redo = handler эффекта `mut`
- Детерминированный запуск = handler `time`+`random` с фиксированными значениями
- Трассировка = handler-обёртка над любым эффектом
- Capability security = разрешённые handler'ы в скоупе

**Killer use-case: AI-first язык.**
LLM пишет 50–80% кода в 2026+. Все существующие языки спроектированы
до этой эпохи, под человека. Nova явно оптимизирует под пару
«LLM пишет, человек ревьюит»:

- **Локальность контекста** — функция понятна без 10 файлов вокруг
  (нет неявных импортов, нет DI через рефлексию, нет хуков-невидимок)
- **Сигнатура = полное описание поведения** — типы + эффекты +
  pre/postconditions, читая сигнатуру знаешь всё
- **Ошибки компилятора как обучающий сигнал** — формат, по которому
  LLM фиксит код за одну итерацию, без человека-посредника
- **Стабильность синтаксиса** — никаких революций каждые 2 года, LLM
  учится на старых данных
- **Проверяемость по фрагменту** — типечекинг одной функции без
  всего проекта
- **Эффекты делают код безопасным для AI-генерации** — невозможно
  «случайно» сходить в сеть/БД, это видно в сигнатуре

**Почему это связано в одно.** Эффекты дают LLM то, чего у него нет
ни в одном языке: **полную видимость побочных действий по сигнатуре**.
Когда LLM пишет функцию, у него нет способа узнать, ходит ли вызываемый
код в сеть. В Nova это в типе. Это превращает «вероятностный код от
LLM» в «верифицируемый код от LLM» — и это **уникальная заявка**.

### Что выкидывается из прежнего дизайна

- «Хороший язык в духе Swift/Kotlin» как цель — заменено на конкретную
  ставку
- «Всего понемногу» (Go + Rust + OCaml + Julia) — теперь Nova следует
  из одной идеи, а не комбинирует пять
- Effects как «ещё одна фича рядом с traits» — теперь это центр

### Что остаётся

- Traits + data вместо классов (D1) — следствие «всё эффект»: trait =
  набор операций, операция может иметь эффект
- Managed memory с современным concurrent GC (D6) — программист пишет
  без префиксов памяти. Real-time зоны через `region { ... }` блок
  с эффектом `Realtime`
- Три режима компиляции (D7) — handler'ы перехватываются и в JIT,
  и в AOT, и в интерпретаторе одинаково
- Структурная типизация — без изменений

### Что добавляется

- **Handler-блоки** — синтаксис подмены эффектов (`with handler { ... }`)
- **Capability-режим** — функция декларирует, какие эффекты разрешены
  в её скоупе
- **Контракты** — `requires`/`ensures`/`invariant` как часть сигнатуры
- **Детерминированный режим тестирования** — встроенный handler-стек,
  фиксирующий время/случайность/IO

### Цена и риски

1. **Сложность реализации компилятора огромна.** Algebraic effects + их
   вывод — задача переднего края PL. Koka работает 10+ лет и всё ещё
   академический.
2. **Сложность понимания.** Хуже монад в 2010. Решается **только**
   качеством сообщений компилятора. Если ошибки про эффекты непонятны —
   язык мёртв.
3. **AI-first — ставка на тренд.** Если LLM-кодинг откатится назад
   (что маловероятно, но возможно) — half реальной заявки исчезает.
4. **Effects performance.** Algebraic effects дороже прямого кода
   (continuation capture). Нужна агрессивная оптимизация: статический
   handler-резолюшн, инлайнинг в чистый код. Это исследовательская
   задача.
5. **Девять из десяти таких попыток проваливаются.** Это нормальный риск
   революционного проекта. Альтернатива — гарантированный «ещё один Nim».

### Правило для дальнейшего дизайна

Каждая новая фича проходит два теста:
1. **Следует ли она из «всё — эффект»?** Если нет — обоснование, почему
   она нужна вне центральной идеи.
2. **Помогает ли она LLM писать корректный код?** Если нет — почему
   она важнее этого.

Фича, не проходящая ни один тест, **не добавляется**, даже если
«хорошая».

## D11. Имена эффектов и синтаксис `with`

**Решение.** Эффект — это **именованный тип**, не ключевое слово.
Имена эффектов пишутся с большой буквы (как все типы).
Синтаксис `with` принимает **либо имя handler'а, либо подмену
вида `EffectName = expr`** через запятую, и ровно **один** блок тела.

### Имена эффектов

```nova
type Logger {
    log(msg str) -> ()
}

type Db {
    query(sql str, args []any) -> []Row
    exec(sql str, args []any) -> ()
}
```

Имя эффекта — обычный идентификатор типа. В сигнатуре функции
эффекты перечисляются по имени:

```nova
fn fetch(url str) Net Time Throws -> Response => ...
```

Стандартный набор эффектов в stdlib:
`Throws[E]`, `Io`, `Net`, `Db`, `Fs`, `Time`, `Random`, `Mut`,
`Alloc[R]`, `Par`, `Log`, `Trace`, `Ask[T]`.

Все с большой буквы. Никакого специального правила «эффекты с
маленькой» — это путало в ранних черновиках.

### Синтаксис `with`

**Отвергнуто** (старый синтаксис из черновиков):

```nova
with handler Logger { log(msg) -> resume(println(msg)) } {
    process(42)
}
```

Два `{...}` блока подряд читаются плохо: непонятно, где кончается
тело handler'а и начинается тело `with`.

**Принято** — две формы:

**Форма 1: подмена через `EffectName = expr`** (основной случай —
тесты, переключение реализации):

```nova
with Logger = console_logger, Db = in_memory, Time = fixed(t0) {
    process_order(o)?
}
```

После `with` — список «эффект = handler-выражение» через запятую,
потом **один** `{ body }`. Парсер однозначен: запятые разделяют
подмены, `{` открывает тело.

**Форма 2: handler как обычное значение** (для сложных или
переиспользуемых handler'ов):

```nova
let audit = Logger {
    log(msg) -> { audit_db.write(msg); resume(()) }
}

with Logger = audit {
    critical_operation()?
}
```

`EffectName { ... }` — выражение-литерал, дающее значение типа
`Handler[EffectName]`. Симметрично record-литералам:

```nova
type User { id u64, name str }              // объявление record-типа
let alice = User { id: 1, name: "alice" }    // record-литерал

type Logger { log(msg str) -> () }          // объявление эффект-типа
let console = Logger { log(msg) => resume(println(msg)) }  // handler-литерал
```

Парсер различает record-литерал и handler-литерал по содержимому
`{...}` ([D18](#d18-унификация-effect--это-обычный-type)):

- `name: value` (двоеточие) → record-литерал
- `name(args) => body` (стрелка) → handler-литерал

Никакого префиксного слова не требуется. См. [D18](#d18-унификация-effect--это-обычный-type)
для полной семантики.

### Имя эффекта в позиции выражения = активный handler

В коде имя эффекта может появляться в **трёх позициях**, каждая
разрешается контекстом:

```nova
// 1. ПОЗИЦИЯ ТИПА (в сигнатуре, в generic-параметре)
fn process(o Order) Db -> Receipt = ...
//                  ^^ здесь Db — имя типа эффекта

// 2. ПОЗИЦИЯ ОПЕРАЦИИ (Db.X — обращение к операции)
Db.query("...", args)
// здесь Db.query — операция активного handler'а

// 3. ПОЗИЦИЯ ВЫРАЖЕНИЯ (одиночное Db в выражении)
let captured_db = Db
//                ^^ здесь Db — значение, текущий активный handler типа Handler[Db]
some_function(Db)
return Db
```

Парсер различает по позиции, никакой неоднозначности нет.

**Никакого `Db.current()` или подобного геттера не существует** —
просто `Db` в выражении. Это симметрично тому, как `User` в выражении
не нуждается в `User.current()`.

### Слово `handler` — НЕ зарезервировано

В ранних черновиках использовался синтаксис `handler EffectName { ... }`
с обязательным префиксом. Он **отвергнут**: имя эффекта + блок
операций уже однозначно говорит «это handler этого эффекта», префикс
лишний.

`handler` свободно для использования как обычный идентификатор —
имя переменной, поля, функции. Зарезервированных слов в Nova
держится минимум.

### Почему именно так

1. **Один блок тела** — нет визуальной путаницы.
2. **Несколько эффектов в одном `with`** — естественно и компактно
   для тестов.
3. **Handler — обычное значение**, не специальная синтаксическая
   форма, привязанная к `with`. Это упрощает композицию.
4. **Симметрия с record-литералами** — `Имя { ... }` для значений
   любых типов, без специальных префиксов.
5. **`with` остаётся примитивом языка**, а не сахаром над функцией —
   потому что он структурно влияет на стек handler'ов.

## D12. Effect erasure и dynamic effects

**Решение.** Статическая типизация эффектов — **дефолт**: очереди,
каналы, планировщики типизированы по эффектам функций, которые
они принимают. Для случаев разнородных задач, плагинов и сериализации
есть **явные** инструменты стирания и динамики.

### Три уровня

**Уровень 1 — статически типизированный планировщик** (дефолт):

```nova
let order_queue Queue[fn(OrderId) Db Log Throws -> ()]
```

Воркер этой очереди статически проверен. Лишний эффект не пройдёт.
Это правильный дефолт для типизированных пайплайнов.

**Уровень 2 — явное стирание через `erase[E]`**:

```nova
fn erase[E](task fn() E -> ()) E -> fn() -> () =>
    let captured = capture_handlers[E]()
    () => with captured { task() }

universal_queue.enqueue(erase(send_email_task))
universal_queue.enqueue(erase(cleanup_db_task))
```

Эффекты захвачены в момент `erase`, тип становится `fn() -> ()`,
очередь принимает разнородные задачи. Цена: handler'ы зашиты, если
они стали невалидными к моменту исполнения — это проблема программиста.

**Уровень 3 — динамические эффекты через `EffectSet` + `DynFn`**:

Runtime-структура `EffectSet`, тип `DynFn` для случаев, когда
эффекты задачи известны только в рантайме (плагины, сериализация
в БД). Используется редко, помечается явно.

### Что НЕ делается

- Стирание не автоматическое — иначе строгая типизация превращается
  в видимость (как Java generic erasure)
- Все очереди не делаются динамическими по умолчанию — потеряется
  главное свойство Nova (видимость эффектов в типе)
- Через границу процесса handler'ы не передаются. Эффекты на этой
  границе становятся протоколом (имена сервисов, типов сообщений) —
  это паттерн «commands + dispatcher», не часть системы эффектов.

### Открытые вопросы

- Конкретный синтаксис `capture_handlers[E]()`
- Семантика `EffectSet` в рантайме (теги типов? vtable?)
- Граничные случаи: эффект выходит за scope и handler уже невалиден

## D13. Panic vs эффекты: что НЕ является эффектом

**Решение.** Не каждое прерывание вычисления — эффект. **Ошибки
аппаратной/математической вычислимости** (деление на ноль, выход
за границы массива, переполнение, OOM, переполнение стека)
**не указываются в сигнатуре** функции. Они образуют общую категорию
`Panic` — runtime-сбоев, перехватываемых на границе.

### Граница

**Видимое (в сигнатуре):** эффекты, описывающие **намерение** —
`Net`, `Db`, `Time`, `Log`, `Throws[BusinessError]`, `Mut`.

**Универсальное (`Panic`, не в сигнатуре):** сбои, описывающие
**невозможность вычисления**:
- Деление на ноль
- Целочисленное переполнение (debug — panic, release — wrapping)
- Выход за границы массива
- Переполнение стека
- Out-of-memory

### Почему так

Если бы `Throws[DivByZero]` был обязателен, он бы появился в
**каждой второй сигнатуре** (любая функция со средним арифметическим,
дисперсией, делением). К нему присоединились бы `Throws[IntegerOverflow]`,
`Throws[ArrayBounds]`. Это **синдром Java checked exceptions** —
информативность сигнатуры исчезает, потому что эффекты везде.

Сознательный компромисс: **строгая теория эффектов уступает
читабельности** в зоне аппаратных сбоев.

### Перехват — на границе fiber'а runtime'ом

`panic` концептуально означает **смерть текущего fiber'а**, не процесса.
В синхронной программе без fiber'ов (CLI, скрипт) fiber один = процесс,
поэтому panic = exit. В серверной программе с fiber-runtime'ом:

- HTTP-handler — это fiber на запрос. Panic в нём = смерть fiber'а,
  runtime возвращает 500, остальные запросы продолжают работать.
- Worker очереди — fiber. Panic = задача упала, scheduler берёт
  следующую.
- Supervised group ([R9](spec/revolutionary.md)) — supervisor видит «fiber
  завершился panic'ом», рестартует по своей стратегии.

**Никакого `try_panic`/`catch` в коде.** Программист **не ловит** panic
в обычной функции — это работа runtime'а на границе fiber'а. Если
программист хочет управляемую ошибку — пишет `throw` + `Throws[E]`,
ловит обычным handler'ом.

```nova
fn handle_request(r Request) Db Log -> Response =>
    process(r)             // если panic — fiber умирает, runtime вернёт 500
                            // если throw — handler выше ловит обычно

fn server() Par Net Throws -> () =>
    supervised {
        spawn handle_requests()
        spawn periodic_cleanup()
    } strategy = one_for_one, max_restarts = 3
    // supervisor рестартует упавшие fiber'ы
```

Это **унификация двух уровней ошибок**:

- **`throw` + `Throws[E]`** — управляемая ошибка, видна в сигнатуре,
  перехватывается handler'ом в коде.
- **`panic`** — сбой fiber'а, перехват только runtime'ом на границе
  fiber'а. В сигнатуре не виден.

Третьего уровня нет. Никаких `try_panic { ... } catch p { ... }` или
`panic_boundary { ... } recover (p) => { ... }` в языке.

### Опция: строгий режим

Для критичного кода (медицина, финансы, авионика):

```nova
@strict_total
fn critical(...) -> Result =>
    // деление на ноль здесь — compile error
    // обязаны checked-операции: safe_div(a, b)?, arr.get(i)?
```

Превращает функцию в тотальную (всегда завершается). Цена — больше
кода, но для 1% случаев это окупается.

### Что НЕ Panic, а обычный эффект

- Бизнес-ошибки парсинга, валидации, аутентификации → `Throws[E]`
- Network failure, DB connection refused → `Throws[NetError]`,
  `Throws[DbError]` внутри эффекта `Net` / `Db`
- Любая ошибка, которую программа **намерена обрабатывать**, —
  это не Panic.

Принцип: **«обработать никак нельзя, надо умереть» → Panic;
«обработать можно и нужно» → Throws**.

## D14. Async как эффект + fiber runtime

**Решение.** `Async` — обычный эффект в сигнатуре. **Без `Future<T>`
в типе возврата.** Без `await` в коде. Цвет функции исчезает —
точнее, переезжает в эффекты, как `Throws`, `Net`, `Db`.

### Внешне

```nova
fn fetch(url str) Net Async -> Response => ...
fn handler(req Request) Net Async Db -> Response =>
    let user = fetch_user(req.id)            // никаких .await
    let posts = fetch_posts(user.id)         // никаких .await
    Response.json(posts)
```

Тип возврата `Response`, не `Future<Response>`. Код пишется как
синхронный.

### Внутри

Под `Async` живёт **fiber-based scheduler** (как в Go или OCaml 5).
Когда операция эффекта `Net` приостанавливается, fiber кладётся
в очередь ожидания, scheduler берёт другой fiber. Память —
сегментированный стек или cactus stack.

### Сравнение с Rust async

|  | Rust async | Nova |
|---|---|---|
| Цвет функции | да (`async fn`) | нет (но эффект `Async` в сигнатуре) |
| `await` нужен | да | нет |
| Тип возврата меняется | `Future<T>` | нет, остаётся `T` |
| Стоимость задачи | ~64 байта (state machine) | ~4–8 KB (fiber stack) |
| Cancellation | ручная (Drop) | structured (`cancel_scope`) |
| C interop blocking | нет проблем | требует `detach to OS thread` |
| Видимость в сигнатуре | есть | есть |

Nova ближе к **Erlang/Go** по runtime, к **Koka** по типам.
Платит **памятью** (fiber stacks) ради **простоты кода**
(невирусность).

### Структурные примитивы (не сводятся к эффекту)

`parallel for`, `race`, `select`, `cancel_scope`, `with_timeout` —
**примитивы языка**, не библиотечные функции. Управление
параллелизмом нельзя выразить только через эффект.

```nova
fn fetch_all(urls []str) Net Async -> []Response =>
    parallel for url in urls {
        fetch(url)
    }
```

### Признание ограничения главного тезиса

«Всё — эффект» — это **типовая модель**, не **runtime-модель**.
На уровне типов `Async` единообразен с `Net`/`Db`. На уровне runtime
async требует fiber-инфраструктуры, как memory regions требуют
allocator'а. Это **исключение, симметричное другим runtime-исключениям**,
а не дыра в концепции.

### Стоимость fiber'а

Каждый fiber — несколько килобайт минимум (растёт по необходимости).
Дороже Rust state machine, дешевле OS thread. **Миллион fiber'ов
на машину — норма** (как Erlang). Миллиард — нет, для таких задач
есть `Stream`/событийная модель.

### Открытые вопросы

- Реализация fiber stacks: segmented? cactus? on-demand growable?
- Дефолтный размер fiber stack
- C interop для синхронных C-вызовов (механизм `detach`)
- Promise как пользовательский тип (когда нужен отложенный результат
  как значение первого класса) — пишется как обычная структура,
  не магия компилятора

## D15. Интерфейсы как структурные типы

**Решение.** Отдельной концепции `interface`/`trait` в языке **нет**.
Их роль играет ключевое слово **`protocol`** ([D42](#d42-protocol-keyword-для-структурных-интерфейсов)).
`type` — для данных, `protocol` — для контрактов. Любой тип со
структурно совпадающими методами автоматически удовлетворяет protocol'у.

### Как это выглядит

```nova
// данные — record-тип
type User { id u64, name str }

// «интерфейс» — protocol с методами
protocol Printable {
    show() -> str
}

// никакого impl: метод просто определяется у типа (D35)
fn User @show() -> str => "User(${@name})"

// функция принимает «что угодно с методом show»
fn log_one(x Printable) Log -> () => Log.info(x.show())

log_one(my_user)   // ok, User автоматически совместим
```

Параметр функции может декларировать требования прямо в типе,
без отдельного имени:

```nova
fn log_one(x { show() -> str }) Log -> () =>
    Log.info(x.show())
```

### Что выкинуто

- `trait` / `interface` как ключевое слово
- `impl Trait for Type` блоки
- `[T: Trait]` bounds в дженериках
- `dyn Trait` vs `impl Trait`
- Ассоциированные типы
- Дефолтные методы
- Номинальная vs структурная типизация (всё структурно)
- Trait-наследование, specialization, HKT

### Что сохранено

- **Эффекты в полях-функциях** — часть сигнатуры, проверяются как обычно.
  Реализация не может привнести эффект сверх объявленного. Это и есть
  ключевое отличие Nova от Go: интерфейс жёстче, потому что эффекты —
  часть контракта.
- **Структурная совместимость** автоматическая, как в Go.
- **Дженерики** без bound'ов — требования описываются типом параметра.

### Цена

- **Нет имени для контракта**, кроме как через `type`-алиас. В IDE
  нельзя «найти всех, кто реализует X», как в Rust/Java.
- **Нет номинальности** — если очень нужна, через newtype-обёртку
  (паттерн, не фича).

### Почему так

1. Следует из принципа «не добавлять фичи без оправдания центральной
   идеей или AI-first». Rust-style traits ни тому, ни другому не служат.
2. Унификация: одна концепция «структурный тип» вместо двух
   («record» + «interface»). Меньше синтаксиса — проще для LLM.
3. Эффекты в сигнатурах методов делают структурный тип строже,
   чем Go interface — это уникальное свойство Nova, которое
   нельзя получить через простое заимствование Go.

## D16. Дженерики через `[T]`, не `<T>`

**Решение.** Параметры типа записываются в **квадратных скобках**:

```nova
fn sort[T](xs []T, less fn(T, T) -> bool) -> []T
type Option[T] = Some(T), None
type HashMap[K, V] = ...
let parsed = parse[int]("42")?
```

**Уточнение.** `[T]` — это **generic-применение** (после имени типа или
функции: `Имя[T]`, `Option[T]`, `HashMap[K, V]`). Само по себе `[T]` не
является типом массива — для массивов используется `[]T` (см.
[D27](#d27-синтаксис-массивов-t-префикс-nt-для-фиксированных)).

Это разделение делает грамматику однозначной:
- `Имя[T]` — generic-применение к именованному типу/функции
- `[]T`, `[N]T` — конструкция нового типа (массив)
- `arr[i]` — индексация (доступ к элементу)

### Отвергнуто: `<T>` (Rust/TS/Java/C# стиль)

`<T>` создаёт **парсер-неоднозначность**:

```
sort<int>(xs)   // дженерик-вызов? или (sort < int) > (xs)?
```

Это известная проблема C++/Java/TS/Rust. Решения, каждое со своей
ценой:
- **Турбофиш** `::<>` (Rust) — синтаксический workaround, признанный
  самим Rust-сообществом проектной ошибкой
- **Контекстный парсинг с backtracking** (TS, C#) — медленнее, ошибки
  непонятнее
- **Запрет некоторых форм** (Rust ограничивает лямбды) — сужает
  выразительность

`>>` в `HashMap<str, Vec<int>>` исторически парсился как сдвиг —
требуются ad-hoc правила парсера. Java/C# годами требовали пробел
между `>` и `>`.

### Почему `[T]` лучше

1. **Парсер однозначен** — `name[T]` после имени функции/типа всегда
   дженерик, никаких backtracking.
2. **Турбофиш не нужен** — `parse[int]("42")` работает напрямую.
3. **Сообщения компилятора чище** — нет режима «ожидался тип, нашли
   выражение».
4. **Скорость компиляции** — важно для AI-first языка, где LLM
   прогоняет компиляцию много раз.
5. **Прецедент** — Go и Scala 3 пришли к тому же решению по тем же
   причинам.

### Цена

Менее знакомо программистам с фоном Rust/TS/Java/C#. Но Go доказал,
что привыкнуть к `[T]` — вопрос недели.

LLM знают `<T>` лучше из обучающих данных. Это слабый минус, но
**однозначный синтаксис** компенсирует — модель делает меньше ошибок
парсинга, которые ловятся компилятором.

### Что не пересматриваем

- Этот выбор — final для v1.0.
- Стабильность синтаксиса (D10) запрещает его менять после релиза.

## D17. Объявление типов: единый синтаксис без `|`

**Решение.** Убираем `|` как разделитель вариантов sum-type, заменяем
запятой. Убираем `=` для record и позиционных структур — `=` остаётся
только для алиасов и sum-types.

### Полный синтаксис объявления типа

```nova
// алиас
type UserId = u64

// record (именованные поля)
type User { id u64, name str }

// позиционная структура
type Point(f64, f64)

// unit-тип (без полей)
type Empty

// sum-type
type Color = Red, Green, Blue

type Shape =
    Circle { radius f64 },
    Square { side f64 },
    Triangle { a f64, b f64, c f64 }

type Result[T, E] = Ok(T), Err(E)
```

### Правила парсинга

| После `type Имя` идёт | Что это |
|---|---|
| `{ ... }` | record-структура |
| `( ... )` | позиционная структура |
| ничего | unit-тип |
| `=` потом тип | алиас |
| `=` потом список вариантов через запятую | sum-type |

Парсер однозначен по первому символу/токену после имени.

### Что отвергнуто

**Старый ML-style синтаксис:**

```nova
type Shape =
    | Circle { radius f64 }       // ОТВЕРГНУТО
    | Square { side f64 }
```

Причины:
- `= |` — два разделителя подряд, дублирование
- `|` начинающий каждую строку чужд для языков не из ML-семейства
- В Nova `,` уже стандартный разделитель списков (параметры, поля
  записи, элементы массива) — добавлять второй разделитель только
  для sum-types нелогично

**Старый дублированный синтаксис для одно-вариантного sum-type:**

```nova
type Point = | Point(f64, f64)     // ОТВЕРГНУТО — дубль
```

Решение: «sum-type с одним вариантом» и «структура» — это одно и то же.
Пишем просто `type Point(f64, f64)`.

**Старый `=` для record:**

```nova
type User = { id u64, name str }   // ОТВЕРГНУТО — `=` лишний
```

`=` означает «справа выражение типа». Когда справа форма данных
(`{...}` или `(...)`) — `=` не нужен.

### Принцип

**Один разделитель списка на весь язык — запятая.** Параметры,
элементы массивов, поля записи, варианты sum-type — везде `,`.
Меньше правил для запоминания, меньше ошибок LLM.

### Цена

- Расхождение с ML-семейством (OCaml/Haskell/Rust/F#) — там `|`.
  Программисту с этим фоном понадобится привыкнуть.
- Менее «функциональный» эстетический вид. Это сознательный выбор
  в пользу согласованности с остальным языком.

### Создание значений и pattern matching — не меняется

```nova
let p = Point(1.0, 2.0)
let u = User { id: 1, name: "alice" }
let c = Circle { radius: 5.0 }
let s = Active

match shape {
    Circle { radius }    => 3.14159 * radius * radius
    Square { side }      => side * side
    Triangle { a, b, c } => heron(a, b, c)
}

match result {
    Ok(value)  => value
    Err(error) => default
}
```

### Shorthand для record-литералов (field punning)

Когда **имя поля совпадает с именем переменной в скоупе**, можно
писать имя один раз:

```nova
let key = "alice"
let value = 42

// длинная форма
let entry = Entry { key: key, value: value }

// shorthand — эквивалент
let entry = Entry { key, value }

// можно смешивать с обычными полями
let entry = Entry { key, value, extra: "data" }
```

**Грамматика record-поля в литерале:**

```
record-field := identifier ":" expr      // полная форма
              | identifier                // shorthand (имя поля = имя в scope)
```

Парсер однозначен:
- `name:` → полная форма с выражением справа
- `name,` или `name}` → shorthand, ищет `name` в scope

Если переменная с таким именем не в scope — compile error:

```
error: shorthand field `key` not in scope
   suggestion: write `key: <expression>` or bring `key` into scope first
```

**Симметрия с pattern matching.** Деструктуризация `Circle { radius }`
извлекает поле `radius` в одноимённую локальную переменную;
конструирование `Circle { radius }` делает наоборот — берёт
переменную `radius` из scope в поле `radius`. Одна и та же запись
работает в обе стороны.

**Прецедент:** Rust 1.17+ (`Point { x, y }`), JS/TS ES6 (`{ x, y }`),
OCaml record punning, Haskell `NamedFieldPuns`. **AI-first аргумент:**
LLM меньше ошибается при переименовании (`oldName: oldName` → `newName:
newName`, забыл одну сторону → bug; с shorthand `oldName` → `newName`
— переименование одно).

### Partial pattern matching — частичная деструктуризация

При деструктуризации в `match` или `if let` можно указать **только
нужные поля** — остальные игнорируются. Две формы, обе валидны:

**Явная** — с `..` маркером:

```nova
match @buckets[idx] {
    Occupied { value, .. } => Some(value)    // явный .. — у Occupied есть ещё поля
    _                      => None
}
```

`..` сигнализирует «здесь ещё что-то есть, мне оно не нужно».
Visual cue.

**Неявная** — без маркера:

```nova
match @buckets[idx] {
    Occupied { value } => Some(value)        // остальные поля автоматически игнорируются
    _                  => None
}
```

Короче, прецедент JS/TS, Haskell `RecordWildCards`.

**Обе формы эквивалентны.** Программист выбирает по контексту:

- **Явная (`..`)** — когда хочется **явный сигнал**, что у типа есть
  ещё поля. Полезно при ревью, при чтении кода после рефакторинга.
- **Неявная** — когда **очевидно**, что других полей много (DTO,
  database row), и упоминать их шум. Краткость.

### Что НЕ pattern-shorthand

**Переименование при деструктуризации** — оставляем явное:

```nova
Occupied { key: k, value }    // key переименовано в k, value совпадает
```

Это другая семантика — программист **специально** хочет другое имя.
Без `:` парсер не отличит от shorthand.

**Construction всегда требует все обязательные поля:**

```nova
let entry = Entry { key }                  // ОШИБКА: value не указано
let entry = Entry { key, value: 0 }        // ок — все поля заполнены
```

`..` для construction (как Rust `Entry { key, ..default }`) —
**отдельный синтаксис**, не зафиксирован в текущем D17. Возможно
дойдём до него позже.

## D18. Унификация: `effect` — это обычный `type`

**Решение.** Ключевое слово `effect` отменяется. Эффекты объявляются
через обычный `type` — компилятор различает «эффект» и «структурный
интерфейс» **по контексту использования**, не по форме объявления.

### Что было

```nova
effect Db {                              // ОТМЕНЕНО
    query(sql str, args []any) -> []Row
    exec(sql str, args []any) -> ()
}

type Printable {
    show() -> str
}
```

Два разных способа объявить «именованный тип с операциями».

### Что стало

```nova
type Db {                                // унифицировано
    query(sql str, args []any) -> []Row
    exec(sql str, args []any) -> ()
}

type Printable {
    show() -> str
}
```

**Один синтаксис** для всех именованных типов с операциями.

### Различение по контексту использования

Один и тот же тип может работать в двух ролях:

```nova
type Logger { log(msg str) -> () }

// А: позиция эффекта — между ) и ->. Активный handler берётся из скоупа.
fn process_a(o Order) Logger -> () =>
    Logger.log("processing")

// Б: позиция типа значения — обычный параметр, передаётся явно.
fn process_b(o Order, logger Logger) -> () =>
    logger.log("processing")
```

Программист выбирает стиль:
- **Эффект** (А) — для пронизывающих контекстов: БД, лог, аутентификация,
  трассировка. Не таскается через 10 функций.
- **Параметр** (Б) — для явных зависимостей одной функции, когда хочется
  локальной видимости.

### Что осталось без изменений

- **Имя типа в позиции эффекта** (между `)` и `->`) — требование
  активного handler'а в скоупе.
- **`Db.operation(args)`** — вызов операции активного handler'а.
- **`Db` в позиции выражения** — активный handler как значение.
- **`with Db = expr { body }`** — подмена handler'а в скоупе.
- **Литерал handler'а** — `Db { query(...) -> ..., exec(...) -> ... }`.

### Различение литералов парсером

Содержимое `{...}` решает однозначно:
- `name: value` (двоеточие) → record-литерал (`User { id: 1 }`)
- `name(args) -> body` (стрелка) → handler-литерал
  (`Db { query(s, a) -> resume(...) }`)

### Цена

1. **Семантическая зависимость в парсинге.** Раньше `effect`
   снимало неоднозначность сразу. Теперь парсер смотрит на содержимое
   `{...}`. Это работает, но добавляет одно правило в грамматику.
2. **Тип может «случайно» использоваться как эффект.** Если у
   функции в сигнатуре стоит `Logger`, и в скоупе нет активного
   handler'а — ошибка компиляции «эффект Logger не предоставлен».
   Это нормально, но раньше было ясно, что Logger — эффект, по
   слову `effect`. Теперь — по сигнатуре функции.
3. **«Эффект» как термин остаётся в документации**, но в коде
   нет соответствующего ключевого слова. Это отражает реальность:
   эффект — это **роль**, не сущность.

### Что это даёт

- Одна концепция «именованный тип с операциями» вместо двух
- Меньше ключевых слов
- Гибкость: один тип — две роли (эффект или параметр) по выбору
- Согласованность с принципом «не плодить специальные сущности»

## D19. Match-arms через `=>`, не `->`

**Решение.** В `match`-выражениях разделитель «образец → результат» —
**`=>`**, не `->`. `->` остаётся **только** для типов и сигнатур.

### Разделение ролей

- **`->`** — декларативно, типы и сигнатуры:
  ```nova
  fn f(x int) -> int                    // тип возврата
  type Handler = (Request) -> Response  // функциональный тип
  fn map[T,U](xs []T, f T -> U) -> []U  // функциональный тип параметра
  ```

- **`=>`** — вычислительно, тело функции и выражения с разветвлением:
  ```nova
  fn double(x int) -> int => x * 2       // тело именованной функции
  let inc = (x) => x + 1                  // лямбда (тело анонимной функции)
  match shape {                           // match-arm
      Circle { r } => 3.14 * r * r
      Square { s } => s * s
  }
  ```

### Семантическая унификация тел функций и match-arm

`params => body`, `pattern => result` — везде одна семантика: «дай мне
это, я отдам тебе то». Тело именованной функции, лямбда, match-arm —
три формы одной идеи, один разделитель.

См. [D22](#d22-анонимная-функция-только-лямбда--тело-именованной--тоже-),
который распространяет `=>` на тело именованной функции и убирает
анонимную форму `fn`.

Прецедент: C#, F#, Scala 3 унифицируют `=>` для лямбд и match-arms.
Nova идёт дальше, унифицируя и тело именованной функции тоже.

### Что отвергнуто

- **`->` (Rust/OCaml/Haskell стиль)** — отвергнуто из-за перегрузки
  символа: тот же `->` означает тип возврата функции. Унификация
  «слева вход, справа выход» теоретически красивая, но практически
  путает: одно и то же символьное обозначение для типов и для
  выражений-разветвлений.
- **`:` (Python стиль)** — отвергнуто из-за конфликта с другими
  использованиями `:`. В Nova `:` остался только в record-литералах
  как разделитель «ключ-значение» (D11/D17).
- **`then`** — отвергнуто, лишнее ключевое слово ради того же,
  что даёт `=>`.

### Цена

- Расхождение с Rust/OCaml/Haskell, где в match `->`/`=>` исторически
  смешивались. Но Rust сам использует `=>` в современном `match`.
- Все примеры с `match ... ->` в документах нужно обновить на `=>`.

## D20. `()` вместо `void` + сводка стрелок

**Решение.** Тип «без значения» — это `()` (unit), не `void`. И сводная
таблица всех стрелочных символов в языке, фиксирующая роли каждого.

### Unit (`()`) — обычный тип

```nova
fn cleanup() Io -> ()                  // явно
fn cleanup() Io                        // -> () можно опустить
let xs [()] = [(), (), ()]             // unit как элемент массива
let r Result[(), str] = Ok(())         // unit как generic-параметр
fn run_all[T](tasks []fn() -> T) -> []T = ...
run_all([cleanup, log_event])          // T = (), всё работает
```

### Что отвергнуто: `void`

`void` в C/C++/Java/C#/JS — это **особый случай**, не нормальный тип:
- Не может быть generic-параметром (`Result<void, E>` не работает)
- Не может быть элементом массива
- Требует обходных путей (`Void` в Java/C#, специальные правила
  для функций без возврата)

Это **дыра в системе типов**, известная боль во всех языках, где
выбрали `void`. В Rust/Haskell/OCaml/Scala/Swift/Kotlin — `()` или
`Unit` как нормальный тип, дыры нет.

Цена `()`: непривычно для C-программистов. Решение: `-> ()` опционально
опускается, и в большинстве кода вопрос не возникает.

### Сводная таблица стрелочных символов

| Символ | Роль | Пример |
|---|---|---|
| `->` | тип возврата функции, функциональный тип | `fn f(x int) -> int`, `T -> U`, `(A, B) -> C` |
| `=>` | тело функции (именованной или анонимной), match-arm | `fn f() => body`, `(x) => x + 1`, `0 => "zero"` |
| `=` | присваивание | `let x = 5` |

Каждый символ имеет **одну роль**.

### Принцип разделения `->` и `=>`

- **`->` декларативно** — описывает форму типа, сигнатуру.
  Артефакт описания, а не исполнения.
- **`=>` вычислительно** — тело функции и match-arm, исполняемый код.
  «Дай мне это, я отдам тебе то.»

Когда в одной строке встречаются оба — видна граница «здесь описание,
а здесь код»:

```nova
fn map[T, U](xs []T, f T -> U) -> []U => [f(x) for x in xs]
//                       ↑↑          ↑↑   ↑↑
//                       тип параметра     тип возврата
//                                          тело
let doubled = map([1, 2, 3], (x) => x * 2)
//                                ↑↑
//                                лямбда, тело тоже через =>
```

### Почему не унификация (везде один символ)

Рассмотрены и отвергнуты:

- **Везде `->`** (Rust/OCaml/Haskell стиль) — символ перегружен,
  одно обозначение для четырёх разных вещей (тип возврата, тип
  функции, тело функции, match-arm). Глаз не видит границы.
- **Везде `=>`** (Scala 3 стиль) — та же проблема в обратную сторону.
- **Любой третий символ** (`~>`, `:>`) — экзотика без выигрыша.

Текущий выбор — **двухсимвольное разделение** — решение C# / Java 17 /
Scala 3 (для типов методов) / Kotlin. Прецедент массовый, AI-friendly,
визуально однозначный.

### Сигнатура функции — почему `=>`, а не `=`

```nova
fn double(x int) -> int => x * 2
```

Тело именованной функции отделяется через `=>`, как и тело лямбды.
Это унификация: «функция → тело» одной стрелкой, независимо от того,
есть у функции имя или нет. См. [D22](#d22-анонимная-функция-только-лямбда--тело-именованной--тоже-).

`=` остаётся **только** для присваивания (`let x = 5`).

### Что отменяет финальность

В предыдущей версии этот пункт был помечен «final для v1.0». [D22](#d22-анонимная-функция-только-лямбда--тело-именованной--тоже-)
пересматривает таблицу стрелок: `=` исключается из ролей «тело функции»,
`=>` принимает её. «Финальность» по правилу D10 относится к
**стабильности после релиза**, не к фиксации в дизайн-стадии. Пересмотр
до релиза устраняет внутреннее противоречие — два синтаксиса для одной
концепции — и поэтому оправдан.

## D21. ОТМЕНЕНО — Opt-in cycle collection снято в пользу managed memory

**Статус: отменено.** Заменено новой версией [D6](#d6-память-managed-по-умолчанию-regions-opt-in-для-real-time)
«managed memory с современным GC + regions opt-in».

### Почему отменено

Исходная идея D21 — программист **сам выбирает** на уровне типа поля,
нужен real-time (`~T`) или удобство (`~&T`). Это давало уникальную
заявку: real-time гарантии без глобального GC.

**Пересмотр в пользу managed memory** (см. discussion-log этап 13):

1. **Целевая ниша Nova — backend + AI-кодинг**, не embedded/real-time.
   Go доказал, что современный concurrent GC не мешает Kubernetes,
   etcd, CockroachDB и доминирующей части инфраструктуры интернета.
2. **AI-first проигрывает** от выбора `~T`/`~&T`/`~weak` для каждой
   структуры — LLM рефлекторно копирует паттерны без понимания.
3. **Когнитивный налог** на программиста: ~80% случаев не знает, нужен
   ли real-time. Опт-ин по дефолту = угадывание.
4. **Прецедент антипаттерна**: Java/Swift/C++ сообщества жалуются на
   misuse weak-ссылок. Nova повторяла бы ту же ошибку.

### Что сохранилось из D21

- Real-time зоны возможны через `region { ... }` блок ([D6](#d6-память-managed-по-умолчанию-regions-opt-in-для-real-time))
  с эффектом `Realtime`
- Borrow `&T` для hot path

### Что отменено полностью

- Префиксы `~T`, `~&T`, `~weak`
- Cycle collector Bacon-Rajan (заменён на единый concurrent GC)
- Эффект `Alloc[Cycle]`
- Compile-time анализ циклов через `~T`
- Тип `Weak[T]` в stdlib не вводится

### Историческое тело D21 (для контекста)

Дальнейший текст этого раздела сохранён как **исторический** —
показывает, как мы пришли к решению и почему пересмотрели. Не
актуален для текущего дизайна Nova.

---

**~~Решение.~~** В Nova два префикса для shared ownership: `~T` и `~&T`.
Cycle collector работает **только** над `~&T` объектами. Real-time
зоны программы используют `~T` (без collector'а), удобные графовые
структуры — `~&T` (с collector'ом).

### Префиксы памяти

| Префикс | Семантика | Real-time | Циклы |
|---|---|---|---|
| `T` | стек или арена (escape analysis) | да | n/a |
| `&T` | borrow с lifetime-проверкой | да | n/a |
| `~T` | Rc, циклы запрещены | да | через `~weak` |
| `~&T` | Tracing Rc, цикл collector | **нет** | автоматически |
| `~weak T` | слабая ссылка (без счётчика) | да | разрывает цикл |

### Главный принцип

**Cycle collector — opt-in.** Программист выбирает на уровне типа поля,
нужен ли ему collector:

```nova
// Real-time дерево: parent через ~weak, без collector
type Tree {
    children [~Tree]
    parent   ~weak Tree
}

// Граф с произвольными циклами: ~& включает collector
type Graph {
    nodes [~&Node]
}
type Node {
    edges [~&Node]
}
```

В первом случае никакой GC не работает. Во втором — collector сканирует
`~&` подграф периодически.

### Real-time гарантия

Если в зоне кода нет `~&T` объектов — **collector никогда не запускается**.
Звуковой движок, торговый поток, embedded — могут полностью избежать
collector'а, оставаясь real-time.

Если другая часть программы (UI, графовые алгоритмы) использует `~&T` —
там collector работает в фоне инкрементально, не затрагивая real-time
зоны.

### Алгоритм

**Bacon-Rajan concurrent cycle collection** (IBM, 2001):
- Каждый `~&T` имеет «цвет» (white/grey/black/purple)
- При уменьшении счётчика, не дошедшем до 0, объект помечается «возможный
  корень цикла»
- Collector периодически (или по triggered'у) сканирует возможные корни,
  ищет циклы, освобождает
- Работа — incrementally, без stop-the-world

Это не требует переноса памяти (как у moving GC) — указатели стабильны,
C-interop работает.

### Эффект `Alloc[Cycle]`

Аллокация `~&T` — это эффект `Alloc[Cycle]`:

```nova
fn build_graph(data Input) Alloc[Cycle] -> ~&Graph = ...
```

Это видно в типе функции, согласовано с принципом D10 «runtime-стоимость
видна в типах». Программист знает, что функция использует «дорогую»
память, и может избежать её в hot path.

### Пересечение `~T` и `~&T`

`~T` объект может содержать `~&T` поле:

```nova
type App {
    config ~Config             // обычный Rc
    graph ~&Graph              // collector-managed
}
```

Collector сканирует **корни** во всех `~T` объектах, находит переходы на
`~&T`. Каждый `~T` имеет небольшой metadata header (~8 байт), позволяющий
collector'у обнаруживать `~&T` ссылки.

Цена: ~8 байт на каждый `~T` объект. Это та же цена, что за Rc-счётчик в
Rust/Swift, плюс один указатель.

### Lint и detection циклов в `~T`

**Compile-time:** компилятор анализирует граф типов. Если возможен цикл
через `~T` без `~weak` — выдаёт ошибку:

```
error: cycle possible in `~T` references between Node and Edge
   suggestion: use `~weak` for one direction, or `~&T` to enable cycle collection
```

**Debug-режим:** runtime detector циклов в `~T` (как ASAN). Найдёт
утечки на этапе тестирования.

**Release-режим:** циклы в `~T` молча текут (как в Rust/Swift). Это
ответственность программиста — но для этого есть compile-time + debug
detection выше.

### Стандартная библиотека

| Тип | Префикс | Причина |
|---|---|---|
| `String`, `Vec`, `HashMap` | без префикса | без ссылок |
| `LinkedList` | `~T` | head/tail модель, циклов нет |
| `Tree` | `~T` + `~weak` для parent | дерево, без циклов |
| `Graph` | `~&T` | возможны произвольные циклы |
| `DOM` | `~&T` | parent ↔ children + listeners |
| `Closure` | захват по умолчанию `~weak` | избегает «retain cycles» как в Swift |

Программист использует stdlib-структуры — не думает о выборе. Свои
структуры — выбирает осознанно.

### Что выбрано отвергая

**Глобальный mark-and-sweep GC** (Java/Go) — отвергнут:
- Stop-the-world паузы
- 1.5-3x overhead памяти
- Real-time невозможен
- Противоречит главному тезису Nova

**Только `~T` + явный `~weak`** (Rust, Swift) — отвергнут:
- Программист обязан думать о циклах в каждой структуре
- Графовые структуры неудобны
- В release циклы текут молча, без помощи

**Автоматическое определение слабых ссылок** — отвергнуто:
- В общем случае невозможно (графы не имеют корня, DLL симметричен)
- Компилятор может ошибиться, программа упадёт при разыменовании

### Сравнение со школой

| Подход | Real-time | Циклы автоматически | Реальный язык |
|---|---|---|---|
| Только Rc + weak | да | нет | Swift, Rust |
| Глобальный GC | нет | да | Java, Go, C# |
| Cycle collector над всем | нет | да | CPython |
| **Opt-in cycles (Nova)** | **да в `~T` зонах** | **да в `~&T` зонах** | **Inko, Midori** |

Nova занимает **уникальную** позицию: real-time гарантии **по выбору
программиста** на уровне типа поля.

### Главный тезис обновляется

Раньше: «Nova не имеет GC».

Теперь: **«Nova имеет opt-in cycle collection. По умолчанию real-time.
Программист выбирает удобство там, где это окупается, через `~&T` тип.
Real-time гарантии остаются доступны для зон без `~&T`.»**

### Цена

1. **Два префикса вместо одного** — программист решает, какой использовать
2. **Усложнение runtime** — Bacon-Rajan collector
3. **Усложнение compile-time анализа** — различение `~T` и `~&T` графов,
   проверка циклов
4. **Метаданные** — ~8 байт на каждый `~T` объект для cross-cutting
   detection

Это обоснованная цена за **полную гибкость**: программист не выбирает
между real-time и удобством на уровне всей программы. Выбирает
**на уровне каждой структуры данных**.

### Финальность

Это решение пересматривает D6. Это **большое** архитектурное изменение,
оправданное запросом «и циклы автоматические, и real-time возможен».
Дальнейшие пересмотры памяти не предусмотрены до v1.0.

## D22. Анонимная функция: только лямбда `=>`. Тело именованной — тоже `=>`

**Решение.** В Nova **нет анонимной формы `fn`**. Любая функция без имени
записывается как лямбда `(params) Effects? -> Type? => body`. Тело
именованной функции тоже отделяется через `=>`, не через `=`. `=` остаётся
исключительно для присваивания (`let x = 5`).

### Что было (отменено)

В ранних примерах ([decisions.md D12](#d12-effect-erasure-и-dynamic-effects)
и [examples/audit.nv](examples/audit.nv)) встречалась анонимная `fn`:

```nova
// ОТМЕНЕНО — два синтаксиса для одной концепции
fn audit_middleware(next Handler) -> Handler =
    fn(req Request) Db Log Throws -> Response = {
        ...
    }

// ОТМЕНЕНО — fn() как анонимная функция-выражение
let captured = fn() -> () = with captured { task() }
```

Получалось два эквивалентных способа создать функциональное значение:
лямбда `(x) => x + 1` и анонимная `fn(x) -> int = x + 1`. Они различались
только наличием явных типов и эффектов, не семантически.

### Что стало

**Анонимная функция = всегда лямбда.** Эффекты и типы опциональны:

```nova
let inc = (x) => x + 1
let parse = (s str) => s.parse[int]()?
let middleware = (req Request) Db Log Throws -> Response =>
    Db.exec("INSERT INTO audit ...")?
    Response.ok()
```

**Тело именованной функции — через `=>`:**

```nova
fn double(x int) -> int => x * 2
fn audit_middleware(next Handler) -> Handler =>
    (req Request) Db Log Throws -> Response => {
        if req.method == "GET" { return next(req) }
        ...
    }

```

### Почему `=>` для тел `fn`, а не `=` (как было)

Было обоснование (D20): «`=` короче, согласован с `let x = 5`».
**Это слабый аргумент.** Реальная цена — две стрелки для одной концепции
«тело функции». LLM и человеку приходилось помнить:

- именованная функция → `=`
- лямбда → `=>`

Семантически это **одна вещь** — связывание параметров с телом. Разделение
по «есть ли имя» не помогает читателю, оно историческая случайность.

Унификация на `=>`:

- **Один символ для тела любой функции.** Именованная или анонимная — `=>`.
- **`=` имеет одну роль** — присваивание значения существующему имени
  через `let`.
- **Параллель с match-arm укрепляется**: `pattern => result`,
  `params => body` — везде «дай мне это, я отдам тебе то».

Цена — теряется внешняя параллель `let x = 5` ↔ `fn f() = body`. Но эта
параллель работала только синтаксически: семантически `let x = 5` —
это «связать имя с уже вычисленным значением», а `fn f() = body` —
это «объявить новое имя и описать его поведение», не вычисление в момент
объявления. Параллель была обманчивой.

### Что отвергнуто

- **Оставить два синтаксиса как есть** — нарушает «один символ — одна
  роль» (D19/D20).
- **Унификация на `=`** (везде `=` для тел) — ломает match-arm, путает
  с присваиванием в guard'ах (`n if n = 0 = ...`).
- **Анонимная `fn` для случаев с эффектами** — провоцирует случай-зависимый
  выбор синтаксиса, не упрощает.

### Обновлённая сводка стрелок

| Символ | Роль |
|---|---|
| `->` | тип возврата, функциональный тип |
| `=>` | тело функции (именованной или анонимной), match-arm |
| `=` | присваивание (`let x = 5`) |

Один символ — одна роль, без исключений.

### Грамматика анонимной функции

```
lambda := "(" params? ")" effects? ("->" type)? "=>" body
```

Скобки вокруг параметров **обязательны всегда**, даже для одного
параметра без типа. `x => x + 1` запрещён, пиши `(x) => x + 1`. Это
устраняет парсер-неоднозначность с другими использованиями идентификаторов
и стандартизирует форму.

### Что НЕ меняется

- `match`-arm: уже использует `=>` (D19), синтаксис тот же.
- `->` для типа возврата и функциональных типов — не меняется (D20).
- `let` — `=` остаётся.
- Именованная `fn` сохраняет ключевое слово `fn`, имя обязательно.

### Запрещено

```nova
fn() => 0                              // НЕТ — fn без имени
fn(x int) -> int => x * 2              // НЕТ — анонимная fn
let f = fn(x) => x + 1                 // НЕТ — fn без имени
fn double(x int) -> int = x * 2        // НЕТ — = вместо => в теле
x => x + 1                             // НЕТ — скобки обязательны
```

Если функция без имени — это лямбда `(params) => body`. Если с именем —
`fn name(params) => body`. Третьего синтаксиса нет.

### Цена

1. **Большая правка примеров.** ~100 примеров `fn ... =` в документах
   и [examples/audit.nv](examples/audit.nv). Механическая, но объёмная.
2. **Расхождение с Rust/Swift/Kotlin** в синтаксисе тела `fn`. Там
   `fn f() -> T { body }` или `fn f() = body`. У нас будет `=>`. Это
   стоит привычки.
3. **Пересмотр D20.** D20 был помечен «final для v1.0», но Nova находится
   в дизайн-стадии без реализации. «Финальность» означает «не пересматривать
   после релиза», не «никогда». Этот пересмотр — до релиза, и он устраняет
   противоречие, а не вводит новое.

### Связь с другими решениями

- **Развивает D19/D20** — устраняет дублирующий синтаксис.
- **Развивает D18** — тот же принцип «не плодить специальные сущности»,
  применённый к синтаксису функций.
- **Поддерживает D10** — AI-first: меньше форм, меньше путаницы у LLM.

## D23. `return` — только для раннего выхода

**Решение.** В Nova **есть** ключевое слово `return`, но используется
исключительно для **раннего выхода** из функции. Последнее выражение
тела — автоматически результат, `return` для него не пишется.

### Что разрешено

```nova
fn classify(x int) -> str =>
    if x < 0  { return "negative" }
    if x == 0 { return "zero" }
    if x < 10 { return "small" }
    "big"                              // последнее выражение = результат

fn process(req Request) Db Throws -> Response =>
    if req.method == "GET" { return next(req) }
    if !authorized        { return Response.error(403) }
    do_work(req)?
```

`return expr` прерывает текущую функцию и возвращает значение. Используется
для guard-clauses и ранних выходов, чтобы избежать глубокой вложенности
`if/else`.

### Что запрещено (стиль)

```nova
fn double(x int) -> int => return x * 2     // ИЗБЫТОЧНО
fn double(x int) -> int => x * 2            // правильно

fn classify(x int) -> str =>
    if x < 0 {
        return "negative"
    } else {
        return "positive"                   // ИЗБЫТОЧНО — обе ветки return
    }

// правильно — match как выражение
fn classify(x int) -> str => match x {
    n if n < 0 => "negative"
    _          => "positive"
}
```

`return` в **последней** позиции функции запрещён `nova lint`-правилом —
он избыточен. Если все ветви функции заканчиваются `return`, значит код
нужно переписать через `match`/`if-else` как выражение.

### Семантика

- `return expr` в теле функции — возврат `expr` из функции.
- `return` в лямбде — возврат из **самой лямбды**, не из enclosing-функции.
  Это упрощает рассуждение: `return` всегда привязан к ближайшей
  функциональной границе.
- `return` в match-arm — возврат из **enclosing-функции**, не из `match`
  (match — выражение, не функциональная граница).
- В `with`-блоках `return` тоже выходит из enclosing-функции,
  пройдя через soft-unwind handler-стека.

### Что отвергнуто

**Полное отсутствие `return` (OCaml/Haskell-стиль).**
Заставляло бы вкладывать `if/else if/else` глубже, ухудшая читаемость
для middleware-стиля и guard-clauses. Audit.nv демонстрирует, что код
с ранними выходами **естественно пишется** даже когда автор не думает
про `return` — это сильный сигнал, что без него язык будет неудобным.

**`break` или `done` вместо `return`.**
Нестандартно, без выгоды. `return` — слово, известное LLM из любого
обучающего датасета.

**`return` обязателен везде (Go/Java-стиль).**
Противоречит «функция = выражение» (тело через `=>`). Если последнее
выражение и так результат, требовать `return` для него — избыточный шум.

### Почему так

1. **Соответствует стилю audit.nv.** `return next(req)`, `return resp`
   уже используются как guard-clauses, естественно для middleware.
2. **AI-first.** LLM генерирует `return` рефлекторно из обучающих данных.
   Полный запрет требовал бы переучивания. Разрешённое использование
   для ранних выходов — компромисс.
3. **Один язык — один стиль.** Линт против избыточных `return` в
   последней позиции даёт единый код, без гибрида «иногда `return`,
   иногда последнее выражение» в одной функции.
4. **Прецедент.** Rust идиоматически использует `return` только для
   ранних выходов, последнее выражение пишется без `;`. Nova следует
   тому же правилу строже — через линт.

### Цена

1. Программисту с фоном Go/Java придётся отвыкать писать `return` в
   конце. Это лёгкая привычка.
2. Линт-правило «нет избыточного `return`» нужно реализовать в
   `nova lint`. Стандартное правило, не сложное.
3. Для функций с одним выражением иногда хочется `return` для
   симметрии с многовариантными — приходится принять, что одна ветвь
   = просто выражение, без `return`.

### Связь с другими решениями

- **Совместимо с D22.** Тело функции через `=>` — это выражение
  (одно или блок). `return` — выход из этого выражения досрочно.
- **Совместимо с D19.** `match`-arm через `=>` остаётся выражением;
  `return` внутри arm выходит из enclosing-функции.
- **Поддерживает D10.** Меньше путаницы для LLM: одно правило
  «return = ранний выход», без двух стилей в одной функции.

## D24. Стратегия SMT-проверки контрактов

**Решение.** Контракты в сигнатуре (`requires`/`ensures`/`invariant`,
[R4](spec/revolutionary.md)) проверяются **SMT-движком на этапе компиляции**
с явным таймаутом и fallback на runtime-проверку. Контракт, который
SMT не смог доказать, **не блокирует** компиляцию по умолчанию — он
становится runtime-assert'ом в debug и тихо игнорируется в release с
предупреждением `@unverified`.

Программист может явно требовать статическое доказательство через
`@must_verify` — тогда компиляция падает, если SMT не справился.

### Что зафиксировано (стратегия)

1. **SMT-кодировка** контрактов из `requires`/`ensures`/`invariant`
   в стандартный формат (SMT-LIB v2). Конкретный движок —
   **выбор реализации**, не дизайна. Дизайн фиксирует **класс движка**:
   поддержка теорий **LIA** (linear integer arithmetic), **EUF**
   (equality + uninterpreted functions), **arrays** базовой
   функциональности. Этим требованиям удовлетворяют Z3, CVC5,
   Bitwuzla — выбор делает компилятор-реализация.

2. **Таймаут на функцию** — рекомендуемый дефолт **2 секунды**.
   Превышение → fallback на runtime-проверку. Программист может
   увеличить через `@verify_timeout(seconds=10)` локально или
   глобально в конфигурации проекта.

3. **Поведение по уровням сборки:**
   - **debug:** SMT-проверка + runtime-fallback для непроверенного.
     Нарушение runtime → panic с указанием контракта и точки.
   - **release:** SMT-проверка. Доказанные контракты — стираются
     полностью (zero cost). Недоказанные — игнорируются молча с
     warning'ом на этапе сборки.

4. **Опт-ин строгости через атрибуты:**
   - **`@must_verify`** на функции → если SMT не доказал контракт,
     компиляция **падает**. Для критичного кода (медицина, финансы,
     авионика).
   - **`@unverified`** на функции → отказ от попытки доказательства
     заранее, всегда runtime-check (чтобы не тратить время
     компиляции на заведомо непроверяемое).

### Что поддерживается в v1.0

**Целевые классы контрактов:**

| Класс | Пример | Решается |
|---|---|---|
| Линейная арифметика над `int`/`money` | `requires amount > 0`, `ensures result == a + b` | да (LIA) |
| Equality для record и sum-type | `requires acc.id == old.id` | да (EUF) |
| Cardinality коллекций | `requires xs.len() > 0`, `ensures result.len() <= xs.len()` | да (через axiomatization) |
| Membership | `ensures result in xs` | да |
| `old(...)` в `ensures` | `ensures balance == old(balance) - amount` | да |
| Условные импликации | `ensures result.is_ok ==> condition` | да |

**Что НЕ поддерживается (research-уровень, отложено):**

- Квантификаторы общего вида (`forall x. P(x) ==> Q(x)`)
- Индукция по структуре данных
- Рекурсивные предикаты
- Нелинейная арифметика
- Floating-point reasoning
- String reasoning сложнее `len()` и equality

Контракты с этими конструкциями принимаются грамматикой, но SMT их не
доказывает → fallback на runtime или ошибка с `@must_verify`.

### Контракты со ссылками на handler-state

Открытый вопрос. Контракт может содержать обращение к операции эффекта:

```nova
fn transfer(...) Db -> ()
    ensures Db.balance(to) == old(Db.balance(to)) + amount
=> ...
```

Это требует **effect-aware SMT-кодировки**: handler-вызов как
неинтерпретированная функция с теоремами о её поведении. В v1.0
поддержка **частичная** — только для эффектов с явным `pure_view`
(чистая проекция состояния handler'а). Полная поддержка — research,
отдельный D-пункт после v1.0.

### AI-first связь

Когда SMT не справился, ошибка компилятора имеет структурированный
формат:

```
warning C0341: contract not verified statically
   in function `withdraw` at src/account.nv:34
   ┌─ src/account.nv:34:5
   │
   34 │     ensures acc.balance == old(acc.balance) - amount
   │             ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^ this contract was not proven

   reason: SMT solver returned 'unknown' after 2.0s
   missing facts:
     - relation between `acc.balance` and arithmetic operations on `money`

   fallback: runtime check inserted in debug build

   suggestions:
     1. add intermediate `assert` to break the proof into steps
     2. use `@must_verify` if static proof is required
     3. simplify the contract if possible
```

Это **обучающий сигнал** для LLM ([R5.3](spec/revolutionary.md)). Модель
получает не «что-то не так», а конкретный класс проблемы и три
предложения. Это и есть AI-first компилятор.

### Что отвергнуто

- **«Всегда статическая проверка, без runtime-fallback».** Сделало бы
  контракты обязательно тотальными — половину прикладного кода
  невозможно было бы аннотировать.
- **«Только runtime-проверка».** Теряется ценность контрактов в
  release (zero cost). Превращает контракты в обычные `assert`.
- **Фиксация конкретного SMT-движка в дизайне.** Дизайн — про
  семантику, не про имя движка. Имя — выбор реализации, как fiber-runtime
  в [D14](decisions.md) (там не сказано «Tokio», сказано
  «fiber-based scheduler»).

### Цена

1. **Реализация требует SMT-интеграции.** Нетривиально, но не research —
   Dafny / F* / Why3 показали, что это работает.
2. **Таймаут зависит от структуры контракта.** Программист иногда
   удивится: «почему не доказывается?». Структурированная ошибка
   должна объяснять.
3. **Effect-aware SMT** — частичная поддержка в v1.0, полная — после.
   Контракты с handler-state — known limitation, не проблема.

### Связь с другими решениями

- **Реализует [R4](spec/revolutionary.md)** — контракты становятся не
  только декларацией, но и инструментом проверки.
- **Поддерживает [R5.3](spec/revolutionary.md)** — структурированные ошибки
  SMT-проверки.
- **Поддерживает [R5.7](spec/revolutionary.md)** — обратимость spec ↔ impl
  опирается на возможность статически проверить контракт.
- **Согласовано с [D10](decisions.md)** — видимость в типах, проверяемость
  по фрагменту, AI-first.

## D25. `throw` и параметризация `Throws[E]`

**Решение.** Бросать ошибку — выражение `throw expr`, прерывающее
функцию через эффект `Throws[E]`. Параметр `E` — тип бросаемого
значения, обычно sum-type. `Throws` без параметра — сахар над
`Throws[Error]` (где `Error` — тип-обёртка из stdlib), допустим только в
приватных функциях. В публичных API `Throws[E]` обязателен.

### Базовое использование

```nova
type DepositError = Closed, NotPositive, OverLimit

fn deposit(mut acc Account, amount money) Throws[DepositError] -> () =>
    if acc.closed   { throw Closed }
    if amount <= 0  { throw NotPositive }
    acc.balance += amount
```

`throw expr` — выражение типа `Never` (никогда не возвращает), прерывает
функцию и передаёт `expr` через эффект `Throws[E]`. Тип `expr` должен
совпадать с `E` в сигнатуре.

### `throw` — операция эффекта `Throws[E]`, не магия

Связь между `throw` и `Throws[E]` — **не специальная проверка
компилятора**, а прямое применение модели алгебраических эффектов
([D10](decisions.md)). `throw expr` — это **операция эффекта**, точно
так же как `Db.query(...)` или `Logger.log(...)`.

Концептуально prelude объявляет:

```nova
type Throws[E] {
    throw(value E) -> Never        // операция, никогда не возвращает
}
```

`throw expr` — сахар для `Throws.throw(expr)`. Компилятор разворачивает
синтаксический `throw` в обычный вызов операции эффекта. Дальше работает
**общее правило для всех эффектов**: использовал операцию — задекларируй
эффект в сигнатуре.

```nova
fn lookup(id u64) Db -> User =>           // Db в сигнатуре — ок
    Db.query("...")

fn lookup(id u64) -> User =>               // Db отсутствует — ошибка
    Db.query("...")
//  ^^^^^^^^^^^^^^^ operation Db.query requires effect Db

fn parse(s str) Throws[ParseError] -> int =>    // Throws в сигнатуре — ок
    throw ParseError.BadFormat

fn parse(s str) -> int =>                       // Throws отсутствует — ошибка
    throw ParseError.BadFormat
//  ^^^^^^^^^^^^^^^^^^^^^^^^^^ throw requires effect Throws[ParseError]
```

Никакой отдельной логики для `throw` нет. Та же проверка, что для
`Db.query`, `Net.get`, `Time.now` и любой другой операции.

### `?` — сахар над `match` + `throw`

`?` тоже не магия. `expr?` разворачивается в:

```nova
match expr {
    Ok(v)  => v
    Err(e) => throw e            // обычный throw, требует Throws[E]
}
```

Поэтому `?` работает только в функциях с `Throws[E]` в сигнатуре —
потому что **раскрывается в `throw`**, а `throw` требует эффект.
Отдельного правила «`?` требует `Throws`» нет, оно вытекает из
обычной проверки эффектов.

### `Never` — почему `throw` совместим с любым типом

`throw expr` имеет тип `Never` — тип, означающий «не возвращает
значение в обычном смысле». `Never` — подтип любого типа (как
`Nothing` в Kotlin/Scala), поэтому `throw` можно использовать как
выражение в любой позиции:

```nova
let x int = if condition { 42 } else { throw NotReady }
//                                     ^^^^^^^^^^^^^^^
//                                     тип Never, совместим с int
```

Это работа `Never`, а не специальное правило для `throw`. То же
поведение у `return` и `panic` — все три имеют тип `Never`. Поэтому
работают и такие выражения:

```nova
let user = lookup(id) ?? return Response.error(404)
//                       ^^^^^^^^^^^^^^^^^^^^^^^^^^^
//                       тип Never, совместим с типом user
```

### Поймать через `?` (проброс) или handler (обработка)

**Проброс через `?`** — сахар «если ошибка, верни её выше». Работает
в функциях с совместимым `Throws[E]` в сигнатуре ([D4](decisions.md)):

```nova
fn pipeline(s str) Throws[ParseError] -> int =>
    let n = parse(s)?              // если parse бросил — pipeline тоже бросает
    n
```

**Обработка через handler** — обычный handler-блок ([D11](decisions.md)):

```nova
fn try_deposit(acc Account, amount money) Log -> bool =>
    with Throws[DepositError] = (err) => {
        Log.error("deposit failed: ${err}")
        false
    } {
        deposit(acc, amount)?
        true
    }
```

Тип результата `with`-блока — общий тип всех веток (тело и handler'ы).
Здесь оба возвращают `bool`. Если разнотипные — обернуть в sum-type/
`Result`/`Option`.

### Три формы handler'а — три семантики

Handler `Throws[E]` может:

1. **Не вызывать `resume`** → прерывание (`with`-блок возвращает значение
   handler'а). Аналог `try/catch` в Java.
2. **Вызывать `resume(value)`** → продолжение с подменой (управление
   возвращается в точку `throw`, как будто `throw` вернул `value`).
   Сильнее `try/catch` — позволяет «починить» ошибку и продолжить.
3. **Делать новый `throw`** → проброс наверх как другой тип (управление
   ищет следующий handler в стеке).

### Один параметр в `Throws[E]`

Параметр **один**. Если функция бросает несколько типов — программист
делает sum-type:

```nova
type TransferError = InsufficientFunds, InvalidAccount, AccountClosed

fn transfer(...) Throws[TransferError] Db -> Receipt => ...
```

**Отвергнуто** — `Throws[E1, E2, E3]` (множественные параметры). Причины:

- Усложняет композицию handler'ов (handler ловит «один из E1/E2/E3»?
  все три? один с union-pattern'ом?)
- Семантически избыточно — sum-type выражает то же чище
- Нарушает простое правило «один эффект — один параметр», как
  `Alloc[R]`, `Ask[T]`

В [revolutionary.md:208](spec/revolutionary.md) исторически было
`Throws[InsufficientFunds, InvalidAccount]` — это нужно поправить под
sum-type стиль.

### `Throws` без параметра — только в приватных функциях

```nova
fn process(o Order) Throws Db -> () =>             // приватно: Throws[Error] выводится
    validate(o)?
    save(o)?

export fn process_pub(o Order) Throws[OrderError] Db -> () => ...
```

Правило:
- **Приватная функция** (без `export`) → `Throws` без `[E]` означает `Throws[Error]`
  (общий тип-обёртка из stdlib)
- **Публичная функция** (`export`) → требует явный `Throws[E]`. Без `[E]`
  — ошибка компиляции

Это **AI-first компромисс**: внутри модуля программист может писать
быстро, не придумывая имя ошибки. На границе модуля LLM (и человек) видят
конкретный тип в сигнатуре.

### Связь с `Result[T, E]`

`Throws[E]` и `Result[T, E]` — **два стиля одного и того же**.
[effects.md:196-204](spec/effects.md) уже намекает; формальное правило:

```nova
fn parse_a(s str) Throws[ParseError] -> int => ...    // throws-стиль
fn parse_b(s str) -> Result[int, ParseError] => ...   // result-стиль
```

Внутри функции с `Throws[E]` можно вернуть `Result[T, E]` через
`.try_unwrap()` или подобную операцию stdlib. Одно конвертируется в
другое.

**Когда использовать Throws:** прикладной код, чтение через `?`,
эффект-handler'ы для retry/log/централизованной обработки.

**Когда использовать Result:** библиотечные функции, где тип ошибки
важен как часть контракта; функции, которые часто ловятся явным `match`
без проброса.

Дефолт — `Throws`. `Result` — когда есть конкретная причина.

### `throw` ≠ `panic`

`throw expr` — обычная ошибка через эффект, **видна в сигнатуре** через
`Throws[E]`. Перехватывается handler'ом в коде.

`panic` ([D13](decisions.md)) — аппаратные/математические сбои (деление
на 0, переполнение, OOM, выход за границы массива) или вызов `panic(msg)`
программистом. **Не виден в сигнатуре**. **Не ловится в коде** —
означает смерть текущего fiber'а, ловится только runtime'ом на границе
fiber'а ([R9 supervised](spec/revolutionary.md)).

Это разные миры:
- «обработать можно и нужно» → `throw` + `Throws[E]`
- «обработать никак нельзя, fiber умирает» → `panic`

### Что отвергнуто

- **`raise` или `error()` вместо `throw`** — `throw` известно из
  Java/JS/C#/Swift, AI-friendly без переучивания.
- **`Throws` всегда без параметра** — теряется видимость типа ошибки
  в сигнатуре, ломает AI-first тезис.
- **`throw` без эффекта в сигнатуре** — невидимое control flow,
  главная проблема Java unchecked exceptions.

### Связь с другими решениями

- **Развивает [D2](decisions.md)** — эффекты вместо ключевых слов;
  `Throws[E]` — один из эффектов.
- **Развивает [D4](decisions.md)** — `?` теперь явно работает с любым
  совместимым `Throws[E]`.
- **Согласовано с [D11](decisions.md)** — handler-литералы для
  `Throws[E]` — обычная форма.
- **Граничит с [D13](decisions.md)** — `throw` ≠ Panic, разные
  механизмы для разных случаев.

### Цена

1. Программист обязан явно описывать тип ошибки в публичных API. Это
   дополнительная работа, оправданная видимостью контракта.
2. Sum-type для нескольких типов ошибок — небольшой синтаксический
   налог, оправданный простотой композиции handler'ов.
3. Граница `throw` vs `Panic` требует понимания — лечится документацией
   и сообщениями компилятора.

## D26. Базовая stdlib и prelude

**Решение.** Базовые типы (`Option[T]`, `Result[T, E]`, `Error`) и их
конструкторы (`Some`, `None`, `Ok`, `Err`) живут в **prelude** —
автоматически в скоупе любого модуля, без `import`. Список prelude
явно зафиксирован в одном месте, не «магия».

### Что в prelude (v1.0)

**Типы:**

```nova
type Option[T] = Some(T), None
type Result[T, E] = Ok(T), Err(E)
type Error                                       // тип-обёртка для Throws
type Ordering = Less, Equal, Greater
type Never =                                     // sum-type БЕЗ вариантов
```

### `Never` — обычный тип без значений, не магия

`Never` объявлен как **sum-type с нулём вариантов** — синтаксически
`type Never =` (после `=` пусто). Это легальная конструкция в системе
[D17](decisions.md): sum-type — список вариантов через запятую, и пустой
список — корректный частный случай.

**Свойства следуют из пустоты, не из специального правила:**

- **Нельзя создать значение типа `Never`** — нет ни одного варианта,
  нет конструктора. Любая попытка → ошибка типизации.
- **`Never` — подтип любого типа.** Если у типа нет значений, функция
  возвращающая `Never` никогда не вернётся; любой контекст, ожидающий
  `T`, может принять `Never`-выражение, потому что несовместимости
  быть не может (нет значений → нет конфликта).
- **Используется в типах не-возвращающих выражений** — `throw expr`,
  `return expr`, `panic(...)`, бесконечный `loop`. Все имеют тип
  `Never`, поэтому совместимы с любым контекстом.

Это **стандартный приём системы типов**, известный как «bottom type»
(⊥). Аналоги: Rust `!`, Haskell `Void`, Kotlin/Scala `Nothing`,
TypeScript `never`. Не уникальная фича Nova.

### Эффекты как обычные типы — `Throws[E]` не магия

Эффекты — это `type` с операциями ([D18](decisions.md)). `Throws[E]`
объявляется в prelude как любой другой эффект:

```nova
type Throws[E] {
    throw(value E) -> Never
}
```

Одна операция `throw`, параметр `E` — generic. Структурно идентично
`Logger` или `Db`:

```nova
type Logger { log(msg str) -> () }
type Db { query(sql str, args []any) -> []Row, exec(sql str, args []any) -> () }
```

`throw expr` в коде — сахар для `Throws.throw(expr)` (вызов операции
активного handler'а), как `Db.query(...)` — вызов операции активного
handler'а `Db`. Никакой специальной обработки для `Throws`. См.
[D25](decisions.md).

**Базовые числовые и строковые типы** (`int`, `i8`-`i64`, `u8`-`u64`,
`f32`, `f64`, `str`, `bool`, `char`, `()`, `byte`) — встроены в язык,
не stdlib, но упомянуты для полноты.

**Стандартные эффекты** ([R2](spec/revolutionary.md)) — `Throws[E]`, `Io`,
`Net`, `Db`, `Fs`, `Time`, `Random`, `Mut`, `Alloc[R]`, `Async`, `Par`,
`Log`, `Trace`, `Ask[T]` — также в prelude по умолчанию.

**Базовые функции:**

```nova
fn print(s str) Io -> ()
fn println(s str) Io -> ()
fn panic(msg str) -> Never
```

### Что НЕ в prelude

Коллекции (`String`, `Vec`, `HashMap`), I/O API (`File`, `Http`), JSON,
SQL, время как библиотека — всё это **обычные модули**, требующие явного
импорта. Имена slim:

```nova
import std.io.{File, read_all}
import std.collections.HashMap
```

Это **не противоречит D10 «локальность контекста»** — prelude
**документирован**, его содержимое фиксированный список, не магия.
LLM знает, что доступно везде. Всё остальное — явный импорт.

### Почему prelude нужен

Без prelude каждый файл начинается с:

```nova
import std.option.{Option, Some, None}
import std.result.{Result, Ok, Err}
```

Это шум на 90% файлов. Прецедент — Rust, Haskell, Swift, Kotlin: все
имеют prelude. AI-first аргумент: LLM не должен генерировать boilerplate-импорты
базовых типов.

### Конкретные ответы на вопросы из обсуждения

- **`None`** — вариант `Option[T]`, объявлен в prelude как часть
  `type Option[T] = Some(T), None`. Не зарезервированное слово, обычное
  имя из stdlib. Используется без скобок (unit-вариант).
- **`Some(token)`** — конструктор позиционного варианта `Some(T)`,
  принимает значение типа `T`.
- **`Ok(...)` / `Err(...)`** — варианты `Result[T, E]`, аналогично.
- **`Void`** — **не существует**, отвергнут в [D20](decisions.md). Тип
  «без значения» — `()` (unit).

### Что отвергнуто

- **Никакого prelude, всё через явный import.** Шум, не выигрыш.
- **Prelude определяется компилятором, без документации.** Магия,
  ломает AI-first тезис.
- **Prelude настраивается per-project.** Усложнение без выгоды; LLM
  должен знать фиксированный набор.

### Открытые вопросы

- **Какие именно методы у `Option`/`Result`** (`unwrap`, `unwrap_or`,
  `map`, `and_then`, `ok_or` и т.д.) — это stdlib API, описывается в
  [Q9](spec/open-questions.md). [audit.nv](examples/audit.nv) использует
  большинство из них де-факто.
- **Семантика `?` для `Option`** — `expr?` на `Option[T]` бросает что?
  Конвертирует через trait-подобный механизм? Сейчас в коде это
  работает (`Option.unwrap_or(default)`, `Option.ok_or(error)?`), но
  правило не зафиксировано.
- **`Error` как универсальный тип** — что в нём? `to_str()`?
  `cause: Option[~Error]`? Похоже на Rust `std::error::Error`. Описать
  в Q9.

### Связь с другими решениями

- **Поддерживает [D10](decisions.md)** — AI-first, локальность
  контекста через **документированный** prelude.
- **Реализует основу для [D25](decisions.md)** — `Throws[Error]`,
  `Result[T, E]`, `Option[T]` — базовые типы для системы ошибок.
- **Согласовано с [D17](decisions.md)** — все типы prelude объявлены
  стандартным синтаксисом sum-type/record.

### Цена

1. **Список prelude нужно поддерживать.** Любое добавление в prelude —
   breaking change после v1.0 (имя становится «зарезервированным» в
   модулях). Поэтому prelude **минимален**.
2. **Импорт-конфликты.** Если программист объявит свой `type Option`,
   получит ошибку «conflicts with prelude". Лечится переименованием
   или явным `import std.option.Option as StdOption`.

## D27. Синтаксис массивов: `[]T` префикс, `[N]T` для фиксированных

**Решение.** Массивы записываются **префиксом**, как в Go:

- **`[]T`** — динамический массив (Vec/slice)
- **`[N]T`** — фиксированный массив длины `N`
- **`[N1][N2]T`** — многомерный фиксированный, **в порядке индексации**
  (внешний размер первым)

Generic-применение остаётся **постфиксом** ([D16](decisions.md)):
`Option[T]`, `HashMap[K, V]`, `Vec[T]`. `[T]` сам по себе **больше не
означает «массив»**.

### Что было (отменено)

```nova
// СТАРО — отменено
let xs [int] = [1, 2, 3]
fn parse(xs [str]) -> [int]
type Matrix = [[f64; 3]; 3]
fn audit_middleware(next Handler) -> Handler =>
    (req Request) Db Log Throws -> Response => ...
```

`[T]` использовался для двух разных вещей: динамический массив **и**
generic-применение (`Option[T]`). Это создавало неоднозначность.

### Что стало

```nova
// НОВО — Go-style массивы
let xs []int = [1, 2, 3]
fn parse(xs []str) -> []int =>
    [parse_one(s)? for s in xs]

// фиксированный
let buf [5]u8 = [0, 0, 0, 0, 0]
let zeros [4]u8 = [0; 4]              // повторение через ;

// многомерный (плоский row-major)
let matrix [2][3]int = [[1, 2, 3], [4, 5, 6]]
matrix[i][j]                           // i от 0 до 1, j от 0 до 2

// generic не меняется
let opt Option[int] = Some(42)
let map HashMap[str, int] = HashMap.new()
```

### Правила парсинга

После `[` парсер видит:

| Что | Это |
|---|---|
| `]` (закрытие сразу) | динамический массив `[]T` |
| число + `]` | фиксированный массив `[N]T` |
| тип + `]` | generic-применение `Имя[T]` (только после имени типа) |
| тип + `,` | generic c несколькими параметрами `Имя[T1, T2]` |

`[`-конструкции **различаются по позиции в грамматике**:
- В позиции типа после имени (`Option[T]`) — generic
- В позиции типа без имени (`[]T`, `[5]T`, `[2][3]T`) — массив
- В позиции выражения (`arr[i]`) — индексация

Парсер однозначен.

### Соответствие индексации

Главный довод за Go-стиль — **порядок размеров в типе и в индексации
совпадает**:

```nova
let arr [2][3]int = ...
arr[i][j]                              // i: 0..2 (внешний), j: 0..3 (внутренний)
```

В Rust-стиле `[[int; 3]; 2]` порядок обратный — внутренний размер пишется
первым. Это известная претензия к Rust, программисты регулярно ошибаются.

### Литералы

```nova
let xs []int = [1, 2, 3]               // динамический литерал
let buf [5]u8 = [0, 0, 0, 0, 0]        // явное перечисление
let zeros [100]u8 = [0; 100]           // повторение, как Rust
let matrix [2][3]int = [[1, 2, 3], [4, 5, 6]]   // вложенный литерал
```

### Пустой массив

```nova
let empty []int = []                   // пустой динамический
let single [1]int = [42]               // фиксированный из одного
```

Пустой фиксированный (`[0]T`) технически возможен, но бесполезен —
обычно `()` или `Option[T]` вместо.

### Расположение в памяти

- **`[N]T`** — N элементов подряд, размер `N * sizeof(T)`. Без указателя.
- **`[]T`** — структура `{ ptr: *T, len: int, cap: int }` (24 байта на
  64-bit), данные на куче.
- **`[N1][N2]T`** — **плоский row-major** массив `N1*N2` элементов. Все
  строки лежат подряд. Доступ `arr[i][j]` = `*(base + i*N2*sizeof(T) + j*sizeof(T))`.
- **`[][]T`** — массив динамических массивов (jagged), внешний массив
  хранит указатели на отдельные внутренние массивы.

Для матриц в hot path использовать `[N1][N2]T` (плоский, cache-friendly).
Для jagged-данных или динамики — `[][]T`.

### Что отвергнуто

**Java-стиль `T[]`/`int[2][3]`:**
- ✗ Парсер сложнее (контекст: индексация vs объявление)
- ✗ Конфликт с `Option[T]` (если оставить generic постфиксом)
- ✗ Трудное чтение сложных типов справа налево

**Rust-стиль `[T]`/`[[T; N]; M]`:**
- ✗ Обратный порядок размеров в многомерных
- ✗ Конфликт `[T]` (массив) vs `Option[T]` (generic) — два разных
  смысла одного символа
- ✗ Многословно для многомерных

**`[T; N]` для одномерного фиксированного (Rust):**
- ✗ `;` отделяет тип от размера, но в многомерных читается странно
- ✗ Нет соответствия индексации

### Почему это правильное решение для Nova

1. **Соответствие индексации** — `[2][3]int` ↔ `arr[i][j]`. Главный
   аргумент.
2. **Парсер однозначен** — `[` всегда начинает тип-конструкцию или
   индексацию, разные позиции грамматики.
3. **Чтение слева направо** — «массив 2×3 целых» без расшифровки.
4. **Generic не страдает** — `Option[T]` остаётся.
5. **Прецедент Go** — backend-программисты привыкают мгновенно.

### Цена

1. **Breaking change** — `[T]` синтаксис из ранних документов и audit.nv
   надо переписать. ~50+ мест.
2. **Расхождение с Rust/Swift** — там `[T]`. Программистам с этим фоном
   нужно переучиться. Цена недели.
3. **Пересмотр D16** — D16 говорил «параметры типа в `[T]`», теперь
   только generic-применение `Имя[T]`. D16 нужно уточнить.

### Связь с другими решениями

- **Уточняет [D16](decisions.md)** — `[T]` теперь только в позиции
  generic-применения (`Имя[T]`), не сам по себе как тип.
- **Согласовано с [D17](decisions.md)** — синтаксис не конфликтует со
  sum-type/record-объявлениями.
- **Согласовано с [D10](decisions.md)** — AI-first, парсер однозначен,
  чтение естественное.

## D28. Вывод эффектов: private — выводится, public — обязательно явно

**Решение.** Эффекты в сигнатуре private-функций (без `export`)
**выводятся компилятором** из тела. Программист может опустить эффекты,
компилятор проанализирует вызовы и добавит. В public API (`export fn`)
эффекты **обязательны явно** — это контракт.

Чистая функция (без эффектов в сигнатуре) — это **проверенный факт**,
не «возможно есть, может быть нет». Если эффектов нет — компилятор
гарантирует детерминированность.

### Базовое использование

```nova
// private — эффекты выводятся
fn helper(x int) =>
    Logger.log("processing ${x}")     // компилятор добавит Logger в сигнатуру
    x * 2

// то же явно — программист тоже может писать
fn helper(x int) Logger -> int =>
    Logger.log("processing ${x}")
    x * 2

// public — должно быть явно
export fn process(x int) Logger -> int =>
    Logger.log("...")
    x * 2

// public БЕЗ эффектов — компилятор проверяет, что их и правда нет
export fn double(x int) -> int =>
    x * 2                              // ок, чистая

export fn bad(x int) -> int =>
    Logger.log("...")                  // ОШИБКА: эффект Logger не объявлен
    x * 2
```

### Гарантия чистоты

Функция **без эффектов в сигнатуре** = компилятор доказал, что она:
- Не делает IO, не ходит в сеть/БД/файлы
- Не yield'ит (нет `Async`)
- Не бросает (нет `Throws`)
- Не мутирует разделяемое состояние (нет `Mut`)
- Не аллоцирует в region'ах с эффектом (нет `Alloc[R]`)
- Детерминирована — равные входы → равные выходы

Это **сильная гарантия**, и она работает потому что эффекты **являются
проверенным фактом**, не дефолтом. Можно мемоизировать, вызвать в
interrupt-handler'е, в hot loop с lock'ом, в spinlock'е, в любом
параллельном контексте.

### Правило вывода

Компилятор анализирует тело функции:
1. Каждый вызов другой функции с эффектами → эти эффекты добавляются
2. Каждый `throw` → `Throws[E]` добавляется
3. Каждое чтение/запись `Mut`-поля или мутация `@field` в
   `mut @method` ([D35](#d35-методы-инстанса-через--self-отменён)) →
   `Mut` добавляется
4. Использование операции эффекта (`Db.query`, `Logger.log`) → этот
   эффект добавляется

Вывод транзитивный: если private-функция `A` вызывает private `B`,
которая вызывает public `C` с `Db Throws`, то `A` получает `Db Throws`.

### Public API — почему обязательно явно

1. **Контракт модуля.** Сигнатура — это интерфейс, который другие
   модули видят. Изменение эффектов = breaking change. Должно быть
   видно в коде, не выводиться невидимо.
2. **AI-first.** LLM, читая сигнатуру публичной функции, должна
   видеть все побочные действия. Public API — точка, где «сигнатура =
   полное описание» работает.
3. **Документация.** Public — это то, что попадает в `nova doc`.
   Эффекты — часть документации, не runtime-деталь.
4. **Случайное расширение.** Если private-функция получила лишний
   эффект (программист добавил `Logger.log` в утилиту), это **не
   должно** автоматически попадать в public — public видит ошибку
   компиляции, программист принимает осознанное решение.

### Случайное расширение в private — известная проблема

Программист добавил вызов `Logger.log(...)` в утилиту → у функции
автоматически появился `Logger` → все вызывающие private-функции тоже
получили `Logger` транзитивно → пока не дойдёт до **public API**, где
будет ошибка компиляции «эффект Logger не объявлен».

Это **корректное поведение**, но цепочка изменений в diff не видна.
Решается тулингом:

- **`nova check --show-effects`** — режим, показывающий выведенные
  эффекты для всех private-функций.
- **`@no_effects` атрибут** на private-функцию — компилятор обязан
  подтвердить, что функция чистая. Если нет — ошибка.
- **`@effects(Logger, Db)` атрибут** — закрепить ожидаемые эффекты для
  private. Расширение → ошибка.

В release-сборке тулинг не нужен, в dev — стандартный механизм
проверки.

### Async — особенно важно

Возникал вопрос «сделать `Async` дефолтным для всех функций, чтобы
не писать его в каждой backend-сигнатуре». **Отвергнуто.** Причина:

Чистая функция `double(x int) -> int` гарантированно **не
приостанавливается**. Можно использовать в hot loop без yield-pauses.
Если бы `Async` был дефолтом — этой гарантии бы не было.

D28 решает «шум `Async`» иначе: в private он **выводится**, программист
не пишет. В public — пишет один раз. Гарантии чистоты сохраняются.

### Связь с другими решениями

- **Развивает [D2](decisions.md)** — «эффекты вместо ключевых слов»,
  с уточнением правила вывода.
- **Унифицировано с [D25](decisions.md)** — то же правило для
  `Throws`: выводится в private, обязателен в public.
- **Поддерживает [D10](decisions.md)** — AI-first: видимость в public
  API сохраняется, шум в private убирается.
- **Поддерживает [R5.2](spec/revolutionary.md)** — «сигнатура = полное
  описание» работает в public, где это и важно.
- **Согласовано с [D5](decisions.md)** — два уровня видимости (`export`
  / приватно), эффект-видимость следует видимости функции.

### Что отвергнуто

- **Везде явно (как Java checked exceptions).** Шум в private-утилитах
  без выгоды.
- **Везде выведено (как Haskell для типов).** Public API теряет
  явный контракт.
- **`Async` по умолчанию для всех функций.** Теряется гарантия
  чистоты, нарушается «нет эффектов = детерминированно».
- **Опт-ин для вывода (`@infer_effects` или подобный).** Программист
  выбирает каждый раз — лишний шум.

### Цена

1. **Качество сообщений компилятора** при ошибке «private-функция
   приобрела эффект, public-вызывающий не объявлен» — критично.
   Программист должен видеть **где** эффект пришёл, **через какую
   цепочку вызовов**.
2. **Цепные изменения в private** — диф не показывает явно, что
   эффекты расширились. Тулинг (`--show-effects`, `@no_effects`)
   компенсирует.
3. **Compile-time стоимость** — анализ эффектов транзитивный,
   увеличивает время компиляции на несколько процентов. Приемлемо.

## D29. Модули и импорты

**Решение.** Файл = модуль. Имя объявляется первой строкой `module path.to.name`.
Экспорт через `export` ([D5](#d5-видимость-только-export-или-приватно)),
импорт через `import path.module.{Names}`. Wildcard и циклические
импорты запрещены.

### Объявление модуля

```nova
module admin.audit                              // первая строка файла
```

Имя модуля — иерархическое через точки. **Соответствует пути в
файловой системе** относительно корня проекта: `admin.audit` →
`admin/audit.nv`.

Один файл = один модуль. Никаких `module X { ... }`-блоков внутри
файла, никаких нескольких объявлений `module` в одном файле.

### Экспорт через `export`

```nova
export fn process(...) -> Response              // экспорт функции
export type AuditEntry { ... }                  // экспорт типа
export const MAX_PAYLOAD = 4096                 // экспорт константы
export let counter = 0                          // экспорт let-binding'а

fn helper(...) -> ()                            // приватно для модуля
type _Internal { ... }                          // приватно (соглашение _-префикс)
```

Без `export` декларация **видна только внутри модуля**. Никаких
четырёх уровней видимости как в Java/C#. См.
[D5](#d5-видимость-только-export-или-приватно).

### Импорт

```nova
// импорт одного имени
import std.io.File

// импорт нескольких имён из модуля
import std.io.{File, Reader, Writer}

// импорт модуля как namespace
import std.collections                          // → доступ через std.collections.HashMap

// алиас одного имени
import std.option.Option as Opt

// алиас модуля
import std.collections as cols                  // → cols.HashMap

// смешанный — несколько имён с алиасами
import std.io.{File as F, read_all}
```

Импортированные имена попадают в **локальный скоуп модуля**. После
импорта используются без префикса (если не объявлен namespace-импорт).

### Wildcard `import x.*` — запрещён

```nova
import std.io.*                                 // ОШИБКА КОМПИЛЯЦИИ
```

Причины запрета:

1. **Скрывает источник.** Программист и LLM не видят, откуда пришло
   имя. Нарушает локальность контекста ([R5.1](spec/revolutionary.md)).
2. **Ломает at-distance.** Добавление имени в импортируемый модуль
   может конфликтовать с локальным — breaking change библиотек.
3. **AI-first.** LLM не должен искать «откуда `File`» по 10 импортам.

Программист пишет явный список или импортирует модуль как namespace.

### Циклические импорты — запрещены

```nova
// модуль A
module app.a
import app.b.SomeType
export type AT { x SomeType }

// модуль B
module app.b
import app.a.AT                                 // ОШИБКА: cycle a → b → a
export type SomeType { ... }
```

Compile error с указанием полного цикла. Решается выделением общих
типов в третий модуль (`app.shared`), который импортируется и `a`, и
`b`.

Это **сильнее, чем в Rust** (где `use` циклы разрешены через forward
declaration), но проще для рассуждения и согласуется с принципом «один
файл = один модуль, без forward declarations».

### Реэкспорт

```nova
module app.facade

export import std.io.{File, Reader}             // File и Reader доступны
                                                //  как app.facade.File, app.facade.Reader
export import std.collections.HashMap as Map    // алиас при реэкспорте
```

`export import X` делает имя `X` доступным **извне моего модуля**, как
если бы оно было объявлено в нём. Полезно для facade-модулей, которые
группируют API из нескольких базовых модулей.

### Конфликты имён

Если импортированное имя совпадает с локальным или другим импортом —
**ошибка компиляции**. Решается алиасом через `as`:

```nova
import std.option.Option
import my.Option as MyOption                    // алиас разрешает конфликт

let x Option[int] = Some(42)                    // std.option.Option
let y MyOption = ...                            // my.Option
```

### Prelude — без `import`

Prelude ([D26](#d26-базовая-stdlib-и-prelude)) — `Option`, `Result`,
`Some`, `None`, `Ok`, `Err`, `Error`, `Never`, базовые типы,
стандартные эффекты, `print`/`println`/`panic` — **автоматически в
скоупе**, без `import`.

Это **не противоречит** правилу «нет неявных импортов» (overview.md):
prelude **документирован** в одном месте ([D26](#d26-базовая-stdlib-и-prelude)),
LLM знает фиксированный список. Это «известная имплицитность», не
магия.

### Соответствие файловой структуры

```
project/
├── src/
│   ├── main.nv                  module main
│   ├── admin/
│   │   ├── audit.nv             module admin.audit
│   │   └── users.nv             module admin.users
│   └── http/
│       └── server.nv            module http.server
└── nova.toml                    манифест проекта
```

Корень — `src/`. Каждый `.nv` файл — отдельный модуль с именем,
соответствующим пути.

**Структура проекта** (где `src/`, как устроен `nova.toml`) — это
**tooling**, см. [Q10 в open-questions](spec/open-questions.md). Здесь
зафиксирована только семантика модулей, не файловая структура.

### Что отвергнуто

- **`pub` + `use`** (Rust-стиль) — `use` уже занят для embed/delegation
  в `type` (см. paradigm). Конфликт смыслов.
- **CamelCase для видимости** (Go-стиль) — мешает естественному
  именованию.
- **`public`/`private`/`protected`/`package`** (Java-стиль) — overkill,
  никто не помнит правила.
- **Wildcard import** — скрывает источник, ломает локальность контекста.
- **Циклические импорты** — усложняют рассуждение, требуют forward
  declarations.
- **Несколько модулей в одном файле** — нарушает «один файл = один
  модуль».
- **`#include` или препроцессор** — отвергнуто давно ([overview.md](spec/overview.md)).

### Связь с другими решениями

- **Развивает [D5](#d5-видимость-только-export-или-приватно)** —
  механизм видимости через `export` ключевое слово.
- **Поддерживает [D10](#d10-революционная-ставка-всё--эффект--ai-first)**
  — локальность контекста, видимость источника каждого имени.
- **Использует [D26](#d26-базовая-stdlib-и-prelude)** — prelude как
  единственное «исключение» из правила явных импортов.
- **Поддерживает [R5.1](spec/revolutionary.md)** — каждый идентификатор
  виден откуда пришёл.

### Цена

1. **Длиннее, чем `pub`** — `export fn` на 3 символа больше. Приемлемо
   для прозрачности.
2. **Циклические импорты требуют рефакторинга** — иногда болезненно
   при миграции с Python/Java. Решается выделением общих типов.
3. **Реэкспорт через `export import`** — двусловная конструкция, чуть
   многословнее, чем Rust `pub use`. Прозрачнее.

## D30. Стиль именования

**Решение.** Один стиль на весь язык, без исключений для акронимов:

| Что | Стиль | Пример |
|---|---|---|
| Типы (record, sum, alias) | PascalCase | `User`, `HashMap`, `Option`, `Result` |
| Варианты sum-type | PascalCase | `Some`, `None`, `Ok`, `Err`, `Red` |
| Эффекты | PascalCase | `Db`, `Net`, `Throws`, `Async`, `Logger` |
| Протоколы | PascalCase | `Hashable`, `Iterator`, `Printable` |
| Generic-параметры | PascalCase, односимвольные | `T`, `K`, `V`, `E` |
| Static-функции (`fn Type.name`) | snake_case | `Account.new`, `HashMap.from`, `Duration.from_secs` |
| Методы инстанса (`fn Type @name`) | snake_case | `@deposit`, `@balance`, `@find_slot` |
| Свободные функции | snake_case | `parse_url`, `next_pow2`, `validate_input` |
| Поля типа | snake_case | `user_id`, `created_at`, `balance` |
| Параметры, локальные переменные | snake_case | `let counter = 0`, `mut acc Account` |
| Константы (`const`) | SCREAMING_SNAKE_CASE | `MAX_PAYLOAD`, `DEFAULT_TIMEOUT` |
| Модули | snake_case через точки | `module admin.audit`, `module std.duration` |

### Акронимы — всегда PascalCase, не UPPERCASE

```nova
type Db { ... }                        // не DB
type Io { ... }                        // не IO
type Fs { ... }                        // не FS
type Url = str                         // не URL
type Http { ... }                      // не HTTP
type JsonValue = ...                   // не JSON
type SqlBuilder { ... }                // не SQL
type ApiClient { ... }                 // не API
type HttpsServer { ... }               // не HTTPS
```

**Правило:** акроним = обычное слово. `Http`, не `HTTP`. `Json`, не
`JSON`. `Url`, не `URL`. `Io`, не `IO`. `Db`, не `DB`.

### Почему так

1. **Одно правило без исключений** — программисту и LLM не нужно
   помнить «акроним из 2 букв капсом, из 3+ PascalCase». Любой
   акроним идёт по правилу PascalCase.
2. **AI-friendly.** LLM плохо угадывает «сколько букв в акрониме».
   Единое правило — нулевой шанс ошибки.
3. **Composability.** `HttpClient`, `JsonParser`, `SqlBuilder`,
   `IoStream` читаются без визуальных «плотностей» из заглавных букв.
   Сравни с `HTTPClient`, `JSONParser`, `SQLBuilder`, `IOStream` —
   глаз спотыкается на стыке акронима и слова.
4. **Прецедент.** Swift API Design Guidelines, Microsoft .NET
   guidelines (с 2010-х), современный Rust — все используют
   PascalCase для акронимов.
5. **Согласованность с уже использованным `Db`.** Сейчас в документах
   `Db` (PascalCase), но `IO`/`FS` (UPPERCASE) — противоречие.
   Зафиксировать единое правило.

### Что отвергнуто

**Java/C# до 2010-х (UPPERCASE для коротких акронимов):**
- `DBException`, `URLConnection`, `XMLParser`, `IOError`
- Минусы: исключения «сколько букв», `parseXMLForJSONFromHTTPResponse`
  превращается в кашу из заглавных
- Современный C# (.NET) сам отказался от этого правила

**snake_case для всего (Python/Ruby):**
- Не подходит для языка с типами как первоклассными — типы и значения
  визуально не отличаются.

**camelCase для функций (Java/C#/JS):**
- `to_str` читается лучше, чем `toStr` для коротких имён
- Snake_case согласован с Rust, ML-семьёй
- Граница слов чётче в snake_case, особенно для длинных имён:
  `parse_http_request` vs `parseHttpRequest` — snake_case легче
  сегментировать

### Цена

1. **Привычка.** Программисты с Java/Go-фоном пишут `IO`, `URL`, `HTTP`
   рефлекторно. Понадобится переучиваться.
2. **Расхождение с C/Java/Python**. `IOError` (Python), `IOException`
   (Java), `os.IsNotExist` (Go) — везде акронимы заглавными. Nova
   расходится.
3. **Breaking change в текущих документах.** `IO`, `FS` нужно
   переименовать в `Io`, `Fs` во всех примерах. ~30 мест.

### Что НЕ меняется

- **`Db`** — уже PascalCase, остаётся как есть
- **`Async`, `Throws`, `Net`** — не акронимы, обычные слова в
  PascalCase
- **`HashMap`, `Option`, `Result`** — обычные имена

### Зарезервированные имена методов (operator overloading)

См. [D46](#d46-перегрузка-операторов-через--методы). Имена `@plus`,
`@minus`, `@times`, `@div`, `@rem`, `@neg`, `@or`, `@and`, `@xor`,
`@shl`, `@shr`, `@eq`, `@lt`, `@le`, `@gt`, `@ge`, `@not`, `@get`, `@set`
**зарезервированы** за operator overloading. Использовать их для других
целей — анти-паттерн (компилятор не запретит, но код становится
сюрпризом).

### Договорные конвенции имён методов

Не enforced, но **рекомендуются** для единообразия stdlib и user code:

| Префикс/имя | Когда |
|---|---|
| `T.new(...)` | Стандартный конструктор (static-функция) |
| `T.from_X(...)` | Конструктор из значения X (`from_secs`, `from_db_row`, `from_pairs`) |
| `T.with_capacity(n)`, `T.empty()`, `T.default()` | Специализированные конструкторы |
| `@show()` | Display: `Self -> str`. Используется в `"${...}"` интерполяции |
| `@hash()` | Хеш: `Self -> u64`. Используется в HashMap, HashSet |
| `@clone()` | Глубокая копия |
| `@iter()` | Возвращает iterator |
| `@next()` | Step iterator'а: `Self -> Option[T]` |
| `@is_X()` | Bool-предикат: `is_empty`, `is_zero`, `is_valid` |
| `@as_X()` | Cheap-конверсия (без аллокации): `as_secs`, `as_str`, `as_nanos` |
| `@to_X()` | Возможно дорогая конверсия: `to_string`, `to_owned`, `to_lower` |

`is_` — bool. `as_` — без аллокации, дешёвая. `to_` — может аллоцировать,
может быть дороже. Семантическая разница важна — следуй ей.

### `_prefix` — только для полей, не для функций

Convention `_prefix` применяется **только к полям record**, как сигнал
«используй методы вместо прямого доступа» (см. [D47](#d47-видимость-деклараций)):

```nova
export type Account {
    readonly owner str         // публичное поле
    _balance money             // используй @balance() / @deposit()
}
```

**Для функций и методов `_prefix` не используется.** Видимость
двухуровневая через `export` (D47):

```nova
export fn process(...) => ...      ✅ публичная
fn validate(...) => ...            ✅ приватная
fn _validate(...) => ...           ❌ не нужно — нет «третьего уровня»
```

### Test-имена — строки естественного языка

`test "..."` принимает **строковый литерал** с описанием поведения,
не идентификатор:

```nova
test "insert and get" { ... }                  ✅ описание
test "withdraw decreases balance" { ... }      ✅
test "test_insert_and_get" { ... }             ❌ snake_case излишне
test "InsertAndGet" { ... }                    ❌ PascalCase излишне
```

### Связь с другими решениями

- **Согласовано с [D5](#d5-видимость-только-export-или-приватно)** —
  `export`/private не зависит от стиля имени.
- **Согласовано с [D10](#d10-революционная-ставка-всё--эффект--ai-first)**
  — AI-first: одно правило, нулевая неоднозначность.
- **Согласовано с [D26](#d26-базовая-stdlib-и-prelude)** — все
  prelude-имена в PascalCase: `Option`, `Result`, `Error`, `Never`,
  `Ordering`.

## D31. Handler-лямбда для эффектов с одной операцией

**Решение.** Если эффект имеет **ровно одну операцию**, handler можно
записать как лямбду — параметры лямбды соответствуют параметрам этой
операции. Для эффектов с двумя и более операциями — handler-литерал
обязателен.

### Базовое использование

```nova
type Throws[E] {
    throw(value E) -> Never
}

// сокращённо — handler-лямбда
with Throws[Error] = (err) => Log.warn("op failed", { "err": err.message }) {
    Db.exec(...)
}

// полная форма — эквивалентна
with Throws[Error] = Throws {
    throw(err) => Log.warn("op failed", { "err": err.message })
} {
    Db.exec(...)
}
```

Для эффекта с несколькими операциями — handler-литерал обязателен:

```nova
type Db {
    query(sql str, args []any) -> []Row
    exec(sql str, args []any) -> ()
}

with Db = Db {
    query(sql, args) => resume(real.query(sql, args))
    exec(sql, args)  => { staged.push((sql, args)); resume(()) }
} {
    transfer(alice, bob, 100)?
}
```

### Какие эффекты попадают под сахар

Из стандартного набора:

| Эффект | Операции | Сахар работает |
|---|---|---|
| `Throws[E]` | `throw(value)` | да — главный win |
| `Random` | `next()` | да (если одна операция) |
| `Logger` (минимальный) | `log(msg)` | да |
| `Time` | `now()`, `sleep(d)` | нет |
| `Db` | `query`, `exec` | нет |
| `Net` | `get`, `post`, ... | нет |
| `Fs` | `read`, `write`, ... | нет |
| Пользовательские | зависит | если одна |

`Throws[E]` — самый частый случай, ради него сахар главным образом
вводится. В backend-коде `with Throws[E] = (err) => ... { ... }`
будет основной формой обработки ошибок через handler.

### Грамматика

В позиции значения после `with EffectName =`:

- **Лямбда** `(params) => body` или `(params) Effects? -> Type? => body`
  → сахар, разворачивается в handler-литерал с одной операцией.
  Компилятор проверяет, что у эффекта **ровно одна операция**, и
  параметры лямбды совместимы с её сигнатурой.
- **Handler-литерал** `EffectName { op() => body, ... }` → используется
  как есть. Работает для любого числа операций.
- **Переменная или выражение** типа `Handler[EffectName]` → используется
  как есть.

Парсер однозначен по первому токену после `=`:
- `(` → лямбда
- `Имя {` → handler-литерал (если содержимое `{...}` начинается с
  `op() =>`) или record-литерал (если с `name:`)
- идентификатор → переменная/выражение

### Что компилятор проверяет

```nova
// ОК — эффект с одной операцией
type Logger { log(msg str) -> () }
with Logger = (msg) => println(msg) { ... }

// ОШИБКА — у Db две операции, лямбда неоднозначна
with Db = (sql, args) => ... { ... }
//        ^^^^^^^^^^^^^^^^^^^^^^^
//        error: handler-lambda requires effect with exactly one operation
//               (Db has 2: query, exec)
//        suggestion: use handler literal — Db { query(...) => ..., exec(...) => ... }

// ОШИБКА — параметры лямбды не совпадают с операцией
with Throws[Error] = () => { ... } { ... }
//                   ^^^^^^^^^^^^^^^
//                   error: handler-lambda parameters mismatch
//                          expected (value Error), got ()
```

### Что отвергнуто

**Полная SAM-conversion (любой type с одной операцией → лямбда).**
Это разрешало бы лямбды и для не-эффектных типов:

```nova
type Comparator { compare(a int, b int) -> int }
let c Comparator = (a, b) => a - b      // ← отвергнуто, не делаем
```

Причина: для **эффектов** сахар сильно окупается (Throws частый,
минимизация строк критична). Для **обычных типов** дублирует
функциональный тип `(int, int) -> int` без выгоды. Граница чёткая —
**сахар работает только в позиции `with EffectName =`**.

### Что меняется в исторических документах

Ранее в [open-questions.md](spec/open-questions.md) была отрицательная
запись «SAM-conversion отвергнут». Сейчас пересмотрена: SAM **принят
с ограничением** — только для эффектов в `with`, только при одной
операции.

Главный аргумент пересмотра: «минимум строк на выходе» — один из
центральных принципов Nova. `with Throws[E] = Throws { throw(err) =>
... }` повторяется в каждой обработке ошибок, что в backend-коде
постоянно. Сахар сокращает в 2-3 раза без потери семантики.

### Цена

1. **Парсер чуть сложнее** — после `with X =` нужно различить
   лямбду и handler-литерал. Не критично, парсер работает по первому
   токену.
2. **Breaking change при добавлении операции** — если эффект расширили,
   все handler-лямбды для него ломаются с compile error. Это
   **корректное** поведение (видимое нарушение контракта), но
   программисту нужно обновить код в нескольких местах.
3. **Два способа делать одно и то же.** Сахар (`(params) =>`) и
   полная форма (`EffectName { op() => }`). Линт может предлагать
   сахар где он короче.

### Связь с другими решениями

- **Развивает [D11](#d11-имена-эффектов-и-синтаксис-with)** —
  добавляет третью форму handler'а (помимо литерала и переменной).
- **Согласовано с [D22](#d22-анонимная-функция-только-лямбда--тело-именованной--тоже-)** —
  лямбды как первоклассные значения, единый синтаксис `(params) =>
  body`.
- **Согласовано с принципом «минимум строк»** — главный win для
  `Throws[E]` обработки.
- **Согласовано с [D25](#d25-throw-и-параметризация-throws)** —
  `throw` как операция эффекта, лямбда естественно её перехватывает.

## D32. Семантика передачи параметров

> **Уточнено в [D36](#d36-поля-типа-дефолт-mutable-у-mut-bindinga-readonly-для-never-mut).**
> Семантика `mut` **на поле типа** изменилась: теперь поле без
> префикса мутируется у `mut`-binding'а; `let` для never-mut, `mut`
> per-field — только для cache/lazy. Семантика `mut` **на параметре**
> (этот D32) — без изменений.

**Решение.** Параметры функций передаются **by reference** в managed
heap (как Java/C# для объектов, Go для maps/slices). Без `mut` —
immutable view, с `mut` — мутации видны вызывающему. Примитивы
(`int`, `bool`, `f64`, ...) — by value в регистре. Borrow `&T` —
opt-in для hot path.

### Базовое поведение

```nova
type Account { mut balance money }

// без mut — функция только читает
fn show(acc Account) Io -> () =>
    println("balance: ${acc.balance}")

// с mut — функция меняет, изменения видны вызывающему
fn deposit(mut acc Account, amount money) =>
    acc.balance += amount

let mut my_acc = Account { balance: 100 }
deposit(my_acc, 50)
// my_acc.balance == 150  ← мутация видна
```

### Примитивы — by value

Числа, bool, char, byte, () — **всегда by value** в регистре.
Эффективно, ожидаемо.

```nova
fn square(x int) -> int => x * x

let n = 5
square(n)               // n не меняется (примитив всегда by value)
```

С `mut int` — это локальная переменная функции, изменения **не видны**:

```nova
fn weird(mut x int) =>
    x = 999             // меняет локально

let n = 5
weird(n)
// n == 5 (примитивы всегда by value, mut — для локальной мутации)
```

### Объекты (record/sum-type/массивы) — managed reference

Структуры, массивы, sum-type значения — передаются как **managed
reference**: указатель в managed heap, отслеживаемый GC. Без `mut`
функция читает, с `mut` мутирует видимо вызывающему.

**Терминология:** «managed reference» — это **деталь runtime'а** (GC
управляет жизнью объекта через указатели). В синтаксисе языка
программист пишет просто `o Order` — никакого `&` или `*` не нужно.

`&T` (borrow в Rust-стиле) **не существует в Nova**. См. [D6](#d6-память-managed-по-умолчанию-regions-opt-in-для-real-time)
— escape analysis закрывает большинство perf-кейсов, для real-time
есть `region { ... }`.

```nova
type Order { items []Item, total money }

fn add_item(mut order Order, item Item) =>
    order.items.push(item)
    order.total += item.price

let mut my_order = Order { items: [], total: 0 }
add_item(my_order, item1)
// my_order содержит item1 и обновлённый total
```

### Иммутабельный binding — что важно

Без `mut` параметр **нельзя** мутировать ни одно поле (кроме помеченных
`mut` per-field, [D36](#d36-поля-типа-дефолт-mutable-у-mut-bindinga-readonly-для-never-mut)):

```nova
type Account { balance money }

fn read_only(acc Account) =>
    acc.balance += 50       // ОШИБКА: acc immutable
    println(acc.balance)    // ок, чтение
```

`mut` на поле в типе — «это поле **можно** мутировать у `mut`-объекта».
`mut` на параметре — «функция **получила** объект как mutable».
Обе нужны вместе для записи.

### Производительность — escape analysis и regions

Когда нужна максимальная производительность без GC overhead:

1. **Escape analysis** (автоматически) — компилятор видит, что объект
   не утекает за пределы функции, оставляет на стеке или в арене
   вызова. Никаких аллокаций в managed heap. Работает для большинства
   локальных значений.

2. **`region { ... }`** ([D6](#d6-память-managed-по-умолчанию-regions-opt-in-для-real-time))
   — явная opt-in зона без GC, для real-time hot path.

```nova
fn process_audio(samples []f32) Realtime -> []f32 =>
    region {
        let buf = []f32.with_capacity(1024)
        // обработка, без GC pauses
        buf.to_owned()
    }
```

Никаких `&T` borrow, никаких lifetime-аннотаций в обычном коде.

### Сводка

| Форма параметра | Передача | Мутация видна снаружи |
|---|---|---|
| `x int` (примитив) | by value | нет (примитив всегда копия) |
| `mut x int` | by value | нет (локальная копия) |
| `o Order` (объект) | managed reference | нет (immutable view) |
| `mut o Order` | managed reference | да |

### Что отвергнуто

**By-value для всех типов (Go-стиль):**
- Копирование больших structs дорого
- Несовместимо с managed heap (D6) — объекты уже в куче, копировать
  бессмысленно
- Программист удивляется «изменил поле, не сохранилось» при не-pointer
  receiver

**By-reference с обязательной explicit мутацией (Rust-стиль `&mut`):**
- Слишком много синтаксиса для прикладного кода
- В Nova есть `mut` префикс уже для других случаев — расширяем его

**Move-семантика (Rust для не-Copy):**
- Сложная для прикладного программиста
- Не нужна с GC — нет ownership-проблем

**Borrow `&T` (Rust-стиль):**
- Скопирован в раннем дизайне рефлекторно. Перевзвешен — **не нужен**.
- В Rust borrow существует потому что нет GC; единственная альтернатива
  move/clone. В Nova с GC передача = указатель, никакого move/clone.
- Escape analysis закрывает большинство perf-кейсов автоматически.
- Slice уже передаётся как `(ptr, len, cap)` структура — дешёво,
  без отдельного borrow.
- Lifetime checker — research-уровень. Цена реализации высокая,
  выгода низкая для прикладного языка с GC.
- Прецедент Go: нет borrow, инфраструктура интернета на нём работает.

### AI-first аргумент

Сигнатура показывает контракт:

```nova
fn deposit(mut acc Account, amount money)    // меняет acc
fn show(acc Account)                          // только читает
```

LLM, читая сигнатуру, **видит**:
- `mut` → функция мутирует, видимо вызывающему
- без `mut` → не мутирует

В Java/C# этого не видно — все объекты mutable references по
умолчанию. Программист должен помнить «эта функция меняет аргумент».

В Nova это в типе.

### Почему AI-first выигрывает

Возьмём `process(order Order)` в Java:
- Меняет `order` или нет? Нужно читать тело
- Если LLM генерирует — может случайно мутировать, не зная

В Nova:
- `process(order Order)` — **гарантированно не мутирует**
- `process(mut order Order)` — **может мутировать**, программист
  принимает осознанно
- LLM не может «случайно» мутировать без `mut` в сигнатуре —
  compile error

### Связь с другими решениями

- **Согласовано с [D6](#d6-память-managed-по-умолчанию-regions-opt-in-для-real-time)** —
  managed heap делает by-reference дешёвым (передача указателя).
- **Согласовано с [D2](#d2-эффекты-вместо-ключевых-слов-asyncthrowsunsafe)** —
  мутация через `Mut` эффект; `mut` параметра + `Mut` эффект в
  сигнатуре — два уровня контроля. См. также [D28](#d28-вывод-эффектов-private--выводится-public--обязательно-явно).
- **Согласовано с [D10](#d10-революционная-ставка-всё--эффект--ai-first)** —
  AI-first видимость в типах.
- **Использует [D17](#d17-объявление-типов-единый-синтаксис-без-)** —
  `mut` поля в типе работают вместе с `mut` параметрами.

### Цена

1. **Программистам с C/Go-фоном** придётся привыкнуть, что объекты
   передаются по ссылке без `*`. Но это **уже привычно** Java/C#/JS/
   Python-программистам.
2. **Программистам с Rust-фоном** покажется упрощённым — нет move,
   нет `&mut`. Это **намеренно** — Nova прикладной язык.
3. **Один синтаксический префикс `mut` для разных вещей** (поле, let,
   параметр). Это **последовательно** — везде «mut = разрешена
   мутация», не разные понятия.

## D33. `const` vs `let` — compile-time vs runtime

**Решение.** `const` — для **compile-time констант**, известных на
этапе компиляции. `let` — для **runtime значений** (с опциональным
`mut`). Это два разных ключевых слова с разной семантикой, не сахар.

### Базовое использование

```nova
// const — compile-time константы
const MAX_PAYLOAD = 4096               // числовой литерал
const PI = 3.14159                      // константа-литерал
const TIMEOUT_SEC = 60 * 5              // арифметика над литералами
const GREETING = "hello"                // строковый литерал

// let — runtime значения (immutable binding)
let now = Time.now()                    // вычисляется в runtime
let user = Db.find(user_id)?            // runtime + эффекты

// let mut — runtime mutable
let mut counter = 0
counter += 1
```

### Правила

**`const` требует:**
- Compile-time computable выражения справа: литералы, арифметика над
  литералами, конструкторы record/sum-type из const-значений
- **Не разрешены** runtime-вызовы, эффекты, обращения к не-const
  значениям
- Тип выводится автоматически из значения

**`let` (без `mut`):**
- Связывает имя со значением, вычисленным **в runtime**
- Имя нельзя переприсвоить (immutable binding)
- Объект, на который указывает имя, может быть мутирован если поля
  `mut` и переменная объявлена `let mut`

**`let mut`:**
- Можно переприсвоить переменную: `counter = 5`
- Поля объекта мутируются если поле объявлено `mut` ([D32](#d32-семантика-передачи-параметров))

### Где живёт значение

- **`const`** — в data-segment бинарника (compile-time встраивается
  в код или загружается из секции `.rodata` по C-аналогии). **Не
  занимает heap**.
- **`let` примитивы** — в стеке или регистрах
- **`let` объекты** — в managed heap (или на стеке через escape
  analysis, [D6](#d6-память-managed-по-умолчанию-regions-opt-in-для-real-time))

### Что отвергнуто

**`:=` в стиле Go (объявление-присваивание одной операцией):**
- Дублирует `let`, два способа сделать одно
- Известный источник багов в Go (shadowing-проблемы)
- AI-first против — два синтаксиса с тонкой разницей

**`final` в стиле Java:**
- Лишнее ключевое слово рядом с `let`
- `let` уже даёт immutable binding

**Без разделения `const`/`let`:**
- Без `const` массивы фиксированного размера `[N]T` ([D27](#d27-синтаксис-массивов-t-префикс-nt-для-фиксированных))
  не имеют способа задать `N` через имя — пришлось бы вписывать
  числа литералами всюду
- Generic-параметры от чисел потребуют compile-time константы рано
  или поздно
- `let MAX = 100` живёт в data-segment всё равно — `const` явно
  отражает это в коде

### Что **не** в v1.0 — `const fn`

`const fn` (функции, вычислимые на этапе компиляции) — отложено до
[Q7 macros/comptime](spec/open-questions.md). До этого `const` ограничен:

- Литералы (числа, строки, bool, char)
- Арифметика над литералами и `const`-значениями
- Конструкторы record/sum-type из `const`-значений

```nova
const HEADER_SIZE = 16
const PAYLOAD_SIZE = 4096
const MAX_FRAME = HEADER_SIZE + PAYLOAD_SIZE     // ок, арифметика над const

const NOW = Time.now()                            // ОШИБКА: runtime-вызов
const USER = Db.find(1)?                          // ОШИБКА: эффекты
```

Когда зафиксируется comptime — `const fn` добавится без breaking
changes (аддитивно).

### Где можно использовать `const`

- **На уровне модуля** — основной случай, для пакетных констант
- **Внутри функции** — для magic numbers внутри тела
- **С `export` префиксом** — экспортируется из модуля ([D29](#d29-модули-и-импорты))

```nova
module config
export const MAX_PAYLOAD = 4096                  // экспорт
const INTERNAL_TIMEOUT = 30                       // приватная

fn process(data []u8) =>
    const THRESHOLD = 100                         // локальная
    if data.len > THRESHOLD { ... }
```

### AI-first аргумент

Сигнатура использования сразу говорит:

- `const X` — программист уверен, значение известно на этапе
  компиляции, никаких runtime-зависимостей
- `let X` — runtime значение, может зависеть от эффектов

LLM, читая `const MAX = compute_max()`, видит **compile error** —
не может вызвать функцию в `const`. Это явный сигнал «вы хотите
runtime значение → используйте `let`».

### Связь с другими решениями

- **Согласовано с [D27](#d27-синтаксис-массивов-t-префикс-nt-для-фиксированных)** —
  `const` нужен для размеров фиксированных массивов `[N]T`
- **Согласовано с [D29](#d29-модули-и-импорты)** — `export const`
  экспортирует константу из модуля
- **Согласовано с [D30](#d30-стиль-именования)** — константы
  пишутся `SCREAMING_SNAKE_CASE`
- **Не блокирует [Q7 comptime/macros](spec/open-questions.md)** — `const fn`
  добавится аддитивно когда comptime готов

### Цена

1. **Два ключевых слова** вместо одного. Но семантика чёткая,
   программист не путается.
2. **`const` ограничен литералами** в v1.0 — некоторые случаи
   потребуют `let` вместо `const` (например, `Duration.seconds(30)`
   до зафиксированного `const fn`).

## D34. `if let` и `while let` для pattern matching в условии

**Решение.** В Nova есть синтаксис `if let pattern = expr { ... }` и
`while let pattern = expr { ... }` — pattern matching прямо в условии,
с локальным binding в скоупе блока. Несколько условий через запятую.

### Базовое использование

```nova
// если Some — извлечь и использовать
if let Some(user) = cache.get(key) {
    process(user)
}

// если Ok — process; иначе ветка else
if let Ok(user) = Db.find(id) {
    process(user)
} else {
    Log.warn("user not found")
}

// while let — итерация пока паттерн совпадает
while let Some(item) = queue.pop() {
    process(item)
}

// несколько условий через запятую
if let Some(user) = lookup(id), user.is_active {
    process(user)
}

// else if let
if let Some(user) = lookup(id) {
    process(user)
} else if let Some(default) = config.default_user {
    process(default)
} else {
    handle_no_user()
}
```

### Грамматика

```
if-expr   := "if" if-cond ("," if-cond)* block ("else" (if-expr | block))?
while-expr := "while" if-cond ("," if-cond)* block

if-cond := "let" pattern "=" expr      // pattern-binding
         | expr                         // обычное condition
```

### Скоуп локальных переменных

Связанные через `let` имена доступны **только в теле блока**. После
закрывающей `}` — недоступны:

```nova
if let Some(user) = lookup(id) {
    process(user)         // ок, user в скоупе
}
process(user)             // ОШИБКА: user не в скоупе
```

В `else if let` — каждый блок имеет свои binding'и:

```nova
if let Some(a) = lookup_a() {
    use(a)                // a доступна
} else if let Some(b) = lookup_b() {
    use(b)                // b доступна, a НЕТ
}
```

### Совместимость с `?`

```nova
if let user = Db.find(id)? {
    // user типа User (не Option/Result)
    process(user)
}
```

`?` работает как обычно — пробрасывает ошибку наверх. Если ошибка
пробросилась — внутрь блока не зашли.

### Какие use-cases закрывает

**1. «Получить и использовать если есть»** — Option/Result без
`match`-блока:

```nova
// СТАРО (только match)
match cache.get(key) {
    Some(data) => return data
    None       => ()
}

// НОВО
if let Some(data) = cache.get(key) {
    return data
}
```

**2. Эквивалент Go `if val, err := f(); err != nil { ... }`** —
скоуп переменной = тело if, не вся функция:

```nova
// Go
if user, err := db.Find(id); err == nil {
    process(user)
}

// Nova
if let Ok(user) = Db.find(id) {
    process(user)
}
```

**3. Условные циклы** — итерация пока есть значение:

```nova
while let Some(line) = reader.read_line()? {
    process(line)
}
```

### Что отвергнуто

**Go-стиль `;`-разделитель:**
```nova
if user = Db.find(id); user.is_active { ... }    // ОТВЕРГНУТО
```

`;` нарушает D17 «один разделитель — запятая». И `=` без `let`
создаёт неоднозначность с присваиванием.

**Полный `:=` оператор** (объявление+присваивание):
- Уже отвергнут ранее (дублирует `let`)
- В Go он есть и **используется в каждой второй функции** — проблема
  shadowing. Nova избегает.

**Smart-cast** (Kotlin-стиль):
```nova
let x = lookup(id)
if x != None {
    use(x.unwrap())          // x всё ещё Option, нужен unwrap
}
```

В Kotlin компилятор «суживает» тип `x` после проверки `!= null`. Это
**магия в типе** — нет в синтаксисе. AI-first против. В Nova
программист **явно пишет** `if let Some(x) = ...` — паттерн виден.

**Без `let` после `if`:**
```nova
if Some(user) = lookup(id) { ... }    // ОТВЕРГНУТО
```

Без явного `let` парсер не отличит binding от сравнения. `let`
обязателен — согласовано с обычным `let`-объявлением ([D33](#d33-const-vs-let--compile-time-vs-runtime)).

### Прецедент

Rust, Swift, Kotlin (через smart-cast) — все имеют этот паттерн.
**Rust** ближе всего к Nova-синтаксису:

```rust
// Rust
if let Some(user) = lookup(id) {
    process(user);
}

while let Some(item) = queue.pop() {
    process(item);
}
```

Один-в-один. Программисты с Rust-фоном привыкают мгновенно.

### AI-first аргумент

Сигнатура if-условия показывает что происходит:

- `if let Some(user) = ...` — **паттерн-матч** с локальным binding
- `if user.is_active` — **обычная проверка**
- `if let Ok(user) = ..., user.is_active` — комбинация

LLM, читая код, **видит** что `user` создаётся внутри if и **только
там** доступна. Никаких смарт-кастов, никакой магии в типе.

### Связь с другими решениями

- **Согласовано с [D17](#d17-объявление-типов-единый-синтаксис-без-)** —
  pattern matching уже работает в `match`, расширяем на `if`/`while`
- **Согласовано с [D33](#d33-const-vs-let--compile-time-vs-runtime)** —
  `let` для runtime binding, скоуп локален. Здесь то же.
- **Согласовано с принципом «минимум строк на выходе»** ([D31](#d31-handler-лямбда-для-эффектов-с-одной-операцией))
  — закрывает частый паттерн без `match`-блока

### Цена

1. **Дополнительная грамматическая форма** — `if let pattern = expr`.
   Не сложно для парсера, но нужно зафиксировать.
2. **Программисту нужно знать**, что `if let` и `if` — две формы.
   Привыкается за день из Rust-документации.

## D35. Методы инстанса через `@`. `self` отменён

**Решение.** Методы инстанса объявляются через `fn Type @method(...)`
с **неявным self**. Доступ к полям — через `@field` (сахар для
self.field). `mut` для мутирующих методов — `fn Type mut @method(...)`.
Конструкторы и static-функции — через точку `fn Type.name(...)` без
`@`. Ключевое слово `self` отменено.

### Базовое использование

```nova
type Account {
    readonly owner str   // immutable invariant (D36)
    _balance money       // мутируется у mut-binding (D36)
}

// конструктор/static — через точку, без @
fn Account.new(owner str) -> Account =>
    Account { _balance: 0, owner: owner }

fn Account.from_json(json str) Throws[ParseError] -> Account =>
    let parsed = Json.parse(json)?
    Account { _balance: parsed.balance, owner: parsed.owner }

// метод инстанса — через пробел и @, неявный self
fn Account @balance() -> money => @_balance

fn Account @is_solvent() -> bool => @_balance > 0

fn Account @summary() -> str =>
    "${@owner}: ${@_balance}"

// мутирующий метод — mut перед @name
fn Account mut @deposit(amount money) =>
    @_balance += amount

fn Account mut @withdraw(amount money) Throws[Overdraft] -> () =>
    if amount > @_balance { throw Overdraft }
    @_balance -= amount
```

Использование:

```nova
let acc = Account.new("alice")
acc.deposit(100)              // вызов метода — через точку, со скобками
acc.deposit(50)
let bal = acc.balance()       // getter, обязательные скобки
let ok = acc.is_solvent()
```

### Грамматика

```
fn-decl := "fn" name-decl
name-decl := free-fn | static-method | instance-method

free-fn          := identifier "(" params ")" effects? "->"-type? "=>" body
static-method    := Type "." identifier "(" params ")" effects? "->"-type? "=>" body
instance-method  := Type ("mut")? "@" identifier "(" params ")" effects? "->"-type? "=>" body

Type := PascalCase identifier ("[" generics "]")?
```

Парсер однозначен после имени типа:
- `.` → static-method или constructor
- `@` или `mut @` → instance-method (с неявным self)

### Доступ к полям внутри метода

Внутри `@method` (или `mut @method`) поля доступны через `@field`:

```nova
fn Account @summary() -> str =>
    "${@owner}: ${@_balance}"          // эквивалент self.owner, self._balance
```

**`@field` — единственная форма.** Точка не работает: `@.field`
**невалидно**, `@field` — единственно верно.

`@` без поля — это **значение текущего инстанса** (`self`):

```nova
fn Account @copy() -> Account => @                // вернуть копию инстанса
fn Account @send_to_channel(ch Channel[Account]) => ch.send(@)
```

### `self` как ключевое слово отменено

В предыдущих версиях дизайна метод объявлялся как
`fn Account.method(self, ...)` с явным `self`. Теперь `self` **не
существует** в Nova:

- В сигнатуре self **не передаётся как параметр** — он маркирован
  через `@method` в декларации
- В теле метода **`@field` единственная форма** доступа к полям
- Старая запись `self.field` — **синтаксическая ошибка**

Если программист хочет alias-имя для текущего инстанса (например,
для clarity), используется `let`:

```nova
fn Account @complex() -> int =>
    let me = @                        // явный alias
    me.foo() + me.bar()
```

### Передача метода как значения

Bound method (захватывает self из инстанса):

```nova
let f = acc.balance              // тип: fn() -> money
f()                               // == acc.balance()
```

Unbound method (self передаётся явно как первый аргумент):

```nova
let g = Account.@balance         // тип: fn(Account) -> money
g(acc)                            // == acc.balance()
```

`Type.@method` — это **обращение к instance-методу как значению**.
Точка-собака читается как «instance-method типа Type». Различается
от static:
- `Account.new` — static-функция, тип `fn(str) -> Account`
- `Account.@balance` — instance-метод как unbound, тип `fn(Account) -> money`

### Чёткое разделение constructor / instance method

Программист и LLM видят:

```nova
fn Account.new(...) -> Account => ...      // constructor — точка, без @
fn Account.from_json(...) -> Account => ...  // static factory — точка, без @
fn Account @balance() -> money => ...       // instance — пробел, @
fn Account mut @deposit(...) => ...         // mutating instance — mut + @
```

Первый символ после имени типа определяет вид:
- `.` → static (constructor, factory, helper)
- `@` или `mut @` → instance method

### Generic'и

```nova
// generics на типе
fn Vec[T].new() -> Vec[T] => ...
fn Vec[T] @len() -> int => ...
fn Vec[T] @push(item T) -> () => ...

// generics на методе
fn Vec[T] @map[U](f T -> U) -> Vec[U] => ...
fn Vec[T] @fold[Acc](init Acc, f (Acc, T) -> Acc) -> Acc => ...
```

`[T]` идёт сразу после имени типа (до точки/пробела). `[U]` — после
`@name`.

### Что отвергнуто

**Сохранение `fn Type.method(self, ...)`** — текущий вариант:
```nova
fn Account.deposit(mut self, amount money) => self.balance += amount  // ОТМЕНЕНО
```

Минусы:
- Длиннее на 9 символов на метод
- `self` повторяется в каждом методе
- `self.field` повторяется в каждом обращении к полю

**Property-механизм** (C#/Kotlin):
```nova
property balance money { get => @_balance; set(new) => @_balance = new }   // ОТВЕРГНУТО
```

Минусы:
- Невидимо: `acc.balance` это поле или вызов?
- Дополнительная концепция в языке
- Прецедент C# — известный источник путаницы

В Nova вместо property — **обычные методы со скобками**:
```nova
fn Account @balance() -> money => @_balance
fn Account mut @set_balance(new money) => @_balance = new

acc.balance()             // явно вызов
acc.set_balance(100)
```

**`@` как параметр** (вариант обсуждался):
```nova
fn Account.deposit(mut @, amount money) => @balance += amount   // ОТВЕРГНУТО
```

Минусы:
- `@` имеет два смысла (параметр в сигнатуре + префикс к полю в теле)
- `(@)` визуально странно для одного-параметра-self

**`fn mut @Type.method(...)`** (`@` перед типом):
```nova
fn mut @Account.deposit(amount money) => @balance += amount   // ОТВЕРГНУТО
```

Минусы:
- `mut` маркирует тип-receiver, а не имя binding'а — расширяет правило
- `@Type` — `@` относится к типу, в теле `@field` — `@` относится к
  полю; разные смыслы

**`fn Type @method` без точки для constructor:**
```nova
fn Account new(owner str) -> Account => ...   // ОТВЕРГНУТО
```

Минусы:
- Расходится с namespace path (`module a.b`, `Color.Red`, `Some(...)`)
- Парсер сложнее различать static и instance

Текущее решение **разделяет**:
- **Точка** = static-функция в namespace типа
- **Пробел + `@`** = метод инстанса с неявным self

Семантическая разница отражается синтаксически.

### AI-first аргумент

LLM, читая декларацию:
- `fn Account.new(...)` → знает: статическая функция, без self
- `fn Account @method(...)` → знает: метод инстанса, self неявный
- `fn Account mut @method(...)` → знает: мутирующий метод

Внутри тела:
- `@field` → поле текущего инстанса
- `name` → локальная переменная или параметр
- `Type.foo` → static-функция
- `obj.foo()` → метод инстанса

Все три уровня **различимы по синтаксису без знания контекста**.
Это AI-friendly.

### Связь с другими решениями

- **Развивает [D32](#d32-семантика-передачи-параметров)** — `mut` для
  параметров остаётся как есть; для методов добавлен `mut @method`
- **Согласовано с [D29](#d29-модули-и-импорты)** — `Type.member` через
  точку для namespace path; instance-методы — другой случай (через `@`)
- **Согласовано с [D17](#d17-объявление-типов-единый-синтаксис-без-)**
  — `Color.Red` constructor через точку; static-функции — тот же
  паттерн
- **Согласовано с принципом «минимум строк»** ([D31](#d31-handler-лямбда-для-эффектов-с-одной-операцией))
  — экономия 6-9 символов на каждый метод
- **Поддерживает [D10](#d10-революционная-ставка-всё--эффект--ai-first)**
  — `@` маркер метода инстанса виден в сигнатуре, локальность
  контекста сохраняется

### Цена

1. **Большая правка примеров.** ~30 мест с `mut self` / `(self)`
   нужно переписать на новый синтаксис. Включая [audit.nv](examples/audit.nv),
   старый [paradigm.md](spec/paradigm.md) (Q8) и сноски в decisions.md.
2. **Расхождение с Rust/Swift синтаксисом.** Программистам с этого
   фона — переучивание. Цена недели.
3. **Новая концепция `@`** — единый смысл «принадлежит self», но
   программист и LLM должны это знать.
4. **Cross-paste из других языков сложнее.** Метод из Rust:
   `fn deposit(&mut self, amount: u64) { self.balance += amount }` →
   в Nova: `fn Account mut @deposit(amount money) => @balance += amount`.
   Не катастрофа, но не 1:1.

## D36. Поля типа: дефолт mutable у `mut` binding'а, `readonly` для never-mut

**Решение.** Пересмотр семантики полей типа из [D32](#d32-семантика-передачи-параметров).
Новое правило:

- **Поле без префикса** — мутируется, **если binding mutable** (`let
  mut acc` или `mut acc Account`). По умолчанию это **большинство
  полей**.
- **Поле с `readonly` префиксом** — **никогда не мутируется**, даже
  у mutable binding'а. Для id, foreign keys, immutable invariants.
- **Поле с `mut` префиксом** — **мутируется всегда**, даже у
  immutable binding'а. Для cache, lazy init, atomic counters
  (аналог C++ `mutable`). Редкий случай.

Также: **group-syntax** для полей одного типа — `a, b, c int` вместо
трёх строк.

### Базовое использование

```nova
// Аккумулятор — все поля мутируемые, никаких префиксов не нужно
type RunAcc {
    att_wins int, def_wins int, draws int
    total_rounds int
    total_moon_chance f64
    atk_lost_m int, atk_lost_s int, atk_lost_h int
}

let mut acc = RunAcc { att_wins: 0, def_wins: 0, ... }
acc.att_wins += 1                  // ок — binding mut, поле без `readonly`

// Структура с invariant'ами — `readonly` для read-only полей
type Account {
    readonly id u64                 // никогда не меняется
    readonly owner str              // тоже
    balance money                    // мутируется у `mut` binding'а
    closed bool                      // тоже
}

let acc = Account.new("alice")
acc.balance = 100                   // ОШИБКА: binding не mut

let mut acc2 = Account.new("alice")
acc2.balance = 100                  // ок
acc2.id = 999                       // ОШИБКА: id объявлено readonly

// Cache/lazy — `mut` для полей, мутируемых через immutable binding
type LazyConfig {
    path str
    mut cached_value Option[str]    // обновляется при первом read
}

fn LazyConfig @get() -> str =>
    if let Some(v) = @cached_value { return v }
    let v = read_file(@path)
    @cached_value = Some(v)         // мутация через @-метод даже у let-binding
    v
```

### Group-syntax

Несколько полей одного типа — через запятую:

```nova
type Point { x, y, z f64 }                          // три f64
type Color { r, g, b u8 }                           // три u8
type RunAcc {
    att_wins, def_wins, draws int                    // три int
    atk_lost_m, atk_lost_s, atk_lost_h int           // три int
    atk_lost_pts, def_lost_pts f64                   // два f64
}
```

С префиксами:

```nova
type Account {
    readonly id, owner_id u64       // два immutable
    balance money                    // дефолт (mutable у mut-binding)
    mut last_access_time time        // mutable всегда
}
```

### Сводная таблица

| Объявление поля | Mutable у `let acc` | Mutable у `let mut acc` | Use case |
|---|---|---|---|
| `field T` (без префикса) | нет | **да** | большинство полей |
| `readonly field T` | **никогда** | **никогда** | id, immutable invariants |
| `mut field T` | **да** | **да** | cache, lazy init, atomic counters |

### Что отвергнуто

**Старая семантика D32** (поле `mut` мутируется только у `mut`-binding):
- Заставляла **писать `mut` перед каждым полем** для аккумуляторов
- 18 раз `mut` в одном типе — шум без пользы
- Семантически: `mut` per-field = «это поле особенное», но если все
  поля mut — особенности нет

**Rust-полное (поле всегда mutable у mut-binding, нет never-mut):**
- Невозможно зафиксировать read-only invariant (id, foreign key)
- Workaround: приватное поле + getter — лишняя возня

**`type X mut { ... }`** (mut на тип):
- Один маркер вместо 18 — короче
- Но: что если 90% mut, 10% нет? Нужно `mut` на тип + опт-аут per
  field. Усложнение.
- Конфликт с современным паттерном «struct + immutable defaults +
  явная мутация» из Swift/Rust

**`final` (Java-стиль) для never-mut полей:**
- Короче на 3 символа (`final` vs `readonly`)
- Прецедент Java/Dart/Kotlin
- Минусы: семантически перегружен — в Java `final method`, `final
  class`, `final var`, `final field` — программист должен помнить
  контекст. Перевешивает однозначность `readonly`.

**`let` для never-mut полей (рассматривался):**
- Короче (3 символа)
- Прецедент Swift
- Минусы: `let` уже значит «binding имени со значением» (D33).
  На поле без `=` — необычно, не самообъясняемо. `readonly` прямо
  говорит «только для чтения».

**`const` (C++-стиль):**
- Конфликт с [D33](#d33-const-vs-let--compile-time-vs-runtime) —
  там `const` = compile-time константа. Здесь — runtime-immutable.
  Перегрузка термина, AI-first против. **Невозможно.**

### AI-first аргумент

Сигнатура типа теперь показывает **свойства мутации** только для
исключений:

```nova
type Account {
    readonly id u64      // явно никогда не меняется
    readonly owner str   // тоже
    balance money        // обычное поле
    closed bool          // обычное
}
```

LLM, читая тип:
- `let id` → «не трогай, это invariant»
- `balance` → «можно мутировать с `mut`-binding'ом»

**Меньше шума, ключевая информация (`let` invariants) выделяется
визуально.** Сравни с D32, где надо было писать `mut balance`,
`mut closed`, и `id` без префикса означал «также не мутирует» —
неотличимо от обычного поля.

### Связь с другими решениями

- **Пересматривает [D32](#d32-семантика-передачи-параметров)** —
  семантика `mut` для полей типа. Передача параметров (`fn f(mut o
  Order)`) **остаётся**: `mut` на параметре = mutable binding,
  внутри — мутации полей по правилу D36.
- **Согласовано с [D33](#d33-const-vs-let--compile-time-vs-runtime)**
  — `let` это immutable binding. На поле тоже — immutable forever.
- **Согласовано с [D35](#d35-методы-инстанса-через--self-отменён)** —
  `fn Type mut @method` mutable-binding self, поля по правилам D36.
- **Прецедент Rust/Go/C++** — поля без префикса мутируются у
  mut-binding'а; `let` для never-mut близко к C++ `const` member.

### Цена

1. **Breaking change в D32** — все примеры с `mut field` нужно
   проверить. Большинство случаев `mut field` теперь = просто `field`
   (если все поля mut). Исключение — `mut` для cache/lazy (редко).
2. **Программистам, привыкшим что Nova "явная везде"** — может
   показаться расслабленным. Но **`let` остаётся явным** для
   important invariants.
3. **Group-syntax** — новая концепция. Аддитивно, не ломает ничего.

## D37. Доступ к полям: `.name` для record, `.N` для позиционных и кортежей

**Решение.** Доступ к полю объекта или элементу позиционной структуры
— через точку:

- **`obj.name`** — поле record по имени
- **`obj.0`, `obj.1`, ...** — поле позиционной структуры или кортежа
  по индексу (0-based)
- **`@name`, `@0`, `@1`** — то же внутри методов инстанса (для self,
  по [D35](#d35-методы-инстанса-через--self-отменён))

### Базовое использование

```nova
// Record — доступ по имени
type User { id u64, name str, email str }

let u = User { id: 1, name: "alice", email: "alice@x.com" }
println(u.name)          // "alice"
println(u.email)         // "alice@x.com"

// Позиционная структура — доступ по индексу
type Point(f64, f64)

let p = Point(1.0, 2.0)
println(p.0)             // 1.0
println(p.1)             // 2.0

// Кортежи — то же
let pair = (1, "alice")
println(pair.0)          // 1
println(pair.1)          // "alice"

let triple = (10, "x", true)
println(triple.2)        // true
```

### Внутри методов инстанса (через `@`)

Согласовано с [D35](#d35-методы-инстанса-через--self-отменён) — `@name`
для record-полей self, `@0`/`@1` — для позиционных:

```nova
type Point(f64, f64)

fn Point @magnitude() -> f64 =>
    math.sqrt(@0 * @0 + @1 * @1)        // позиционные поля self

type Account { balance money, owner str }

fn Account @summary() -> str =>
    "${@owner}: ${@balance}"             // record-поля self
```

### Mutation

При `mut`-binding поля можно записывать (по [D36](#d36-поля-типа-дефолт-mutable-у-mut-bindinga-readonly-для-never-mut)):

```nova
let mut p = Point(1.0, 2.0)
p.0 = 5.0                // ок, позиционное поле, mut binding

let mut u = User { id: 1, name: "alice", email: "..." }
u.name = "bob"           // ок, record, mut binding и поле без readonly
```

Не работает для `readonly` полей (D36) и без `mut` binding'а.

### Pattern matching — альтернатива доступу

Pattern matching работает как раньше ([D17](#d17-объявление-типов-единый-синтаксис-без-)):

```nova
match p {
    Point(x, y) => x + y                 // деструктуризация позиционной
}

let Point(x, y) = p                       // let-pattern, деструктуризация
let (a, b, c) = triple                    // деструктуризация кортежа
```

Программист выбирает: `.0`/`.1` для **точечного доступа**, pattern
matching для **извлечения нескольких полей сразу**.

### Парсер — однозначность `.N`

Между идентификатором (или закрывающей скобкой выражения) и `.N` —
это **доступ к полю**, не decimal:

```nova
p.0                      // p — идентификатор, .0 — поле 0
0.0                      // числовой литерал f64
(get_pair()).0           // выражение в скобках, .0 — поле
1.5                      // числовой литерал
1.foo                    // ОШИБКА: после литерала идентификатор
                         //   — парсер не знает, число это или
                         //   `1` метод вызова `.foo`. Запретить.
```

Правило: `.N` или `.name` после **выражения, которое не литерал
числа** — это field access. После числового литерала точка только
для decimal или ничего.

### Многократный доступ

```nova
type Tree { children []Tree, value int }
let root = Tree { children: [...], value: 1 }
println(root.children.0.value)            // первый ребёнок, его value

type Pair = (Tree, Tree)
let pair = (left_tree, right_tree)
println(pair.0.value)                     // value первого дерева
```

### Что отвергнуто

**Только pattern matching, без direct access:**
```nova
let Point(x, _) = p; use(x)              // ОТВЕРГНУТО как единственная форма
```
Слишком многословно для простых случаев. Прецедент Haskell (там
есть `fst`/`snd` для пар) показывает: программисты всё равно создают
утилиты для direct access.

**Функции-аксессоры (Haskell-style `fst`, `snd`):**
- Работает только для пар
- Не масштабируется для 3+ кортежей
- Конфликтует с прецедентом современных языков

**Только именованные record (без позиционных):**
```nova
type Point { x f64, y f64 }              // ОТВЕРГНУТО как единственный вариант
```
Кортежи нужны для **временных** группировок (возврат пары из функции,
итератор `(K, V)`). Объявлять тип для каждой пары — шум.

**`obj[0]` (TypeScript array-style):**
```nova
pair[0]                                   // ОТВЕРГНУТО
```
- Конфликт с индексацией массивов: `arr[0]` это runtime-операция с
  возможной ошибкой выхода за границы. `pair.0` — compile-time
  проверка, всегда безопасна.
- В Rust/Swift `.0` — стандарт, AI-friendly.

### Связь с другими решениями

- **[D17](#d17-объявление-типов-единый-синтаксис-без-)** — позиционные
  структуры объявляются как `type Point(f64, f64)`. D37 описывает
  как к ним обращаться вне pattern matching.
- **[D35](#d35-методы-инстанса-через--self-отменён)** — `@name`/`@0`
  внутри методов инстанса.
- **[D36](#d36-поля-типа-дефолт-mutable-у-mut-bindinga-readonly-для-never-mut)**
  — правила мутации работают одинаково для именованных и позиционных.

### Цена

1. **`.0`, `.1`** не self-документированы — программист не помнит
   что значит `p.0` без знания типа `Point`. Аргумент против:
   позиционные структуры **редко используются** для важных доменных
   типов. Для важных — record с именованными полями. Кортежи —
   локальные временные группировки, где `.0`/`.1` нормально.
2. **Парсер чуть сложнее** — `.N` после идентификатора и после
   числового литерала различаются по контексту.

## D38. Создание массивов и turbofish для дженериков

**Решение.** Создание пустых массивов через **литерал** (с
annotation) или через **static-методы на типе массива**. Для случаев
когда type inference не справляется — **turbofish**: явная передача
generic-параметров через `[T]` после имени типа или функции.

### Создание массивов

**1. Литерал с annotation на `let`:**

```nova
let mut buckets []Slot[K, V] = []        // пустой, тип из annotation
let xs []int = [1, 2, 3]                  // непустой
```

Простейшая форма. Annotation **обязательна**, если массив пустой —
inference не может вывести тип элемента.

**2. Inference из контекста:**

```nova
fn first(xs []int) -> Option[int] => ...

let result = first([])                    // []: tип выводится из аргумента
```

Когда литерал передаётся в функцию или присваивается в типизированное
поле — компилятор выводит тип элемента из контекста.

**3. Static-методы на типе массива:**

```nova
let mut buckets = []Slot[K, V].with_capacity(cap)
let empty = []int.new()
let zeros = []u8.filled(0, 1024)
```

Тип явный. Полезно когда:
- Нужна pre-allocation (`with_capacity(n)`)
- Нет контекста для inference (например, в вычислении выражения,
  передаваемого в `?`)
- Код читается лучше когда тип явно сразу

stdlib-методы на массиве (предполагаемые в [Q9](spec/open-questions.md)
stdlib):
- `[]T.new() -> []T` — пустой
- `[]T.with_capacity(cap int) -> []T` — пустой с pre-allocation
- `[]T.filled(value T, count int) -> []T` — заполненный значением

### Turbofish — явные generic-параметры

Когда type inference не справляется, generic-параметры передаются
**после имени** в `[T]`:

```nova
// Generic-функция
fn parse[T](s str) Throws -> T => ...

let n = parse[int]("42")?                 // явно T = int

// Generic-тип со static-методом
let buckets = []Slot[K, V].with_capacity(16)

// Generic-конструктор record-типа
type Cache[K, V] { ... }
let c = Cache[str, int].new()             // явно K=str, V=int

// Generic-метод
fn HashMap[K, V] @get(key K) -> Option[V] => ...

let result = m.@get[int]("key")           // явно тип ключа (если нужно)
```

Грамматика turbofish — **то же самое `[T]`** что в декларации generic.
Согласовано с [D16](#d16-дженерики-через-t-не-t):

```
generic-application := identifier "[" type ("," type)* "]"
```

После идентификатора `[T]` в позиции выражения / типа = **передача
generic-параметра**. Это работает для:
- Функций (`parse[int]("42")`)
- Static-методов типа (`[]Slot[K, V].with_capacity(16)`)
- Конструкторов (`HashMap[str, int].new()`)

### Никакого `::` (Rust-стиль)

В Rust turbofish это `::<>`:

```rust
parse::<i32>("42")
Vec::<i32>::new()
```

Это **известная проектная ошибка** Rust — сами признают. Сделано чтобы
обойти `<T>` парсер-ambiguity. В Nova `[T]` — парсер однозначен ([D16](#d16-дженерики-через-t-не-t)),
**`::` не нужен**.

```nova
parse[int]("42")                          // ← без ::
[]Slot[K, V].with_capacity(16)            // ← без ::
HashMap[str, int].new()                   // ← без ::
```

### Сводка форм для пустого/пред-аллоцированного массива

| Форма | Когда использовать |
|---|---|
| `let xs []T = []` | Тип ясен из annotation, без pre-allocation |
| `let xs []T = [a, b, c]` | Известны элементы, type из annotation или inference |
| `[]T.new()` | Static-метод, пустой, тип явный |
| `[]T.with_capacity(n)` | Static-метод, с pre-allocation |
| `[]T.filled(v, n)` | Заполненный одним значением |

### Пример из stdlib HashMap

До D38 (неоднозначно):
```nova
fn new_buckets[K, V](cap int) -> []Slot[K, V] =>
    let mut buckets = [].with_capacity(cap)    // тип неясен
    ...
```

После D38 (два валидных варианта):

```nova
// Вариант A — annotation на let
fn new_buckets[K, V](cap int) -> []Slot[K, V] =>
    let mut buckets []Slot[K, V] = []
    for _ in 0..cap { buckets.push(Slot.Empty) }
    buckets

// Вариант B — turbofish + with_capacity
fn new_buckets[K, V](cap int) -> []Slot[K, V] =>
    let mut buckets = []Slot[K, V].with_capacity(cap)
    for _ in 0..cap { buckets.push(Slot.Empty) }
    buckets
```

**Вариант B предпочтительнее** для HashMap — `with_capacity` даёт
pre-allocation без resize при первых `push`.

### Что отвергнуто

**Rust-style `::<T>` (turbofish):**
- В Rust нужен из-за `<T>` ambiguity. Nova использует `[T]` (D16) —
  `::` не нужен.
- Вводить `::` только для одной конструкции — лишняя сущность.

**Implicit `make[T](n)` или `Array.new[T](n)`:**
- Глобальная функция `make` — Go-style, в Nova не вписывается
- `Array.new` подразумевает что массив это namespace `Array`. У нас
  `[]T` — встроенный синтаксис ([D27](#d27-синтаксис-массивов-t-префикс-nt-для-фиксированных)),
  не type `Array`. Static-методы вызываются на `[]T.method`, не на
  `Array.method`.

**`Vec[T].new()` (Rust-style имя):**
- В Nova `[]T` это **встроенный синтаксис** для динамических
  массивов, не отдельный type `Vec`. Альтернативное имя `Vec[T]`
  ввело бы дублирование.

### Связь с другими решениями

- **Согласовано с [D16](#d16-дженерики-через-t-не-t)** — generic
  применяется через `Имя[T]` без turbofish-операторов.
- **Использует [D27](#d27-синтаксис-массивов-t-префикс-nt-для-фиксированных)**
  — `[]T` это тип, у него можно вызывать static-методы через `.`.
- **Согласовано с [D35](#d35-методы-инстанса-через--self-отменён)** —
  `Type.method` для static (constructor), `Type @method` для
  instance. Здесь `[]T.with_capacity(n)` — static на типе массива.

### Цена

1. **`[]T.method()` — новая синтаксическая роль** для типа массива
   как namespace. Парсер должен распознать `[]T` или `[N]T` перед
   точкой.
2. **`[]Slot[K, V].with_capacity(16)`** визуально плотно — `[]` +
   generic + точка + метод. Терпимо, прецедент Rust `Vec::<T>::new`
   ещё хуже.
3. **stdlib должен предоставить** `new()`, `with_capacity(n)`,
   `filled(v, n)` для `[]T`. См. [Q9](spec/open-questions.md).

## D39. Embed и delegation: `use Type` и `use name Type`

**Решение.** Композиция типов через `use Type` внутри декларации
record. По умолчанию имя встроенного — **имя типа** (`Account` →
`@Account`). Для алиаса — `use name Type` (как обычное объявление
поля: `имя тип`).

### Базовое использование

```nova
type AuditedAccount {
    use Account                     // имя = "Account"
    audit_log []AuditEntry
}

let acc AuditedAccount = ...

// Прямой доступ к полям Account через AuditedAccount (auto-proxy)
println(acc.balance)                // = acc.Account.balance
println(acc.owner)                  // = acc.Account.owner

// Прямой вызов методов Account (auto-proxy)
acc.is_solvent()                    // = acc.Account.is_solvent()

// Доступ к встроенному объекту целиком
let just_account = acc.Account
```

### Алиас для имени

```nova
type Wrapper {
    use w HashMapIter[K, V]         // имя = "w"
    extra int
}

fn Wrapper mut @next() -> Option[K] =>
    match @w.next() {                // обращение через alias
        Some((k, _)) => Some(k)
        None         => None
    }
```

Грамматика согласована с остальной частью Nova: везде **имя тип**
(параметры функций, поля record, let-bindings, for-loop, embed).

### Конфликт имён — обязательный alias

Если два `use` вводят одинаковые имена методов, и оба не
переопределены в типе-обёртке — **compile error**:

```nova
type Logger { log(msg str) -> () }
type Auditor { log(msg str) -> () }

type Combined {
    use Logger
    use Auditor
}

let c = Combined { ... }
c.log("...")                        // ОШИБКА: ambiguous, есть в Logger и Auditor
```

Решение — alias или явный вызов:

```nova
// Alias
type Combined {
    use console Logger
    use audit Auditor
}

fn Combined @log_all(msg str) =>
    @console.log(msg)
    @audit.log(msg)

// Или явный вызов
let c = Combined { ... }
c.Logger.log("...")                 // явно Logger
c.Auditor.log("...")                // явно Auditor
```

### Override метода

Если тип-обёртка определяет метод с тем же именем — он **затмевает**
делегированный:

```nova
type AuditedAccount {
    use Account
    audit_log []AuditEntry
}

// Override deposit — добавляет audit-запись
fn AuditedAccount mut @deposit(amount money) =>
    @Account.deposit(amount)        // явный вызов «родителя»
    @audit_log.push(AuditEntry.deposit(amount))

let acc AuditedAccount = ...
acc.deposit(100)                    // вызывает AuditedAccount.deposit, не Account.deposit
```

Без `@Account.` в теле — **бесконечная рекурсия** (вызов сам себя).
Программист должен явно обращаться к встроенному.

### Что это НЕ

**Не наследование.** `AuditedAccount` не **является** Account:

```nova
fn process(a Account) -> () => ...

let aa AuditedAccount = ...
process(aa)                         // ОШИБКА: AuditedAccount это не Account
process(aa.Account)                 // ок: извлекли Account-часть
```

`use` это **delegation** (как Go embed), не subtyping. Подтипа нет.
Если нужен **полиморфизм** — структурный тип ([D15](#d15-интерфейсы-как-структурные-типы)):

```nova
fn process(a { balance() -> money }) -> () => ...
process(aa)                         // ок: AuditedAccount имеет balance()
                                     //  через delegation
```

**Не множественное наследование.** Можно `use` несколько типов, но
конфликты решаются alias'ом или explicit. Diamond-problem не
существует — нет иерархии.

### Прокси-методы — auto-generated

При `use Type` компилятор генерирует **прокси-методы** для каждого
метода `Type`:

```nova
type Account { balance money }
fn Account @balance_pct(of money) -> f64 => @balance / of * 100.0

type AuditedAccount { use Account, audit_log []AuditEntry }

// Компилятор генерирует:
// fn AuditedAccount @balance_pct(of money) -> f64 =>
//     @Account.balance_pct(of)

let aa AuditedAccount = ...
aa.balance_pct(1000.0)              // работает через auto-proxy
```

Это **zero-cost** — компилятор инлайнит вызов. Никакой vtable.

### Что отвергнуто

**`use Type as name` (Rust import-style):**
- `as` в Nova зафиксирован для **импортов** ([D29](#d29-модули-и-импорты))
  — там семантика «alias имени извне». В embed — «объявление поля».
  Разные случаи.
- `use name Type` согласован с **остальным синтаксисом** Nova:
  параметры (`fn f(x int)`), поля (`{ name str }`), let (`let mut acc
  Account`). Везде «имя тип».

**Только `use Type` без alias (Go embed):**
- Не покрывает конфликт имён без переименования встроенного типа.

**Subtyping:**
- Противоречит [D1](#d1-парадигма-traits--data-без-классов) — без
  наследования. Полиморфизм через структурные типы ([D15](#d15-интерфейсы-как-структурные-типы)).

**Множественное наследование:**
- Известный антипаттерн (diamond, fragile). Embed решает 80% задач
  без сложности.

### Связь с другими решениями

- **Развивает [D1](#d1-парадигма-traits--data-без-классов)** —
  наследования нет, есть delegation через `use`.
- **Использует [D17](#d17-объявление-типов-единый-синтаксис-без-)**
  — `use` внутри `type Name { ... }` блока.
- **Согласовано с [D35](#d35-методы-инстанса-через--self-отменён)**
  — `@Type.method()` или `@alias.method()` для вызова из метода
  обёртки.
- **Согласовано с [D32](#d32-семантика-передачи-параметров)** —
  везде в Nova порядок `имя тип`.
- **Закрывает open-question** «Точная грамматика для embed/delegate
  при конфликте имён методов» из ранних версий decisions.md.

### Цена

1. **Auto-proxy methods** — компилятор должен генерировать. Не
   сложно, но требует resolve фазы.
2. **Конфликт имён ловится в compile-time** — программист видит
   ошибку и пишет alias. Не silent — нормальное поведение.
3. **Generic'и в embed** — `use HashMapIter[K, V]` — параметры
   передаются как обычно. Парсер уже умеет [generic-application
   D38](#d38-создание-массивов-и-turbofish-для-дженериков).

## D40. Тело функции: `=>` для одного выражения, `{}` для блока

### Решение

Два **взаимоисключающих** способа задать тело функции/метода:

1. **Expression-body** — `=> expr`, ровно одно выражение.
2. **Block-body** — `{ stmt; ...; expr }`, последовательность шагов;
   значение блока = последнее выражение (если оно есть).

Indentation **не значим**. Перевод строки и отступы — оформление, не
синтаксис. Компилятор не догадывается «всё, что отступлено сильнее
сигнатуры — тело функции».

### Грамматика

```
fn-decl    = 'fn' name '(' params ')' [effects] ['->' type] body
body       = '=>' expression
           | block
block      = '{' { statement } [ expression ] '}'
```

`=>` и `{` **никогда не сочетаются**: ни `fn f() => { ... }`,
ни `fn f() { => x }`.

### Примеры

```nova
// expression-body
fn double(x int) -> int => x * 2
fn Account @balance() -> money => @_balance
fn HashMap[K, V].new() -> HashMap[K, V] =>
    HashMap[K, V].with_capacity(16)        // одно выражение, перенесённое

// block-body
fn next_pow2(n int) -> int {
    if n <= 1 { return 1 }
    let mut p = 1
    while p < n { p *= 2 }
    p
}

fn process(r Request) Db Log -> Response {
    let user = load_user(r.user_id)?
    log("processing ${user.name}")
    build_response(user)
}
```

`if`/`match`/`block` сами по себе — выражения, поэтому такое валидно
как expression-body:

```nova
fn classify(n int) -> str => match n {
    0 => "zero",
    n if n > 0 => "positive",
    _ => "negative",
}

fn abs(x int) -> int => if x < 0 { -x } else { x }
```

Здесь `=>` пишется один раз — открывает тело функции, а внутри `match`
и `if` — это разные `=>` (стрелка ветви match) и обычное выражение if.

### Почему строго

Альтернатива — позволить `=> indented-block` (как F# `let f x =`,
OCaml `let f x =`, Python `def f():`) — отвергнута:

- **Indentation-significant грамматика** ломает copy-paste, требует
  единого настроя на табы/пробелы, плохо переживает auto-format.
- **Парсер сложнее.** Надо считать столбцы, обрабатывать смешанные
  отступы, давать осмысленные ошибки.
- **AI-инструменты** часто переформатируют код — невидимая разница
  между телом функции и следующим выражением module-level становится
  багом.
- **Явные `{}`** = ноль двусмысленности. Инструменты (форматер,
  линтер, LSP, парсер) живут проще.

Цена — пара лишних скобок в многошаговых функциях. Это дёшево.

### Связь с другими решениями

- **D2** (`=>` для single-expression bodies) — D40 это уточнение и
  ужесточение D2: `=>` принимает **ровно** одно выражение, не блок.
- **D31** (handler-lambda `(x) => expr` или `(x) => { ... }`) — то же
  правило для лямбд.

### Цена

1. **В примерах надо исправить multiline `=>` без `{}`.** Я писал
   так в HashMap, oxsar_port, audit и в нескольких драфтах. После
   D40 это нелегально и переписывается.
2. **Длинные one-liner'ы** (`HashMap[K, V].with_capacity(16)` после
   `=>` с переносом) остаются legal — это всё ещё одно выражение,
   просто перенесённое.

## D41. Static-функции есть, static-состояния нет

### Решение

У типа есть **static-функции** (`fn Type.name(...)`), но **нет static-полей**,
**нет static-переменных**, **нет static initializer'ов**.

Если нужны константы, ассоциированные с типом, — это `const` в том же
модуле. Если нужно «глобальное» изменяемое состояние — это **handler**
(эффект-capability), не static.

### Как именно

**Static-функции** — обычные функции в namespace типа. Внутри одной
static-функции другие static-функции того же типа вызываются **через
полное имя**, без сокращений:

```nova
fn Account.new(owner str) -> Account =>
    Account { _balance: 0, owner: owner }

fn Account.from_balance(owner str, initial money) -> Account {
    let acc = Account.new(owner)             // явное Account.new, не self.new
    Account.deposit_static(acc, initial)     // тоже явно
    acc
}
```

Никакого `Self::new` (Rust) или просто `new` (Java/C#). Один способ
вызова static-функции — через имя типа, что внутри типа, что снаружи.

**Константы рядом с типом** — отдельные `const` в модуле:

```nova
const ACCOUNT_MIN_BALANCE money = 0
const ACCOUNT_MAX_OVERDRAFT money = 1000

fn Account.new(owner str) -> Account =>
    Account { _balance: ACCOUNT_MIN_BALANCE, owner: owner }
```

Если нужна группировка — модуль:

```nova
module account_limits

export const MIN_BALANCE money = 0
export const MAX_OVERDRAFT money = 1000

// использование:
import account_limits
let acc = Account.new_with(account_limits.MIN_BALANCE)
```

**Глобальное изменяемое — через handler:**

```nova
// Вместо static counter / static config
type Counter { count int }

effect IdGen { fresh() -> u64 }

handler counter_id_gen(c mut Counter) IdGen {
    fresh() => {
        c.count += 1
        c.count
    }
}

// в main:
fn main() {
    let mut counter = Counter { count: 0 }
    with IdGen = counter_id_gen(counter) {
        run_app()
    }
}
```

Тестируется тривиально — другой handler в `with`-блоке.

### Почему

- **Static state — главный источник скрытых багов.** Глобальный
  изменяемый стейт не виден в сигнатурах, ломает параллельность,
  невозможно тестировать без хаков.
- **Тесты.** Static-поле = разделяемое состояние между тестами.
  Каждый тест должен либо ресетить его (хрупко), либо запускаться
  изолированно (медленно). Handler — `with`-блок изолирует
  автоматически.
- **Параллелизм.** Несколько fiber'ов на одном static-поле = data race
  по умолчанию. Handler-state живёт в scope и не делится случайно.
- **DI is the language.** Передача зависимостей — это handler. Не
  нужен отдельный фреймворк для DI, не нужны static-singleton'ы как
  замена.
- **Единственный путь.** Нет «иногда static, иногда handler» — всегда
  handler. Меньше способов сделать неправильно.

### Что отвергнуто

- **Static mutable поля** (Java `static int counter`, Python class
  variable). Силно мешает тестам и параллелизму.
- **Static immutable поля как `const`** на типе (`const Account.MIN`).
  Технически безопасно, но добавляет второй способ объявить
  константу. Один способ — `const` в модуле.
- **Companion-object** (Kotlin). То же что и static, просто в обёртке.
  Не нужен.
- **Lazy static** (Rust `lazy_static!`). Скрытое глобальное состояние
  с инициализацией. Если нужна ленивость — handler с lazy полем.

### Связь с другими решениями

- **D6** (managed memory) — глобального mutable state не предусмотрено
  в модели памяти; всё живёт в fiber-scope или handler-scope.
- **D27/D31** (handlers) — основной механизм для зависимостей и
  «глобальных» состояний.
- **D33** (`const` vs `let`) — `const` — единственный способ объявить
  immutable «глобальную» константу.

### Цена

1. **Привычка из Java/C#/Python ломается.** Нет `Account.MAX_BALANCE`
   как поля, есть `MAX_BALANCE` как `const` в модуле. Чуть длиннее,
   но единообразнее.
2. **Singleton'ы переписываются как handler.** Это не цена, а фича —
   но мигрирующий код придётся переделать.
3. **Counter / cache / pool** требуют явного создания и проброса в
   `with`-блок. Не «само работает», а явное жизненное циклы.

## D42. `protocol` keyword для структурных интерфейсов

### Решение

Структурные интерфейсы объявляются отдельным keyword **`protocol`**, не
через `type X = { ... }`.

```nova
protocol Hashable {
    hash() -> u64
    eq(other Self) -> bool
}

protocol Iterator[T] {
    next() -> Option[T]
}

type Login {                        // record (данные) — keyword type
    username str
    password str
}
```

`type` — для **данных**: record, sum-type, alias.
`protocol` — для **поведения**: набор методов как контракт.

### Почему отдельный keyword

Старая форма `type X = { методы }` визуально совпадала с record-формой
`type X { поля }`, различаясь только знаком `=`. Это сбивало:

```nova
// record (поля)
type Login { username str, password str }

// протокол (методы) — раньше
type Hashable = { hash() -> u64, eq(other Self) -> bool }
```

LLM и человек различают намерение по `=`, что хрупко. Замена на
ключевое слово делает намерение явным:

```nova
type Login { username str, password str }       // данные
protocol Hashable { hash() -> u64 }             // контракт
```

### Что меняется

1. **Объявление** — `type X = { методы }` заменяется на `protocol X
   { методы }`. `=` больше не нужен.
2. **`Self`** валиден только в `protocol`-блоке.
3. **Структурная семантика остаётся** — любой тип, у которого
   совпадают сигнатуры, **автоматически** удовлетворяет protocol'у.
   Никаких `impl Hashable for MyType`.
4. **`type X = T`** (alias) и `type X = A | B` (sum-type) **остаются** —
   `=` после `type` ассоциируется только с типом-выражением, не с
   объявлением методов.
5. **`fn`-префикс** для поля-функции в record не отменяется:
   ```nova
   type Button {
       text str
       on_click fn() Io -> ()      // поле-функция в record
   }
   ```
   В `protocol` `fn`-префикс не нужен — там и так все «члены» это
   методы.

### Сравнение с другими языками

`protocol` как keyword для интерфейсов используется в:
- **Swift** — `protocol` (nominal, с extension'ами).
- **Objective-C** — `@protocol`.
- **Clojure** — `defprotocol`.
- **Elixir** — `defprotocol`.
- **Python** — `typing.Protocol` (с PEP 544, structural).

Nova семантически ближе всего к Python `typing.Protocol` — чисто
структурный subtyping. Любой тип со совпадающей сигнатурой
автоматически удовлетворяет.

### Что отвергнуто

- **`contract`** — слово занято в Nova под pre/post-условия (D24).
- **`promise`** — массовая ассоциация с async (JS Promise).
- **`interface`** — слишком сильный nominal-bias (Java/C#).
- **`trait`** — обещает Rust-фичи (default impl, supertraits, blanket
  impl), которых в Nova нет.
- **`shape`** — короче, но менее знакомо как keyword.
- **`ability`** — образно, но без знакомства; навязывает `-able`
  суффикс именам.

### Связь с другими решениями

- **D9** (структурные интерфейсы) — этот D42 уточняет грамматику.
- **D39** (`use`/embed) — `use Type` остаётся для делегации между
  record-типами; `protocol` не embed'ится (см. ниже про bounds).
- **D35** (методы) — vtable layout для `protocol`-значений описан в
  диалоге, фиксируется отдельным D в будущем (fat pointer ABI).

### Открытые вопросы

- **Bounds на дженерики** — `HashMap[K: Hashable, V]` требует
  отдельного решения (Q1 / open-questions). Сейчас `K` без bound,
  компилятор полагается на структурное соответствие при
  использовании.
- **Default-методы в protocol** — пока запрещены. Отдельное расширение
  при необходимости.
- **Inheritance protocol'ов** — `protocol A : B` пока запрещено.
  Эквивалент достигается явным включением методов `B` в `A`.

### Цена

1. **Миграция существующих доков и примеров** — `type X = { ... }`
   заменить на `protocol X { ... }`. ~5 файлов, мелкая правка.
2. **Новый keyword** — `protocol` добавляется в keyword.declaration в
   VSCode grammar, парсер.

## D43. Trailing lambda с обязательными `()`

### Решение

Если последний параметр функции — функционального типа (`fn(...) -> T`),
лямбда может быть вынесена за скобки вызова:

```nova
spawn() {
    handle_request(req)
}

with_timeout(2.seconds) {
    Db.exec("...")
}

retry(3) {
    Net.get(url)
}

list.filter() { x => x > 0 }
list.fold(0) { (acc, x) => acc + x }
```

`()` **всегда обязательны**, даже если других аргументов нет. Это
ключевое отличие от Kotlin/Swift/Ruby и плата за простой парсер.

### Грамматика

```
call           = primary '(' args ')' [ trailing-lambda ]
trailing-lambda = '{' [ params '=>' ] body '}'
params         = identifier | '(' identifier { ',' identifier } ')'
```

**Правила:**

1. **`()` обязательны.** `spawn { body }` — синтаксическая ошибка.
   Только `spawn() { body }`.

2. **`{` на той же строке, что и `)`.** Перенос между `)` и `{`
   **запрещён** — иначе парсер не различит trailing lambda от
   следующего statement-блока.

   ```nova
   spawn(arg) {           // OK
       body
   }

   spawn(arg)             // ❌ ошибка
   {                      // парсер не свяжет с вызовом выше
       body
   }
   ```

3. **Тип последнего параметра должен быть функциональным.** Иначе —
   ошибка типизации. Парсер строит AST синтаксически, типизатор
   проверяет согласованность.

4. **Один trailing на вызов.** Не два, не три.

5. **Параметры лямбды через `=>`:**
   - `{ body }` — лямбда без параметров.
   - `{ x => body }` — один параметр, без скобок.
   - `{ (a, b) => body }` — два или больше, в скобках.

6. **Implicit `it` запрещён.** Параметр всегда именован.

7. **Method chain — те же правила:** `list.filter() { x => x > 0 }`.
   Никаких сокращений.

### Дисамбигуация с record-литералами

`{` после `)` = trailing lambda.
`{` после имени типа без `()` = record-литерал.

```nova
let u = User { name: "alice" }       // record (имя типа, без ())
fn_call(arg) { name: "alice" }       // trailing lambda (после `)`)
fn_call(arg, User { name: "a" })     // record внутри args (между `,` и `)`)
```

Парсер различает по контексту синтаксически. **Type-directed parsing
не нужен.**

### Примеры миграции

Короткие лямбды остаются в скобках — convention, не mandatory:

```nova
list.filter((x) => x > 0)               // короткая, остаётся inline
m.get_or_insert("counter", () => 0)     // короткая, остаётся inline
```

Длинные лямбды переписываются на trailing для читаемости:

```nova
// БЫЛО
spawn(() => {
    handle_request(req)
    log_completion()
})

// СТАЛО (по желанию)
spawn() {
    handle_request(req)
    log_completion()
}
```

### Что становится возможным

Многие операции, бывшие language primitives, могут стать обычными
stdlib-функциями. Это убирает специальный синтаксис из компилятора:

```nova
fn with_timeout[T](dur Duration, body fn() Async -> T) Async Throws -> T
fn transaction[T](db mut Db, body fn() Db Throws -> T) Db Throws -> T
fn retry[T](attempts int, body fn() Throws -> T) Throws -> T
fn region_alloc[T](body fn() Realtime -> T) Realtime -> T
```

Используются **как primitives** на сайте вызова:

```nova
with_timeout(2.seconds) { Db.exec("...") }
transaction(db) { update(alice); update(bob) }
retry(3) { Net.get(url) }
```

### Что остаётся keyword-блоками

Некоторые конструкции **остаются** специальным синтаксисом языка
(не функциями), потому что:
- Требуют compile-time анализа, недоступного через сигнатуру функции.
- Имеют особую семантику для эффект-системы или scheduler'а.
- Без `()`, в отличие от function calls.

```nova
with X = handler { body }            // эффект-binding
parallel for x in xs { body }        // structured concurrency
supervised { body } strategy = ...   // supervision tree
forbid Net, Fs { body }              // capability scope
race { body }, select { body }       // concurrency primitives
region { body }                      // realtime memory
match expr { ... }, if, for, while   // control flow
```

Различие между **функцией с trailing lambda** и **keyword-блоком** —
наличие `()`. Это явный визуальный сигнал:

```nova
spawn() { ... }                      // функция (есть `()`)
parallel for x in xs { ... }         // keyword (нет `()`)
```

### Что отвергнуто

1. **Опциональные `()`** (Kotlin-style `spawn { body }`) — нет
   локального способа развести с record-литералами без
   type-directed parsing.

2. **`()` опционально в method chain** (`list.filter { ... }`) —
   нарушение единого правила, лишнее исключение.

3. **Implicit `it` параметр** (Kotlin `{ it > 0 }`) — нелокальный
   reasoning, плохо для AI.

4. **Trailing lambda через keyword `do`** (`fn_call(arg) do { body }`) —
   новый keyword без необходимости.

5. **Indentation-significant вариант** (Python-style блок-через-отступ
   после `)`) — конфликт с D40.

### Связь с другими решениями

- **D40** (`=>` для одного выражения, `{}` для блока) — trailing
  lambda использует `{}`-блок-форму. Параметры через `=>` — та же
  стрелка, что в expression-body, но в позиции «параметры лямбды».
  Контекст однозначный.
- **D31** (handler-lambda) — для эффектов сахар `with X = (e) => expr`
  остаётся. Trailing lambda — для **обычных** функций с лямбдой-аргументом.
- **D27** (массивы) и **D14** (concurrency) — `parallel for`, `supervised`
  и т.п. остаются keyword'ами.

### Цена

1. **Обязательные `()`** добавляют 2 символа к каждому вызову с
   trailing lambda. `spawn() { body }` вместо Kotlin-стиля
   `spawn { body }`. Цена приемлема за локальный парсер.
2. **Конвенция «короткие inline, длинные trailing»** — мягкое правило,
   не enforced. Зависит от вкуса и команды.
3. **Парсер должен научиться распознавать trailing lambda** — но это
   локальное правило, не сильно усложняет грамматику.

## D44. Числовые литералы

### Решение

Полный набор форм для числовых литералов в Nova:

```nova
// ── Целые: десятичные ────────────────────────────────────────────
1
1_000_000_000              // разделитель `_` между цифрами
0                          // ноль

// ── Целые: hex / binary / octal ──────────────────────────────────
0xFF                       // hex, любой регистр (0xff = 0xFF)
0xFF_FF_FF_FF              // hex с разделителем
0b1010_0001                // binary
0o755                      // octal (Unix permissions)

// ── Float ────────────────────────────────────────────────────────
1.5
1_234.567_89               // разделители с обеих сторон точки
1e10                       // научная нотация
1.5e-3                     // дробная + экспонента
1_000.5e6                  // комбинация

// ── Type через cast или аннотацию (suffixes НЕ вводим) ───────────
let x i32 = 100            // через аннотацию
100 as u8                  // через cast
0xFF as u32                // hex как u32
```

### Default-типы

- **Integer literal** без контекста: **`int`** (платформенно-зависимая
  ширина: i64 на 64-bit, i32 на 32-bit; см. список primitive types).
- **Float literal** без контекста: **`f64`**.

Если контекст определяет тип (аннотация, тип параметра функции, тип
поля при assign'е) — литерал берёт тип из контекста:

```nova
let x u8 = 200             // 200 это u8
fn write(b u8) -> () => ...
write(0xFF)                // 0xFF это u8

let arr []f32 = [1.0, 2.0]  // элементы f32
```

### Разделитель `_`

**Разрешено:**
- Только **между цифрами**.
- В любом числе: десятичном, hex, binary, octal, float (целая и
  дробная части), экспоненте.

**Запрещено:**
- В **начале** литерала: `_1_000` ❌.
- В **конце** литерала: `1_000_` ❌.
- **Подряд**: `1__000` ❌.
- **Сразу после префикса**: `0x_FF` ❌, `0b_1010` ❌.
- **Вокруг точки** в float: `1_.5` ❌, `1._5` ❌.
- **Вокруг `e`** в экспоненте: `1_e10` ❌, `1e_10` ❌.

### Грамматика (regex)

```
decimal-int     = [0-9] (_? [0-9])*
hex-int         = "0x" [0-9a-fA-F] (_? [0-9a-fA-F])*
binary-int      = "0b" [01] (_? [01])*
octal-int       = "0o" [0-7] (_? [0-7])*

float           = decimal-int "." decimal-int (("e"|"E") ("+"|"-")? decimal-int)?
                | decimal-int ("e"|"E") ("+"|"-")? decimal-int

literal-int     = decimal-int | hex-int | binary-int | octal-int
literal-number  = float | literal-int
```

`(_? [0-9])*` означает «опциональный `_`, затем цифра» — это запрещает
двойные `_` и `_` в начале/конце.

### Почему без type-suffixes (`100u32`, `1.5f32`)

Отвергнуто по причинам:

1. **Шум.** `100u32`, `0xFFu8`, `1.5f32` визуально хуже `100 as u32`.
2. **Дублирование.** `let x u32 = 100` уже работает через inference.
3. **Тренд новых языков** (Swift, Go, Zig) — без суффиксов.
4. **AI-friendly.** Меньше форм записи = LLM стабильнее.
5. **Cast как явная альтернатива** (`100 as u32`, `0xFF as u8`)
   достаточен для редких случаев, где inference не справляется.

Если в будущем выяснится, что `as`-cast недостаточно эргономичен —
вернёмся к suffixes отдельным D-решением.

### Сравнение с другими языками

| Язык | Разделитель | Type suffixes | Default int | Default float |
|---|---|---|---|---|
| **Nova** | `_` | ❌ (через `as`) | `int` | `f64` |
| **Rust** | `_` | ✅ (`u32`, `f64`) | `i32` | `f64` |
| **Swift** | `_` | ❌ | `Int` | `Double` |
| **Go** | `_` | ❌ | `int` | `float64` |
| **Kotlin** | `_` | частично (`L`, `F`) | `Int` | `Double` |
| **Java** | `_` | `L`, `F`, `D` | `int` | `double` |
| **C#** | `_` | `u`, `l`, `m` | `int` | `double` |
| **Python** | `_` | ❌ | `int` (bigint) | `float` |
| **C++14** | `'` | `ull`, `f`, `L` | `int` | `double` |

Nova ближе всего к **Go**: `_` как разделитель, `int` как дефолт, без
суффиксов. Платформенно-зависимый `int` — компромисс между Rust
(`i32` фикс) и Python (bigint).

### Связь с другими решениями

- **D27** (массивы): `[10]u32` — `10` берёт тип `int` для длины.
- **D40** (тело функции): литералы в expression-body работают как обычно.
- **D33** (`const` vs `let`): `const N int = 1_000_000` — литерал в
  const-контексте.

### Цена

1. **Парсер строже** — regex с `(_? digit)*` сложнее `[0-9_]+`.
   Но это локальное правило, не влияет на остальную грамматику.
2. **VSCode grammar** надо обновить — текущая `[0-9_]+` пропускает
   `1__000` и `_1`. Заменить на правильный regex.
3. **Без suffixes** — иногда придётся писать `100 as u32` вместо
   `100u32`. Редко.

## D45. Inferred return type для expression-body

### Решение

В **expression-body** (`=> expr`) тип возврата `-> T` **опционален** —
выводится из тела:

```nova
fn double(x int) => x * 2                          // -> int выведен
fn Duration @as_nanos() => @nanos                  // -> i64 выведен из @nanos
fn Duration @is_zero() => @nanos == 0              // -> bool выведен
fn HashMap[K, V] @len() => @count                  // -> int выведен
fn int @seconds() => Duration.from_secs(@)         // -> Duration выведен
```

В **block-body** (`{ ... }`) `-> T` **обязателен**, если не unit:

```nova
fn next_pow2(n int) -> int {                       // обязателен
    if n <= 1 { return 1 }
    let mut p = 1
    while p < n { p *= 2 }
    p
}

fn process() {                                     // -> () опускаем (D32)
    Log.info("hello")
}
```

### Правила

1. **Expression-body** (`=> expr`): `-> T` опционален. Если опущен —
   тип берётся из выражения.
2. **Block-body** (`{ ... }`): `-> T` обязателен, если не unit.
3. **Unit `-> ()`**: всегда опускается (D32, не меняется).
4. **`export` функции**: style guide рекомендует **писать `-> T` явно**
   для public API. Линтер предупреждает, компилятор не enforces.

### Inference — локальный, простой

Компилятор выводит тип возврата из выражения тела:

- Литерал → его тип (`100` → `int`, `1.5` → `f64`).
- `@field` → тип поля.
- Вызов функции → тип возврата той функции.
- Record-литерал `T { ... }` → `T`.
- Match → тип всех веток (через unification).
- If/else → тип обеих веток.

Это **локальный inference** (одна функция, одно выражение), не
full Hindley-Milner. Не требует анализа всего модуля.

### Когда писать `-> T` явно (рекомендация)

- **`export` функции** — public API. Изменение тела не должно молча
  менять тип возврата.
- **Сложные выражения** — match с большим числом веток, где беглый
  читатель не сразу видит тип.
- **Generic-функции** — `fn map[T, U](xs []T, f T -> U) -> []U` — `-> []U`
  явно показывает связь с параметром.

### Когда опускать `-> T` (рекомендация)

- **Простые геттеры** — `fn @len() => @count`.
- **Однозначные конструкторы** — `fn from_secs(n i64) => Duration { nanos: n * 1e9 }`.
- **Тривиальные предикаты** — `fn @is_zero() => @nanos == 0`.
- **Stdlib helpers** внутри модуля.

### Сравнение с другими языками

| Язык | Expression-body inference | Block-body inference |
|---|---|---|
| **Nova** | ✅ да | ❌ |
| **Rust** | ❌ | ❌ |
| **Swift** | ❌ | ❌ |
| **Go** | ❌ | ❌ |
| **Kotlin** | ✅ да | ❌ |
| **Scala** | ✅ да | ✅ да |
| **TypeScript** | ✅ да | ✅ да |
| **OCaml/Haskell/F#** | ✅ всегда | — |

Nova следует **Kotlin-стилю** — inference только для compact form.

### Связь с другими решениями

- **D32** (опциональный `-> ()`) — оставляется без изменений; D45
  расширяет на не-unit типы для expression-body.
- **D40** (`=>` для одного выражения, `{}` для блока) — граница
  применимости D45 точно совпадает с D40.

### Цена

1. **Inference logic** в компиляторе — ~200 LOC, локальный.
2. **Style guide** для public API — рекомендация, не enforcement.
3. **Linter rule** — предупреждение `export fn ... =>` без `-> T`.

## D46. Перегрузка операторов через @-методы

### Решение

Стандартные операторы автоматически вызывают методы инстанса с
**фиксированными именами**. Если у типа есть метод нужного имени,
оператор работает.

```nova
fn Duration @plus(other Duration) -> Duration =>
    Duration { nanos: @nanos + other.nanos }

fn Duration @times(n i64) -> Duration =>
    Duration { nanos: @nanos * n }

let total = 1.hour() + 30.minutes()       // вызывает @plus
let triple = 5.seconds() * 3              // вызывает @times
```

### Mapping операторов и методов

| Оператор | Метод | Возврат |
|---|---|---|
| `a + b` | `a.plus(b)` | свободный |
| `a - b` | `a.minus(b)` | свободный |
| `-a` (унарный) | `a.neg()` | обычно `Self` |
| `a * b` | `a.times(b)` | свободный |
| `a / b` | `a.div(b)` | свободный |
| `a % b` | `a.rem(b)` | свободный |
| `a \| b` (bitwise) | `a.or(b)` | свободный |
| `a & b` (bitwise) | `a.and(b)` | свободный |
| `a ^ b` (bitwise) | `a.xor(b)` | свободный |
| `a << n` | `a.shl(n)` | свободный |
| `a >> n` | `a.shr(n)` | свободный |
| `a == b` | `a.eq(b)` | `bool` |
| `a != b` | `!a.eq(b)` | `bool` (выводится) |
| `a < b` | `a.lt(b)` | `bool` |
| `a <= b` | `a.le(b)` | `bool` |
| `a > b` | `a.gt(b)` | `bool` |
| `a >= b` | `a.ge(b)` | `bool` |
| `!a` (унарный) | `a.not()` | обычно `bool` или `Self` |
| `a[i]` (read) | `a.get(i)` | свободный |
| `a[i] = v` | `a.set(i, v)` | `()` |

### Правила

1. **Только методы инстанса** (`fn Type @method`). Не свободные
   функции, не static-функции. Привязывает оператор к **первому**
   операнду — никаких неоднозначностей.

2. **Тип возврата свободный.** `Vector @times(scalar f64) -> Vector` или
   `Duration @times(n i64) -> Duration` — операнды и результат
   могут быть разных типов.

3. **Bool short-circuit (`&&`, `||`) НЕ перегружаются.** Они остаются
   строго булевыми, чтобы short-circuit семантика была предсказуемой.

4. **`!=` выводится из `@eq`.** Не нужно объявлять отдельно.

5. **Никаких custom-операторов.** Нельзя ввести `@@`, `:+`, `**` для
   своих типов. Фиксированный набор символов из таблицы выше.

6. **Никакой автоматической рефлексии.** Структурное соответствие
   protocol'у `Add[T]` (как в Rust/Swift) не требуется — компилятор
   просто проверяет наличие метода с правильным именем при разборе
   оператора.

### Что отвергнуто

- **Через protocol/trait** (Rust `impl Add for T`, Swift `+ func`) —
  избыточно для нужд Nova. Достаточно структурного matching по имени
  метода.
- **Custom-операторы Scala-стиля** (`:+`, `<>`, `>>=`) — известный
  источник нечитаемого кода. Запрещены.
- **Свободные функции `fn plus(a, b)`** — overloading через свободные
  функции создаёт ambiguity при unification аргументов. Только
  методы.
- **Перегрузка `&&`/`||`** — нарушает short-circuit семантику.
- **Авто-генерация `@eq`/`@lt` через `derive`** — отдельный механизм,
  не часть D46. Сейчас требуется писать вручную.

### Type coercion

`a + b` требует, чтобы у типа `a` был метод `@plus(other T)` где `T` —
тип `b`. Никакой автоматической coercion (`int → f64`, etc.) при
вызове через оператор:

```nova
let d = 1.hour()                  // Duration
d + 30                            // ❌ Duration @plus принимает Duration, не int
d + 30.seconds()                  // ✅ оба Duration
```

Если нужно перегрузить для разных правых типов — несколько методов:

```nova
fn Vector @times(s f64) -> Vector =>     // умножение на скаляр
    Vector { x: @x * s, y: @y * s }

fn Vector @times(other Vector) -> f64 =>  // dot product
    @x * other.x + @y * other.y
```

В Nova **overloading методов разрешён по типу аргумента** (если
сигнатуры различимы). Это уточнение для D46 — фиксируем здесь:
методы могут иметь одно имя при разных типах аргументов. Для **обычных**
методов overloading тоже доступен.

### Сравнение с другими языками

| Язык | Overloading | Механизм | Custom-операторы |
|---|---|---|---|
| **Nova** | ограниченный | имена методов | ❌ |
| **Rust** | ограниченный | trait `Add`/`Sub`/... | ❌ |
| **Swift** | ограниченный | `func +(...)` | ✅ через protocol'ы |
| **Kotlin** | ограниченный | `operator fun plus(...)` | ❌ |
| **C#** | ограниченный | `static operator+(...)` | ❌ |
| **Scala** | полный | методы могут быть символьными | ✅ |
| **C++** | полный | `operator+(...)` | ✅ |
| **Go** | ❌ | — | ❌ |
| **Java** | ❌ | — | ❌ |

Nova ближе всего к **Kotlin** — фиксированный набор имён методов,
без custom-операторов.

### Примеры использования

```nova
// Duration арифметика
let d = 1.hour() + 30.minutes()
let backoff = 100.millis() * 2

// Сравнение
if elapsed > 1.second() { Log.warn("slow") }
if a == b { ... }

// Индексация (для коллекций)
fn HashMap[K, V] @get(key K) -> Option[V] => ...
fn HashMap[K, V] mut @set(key K, value V) -> () => @insert(key, value)

let m = HashMap[str, int].new()
m["counter"] = 1                          // m.set("counter", 1)
let v = m["counter"]                      // m.get("counter")

// Vector
fn Vector @plus(o Vector) => Vector { x: @x + o.x, y: @y + o.y }
fn Vector @times(s f64) => Vector { x: @x * s, y: @y * s }

let v = Vector { x: 1.0, y: 2.0 } + Vector { x: 3.0, y: 4.0 }
let scaled = v * 2.5
```

### Связь с другими решениями

- **D35** (методы через `@`) — D46 использует тот же механизм.
- **D9** (нет trait/interface) — overloading **не** через protocol.
  Чисто по имени метода.
- **D45** (inferred return) — методы операторов — обычные expression-body,
  возвращаемый тип может быть выведен.
- **Q16 (bitflags)** — теперь решаемо: type Permission(int) с
  методами `@or`, `@and` для `|`, `&`. Закрывает Q16 как
  «overloading доступен».

### Цена

1. **Парсер должен уметь связывать оператор с методом.** В момент
   разбора `a + b` компилятор смотрит метод `@plus` у типа `a`. Не
   найден — ошибка типизации, не парсинга.
2. **Возможна типовая ambiguity** при overloading методов с одним
   именем. Решается обычными правилами разрешения (точное
   совпадение типа, без implicit conversion).
3. **Закрытый список операторов.** Custom-операторы запрещены.
   Если в будущем потребуется (rare) — отдельным D-решением.

### Что закрывает

- **Q16 (bitflags)** — теперь можно сделать `type Permission(int)` с
  `@or`/`@and`/`@not` для `|`/`&`/`!`. Open-question закрывается.

## D47. Видимость деклараций

### Решение

**`export`** перед декларацией = публичная (видна снаружи модуля).
**Без `export`** = приватная (видна только внутри модуля).

Применяется единообразно ко всем top-level конструкциям и методам:

```nova
module account

// ── Типы ──────────────────────────────────────────────────────────
export type Account {                    // публичный тип
    readonly owner str
    balance money
    _internal_id u64                     // convention: _prefix = приватное-по-договору
}

type InternalState {                     // приватный тип
    pending_ops []Op
}

// ── Константы ─────────────────────────────────────────────────────
export const ACCOUNT_MIN_BALANCE money = 0
const _INTERNAL_TIMEOUT_MS int = 5_000

// ── Static-функции и конструкторы ─────────────────────────────────
export fn Account.new(owner str) -> Account =>
    Account { owner, balance: money.zero, _internal_id: gen_id() }

fn Account.from_db_row(row Row) Throws -> Account => ...   // приватная

// ── Методы инстанса ───────────────────────────────────────────────
export fn Account @balance() => @balance                   // публичный
export fn Account mut @deposit(amount money) {             // публичный
    @validate_amount(amount)?
    @balance += amount
}

fn Account @validate_amount(amount money) =>               // приватный
    amount > 0 && amount < money.MAX

// ── Протоколы ─────────────────────────────────────────────────────
export protocol Hashable {
    hash() -> u64
    eq(other Self) -> bool
}

protocol _InternalIter[T] {                // приватный protocol
    next() -> Option[T]
}
```

### Видимость полей record

**MVP:** все поля **`export`-типа публичны** для упрощения. Convention
**`_prefix`** для полей, которые автор считает приватными:

```nova
export type Account {
    readonly owner str       // публичное
    balance money            // публичное
    _internal_id u64         // приватное-по-конвенции (НЕ enforced)
}
```

Convention `_prefix` — **не enforced** компилятором. Программист может
обратиться к `acc._internal_id` извне, компилятор не остановит. Это
**сознательный компромисс MVP**.

**`_prefix` применяется только к полям**, не к функциям и методам.
Для функций/методов видимость **двухуровневая** через `export`/без —
третьего «совсем-приватного» уровня нет:

```nova
export fn public_op(...) => ...       ✅ публичная
fn private_helper(...) => ...         ✅ приватная для модуля
fn _internal_helper(...) => ...       ❌ не нужно — нет третьего уровня
```

Сознательный компромисс MVP по причинам:

- В реальности 90% случаев инкапсуляция полей делается через **методы**
  (`@balance()` геттер, `@deposit()` сеттер). Поля редко прямо
  модифицируются снаружи.
- Per-field `export` усложняет грамматику — добавляется keyword в
  каждое поле.
- Если в будущем покажется необходимым enforcement — расширим (per-field
  `export` или keyword `private`).

### Метод приватного типа

Если тип приватный (`type X` без `export`), все его методы фактически
**недоступны снаружи модуля**, даже если помечены `export`:

```nova
type InternalState { ... }

// синтаксически валидно, но `export` бесполезен — тип всё равно приватный
export fn InternalState @do_thing() => ...
```

Компилятор может выдать **предупреждение** «`export` method on
non-exported type — `InternalState` is private, method visibility
limited to module».

### Re-export через `export import`

Зафиксировано в [D29](#d29) — модуль может re-export'ить чужие
декларации:

```nova
module my_lib

export import std.duration              // вся duration видна пользователям my_lib
import std.internal_helpers             // только внутри my_lib
```

### Почему именно так

1. **Один keyword `export`** работает для всех видов деклараций — type,
   fn, const, protocol, метод. Учить нечего.
2. **Default приватный** соответствует трендам современных языков
   (Rust, Swift, Java/C# для классов). Безопасно по умолчанию.
3. **Convention `_prefix`** для полей — Python/Ruby-стиль, знакомо
   LLM, не требует языковых средств.
4. **Постепенная эволюция:** если понадобится enforcement полей —
   расширим без breaking changes (новое keyword добавит strictness,
   старый код останется валидным).

### Что отвергнуто

- **Per-field `export`** (Вариант B) — многословно, преждевременная
  оптимизация для MVP.
- **`private` keyword** (Вариант D) — Java-стиль, конфликтует с
  философией Nova «явно публичное». Не нужен.
- **Видимость по регистру** (Go-стиль `Capital = public`) — Nova
  использует разные conventions для разных уровней (см. D30 про
  именование), смешивать с visibility сделает грамматику зависимой
  от регистра идентификаторов.

### Сравнение с другими языками

| Язык | Default | Public marker | Field-level visibility |
|---|---|---|---|
| **Nova** | private (модуль) | `export` | все public, `_prefix` convention |
| **Rust** | private | `pub` | per-field `pub` |
| **Go** | regex-based | Capital letter | то же |
| **Swift** | internal | `public`/`private`/`fileprivate` | per-field |
| **Kotlin** | public | `private`/`internal` | per-field |
| **Java** | package-private | `public`/`private` | per-field |
| **Python** | public | — | `_prefix` convention |
| **OCaml** | через signature | в `.mli` файле | через signature |

Nova ближе всего к **Python** для полей и к **Rust** для top-level.

### Связь с другими решениями

- **D29** (модули) — D47 уточняет визуальные правила для всех видов
  деклараций.
- **D30** (имена) — `_prefix` для приватных полей дополняет правила
  именования.
- **D35** (методы через `@`) — методы получают тот же `export`
  механизм.
- **D9, D42** (структурные типы / `protocol`) — `export protocol X`
  делает контракт публичным.

### Цена

1. **Convention `_prefix` не enforced** — опытные программисты
   напомнят друг другу в code review. Linter может предупреждать.
2. **Будущее расширение** — если потребуется per-field enforcement,
   добавим `export` для полей. Не break.
3. **Документация** — нужно зафиксировать convention в стайл-гайде,
   чтобы не было сюрприза «почему `_balance` всё-таки доступен
   снаружи».

## D48. Tagged template literals

### Решение

Литералы вида `` tag`raw_text` `` — синтаксический сахар над вызовом
функции `tag` со специальной сигнатурой, принимающей сегменты текста и
интерполированные значения раздельно.

```nova
let j = json`{"name": "alice"}`
let q = sql`SELECT * FROM users WHERE id = ${user_id}`
let h = html`<div>${escape(name)}</div>`
let r = regex`\d{3}-\d{4}`
let b = bytes`deadbeef`
```

### Грамматика

```
tagged-template = identifier '`' template-body '`'
template-body   = ( raw-char | escape-seq | interpolation )*
raw-char        = любой символ кроме '`', '\\', '${'
escape-seq      = '\\' ( '`' | '\\' | '${' | 'n' | 't' | 'r' | ... )
interpolation   = '${' expression '}'
```

`identifier` — обычный snake_case (имя функции в скоупе).

### Семантика — desugar в вызов

```nova
sql`SELECT * FROM users WHERE id = ${user_id} AND name = ${name}`
```

Эквивалентно:

```nova
sql(
    ["SELECT * FROM users WHERE id = ", " AND name = ", ""],
    [user_id, name]
)
```

То есть tag-функция получает:
- **`parts: []str`** — сегменты текста между интерполяциями. Если в
  начале/конце нет текста — пустая строка.
- **`args: []T`** — значения интерполированных выражений, где `T` —
  тип, который выбирает tag-функция (часто `any` или специфичный).

Длина `parts` всегда **на 1 больше** длины `args`.

### Сигнатура tag-функции

Единый шаблон для всех тегов:

```nova
fn tag_name(parts []str, args []T) -> ResultType => ...
```

Конкретный тип `T` и `ResultType` — на усмотрение функции:

```nova
fn json(parts []str, args []Json) -> Json
fn sql(parts []str, args []SqlParam) -> Sql
fn html(parts []str, args []HtmlSafe) -> Html
fn regex(parts []str, args []str) -> Regex
fn bytes(parts []str, args []byte) -> Bytes
```

**Без интерполяций** — `args` пуст, `parts` содержит один элемент:

```nova
json`{"x": 1}`
// → json(["{\"x\": 1}"], [])
```

### Compile-time валидация

Для тегов с **только литеральным содержимым** (пустой `args`)
компилятор может **вызвать tag-функцию во время компиляции**, если
функция помечена `@comptime` (механизм будет уточнён в D-решении про
макросы). Это даёт:

- **Compile-time проверку** валидности (некорректный JSON → ошибка
  компиляции, не runtime).
- **Zero-cost** — литерал превращается в готовое значение.

```nova
@comptime
fn json(parts []str, args []Json) -> Json => ...

// в момент компиляции:
const SCHEMA Json = json`{"type": "object"}`     // парсится, проверяется
let bad = json`{not valid}`                      // ❌ compile-time error
```

Если интерполяции есть — выполнение runtime.

В MVP `@comptime`-теги откладываются — все теги работают runtime.
Compile-time — оптимизация v2.

### Escape-последовательности

Внутри template:
- `` \` `` — буквальный backtick.
- `\\` — буквальный backslash.
- `\${` — буквальный `${`, не интерполяция.
- `\n`, `\t`, `\r`, `\u{...}` — стандартные escapes.

Остальные символы — **raw**, как написано. Это удобно для regex и SQL,
где `\d`, `\\` встречаются часто:

```nova
let r = regex`\d+\.\d+`               // не нужно дважды экранировать
let q = sql`SELECT name LIKE 'a\%'`   // % не интерпретируется парсером
```

### Multiline

Естественно поддерживается — `` ` `` не имеет ограничения на одну строку:

```nova
let q = sql`
    SELECT id, name
    FROM users
    WHERE created_at > ${cutoff}
    LIMIT 100
`
```

Whitespace между `` ` `` и текстом — **часть содержимого**. Tag-функция
сама решает, нормализовать его или сохранить.

### Стандартные теги в stdlib

Минимальный набор для MVP:

| Тег | Тип результата | Назначение |
|---|---|---|
| `json` | `Json` | JSON-литерал, parsed |
| `sql` | `Sql` | SQL-запрос с параметрами (защита от injection) |
| `regex` | `Regex` | Compiled regex |
| `bytes` | `Bytes` | Бинарный литерал из hex |

**Не входят в MVP** (можно добавить в user-space):
- `html`, `css` — domain-specific, в отдельных пакетах.
- `graphql`, `cypher` — то же.

Программист может **объявить свой тег** — это обычная функция:

```nova
export fn url(parts []str, args []str) -> Url {
    let buf = StringBuilder.new()
    for i in 0..parts.len {
        buf.append(parts[i])
        if i < args.len { buf.append(url_encode(args[i])) }
    }
    Url.parse(buf.to_string())
}

let u = url`https://api.example.com/users/${user_id}?q=${query}`
```

### SQL-injection и safe interpolation

Главное преимущество tagged templates — **типобезопасная интерполяция**:

```nova
// небезопасно — обычная строка
let q = "SELECT * FROM users WHERE name = '${name}'"
// → SQL injection, если name содержит '

// безопасно — tagged template
let q = sql`SELECT * FROM users WHERE name = ${name}`
// → sql() видит name как параметр, эскейпит или передаёт
//   через prepared statement
```

Tag-функция **не получает интерполированную строку** — она получает
**raw parts** и **args отдельно**, и сама решает, как их соединить.
Для SQL — через prepared statement parameters (`$1`, `$2`).

### Сравнение с другими языками

| Язык | Синтаксис | Compile-time? |
|---|---|---|
| **Nova** | `` tag`...` `` | через `@comptime` (опционально) |
| **JavaScript** | `` tag`...` `` | нет |
| **Scala** | `` s"..." ``, `` raw"..." `` | для `s`/`raw` да |
| **C# 11+** | `` $$"""...""" ``, raw strings | частично |
| **Python** | `f"..."` (только interp) | runtime |
| **Rust** | `format!`, `println!` через макросы | compile-time |
| **Kotlin** | `"...$x..."` | runtime |
| **Swift** | `"\(x)"` | runtime |

Nova ближе всего к **JavaScript** по синтаксису, к **Scala/Rust** по
compile-time возможностям.

### Что отвергнуто

1. **Префикс через символ** (`s"..."`, `r"..."` Scala/Python-стиль) —
   ограничивает имя одним символом, не позволяет user-defined теги.
   `tag\`...\`` принимает любой identifier.

2. **Раздельные строковый и интерполяционный конструкторы** —
   `tag.raw("...")` + `tag.interp("...", args)` — слишком многословно
   для частого использования.

3. **Macros вместо tagged templates** (Rust-стиль `sql!("...")`) —
   требует механизма макросов, который ещё не зафиксирован. Tagged
   templates проще: обычная функция с особым синтаксисом вызова.

4. **Implicit tag** (без префикса — для дефолтного типа) — вводит
   ambiguity со строковыми литералами. Tag всегда явный.

### Связь с другими решениями

- **D33** (`const` vs `let`): теги с `@comptime` и без интерполяции
  могут быть значениями `const`.
- **D24** (контракты): tag-функция может использовать `requires` для
  валидации parts и args.
- **D27** (массивы): `parts` и `args` — обычные массивы.
- **D40** (тело функции): tag-функции — обычные функции, body как
  любая другая.
- **D42** (`protocol`): можно объявить `protocol Tag[T, R]` для
  сигнатуры tag-функции, но это не обязательно.

### Цена

1. **Парсер** должен распознавать `identifier` `` ` `` и переключаться
   в template-mode. Локальная грамматика, не сложная.
2. **Стандартные теги** (`json`, `sql`, `regex`, `bytes`) — реализуются
   в stdlib, ~500 LOC каждый.
3. **`@comptime`-механизм** — отдельный D-решение, откладывается на v2.
4. **Документация** — нужно объяснить безопасность interpolation,
   особенно для SQL/HTML.

## D49. Statement separator и парсинг выражений

### Решение

**Перенос строки** — основной разделитель statement'ов. **`;`** —
опциональный, нужен только для нескольких statement'ов на одной строке.

```nova
let x = 1                        // newline разделяет
let y = 2
foo(x, y)

let a = 1; let b = 2; foo(a, b)  // ; для одной строки (редко)
```

### Лексер: какие newlines игнорируются

Лексер выдаёт токен `NEWLINE`, но **игнорирует** его в позициях, где
statement очевидно продолжается:

1. **После висящего бинарного оператора** в конце предыдущей строки:
   ```nova
   let total = a +
               b +
               c              // ✅ продолжение
   ```

2. **Внутри открытых `(`, `[`, `{`:**
   ```nova
   let user = User {
       name: "alice",
       age: 30,
   }                          // ✅ внутри `{}` newlines игнорируются

   let result = foo(
       arg1,
       arg2,
   )
   ```

3. **Перед `.`** (method chain):
   ```nova
   let result = list
       .filter() { x => x > 0 }
       .map() { x => x * 2 }
       .sum()                 // ✅ `.method()` продолжает выражение
   ```

4. **Перед `?`** (error propagation):
   ```nova
   let user = find_user(id)
       ?                      // ✅ продолжение
   ```

5. **После `,` в списках** (внутри `(...)`, `[...]`, `{...}`).

### Стиль переноса бинарных операторов

**Бинарные операторы переносятся в КОНЕЦ предыдущей строки**, не в
начало следующей (Go-стиль):

```nova
// ✅ OK
let total = a +
            b +
            c

// ❌ НЕ OK — `+` в начале новой строки воспринимается как унарный
let total = a
          + b
          + c
```

Причина: при `\n + b` парсер видит начало нового statement'а
с унарным `+b`. Стандартная Go-практика — операторы в конце.

Исключение — **`.method()`** и **`?`** допустимы в начале строки, потому
что `.` и `?` не могут начать самостоятельный statement.

### Edge cases

**1. Идентификатор на одной строке, `(` на следующей.**

```nova
let x = foo
(arg)                        // ❌ два statement'а: `let x = foo` и `(arg)`
```

Парсер видит после `foo` newline → конец statement. `(arg)` следующий
statement (выражение в скобках). Чтобы вызвать — на одной строке или
с открытой скобкой:

```nova
let x = foo(arg)             // ✅ одна строка

let x = foo(                 // ✅ открытая `(` игнорирует newline
    arg
)
```

**2. Trailing lambda (D43):** `)` и `{` **на одной строке**, перенос
между ними запрещён.

```nova
spawn() {                    // ✅
    body
}

spawn()                      // ❌ D43 — `{` на новой строке
{ body }
```

**3. Block-as-expression:**

```nova
let x = {
    let a = 1
    let b = 2
    a + b                    // последнее выражение = значение блока
}
```

В блоке-выражении те же правила: newlines разделяют statement'ы,
последнее выражение — значение. Если хочется выкинуть результат:

```nova
let _ = {
    expensive_computation()
    log_event()              // последнее выражение () — блок возвращает ()
}
```

**4. Match-arms:**

```nova
match x {
    Some(v) => v * 2          // одна arm на строку, `,` опционален
    None    => 0
}

match x {
    Some(v) => v * 2,         // `,` тоже работает
    None    => 0,
}
```

`,` или `\n` — оба разделяют arms.

**5. Пустой statement:**

```nova
;                            // ❌ ошибка — `;` без выражения
;;                           // ❌ ошибка
```

Пустые `;` запрещены — это всегда баг.

### Иерархия приоритетов операторов

Стандартная C-подобная, от **низкого** к **высокому**:

| Уровень | Операторы | Ассоциативность |
|---|---|---|
| 1 | `=`, `+=`, `-=`, `*=`, `/=`, `%=`, `&=`, `\|=`, `^=`, `<<=`, `>>=` | right |
| 2 | `..`, `..=` (range) | non-associative |
| 3 | `\|\|` | left |
| 4 | `&&` | left |
| 5 | `==`, `!=` | left |
| 6 | `<`, `<=`, `>`, `>=` | left |
| 7 | `\|` (bitwise or) | left |
| 8 | `^` (bitwise xor) | left |
| 9 | `&` (bitwise and) | left |
| 10 | `<<`, `>>` | left |
| 11 | `+`, `-` (binary) | left |
| 12 | `*`, `/`, `%` | left |
| 13 | `as` (cast) | left |
| 14 | `!` (logical not), `-` (unary), `&` (если будет) | right |
| 15 | `?` (error propagation), `()` (call), `[]` (index), `.` (member) | left |

`?` и `.` — **post-fix**, привязываются крепко. `! -` — **pre-fix**.

### Грамматика (упрощённо)

```
program       = statement*
block         = '{' statement* '}'
statement     = ( decl | expr ) statement-end
statement-end = ';' | NEWLINE | look-ahead '}'

decl          = let-decl | const-decl | type-decl | fn-decl | ...

expr          = assign-expr
assign-expr   = range-expr ( assign-op range-expr )?
range-expr    = or-expr ( '..'|'..=' or-expr )?
or-expr       = and-expr ( '||' and-expr )*
and-expr      = eq-expr ( '&&' eq-expr )*
eq-expr       = cmp-expr ( ('=='|'!=') cmp-expr )*
cmp-expr      = bitor-expr ( ('<'|'<='|'>'|'>=') bitor-expr )*
bitor-expr    = bitxor-expr ( '|' bitxor-expr )*
bitxor-expr   = bitand-expr ( '^' bitand-expr )*
bitand-expr   = shift-expr ( '&' shift-expr )*
shift-expr    = add-expr ( ('<<'|'>>') add-expr )*
add-expr      = mul-expr ( ('+'|'-') mul-expr )*
mul-expr      = cast-expr ( ('*'|'/'|'%') cast-expr )*
cast-expr     = unary-expr ( 'as' type )*
unary-expr    = ('!'|'-') unary-expr | postfix-expr
postfix-expr  = primary ( '.' name | '[' expr ']' | '(' args ')' | '?' )*
primary       = literal | identifier | '(' expr ')' | block | if | match | ...
```

### Сравнение с другими языками

| Язык | `;` обязателен | Newline = разделитель |
|---|---|---|
| **Nova** | ❌ опциональный | ✅ |
| **Go** | ❌ (вставляется автоматически) | ✅ |
| **Python** | ❌ | ✅ |
| **Kotlin** | ❌ | ✅ |
| **Swift** | ❌ | ✅ |
| **TypeScript** | ❌ | ✅ |
| **JavaScript** | ❌ (ASI) | частично |
| **Rust** | ✅ | ❌ |
| **C/C++/Java/C#** | ✅ | ❌ |

Nova следует **современному тренду** (Go/Kotlin/Swift): newline-сигнализация
с предсказуемыми правилами.

### Что отвергнуто

1. **Обязательный `;` (Rust/C-стиль)** — лишний шум, не нужен в
   современных языках.
2. **Indentation-significant** (Python-стиль для блоков) — конфликтует
   с D40 (`{}` для блока).
3. **JavaScript ASI** (Automatic Semicolon Insertion с edge cases) —
   **известный** источник багов (`return\n{...}` возвращает undefined).
   Nova использует **более** простые правила: явное «висящий оператор»,
   «незакрытая скобка», «.method()/?».
4. **Перенос бинарных операторов в начало строки** (некоторые
   стайл-гайды JS) — отвергнут в пользу Go-стиля «оператор в конце».

### Связь с другими решениями

- **D40** (тело функции `=>` или `{}`) — D49 уточняет, что внутри
  `{}` newlines разделяют statement'ы.
- **D43** (trailing lambda) — `)` и `{` на одной строке, частный
  случай D49.
- **D45** (inferred return) — последнее выражение блока становится
  возвратом, как раз через D49 (newline-разделитель).
- **D31, D32** — handler-литералы используют те же правила
  разделения внутри `{...}`.

### Цена

1. **Лексер** должен учитывать контекст для NEWLINE — сложнее
   простого whitespace-skipping. ~100 LOC дополнительно.
2. **Edge cases** требуют документации — особенно про висящие
   операторы и `f\n(x)` ловушку.
3. **IDE/форматер** — должен следовать «оператор в конце строки»,
   иначе пользователи получают синтаксические ошибки от
   автоформатирования.
