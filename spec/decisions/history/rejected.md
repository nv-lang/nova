# Отвергнутые альтернативы

Идеи, которые рассматривались, но не приняты в дизайн Nova. Каждая
альтернатива — с причиной отказа и ссылкой на D-решение, где было
принято финальное правило.

> **Зачем сохранять.** При следующих обсуждениях не возвращаться к
> уже отвергнутым идеям, или вернуться **с новыми аргументами** —
> зная, почему отказались раньше.

## Парадигма и типы

### Классы и наследование (Java/C#/Python)
**Отвергнуто.** Fragile base class, diamond problem, божественные
классы. См. [D1](../01-philosophy.md#d1).

### `trait` + `impl Trait for Type` (Rust)
**Отвергнуто.** Заменено структурным `protocol` ([D42](../02-types.md#d42)).
Структурно — любой тип со совпадающими методами автоматически
удовлетворяет, без `impl`-блоков.

### Bounds на дженериках (`fn f[T: Bound](x T)`)
**Отвергнуто в MVP.** Дженерики без bound'ов; требования описываются
анонимным структурным типом параметра. См. [D15](../02-types.md#d15).

### `dyn Trait` vs `impl Trait` различение (Rust)
**Отвергнуто.** Усложняет язык, не нужно для прикладного.

### Ассоциированные типы и default-методы протоколов
**Отвергнуто в MVP.** Может быть пересмотрено. См. [D42](../02-types.md#d42).

### Trait-наследование, specialization, HKT
**Отвергнуто.** Research-уровень, не нужно для прикладного.

### Sum-type c `|` (Haskell-style)
**Отвергнуто.** Один разделитель — запятая. См. [D17](../02-types.md#d17).

### `type X = { поля }` со знаком равенства для record
**Отвергнуто.** `=` означает «справа выражение типа», для record
формы данных не нужен. См. [D17](../02-types.md#d17),
[history/evolution.md](evolution.md).

### `type X = { методы }` для протоколов
**Отвергнуто.** Заменено keyword `protocol` ([D42](../02-types.md#d42)).
Визуально путалось с record.

### Per-field `mut`/`final` модификаторы
**Отвергнуто.** Слишком многословно (~18 `mut` в большом record).
Заменено на «mut по умолчанию + `readonly`». См.
[D36](../02-types.md#d36).

### `mut self` как параметр метода
**Отвергнуто.** Заменено на `mut @method` модификатор перед `@`. См.
[D35](../03-syntax.md#d35).

### Subtyping для embed/delegation
**Отвергнуто.** `use Parent` — это композиция, не наследование.
`AuditedAccount` не подтип `Account`. См. [D39](../02-types.md#d39).

## Эффекты

### Lowercase эффекты (`throws io async`)
**Отвергнуто.** Эффекты — обычные типы, PascalCase.
См. [D2](../04-effects.md#d2), [D11](../04-effects.md#d11).

### `!throws io async` (с восклицательным знаком)
**Отвергнуто.**
- `!` похож на `not` (логическое отрицание), сбивает чтение.
- Только перед первым эффектом — непоследовательно с несколькими.

См. [D3](../04-effects.md#d3).

### `<throws, io, async>` (Koka-style в угловых скобках)
**Отвергнуто.** Угловые скобки нужны для дженериков, тяжелее читается.

### `~throws io async` (тильда)
**Отвергнуто.** Неудобно набирать на части клавиатур, визуально слабее
слов.

### Атрибуты `@throws @io @async`
**Отвергнуто.** Лишние символы, `@` ассоциируется с метаданными.

### Эффекты выводятся молча, без видимости в публичном API
**Отвергнуто.** Эффекты должны быть **видны на глаз** в публичной
сигнатуре. См. [D28](../04-effects.md#d28).

### `effect X { ... }` keyword
**Отвергнуто.** Заменено на обычный `type X { ... }`. Эффект — частный
случай типа. См. [D18](../04-effects.md#d18).

### `handler X = ...` keyword
**Отвергнуто.** Handler — это значение, не отдельная категория.
Использует record-литерал или handler-лямбду. См.
[D11](../04-effects.md#d11), [D31](../04-effects.md#d31).

### `Throws` без параметра E в публичных функциях
**Отвергнуто.** Бесконтекстный `Throws` уровня «может бросить что-то»
не информативен. В публичных API параметр обязателен. См.
[D25](../04-effects.md#d25), [D28](../04-effects.md#d28).

### `try { ... } catch (e) { ... }` блок
**Отвергнуто.** Используется handler через `with Throws[E] = ... { ... }`
— тот же механизм, что для всех эффектов.

### `try_panic`/`catch panic` в коде
**Отвергнуто.** Panic перехватывается **только runtime'ом** на границе
fiber'а. См. [D13](../08-runtime.md#d13).

## Память

### Opt-in cycle collection (`~T`, `~&T`, `~weak`)
**Отвергнуто.** Заменено managed concurrent GC по умолчанию. См.
[D6](../05-memory.md#d6), отменённый
[D21](../05-memory.md#d21-отменено-opt-in-cycle-collection).

### `&T` / `mut &T` borrow (Rust-style)
**Отвергнуто.** Не нужно при managed GC. Передача объекта = передача
указателя. Slice уже эффективен. Lifetime checker — research-уровень.

### `Weak[T]` тип в stdlib
**Отвергнуто.** Use cases решаются иначе — `Cache[K, V]` с TTL/LRU,
handler-механизм для observer pattern.

### Эффект `Alloc[Cycle]`
**Отвергнуто.** Аллокации в managed heap не отдельный эффект. См.
[D6](../05-memory.md#d6).

### Cycle collector Bacon-Rajan
**Отвергнуто.** Заменён на единый concurrent GC.

### Compile-time анализ циклов через `~T`
**Отвергнуто.** Не нужен — GC справляется.

### Realtime: всегда явный `region { ... }` блок
**Отвергнуто.** Implicit region для тела `Realtime`-функции —
компилятор оборачивает сам, дублирование контракта не нужно.

## Синтаксис

### `:` в type annotations (`x: int`, `let y: u64 = 1`)
**Отвергнуто.** Где компилятор однозначно знает «дальше тип» —
двоеточие опускается. См. [D17](../02-types.md#d17), `spec/syntax.md`.

### Match-arms через `->` (Rust-стиль)
**Отвергнуто.** `->` уже занят для возвращаемого типа. Match arms —
через `=>`. См. [D19](../03-syntax.md#d19).

### Тело функции через `=` (OCaml/F#-стиль)
**Отвергнуто.** `=` ассоциируется с присваиванием. Тело — через `=>`.
См. [D20](../03-syntax.md#d20), [D22](../03-syntax.md#d22).

### Дженерики через `<T>` (Rust/TS/Java-стиль)
**Отвергнуто.** Парсер-неоднозначность (`sort<int>(xs)` vs
`(sort < int) > (xs)`). Используется `[T]`. См.
[D16](../03-syntax.md#d16).

### Type suffixes `100u32`, `1.5f32`
**Отвергнуто.** Шум, дублирует inference. Используется `as`-cast
(`100 as u32`). См. [D44](../03-syntax.md#d44).

### Implicit `it` в лямбдах (Kotlin)
**Отвергнуто.** Нелокальное поведение. Параметр всегда именован.
См. [D43](../03-syntax.md#d43).

### Опциональные `()` в trailing lambda (`spawn { body }`)
**Отвергнуто.** `()` обязательны (`spawn() { body }`). Иначе
парсер не различает с record-литералом. См. [D43](../03-syntax.md#d43).

### Custom-операторы (`:+`, `<>`, `>>=` Scala-стиль)
**Отвергнуто.** Известный источник нечитаемого кода. Только
фиксированный набор. См. [D46](../03-syntax.md#d46).

### `&&` / `||` для overloading
**Отвергнуто.** Short-circuit семантика должна быть строгой и
предсказуемой. См. [D46](../03-syntax.md#d46).

### `cond && throw X` как guard-pattern
**Отвергнуто.** Перегрузка `&&` (bool / control flow / side-effect) —
анти-паттерн. Используется `validate()?` или `if cond { throw X }`.

### Indentation-significant блоки (Python-стиль)
**Отвергнуто.** `{}` обязательны для блоков. `=>` принимает только
одно выражение. См. [D40](../03-syntax.md#d40).

### Обязательный `;` в конце statement (C/Rust-стиль)
**Отвергнуто.** Newline разделяет statement'ы; `;` опционален. См.
[D49](../03-syntax.md#d49).

### Обязательный `return` в block-body
**Отвергнуто.** Последнее выражение блока — значение функции
(Rust/Scala-стиль). `return` — для раннего выхода. См.
[D23](../03-syntax.md#d23).

### `ret` вместо `return`
**Отвергнуто.** Отступление от 12 мейнстрим-языков без веской
причины. Экономия 3 символов не оправдывает.

### `Void`
**Отвергнуто.** Тип «без значения» — `()` (unit). См.
[D20](../03-syntax.md#d20).

## Видимость и модули

### `pub` (Rust-style)
**Отвергнуто.** Заменено на `export` для симметрии с `import`. См.
[D5](../07-modules.md#d5).

### Видимость по регистру (Go: Capital = public)
**Отвергнуто.** Мешает естественному именованию. См.
[D5](../07-modules.md#d5).

### `public`/`private`/`protected`/`package` (Java)
**Отвергнуто.** Overkill, никто не помнит правила. Два уровня
достаточно. См. [D5](../07-modules.md#d5), [D47](../07-modules.md#d47).

### `private` keyword
**Отвергнуто.** Конфликтует с философией Nova «явно публичное».
Default — приватный, явный — `export`. См. [D47](../07-modules.md#d47).

### Per-field `export` для record
**Отвергнуто в MVP.** Все поля публичны, convention `_prefix` для
приватных. Может быть пересмотрено в будущем. См.
[D47](../07-modules.md#d47).

### Wildcard import `import x.*`
**Отвергнуто.** Скрывает источник, ломает локальность контекста.
См. [D29](../07-modules.md#d29).

### Циклические импорты
**Отвергнуто.** Compile error. Решается выделением в третий модуль.
См. [D29](../07-modules.md#d29).

### Несколько модулей в одном файле
**Отвергнуто.** Один файл = один модуль. См. [D29](../07-modules.md#d29).

### `#include` или препроцессор
**Отвергнуто.** Только `import` через семантику модулей.

### Static-поля и static-переменные
**Отвергнуто.** Глобальное mutable state — через handler. См.
[D41](../08-runtime.md#d41).

### Companion-object (Kotlin)
**Отвергнуто.** То же что static, просто в обёртке.
См. [D41](../08-runtime.md#d41).

### Lazy static (Rust `lazy_static!`)
**Отвергнуто.** Скрытое глобальное состояние с инициализацией. Если
нужна ленивость — handler с lazy полем.

## Operator overloading и литералы

### Operator overloading через protocol/trait
**Отвергнуто.** Через имена методов (`@plus`, `@eq`) — структурно,
без `impl Add for T`. См. [D46](../03-syntax.md#d46).

### Tagged template через макросы (Rust `sql!("...")`)
**Отвергнуто.** Tagged template — обычная функция с особым синтаксисом
вызова. См. [D48](../03-syntax.md#d48).

### Префикс через символ (`s"..."`, `r"..."` Scala-стиль)
**Отвергнуто.** Ограничивает имя одним символом, не позволяет
user-defined теги. См. [D48](../03-syntax.md#d48).

### Implicit tag для строки (без префикса)
**Отвергнуто.** Создаёт ambiguity со строковыми литералами. Tag
всегда явный. См. [D48](../03-syntax.md#d48).

## Concurrency

### `async` / `await` keywords (Rust/JS-стиль)
**Отвергнуто.** `Async` — обычный эффект, без `Future<T>` в типе и
без `await` в коде. Цвет функции исчезает. См.
[D14](../06-concurrency.md#d14).

### `Future<T>` / `Promise<T>` в типе возврата
**Отвергнуто.** Тип остаётся `T`, эффект `Async` в сигнатуре.

### Stackless coroutines (Rust state machines)
**Отвергнуто.** Экономят память, но требуют `Pin`/`Send`/`Sync`
бойлерплейта; не подходят для AI-кодинга.

### OS threads as default
**Отвергнуто.** Слишком тяжёлые для миллионов задач.

### Custom Promise как магия компилятора
**Отвергнуто.** `Promise[T]` если нужен — пишется обычным кодом
через handler-обёртку.

## Контракты

### «Всегда статическая проверка, без runtime-fallback»
**Отвергнуто.** Сделало бы контракты обязательно тотальными,
половину прикладного кода невозможно было бы аннотировать. См.
[D24](../09-tooling.md#d24).

### «Только runtime-проверка»
**Отвергнуто.** Теряется ценность контрактов в release. Превращает
контракты в обычные `assert`.

### Фиксация конкретного SMT-движка в дизайне
**Отвергнуто.** Дизайн фиксирует **класс** движка, имя — выбор
реализации.
